## Problem Statement

This repo has 4 local skills (workflow, contracts, skill-author, system-health), 15+ templates, and extensive tooling, but no documentation that captures the *why*. CLAUDE.md has conventions. README.md has install steps. SKILL.md files describe what each skill does. None of these explain the design philosophy, the tradeoffs that shaped the architecture, or how the pieces relate to each other.

The portfolio repo is worse: 30+ design docs from office hours sessions live in `~/.gstack/projects/`, but the repo itself has no overview doc.

Existing tools handle format enforcement (`/contracts`) and post-ship updates (`/document-release`), but neither can:
- Generate narrative documentation from scratch for a repo with zero philosophy docs
- Synthesize the "why" behind patterns and decisions into a coherent story
- Detect when docs are stale because code changed under them
- Check that all docs in a project tell a coherent, non-contradictory story

The primary audience is future-self: coming back to a repo in 6 months and needing to remember why decisions were made.

## What Makes This Cool

A "repo memory compiler" (Codex's framing). Not a doc generator that produces boilerplate, but a skill that reads the codebase, git history, existing docs, and prior design docs, then synthesizes a narrative that captures intent. Combined with a claims sidecar that maps each doc assertion to code evidence, enabling diff-based staleness detection.

The loop: `repo -> generate meaningful philosophy doc -> diff detects code changes -> flag stale sections -> human reviews and regenerates`. Docs that know when they're wrong.

## Constraints

- Must live in `skills/docs/` following the existing skill directory structure
- SKILL.md with YAML frontmatter (name, description, version, allowed-tools)
- Absorbs `/contracts` template enforcement logic (owned by this repo)
- Composes with gstack's `/document-release` as an external dependency (not absorbed)
- Optimized for user's repos and conventions (CLAUDE.md, doc triplets, gstack patterns), not universal
- No new runtime dependencies (pure markdown skill, bash scripts if needed)
- Must work in the existing skills-deploy pipeline

## Premises

1. **New doc type needed** — PHILOSOPHY.md / OVERVIEW.md are genuinely different artifacts from CLAUDE.md and README. They capture intent, tradeoffs, and design rationale.
2. **Diff-based staleness detection** — Watch git diffs against a claims sidecar (`.docs/claims.json`) that maps doc sections to evidence files. When evidence files change, flag the corresponding doc sections as potentially stale.
3. **Absorb contracts, compose document-release** — Own the template enforcement logic from `/contracts` (it's in this repo anyway). Call gstack's `/document-release` for post-ship updates rather than reimplementing.
4. **My repos only** — Optimized for repos that use CLAUDE.md, doc triplets, and gstack conventions. Not designed for universal portability.
5. **Flag-only staleness** — Detect and report stale sections. Human decides what to regenerate. No auto-healing without explicit opt-in.

## Cross-Model Perspective

Codex (independent cold read) reframed the skill as a "repo memory compiler" rather than a doc generator:

- **Coolest version:** Maintain a machine-readable claim graph where each doc claim maps to evidence in code, config, file structure, and commit history. ## Approaches Considered

### Approach A: Narrowest Wedge — /docs init
Single subcommand generating PHILOSOPHY.md with claims sidecar. Proves the generation loop. Ships in ~15 min. Completeness: 4/10. Too narrow for the stated vision.

### Approach B: Full Doc Intelligence (CHOSEN)
4 subcommands (init, check, enforce, sync), claims sidecar. Absorbs contracts, composes with document-release. Complete vision without over-engineering. ~30 min. Completeness: 8/10.

### Approach C: Layered Architecture
Plugin system with doc-type providers. Most extensible (9/10) but the "my repos only" constraint makes the extensibility unnecessary complexity.

## Recommended Approach

**Approach B: Full Doc Intelligence** with 4 subcommands and two supported doc types (philosophy, overview).

### Skill Structure

```
skills/docs/
  SKILL.md           # Router + preamble + subcommand dispatch
  init.md            # Generate docs from scratch (PHILOSOPHY.md, OVERVIEW.md)
  check.md           # Coherence checking + staleness detection
  enforce.md         # Absorbed contracts logic (template enforcement)
  sync.md            # Compose with /document-release for post-ship updates
  CHANGELOG.md       # Version history
```

**allowed-tools:** `Bash, Read, Write, Edit, Glob, Grep, AskUserQuestion`. The `init` and `sync` subcommands need Write/Edit to create and update doc files and the claims sidecar. The `enforce` subcommand inherits these permissions but preserves the "report first, fix only on request" behavior from the original `/contracts` — it never writes without explicit user confirmation via AskUserQuestion.

### Subcommands

**`/docs init [type]`** — Generate documentation from scratch
- Reads: codebase structure, CLAUDE.md, README.md, git log, existing docs, design docs in `~/.gstack/projects/`
- Generates: PHILOSOPHY.md (default) or OVERVIEW.md
- Outputs: The doc file + `.docs/claims.json` sidecar mapping sections to evidence
- **Re-run behavior:** If the target doc already exists, generate a `.docs/claims.json` for it without overwriting. Present a diff of what would change and ask the user whether to update specific sections, append new sections, or skip. Never silently overwrite existing narrative docs.
- PHILOSOPHY.md schema:
  - Why this repo exists
  - Design principles and tradeoffs
  - What it intentionally does NOT optimize for
  - Key patterns and conventions (with rationale)
  - How to extend without breaking its character
  - Dependencies and assumptions
  - Failure modes and maintenance risks
- OVERVIEW.md schema:
  - What this project is (one paragraph)
  - Architecture at a glance
  - Key components and how they relate
  - Getting started (from the perspective of someone who will work on this, not just use it)
  - Current state and known limitations

**`/docs check`** — Coherence and staleness checking
- Reads: All docs in the repo + `.docs/claims.json`
- Staleness: Compares evidence files in claims.json against `git diff <stored-sha>..HEAD -- <evidence-path>` for each evidence entry. If any evidence file changed since the stored commit, the corresponding doc section is flagged as potentially stale. If an evidence file was deleted, always flag as stale.
- Coherence (v1 — mechanical checks only): Broken internal links between docs, duplicate section headers across docs, conflicting version numbers, references to files/functions that no longer exist. LLM-based semantic coherence checking (contradictions, terminology drift) is deferred to v2.
- Output: Report with staleness flags and coherence issues, grouped by severity.

**`/docs enforce [path]`** — Template enforcement (absorbed from /contracts)
- Reads: Doc triplets (PRD + ARCHITECTURE + TEST-SPEC) against templates
- Checks: Structural validity, cross-doc traceability, required sections
- Template resolution: Same fallback chain as current /contracts ($REPO_ROOT/templates/ -> ~/.claude/spec/templates/ -> ~/.claude/templates/)
- This is the existing /contracts check logic, owned and maintained here

**`/docs sync`** — Post-ship documentation updates
- Composes with gstack's `/document-release` skill
- Reads the recent diff, identifies which docs need updating
- Delegates the actual update work to /document-release
- Then runs `/docs check` to verify coherence after updates

### Claims Sidecar (`.docs/claims.json`)

```json
{
  "version": 1,
  "generated_at": "2026-04-11T00:00:00Z",
  "generated_commit": "949ed38",
  "docs": {
    "PHILOSOPHY.md": {
      "sections": {
        "Why this repo exists": {
          "evidence": ["CLAUDE.md", "README.md", "skills-catalog.json"],
          "commit": "949ed38"
        },
        "Design principles": {
          "evidence": ["CLAUDE.md", "templates/GENERATION-GUIDE.md"],
          "commit": "949ed38"
        },
        "Key patterns": {
          "evidence": ["skills/workflow/SKILL.md", "skills/contracts/SKILL.md", "templates/"],
          "commit": "949ed38"
        }
      }
    }
  }
}
```

Evidence is tracked at file-level only (not line ranges). Line-level tracking is too brittle — any edit to the file, even irrelevant ones, triggers a false positive. File-level is the right granularity for narrative docs.

Evidence paths can be files or directories. When `git diff` runs against a directory path (e.g., `templates/`), it flags any change to any file within that directory. Use directory-level evidence sparingly — only when the entire directory's structure is relevant to the claim (e.g., the "Key patterns" section references the full templates directory because the set of templates IS the pattern). For most sections, list specific files to avoid false positives from unrelated changes within a directory.

When `/docs check` runs, it runs `git diff <commit>..HEAD -- <evidence-path>` for each section. If any evidence file changed, the section is flagged as potentially stale. If an evidence file was deleted, always flag as stale and warn.

### Supported Doc Types

v1 ships with two doc types:

| Type | Output | When to use |
|------|--------|-------------|
| `philosophy` | PHILOSOPHY.md | Capture design rationale, tradeoffs, principles |
| `overview` | OVERVIEW.md | Project architecture, components, getting started |

Each type is defined inline in init.md with its schema and evidence-gathering strategy. Adding a third type means adding another section to init.md following the same pattern. No abstraction layer needed.

## Migration Plan

The `/workflow` skill currently depends on `/contracts` (catalog entry lists it in `depends`). When `/docs enforce` absorbs `/contracts` logic:

1. **Phase 1 (v0.1.0):** Ship `/docs` with enforce subcommand. Keep `/contracts` as-is. Both work independently.
2. **Phase 2 (v0.2.0):** Update `/workflow` review.md and ship.md to check for `/docs enforce` first, fall back to `/contracts`. Update catalog `depends` from `contracts` to `docs`.
3. **Phase 3 (v0.3.0):** Remove `skills/contracts/` directory and catalog entry. `/docs enforce` is the canonical location.

Each phase ships as a separate PR. No breaking changes between phases.

## Open Questions

1. **Where does `.docs/claims.json` live?** In the repo root (committed, visible to collaborators) or in `~/.gstack/projects/` (local-only, invisible)? Repo root makes staleness checking work across machines. Local keeps the repo clean.
2. **Should coherence checking expand to LLM-based semantic analysis in v2?** v1 handles mechanical checks (broken links, duplicate headers, conflicting versions). Semantic checks (contradictions, terminology drift) would require prompting the LLM to compare doc sections, which is powerful but slower.
