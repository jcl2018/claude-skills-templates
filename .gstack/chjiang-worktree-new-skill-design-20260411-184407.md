## Problem Statement

`/skill-author` is a 6-stage pipeline (intake, scaffold, author, check, ship, install) that wraps bash scripts into a guided workflow. The actual workflow for creating skills is: `/office-hours` (design), implement (write SKILL.md directly), `/ship` (commit + PR). The skill-author pipeline duplicates this with extra ceremony. Same pattern as the /workflow and /contracts sunset in sessions 11-12.

The bash scripts it wraps (create-skill.sh, skill-design.sh, skill-check.sh, skill-version.sh, skill-ship.sh) also duplicate what CC does natively when guided by CLAUDE.md instructions.

## What Makes This Cool

Convergence. Seven skills became two, two became zero plus templates, now the last standalone skill pipeline goes the same way. The repo's real product is templates + conventions + docs skill. Everything else is wrappers CC doesn't need.

## Constraints

- Must not break validate.sh (the core validation tool)
- Must preserve template knowledge (SKILL.md frontmatter format, catalog schema)
- test.sh integration tests that exercise deleted scripts need rewriting
- The doc-SKILL-DESIGN.md template stays in templates/
- skills-deploy pipeline continues to work

## Premises

1. /skill-author adds no value over /office-hours + implement + /ship
2. The bash lifecycle scripts also go (CC does this natively with CLAUDE.md guidance)
3. Templates stay, delivered via skills-deploy
4. CLAUDE.md gets a "Creating a new skill" section documenting frontmatter, catalog format, and validation rules

## Recommended Approach

**Approach C: Full cleanup + CLAUDE.md guide**

### What to delete
- `skills/skill-author/` directory (SKILL.md, DESIGN.md, CHANGELOG.md)
- `scripts/skill-design.sh`
- `scripts/create-skill.sh`
- `scripts/skill-check.sh`
- `scripts/skill-version.sh`
- `scripts/skill-ship.sh`
- `scripts/skill-migrate.sh`
- `skills-catalog.json` entry for skill-author
- Template references in catalog for skill-author (doc-SKILL-DESIGN.md, GENERATION-GUIDE.md, etc.)

### What to keep
- `templates/doc-SKILL-DESIGN.md` (template file stays, but catalog ownership moves)
- `templates/GENERATION-GUIDE.md` and other generation guides
- `scripts/validate.sh` (core validation, does not call deleted scripts)
- `scripts/test.sh` (needs rewriting to remove integration tests for deleted scripts)
- `scripts/skills-deploy` (template deployment, unaffected)
- `scripts/collection-version.sh` (collection versioning)
- `scripts/lib.sh` (shared utilities)
- `scripts/doctor.sh`, `scripts/lint-skill.sh`, `scripts/deps.sh`, `scripts/generate-readme.sh`, `scripts/sync-upstream.sh`, `scripts/setup-hooks.sh`, `scripts/setup.sh`, `scripts/test-deploy.sh` (ancillary scripts, unaffected by this change)

### What to add
- CLAUDE.md section: "Creating a new skill" with:
  - Directory structure: `skills/{name}/SKILL.md`
  - Required YAML frontmatter: name, description, version, allowed-tools
  - Catalog entry format (JSON schema with example)
  - Validation: run `./scripts/validate.sh` before committing
  - Template: use `templates/doc-SKILL-DESIGN.md` for DESIGN.md

### What to rewrite
- `test.sh`: remove integration tests for skill-design.sh, create-skill.sh, skill-check.sh, skill-version.sh, skill-ship.sh. Keep validate.sh + smoke tests.
- `CLAUDE.md`: update "3 custom skills" to "2 custom skills" (line 5), remove skill-author from "Quick start" and "Scripts reference"
- `README.md`: update skills table (now 2 skills: docs, system-health)
- `CONTRIBUTING.md`: remove references to skill-design.sh, create-skill.sh, skill-check.sh, skill-version.sh, skill-ship.sh. Replace with CLAUDE.md-guided workflow.

### Template catalog reassignment
Templates currently owned by skill-author in the catalog:
- `doc-SKILL-DESIGN.md`, `GENERATION-GUIDE.md`, `architecture-GENERATION-GUIDE.md`, `prd-GENERATION-GUIDE.md`, `test-spec-GENERATION-GUIDE.md`

(Note: TRACKER-TEMPLATE.md is NOT in skill-author's catalog entry. It's in the "templates" catalog entry.)

**Decision:** Move these 5 templates to the existing "templates" catalog entry. That entry already exists and serves as the catch-all for shared templates. This is the simplest path, no new abstractions needed.

Generation guide templates (GENERATION-GUIDE.md, prd-GENERATION-GUIDE.md, etc.) are general-purpose meta-templates for doc generation. They stay. They're useful for any doc-first workflow, not just /workflow track.
