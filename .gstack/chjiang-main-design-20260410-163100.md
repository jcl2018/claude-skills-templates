# Design: Claude Skills Templates — Skill Workbench

Status: APPROVED

## Problem Statement

Claude Code skills are just SKILL.md files with optional scripts — there's no standard way to author, validate, test, or distribute custom skills across projects. This repo has 8 custom skills and 20+ templates already, but no structure to make them installable by other repos, no scaffolding for creating new skills, and no validation that skills conform to their templates. The goal is to turn this repo into a skill development workbench: a place to build skills like real software (with tests, contracts, and a distribution format) that other repos can pull from.

## What Makes This Cool

The inner loop. Right now, creating a Claude Code skill means writing a SKILL.md and hoping it works. This repo would make skill development feel like actual software engineering: scaffold a new skill from a template, validate it against contracts, run smoke tests, and publish it so any repo can install with one command. Skills that ship with evidence, not just prompts.

## Terminology

- **Doc triplet:** A set of three documents (PRD.md, ARCHITECTURE.md, TEST-SPEC.md) that together specify a feature. Each is generated from a corresponding template. The triplet pattern ensures every feature has requirements, architecture, and test coverage defined together.
- **Skill directory:** A folder under `skills/` containing at minimum a SKILL.md file. May also contain skill-specific templates, generation guides, or helper scripts.
- **Skills catalog:** `skills-catalog.json` at the repo root. The authoritative registry of all skills in this repo, their metadata, and their status.

## Constraints

- Must work as a Claude Code plugin (native install via `/plugin marketplace add`). **Fallback:** If the plugin.json schema differs from what's documented here, the repo still works via direct git clone. The plugin manifest is additive, not load-bearing for the authoring workflow.
- No custom distribution infrastructure — git pull is the update mechanism
- Follow gstack's conventions (SKILL.md structure, skill directory layout) but remain fully independent
- No build system required — skills are Markdown + shell scripts. Shell scripts use only POSIX utilities plus `grep`, `sed`, and `awk` for YAML frontmatter parsing (no external YAML parser dependency).
- Must be usable by one person (the author) before it's useful to a community

## Premises

1. This repo should be structured as a Claude Code plugin for native distribution
2. The real value is the skills themselves and the authoring/testing workflow, not distribution plumbing
3. No deploy.sh — git pull handles updates for both the author's machine and consuming repos
4. Follow gstack's pattern conventions; use skills-catalog.json as the registry
5. The upstream-skills architecture (catalog + clean-overwrite + version tracking) is the distribution model

## Recommended Approach

**Approach B: Skill Workbench** — delivers the authoring inner loop without over-building for a community that doesn't exist yet. Can grow into C.

### What to build

**1. Plugin manifest (`plugin.json`)**

Standard Claude Code plugin format. Declares the repo as installable. **Auto-generated from skills-catalog.json** — never manually edited. A small jq command in `scripts/generate-plugin.sh` reads active skills from the catalog and writes plugin.json:
```json
{
  "name": "claude-skills-templates",
  "version": "0.1.0",
  "description": "Work lifecycle pipeline, doc contract enforcement, and skill authoring workbench for Claude Code",
  "skills": [
    "skills/work",
    "skills/work-track",
    "skills/work-implement",
    "skills/work-review",
    "skills/work-ship",
    "skills/system-health",
    "skills/align-feature-contract",
    "skills/test-align-contract"
  ]
}
```
Note: The exact plugin.json schema needs verification against Claude Code docs. The authoring workflow works regardless of plugin format. See Open Question #1.

**2. Skills catalog (`skills-catalog.json`)**

The authoritative registry. `plugin.json` is derived from the catalog (lists skill paths for the plugin system), but `skills-catalog.json` is the source of truth. `validate.sh` validates against the catalog only.

Each skill entry declares:
- name, version, description
- source: "local" (authored here) vs "upstream" (copied from gstack)
- depends: object with two keys: `skills` (list of skill names from this catalog) and `tools` (list of external tool names like "jq", "git"). `validate.sh` checks only that skill-name dependencies exist in the catalog. External tool deps are documentation-only.
- portability: "standalone" (the skill's SKILL.md is self-contained — no references to other skills, no assumptions about work-item state) vs "pipeline" (the skill reads output from or assumes prior execution of another skill in this repo). Decision rule: if the SKILL.md references another skill by name or reads files created by another skill, it's "pipeline."
- files: list of paths relative to repo root owned by this skill (SKILL.md, scripts, templates)
- templates: list of template file names in `templates/` that this skill uses (e.g., `["doc-PRD.md", "doc-ARCHITECTURE.md"]`). Validated against the `templates/` directory.
- status: active | experimental | deprecated

Example catalog entry:
```json
{
  "name": "work-track",
  "version": "0.1.0",
  "description": "Work item creation and scaffolding",
  "source": "local",
  "depends": {
    "skills": [],
    "tools": ["jq"]
  },
  "portability": "standalone",
  "files": ["skills/work-track/SKILL.md"],
  "templates": ["doc-PRD.md", "doc-ARCHITECTURE.md", "doc-TEST-SPEC.md", "tracker-feature.md"],
  "status": "active"
}
```

**3. Per-skill manifest format**

Each skill directory gets structured metadata. The SKILL.md frontmatter already has `name` and `description`. All frontmatter values are single-line scalars (no YAML lists in SKILL.md — lists live in skills-catalog.json where they're parsed as JSON). Frontmatter fields:
- `name` — required, must match catalog entry
- `description` — required, one-line summary
- `version` — optional in v1 (catalog tracks version authoritatively). Add when the skill stabilizes.

**4. Scaffold script (`scripts/create-skill.sh`)**

```bash
./scripts/create-skill.sh my-new-skill
```
Guards: exits with code 1 and prints error to stderr if skill name already exists in catalog or as a directory under `skills/`.

Creates:
- `skills/my-new-skill/SKILL.md` — skeleton with frontmatter (name, description placeholder)
- `docs/my-new-skill/PRD.md` — from `templates/doc-PRD.md`
- `docs/my-new-skill/ARCHITECTURE.md` — from `templates/doc-ARCHITECTURE.md`
- `docs/my-new-skill/TEST-SPEC.md` — from `templates/doc-TEST-SPEC.md`
- Appends new entry to `skills-catalog.json` using the example format from section 2 (source: "local", status: "experimental", empty depends, portability: "standalone")
- Regenerates `plugin.json` from catalog (see section 1)

**5. Validation script (`scripts/validate.sh`)**

Pure shell script (no Claude invocation). Uses grep/sed for YAML frontmatter (single-line `key: value` scalars only — no nested YAML, no lists; lists live in the JSON catalog) and jq for JSON (catalog and plugin.json).

**Error checks** (exit 1 on any failure):
- Every skill in skills-catalog.json has a SKILL.md on disk
- Every SKILL.md has required frontmatter fields (name, description)
- Every skill's `templates` list in the catalog has those files present in `templates/`
- No orphan skill directories (directories under `skills/` with no matching catalog entry)
- Doc triplets in docs/ have all three files (PRD.md, ARCHITECTURE.md, TEST-SPEC.md) and each has a `type` frontmatter field
- plugin.json `skills` array matches the catalog's active skill paths (or regenerate if stale)
- All declared skill-name dependencies (from `depends.skills`) exist as entries in the catalog. External tool deps (`depends.tools`) are not checked.

**Warning checks** (printed with WARNING prefix, do not affect exit code):
- Template sync: `skills/align-feature-contract/PRD-TEMPLATE.md` content matches `templates/doc-PRD.md` (and same for ARCHITECTURE and TEST-SPEC variants). Warns on drift but does not fail — the skill-internal copies may intentionally diverge.
- Orphan doc triplet directories: directories under `docs/` that don't match any skill name in the catalog. These may be feature-family docs (like `docs/infrastructure/`) that aren't skills, so this is advisory only.

Exit codes: 0 = all error checks pass (warnings may be present), 1 = one or more error checks failed. Each check prints `PASS`, `FAIL: reason`, or `WARNING: reason`.

**6. Smoke test runner (`scripts/test.sh`)**

`test.sh` is a superset of `validate.sh` — it runs `validate.sh` first, then adds structural content checks. Same tooling: shell + jq, YAML frontmatter parsed as single-line `key: value` scalars only.

**Additional checks beyond validate.sh:**
- No duplicate skill names in catalog
- All SKILL.md files have parseable YAML frontmatter (text between first two `---` lines contains at least `name:` and `description:`)
- Doc triplets have required sections: PRD.md has `## Problem Statement`, ARCHITECTURE.md has `## Overview` or `## Architecture`, TEST-SPEC.md has `## Test Matrix`

Exit codes: 0 = validate.sh passes AND all additional tests pass, 1 = any failure. Output format: one line per test, `OK` or `FAIL: reason`.

**7. Template consolidation**

Currently templates exist in two places: `templates/` and `spec/templates/`. Consolidate to `templates/` as the single canonical location, using the `doc-` prefix naming convention (doc-PRD.md, doc-ARCHITECTURE.md, etc.) which aligns with the align-feature-contract skill's template resolution order. Remove `spec/templates/` after migration.

**Template copies inside skill directories:** The templates inside `skills/align-feature-contract/` (PRD-TEMPLATE.md, ARCHITECTURE-TEMPLATE.md, TEST-SPEC-TEMPLATE.md) are kept there because the align-feature-contract SKILL.md references them as a fallback resolution path (see its Preconditions section). These are the SAME templates as `templates/doc-*.md` — just with legacy naming. Decision: keep the skill-internal copies for now (the skill's resolution logic expects them), and add a validate.sh check that they stay in sync with the canonical `templates/` versions.

**8. CLAUDE.md for the repo**

Project-level CLAUDE.md with:
- Repo purpose and quick-start (clone, validate, create a skill)
- Skill routing rules: map user intents to skills (e.g., "scaffold a work item" -> /work-track, "check doc quality" -> /align-feature-contract)
- Repo conventions: skill directory structure, template naming (`doc-` prefix), catalog format, frontmatter requirements
- Principles: P1 (repo is authority), P2 (upstream skills are copies not forks), P4 (catalog is source of truth for skill inventory). P3 (personal-native: no $AI_CONTENT_DIR indirection) is inherited from home-setup but not restated here — it's an environment constraint, not a repo convention.

## Open Questions

1. **Plugin format details:** The plugin.json schema shown here is a best-effort based on search results. Verify against Claude Code docs before publishing. The authoring workflow (scaffold, validate, test) works regardless of plugin format — plugin.json is additive distribution, not a prerequisite.
2. **Skill versioning:** Start with 0.1.0 for all skills (single repo version). Version skills independently only when a consumer needs to pin a specific skill version — that complexity isn't justified until there are downstream users. Track in catalog but don't enforce semver semantics in v1.
3. **Upstream consumption:** This repo is designed to BE an upstream (other repos pull from here). The catalog's `source` field already distinguishes local vs upstream skills. A downstream repo would clone/submodule this repo and reference skill paths. Detailed downstream consumption docs deferred until a second repo actually consumes these skills.

## Success Criteria

- `./scripts/create-skill.sh test-skill` produces a valid skill directory + doc triplet, and the new skill appears in skills-catalog.json
- `./scripts/validate.sh` exits 0 on the current repo state (all checks pass)
- `./scripts/test.sh` exits 0 (all structural smoke tests pass)
- CLAUDE.md exists and contains: repo purpose, quick-start instructions, skill routing rules, and conventions — a developer can read it and create a new skill without asking questions
- Another repo can clone this repo and reference a skill path to use it (plugin system install is a stretch goal pending format verification)

## Distribution Plan

- Primary: Claude Code plugin system (`/plugin marketplace add username/claude-skills-templates`)
- Secondary: Direct git clone for consumers who want the full repo (templates, docs, scripts)
- Updates: git pull from origin — no custom sync tooling needed
- CI/CD: GitHub Actions for running validate.sh and test.sh on every PR (future, not in v1)
