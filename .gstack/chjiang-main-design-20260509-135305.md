## Problem Statement

TODOS.md:20 captures a deferred decision: build a `/personal-pipeline` orchestrator wrapping the three pipeline skills (`/scaffold-work-item`, `/implement-from-spec`, `/qa-work-item`) shipped in F000010. The original gate was "2+ weeks of real use." That gate has not literally been met (1 calendar day), but the behavioral proxy has: F000011 + F000012 + F000013 all shipped through the manual three-step pipeline in the past week, so the user has signal about which gates matter.

Two recurring frictions surfaced through F000011-F000013:

1. **Three-keystroke ergonomics.** The user invokes `/scaffold-work-item <doc>`, then `/implement-from-spec <dir>`, then `/qa-work-item <dir>` separately. Manual context switching, no enforced gate between phases.
2. **Trust in upstream skill self-checks is uneven.** `/scaffold-work-item` runs `/personal-workflow check` at boundary (Step 9), but TODOS.md:26 documents a real bug: the Step 5 idempotency hole means scaffold writes a duplicate `F000011` next to `F000010` and the boundary check passes vacuously because it's looking at the new dir. The orchestrator can fail-close on this by running an *independent* gate that scans the source design doc → scaffolded dir mapping.

The user explicitly raised both concerns: file-only boundaries between phases, and a quality reviewer between each step.

## What Makes This Cool

A single keystroke runs the full personal-workflow phase 2-4 loop with subagent isolation per phase. Each subagent gets a tiny prompt (a few hundred tokens, mostly file paths) and returns a 1-2 sentence verdict. The orchestrator stays under ~5K tokens of context across all three phases. Independent quality gates between phases catch upstream-skill bugs before they propagate.

The orchestrator is structurally different from the deleted /workflow because /workflow was a prose wrapper around what Claude Code does naturally given CLAUDE.md rules. This orchestrator's value-add is `Agent` tool dispatch with `subagent_type` per phase — fresh context, file-paths-only handoff, halt-on-red gates. That's plumbing, not prose.

## Constraints

- **Personal-workflow only.** The orchestrator does not extend to company-workflow / work-copilot (deprecated path).
- **No logic duplication.** Orchestrator dispatches to `/scaffold-work-item`, `/implement-from-spec`, `/qa-work-item` via Agent tool — does not re-implement their internals. If those skills change, orchestrator inherits.
- **File-only handoff between phases.** Subagents communicate only through filesystem (handoff docs in `work-items/`). No in-memory state crossing.
- **Independent inter-step quality gates.** Orchestrator runs its own checks between phases — does not trust upstream skill self-checks.
- **Human gates at decision points only.** AskUserQuestion at: scaffold approval, taste forks during implement, red/ambiguous QA results, ship-readiness.
- **Existing infra only.** `skills-deploy install` deploys it. `validate.sh` + `test.sh` covers it. No new pipelines.
- **Sunset criterion baked in.** If the orchestrator is skipped or produces worse results 3 times in 5 runs, delete it. Avoids /workflow's slow death.

## Premises (locked with user 2026-05-09)

1. **PHILOSOPHY override is structural.** The orchestrator uses `Agent` tool with `subagent_type` for fresh-context phase isolation. If implemented as inline prose, it IS the /workflow anti-pattern and should be deleted. The structural distinction is the load-bearing premise.
2. **File-only handoff between subagents; orchestrator-as-broker.** Subagents read inputs from disk with fresh context, write outputs to disk, return a short verdict (file path + 1-2 sentences). The orchestrator passes file paths between phases as Bash arguments — that is the broker layer, not in-memory state crossing *between* subagents. The premise constrains subagent-to-subagent communication, not orchestrator-to-subagent.
3. **Independent inter-step quality gates** (user-added). Orchestrator runs:
   - **Pre-scaffold idempotency check:** read source design doc footer (`**Status: SCAFFOLDED → <path>**` line; Step 12 of scaffold writes this). If present and path exists → reuse, skip Phase 1. If absent → dispatch Phase 1. Decouples from the scaffold skill's known Step 5 hole (TODOS.md:26) — the orchestrator's check uses a signal scaffold already maintains.
   - **Post-scaffold gate:** `/personal-workflow check {dir}` (fresh-context structural re-verification) + footer-write-back confirm (the design doc footer must have been updated; if not, partial-write halt with manual-cleanup AUQ).
   - **Post-implement gate:** `/personal-workflow check` + `scripts/validate.sh`. Does NOT run `scripts/test.sh` in v1 (slow; revisit in v2 if real-world runs catch issues only test.sh would have caught). Gate scripts exit code != 0 → halt with output captured to tracker journal; user re-runs orchestrator to retry.
   - **Post-QA gate:** smoke green parseable from tracker (`[smoke-pass]` journal entry); E2E findings parseable (`[qa-pass]` or `[qa-red]` entries); Phase 2 gates green where qa-work-item is supposed to mark them. Red → AskUserQuestion with the specific findings, halt by default.
4. **Signal sufficiency.** Three real pipeline runs (F000011-F000013) provide enough behavioral signal about which gates matter. The original 2-week calendar gate was a proxy for usage; usage is now sufficient. Locked, not retro-litigated.
5. **No logic duplication.** Orchestrator calls 3 existing skills via Agent tool; orchestrator does not re-implement their logic.
6. **Human gates at decision points only.** AskUserQuestion at scaffold approval (after gate), taste forks in implement, red/ambiguous QA, ship-readiness.
7. **Sunset criterion: 5 runs, mechanically defined.** Orchestrator appends one line per invocation to `~/.gstack/analytics/personal-pipeline.jsonl` with `{run_id, design_doc, end_state}` where `end_state ∈ {green, halted_at_gate, user_aborted, subagent_crashed}`. On the 6th invocation, orchestrator prints the run history and AskUserQuestions: keep / delete. **Trip-wire for delete recommendation:** ≥3 of 5 ended in `halted_at_gate`. No qualitative leg — the user keeps or deletes at the AUQ based on what they see, but the recommendation is not gated on a self-report memory test about whether they fell back to manual. The unenforceable "skipped" outcome is dropped (orchestrator can't observe being skipped from inside).
8. **Distribution.** `skills/personal-pipeline/SKILL.md` + catalog entry. Existing `skills-deploy` + `validate.sh` + `test.sh`.

## Approaches Considered

### Approach A: Full Agent-tool orchestrator (recommended; chosen)
Single `/personal-pipeline` skill. Each phase runs in a fresh Agent subagent with file-paths-only prompt. Orchestrator runs independent gates between phases. AskUserQuestion at decision points. ~2 weekends. Honors all 8 premises by construction.

### Approach B: Inline prose orchestrator (NOT chosen — anti-pattern)
Single skill, prose-only: "now run /scaffold, run gate, run /implement, ..." Claude executes inline in one conversation. Smallest code (~200-300 lines, 1 weekend), but structurally identical to the deleted /workflow. Violates P1 + P2. Listed for completeness; should be deleted on sight.

### Approach C: Shell harness `scripts/run-pipeline.sh` (NOT chosen)
Bash chains `claude -p /scaffold` → `claude -p /implement` → `claude -p /qa`. Strongest isolation by construction (OS processes, not Agent subagents). Bash gates between. ~1.5 weekends. Loses interactive AskUserQuestion (would need file-based prompts), and bash duplicates parts of `/personal-workflow check` parsing for gates. Trades P4 (interactive gates the user explicitly wants) for marginal isolation gains over Agent subagents.

## Recommended Approach

**Approach A: Full Agent-tool orchestrator.**

### Skill structure

`skills/personal-pipeline/`
- `SKILL.md` — frontmatter (name, description, version, allowed-tools incl. `Agent`), preamble, path resolution, usage, error handling. Entry point.
- `pipeline.md` — step-by-step orchestration logic. The meaty file.
- `fixtures/example-design-doc/` — synthetic design doc that produces a known scaffold + implement + qa output for testing.

### `pipeline.md` orchestration (high level)

The orchestrator does NOT pre-parse the design doc. It delegates type/shape detection to the Phase 1 subagent, which already has that logic in `/scaffold-work-item`.

**Subagent return contract (universal).** Every subagent's final message MUST end with a single line of the form `RESULT: <key>=<value>` (one or more KV pairs, semicolon-separated). The orchestrator parses the final message with `grep -E '^RESULT: '` and refuses to proceed if the line is absent. Concretely:

- Phase 1 subagent: `RESULT: WORK_ITEM_DIR=<absolute_path>`
- Phase 2 subagent: `RESULT: STATUS=<green|halted>; FILES_CHANGED=<n>; AUQ_NEEDED=<none|sensitive-surface|taste-fork>`
- Phase 3 subagent: `RESULT: SMOKE=<green|red>; E2E=<green|red|ambiguous>; PHASE2_GATES=<green|partial|red>`

**Orchestration steps:**

1. **Input validation.** Argument is a design-doc path under `~/.gstack/projects/`. Refuse if not. Refuse on multiple paths (one design doc per run).

2. **Pre-scaffold idempotency check.** Read source design doc. Search for footer line matching `^\*\*Status: SCAFFOLDED → (.+)\*\*$`.
   - **(a) Footer found, path exists, `/personal-workflow check` green** → set `WORK_ITEM_DIR` to that path, skip to Step 5.
   - **(b) Footer found, path missing** → halt with manual-cleanup AUQ ("design doc claims scaffolded but dir is gone — investigate").
   - **(c) Footer absent BUT a work-item tracker references this design-doc path** → halt with manual-cleanup AUQ ("scaffold appears to have halted between its Step 9 boundary check and Step 12 footer write — either delete the partial dir or hand-write the footer, then re-run"). Cheap scan: `grep -l "$DESIGN_DOC_PATH" work-items/**/tracker-*.md`. Catches the Step 9/Step 12 ordering hole — without this branch, re-run would dispatch Phase 1 and risk a duplicate dir via TODOS.md:26.
   - **(d) Footer absent AND no tracker references this design doc** → continue to Step 3 (fresh scaffold).

3. **Phase 1 — Scaffold (subagent).** Spawn Agent with `subagent_type: general-purpose`. Prompt structure: stable preamble (role: scaffold runner; instruction: invoke `/scaffold-work-item` with the path; return contract) + variable tail (the design-doc absolute path). Auto-accept ID/slug/shape AUQs ONLY (those are mechanical). Surface sensitive-surface AUQs (manifest/catalog/validator changes) by halting and reporting `AUQ_NEEDED=...` so the orchestrator re-AUQs to the human.

4. **Post-scaffold gate** (orchestrator runs; not the subagent):
   - Re-read source design doc footer. Confirm `Status: SCAFFOLDED → <WORK_ITEM_DIR>` is present and matches the subagent's `RESULT`. If mismatch or missing → partial-write halt with manual-cleanup AUQ.
   - `/personal-workflow check "$WORK_ITEM_DIR"` — refuse on red, surface findings.
   - Detect multi-story feature: if `WORK_ITEM_DIR` is a feature with ≥1 user-story child dir, halt after gate with message "feature-shaped work-item; invoke /implement-from-spec + /qa-work-item per child manually" and exit clean. (v1 scope.)
   - AskUserQuestion to confirm shape. On approve → continue.

5. **Phase 2 — Implement (subagent).** Spawn Agent. Prompt: stable preamble (role: implementer runner; invoke `/implement-from-spec`; surface AUQs via `RESULT: AUQ_NEEDED=...`; do not auto-accept sensitive-surface or taste-fork) + variable tail (`WORK_ITEM_DIR`). On `AUQ_NEEDED != none` → orchestrator re-AUQs to human, then re-dispatches Phase 2 with the human's answer threaded into the prompt tail.

6. **Post-implement gate:**
   - `/personal-workflow check "$WORK_ITEM_DIR"`.
   - `scripts/validate.sh` (fast). Capture stdout+stderr; on exit != 0 → halt with output appended to tracker journal as `[gate-red]`.
   - On red: AskUserQuestion "implement broke X — see tracker journal. Abort, attempt fix (re-dispatch Phase 2 with fix prompt), or override?"

7. **Phase 3 — QA (subagent).** Spawn Agent. Prompt: stable preamble (role: QA runner; invoke `/qa-work-item`; return structured verdict) + variable tail (`WORK_ITEM_DIR`).

8. **Post-QA gate:**
   - Parse tracker `[smoke-pass]`/`[smoke-fail]` and `[qa-pass]`/`[qa-red]` journal entries written by `/qa-work-item`.
   - Confirm Phase 2 gates green where qa-work-item is supposed to mark them.
   - On red/ambiguous: AskUserQuestion with the specific findings. Default = halt.

9. **Final summary + telemetry write.** Print `WORK_ITEM_DIR`, suggest `/ship`. Append `{run_id, design_doc, end_state}` line to `~/.gstack/analytics/personal-pipeline.jsonl`. On 6th run, AskUserQuestion sunset decision (per Premise 7).

### Error/abort contract

- **Idempotent.** Re-running on the same design doc resumes from the first incomplete phase. Pre-scaffold check (Step 2) handles "already scaffolded" without re-running scaffold.
- **Halt-on-red, no rollback.** On subagent crash or gate red without user override: write `[gate-red]` or `[subagent-crash]` to tracker journal, append run line to telemetry with `end_state=halted_at_gate` or `subagent_crashed`, exit non-zero.
- **Resume.** User re-invokes orchestrator (which re-checks pre-scaffold, then resumes from first incomplete phase) OR invokes the next individual skill manually.
- **Concurrent invocation.** Two runs on different design docs can race on `work-items/` ID generation. Inherits the workbench's existing accepted-risk posture (PHILOSOPHY.md does not gate on this); document but do not lock.
- **Cancel (Ctrl-C).** Kills orchestrator; subagent state is whatever was last written to disk. Re-invoke to resume.

### Decision gates (AskUserQuestion)

- Post-scaffold-gate (confirm shape)
- Phase 2 sensitive-surface or taste-fork (subagent reports `AUQ_NEEDED=...`; orchestrator re-AUQs)
- Post-implement-gate red
- Post-QA-gate red/ambiguous
- Sunset check on 6th invocation
