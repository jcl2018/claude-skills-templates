## Affected Skills
- `/scaffold-work-item` (or `/personal-workflow scaffold`) — input: design-doc-path → output: work-item directory tree
- `/implement-from-spec` (or `/personal-workflow implement`) — input: user-story dir → output: code changes + tracker updates
- `/qa-work-item` (or `/personal-workflow qa`) — input: user-story dir → output: smoke + E2E results in tracker

## Key Interactions to Verify
- Scaffold a feature design doc → confirm feature dir + at least one user-story child created with all required artifacts (TRACKER + DESIGN + ROADMAP for feature; TRACKER + DESIGN + SPEC + TEST-SPEC for user-story)
- Scaffold a user-story design doc → confirm a single user-story dir with 4 artifacts
- Implement on a user-story dir → confirm code is written matching SPEC, tracker journal records changes, Phase 1 → Phase 2 gates transition
- QA on a user-story dir → confirm smoke runs, QA engineer subagent runs E2E, results written to tracker
- Pass a feature dir to /implement or /qa → confirm AskUserQuestion lists child user-stories (1.2A behavior)
- Re-invoke any skill on already-completed input → confirm NO-OP idempotency (1.1A)
- Boundary check at start of implement/QA: pass an unscaffolded dir → confirm clean refusal (1.3A)

## Edge Cases
- Branch name doesn't match {feature/feat/story/task/chore/fix/defect/bugfix}: scaffold should AskUserQuestion to confirm type
- ID collision (work-items/ already has the next sequential ID): should be impossible by design but verify increment logic
- Template missing from both fallback locations: existing /personal-workflow check error path
- LLM produces invalid output (e.g., scaffold writes a SPEC missing required sections): boundary check at end (1.3A) catches it; AskUserQuestion to surface
- QA engineer subagent times out at 5 min: tracker records timeout, AskUserQuestion to re-run
- QA engineer subagent returns ambiguous results: AskUserQuestion to interpret green/red
- Smoke fails before E2E: write failures to tracker, abort QA before E2E run
- Multi-story feature with N user-stories: scaffold creates all N children; implement and QA must be invoked once per child (1 + 2N total invocations)

## Critical Paths
- **Bootstrap (chicken-and-egg):** First /scaffold-work-item run must be on its own design doc; verify it can scaffold the F000010 work item that contains itself
- **End-to-end pipeline on a real work item:** Run scaffold → implement → qa on a full design doc; confirm work item ends in Phase 2 with green smoke + green E2E ready for /ship
- **Drift detection at boundary:** Modify a template, re-run any skill against an existing work item, confirm boundary check catches the drift

## Test Approach (manual, per Step 0 decision)
