## Problem Statement

The company-workflow skill validates work items but doesn't teach the AI how to CREATE them. On a fresh machine (no gstack, no `~/.claude/rules/work-items.md`), the AI has templates and validate, but Step 1 of the doc-driven workflow is broken: the AI doesn't know scaffolding conventions (IDs, directory layout, type-to-artifact mapping, placeholder replacement).

The skill needs to be fully self-contained. One `skills-deploy install`, and the 3-step workflow works on any machine.

## What Makes This Cool

One skill package = complete doc-driven development system. Templates for structure, examples for tone, WORKFLOW.md for conventions, validate for enforcement. No external dependencies. Install it on your work machine and start creating company-spec work items immediately.

## Prior State

- SKILL.md has unified validate (file + dir modes) — shipped
- 13 templates in templates/company-workflow/ — shipped
- company-artifact-manifests.json with 5 type entries — shipped
- PRD documents 3-step workflow with Step 3 as "follow-up"
- Scaffolding rules live in ~/.claude/rules/work-items.md (workbench-only)
- No examples exist yet
- No WORKFLOW.md exists yet

## Constraints

- Zero gstack dependencies (standalone skill)
- SKILL.md stays focused on validation (no scaffolding logic in SKILL.md)
- Company 4-phase lifecycle preserved (Track, Implement, Review, Ship)
- Deployable via skills-deploy install
- Must work on a fresh machine with no prior setup

## Premises

1. The skill must be fully self-contained. One `skills-deploy install` and the 3-step workflow works. No gstack, no work-items.md, no external rules.
2. Examples (1 per template, 13 total) are the primary mechanism for teaching AI content style. Templates teach structure, examples teach tone/depth.
3. The skill needs a scaffolding section (in WORKFLOW.md) that tells the AI how to generate work items: type-to-artifact mapping, ID conventions, directory layout.
4. The validate command stays read-only. Scaffolding is done by the AI following WORKFLOW.md instructions, not by the validate command itself.
5. The company 4-phase lifecycle is preserved in the self-contained package (not the workbench 3-phase).

## Approaches Considered

### Approach A: All-in-SKILL.md
Add scaffold section directly to SKILL.md. One file does everything.
Effort: M. Risk: Med (SKILL.md bloat). Completeness: 8/10.

### Approach B: Companion WORKFLOW.md + SKILL.md
Keep SKILL.md focused on validation. Create WORKFLOW.md for the 3-step workflow and scaffolding conventions.
Effort: M. Risk: Low (clean separation). Completeness: 9/10.

### Approach C: Self-contained rule file
Create a global rule at ~/.claude/rules/company-work-items.md, auto-loaded by AI.
Effort: M. Risk: Med (hidden behavior). Completeness: 7/10.

## Recommended Approach

**Approach B: WORKFLOW.md + Examples**

Clean separation. SKILL.md validates. WORKFLOW.md teaches. Examples demonstrate.

### Deliverables

#### 1. WORKFLOW.md (`skills/company-workflow/WORKFLOW.md`)

The "how to use this system" doc. Contains:

**Section 1: Doc-Driven Development Workflow**
- Step 1: Generate Initial Docs — AI reads templates + examples, generates work item docs
- Step 2: Align the Big Picture — engineer edits, runs validate iteratively
- Step 3: Implement and Iterate — code follows aligned docs, validate enforces compliance

**Section 2: Scaffolding Conventions**
- Type-to-artifact mapping (reference company-artifact-manifests.json):
  - feature: TRACKER + milestones + PRD + ARCHITECTURE + TEST-SPEC (5 artifacts)
  - defect: TRACKER + RCA + test-plan (3 artifacts)
  - task: TRACKER + test-plan (2 artifacts)
  - user-story: TRACKER + PRD + ARCHITECTURE + TEST-SPEC + milestones (5 artifacts)
  - review: TRACKER + review-notes (2 artifacts)
  - Note: `userstory` (no hyphen) is accepted as an alias for `user-story`. Both normalize to the same type.
  - Note: The company manifest gives features 5 artifacts vs. the workbench's 2 (tracker + milestones only). The company spec treats features as full-scope items; the workbench delegates PRD/ARCH/TEST-SPEC to user stories. WORKFLOW.md documents the company mapping.
  - Note: `doc-scrum.md` and `doc-review-notes.md` are freestanding templates, not tied to a specific type's required artifacts. They can be created ad-hoc (e.g., scrum meeting notes, standalone reviews) in any work item directory or at the repo root, with no ID prefix required.
- ID generation: `{TYPE_PREFIX}{NNNNNN}` (F=feature, S=user-story, T=task, D=defect, R=review). Increment from highest existing ID in work-items/.
- Directory layout:
  ```
  work-items/
    features/{slug}/
      {ID}_TRACKER.md
      {ID}_{artifact}.md
      {child-slug}/          # nested: feature > userstory > task
    defects/{slug}/
      {ID}_TRACKER.md
      {ID}_{artifact}.md
  ```
- Placeholder replacement: `{ITEM_NAME}`, `{ITEM_ID}`, `{PARENT_ID}`, `{FEATURE_ID}`, `{YYYY-MM-DD}`, `{BRANCH_NAME}`, `{author}`
- 4-phase lifecycle: Track, Implement, Review, Ship (preserved from company spec)

**Section 3: Using validate**
- File mode: `company-workflow validate <file>` — structural check
- Directory mode: `company-workflow validate <dir>` — artifact completeness
- Run after Step 1 (check generated docs), during Step 2 (iterative alignment), before Step 3 (pre-implementation gate)

**Section 4: Installation**
```
skills-deploy install
# Copies to:
#   ~/.claude/skills/company-workflow/   (SKILL.md, WORKFLOW.md, contract.json, manifests, examples/, reference/, philosophy/)
#   ~/.claude/templates/company-workflow/ (13 templates)
# Use --overwrite to force-replace files with local modifications
```

#### 5. skills-deploy Subdirectory Fix

**Problem:** `skills-deploy` (line 196-203) only symlinks top-level `*.md` and `*.json`
files from skill directories. It does NOT handle subdirectories. This means `examples/`,
`reference/`, `philosophy/`, and `fixtures/` are never deployed, despite the docs claiming
they are.

**Fix:** Extend the skill installation loop in `scripts/skills-deploy` to symlink
subdirectories. After the existing top-level file loop (line 196-203), add:

```bash
# Symlink subdirectories from source (examples/, reference/, philosophy/, fixtures/)
for d in "$src"/*/; do
  [ -d "$d" ] || continue
  local dname
  dname=$(basename "$d")
  ln -snf "$d" "$SKILLS_TARGET/$name/$dname"
done
```

This symlinks each subdirectory as a whole (not individual files), keeping the deploy
fast and atomic. The symlink points to the repo source, same pattern as the top-level
`.md` files.

**Scope:** This fix affects ALL skills with subdirectories, not just company-workflow.
Currently only company-workflow has subdirectories (examples/, reference/, philosophy/,
fixtures/), so no regressions expected.

**Acceptance tests:**
