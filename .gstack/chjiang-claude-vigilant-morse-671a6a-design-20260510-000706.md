## Problem Statement

`/CJ_suggest` (`skills/CJ_suggest/SKILL.md`) embeds a ~150-line bash block inside its routing instructions and tells the model to "Run the bash block below." On this machine the Claude Code Bash tool dispatches that block through **zsh** (`Shell: zsh` in the env), not bash.

zsh treats `status` as a built-in **read-only** parameter (alias of `$?`). The skill's tracker-join code at line 33 of the heredoc'd block does `status=$(echo "$row" | awk -F'\t' '{print $2}')` to bind the work-item's frontmatter `status:` field. zsh refuses with:

```
(eval):33: read-only variable: status
```

…and the skill exits 1 before printing the top-5 table.

Today's workaround: write the body to a tempfile and invoke `bash /tmp/suggest.sh` explicitly. Verified working this session — produced the expected ranked table.

## Root Cause

The skill is bash-shaped (uses `<( )` process substitution, `<<<` here-strings, `[[ … ]]` would also be bash-flavored but isn't actually used) but is **eval'd by whichever shell the harness picks**. The Claude Code Bash tool on this machine runs zsh. Three of zsh's read-only specials would collide if used as plain variables: `status`, `pipestatus`, `LINENO`. The skill happens to use `status`.

This is the first concrete bug for the latent class "skill heredoc bash blocks assume bash but get zsh." Other skills in this repo (`CJ_personal-workflow`, `CJ_company-workflow`, `CJ_system-health`) may have similar latent collisions but none are known to crash today. Out of scope for this defect — flag separately if surfaced.

## Reproduction

```bash
cd <repo-root>
# trigger /CJ_suggest however the harness invokes it (here: paste the SKILL.md
# bash block into Bash tool with run_in_background=false, on a machine where
# the harness shell is zsh)
# → (eval):33: read-only variable: status
# → exit 1, no table
```

## Constraints

- Fix must be permanent in `skills/CJ_suggest/SKILL.md` — not a per-invocation workaround.
- Must keep the `/CJ_suggest` user-facing surface unchanged (still produces the same markdown table; same routing rules in CLAUDE.md and `skills-catalog.json`).
- Must preserve the "this-repo only, macOS only" scope from v1 design premises.
- Must not regress the existing /CJ_suggest behavior on machines where the harness is already bash.
- Skill author already anticipated this evolution: `skills/CJ_suggest/SKILL.md` Notes section says **"Single-file by design. No `scripts/suggest.sh` in v1. Promote to Approach B (script + eval case) post-soak if needed."** This defect IS the soak.

## Premises

1. Bug reproduces deterministically under zsh — confirmed in-session (D0 implicit).
2. Fix scope is `/CJ_suggest` only; latent zsh-vs-bash mismatches in other skills are a separate work item — confirmed.
3. Permanent fix is preferred over inline rename — confirmed (D1).
4. Approach B aligns with the skill author's already-planned promotion path and matches the workbench's `scripts/` convention (validate.sh, test.sh, eval.sh, etc.) — confirmed (D1).

## Approaches Considered

### Approach A: Rename `status` → `wi_status` inline

One-line edit to SKILL.md heredoc. Ships in 30 seconds.

- Effort: XS (~5 min)
- Risk: Low surface, but **does not address root cause**
- Pros: smallest diff; preserves single-file skill; zero new files/tests
- Cons: only patches the one variable that crashed today; any future zsh reserved word in the skill silently re-crashes; doesn't match the "post-soak" plan
- Reuses: nothing

### Approach B: Extract to `skills/CJ_suggest/scripts/suggest.sh` (chosen)

Move the entire bash body to a sibling script with `#!/usr/bin/env bash`. SKILL.md routing collapses to one line:

```bash
bash "$(git rev-parse --show-toplevel)/skills/CJ_suggest/scripts/suggest.sh"
```

The shebang pins execution to bash regardless of harness shell.

- Effort: S (~30 min: extract, test, add eval case, update SKILL.md)
- Risk: Low (script semantics identical to today's heredoc)
- Pros: pinned `#!/usr/bin/env bash` immune to harness-shell drift; matches `scripts/` convention; aligns with skill author's documented "post-soak promotion"; testable in isolation; eval-harness can exercise the script directly
- Cons: SKILL.md no longer literally self-contained; one new file + one new eval case
- Reuses: existing `scripts/` directory + script-deployment patterns; eval harness from `scripts/eval.sh`

### Approach C: Wrap inline block in `bash -c <<'BASH' … BASH`

Keep single-file by wrapping the entire body in a `bash -c "$(cat <<'BASH' … BASH)"` heredoc inside SKILL.md.

- Effort: S
- Risk: Med (quoting fragility)
- Pros: stays single-file; works under any harness shell
- Cons: heredoc-in-`bash -c` is double-escape territory — easy to break on routine edits; ugly; future maintainers misread it; no isolation for testing
- Reuses: nothing

## Recommended Approach

**B: Extract to `skills/CJ_suggest/scripts/suggest.sh`.** The skill author explicitly planned for this promotion when the inline form's tradeoffs surfaced ("Promote to Approach B post-soak if needed" in SKILL.md Notes). The defect IS that surfacing. B is also the only approach that's **structurally** robust to harness-shell drift — A patches one symptom, C is brittle.

## Components Affected

- `skills/CJ_suggest/SKILL.md` — replace bash heredoc body with one-line `bash …/scripts/suggest.sh` invocation; trim Notes section's now-obsolete "Single-file by design" caveat.
- `skills/CJ_suggest/scripts/suggest.sh` — new file. Contains the existing bash body verbatim with `#!/usr/bin/env bash` shebang and `set -euo pipefail` (slight upgrade from current `set -u`).
- `skills-catalog.json` — `CJ_suggest` entry's `files` array gains `skills/CJ_suggest/scripts/suggest.sh`.
- `scripts/validate.sh` — no change; existing catalog/filesystem coherence check covers the new file once cataloged.
- `tests/eval/CJ_suggest/` (if eval harness has a case for /CJ_suggest already) — add or update case to invoke the script directly. If not yet present, defer to F000013 V1 eval expansion (top-of-queue work item) rather than block this defect.

## Data / Control Flow

Unchanged from today. The script reads `TODOS.md` and `work-items/**/*_TRACKER.md`, joins, scores, prints markdown. Single behavioral change: `set -euo pipefail` (was `set -u`) means an unhandled non-zero exit anywhere now aborts the script with a stack-trace-ish error rather than continuing. Acceptable because every existing failure mode in the script already exits explicitly with a clear message; pipefail just hardens the implicit ones.

## Success Criteria

1. `/CJ_suggest` invocation under both zsh and bash harnesses produces an identical markdown table — no shell-specific divergence.
2. The defect's exact reproduction (paste old SKILL.md body into zsh) no longer occurs because the body is no longer pasted — the routing dispatches to bash via shebang.
3. `scripts/validate.sh` passes after the catalog update.
4. `scripts/test.sh` passes (covers validate + any existing eval case for /CJ_suggest).
5. `bash skills/CJ_suggest/scripts/suggest.sh` from the repo root, on a machine with the standard fixture (TODOS.md + work-items/), produces the same top-5 ordering observed in this session.

## Out of Scope

- Auditing other skills (`CJ_personal-workflow`, `CJ_company-workflow`, `CJ_system-health`, `CJ_personal-pipeline`, etc.) for the same latent class. Track separately if any other skill surfaces a zsh collision.
- Cross-platform support (Linux). The skill is already explicitly macOS-only via the BSD `date -j -f` check; the script inherits that.
- Replacing `awk`/`sed` YAML parsing with `yq` (the SKILL.md FRAGILITY NOTE flags this as a future migration if a consumed field starts containing `": "`). Not triggered today.

## Risks

- **R1: Catalog drift.** If `skills-catalog.json` isn't updated to include the new script in `CJ_suggest`'s `files` array, `scripts/validate.sh` will FAIL on missing/orphaned files. Mitigation: include the catalog edit in the same PR.
- **R2: `set -euo pipefail` regression.** A previously-tolerated non-zero exit somewhere in the script could now abort. Mitigation: smoke-test with the canonical fixture (this repo's current TODOS.md + work-items/) before landing.
- **R3: skills-deploy.** `~/.claude/skills/CJ_suggest/scripts/suggest.sh` must end up on disk after `skills-deploy install`. Verify the deploy script copies subdirectories under `skills/CJ_suggest/`, not just SKILL.md. (Per `scripts/skills-deploy` source, files are taken from the catalog's `files` array — so as long as the catalog lists the script, deploy picks it up.)
