## Problem Statement

`philosophy.md` at repo root (350 lines, 19922 bytes) is the workbench's "why this exists + how the pieces fit together" entry-point doc, intended for the operator and future contributors. It has drifted significantly across the last 5 weeks of shipping. Concrete drift:

- References three skills that no longer exist (`/workflow`, `/contracts`, `/docs`) as primary evidence for design principles (lines 5, 11, 13, 19, 336, 342, 348).
- Names `/CJ_goal_auto` and `/CJ_goal_run` as the primary entry points across the decision tree (lines 58-93), per-skill pipelines (lines 97-119, 121-143), and "Quick rule of thumb" table (lines 305-313) — both deprecated by F000027 / S000060 (PR #173, v5.0.6) and scheduled for sunset v6.0.0.
- Zero mention of the actual current front doors `/cj_goal_feature` (build a feature: topic → PR) and `/cj_goal_defect` (fix a bug: description → shipped fix), which have shipped 10 and 4 telemetry entries respectively this week alone.
- Zero mention of three mechanism layers landed in the last 14 days: `cj-goal-common.sh` shared helper (S000057), F000028 post-merge/post-rewrite doc-sync hooks, F000029 marker-pickup AUQ in cj_goal preambles.
- `/CJ_goal_run` Phase 4 still listed as `/land-and-deploy` tail (line 137); the new verbs PR-stop by design and `/land-and-deploy` is now a separate human step.

`/document-release` (post-ship doc updater) already finds `philosophy.md` via its `find . -maxdepth 2 -name "*.md"` glob, but falls under the generic "Any other .md files" treatment in Step 2 — which doesn't have specific skill-routing-drift checks (catalog `status: deprecated` rows, new `skills/*` directories not yet in the decision tree). So `/document-release` would "review" philosophy.md but wouldn't catch the drift class that just accumulated. The current drift accumulated specifically because that audit heuristic is generic.

Repo layout discovery: there is no `doc/` folder yet. Root currently carries six convention-anchored docs (README.md, CLAUDE.md, CONTRIBUTING.md, CHANGELOG.md, TODOS.md, philosophy.md) plus the machine-readable manifest (skills-catalog.json). `work-copilot/philosophy/` and `work-copilot/reference/` exist but are scoped to the Copilot bundle — out of scope here.

## What Makes This Cool

Closing a real feedback loop. F000028 + F000029 (shipped today, v5.0.8 + v5.0.9) wired the doc-sync mechanism: hooks drop a marker when main moves, preambles surface an AUQ to run `/document-release`. But the actual content `/document-release` audits has no specific named-doc treatment for the workbench's signature explanation — the "why this exists + which skill does what" surface. This feature gives that surface its own named-doc audit so the loop becomes: ship a routing change → hook fires → AUQ surfaces → `/document-release` runs → named-doc check catches skill-routing drift → operator sees and fixes. The novel pattern callout in CLAUDE.md ("script-output-drives-AUQ" from F000029) gets a content-side analogue: "named-doc audit catches drift class the generic case misses."

## Constraints

- **Root-convention files do not move.** README.md, CLAUDE.md, CONTRIBUTING.md, CHANGELOG.md, TODOS.md, skills-catalog.json all have hard reasons to stay at root: GitHub display (README, CONTRIBUTING, CHANGELOG), Claude Code autodiscovery (CLAUDE.md), runtime consumers like `/ship` Step 14 + `/CJ_suggest` + `/CJ_goal_todo_fix` reading `./TODOS.md`, and `validate.sh` / `skills-deploy` / `test.sh` reading `./skills-catalog.json`. Touching these is out of scope.
- **`philosophy.md` and `PHILOSOPHY.md` are the same inode** (58122076) on this case-insensitive APFS; there is only one file. The move-into-doc/ step is the literal command `mkdir -p doc && git mv philosophy.md doc/PHILOSOPHY.md` — destination path differs, so the rename works in one shot.
- **`/document-release` is upstream gstack — we do NOT edit its SKILL.md.** The skill lives only at `~/.claude/skills/document-release/SKILL.md` (gstack-deployed); there is no `skills/document-release/` directory in this repo to fork. Workbench-specific audit conventions go into `CLAUDE.md` (a new `## /document-release workbench audit conventions` section); the upstream skill reads CLAUDE.md as project context during its Step 2 audit pass, so the workbench conventions get picked up without modifying the skill. This pattern (project instructions teach generic upstream skills repo-specific rules) is the same one already used by the existing CI/CD merge convention section that teaches `/ship` + `/land-and-deploy` to skip `--auto` in this repo.
- **`/document-release` Step 1 aborts on the base branch** (`If on the base branch, abort: 'You're on the base branch. Run from a feature branch.'`). This is an F000029 contract gap — the cj_goal_feature "On A (and on main)" path documents invoking `/document-release` inline post-merge, but `/document-release` refuses to run on main. This is **out of scope** for this feature (it would be its own user-story or defect). The CLAUDE.md convention this feature adds is read by `/document-release` whenever it runs on any feature branch in this workbench — so the wiring lands on the path Step 1 actually allows.
- **No new manifest layer.** A declarative `skills-manifest.yaml` that drives philosophy.md + rules/skill-routing.md generation (the rejected Option C from the scope AUQ) is out of scope. Skill routing already lives in `rules/skill-routing.md`; having `doc/PHILOSOPHY.md` and `doc/ARCHITECTURE.md` narrate it in prose is fine.
- **Telemetry surface stays the same.** The feature touches docs + one CLAUDE.md section. It does not change cj_goal_feature, `/document-release`, or any other orchestrator.

## Resolved decisions

- **CLAUDE.md mechanism duplication is acceptable for v1.** The F000009 / F000028 / F000029 / TODOS-hygiene mechanism sections in CLAUDE.md stay (they are agent-relevant); the new `doc/ARCHITECTURE.md` gets its own operator-facing versions that may overlap by ~30% in content. The "right" answer would be to extract CLAUDE.md mechanism prose into `doc/ARCHITECTURE.md` (Approach C from scope AUQ), but the operator rejected that at scope AUQ. Soft duplication is the cost of declining the bigger refactor — accept it and revisit if CLAUDE.md grows past ~500 lines.
- **`doc/README.md` is dropped from v1.** With only two files in `doc/` (PHILOSOPHY.md + ARCHITECTURE.md), an index file is a YAGNI cost. Discovery happens via root `README.md` "Deeper reading" section + GitHub's directory rendering. Revisit if `doc/` grows past 4 files.
- **Drop-vs-tombstone rule for retired skill names.** Three retired skills (`/workflow`, `/contracts`, `/docs`) get a single `## Retired skills` subsection in `doc/PHILOSOPHY.md` with one paragraph per name (what it was, when retired, why, what replaced it). All OTHER mentions throughout PHILOSOPHY/ARCHITECTURE are dropped. The same rule applies to the F000027-deprecated `/CJ_goal_auto` + `/CJ_goal_run` — tombstone once, drop other mentions.

## Premises

All five confirmed in chat before this doc was written:

1. Scope is rewrite + wiring, not just rewrite. The user's stated goal includes "to be updated by document-release"; pure rewrite undershoots.
2. The "which skill does what" surface (current philosophy.md lines 52-314) is the most-drifted and most-valuable content. Treat it as the centerpiece of the rewrite.
3. `philosophy.md` belongs at root only by accident. Moving it to `doc/PHILOSOPHY.md` lets `doc/` become the home for explanation/reference deep-dives that don't belong at root.
4. Add NO new manifest layer. Prose docs that cross-link to `skills-catalog.json`, `rules/skill-routing.md`, and the work-item trees are enough.
5. `/document-release` Step 1 base-branch abort is a separate F000029 contract gap. File as TODOS follow-up; do not gate this feature on fixing it.

## Approaches Considered

### Approach A: Rewrite philosophy.md in place + add to /document-release Step 2 named docs

Keep philosophy.md at root, rewrite it end-to-end, add it as a NAMED doc in `/document-release` Step 2 audit with explicit skill-routing-drift checks. Smallest diff. Loses the consolidation benefit the user's reframe was pointing at.

- Effort: human ~3 hours / CC ~15 min
- Reuses: existing /document-release Step 2 structure (which already names README/ARCHITECTURE/CONTRIBUTING/CLAUDE.md)
- Pros: minimal blast radius; one file moved is one fewer thing to break; passes immediately.
- Cons: doesn't establish a `doc/` home for future explanation content (ARCHITECTURE, HOW-TO would have nowhere to land cleanly); root-level doc/file count grows over time instead of being organized.

### Approach B (RECOMMENDED): Create doc/ folder; move PHILOSOPHY + add ARCHITECTURE; wire workbench audit via CLAUDE.md convention

1. Create `doc/` at repo root: `mkdir -p doc`.
2. Move philosophy.md → doc/PHILOSOPHY.md: `git mv philosophy.md doc/PHILOSOPHY.md` (preserves history; one shot on the case-insensitive APFS because destination path differs).
3. Rewrite `doc/PHILOSOPHY.md` to current state. Concrete content requirements:
   - Replace `/CJ_goal_auto` + `/CJ_goal_run` as primary entry points with `/cj_goal_feature` + `/cj_goal_defect`.
   - Add a `## Retired skills` subsection at the end (above Failure modes) with one paragraph for each retired name: `/workflow`, `/contracts`, `/docs`, `/CJ_goal_auto`, `/CJ_goal_run`. Each paragraph names: what it was, when retired (PR #, version), why, what replaced it. All OTHER mentions of these names elsewhere in PHILOSOPHY are dropped.
   - Preserve the design-principles 1-5, "What this intentionally does NOT optimize for", "Key patterns and conventions", "How to extend without breaking its character", "Dependencies and assumptions", and "Failure modes and maintenance risks" sections (still load-bearing — verify against current repo state but expect light edits only).
   - Add a `## Decision tree` heading (anchor target for the named-doc audit) that contains the routing diagram for the active CJ_ skills (cj_goal_feature, cj_goal_defect, CJ_goal_investigate, CJ_goal_todo_fix, CJ_suggest, CJ_system-health, CJ_improve-queue). Internal phase-step skills (CJ_personal-pipeline, CJ_scaffold-work-item, CJ_implement-from-spec, CJ_qa-work-item, CJ_personal-workflow) listed in a separate "Called transitively" table.
4. Add NEW `doc/ARCHITECTURE.md` — mechanism reference. Five required sections, each must answer specific questions (no word-count gate):
   - `## The shared cj-goal-common.sh helper (S000057)` — must name the phases it owns (worktree, pr-check, telemetry), the modes it dispatches on (feature, defect, investigate), and the two consumers (cj_goal_feature, cj_goal_defect — and CJ_goal_investigate, which adopts it on its own update cadence).
   - `## F000028 doc-sync hooks (post-merge + post-rewrite)` — must name the marker file path schema (`~/.gstack/doc-sync-pending/<repo-slug>.json`), the marker fields (repo, head_sha, main_moved_at, diff_base, changed_files), the two git hooks (post-merge, post-rewrite), and what they don't fire on (trivial main-moving merges).
   - `## F000029 marker-pickup AUQ (cj_goal preambles)` — must explain the script-output-drives-AUQ split (script detects + prints `DOC_SYNC_PENDING <path>`, SKILL.md prose owns AUQ template + branch logic), the branch-aware A/B ordering (A on main, B on feature branch), and the `--resolved` / `--snooze` / `--skip` lifecycle.
   - `## Decision tree mirror` — points readers back to `doc/PHILOSOPHY.md ## Decision tree` (single source of truth; ARCHITECTURE summarizes, does not duplicate).
   - `## Deprecation tombstones` — cross-reference to `doc/PHILOSOPHY.md ## Retired skills` plus an explanation of the workbench's "three-shape deprecation" pattern (catalog `status: deprecated` + skill source relocation to `deprecated/<name>/` + work-item history relocation). Names F000005, F000006, F000007 as the original instances; F000027 + S000060 as the recent.
5. Update root-level `README.md` to add a `## Deeper reading` section linking to `doc/PHILOSOPHY.md` and `doc/ARCHITECTURE.md` (3 lines added).
6. Add a NEW section to root-level `CLAUDE.md`: `## /document-release workbench audit conventions`. This section teaches `/document-release` what to do when it runs in this repo. Content requirements:
   - States that `doc/PHILOSOPHY.md` and `doc/ARCHITECTURE.md` are NAMED audit surfaces (not generic "Any other .md files") and lists the specific drift class.
   - Provides the literal commands for the audit:
     - **Retired-skill drift check:** `jq -r '.[] | select(.status=="deprecated") | .name' skills-catalog.json` to extract retired skill names. For each name, `grep -n` in `doc/PHILOSOPHY.md` + `doc/ARCHITECTURE.md`. A mention is **annotated** (and skipped) if it appears inside the `## Retired skills` subsection of PHILOSOPHY, OR inside a `~~strikethrough~~` span, OR within 200 characters of the word `DEPRECATED` or `sunset` or `tombstone` (case-insensitive). All OTHER mentions are drift findings.
     - **New-skills check:** `jq -r '.[] | select(.status=="active") | .name' skills-catalog.json` to extract active skill names. For each name, grep `doc/PHILOSOPHY.md ## Decision tree` (anchor heading) for a case-sensitive literal match. Missing → drift finding ("active skill not in decision tree"). This check runs on `doc/PHILOSOPHY.md` only (not ARCHITECTURE — which is mechanism, not routing).
   - States what to do with findings: surface them in the `## Documentation` section of the PR body under a `### Skill-routing drift` subheading.

- Effort: human ~1.5 days / CC ~60-75 min (revised up from 35 min — net-new prose for PHILOSOPHY rewrite + 5-section ARCHITECTURE + CLAUDE.md convention with literal jq + grep specs is closer to S000061's effort profile than S000062's).
- Reuses: F000028+F000029 mechanism (already shipped); the existing CI/CD merge convention section in CLAUDE.md as the "project-instructions-teaching-upstream-skill" precedent; the existing routing prose in `rules/skill-routing.md` as input fidelity check for the Decision tree section.
- Pros: gives `doc/` a real home + a real reason to exist; the workbench audit convention specifically catches the drift class that just accumulated; preserves git history of philosophy.md via `git mv`; doesn't touch any runtime files; doesn't fork the upstream `/document-release` skill.
- Cons: 2 new files (doc/PHILOSOPHY.md, doc/ARCHITECTURE.md) + 2 edited files (README.md, CLAUDE.md); CLAUDE.md gains a section (already 389 lines, grows ~40 lines); reliance on `/document-release` actually reading + executing the CLAUDE.md convention section (untested until first /document-release run on a workbench feature branch — see Post-merge verification).

### Approach C: Approach B + CLAUDE.md mechanism extraction

Approach B plus: extract the four multi-paragraph mechanism sections currently in CLAUDE.md (Update-check F000009, Doc-sync check F000028, F000029 follow-up novel-pattern callout, TODOS.md hygiene conventions — roughly lines 100-300 of CLAUDE.md) into `doc/ARCHITECTURE.md`. CLAUDE.md slims down to ~200 lines (from 389), keeping just agent-action guidance.

- Effort: human ~2 days / CC ~55 min
- Reuses: same as B
- Pros: CLAUDE.md becomes much cheaper for Claude to load + scan; explanation content has a real home; no content lost (just relocated).
- Cons: REAL refactor — would need to verify nothing downstream greps CLAUDE.md for those exact section headings (manifest sync hooks, F000028's hook itself may grep for "## Doc-sync check mechanism" as a sanity check). Higher coordination cost. Rejected at scope AUQ in favor of B.
