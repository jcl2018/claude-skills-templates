## Problem Statement

The company-workflow skill needs to consult external knowledge (coding guidance, domain docs) but has no mechanism to discover where that knowledge lives. Without it, users paste context into every prompt and Claude defaults to generic patterns that clash with house style.

S000004 is the first vertical slice of F000004 knowledge integration. It teaches the skill WHERE knowledge lives via the `AI_KNOWLEDGE_DIR` environment variable. Resolution only. Actual reading of knowledge files ships in S000005 (always-on category loading) and S000006 (on-demand trigger matching).

This is an audit, not a fresh plan. T000003 already landed at commit 6265249 with the resolution bash block in `skills/company-workflow/SKILL.md` and the configuration docs in `skills/company-workflow/WORKFLOW.md`. This design doc captures the premises, alternatives considered, and the chosen approach so future readers see the decision trail.

## What Makes This Cool

Ten lines of bash unlock a whole feature. The env var + preamble pattern is trivially small, but once S000005/S000006 land, the skill auto-injects house coding style on every invocation and matches domain triggers against the user's prompt without any per-command setup. One env var opens the door.

The three-story decomposition (resolve, then always-on, then on-demand) means each slice is reviewable on its own. No giant PR, no "everything or nothing" ship risk.

## Constraints

- Zero regression on existing `/company-workflow validate` stdout for any fixture (stderr is expected to differ when the env is unset)
- Exit code stays 0 even when resolution fails (CI cannot break on missing knowledge)
- Warnings go to stderr, not stdout (validate stdout must stay clean)
- Runs safely under `set -e` / `set -o pipefail` in a parent shell. The `[ -d ... ]` tests live inside `if`/`elif` guards, which neutralize `set -e` by consuming the failure status. No `|| true` wrapper needed, but the structural invariant must be preserved in future edits to the block.
- Runs under the existing 2-level fallback path resolution (workbench repo → `~/.claude/skills/`)
- Fits the existing `skills-deploy install` pipeline with no new infra

## Premises

All four held in Phase 3. No challenges from the user.

1. AI coding assistants need explicit access to user-owned coding conventions and domain knowledge. Paste-into-prompt is the status quo and it hurts.
2. Env var is the right seam. One knob, works across repos, overridable in CI. Beats fixed path (couples to `$HOME`), per-repo `.knowledge/` (forces duplication), and a config file (adds complexity).
3. Resolution decouples from loading. S000004 answers "where does knowledge live?" S000005/S000006 answer "how do I read it?" Three small PRs beat one big one.
4. Warn-every-invocation when unset is deliberate. It is the nudge to configure, not an error to silence.

## Approaches Considered

### Approach A: Env var + bash preamble in SKILL.md (CHOSEN)

The preamble resolves `AI_KNOWLEDGE_DIR`, runs `[ -e ... ]` then `[ -d ... ]` checks, sets `$_KNOWLEDGE_DIR` (empty string on failure), emits one of three distinct warnings on stderr when resolution fails: unset/empty, path-not-found, path-not-a-directory. The valid case emits no warning and populates `$_KNOWLEDGE_DIR` with the path. Four input states → three warning messages (valid case is silent), mapping 1:1 to T000004 Tier 2 scenarios.

Reuses the existing Path Resolution block pattern. `$_KNOWLEDGE_DIR` follows the leading-underscore convention already established in SKILL.md's Path Resolution section (see `$_SKILL_DIR`, `$_TMPL_DIR`).

The warning fires whenever SKILL.md's preamble runs, which is every `/company-workflow` subcommand invocation (currently only `validate`, plus any future subcommands). Hook point = preamble, not per-subcommand.

- Effort: S (done, shipped in `6265249`)
- Risk: Low. Five scenarios verified end to end, exit 0 in all cases
- Pros: single knob, overridable in CI, decoupled from home-dir layout, matches existing `$_SKILL_DIR`/`$_TMPL_DIR` naming convention
- Cons: one more env var to document, warning is noisy by design

### Approach B: Convention-only, no env var

Skill checks `$HOME/.claude/knowledge/` unconditionally. Directory exists or it does not. No configuration.

- Effort: S
- Risk: Med
- Pros: zero config, nothing for the user to set up, fewer failure modes to test
- Cons: couples to `$HOME` layout, can't relocate without symlinks, CI can't override, can't support more than one knowledge root ever
- Explicitly rejected in Phase 1 Journal (2026-04-16): "no way to relocate without symlinks; couples knowledge to home dir layout"

### Approach C: Multi-source with XDG fallback

Check `$AI_KNOWLEDGE_DIR` first. If unset, fall back to `$XDG_CONFIG_HOME/claude/knowledge`, then `$HOME/.claude/knowledge`.

- Effort: M (more branches in the bash block, 3x the test matrix)
- Risk: Low-Med
- Pros: standard XDG conventions, env var still overrides
- Cons: 3x resolution surface, more test rows, no user asked for it, fallback paths can mask misconfiguration
- Deferred per Phase 1 Journal: "potential follow-up; ship single-source first"

## Recommended Approach

Approach A. It is already shipped and verified. B and C were considered and rejected in Phase 1 with recorded rationale. The office-hours value here is the explicit confirmation, not a course change.

## Open Questions

- Accept `$XDG_CONFIG_HOME/claude/knowledge` as a fallback eventually? Deferred. Revisit if users ask.
- Does the warn-every-invocation cadence become annoying in practice? Deferred. Suppression is out of scope for v1; if real users file a complaint, spin a follow-up story for an opt-out env var.
- Do we need `[ -r "$AI_KNOWLEDGE_DIR" ]` in addition to `[ -d ... ]`? Currently no. macOS + Linux default perms make the extra check rarely useful. If the dir exists but is unreadable, S000005/S000006's category enumeration (`ls`, `find`) will fail with a stderr error when it runs; T000004 should decide whether to pin that failure mode in its own test case or defer it to S000005's test matrix (preferred: defer; it's a loading-phase concern, not resolution).

## Distribution Plan

No new distribution mechanism. The skill already ships via `scripts/skills-deploy install`, which copies `SKILL.md` and `WORKFLOW.md` to `~/.claude/skills/company-workflow/`. Users add `export AI_KNOWLEDGE_DIR=...` to their shell profile. The configuration docs live in `WORKFLOW.md §Knowledge Configuration`.

## What I noticed about how you think

- **You push back on sequencing.** When I proposed three vertical slices, you accepted the decomposition but the framing you locked in was "smallest increment first." Not "break it up," but a principle. That shapes every future story.
- **You challenge naming without drama.** You caught the env var casing (`AI_Knowledge_DIr`) and let me normalize to `AI_KNOWLEDGE_DIR` without arguing. You also didn't just accept my first proposal of `CLAUDE_KNOWLEDGE_DIR`. You chose `AI_` to keep the knowledge store provider-agnostic. That is a small taste call that will matter a year from now when someone is running this against a different model.
- **You default conservative.** Missing `.knowledge.yml` → on-demand with empty triggers (dark by default) was your pick. The skill does nothing surprising unless the user explicitly asks. That's a mature position for a feature that injects content into an LLM's context.
- **You do not pre-decide implementation.** You kept the question of how knowledge actually reaches Claude (preamble-cat vs. Claude Read tool) open until architecture time, then resolved it in the ARCHITECTURE doc, not in the TRACKER. You let decisions live in the right artifact.
