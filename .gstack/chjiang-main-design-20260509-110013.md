## Problem Statement

The skill workbench has comprehensive metadata validation (`scripts/validate.sh`) and structural smoke tests (`scripts/test.sh`, `scripts/test-deploy.sh`), but no automated check that a skill *behaves correctly at runtime* against a fixture. Recent regressions (S000022 Step 18 traceability parser, D000016 stale template references) were caught by manual smoke testing or in-the-wild use, not by CI. The "Behavioral eval harness (P1, M)" TODO has been deferred since at least 2026-04-10 because scratch-workspace invocation of skills via Claude was previously "uncharted in Claude Code" — that has since changed.

## What Makes This Cool

The 2026 `claude` CLI exposes a stack of flags purpose-built for evals: `--print` (headless), `--output-format json` (parseable), `--json-schema` (structured assertion), `--plugin-dir` (session-scoped plugin loading), `--max-budget-usd` (per-run cost cap), `--bare` (minimal startup), `--no-session-persistence`. Combined, these collapse what was previously a custom-runner project into a few-hundred-line bash script that spawns the real CLI against scratch worktrees. The eval tests *the skill the user actually experiences*, not a simulation of it — the same skill loader, the same tool permissions, the same model.

## Constraints

- Stack must remain bash-friendly — workbench is bash-heavy by convention; new contributors read `scripts/` and infer the patterns.
- API token cost must be bounded — eval runs cannot block PR CI on cost grounds.
- Output assertion must be robust to LLM variance — prose golden-diff is brittle; structured JSON output validated against schema is the chosen approach.
- V1 ships in roughly one weekend — wedge-first; resist platform-building.

## Premises

1. **Variance is handled via structured JSON output, not prose golden-diff.** Eval prompts ask Claude to emit a JSON summary; `--json-schema` validates shape. Prose drift is irrelevant.
2. **V1 covers skills whose primary user-facing output is a structured report Claude returns to the user** (`personal-workflow`, `system-health`). Skills whose primary output is filesystem mutations (`scaffold-work-item`, `implement-from-spec`, `qa-work-item`) defer to V2's structural-assertion helpers. `deprecated/company-workflow` is out of scope permanently.
3. **Cadence is nightly on `main` plus manual local invocation.** Per-PR not in V1 — adds 30–90s + token cost to every CI run, dominated by lint-only / docs-only PRs that touch zero skills. Add per-PR later (with `paths: ['skills/**', 'templates/**']` filter) if the signal/cost ratio justifies it.
4. **Eval tests the in-repo skill source.** `--plugin-dir <repo>/skills` so a PR's skill changes are what gets tested. Deploy-side correctness is already covered by `scripts/test-deploy.sh`. **Note:** see Spike 0 and Open Question Q1 — `--plugin-dir` was designed for plugin directories, and the workbench's `skills/` is a flat skill directory; the spike resolves whether this works directly, requires a plugin manifest wrapper, or needs a different loading path.

## Approaches Considered

### Approach A: Bash + jq, lives next to test.sh (CHOSEN)

`scripts/eval.sh` orchestrator. `tests/eval/<skill>/<case>/` directory layout per case: `prompt.md` (the eval prompt for Claude), `fixture/` (seeded files copied to a tmpdir), `expected.schema.json` (JSON Schema asserting on the output shape). Runner spawns headless `claude` per case, captures stdout, validates against schema.

- Effort: S–M (one weekend if Spike 0 resolves cleanly; up to two weekends if it surfaces fallback work)
- Risk: Low–Med (pending spike outcome)
- Pros: matches existing bash conventions; one new script + one new directory; reuses `scripts/lib.sh` and existing fixture patterns; clean migration path to Approach B if it gets painful
- Cons: bash debugging UX is rough; pretty diffs / parallelism / retry-on-flake take real effort to do well in bash
- Reuses: `scripts/lib.sh`, `scripts/test-helpers/`, the `fixtures/` and `skills/personal-workflow/fixtures/` directory conventions

### Approach B: Bun + TypeScript runner with `bun test`

`tests/eval/` is a Bun project. Each eval case is a `.eval.ts` file using `bun test`'s API: declares fixture seeding (TS helper writing files to a tmpdir), spawns `claude`, parses JSON, asserts via Zod schemas. Real test runner UX (parallelism, `.only`/`.skip`, watch mode for local dev). Type-safe schemas — schema and TS type stay in sync.

- Effort: M (2–3 weekends)
- Risk: Med (introduces Bun dep to a bash workbench, but `work-copilot/` already uses Bun so it's not foreign)
- Pros: best long-term DX; type safety; scales cleanly to V2 (LLM-judge, structural assertions)
- Cons: more plumbing before first passing eval; adds Bun runtime dep for contributors

### Approach C: Co-located eval blocks inside SKILL.md (creative / lateral)

Each `SKILL.md` gets an optional `## Eval Cases` appendix containing markdown-formatted eval cases (frontmatter declares fixture / prompt / schema). Runner parses all SKILL.md files, extracts blocks, runs them. Templates ship with their own evals.

- Effort: S–M
- Risk: High (novel pattern, no precedent)
- Pros: ownership co-location (fix skill + eval in same edit); skills become self-documenting; templates carry their own behavioral spec
- Cons: SKILL.md files balloon (already long); reinvents diff/error/parallelism in bash; unclear scaling past 2–3 cases per skill

## Recommended Approach

**Approach A.** Reasons: (1) ships fastest and matches the wedge-first principle, (2) the rest of `scripts/` is exactly this shape (`test.sh` + `lib.sh` + bash helpers) so a new contributor reads `eval.sh` and immediately understands the conventions, (3) if A becomes painful, the migration to B preserves the eval cases — only the runner swaps. C's co-location idea is interesting but the failure modes (SKILL.md bloat, novel pattern) outweigh the benefit at V1 scale.

### Spike 0 (prerequisite, ~30–60 min, blocks the rest of V1)

Three CLI behaviors are unverified and load-bearing. Resolve before committing the runner to a final shape.

- **S0.1:** `--plugin-dir <repo>/skills` actually loads the workbench's skills in `--print --bare` mode such that `/personal-workflow check` is invocable. Test: `claude --bare -p "/personal-workflow check work-items/" --plugin-dir ./skills --print --output-format json --no-session-persistence` against a known-valid work-item tree. If skills are not discovered, see fallback in S0.1-fallback below.
- **S0.2:** `--json-schema` accepts the schema as an inline JSON string only (no `@<file>` shorthand was found in `claude --help`). Confirm: `claude -p ... --json-schema "$(cat schema.json)"` works; record whether `@<file>` is silently accepted, silently ignored, or errors. The runner uses inline `$(cat ...)` regardless.
- **S0.3:** When the model emits JSON that fails schema validation, what does the CLI do? Three possible behaviors: (a) non-zero exit with no stdout — runner asserts via exit code; (b) zero exit with the JSON written + a warning on stderr — runner pipes stdout through `ajv-cli` and asserts via that; (c) zero exit, no warning, schema silently ignored — runner always pipes through `ajv-cli`. Run a known-bad case and observe.

**S0.1-fallback (in priority order):**
1. Wrap `<repo>/skills/<name>/SKILL.md` in a minimal plugin manifest at runtime (write `<tmp>/plugins/<name>/plugin.json` + symlink the skill dir). Pass `--plugin-dir <tmp>/plugins`.
2. Use `skills-deploy install --to <tmp>/.claude/skills` (extending `skills-deploy` with `--to` if needed) and pass `HOME=<tmp>` to the `claude` invocation so `~/.claude/skills/` resolves to the seeded directory. Concurrency-safe: every eval case has its own `$HOME`. **Recommended fallback.**
3. Inline the SKILL.md content via `--system-prompt-file <SKILL.md path>`. Lowest fidelity (no slash-command discovery, no template resolution), accept only if 1 and 2 are both blocked.

**S0.3-fallback:** if `--json-schema` is enforcement-soft (cases b or c above), add `ajv-cli` as a dev dependency (`bunx ajv-cli@5` or npm install — the `claude` CLI is already a node-runtime prerequisite for contributors, so npm/bun is available). Document in `tests/eval/README.md`.

### Concrete shape of Approach A (assumes Spike 0 resolves to S0.1 working or S0.1-fallback #2; S0.3 worst case)

**`scripts/eval.sh` shape (top-level, with positional filter args):**

```bash
#!/usr/bin/env bash
# Run behavioral eval cases against in-repo skills.
# Usage:
#   bash scripts/eval.sh                              # all skills, all cases
#   bash scripts/eval.sh personal-workflow            # all cases for one skill
#   bash scripts/eval.sh personal-workflow case-name  # single case
# Exit 0 = all pass. Exit 1 = one or more failures.
set -euo pipefail
. "$(dirname "$0")/lib.sh"
init

EVAL_ROOT="$REPO_ROOT/tests/eval"
SKILLS_DIR="$REPO_ROOT/skills"
TEMPLATES_DIR="$REPO_ROOT/templates"

skill_filter="${1:-}"
case_filter="${2:-}"

tmp_results=$(mktemp)
trap 'rm -f "$tmp_results"' EXIT

# Pre-warm ajv-cli once so concurrent xargs children share the npm cache.
# Without this, 4 parallel `npx ajv-cli@5` first-time installs race the cache.
npx --prefer-offline --yes ajv-cli@5 --version >/dev/null 2>&1 || \
  npx --yes ajv-cli@5 --version >/dev/null 2>&1

discover_cases() {
  for skill_dir in "$EVAL_ROOT"/*/; do
    [ -d "$skill_dir" ] || continue
    local skill; skill=$(basename "$skill_dir")
    [ "$skill" = "lib" ] || [ "$skill" = "schemas" ] && continue
    [ -n "$skill_filter" ] && [ "$skill" != "$skill_filter" ] && continue
    for case_dir in "$skill_dir"*/; do
      [ -d "$case_dir" ] || continue
      local case_name; case_name=$(basename "$case_dir")
      [ -n "$case_filter" ] && [ "$case_name" != "$case_filter" ] && continue
      printf '%s\t%s\n' "$skill" "$case_dir"
    done
  done
}

# Parallelism: xargs -P N runs N cases concurrently. Each case has its own
# $HOME via the fallback loader, so concurrent runs cannot corrupt skill
# state. N=4 keeps cost predictable and avoids API rate-limit clustering.
discover_cases | \
  xargs -P 4 -L 1 -I '{}' \
    bash "$EVAL_ROOT/lib/run-case.sh" '{}' "$SKILLS_DIR" "$TEMPLATES_DIR" || FAIL=$?

# run-case.sh writes per-case PASS/FAIL lines to stdout. xargs exits non-zero
# if any child did. Capture, print summary, propagate exit status.
xargs_rc=0
discover_cases | xargs -P 4 -L 1 -I '{}' \
  bash "$EVAL_ROOT/lib/run-case.sh" '{}' "$SKILLS_DIR" "$TEMPLATES_DIR" \
  | tee "$tmp_results" || xargs_rc=$?

PASS=$(grep -c '^PASS:' "$tmp_results" || true)
FAIL=$(grep -c '^FAIL:' "$tmp_results" || true)
echo "----"
echo "PASS: $PASS  FAIL: $FAIL"
if [ "$FAIL" -gt 0 ] || [ "$xargs_rc" -ne 0 ]; then
  grep '^FAIL:' "$tmp_results" >&2 || true
  exit 1
fi
exit 0
```

**`tests/eval/lib/run-case.sh` shape (per-case runner):**

```bash
#!/usr/bin/env bash
# Args: <skill>\t<case_dir> <skills_root> <templates_root>
# Behavior: seed fixture, spawn claude headless, validate JSON output against schema.
set -euo pipefail
IFS=$'\t' read -r skill case_dir <<< "$1"
skills_root="$2"; templates_root="$3"
case_name=$(basename "$case_dir")
prompt_file="$case_dir/prompt.md"
fixture_dir="$case_dir/fixture"
schema_file="$case_dir/expected.schema.json"

tmpdir=$(mktemp -d)
fake_home=$(mktemp -d)
trap 'rm -rf "$tmpdir" "$fake_home"' EXIT

# Seed fixture into tmpdir + git init (so the skill's `git rev-parse` resolves
# to the fixture root, not the workbench, matching Premise 4).
[ -d "$fixture_dir" ] && cp -R "$fixture_dir/." "$tmpdir/"
( cd "$tmpdir" && git init -q && git add -A && git -c user.email=eval@local -c user.name=eval commit -q -m init ) || true

# Bind workbench skills + templates into the fake home so the skill resolves
# its own tooling correctly. (S0.1-fallback #2 — the recommended fallback;
# update this when Spike 0 resolves to a different loader path.)
# Symlink ONLY the skill under test to keep --bare's minimal-context intent.
mkdir -p "$fake_home/.claude/skills" "$fake_home/.claude/templates"
ln -s "$skills_root/$skill" "$fake_home/.claude/skills/$skill"
[ -d "$templates_root/$skill" ] && ln -s "$templates_root/$skill" "$fake_home/.claude/templates/$skill"

# Inline the schema (no @file syntax; CLI accepts inline JSON only).
schema_json=$(cat "$schema_file")
prompt=$(cat "$prompt_file")

# Run claude headless against fixture.
output=$(
  HOME="$fake_home" claude --bare \
    --print \
    --output-format json \
    --json-schema "$schema_json" \
    --add-dir "$tmpdir" \
    --model sonnet \
    --max-budget-usd 0.15 \
    --no-session-persistence \
    --permission-mode bypassPermissions \
    --allowedTools "Bash,Read,Glob,Grep" \
    "$prompt" 2>&1
) && rc=0 || rc=$?

# --output-format json wraps the model's response: { "result": "<model text>", ... }
# When the model emits JSON (as our prompts mandate), .result is a JSON-encoded
# string, so we use `fromjson` to parse it into the actual object before validating.
# S0.3 worst case: --json-schema may not exit-fail on mismatch. Always re-validate
# with ajv-cli for belt-and-braces.
if ! echo "$output" | jq -r '.result | fromjson' > "$tmpdir/output.json" 2>/dev/null; then
  echo "FAIL: $skill/$case_name (no parseable JSON object in .result)"
  echo "$output" >&2
  exit 1
fi
ajv_err=$(npx --prefer-offline --yes ajv-cli@5 validate -s "$schema_file" -d "$tmpdir/output.json" 2>&1) || {
  echo "FAIL: $skill/$case_name (schema validation)"
  echo "  ajv: $ajv_err" >&2
  echo "  output: $(cat "$tmpdir/output.json")" >&2
  exit 1
}

if [ "$rc" -ne 0 ]; then
  echo "FAIL: $skill/$case_name (claude exit $rc)"
  exit 1
fi
echo "PASS: $skill/$case_name"
```

**Eval prompt convention** (`prompt.md`). Every prompt:

1. Opens with the explicit slash-command invocation: `/personal-workflow check work-items/` (or whatever the skill's natural entry point is — `--bare` skips CLAUDE.md auto-discovery, so the prompt must spell out the skill invocation).
2. Closes with the JSON contract: *"Run the validation above. Then output a JSON object matching the provided schema. Do not include prose outside the JSON object — your response must be a valid JSON object only."*

Example `prompt.md` for `check-flags-missing-lifecycle`:

```markdown
/personal-workflow check work-items/

The work item under work-items/tasks/T000099_broken/ has a TRACKER.md missing
required lifecycle gate rows. Run the standard validation, then output a JSON
object matching the schema:

{
  "overall": "FAIL" | "PASS",
  "checks": {
    "lifecycle_gates": { "status": "PASS" | "FAIL", "missing": [<gate names>] }
  }
}

Output only the JSON object. No prose.
```

**`expected.schema.json` example** (for `check-flags-missing-lifecycle`):

```json
{
  "type": "object",
  "required": ["overall", "checks"],
  "properties": {
    "overall": { "enum": ["PASS", "FAIL"] },
    "checks": {
      "type": "object",
      "required": ["lifecycle_gates"],
      "properties": {
        "lifecycle_gates": {
          "type": "object",
          "required": ["status"],
          "properties": {
            "status": { "enum": ["PASS", "FAIL"] },
            "missing": { "type": "array", "items": { "type": "string" } }
          }
        }
      }
    }
  }
}
```

**Schema reuse:** `tests/eval/schemas/common-frags.json` holds shared fragments (overall PASS/FAIL enum, ac_coverage shape, lifecycle_gates shape) referenced via `$ref`. V1 is OK with some hand-written drift across cases — explicit deliberate trade-off, not oversight. Consolidation pass scheduled into V2.

**S000022 case caveat — explicit acknowledgment:** the `check-step18-faithful-comma-split` case tests *"Claude faithfully executes the Step 18 comma-split spec as written in `check.md`"*. It does NOT test the comma-split logic in isolation — that logic is described in English prose inside `check.md`, not extracted into a callable bash function, so a unit test is impossible without first refactoring. The eval case catches one failure mode (Claude misreads the spec) but not another (the spec itself is wrong while Claude executes it faithfully). Closing the second gap requires either extracting the parser into `scripts/check-helpers/parse-traceability.sh` (a separate refactor, out of V1 scope) or running the eval against an oracle case where the correct output is hand-known. V1 ships with the weaker signal acknowledged.

**CI integration:** New `.github/workflows/eval-nightly.yml`. Cron `0 9 * * *` UTC. Steps: checkout, install `claude` CLI, run `bash scripts/eval.sh`, report. Requires `ANTHROPIC_API_KEY` repo secret. Does NOT block PRs in V1.

**Local invocation:**
- `bash scripts/eval.sh` — all cases
- `bash scripts/eval.sh personal-workflow` — one skill
- `bash scripts/eval.sh personal-workflow check-flags-missing-lifecycle` — one case

## Open Questions

The three Spike 0 questions (S0.1 plugin-dir, S0.2 json-schema syntax, S0.3 json-schema enforcement) are the load-bearing unknowns. They are scheduled as the first build task; the design above includes fallbacks for each. Other questions:

- **Q4: How are golden schemas updated when skill behavior intentionally changes?** Convention: edit the schema in the same PR as the skill change. No `--update-snapshot` flow in V1 — schemas are hand-written and small. Reviewers verify the schema diff matches the skill intent.
- **Q5: What if `--max-budget-usd` is hit mid-case?** Probably partial output, no schema match, runner reports FAIL. Confirm behavior in Spike 0 alongside S0.3. Per-case cap of $0.15 means budget exhaustion is itself signal — if a case routinely costs more than $0.15, the prompt is too open-ended.

## Success Criteria

- `bash scripts/eval.sh` runs end-to-end on a clean checkout against personal-workflow + system-health, reporting PASS/FAIL per case
- The S000022 Step 18 case catches the original regression when reverted in a test branch
- A new GitHub Actions workflow runs nightly on `main` and reports failures via the existing notification surface
- Total V1 case count: 6–10 cases across the 2 skills
- **Cost per nightly run: under $1.50 USD** (10 cases × $0.15 cap = $1.50 worst case; expected average $0.40–$0.80)
- **Wall-clock per nightly run: under 12 minutes** (10 cases ÷ 4 parallel × ~3 min/case worst; expected 4–8 min)

## Distribution Plan

The eval harness is in-repo tooling, no separate distribution. Existing deployment pipelines cover everything:

- `scripts/eval.sh` and `tests/eval/` ship with the workbench repo via `git pull`
- The `claude` CLI is a contributor prerequisite (already true for the workbench)
- `npx ajv-cli@5` is available because the `claude` CLI installs node alongside itself (no new system dep)
- `ANTHROPIC_API_KEY` is a repo secret managed by the maintainer
- No new package, binary, container, or external service introduced

## V2 trajectory (out of scope for V1)

- Add `scaffold-work-item`, `implement-from-spec`, `qa-work-item` to scope. These produce filesystem side effects more than parseable output — needs structural assertion helpers (file tree shape, frontmatter present, journal entry written) layered on top of the V1 runner.

- Migrate to Approach B (Bun + bun test) if bash debugging UX becomes a sustained pain point. Eval cases (prompts, fixtures, schemas) are runner-agnostic — only `eval.sh` and `run-case.sh` get rewritten.
- Add LLM-judge for cases where `--json-schema` is too rigid (e.g., evaluating prose output quality of design-doc-generating skills).
- Schema consolidation: lift shared shapes from per-case `expected.schema.json` files into `tests/eval/schemas/` with `$ref`s.
- Refactor parser-shaped logic in `check.md` into callable bash helpers under `scripts/check-helpers/`, then unit-test those directly in `scripts/test.sh` for deterministic regression coverage of the parser logic itself (closes the gap acknowledged in the S000022 case caveat).

## Reviewer Concerns

Adversarial review (2 iterations) closed all 12 first-round findings. The following remained at iteration 2 and persist into V1 implementation deliberately — they are acknowledged limits, not unresolved questions:

- **R1 (design-level):** The `run-case.sh` sketch in this doc is shaped specifically for S0.1-fallback #2 (fake `$HOME` + symlinks). If Spike 0 resolves to S0.1 working directly via `--plugin-dir`, the runner simplifies — the `fake_home` setup goes away and `--plugin-dir "$skills_root"` replaces the `HOME=$fake_home` indirection. The first PR after the spike should pick the simpler shape if available; the fallback shape is the worst-case assumption baked into this doc.
- **R2 (design-level):** The S000022 Step 18 case tests *Claude faithfully executing the comma-split spec*, not the comma-split logic itself. To close that gap, the parser-shaped logic in `check.md` must be extracted into a callable bash helper under `scripts/check-helpers/parse-traceability.sh` and unit-tested directly in `scripts/test.sh`. Scheduled into V2; flagged here so the V1 case author writes the eval prompt + schema in a way that doesn't pretend to cover what it can't.
- **R3 (operational):** First nightly CI run will reveal real per-case wall-clock and cost numbers. The success criteria ("under 12 min, under \$1.50") are estimates derived from a 3-min/case worst case and a \$0.15 per-case cap. If empirical numbers exceed these by more than 50%, V1 needs either fewer cases or tighter prompts before the harness can be trusted as a regression signal.

## What I noticed about how you think

- You picked the high-fidelity path in D4 ("Test via Claude invocation") over my recommendation of script-level testing. You said in effect *if I'm building an eval harness, I want it to test the actual skill the user runs, not the deterministic core that happens to back it.* That's a taste call worth honoring — and it's why this design ended up shaped around the `claude` CLI rather than around the bash scripts the skills delegate to.
- Then you picked the wedge-first path in D8 ("Bash + jq") over the more sophisticated TypeScript option. You're consistent: highest fidelity at the eval *target* (real skill invocation), lowest ceremony at the eval *runner* (bash like the rest of the workbench). Two distinct dimensions, optimized independently — that's the same thinking pattern that produced the per-skill fixture conventions in `skills/personal-workflow/fixtures/` and the `MIRROR_SPECS` array in `validate.sh` (one line to add a new mirror dir).
- "All good" on the four premises after one round. You read them, didn't push back. Either you're calibrated and they're correct, or you're eager to move on. Both are fine — but worth flagging that the most likely premise to actually hold the design back is P3 (nightly cadence). If a regression makes it to `main` and sits there overnight before being caught, you'll be tempted to switch to per-PR. The design accommodates that — `paths` filter is a one-line workflow change — but be aware you may want it sooner than you think.
- The saved memory "skip design for small TODOs" was real signal. You overrode it for this TODO specifically because the eval harness *isn't* small — it's the foundation that every future skill correctness claim rests on. That's exactly the right time to override the heuristic.
