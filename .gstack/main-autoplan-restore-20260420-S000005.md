## Problem Statement

S000004 shipped the `AI_KNOWLEDGE_DIR` env-var resolution block. `/company-workflow`
now knows **where** the engineer's knowledge folder lives. It does nothing with the
contents.

S000005 closes the loop. Two surfacing paths ship together in one story:

- **Always-on** — categories marked `surface: always` have every `.md` file injected
  into Claude's context on every skill invocation. Used for house-style guidance
  (e.g. cpp error-handling conventions) the engineer wants applied by default,
  without copy-paste.
- **On-demand** — categories marked `surface: on-demand` with a `triggers` list load
  only when the user's latest message mentions a trigger. Used for situational
  material (domain deep-dives, runbooks) that's only relevant some of the time.

Both paths share the `.knowledge.yml` parser, file enumeration, per-repo opt-in gate,
and fixture builder. Prior vertical slicing (former S000005 always-on + S000006
on-demand) was merged on 2026-04-19 because the shared helper layer turned the
second "slice" into a refactor inside the first, not a real PR boundary.

This design doc captures **how** to execute T000006 — the sequencing, the file
changes, the commit boundaries, what's deliberately deferred, and the tripwires
that prove the contract is real.

## What Makes This Cool

**Knowledge shows up where it's needed without the engineer copy-pasting.** Three
classes of user experience unlock in one ship:

1. An engineer drops `coding/.knowledge.yml { surface: always }` + a cpp style guide
   under `$AI_KNOWLEDGE_DIR`. From that moment on, every `/company-workflow`
   invocation in an opted-in repo includes that style in Claude's context. Style
   violations drop to near-zero without the engineer ever asking.
2. An engineer drops `runbooks/.knowledge.yml { surface: on-demand, triggers: ["pricing engine", PE] }`
   + their pricing runbook. When they ask about the pricing engine, Claude reads
   the runbook *before* answering. When they ask about something else, no tokens wasted.
3. The per-repo opt-in marker (`.claude/knowledge-enabled`) means the engineer's
   global `$AI_KNOWLEDGE_DIR` pointed at Company A's knowledge folder doesn't leak
   into Company B's repo. Cross-context contamination is architecturally impossible,
   not just policy.

The interesting implementation move is that bash does discovery + enumeration
(deterministic, testable), but Claude does prompt matching (bash can't see the
prompt). The skill emits structured blocks (`## Always-On Knowledge`,
`## On-Demand Knowledge Candidates`) that encode the contract: paths bash
enumerated, triggers bash parsed, matching rule bash declared. Claude consumes
those blocks and either Reads the files or matches triggers and then Reads them.
No prompt-length blowup from `cat`-ing files into the preamble.

## Constraints

- **S000004 must stay on `main`.** Resolution block is the input. It's there (PR #38).
  No retouching in this PR.
- **Zero regression.** `/company-workflow validate` on existing `fixtures/valid-feature-dir/`
  must produce byte-identical output with env unset vs set-to-empty-dir.
- **No new system deps.** Bash + coreutils (`find`, `sort`, `grep`) only. No `yq`.
- **No committed fixtures under `skills/`.** The knowledge dir is user-owned
  (external by design). Test fixtures materialize in `mktemp -d` via a shared
  helper at `scripts/test-helpers/knowledge.sh`.
- **Per-repo opt-in gate is P0.** The marker (`.claude/knowledge-enabled` in repo

## Premises

Inherited from S000005_ARCHITECTURE.md (all pressure-tested by prior /plan-eng-review
+ Codex outside-voice). Two worth naming explicitly for execution:

1. **Inline bash `grep` yml parser, not `yq`.** The supported subset is
   `surface: always|on-demand` + `triggers: [a, "b, c"]` (flat keys; list via
   `[a, b]` or `- a\n- b`). Out-of-subset inputs → malformed-yml warning + skip.
   WORKFLOW.md must document this subset explicitly so users don't file bugs for
   unsupported yaml features.
2. **Skill emits paths; Claude Reads them.** Not `cat`-inline into the preamble.
   Keeps preamble small; uses Claude's native Read tool; lets Claude paginate huge
   files if needed. The contract is the `## Always-On Knowledge` and
   `## On-Demand Knowledge Candidates` sections. E2E canary tests (see §Success
   Criteria) prove Claude actually performs the Reads.
3. **On-demand matching happens in Claude, not bash.** Bash doesn't have access to
   the user's current message in a skill preamble. Claude does. SKILL.md specifies
   the rules explicitly: case-insensitive whole-word for single tokens, phrase
   match at token boundaries for quoted multi-word triggers, load-all-matched.
4. **`.claude/knowledge-enabled` marker gates BOTH paths.** Missing marker → neither
   `## Always-On Knowledge` nor `## On-Demand Knowledge Candidates` is emitted.
   Gate check runs first in both sections. Prevents cross-context contamination.
5. **Missing `.knowledge.yml` is silent skip.** A category with no yml is "in
   progress" from the user's POV. Warnings are for broken states, not unconfigured
   ones.
6. **Malformed `.knowledge.yml` is loud skip.** One stderr warning line naming
   the file; sibling categories unaffected; exit 0.
7. **Load order is deterministic.** Categories lex-sorted; files within a category
   lex-sorted by relative path. Re-runs produce byte-identical output given the
   same filesystem state.
8. **Soft size warn ≥ 50KB total always-on.** No hard cap; user opted into
   always-on, they bear the context cost. Warn gives observability.

## Approaches Considered

### Approach A: Sequential as T000006 Todos describe

Follow T000006's Todos in order: test helper → shared helpers → always-on →
on-demand → opt-in gate → tests. Ship as one PR with a sweeping commit or a few
loosely sequenced commits.

- Effort: ~400 lines bash + tests (S-M)
- Risk: Low — matches the scoped plan
- Pros: No re-planning overhead; matches what's already written
- Cons: Single large PR is harder to review atomically; bisecting post-merge
  (e.g., "where did the opt-in gate break?") requires re-reading the diff

### Approach B: Canary-driven TDD

Write the test helper first, then the E2E tests (E1–E14 from TEST-SPEC) as
failing, then build bash until each canary surfaces.

- Effort: +~30% upfront (scaffolding + failing tests first), -15% at the end (no
  bug-chasing because every test was specified before impl)
- Risk: Medium — can't write meaningful canary tests without the helper first;
  chicken-and-egg on the first commit
- Pros: The hardest-to-prove AC — "does Claude actually Read the listed paths?" —
  is validated before any bash lands. Prevents a class of subtle contract breaks.
- Cons: Over-discipline for a well-specified spec; TEST-SPEC is already binding

### Approach C: Approach A + strict bisectable commits (chosen)

Same file scope as A, sequenced as **3 bisectable commits inside one PR**:

- **c1: helpers** — `scripts/test-helpers/knowledge.sh` + SKILL.md
  `## Knowledge Helpers` section. Introduces no user-visible behavior; just infra.
- **c2: always-on + docs** — SKILL.md `## Knowledge Loading` + WORKFLOW.md
  `.knowledge.yml` schema docs + always-on tests (A1–A10 from TEST-SPEC).
- **c3: on-demand + opt-in gate + full suite** — SKILL.md `## On-Demand Matching` +
  `.claude/knowledge-enabled` gate + WORKFLOW.md on-demand + security callout +
  tests O1–O12 + G1–G3 + E1–E14.

- Effort: same as A (~400 lines) + ~5 min of commit-message work
- Risk: Low
- Pros: Reviewer can land-and-verify commit-by-commit; drift between docs and
  code is caught at each commit boundary; if on-demand matching stalls in review,
  c1+c2 are still shippable independently; security control (opt-in gate) lands
  with the logic it gates
- Cons: Marginally more discipline in commit boundaries

## Recommended Approach

**Approach C.** Same scope as A, better review surface. Ships as one atomic PR to
keep the F000004 feature story coherent, but the internal commit boundaries let
the reviewer follow the argument:

1. "Here are the helpers, no behavior change."
2. "Here's always-on loading using those helpers, and here are docs + tests that
   prove it works."
3. "Here's on-demand matching (consumes the same helpers), and here's the opt-in
   gate that protects both paths, and here's the full test suite."

Approach B was tempting but adds discipline the TEST-SPEC already provides —
TEST-SPEC was written as the binding contract, so writing failing tests from it
first is mechanical, not generative.
