## Problem Statement

The company follows a formal work item specification (the "spec") that defines structural validation rules via contract.json, tracker templates with fields like `workflow_type` and `url` (Jira/TFS integration), reference guides for AI doc generation, philosophy docs explaining lifecycle rationale, and validation test fixtures. The workbench repo has simpler personal-dev templates that diverge from the company spec: shorter lifecycle checkboxes (gstack `/ship` shorthand vs. explicit "Linux branch build passes"), no `workflow_type` field, no external tracker URL, no review type, no scrum doc type.

The user wants to build a separate, company-grade skill + template system within this same repo, deployable to company machines via `skills-deploy`, without touching the existing personal-dev system.

## Demand Evidence

"The spec folder is the truth. That's what the company follows." The company spec exists, is authoritative, and is the standard work item format on company machines. The user has been using personal-dev templates for this workbench, but the company system is what gets enforced on team machines.

## Status Quo

Today the company spec lives as a standalone directory (`~/Downloads/spec/`) with no deployment or enforcement mechanism. Engineers on company machines follow the spec manually or through whatever tooling the company provides. The workbench repo has no awareness of the company spec. Two parallel systems, no bridge.

## Target User & Narrowest Wedge

**Target user:** The repo maintainer (and eventually other engineers) who needs work items that comply with the company spec while developing in this workbench.

**Narrowest wedge:** A new skill (`company-workflow`) that ships the full spec as a deployable template set with validation, plus a `template-registry.json` that formally declares both template systems and their boundaries.

## Constraints

- Zero breakage to the existing personal-dev templates, skills, work items, validation, or deployment
- Company templates must coexist alongside personal-dev templates in `templates/` using subfolder namespacing
- The new skill must deploy via `skills-deploy` (same pipeline as existing skills)
- Validation against `contract.json` lives in the skill with a callable entry point (CI can invoke it)
- Work item directory structure and ID generation rules (from `~/.claude/rules/work-items.md`) still apply
- No modification to existing `validate.sh`, `test.sh`, or catalog validation logic

## Premises

1. The company spec becomes a NEW skill in this repo, not a replacement for personal-dev templates
2. Templates coexist in `templates/` using subfolders (`templates/company/` alongside root-level personal-dev templates)
3. contract.json validation lives inside the new skill with a callable entry point (CI-friendly)
4. Reference guides + philosophy docs ship as part of the skill (not standalone templates)
5. Existing `work-items/` structure and ID rules still apply; the company skill uses its own templates when scaffolding
6. A `template-registry.json` at repo root declares named template sets with metadata (version, contract path, description)

## Approaches Considered

### Approach A: Minimal Skill + Subfolder
New skill at `skills/company-workflow/` with templates at `templates/company/`. Skill has SKILL.md, contract.json, and reference guides. Simple, follows existing patterns.
- Effort: M (CC: ~30 min)
- Risk: Low
- Pros: Proven pattern. Catalog entry + skills-deploy = done.
- Cons: No versioning boundary between template sets beyond directory names.
- Completeness: 7/10

### Approach B: Skill + Template Registry (CHOSEN)
Same as A, plus a `template-registry.json` at repo root declaring named template sets (`personal-dev`, `company-spec`) with metadata. The skill reads the registry to resolve templates.
- Effort: L (CC: ~45 min)
- Risk: Med — new abstraction, but small and self-contained
- Pros: Clean versioning. Explicit template set switching. Future-proof.
- Cons: Adds one new config file.
- Completeness: 9/10

### Approach C: Spec-as-Submodule
External spec via git submodule. Auto-sync, single source of truth.
- Effort: S (CC: ~15 min)
- Risk: High — submodule pain, CI complexity, unclear deploy path
- Completeness: 5/10

## Recommended Approach

**Approach B: Skill + Template Registry.** The user explicitly said this is the opportunity to define template boundaries and versions. A registry makes that explicit. The marginal cost over A is minimal.

## Source of Truth Import

The company spec source lives at `~/Downloads/spec/` on the maintainer's local machine. This is a **one-time manual copy** into the repo:
- Templates: `~/Downloads/spec/templates/*.md` → `templates/company/`
- Contract: `~/Downloads/spec/contract.json` → `skills/company-workflow/contract.json`
- Guides: `~/Downloads/spec/reference/guide-*.md` → `skills/company-workflow/reference/`
- Philosophy: `~/Downloads/spec/philosophy/rationale-*.md` → `skills/company-workflow/philosophy/`
- Fixtures: `~/Downloads/spec/reference/invalid-*.md` → `skills/company-workflow/fixtures/`

Once committed to the repo, the repo becomes the source of truth for this workbench. Future spec updates from the company are manual syncs (diff and copy). The `~/Downloads/spec/` path does not need to exist on CI or teammate machines.

## Validation Entry Point

The skill exposes a validation subcommand callable from CI or the command line:

```
company-workflow validate <work-item-path>
```

Behavior:
- Reads `contract.json` from the skill directory
- Checks required frontmatter fields (`name`, `type`, `status`, `created`, `updated`, `workflow_type`, `url`)
- Validates required sections exist in expected order (`Lifecycle`, `Todos`, `Log`, `PRs`, `Files`, `Journal`)
- Confirms lifecycle has >= 4 checkboxes across 4 phases (Track, Implement, Review, Ship)
- Exit code 0 = valid, exit code 1 = invalid (issues printed to stderr)

This follows the `/docs check` pattern: the skill owns the logic, but it's invocable standalone.

## Open Questions

1. **Template resolution override:** When the company skill is active, should it override the fallback chain in `work-items.md` (repo templates/ → ~/.claude/spec/templates/ → ~/.claude/templates/)? Or should the skill handle resolution internally?
2. **Naming decision:** The spec uses `type: userstory` (no hyphen). The personal-dev system uses `user-story` (hyphen). **Decision for company templates: preserve the spec's exact spelling (`userstory`).** The company skill's routing logic must normalize when matching branch names (`story-*` → `userstory`). The two template sets intentionally use different spellings because they are different systems. The template-registry.json `types` array is authoritative per set.
3. **Deploy target:** `skills-deploy` currently deploys root-level templates. Subfolder support (deploying `templates/company/` → `~/.claude/templates/company/`) may need a minor update. **If skills-deploy cannot handle subfolders, create T000003 to patch it before this feature ships.**
4. **Review type routing:** The company skill handles `review` type routing internally. Branch pattern `review-*` maps to type `review`. This does NOT require changes to `~/.claude/rules/work-items.md` (which governs personal-dev only). The company skill's SKILL.md includes its own routing table.
5. **Uninstall behavior:** `skills-deploy remove company-workflow` removes the skill symlink. Company templates at `~/.claude/templates/company/` require manual cleanup for now. Document in skill README.
