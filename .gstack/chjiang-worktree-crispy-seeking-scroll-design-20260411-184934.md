## Problem Statement

system-health is a working, deployed skill at v0.3.0. It has a SKILL.md (695 lines),
DESIGN.md, and CHANGELOG.md. But it has no formal feature work item. Every other
tracked feature in this repo (workflow-alpha, F000001) has a full artifact set:
TRACKER.md, PRD, ARCHITECTURE, TEST-SPEC, and milestones. system-health shipped
through three major versions without that structure.

The goal is to formalize system-health as a V1 feature work item (F000002), generate
the standard artifact set from the existing implementation, bump the version to 1.0.0,
and close it out.

## What Makes This Cool

This is a retroactive formalization of a feature that already shipped. The work items
won't drive implementation (it's done), they'll serve as structured reference. The
milestones tell the real story: initial import, graph rewrite (v0.2), usage analytics
overlay (v0.3), and V1 cut. The PRD captures user needs that were never formally
written down. The architecture doc captures the 5-step pipeline and 4-bucket scoring
model in a format consistent with the rest of the repo.

## Constraints

- All artifacts use the existing templates from `~/.claude/templates/`
- Feature ID is F000002 (next after F000001_workflow_alpha)
- Work item directory: `work-items/F000002_system_health_v1/`
- No new code. The design doc describes what exists at v0.3.0
- Version bump from 0.3.0 to 1.0.0 in SKILL.md, DESIGN.md, CHANGELOG.md, and skills-catalog.json
- TRACKER.md phases should reflect completed work (all checkboxes checked)

## Premises

1. system-health v0.3.0 IS the V1 implementation. No new features needed.
2. Work item docs describe what already exists. They don't propose changes.
3. Feature ID = F000002 (next sequential after F000001).
4. Version bump 0.3.0 -> 1.0.0 is part of the V1 cut.
5. The existing DESIGN.md and CHANGELOG.md remain in `skills/system-health/`. The work item docs are the formal project-management layer on top.

## Approaches Considered

### Approach A: Retroactive Snapshot
Document v0.3.0 exactly as-is. All milestones pre-marked DONE. Minimal effort.
Rejected because milestones feel artificial and provide no contributor context.

### Approach B: V1 Feature Definition (chosen)
Design doc defines the V1 feature boundary. Milestones reflect the real journey
(v0.1 -> v0.2 -> v0.3 -> 1.0.0) with actual commit SHAs as evidence. PRD captures
user needs properly. Architecture doc captures the 5-step pipeline.

### Approach C: V1 + Roadmap
Same as B plus future ideas (auto-fix mode, MCP health, cross-machine sync).
Rejected because speculative roadmap items create stale docs, and the user wants
to "call it a day."

## Recommended Approach

Approach B. Scaffold `work-items/F000002_system_health_v1/` with:

1. **F000002_TRACKER.md** — Feature tracker with all 4 phases checked complete.
   Acceptance criteria derived from DESIGN.md test criteria.

2. **F000002_PRD.md** — Product requirements capturing:
   - User: Claude Code power user with 40+ skills installed
   - Problem: No visibility into ~/.claude/ health, dependency topology, or usage patterns
   - Solution: Scored dashboard with dependency graph, filesystem health, usage trends
   - User stories: "As a CC user, I want to see which skills depend on which"
   - Success metrics: Zero broken symlinks, zero dead references, usage data visible

3. **F000002_ARCHITECTURE.md** — Technical architecture capturing:
   - 5-step pipeline: Scan -> Graph -> Filesystem -> Waza -> Score+Trend
   - 4-bucket scoring: Structure (25%), References (35%), Integrity (25%), Hygiene (15%)
   - Data flow: bash/jq computes, Claude interprets (never computes)
   - Usage analytics: JSONL normalization, 3-schema support, anomaly detection
   - Dependencies: waza (optional runtime), jq (recommended)

4. **F000002_TEST-SPEC.md** — Test specification:
   - SKILL.md frontmatter validation
   - Catalog dependency declaration
   - E2E: all 5 steps produce output
   - Waza graceful degradation
   - Usage trends with missing/corrupt data
   - Scoring calibration

5. **F000002_milestones.md** — Real milestones from git history:
   - M1: Initial import (fbfc9ba) — v0.1.0, 9-layer model from home-setup
   - M2: Graph rewrite (107031e) — v0.2.0, 5-step pipeline, dependency graph, 4-bucket scoring
   - M3: Usage analytics (0659c00) — v0.3.0, skill-usage.jsonl overlay, anomaly detection
   - M4: V1 cut — version bump to 1.0.0, work item formalization

After scaffolding, bump version in:
- `skills/system-health/SKILL.md` (version: 0.3.0 -> 1.0.0)
- `skills/system-health/DESIGN.md` (version: 0.2.0 -> 1.0.0)
- `skills/system-health/CHANGELOG.md` (backfill missing [0.3.0] entry from commit 0659c00, then add [1.0.0] section noting V1 formalization)
- `skills-catalog.json` (version: "0.3.0" -> "1.0.0")

Note: CHANGELOG currently has [0.2.0] and [0.1.0] but no [0.3.0] entry despite
SKILL.md already being at 0.3.0. The [0.3.0] entry should document the usage
trends addition (Step 4.5). The [1.0.0] entry marks V1 cut with work item formalization.
