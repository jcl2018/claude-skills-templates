## Problem Statement

After every successful `/ship` + `/land-and-deploy` cycle on a personal-workflow work item, the work item's Phase 3 lifecycle gates stay UNCHECKED in the tracker. The workflow ran successfully, the PR merged, the deploy verified, but the tracker doesn't reflect any of it. As of 2026-05-08, this has happened on every shipped PR in the F000010 pipeline (S000017, S000019, S000018, plus D000016): 4 work-item trackers carry stale Phase 3 state.

The original "two manual steps" formulation in the personal-workflow design ("design-doc → work-item, E2E testing") was actually three: manually updating Phase 3 gates after `/ship` and `/land-and-deploy` is the third. Today's session shipped 4 PRs and every one of them left the gates blank — direct evidence that the manual step is unmemorable in practice.

`/implement-from-spec` (S000018, v1.9.0) and `/qa-work-item` (S000019, v1.8.0) handle Phase 1→2 and Phase 2→3 transitions cleanly. Phase 3 (post-ship) is the remaining gap.

## Premises (confirmed in-session)

1. **The drift is real friction worth solving** — confirmed by 4-PR direct experience this session.
2. **Phase 3 gates are load-bearing** — half-checked is worse than no gates. Either keep them accurate or delete them; "delete" loses audit trail and was rejected.
3. **5 of 6 Phase 3 gates are inferable from external state** — `/personal-workflow check`, `Smoke tests pass in CI`, `All children shipped`, `/ship — PR created`, `/land-and-deploy — merged and deployed` are all readable via `gh`/git/process-rerun. Only `E2E walked manually` is purely human-driven.
4. **The tracker is process evidence, not ship evidence** — git/PR/deployment is authoritative for "did this ship". The tracker says "did we follow the process". Drift is documentation lag, not a data integrity bug.
5. **Manual `check --update` would have the same forgetting problem** — solution must be auto-triggered. Direct experimental evidence: I forgot 4 times today.
6. **/ship and /land-and-deploy ownership is fixed** — upstream gstack, slow contribution loop. Local-side fix ships first; option 4 (upstream) is deferred-not-killed.

## Recommended Approach

**Option B: Engine + git `post-merge` hook.**

Two pieces working together:

### Piece 1: Inference engine — `/personal-workflow check --update <work-item-dir>`

Add a `--update` flag to `/personal-workflow check`. When invoked, the skill:

1. Runs the standard structural validation (existing behavior — Steps 1-13).
2. **Phase 3 gate inference** (new — Step 13.5):
   - For each Phase 3 gate label in the tracker (matched by substring), determine the corresponding external signal:

| Gate label substring | External signal | Inference |
|---|---|---|
| `/personal-workflow check` validation passed | re-run `/personal-workflow check` itself | PASS → check |
| `Smoke tests pass in CI` | `gh pr checks <PR>` for the work-item's PR | all checks pass → check |
| `E2E walked manually` | (no signal) | **never auto-check** — leave as-is or AUQ |
| `All children shipped` | recurse into child work-item dirs, verify their PRs are merged | all merged → check |
| `/ship — PR created` | `gh pr view` for the branch | PR exists (any state) → check |
| `/land-and-deploy — merged and deployed` | `gh pr view --json state` AND deploy probe | state=MERGED → check |
| `/document-release` | look for a recent `docs:` commit on main referencing this work item | found → check |

   - For each inferable gate, write `[x]` if the signal confirms; leave `[ ]` if not. Do NOT downgrade `[x]` to `[ ]` (idempotent, additive only).
   - Skip `E2E walked manually` entirely — never auto-check that one. If user wants it checked, they edit the tracker by hand or use a separate flag.
3. **PR linking** (new — Step 13.6):
   - Find the work-item's PR via `gh pr list --search "<work-item-id>" --state all`.
   - Append PR link + status to the tracker's `## PRs` section if not already present.
4. **Journal entry** (new — Step 13.7):
   - Append `[gates-update]` journal entry summarizing what changed: `2026-05-08 [gates-update] Phase 3: marked /ship + /land-and-deploy + smoke tests pass green (PR #66 MERGED, all checks pass).`
5. Boundary check (existing) confirms the tracker is still template-compliant after writes.

### Piece 2: Git `post-merge` hook trigger

Add a hook script at `scripts/post-merge-hook.sh` and wire it into `scripts/setup-hooks.sh`:

1. The hook fires after `git pull` / `git merge` on any branch.
2. Inside the hook:
   - Check we're on `main` (the branch where merges land). Skip if on a feature branch.
   - Read `git log @{1}..HEAD --name-only` to find files touched by the incoming commits.
   - Filter to paths matching `work-items/**/*_TRACKER.md` (work-item dirs).
   - For each touched work-item dir, invoke `/personal-workflow check --update <dir>` (via shell call to the deployed skill).
3. The hook is best-effort: failures don't block the merge. Errors print a one-line warning and the user can re-run manually.

### Why this combination

- **Engine alone** (option A) would require the user to remember a manual command — same forgetting problem (P5 violation).
- **Hook alone** is structurally impossible — there's no inference engine to call.
- **Together**: engine does the work, hook fires it automatically when the user pulls main after a merge. The user already runs `git pull` to get back to clean main after every ship; the hook piggybacks on that natural step.

### What ships in v1

- New `--update` flag on `/personal-workflow check`
- New script `scripts/post-merge-hook.sh`
- `scripts/setup-hooks.sh` extended to install the post-merge hook
- New `## PRs` and `## Journal [gates-update]` writebacks per the engine spec
- Tests covering both pieces

### What's deferred

- `E2E walked manually` auto-detection — defer; would require a manual-test-log file or explicit `--mark-e2e` flag.
- Upstream gstack contributions (option 4 from original TODO) — defer per P6.
- Web UI / cross-machine merge coverage — accept as a known gap; the user can run `/personal-workflow check --update` manually after a remote merge if needed.

## Constraints

- **Must work on existing personal-workflow work-items.** No template changes — Phase 3 gate labels are already in `tracker-feature.md`, `tracker-user-story.md`, `tracker-task.md`, `tracker-defect.md`.
- **Idempotent** — running `--update` multiple times must not corrupt state. Existing `[x]` checks stay; only `[ ]` → `[x]` transitions on positive signal.
- **Boundary-safe** — must not break `/personal-workflow check`'s own validation. Writes happen between Steps 13 (Directory Mode report) and 14 (Tier 2). Tier 2 confirms no DRIFT after writes.
- **Don't touch QA-owned or implementer-owned Phase 2 gates.** Those are owned by `/qa-work-item` (S000019) and `/implement-from-spec` (S000018). The new logic operates on Phase 3 only.

## Approaches Considered

### Approach A: Engine only — manual habit
Add `--update` flag to `/personal-workflow check`. Document manual usage in CLAUDE.md.
**Pros:** smallest scope; engine ships; engine is needed regardless.
**Cons:** P5 violation — user forgets. Drift partially reduced for disciplined users only.
**Completeness: 5/10.**

### Approach B: Engine + git post-merge hook (RECOMMENDED)
A's engine plus auto-trigger via git `post-merge` hook on main.
**Pros:** auto-triggered (P5 satisfied); piggybacks on natural `git pull` step; uses existing hook infra (`scripts/setup-hooks.sh`).
**Cons:** only fires for local merges; web UI / cross-machine merges miss the trigger.
**Completeness: 8/10.**

### Approach C: Engine + `/personal-workflow ship` wrapper
A's engine plus a thin wrapper skill that does `/ship` then `--update`.
**Pros:** works regardless of merge mechanism (wrapper fired before merge); fits F000010 naming convention.
**Cons:** P5 partial — user must remember `/personal-workflow ship` instead of `/ship`. Today's habit (`/ship` directly) needs change.
**Completeness: 7/10.**

## Recommended Approach

Approach B. Auto-triggered satisfies P5; the local-merge-only constraint is acceptable for solo workflows where merges happen on the same machine that runs `git pull`. The engine-only foundation (Approach A) ships as part of B; if Approach C becomes valuable later, the wrapper can layer on top of the same engine.

## Open Questions

- **Hook scope: post-merge only, or also post-checkout / post-rewrite?** Post-merge handles the `git pull` after ship. Post-checkout might re-fire on branch switches (unwanted). Decision: post-merge only for v1.
- **What happens if `gh` is offline / unauthenticated?** The engine falls back to a partial inference (just `/ship` PR existence and structural check). Document this; don't fail.
- **Children inference for features with multiple user-stories: how deep?** v1: one level (direct children). Recursion comes later if multi-level features ship.
- **`/document-release` gate signal: how to detect "post-ship doc audit done"?** v1 heuristic: look for a `docs:` commit on main authored after the work-item's PR merged that touches paths the work-item modified. If no such commit, leave gate unchecked. Could over-mark on coincidence; accept as low-risk.

## Success Criteria

- After a ship + merge + `git pull`, the touched work-item's Phase 3 gates that ARE inferable get marked `[x]` automatically.
- `E2E walked manually` stays `[ ]` (never auto-marked).
- A `[gates-update]` journal entry summarizes what changed.
- The `## PRs` section gets the merged PR link with status.
- Re-running `/personal-workflow check --update` is a no-op if state already converged.
- Existing `/personal-workflow check` (no `--update` flag) behavior is unchanged.
- Hook failures don't block git merges; they print a warning and the user can re-run manually.

## Distribution Plan

This is a workbench-internal change; no new distribution surface. Deployment via existing `skills-deploy install` (which symlinks `personal-workflow` skill assets) + manual `scripts/setup-hooks.sh` re-run on existing clones to install the post-merge hook.

## Dependencies

- `gh` CLI (already required by personal-workflow check for related logic)
- Git ≥ 2.x (post-merge hook standard)
- No new package dependencies

## The Assignment

After v1 ships: run a real `/ship` cycle on a small change, then `git pull main`, then `/personal-workflow check work-items/.../some-tracker-dir` and verify Phase 3 gates auto-update without any manual checkbox edits. The point of this design is that the verification is "ship something, then look at the tracker — gates are right."
