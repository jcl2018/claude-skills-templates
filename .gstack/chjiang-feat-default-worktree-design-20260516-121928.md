# Design: Default worktree for /CJ_goal_* sessions

Generated by /office-hours on 2026-05-16
Branch: main
Repo: jcl2018/claude-skills-templates
Status: APPROVED
Mode: Builder

## Problem Statement

The three top-level orchestrators in the CJ_goal_* family — `/CJ_goal_run`
(new features), `/CJ_goal_investigate` (defects), `/CJ_goal_todo_fix` (TODOs) —
currently assume current-branch execution. (Scope note: `/CJ_goal_investigate`
is in-flight on an unmerged worktree and is therefore **out of scope for this
PR** — see Open Question #5 — but the design is written for all three so the
followup is a copy-paste once its parent worktree lands.) When invoked from the main
checkout on `main`, they create a feature branch on top of `main` directly
and run the whole pipeline (autoplan → scaffold → impl → QA → ship → deploy)
there. Three concrete problems with that:

1. **Pollutes the main checkout.** Untracked files, branch switches, build
   artifacts all land in the same working tree the user uses for everything
   else. `git status` becomes noisy mid-pipeline; an interruption mid-scaffold
   leaves the main checkout in a half-modified state.
2. **Blocks parallel sessions.** Only one CJ_goal_* run can hold the main
   checkout at a time. The user already runs multiple sessions in parallel
   via Conductor — 17+ active worktrees under `.claude/worktrees/` at
   session start. The orchestrators are the one entry point that doesn't
   honor that pattern.
3. **Inconsistent with the established convention.** CLAUDE.md documents:
   "This repo's day-to-day work happens inside a git worktree under
   `.claude/worktrees/{name}/`, while the parent repo at the root has `main`
   checked out." That convention is enforced by hand and by Conductor for
   most flows; the CJ_goal_* orchestrators are the obvious gap.

The fix is "default to a worktree": each orchestrator detects whether it
is already inside a worktree (Conductor case), and if not, creates one
under `.claude/worktrees/cj-{type}-{timestamp}/` before any code-changing
phase fires. The user gets isolation for free; Conductor-spawned sessions
no-op the detection and keep their existing worktree.

## What Makes This Cool

The fix turns the orchestrators into "isolated by default": `/CJ_goal_run`
on `main` Just Works without thinking about it. No "oh wait, I forgot to
worktree first." The work goes into its own checkout, the PR goes up from
that worktree's branch, and the main checkout stays clean. Conductor flows
keep working unchanged (detection skips the wrap). One shared helper script,
three callers, ~60 lines total.

The detection logic is the trick that makes it cheap: `git rev-parse
--git-dir` versus `--git-common-dir` — if they differ, you're already in
a worktree, do nothing. No state file, no config, no marker. The repo
itself tells you whether you need to wrap.

## Constraints

- **Detect already-in-worktree (Conductor case) and skip creation.** The
  17+ active worktrees under `.claude/worktrees/` are mostly Conductor-spawned.
  Wrapping them again would create nested worktrees, fail on `git worktree add`,
  or worse, succeed silently and confuse `/ship`.
- **Must support `--quiet` mode without AUQ.** `/CJ_goal_todo_fix --quiet`
  is the cron-eligible drain mode (CLAUDE.md "Schedule-friendly drain"). It
  cannot surface an interactive prompt — auto-create or auto-skip silently.
- **Must support `--no-worktree` opt-out.** Users who explicitly want
  current-branch execution (debugging, intentional main edit) need an escape.
- **Worktree path.** `.claude/worktrees/{name}/` per CLAUDE.md convention.
  No new directory tree.
- **Branch naming.** `cj-{type}-{YYYYMMDD-HHMMSS}` where type ∈
  {`run`, `investigate`, `todo`}. The branch name equals the worktree leaf
  name (matches the existing convention; one name to track).
- **Workbench-resolvable helper.** The shared script must be resolvable
  from any consumer repo (CJ_goal_* skills work in any repo with `TODOS.md`
  + `work-items/`, not just the workbench). Use the same pattern as
  `skills-update-check`: resolve via `.skills-templates.json`'s `source`
  field, source the helper from `$source/scripts/cj-worktree-init.sh`.
- **No new always-on cost.** The helper runs only when the orchestrator
  fires, not on every skill preamble.
- **Cleanup is out of scope for v1.** CLAUDE.md already documents the
  worktree-cleanup convention (`gh pr merge --delete-branch` errors in a
  worktree; `gh api -X DELETE` is the workaround). Auto-cleanup after
  `/land-and-deploy` is a follow-up.
- **The `skills-deploy` manifest-pinning bug (TODOS.md:110) is unrelated.**
  That bug pins `manifest.source` to the worktree path when `skills-deploy
  install` is run from a worktree. The orchestrators don't call
  `skills-deploy install`, so this design doesn't compound that bug. (It
  also doesn't fix it — separate work-item.)

## Premises

All four confirmed by the user's request + observable workbench state:

1. **The user wants worktree-default for /CJ_goal_run, /CJ_goal_investigate,
   /CJ_goal_todo_fix.** The user's framing "(like developing a new feature,
   fixing a bug, fixing an todo)" maps 1:1 to these three orchestrators —
   they are the public CJ_goal_* family per CLAUDE.md.
2. **Worktrees are already the user's working pattern.** 17 active worktrees
   under `.claude/worktrees/` at session start, mostly Conductor-managed.
   `.claude/worktrees/{name}/` is the established path and branch convention.
3. **Conductor-spawned sessions are already in worktrees.** Auto-creation
   must detect (`--git-dir != --git-common-dir`) and skip — otherwise
   `git worktree add` collides or nests.
4. **The orchestrators do NOT share a common scaffold entry point.**
   `/CJ_goal_todo_fix` has its own internal T-task scaffold (TRACKER +
   test-plan written directly by `scripts/todo_fix.sh`); only
   `/CJ_goal_run` and (some paths of) `/CJ_goal_investigate` call
   `/CJ_scaffold-work-item`. Worktree creation must therefore live at the
   orchestrator preamble level, not inside `/CJ_scaffold-work-item`. That
   rules out a clever-looking "move worktree creation into scaffold"
   approach.

## Approaches Considered

### Approach A: Per-skill preamble (minimal viable)
Each of the three `SKILL.md` files grows ~15 lines in its preamble:
detect-in-worktree, detect-on-main, `git worktree add`, `cd`. Self-contained
per skill; no shared state.

- Effort: **S**
- Risk: **Low**
- Pros:
  - Simple; no new script files.
  - Each skill stays fully self-contained.
  - No cross-skill resolution concern.
- Cons:
  - Logic duplicated 3x. Bug-fix in one skill silently diverges from the
    others (the `/CJ_personal-workflow` family has been bitten by this
    multiple times — TODOS.md "ID-picker source-of-truth" is the same
    shape of debt).
  - Each preamble grows 3× the SKILL.md gets harder to skim.
  - Future fourth caller (if a `/CJ_goal_spike` ever lands) gets a copy-paste.
- Reuses: existing `_REPO_ROOT` resolution in each SKILL.md.

### Approach B: Shared helper script `scripts/cj-worktree-init.sh` (RECOMMENDED)
One bash script in the workbench at `scripts/cj-worktree-init.sh`. Each of
the three `SKILL.md` preambles resolves the workbench source path via
`.skills-templates.json`'s `source` field (same pattern as
`skills-update-check`) and runs the script. Script outputs structured
key=value lines on stdout; preamble parses, optionally surfaces a one-line
"created worktree at X" message, then `cd`s.

- Effort: **M** (~60 lines of bash + 3× ~5-line preamble integrations)
- Risk: **Low–Medium** (one risk: workbench-source resolution failing in a
  consumer repo without `.skills-templates.json` — fallback path needs to
  no-op gracefully)
- Pros:
  - Single source of truth for: in-worktree detection, branch generation,
    `git worktree add` + failure handling, `--no-worktree` / `--quiet`
    parsing.
  - Testable in isolation (`bash scripts/cj-worktree-init.sh --dry-run`).
  - Future fourth caller opts in with ~3 lines.
  - Mirrors the existing pattern (`skills-update-check`, the per-skill
    `scripts/todo_fix.sh` + `drain-one-todo.sh` extraction).
- Cons:
  - One new script file.
  - Each preamble depends on workbench-source resolution succeeding (graceful
    no-op required when it doesn't — same failure mode as
    `skills-update-check` today).
- Reuses: `_S=$(jq -r '.source ...)` resolution pattern from
  `skills-update-check`; `_REPO_ROOT` from each `SKILL.md`; CLAUDE.md
  worktree-path convention.

### Approach C: New wrapper skill /CJ_worktree_enter (lateral / over-build)
A new skill that the three orchestrators dispatch to. Most composable in
principle, but:
- adds a catalog entry, a `SKILL.md`, deployment surface;
- introduces a recursion concern (does the wrapper itself worktree if invoked
  directly? must not);
- inverts the dependency direction (orchestrators now depend on yet another
  CJ_* skill);
- the value-add over a shared script is theoretical — there is no AUQ flow
  or multi-step interactive logic to justify a full skill.

- Effort: **L**
- Risk: **Medium** (catalog churn, recursion edge cases, deploy ordering)
- Cons: rejected as over-build; bash helper covers the actual need.

## Recommended Approach

**Approach B — `scripts/cj-worktree-init.sh`.**

Rationale: three callers means the duplication tax in (A) is real, not
hypothetical. A bash helper is the cheapest abstraction that pays for
itself by the second caller. Approach (C) adds skill-catalog surface
without solving a problem (A) and (B) don't already solve. (B) also
mirrors the existing extraction patterns (`scripts/todo_fix.sh`,
`drain-one-todo.sh`, `skills-update-check`) — same shape, same resolution
mechanism, no new ceremony.

### Design specifics

**1. New script: `scripts/cj-worktree-init.sh`.**

Contract (stdout = key=value lines, exit 0 unless catastrophic):

```
WORKTREE_STATE=created|detected|skipped|opted_out|failed
WORKTREE_PATH=<absolute path>
WORKTREE_BRANCH=<branch name>
WORKTREE_NOTE=<one-line human-readable note, e.g. "created cj-run-..." or "already in worktree X">
```

Args:
```
cj-worktree-init.sh --caller {run|investigate|todo} [--no-worktree] [--quiet] [--dry-run]
```

Logic (~60 lines):
```
1. Parse --caller (required), --no-worktree, --quiet, --dry-run.
2. If --no-worktree: emit STATE=opted_out, PATH=$(pwd), return 0.
3. Detect in-worktree:
     gitdir=$(git rev-parse --git-dir)
     common=$(git rev-parse --git-common-dir)
     if [ "$gitdir" != "$common" ]; then
       STATE=detected, PATH=$(pwd), return 0
     fi
4. Detect on-main:
     branch=$(git branch --show-current)
     if [ "$branch" != "main" ] && [ "$branch" != "master" ]; then
       STATE=skipped (on feature branch, respect user choice), PATH=$(pwd), return 0
     fi
5. Compose name:
     name="cj-${CALLER}-$(date +%Y%m%d-%H%M%S)"
     path=".claude/worktrees/${name}"
6. If --dry-run: emit STATE=created, PATH=<path>, BRANCH=<name>, NOTE="(dry-run)"; return 0.
7. mkdir -p .claude/worktrees
   git worktree add "$path" -b "$name"
   if non-zero: emit STATE=failed, PATH=$(pwd), NOTE="<git error>"; return 0 (fallback to current)
8. Emit STATE=created, PATH=<abs path>, BRANCH=<name>, NOTE="created $name".
```

No AUQ. Auto-create silently when on main. Opt out with `--no-worktree`.
This keeps `--quiet` mode (cron) safe without special-casing.

**2. Caller integration — three SKILL.md preambles get the same block.**

After the existing "Path Resolution" / `_REPO_ROOT` block, before the first
phase that mutates state (Phase 1 / autoplan / scaffold):

```bash
# Default-worktree (workbench-resolvable, no-op if workbench not deployed)
_S=$(jq -r '.source // empty' "$HOME/.claude/.skills-templates.json" 2>/dev/null)
if [ -n "$_S" ] && [ -x "$_S/scripts/cj-worktree-init.sh" ]; then
  eval "$("$_S/scripts/cj-worktree-init.sh" --caller {run|investigate|todo} "$@")"
  if [ "$WORKTREE_STATE" = "created" ] || [ "$WORKTREE_STATE" = "detected" ]; then
    cd "$WORKTREE_PATH" || { echo "ERROR: cd $WORKTREE_PATH failed"; exit 1; }
  fi
  [ -n "$WORKTREE_NOTE" ] && echo "[worktree] $WORKTREE_NOTE"
fi
```

Three preamble inserts, ~10 lines each. The `eval` consumes the
`KEY=VALUE\n` stdout. Graceful no-op when workbench source isn't
resolvable (downstream consumer repo without the workbench cloned).

**3. Per-caller `--caller` values:**
- `/CJ_goal_run` → `--caller run` → branch prefix `cj-run-`
- `/CJ_goal_investigate` → `--caller investigate` → branch prefix `cj-inv-`
- `/CJ_goal_todo_fix` → `--caller todo` → branch prefix `cj-todo-`

(`inv` over `investigate` for branch-name compactness; same convention
the existing worktrees use — short names.)

**4. `/CJ_goal_todo_fix` drain mode special case.**

Drain mode loops over N TODOs; each TODO ships its own PR. Two options:
- **(a)** One worktree per drained TODO (created inside the loop, per
  iteration).
- **(b)** One worktree per drain session, reused.

(a) matches the "one PR per TODO" model already enforced by /ship Gate #2
firing per drained PR — each TODO is independent work, each PR has its
own branch. (b) would force all drained PRs onto the same branch, which
breaks `/ship`.

So: for drain mode, worktree creation moves from preamble into the
per-TODO loop in `scripts/drain-one-todo.sh` (or `todo_fix.sh`). The
preamble handles single-TODO mode; the loop handles drain mode. One
caller, two integration points.

**Single-TODO mode** (e.g. `/CJ_goal_todo_fix T000045`): preamble creates
one worktree, the whole flow runs there. Same shape as `/CJ_goal_run`.

**Drain mode** (no args, `--max-drain N`): preamble does NOT create a
worktree (the loop will). Each iteration of `drain-one-todo.sh` calls
`cj-worktree-init.sh` before scaffolding the T-task. Each drained TODO
gets its own worktree + branch + PR.

  - `cd ~/.../claude-skills-templates && /CJ_goal_run <design-doc>` from
    `main` creates `.claude/worktrees/cj-run-{ts}/` on branch
    `cj-run-{ts}`, cds in, runs the full pipeline there. Main checkout

  - `/CJ_goal_todo_fix T000045` → `cj-todo-{ts}`.
- **Conductor / already-in-worktree no-ops:**
  - Inside `.claude/worktrees/cheeky-popping-kahn/`, `/CJ_goal_run`
    detects the existing worktree, emits a `[worktree] already in
    cheeky-popping-kahn` note, runs in-place.
- **Opt-out works:**
  - `/CJ_goal_run --no-worktree <design-doc>` runs on the current branch
    with no worktree creation.
- **Drain mode per-TODO worktrees:**
  - `/CJ_goal_todo_fix --max-drain 3` creates three distinct worktrees,
    one per drained TODO, each with its own PR.
- **Cron-safe:**
  - `/CJ_goal_todo_fix --quiet --max-drain 3` runs with no AUQ, auto-creates
    each worktree silently, no stdout noise beyond the existing telemetry
    journal.
- **Graceful degradation:**
  - In a consumer repo without `.skills-templates.json`, the preamble
    silently no-ops and runs on the current branch (today's behavior).
- **Test suite stays green:** `scripts/validate.sh` and `scripts/test.sh`
  pass after the change. Fixtures that assert on current-branch behavior
  may need a `--no-worktree` flag added.

### Revised Design Specifics (post-Eng-review)

This section supersedes the original "Design specifics" further up the doc. Where the two conflict, this section wins.

**1. Helper output: JSON, not `KEY=VALUE` + eval.**

Helper emits one line of JSON on stdout. Preamble parses with `jq -r` per field. Zero shell injection surface:

```json
{"state":"created","path":"/Users/.../cj-run-20260516-123456-78901","branch":"cj-run-20260516-123456-78901","note":"created cj-run-20260516-123456-78901"}
```

**2. Helper args (revised):**

```
cj-worktree-init.sh --caller {run|investigate|todo} [--no-worktree] [--quiet] [--dry-run] [--force-create]
```

`--force-create` (NEW): bypass in-worktree detection. For drain-loop invocations from `drain-one-todo.sh` where the parent is already inside a worktree but each drained TODO needs its own.

**3. Helper logic (revised):**

```
1. Parse --caller (required, enum {run,investigate,todo}),
   --no-worktree, --quiet, --dry-run, --force-create.
2. If --no-worktree: emit STATE=opted_out, PATH=$(pwd), return 0.
3. Resolve repo top via `git rev-parse --show-toplevel` — error if not in repo.
4. Unless --force-create, detect in-worktree:
     gitdir=$(git rev-parse --git-dir)
     common=$(git rev-parse --git-common-dir)
     if [ "$gitdir" != "$common" ]; then
       STATE=detected, PATH=$(pwd), return 0
     fi
5. Detect on-main:
     branch=$(git branch --show-current)
     if not main/master AND not --force-create:
       STATE=skipped (on feature branch), PATH=$(pwd), return 0
6. **Dirty-state check** (NEW): if `git diff --quiet && git diff --cached --quiet` fails:
     if --quiet: STATE=skipped, note="dirty checkout; ran in-place", return 0
     else:       STATE=failed, note="dirty checkout: stash/commit or pass --no-worktree", return 1
7. Compose name:
     prefix per --caller: run→cj-run, investigate→cj-inv, todo→cj-todo
     ts=$(date +%Y%m%d-%H%M%S)
     name="${prefix}-${ts}-$$"   # PID suffix prevents same-second collision
     path="$(git rev-parse --show-toplevel)/.claude/worktrees/${name}"
8. If --dry-run: STATE=created, PATH=<path>, BRANCH=<name>, note="(dry-run)", return 0.
9. mkdir -p "$(dirname "$path")"
   git worktree add "$path" -b "$name" 2>err.log
   exit_code=$?
   if exit_code != 0:
     # Retry once with extended PID-+random suffix to dodge transient collision
     name="${prefix}-${ts}-$$-$RANDOM"
     path="$(git rev-parse --show-toplevel)/.claude/worktrees/${name}"
     git worktree add "$path" -b "$name" 2>err.log
   if still != 0:
     # Sanitize git error before emitting (NUL/control/quote stripped, 200-char cap)
     STATE=failed, PATH=$(pwd), note="<sanitized error>", return 1
10. Emit STATE=created, PATH=<abs path>, BRANCH=<name>, note="created $name", return 0.
```

**Output is always single-line JSON.** `note` is sanitized to ASCII printable + safe punctuation, capped at 200 chars. Even if a future bug lets uncontrolled text leak into `note`, JSON encoding + jq parsing keeps it strictly a string value — no eval.

**4. Caller integration — preamble block (revised):**

This block goes **BEFORE** the existing `_REPO_ROOT` / Path Resolution block in each `SKILL.md`, so paths resolve post-cd:

```bash
# Default-worktree (BEFORE path resolution — variables get re-resolved post-cd)
if [ $# -gt 0 ]; then  # Skip helper on no-arg (Branch g auto-resume must run on current branch)
  _S=$(jq -r '.source // empty' "$HOME/.claude/.skills-templates.json" 2>/dev/null)
  if [ -n "$_S" ] && [ -x "$_S/scripts/cj-worktree-init.sh" ]; then
    _WT_JSON=$("$_S/scripts/cj-worktree-init.sh" --caller run "$@" 2>/dev/null)
    if [ -n "$_WT_JSON" ]; then
      _WT_STATE=$(echo "$_WT_JSON" | jq -r '.state // "failed"' 2>/dev/null)
      _WT_PATH=$(echo "$_WT_JSON" | jq -r '.path // empty' 2>/dev/null)
      _WT_NOTE=$(echo "$_WT_JSON" | jq -r '.note // empty' 2>/dev/null)
      if [ "$_WT_STATE" = "created" ] || [ "$_WT_STATE" = "detected" ]; then
        cd "$_WT_PATH" || { echo "[worktree] ERROR: cd $_WT_PATH failed"; exit 1; }
      elif [ "$_WT_STATE" = "failed" ]; then
        echo "[worktree] ERROR: $_WT_NOTE"
        exit 1  # halt; the feature's job is isolation, fallback to dirty main violates it
      fi
      # On opted_out / skipped: no cd, no halt; just continue
      [ "${QUIET:-0}" != "1" ] && [ -n "$_WT_NOTE" ] && echo "[worktree] $_WT_NOTE"
    fi
  elif [ -z "$_S" ] || [ ! -x "$_S/scripts/cj-worktree-init.sh" ]; then
    # Visible warning, not silent no-op (E9)
    [ "${QUIET:-0}" != "1" ] && echo "[worktree] WARN: helper unreachable; running on current branch"
  fi
fi
# Existing _REPO_ROOT / Path Resolution block follows here — resolves against worktree post-cd
```
