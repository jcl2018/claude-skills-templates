## Problem Statement

`/CJ_suggest` (v1.1.0) is a tight, deterministic, stateless TODO ranker — exactly what its downstream callers (`/CJ_goal_todo_fix --for-skill cj-goal --limit 15`, `/loop /CJ_goal_todo_fix`) need. It tells you *what's next from the backlog*.

It does not tell you *what should be in the backlog in the first place*. This workbench builds Claude Code skills. Anthropic ships new patterns continuously (tool use, hooks, AskUserQuestion shapes, agent SDKs, claude.md conventions). The repo's skills drift from current best practices, conventions diverge across skills, and external articles that should trigger consolidation work go unnoticed because there's no surface for "here's a URL — does this apply to us?"

The user wants a "proactive improver" that scans three input dimensions — this repo's state, online Claude best-practice sources, and user-supplied URLs — and surfaces improvement work.

## What Makes This Cool

**Improvement-as-TODO**. Instead of building a second ranking system or a new query surface, `/CJ_improve-queue` writes new rows into `TODOS.md` that the existing `/CJ_suggest → /CJ_goal_todo_fix → /ship → /land-and-deploy` pipeline already knows how to consume. Improvement work becomes indistinguishable from regular work-item work, ranked by the same scoring, gated by the same hygiene conventions (strikethrough = DONE), burned down by the same `/loop /CJ_goal_todo_fix` flow.

The compound effect: the more best-practice URLs you feed it, the more your skill collection auto-aligns to evolving Claude patterns — through the same shipping pipeline you already use.

## Constraints

- **Workbench-only**. Lives in `skills/CJ_improve-queue/`. Catalog status `experimental` initially.
- **Read-only on the world; mutating on TODOS.md only**. `allowed-tools` frontmatter: `Bash`, `Read`, `Edit`, `WebFetch`, `WebSearch`, `Agent`. No `AskUserQuestion` in v1 — the skill emits rows and exits; user decisions happen via TODOS.md review + the existing `/CJ_goal_todo_fix` flow.
- **Atomic writes to TODOS.md**. Uses `mktemp` + `mv` (same-volume rename is atomic). Pre-write: `git status --porcelain TODOS.md` must be clean; refuse with clear error if dirty (prevents editor-buffer overwrite). Pre-write backup to a `.gitignored` workbench path `.claude/tmp/TODOS.md.bak.<timestamp>` (NOT `/tmp`, which doesn't survive reboot and accumulates forever).
- **Concurrency**: file-lock via `flock` on `.claude/tmp/CJ_improve-queue.lock`. Two parallel `evaluate` invocations serialize; second prints "in-progress, retry" and exits 0.
- **Idempotent per source**. Each generated row carries a stable signature (truncated sha256 of canonical URL + subagent-emitted pattern_name). Idempotency probe: `grep -Fq 'impr-sig=<hash>' TODOS.md` (signature lives in an HTML comment footer, not in heading text — see Output Format below).
- **macOS-pinned, like `/CJ_suggest`**. `uname -s = Darwin` gate at script start. (Inherited from `/CJ_suggest`; no new BSD-specific usage in Phase 1 beyond the timestamp format for backup filenames, which is cross-compatible.)
- **No network calls without explicit user invocation**. `audit` mode (Phase 2) is offline-only. `research` and `evaluate` modes are user-triggered with URL/topic args.
- **`skills/**` scan scope**: `skills/*/SKILL.md` only. Explicitly excludes `deprecated/**`, `work-copilot/**`, and `.claude/**`. The deprecated/work-copilot exclusion matters because those mirror upstream and shouldn't be reshaped by improver suggestions.

## Premises

1. `/CJ_suggest`'s deterministic, stateless, fast contract is load-bearing for `/CJ_goal_todo_fix` and `/loop /CJ_goal_todo_fix`. Adding AI-reasoning or network to it would break those callers. **Keep `/CJ_suggest` untouched.**
2. The killer use case is *alignment-check between this repo's skills and evolving Claude best practices* — not generic TODO ideation. The workbench IS skills; drift from Anthropic's current patterns is the real cost.
3. URL evaluation, repo audit, and topic research are three input dimensions that converge on one output type: improvement TODO rows. Sharing output format is the unification, not sharing query infrastructure.
4. Mutation to `TODOS.md` is acceptable because rows land with a `DRAFT —` heading prefix (see Output Format) until the user removes it. `/CJ_suggest` ignores `DRAFT —` rows; `/loop /CJ_goal_todo_fix` therefore won't auto-pick them. Promotion to active is a one-line manual edit. This decouples improvement-row generation from auto-shipping.
5. Existing signals are sufficient as inputs: `~/.gstack/analytics/skill-usage.jsonl` (which skills are used), `git log` (recent churn), `skills/**/SKILL.md` (repo content), `WebFetch`/`WebSearch` (external).

## Approaches Considered

### Approach A: Extend `/CJ_suggest` with `--mode improve|research|evaluate`
Bolt new modes onto the existing ranker. Single command, four behaviors.

**Pros**: One mental model, one entry point. Reuses TODOS join logic.

**Cons**: Compromises `/CJ_suggest`'s stateless deterministic contract — now optionally network-bound, AI-reasoning, slow. Breaks the implicit interface its downstream callers (`/CJ_goal_todo_fix`) rely on for predictable preflight behavior.

Rejected.

### Approach B: New sister skill `/CJ_improve` with three sub-modes (audit / research / evaluate)
Add `/CJ_improve audit`, `/CJ_improve research <topic>`, `/CJ_improve evaluate <url>`. Each emits human-readable findings directly to stdout. `/CJ_suggest` untouched.

**Pros**: Direct one-keystroke URL-eval surface. `/CJ_suggest` stays pure.

**Cons**: Findings are ephemeral — there's no shipping pipeline behind them. User has to manually translate "this article suggests X" into TODOS.md rows or one-off PRs. Reinvents queue management.

Rejected (after Phase 4 discussion).

### Approach C: `/CJ_improve-queue` writes improvement rows into `TODOS.md` (chosen)
A new skill that scans inputs and *appends* improvement TODOs to `TODOS.md` matching the existing schema. `/CJ_suggest` picks them up unchanged. The full `/CJ_suggest → /CJ_goal_todo_fix → /ship` pipeline carries them to merged PRs without new ranking, dispatch, or shipping code.

**Pros**: Maximum pipeline reuse. Improvement work compounds through the same loop the user already runs. TODOS.md hygiene conventions (strikethrough = DONE, PARTIAL annotations, post-bundle cleanup) automatically govern improvement rows.

**Cons**: URL-evaluation is technically two-step (`/CJ_improve-queue evaluate <url>` writes a row; `/CJ_suggest` ranks it). Mitigated because the user already runs `/CJ_suggest` (or `/loop /CJ_goal_todo_fix`) routinely. Also: TODOS.md mutation requires atomic-write discipline.

## Recommended Approach

**Approach C**, with phased rollout.

### Phase 1 (MVP): `evaluate <url>` mode only

Single sub-command. Highest-value, smallest surface, fastest to ship.

```bash
/CJ_improve-queue evaluate https://docs.anthropic.com/claude-code/some-new-pattern
```

**Flow** (two halves: shell envelope + Agent subagent):

1. **Shell envelope (`scripts/improve_queue.sh`)**:
   - Parse `evaluate <url>` args. Canonicalize URL (lowercase host, strip trailing `/`, strip query params: `utm_*`, `source`, `ref`, `fbclid`).
   - Pre-flight: `uname -s = Darwin`; `git status --porcelain TODOS.md` clean; `flock` on lock file.
   - Compute interim hash from canonical URL alone (subagent's `pattern_name` extends it later).
   - Hand off to SKILL.md, which instructs the main agent to dispatch the subagent (see "Subagent dispatch protocol" below). Bash CANNOT dispatch agents — the script writes a JSON request file `.claude/tmp/CJ_improve-queue.request.json` and exits 0; SKILL.md's prose instructs the main agent to call `Agent` with the request payload and then re-invoke `scripts/improve_queue.sh apply <verdict-file>` to perform the append.
2. **Subagent (Agent tool, general-purpose, fresh context)**: reads URL via WebFetch + scans in-scope SKILL.md files + returns structured verdict (see Subagent Contract). On WebFetch failure (rate-limit, 4xx, paywall): subagent returns `{verdict: "fetch_failed", error: "..."}`; envelope emits a stderr line and exits 0 without appending a row.
3. **Apply step (`scripts/improve_queue.sh apply <verdict-file>`)**:
   - Read verdict JSON.
   - If `verdict ∈ {"match", "reject"}`: emit stderr log line, NO row append, exit 0.
   - If `verdict ∈ {"conflict", "novel"}`: compute final signature `sha256(canonical_url + pattern_name)[:16]`. Idempotency probe: `grep -Fq "impr-sig=$SIG" TODOS.md` → if hit, NO-OP exit 0.
   - Otherwise: mint synthetic ID `I<NNNNNN>` where N is `(max existing I-ID in TODOS.md) + 1`, defaulting to 1. Teach `/CJ_suggest` (S000045, follow-on) to treat the `I` prefix as orphan-equivalent for ranking (same scoring path as today's orphan rows, no tracker join attempted).
   - Build the row (see Output Format below). Append via atomic mktemp + mv. Validation: post-write, run `/CJ_suggest --limit 1`; **assert exit 0**. Non-zero → restore from backup, exit 1 with parser-error stderr.
4. Print stdout summary: `[CJ_improve-queue] appended TODOS.md row I<NNNNNN>: "<pattern-name>" (P3, M, affects N skills, verdict=<v>, confidence=<C>/10). Run /CJ_suggest to see ranking.`

**Output format (final row in TODOS.md)**:

```markdown
### DRAFT — I<NNNNNN> Adopt <pattern-name> from <short-source-name> (P3, M)
**Source:** <canonical-url>
**Verdict:** <novel|conflict>
**Why:** <one-line summary of the source's recommendation>
**Affected skills:** skills/<skill-a>/SKILL.md, skills/<skill-b>/SKILL.md
**Suggested change:** <one-line>
**Source quote:** "<verbatim quote, hard-cut at 200 chars, ellipsis appended if truncated>"
**Generated by:** /CJ_improve-queue evaluate on 2026-05-15
<!-- impr-sig=<sha256-truncated-16> impr-conf=<N>/10 -->

```

The `DRAFT —` heading prefix opts the row out of `/CJ_suggest` ranking. The signature + confidence live in a trailing HTML comment so they never appear in the rendered top-5 markdown table. User promotes by removing `DRAFT —`; that's the explicit consent gate. `/CJ_suggest` will need a one-line filter addition (S000045, follow-on) to skip `^### DRAFT —` headings the same way it skips strikethrough.

**Default priority/size**: P3, M. Priority chosen so that even after the user promotes from DRAFT, the row sits below P1/P2 user-authored work and above P4 orphans — visible but not displacing real work.

### Subagent dispatch protocol

The SKILL.md preamble instructs the main agent to:

1. Run `scripts/improve_queue.sh evaluate-prepare <url>`. This script does canonicalization, preflight, and writes `.claude/tmp/CJ_improve-queue.request.json` containing:
   ```json
   {
     "canonical_url": "https://docs.anthropic.com/claude-code/some-pattern",
     "in_scope_skill_files": ["skills/CJ_suggest/SKILL.md", "skills/CJ_goal_todo_fix/SKILL.md", ...],
     "request_id": "<uuid>"
   }
   ```
2. Invoke `Agent` tool with `subagent_type: general-purpose`, prompt: (see Subagent Contract below). Agent returns the verdict JSON.
3. Write the verdict to `.claude/tmp/CJ_improve-queue.verdict-<request_id>.json`.
4. Run `scripts/improve_queue.sh apply .claude/tmp/CJ_improve-queue.verdict-<request_id>.json`.

This split lets bash own the deterministic parts (canonicalization, atomic write, idempotency probe, validation) and the Agent own the reasoning. Each side has a strict contract.

### Subagent Contract

**Input prompt template** (passed to `Agent` by the main agent):

```
You are reviewing an article about Claude Code best practices to determine
whether it suggests a pattern this skill workbench should adopt.

Article URL: <canonical_url>
Scope of skills to compare against: <list of in_scope_skill_files>

Steps:
1. WebFetch the URL.
2. Identify the single primary recommended pattern. Name it concisely
   (kebab-case, ≤40 chars), e.g. "explicit-allowed-tools-list" or
   "post-edit-hooks".
3. Read each in-scope SKILL.md file.
4. Classify:
   - "match"    — pattern already present in ≥1 skill, no action needed
   - "conflict" — pattern conflicts with current approach in ≥1 skill
   - "novel"    — pattern not present, would apply to ≥1 skill
   - "reject"   — pattern not applicable to this workbench
   - "fetch_failed" — WebFetch errored or returned paywall/404

Return ONLY a JSON object (no prose, no markdown fences) with this schema:

{
  "verdict": "match"|"conflict"|"novel"|"reject"|"fetch_failed",
  "pattern_name": "kebab-case-name",
  "confidence": <integer 1-10>,
  "affected_skills": ["skills/CJ_X/SKILL.md", ...],   // empty array if verdict=reject|fetch_failed
  "source_quote": "verbatim quote from article, ≤200 chars",  // empty string if fetch_failed
  "suggested_change": "one-line description of the change to make",  // empty if match|reject|fetch_failed
  "error": "..."   // present only when verdict=fetch_failed
}

If confidence <7, prepend the suggested_change with "REVIEW: ".
Do not invent skill paths — only reference files actually present in the
in_scope_skill_files list.
```

**Fallback when verdict is unparseable**: shell envelope's `apply` step validates the JSON via `jq -e '.verdict'`. On parse failure: emit stderr line `[CJ_improve-queue] subagent returned unparseable verdict; no row appended`, exit 0 (non-fatal — user can retry).

**Test fixture for `/CJ_qa-work-item`**: check in `tests/fixtures/CJ_improve-queue/sample-verdict-novel.json` and a frozen snapshot of the URL fetch result `tests/fixtures/CJ_improve-queue/sample-fetch-anthropic-skills-page.html`. QA bypasses the live WebFetch (and the subagent) by stubbing both via env vars `CJ_IMPROVE_QUEUE_VERDICT_FILE` and `CJ_IMPROVE_QUEUE_FETCH_FILE`. Live network is exercised manually in the killer-test (Next Steps #5), not in CI.
