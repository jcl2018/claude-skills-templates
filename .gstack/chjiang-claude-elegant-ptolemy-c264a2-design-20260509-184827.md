## Problem Statement

`/personal-pipeline` (v1.13.0, F000014) compresses `/scaffold-work-item` →
`/implement-from-spec` → `/qa-work-item` into one keystroke, but every run
still requires answering 5+ AskUserQuestion gates inline:

- Step 4: confirm scaffolded shape
- Step 5.2: pre-collected sensitive-surface AUQs (catalog/manifest/validator)
- Step 5.2: pre-collected taste-fork AUQs from SPEC tradeoffs
- Step 5.3: ESCALATION_NEEDED from implement subagent
- Step 6: post-implement gate red (validate.sh fail)
- Step 8: post-QA gate red
- Step 9: sunset checkpoint (every 5 invocations from the 6th onward)

The orchestrator removed the manual scaffolding/implement/QA labor; the
human-decision labor stayed. Each pipeline run is 5-30 minutes wall-clock
with 5+ interruptions.

`/autoplan` (gstack) solved the same shape problem for plan reviews: read the
relevant skills at full depth, but auto-decide intermediate AUQs using 6
principles, classify each decision (Mechanical / Taste / User Challenge), and
surface only the close calls at one final approval gate. This design ports
that contract to /personal-pipeline as a `--auto` flag.

This extends [chjiang-main-design-20260508-102829.md](chjiang-main-design-20260508-102829.md)
(the F000014 source design that established the 3-skill pipeline + sensitive-surface
pre-collection contract).

## What Makes This Cool

On a green-or-recoverable path: one keystroke from design doc to "ready to
/ship" review. The pipeline runs end-to-end answering its own AUQs by
principle, halts only on structural drift, gate-red, or hard-stop User
Challenges, and presents one terminal review gate showing every Taste +
Approve-with-surfacing User-Challenge decision it made (Mechanical
decisions are silent + counted only). The human reads ONE structured
summary instead of answering inline questions across a 5-30 minute run.

**Honest contract bounds:**
- **Not fire-and-forget.** Step 8.5 final approval gate is mandatory on the
  green-or-recoverable path.
- **Not a separate skill.** `--auto` is a flag on the existing /personal-pipeline.
- **Step 8.5 is conditional.** Halt-at-Gate User Challenges (gate-red, hard
  ESCALATION_NEEDED) halt mid-pipeline — they never reach 8.5. The "single
  final gate" promise applies only to runs where every User Challenge is an
  Approve-with-surfacing type (sensitive-surface approvals).

## When auto mode is worth it (effort/pain ratio)

The reviewer rightly flagged that the median single-work-item run hits only
1 AUQ (Step 4 confirm-shape) — saving "5+ AUQs" overstates the median win.
The real wins compound:

| Use case | Manual AUQs | Auto AUQs | Win |
|---|---|---|---|
| Single SPEC-light user-story | 1 (Step 4) | 0-1 (one final gate, possibly empty) | marginal |
| SPEC with N sensitive-surface paths | 1 + N | 1 (final gate, N items) | +N→1 |
| SPEC with M Tradeoff taste-forks | 1 + M | 1 (final gate, M items) | +M→1 |
| Back-to-back K work items × J AUQs each | K × J | K (one gate per run) | K(J-1) |
| Bad-path run (gate red) | 1 + interruption | halt at gate (no win) | 0 |

The design pays for itself most when: (a) you're doing multi-tradeoff SPECs,
(b) you're working through a queue of related work items, or (c) you want
the "audit trail" of decisions logged for review. The single SPEC-light
user-story case stays a thin win — that's honest, not a regression.

## Constraints

- **Manual mode contract preserved.** v1.13.0's behavior is byte-identical
  when `--auto` is absent. Auto mode is purely additive code.
- **Sensitive surfaces always reviewed.** Catalog, manifest, validator,
  template, git-hook changes are User Challenges by design — surfaced at the
  final gate, never auto-bypassed.
- **Boundary checks mandatory.** `/personal-workflow check` red is a halt
  regardless of mode.
- **No chain into /ship.** Auto mode stops at "QA green + final gate
  approved". /ship is a separate user invocation with its own adversarial
  review.
- **Subagents still cannot call AUQ** (S000026 spike). Auto-mode doesn't
  change that; pre-collected AUQs at the orchestrator still apply, just with
  auto-decided answers in auto mode.
- **Workbench-only scope.** This is a skill in claude-skills-templates.
  Distribution is via existing `skills-deploy install`. No new artifact, no
  new CI plumbing.

## Premises (all approved in D3)

1. **Additive flag, not separate skill.** `--auto` parses at Step 1; manual
   path is the default.
2. **6 principles tilt safer.** Replace /autoplan's P6 ("Bias toward action")
   with "Bias toward halt-on-doubt" — code-mutating pipeline has higher blast
   radius than plan review.
3. **Sensitive-surface AUQs are User Challenges.** Catalog / manifest /
   validator / template / git-hook surfaces always flow to the final
   approval gate with full context.
4. **Boundary check red is mandatory halt.** `/personal-workflow check` red
   does not get auto-overridden.
5. **No chain into /ship.** Auto mode stops at green pipeline + final gate.
6. **Shared telemetry + sunset.** Same JSONL, one new field `mode: auto|manual`.
   Sunset trip-wire counts both modes.

## Approaches Considered

### Approach A: `--auto` flag on /personal-pipeline (recommended)

- **Summary:** SKILL.md parses `[--auto]`, pipeline.md gains an "Auto Mode
  Overlay" section + a new Step 8.5 final approval gate. Each existing AUQ
  wraps `if AUTO_MODE: auto_decide else: auq`.
- **Effort:** human ~3h / CC ~30 min
- **Risk:** Low — manual path untouched, auto path additive
- **Pros:** consistent with premise 1; minimum maintenance surface; reuses
  pipeline.md's 9 steps; zero regression risk on manual mode
- **Cons:** Auto Mode Overlay section is dense (6 principles + classification
  + per-gate wrapping in one place); pipeline.md grows ~25%
- **Reuses:** 100% of pipeline.md's 9 steps, /autoplan's 6-principle +
  classification framework, F000014 sensitive-surface pre-scan logic

### Approach B: New `/personal-pipeline-auto` skill

- **Summary:** separate skill at `skills/personal-pipeline-auto/`, mirrors
  /autoplan as its own command. Either calls /personal-pipeline as subagent
  (broken — AUQ unreachable per S000026) or duplicates orchestration logic.
- **Effort:** human ~6h / CC ~1h
- **Risk:** Medium — double-maintenance burden; new test surface
- **Pros:** discoverable in skills-catalog.json as its own command; mirrors
  gstack's `/autoplan` convention exactly; cleaner separation
- **Cons:** violates premise 1; subagent-wraps-skill pattern is broken
  (S000026); pipeline.md changes must be replicated; double the test work
- **Reuses:** ~70% if pipeline.md is shared via path-resolution fallback;
  ~0% if duplicated

### Approach C: Per-user opt-in via gstack-config

- **Summary:** no flag. /personal-pipeline reads
  `gstack-config get personal_pipeline_auto`; if true, auto mode runs
  automatically. Override: explicit `--manual` flag.
- **Effort:** human ~4h / CC ~45 min
- **Risk:** Medium — "set and forget" surprise on sensitive-surface
  decisions; partially violates premise 1
- **Pros:** zero invocation friction once configured; matches existing
  gstack-config patterns (checkpoint_mode, telemetry)
- **Cons:** subtle invocation semantics — same command does different things
  on different machines; conflicts with F000014's "user actively chooses for
  this invocation" safety motivation
- **Reuses:** same as A plus gstack-config's get/set scaffolding

## Recommended Approach

Approach A — `--auto` flag on /personal-pipeline.

### The 6 Decision Principles (auto-mode tuned)

Direct port of /autoplan's principles with one substitution: P6 becomes
halt-on-doubt instead of bias-toward-action, reflecting the higher blast
radius of code-mutating pipeline vs plan-review.

1. **Choose completeness** — pick the option that covers more edge cases /
   acceptance criteria.
2. **Boil lakes (in-blast-radius)** — auto-approve in-radius expansions
   < 5 files, no new infra.
3. **Pragmatic** — between two approaches that fix the same thing, pick the
   cleaner one.
4. **DRY** — duplicates existing functionality? Pick reuse.
5. **Explicit over clever** — 10-line obvious > 200-line abstraction.
6. **Bias toward halt-on-doubt** *(REPLACED P6)* — when uncertain, halt and
   surface at final gate, do not auto-approve. Code-mutating pipeline has
   higher blast radius than plan review.

**Per-gate classification table:**

| Gate | Dominating principles | Auto-mode classification | Halts pipeline? |
|---|---|---|---|
| Step 2 branch (b) — footer present, path missing | P6 | User Challenge — Halt-at-Gate (recommend "Re-scaffold") | yes — never reaches 8.5 |
| Step 2 branch (c) — footer absent, path present | P6 | User Challenge — Halt-at-Gate (recommend "Halt for manual cleanup") | yes — never reaches 8.5 |
| Step 4 scaffold-shape (single-story, check green) | P5 + P3 | Mechanical (Approve, silent) | no |
| Step 4 scaffold-shape (feature with children) | (existing pipeline halts before AUQ) | n/a — multi-story branch already halts at sub-step 3 with `end_state=green` | yes (manual mode parity) |
| Step 5.2 sensitive-surface | P6 + premise 3 | User Challenge — Approve-with-surfacing (auto-pick approve, log, surface at 8.5) | no |
| Step 5.2 taste-fork | P5 + P1 | Taste (auto-pick per P3, surface at 8.5) | no |
| Step 5.3 ESCALATION_NEEDED — first occurrence | (existing one-retry per pipeline.md:271 fires first) | retry once silently | no (yet) |
| Step 5.3 ESCALATION_NEEDED — retry also failed | P6 | User Challenge — Halt-at-Gate (recommend abort) | yes — never reaches 8.5 |
| Step 6 post-implement validate.sh red | P6 | User Challenge — Halt-at-Gate (recommend abort) | yes — never reaches 8.5 |
| Step 8 post-QA red | P6 | User Challenge — Halt-at-Gate (recommend abort) | yes — never reaches 8.5 |
| Step 9 sunset checkpoint | (always-AUQ in both modes — by design) | Always interactive (per-invocation visible regardless of mode) | no (gate at end of run, after 8.5 if 8.5 fired) |

**Two halt categories — distinct logging contracts:**

1. **Halt-regardless** (do NOT log to `$DECISION_LOG`; not a "decision" the
   orchestrator made):
   - Boundary check red — `/personal-workflow check` at Steps 2(a)/4/6/8
     red is a hard halt. Tracker journal + telemetry only;
     `end_state=halted_at_gate`.
   - Subagent crash — empty/no RESULT line. Tracker journal + telemetry
     only; `[subagent-crash]` entry; `end_state=subagent_crashed`.

2. **Halt-at-Gate User Challenge** (DO log to `$DECISION_LOG` with class
   `user_challenge_halt` for audit, then halt):
   - Step 2 b/c partial-write recovery
   - Step 5.3 ESCALATION_NEEDED retry-also-failed
   - Step 6 post-implement validate.sh red
   - Step 8 post-QA red

The distinction: halt-regardless is a structural/integrity failure (the
orchestrator didn't make a decision; the system did). Halt-at-Gate is a
decision the orchestrator made (recommend abort) that the user could in
principle override but won't, so the run halts. Logged so 8.5 reviewers
of *prior* runs can audit what halted them.

**Concrete P6-vs-old-P6 example.** Suppose a SPEC has a Tradeoff row with
two implementations: (i) reuse existing helper at +5 lines, (ii) inline
specialization at +20 lines. Old-P6 (bias toward action) and the cleaner-
option principle both pick (i). No conflict. But suppose row reads:
"reuse helper (+5 lines, requires changing helper signature affecting 3
other callers) vs inline (+20 lines, isolated)." Old-P6 would still pick
(i) — bias toward shipped action. New-P6 (halt-on-doubt) sees blast
radius hitting 3 unrelated callers and recommends inline + surface as
Taste. P6 changes the answer here. This is the contrast you bought.

### Decision Classification (User Challenge split into two sub-types)

- **Mechanical** — auto-decide silently, log to `$DECISION_LOG` with class
  `mechanical`, count only at 8.5. Examples: scaffold-shape Approve when
  `/personal-workflow check` passes + single-story shape; sunset KEEP when
  trip-wire below threshold.
- **Taste** — auto-decide with recommendation, log to `$DECISION_LOG` with
  class `taste`, surface at 8.5 with reasoning. Examples: in-blast-radius
  taste-fork choices in SPEC tradeoffs; feature shapes with multiple
  children.
- **User Challenge — Approve-with-surfacing** — auto-pick "approve"
  forward (so the implement subagent can proceed), log with class
  `user_challenge_approved`, surface at 8.5 with full context. If the user
  rejects at 8.5, end_state = `user_aborted`; user reverts manually
  (`git restore`, `git stash`, or revert the work-item dir). Examples:
  sensitive-surface approvals at Step 5.2 (catalog/manifest/validator/
  template/git-hook).
- **User Challenge — Halt-at-Gate** — halt at the originating step;
  surface inline, do NOT auto-pick "approve" (subagent has nothing to
  proceed with). Step 8.5 never fires for these runs. Examples:
  ESCALATION_NEEDED (after the existing one-retry already failed),
  post-implement validate.sh failure, post-QA red.

**Why two User-Challenge sub-types:** the gate at Step 5.2 has a
"forward-thread" decision — auto mode picks an answer, the subagent uses
it, code mutates. Reviewing at 8.5 lets you confirm or revert. The gates
at Steps 5.3/6/8 don't have a "forward" — gate-red can't be auto-decided
because there's nothing TO decide; the next phase has nothing to consume.
Conflating both as "User Challenge" was the doc's first-draft bug.

**Rollback story (User-Challenge Approve-with-surfacing rejected at 8.5):**
v1 = abort + manual revert. We do NOT attempt programmatic rollback
(git-restore is fragile across mid-pipeline subagent edits, and the work-
item dir + repo files are mutated together). The user runs
`git status` + `git restore` themselves. Step 8.5's "Reject specific
decisions" option is therefore renamed to **"Abort + show what to revert"**
in v1: prints the list of files mutated under each rejected decision.
Reviser-friendly, but no auto-rollback.

### Pipeline.md changes

#### 1. New section "Auto Mode Overlay" (top of pipeline.md, after the
universal RESULT contract)

Defines `$AUTO_MODE`, the 6 principles, the classification rules, the
$DECISION_LOG path, and the halt-regardless exceptions (boundary check
red, subagent crash — both skip $DECISION_LOG entirely). Cross-references
each step's specific auto-mode behavior in the table above.

**`$DECISION_LOG` is a single shared file** (consistent with
`~/.gstack/analytics/personal-pipeline.jsonl` telemetry, which is also
single-file across all runs):

```
~/.gstack/analytics/personal-pipeline-auto-decisions.jsonl
```

Every line tagged with `run_id` so Step 8.5 grep-filters to the current
run only. No rotation needed for v1 (one-line-per-decision append, dwarfed
by repo size); revisit if the file exceeds 1MB.

**Path is computed once at Step 1** and stored in the orchestrator's
context as `$DECISION_LOG`; any later Bash block can re-derive it from
the constant string (no shell-variable persistence assumed across Bash
calls — model carries the literal path).

One line per decision (jq-emitted, JSON-safe):

```json
{"run_id":"20260510-093015-12345","step":"5.2","gate_id":"sensitive-surface-catalog","classification":"user_challenge_approved","decision":"approve","recommendation":"approve","reasoning":"in-blast-radius catalog wiring; SPEC matches existing pattern","context_missing":"none flagged","files_affected":["skills-catalog.json","README.md"],"ts":"2026-05-10T09:32:11Z"}
```

Classification values: `mechanical`, `taste`, `user_challenge_approved`,
`user_challenge_halt` (the last one is logged for audit even though it
halts the run; `end_state=halted_at_gate`).

#### 2. New Step 8.5: Final Approval Gate (auto mode only)

Runs only if `$AUTO_MODE=true` AND we've reached this step (no
halt-at-gate User Challenges fired). The orchestrator has AUQ available
(SKILL.md `allowed-tools` includes AskUserQuestion) — this step is at the
orchestrator level, not in a subagent.

1. Filter `$DECISION_LOG` to this run:
   ```bash
   jq -c "select(.run_id == \"$RUN_ID\")" "$DECISION_LOG" > /tmp/decisions-$RUN_ID.jsonl
   ```
2. **Empty-state fallback:** if the filtered file is empty (no Taste, no
   User-Challenge-Approved fired — only Mechanical decisions and they're
   silent + count-only), write `[auto-pipeline-clean]` to tracker journal
   and short-circuit to Step 9 (no AUQ surfaces). The auto run was
   genuinely uneventful; nothing to confirm.
3. Otherwise, group by classification: count `mechanical`, list `taste`,
   list `user_challenge_approved`. Render via the gstack AUQ format with
   ALL required fields (D-numbered header, project line, ELI10, Stakes,
   Recommendation, Completeness/kind-note, ≥40-char Pros/Cons per option,
   Net):

```
D-AUTO-FINAL — Confirm auto-mode decisions
Project/branch: <repo>/<$_BRANCH> | Run: $RUN_ID
ELI10: I made $((M+K)) close-call decisions while running your pipeline.
The Mechanical ones were obvious and silent. The $M Taste and $K
User-Challenge-Approved decisions below are the close calls I'd like you
to confirm before you /ship. Each User Challenge auto-picked "approve"
and threaded that answer to the implement subagent — files were already
written. If any of these picks was wrong, you Abort here and I print the
exact file list for you to revert by hand.
Stakes if we pick wrong: Approve = you live with the auto-picks (revert
later via /ship review or normal git workflow). Abort = you revert now
and re-run manually; ~3-5 min lost.

Recommendation: Approve (recommended) — all $K User Challenges were
auto-picked "approve" with reasoning; reversible by `git restore` on the
affected files if you disagree.
Note: options differ in kind, not coverage — no completeness score.

Decision summary:
- $N mechanical (silent, logged; expand on request)
- $M taste:
    [for each: Gate <step.gate_id> | Pick: <decision> | Reasoning:
     <one sentence per dominating principle>]
- $K user_challenge_approved:
    [for each: Gate <step.gate_id> | Pick: approve | Files: <list>
     | What we approved: <one-liner> | Why approve looked right: <reasoning>
     | Context we could be missing: <context_missing>
     | If we're wrong, cost is: <one sentence>]

Log: $DECISION_LOG (filter run_id=$RUN_ID)

A) Approve all auto-decisions (recommended)
  ✅ Pipeline ends green, ready for /ship review with full audit trail
  ✅ Every User Challenge has a logged file list — easy to spot-revert later
  ❌ You commit to $K sensitive-surface picks at once instead of one-by-one

B) Abort + show what to revert
  ✅ Prints per-decision files_affected list grouped by gate for easy revert
  ✅ Pipeline state preserved — no programmatic rollback gone wrong
  ❌ Manual revert + re-run loses ~3-5 minutes vs surgical "reject this one"

Net: Approve when you trust the reasoning summaries; Abort when any one
User Challenge surfaces a context the orchestrator clearly didn't have.
```

4. AskUserQuestion (single AUQ, two options — gstack format above):

   v1 deliberately drops "Reject specific decisions" — see Rollback story
   above. We don't have a programmatic rollback for partial Approve.

5. **On Approve:** write `[auto-final-gate-approved]` to tracker journal;
   set `$END_STATE=green`; continue to Step 9.
6. **On Abort:** print the file-revert summary (group `files_affected` by
   decision); `$END_STATE=user_aborted`; write telemetry; exit.

#### 3. Existing steps: minimal touches

- **Step 1 (input validation):**
  - Parse `--auto` flag at the front of `$@` before `$DESIGN_DOC`. Set
    `$AUTO_MODE=true|false`.
  - Derive `$RUN_ID` (existing).
  - Set `$DECISION_LOG=~/.gstack/analytics/personal-pipeline-auto-decisions.jsonl`
    (constant path; no `$SLUG` indirection — analytics is workbench-flat).
  - `mkdir -p ~/.gstack/analytics` (idempotent, matches telemetry's
    existing call at Step 9.1).
- **Step 2 b/c AUQs:** classify as User Challenge — Halt-at-Gate (per
  table). Auto mode does NOT auto-decide partial-write recovery — these
  states require manual cleanup.
- **Step 4 AUQ:** classify per the table.
- **Step 5.2 sensitive-surface AUQs:** ALWAYS User Challenge —
  Approve-with-surfacing. Auto-pick "approve" forward (subagent proceeds
  in Step 5.3); log with `files_affected` from the SPEC's Components
  Affected table; surface at 8.5. v1 ships the simple rule (always
  approve, always surface); the validate.sh-without-TODOS-entry carve-out
  is deferred to Open Q4.
- **Step 5.2 taste-fork AUQs:** Taste. Auto-pick per P3 (cleaner option,
  fallback to first-listed); thread the answer to subagent; surface at
  8.5.
- **Step 5.3 ESCALATION_NEEDED:** the existing one-retry per
  `pipeline.md:271` fires first (silently in auto mode). On retry-also-
  fails: classify as User Challenge — Halt-at-Gate; halt with `[gate-red]`.
- **Step 6 post-implement red:** User Challenge — Halt-at-Gate; halt with
  `[gate-red]`. Log decision before halting.
- **Step 8 post-QA red:** User Challenge — Halt-at-Gate; halt. Log decision
  before halting.
- **Step 9 sunset checkpoint:** always-AUQ in both modes. Not a contract
  exception — sunset is per-invocation visible by design (counts pooled
  across modes per premise 6).

### Acceptance Criteria

1. `/personal-pipeline <doc>` (no flag) executes the same Bash blocks,
   dispatches the same subagent prompts, and AUQs at the same gates as
   v1.13.0. Verification: spot-check by diffing the manual code path
   regions of pipeline.md against the v1.13.0 baseline; all auto-mode
   logic is gated on `$AUTO_MODE=true`.
2. `/personal-pipeline --auto <doc>` runs through scaffold/implement/QA
   without per-gate user prompts on the green-or-recoverable path.
3. Step 5.2 sensitive-surface AUQ in auto mode writes a
   `user_challenge_approved` line to `$DECISION_LOG` (with `run_id`,
   `files_affected`, reasoning) and surfaces at Step 8.5.
4. Boundary check red halts auto mode the same as manual mode: no
   `$DECISION_LOG` write; tracker journal + telemetry only;
   `end_state=halted_at_gate`.
5. Subagent crash halts auto mode the same as manual mode: no
   `$DECISION_LOG` write; `[subagent-crash]` journal entry;
   `end_state=subagent_crashed`.
5b. Halt-at-Gate User Challenges (Step 5.3 retry-failed, Step 6, Step 8)
    DO write a `user_challenge_halt` line to `$DECISION_LOG` before
    halting; `end_state=halted_at_gate`. Distinct from halt-regardless
    above.
6. Step 8.5 final gate shows: count of `mechanical` decisions, full list
   of `taste` decisions with reasoning, full list of `user_challenge_approved`
   with full context block (per the gstack AUQ format above).
7. Empty-state at Step 8.5 (no Taste, no User-Challenge-Approved
   decisions): writes `[auto-pipeline-clean]` to tracker journal and
   short-circuits to Step 9 without surfacing an AUQ.
8. Abort at Step 8.5 sets `end_state=user_aborted`; prints per-decision
   `files_affected` summary for the user to revert manually;
   telemetry written; pipeline state preserved.
9. Telemetry line at Step 9.1 includes `mode: auto|manual` field.
10. Sunset checkpoint counts both modes (`auto`+`manual` lines pooled);
    trip-wire fires at invocation 6 then every 5 thereafter regardless
    of mode.
11. **Multi-classification synthetic SPEC test:** SPEC with 1 sensitive
    surface (catalog) + 2 Tradeoff taste-forks + green QA produces
    Step 8.5 summary with $N≥1 mechanical, $M=2 taste, $K=1
    user_challenge_approved. Counts match `jq` filter on
    `$DECISION_LOG`.

## Open Questions

1. **Auto mode + dry run.** Out of scope for v1; future flag combination
   if we want to preview decisions without applying.
2. **Multi-User-Challenge UX at final gate.** If $K is large (say > 5),
   the single AUQ becomes a wall of text. Acceptable for v1; revisit if
   real usage hits this regularly. Possible v1.1: collapse $N+$M into a
   table, expand $K full-context blocks one-at-a-time.
3. **`$DECISION_LOG` size growth.** Single shared file at
   `~/.gstack/analytics/personal-pipeline-auto-decisions.jsonl`. Append-
   only, no rotation in v1 (matches the existing telemetry pattern). If
   the file exceeds 1MB (~10k decisions), revisit.
4. **Carve-out heuristic for `validate.sh` sensitive surface.** The
   "downgrade to flag context_missing if no TODOS.md entry" rule is
   admittedly hand-wavy. Could be replaced with a simpler
   "always recommend approve, never abort, surface at 8.5 unconditionally"
   rule. Default to the simpler rule for v1; revisit if real runs show
   sensitive-surface false-approves.

## Success Criteria

- One real run of `/personal-pipeline --auto` on a TODOS.md item completes
  end-to-end with one final gate AUQ.
- Decision log is scannable in <30s before approving.
- Manual mode is provably unchanged (skills-catalog.json + pipeline.md diff
  inspection on the manual code path).
