## Problem Statement

The /docs check skill validates work items for template compliance (Check 1), lifecycle consistency (Check 2), and PRD-to-TEST-SPEC traceability (Check 3). But it does NOT check structural completeness. F000002_system_health_v1 has all its doc artifacts (TRACKER, PRD, ARCHITECTURE, TEST-SPEC, milestones) but zero child user stories or tasks. It passes all three existing checks. Nobody flags that a feature without decomposition is structurally incomplete.

F000001_workflow_alpha is the exemplar: feature with two user stories (S000001, S000002) and a task (T000001) nested properly. F000002 should fail because it has none.

## What Makes This Cool

This turns `/docs check` from "did you fill out the forms correctly" into "is your work actually decomposed." The tree report shows the entire hierarchy at a glance with per-node badges. And the graph artifact (`work-item-graph.json`) is the foundation for the dependency graph north star — traceability links, blocked-by relations, impact analysis. Build the graph now, visualize it later.

Codex nailed it: "They're not excited by validation. They're excited by having a navigable model of the work."

## Constraints

- Must extend existing check.md (new Check 4 + new tree output section)
- Must reuse Step 10's Actual Model — no separate parser
- Must handle the same normalization rules from Step 8 (type spelling, filename matching). Note: Step 8 defines a 2-level template fallback chain, while rules/work-items.md and CLAUDE.md document a 3-level chain (adding `~/.claude/spec/templates/`). This pre-existing inconsistency is outside scope of this design but should be tracked for a future fix.
- Structural rules are absolute: no escape hatch, no `atomic: true` override
- If something doesn't need decomposition, it's the wrong type (use task, not feature)
- Tree report format must be human-readable in terminal output
- Graph artifact must be machine-readable JSON
- Must gracefully handle repos with no work-items/ (skip, same as existing checks)

## Premises

1. A feature MUST have >= 1 user-story child. Always flagged, no escape hatch.
2. A user-story MUST have >= 1 task child. Same rule.
3. Validation only — flag violations, don't block scaffolding.
4. Lives in /docs check as Check 4, reusing the Actual Model already built by Step 10.
5. Output is a tree report showing the full hierarchy with completeness indicators.
6. Dependency graph (blocked-by, depends-on, traceability) is north star, ships later.

## Cross-Model Perspective

Codex (independent cold read) proposed a "project topology compiler" — not just a lint check but an executable delivery model with impact analysis (change a PRD story and it tells you which TEST-SPEC rows and tasks are stale) and topology diff vs merge-base (showing what a PR added or broke structurally).

Key Codex insight: "They're not excited by validation — they're excited by having a navigable model of the work." This confirmed the tree report approach over flat section output.

Codex also suggested emitting `work-item-graph.json` alongside the human report — build the graph artifact now so the north star is cheap later. This was adopted into Approach C.

Closest existing tool: OpenProject (typed work items, parent/child, dependency relations) — but missing repo-native markdown as source of truth and hard structural invariants.

## Approaches Considered

### Approach A: Minimal Check Extension
Add Check 4 to check.md. Walk the Actual Model, flag features with 0 user-story children and user-stories with 0 task children. Section-based output like existing checks.
- Effort: S (CC: ~15 min)
- Risk: Low
- Completeness: 6/10
- Rejected: no tree visualization, no graph artifact, no path to north star

### Approach B: Tree Report + Structural Check
Add Check 4 with tree-first output. Walk the hierarchy depth-first, print each node with completeness indicators. Flag structural violations inline in the tree.
- Effort: M (CC: ~25 min)
- Risk: Low
- Completeness: 8/10
- Considered: good visualization but no graph artifact

### Approach C: Tree Report + Graph Artifact (Chosen)
Approach B plus emit `work-item-graph.json` — a machine-readable graph of the full hierarchy with node types, states, children, and structural completeness status.
- Effort: M (CC: ~30 min)
- Risk: Low
- Completeness: 9/10
- Chosen: graph artifact is ~5 extra minutes and sets up the navigable model north star

## Recommended Approach

Approach C: Tree Report + Graph Artifact

### Architecture

#### Check 4: Structural Completeness (check.md Steps 15-16)

**Step 15: Define structural rules**

Hard-coded type hierarchy requirements (derived from rules/work-items.md directory structure documentation):

```
required_child_type:
  feature    -> user-story  (min: 1)
  user-story -> task        (min: 1)
  task       -> (none)
  defect     -> (none)
```

For each work item in the Actual Model:
1. Look up its normalized type in the rules above
2. If the type has a required child type:
   - Count children of the required type (using normalized type comparison)
   - If count == 0: flag `[INCOMPLETE] {slug} — {type} has 0 {required_child_type} children (minimum: 1)`
   - If count >= 1: `[PASS] {slug} — {count} {required_child_type} children`

Note: always use the singular form of the type name regardless of count (e.g., "2 user-story children", not "2 user-stories children"). Use "child" when count == 1, "children" when count > 1 (e.g., "1 user-story child", "2 user-story children").
3. If the type has no required child type: `[PASS] {slug} — no structural requirements`

**Step 16: Tree Report**

After all four checks run, emit a unified tree view of the work item hierarchy. Walk work-items/ depth-first, sorting siblings alphabetically by slug at each level. Print each node with badges:

```
WORK ITEM TREE:
  F000001_workflow_alpha (feature) [Open]
    template: PASS  lifecycle: PASS  traceability: PASS  structure: PASS
    S000001_four_phase (user-story) [In Progress]
      template: PASS  lifecycle: PASS  traceability: PASS  structure: PASS
      T000001_router_implementation (task) [Open]
        template: PASS  lifecycle: PASS  structure: PASS
    S000002_template_consolidation (user-story) [Open]
      template: PASS  lifecycle: PASS  traceability: PASS  structure: INCOMPLETE (0 task children)

  F000002_system_health_v1 (feature) [Open]
    template: PASS  lifecycle: PASS  traceability: PASS  structure: INCOMPLETE (0 user-story children)
```

Badge definitions (ordered by severity, lowest to highest):
1. `PASS` — all checks passed for this dimension
2. `WARN` — advisory issue (lifecycle warning, extra sections, etc.)
3. `DRIFT` — template compliance drift (missing fields/sections)
4. `INCOMPLETE` — structural completeness violation (missing required children)

Each badge summarizes the worst result (highest severity) from its check category for that node. For example, if Check 1 found 2 DRIFT issues and 1 PASS, the template badge shows `DRIFT`.

**Step 17: Graph Artifact**

After the tree report, emit `work-item-graph.json` to the `.docs/` directory (alongside claims.json). Create `.docs/` if it does not exist:

```json
{
  "version": "1.0.0",
  "generated_at": "2026-04-11T23:00:00Z",
  "generated_commit": "8cad743",
  "nodes": [
    {
      "id": "F000001",
      "slug": "F000001_workflow_alpha",
      "type": "feature",
      "state": "Open",
      "path": "work-items/F000001_workflow_alpha",
      "parent": null,
      "children": ["S000001", "S000002"],
      "badges": {
        "template": "PASS",
        "lifecycle": "PASS",
        "traceability": "PASS",
        "structure": "PASS"
      }
    },
    {
      "id": "S000001",
      "slug": "S000001_four_phase",
      "type": "user-story",
      "state": "In Progress",
      "path": "work-items/F000001_workflow_alpha/S000001_four_phase",
      "parent": "F000001",
      "children": ["T000001"],
      "badges": {
        "template": "PASS",
        "lifecycle": "PASS",
        "traceability": "PASS",
        "structure": "PASS"
      }
    }
  ],
  "edges": [],
  "structural_rules": {
    "feature": {"required_child": "user-story", "min": 1},
    "user-story": {"required_child": "task", "min": 1}
  }
}
```

The `edges` array is empty in v1 — reserved for blocked-by, depends-on, and traceability links in the north star version.

Schema:
- `nodes[].id`: work item ID (e.g., "F000001")
- `nodes[].slug`: directory name
- `nodes[].type`: normalized type
- `nodes[].state`: Open / In Progress / Closed
- `nodes[].path`: relative path from repo root
- `nodes[].parent`: parent ID or null
- `nodes[].children`: array of child IDs
- `nodes[].badges`: per-check summary (PASS/WARN/DRIFT/INCOMPLETE, severity-ordered)
- `edges`: reserved for dependency graph (v2)
- `structural_rules`: the hierarchy rules being enforced

Output path: `.docs/work-item-graph.json`

### Files to modify

### North star path (not this iteration)

The graph artifact sets up three future capabilities:
1. **Traceability edges**: PRD story → TEST-SPEC AC, user-story → task mapping
2. **Dependency edges**: blocked-by, depends-on between work items
3. **Impact analysis**: given a file change, which graph nodes are affected (combines claims.json evidence with graph topology)
4. **Topology diff**: show what a PR changed structurally (new nodes, removed edges, broken invariants)

## Open Questions

1. Should `work-item-graph.json` be gitignored (generated artifact) or committed (shared state)?
   Recommendation: gitignore it, regenerate on `/docs check`. Same pattern as claims.json.
2. Should the tree report appear in ALL `/docs check` runs, or only when work-items/ exists?
   Recommendation: only when work-items/ exists (consistent with existing skip behavior).
3. Should defects have structural requirements? (e.g., defect must have >= 1 task)
   Recommendation: no, defects and tasks are leaf types. Keep it simple for now.
