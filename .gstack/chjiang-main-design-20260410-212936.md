## Problem Statement

After `/skill-author` completes (scaffold, author, check, ship), the newly created
skill exists in the repo but isn't deployed to `~/.claude/skills/`. The user has to
manually run `scripts/skills-deploy install <name>` to make it usable. This is a gap
in an otherwise end-to-end pipeline.

## What Makes This Cool

Zero-friction skill creation. You invoke `/skill-author my-skill`, answer a few
questions, and when it's done the skill is committed, PR'd, AND immediately usable.
No manual deploy step. The pipeline does the complete thing.

## Constraints

- Must use existing `scripts/skills-deploy install` ... no new install logic.
- Install failure must not fail the overall pipeline (skill is already committed).
- Must respect the resume-on-failure pattern (file existence checks).

## Premises

1. The install step belongs AFTER `/ship` completes (Stage 6), not before ... because
   until the code is committed, the symlink target could change.
2. `skills-deploy install` already handles collision detection with gstack ... no new
   safety logic needed.
3. If install fails (e.g., permission error), it should warn but not fail the overall
   pipeline ... the skill is already committed and PR'd.

## Approaches Considered

### Approach A: New Stage 6 (chosen)

Add a "Stage 6: Install" section to skill-author's SKILL.md. After `/ship` completes,
run `scripts/skills-deploy install <name>` to deploy the symlink.

- Clean separation of concerns (ship and install are different operations)
- Easy to skip or re-run independently
- Pipeline stage count (6) matches the logical steps
- Resume-on-failure: check if symlink already exists before running

### Approach B: Append to Stage 5

Add install as the final step of Stage 5 (Ship), after `/ship` completes.

- Fewer structural changes
- But Stage 5 would do two things with different failure modes
- Harder to skip install without skipping ship

## Recommended Approach

**Approach A: New Stage 6.** Ship and install have different failure modes and different
skip conditions. Keeping them separate preserves the resume pattern where file existence
determines progress.

### Implementation

Add this after Stage 5 in `skills/skill-author/SKILL.md`:

Note: `<name>` is the placeholder pattern used throughout the existing SKILL.md.
Claude substitutes the actual skill name at runtime. `$REPO_ROOT` is defined in
Stage 2 (`REPO_ROOT=$(git rev-parse --show-toplevel)`) and reused by all stages.

```markdown
## Stage 6: Install

Deploy the newly created skill to `~/.claude/skills/` so it's immediately usable.

1. **Re-derive REPO_ROOT** (Claude does not share shell state between tool calls):
   ```bash
   REPO_ROOT=$(git rev-parse --show-toplevel)
   ```

2. **Resume guard.** Check if already installed:
   ```bash
   if [ -L "$HOME/.claude/skills/<name>/SKILL.md" ]; then
     echo "Skill <name> already installed, skipping."
   fi
   ```
   If the symlink exists, print the step 4 success message and skip to end of stage.

   Note: `skills-deploy install` is idempotent (skips if symlink already points to
   the correct target), but the explicit guard avoids a misleading "installed" message
   on resume.

3. **Run install:**
   ```bash
   "$REPO_ROOT/scripts/skills-deploy" install <name>
   ```

4. If install succeeds, print (user-facing instruction, not executed by Claude):
   ```
   Skill <name> installed to ~/.claude/skills/<name>/
   Tag the release: git tag <name>-v<version> && git push --tags
   ```

5. If install fails (non-zero exit), print a warning but do NOT exit with error:
   ```
   WARNING: Auto-install failed. The skill is committed and shipped.
   Run manually: scripts/skills-deploy install <name>
   ```

   Any failure reason (permissions, missing jq, broken symlink) is covered by the
   non-zero exit path. The skill is already committed, so install is best-effort.
```

Also update:

- **Stage 1 print statement:** Change from
  `"Starting /skill-author for <name>. Stages: scaffold, author, check, ship."`
  to `"Starting /skill-author for <name>. Stages: intake, scaffold, author, check, ship, install."`
  (6 stages, matching the actual pipeline).

- **Stage 5 step 4:** Remove the git tag instruction
  (`Tag the release: git tag <name>-v<version> && git push --tags`).
  This moves to Stage 6 step 3 where it belongs, since tagging happens after
  the skill is both shipped and installed. Stage 5 step 4 becomes just:
  `"Skill <name> shipped via /ship."`

Note on symlink stability: the symlink points to the repo working tree, not a
specific git commit. A branch switch changes what the symlink resolves to. This
is the same behavior as gstack's symlink model and is by design (update = git pull).
