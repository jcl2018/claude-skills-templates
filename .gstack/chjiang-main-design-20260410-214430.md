## Problem Statement

Templates in `templates/` are tied to this repo. When someone installs skills into another repo via `skills-deploy install`, only SKILL.md gets symlinked. The templates stay behind. The skills have a three-level fallback chain (`$REPO_ROOT/templates/` -> `~/.claude/spec/templates/` -> `~/.claude/templates/`) but nothing populates the fallback locations. Anyone using workflow or contracts in a different repo gets template-not-found errors.

## What Makes This Cool

Zero-friction skill portability. Install a skill, its templates just work. No manual copying, no "did you remember to set up templates?" docs. The skill system feels ambient, not repo-invasive. Codex nailed it: the excitement is clean ergonomics.

## Constraints

- Must extend existing `scripts/skills-deploy` (370 lines of bash), not create a new tool
- Must use `skills-catalog.json` `templates` arrays as source of truth (already populated)
- Templates deploy to `~/.claude/templates/` (global, invisible to target repos)
- Static file copy, no variable substitution at deploy time
- Never overwrite existing templates without warning
- Shared ownership: remove only cleans up templates no other installed skill needs
- Must work with existing manifest at `~/.claude/.skills-templates.json`

## Premises

1. Templates deploy to `~/.claude/templates/` during `skills-deploy install`
2. The catalog's per-skill `templates` array is the source of truth for which templates each skill needs
3. Template files are static (straight file copy, no templating)
4. Existing templates at the destination are not overwritten without warning
5. `skills-deploy remove` cleans up templates only if no other installed skill still needs them

## Cross-Model Perspective

Codex's cold read identified the key user insight: "global, invisible to target repos" reveals the builder cares about ambient ergonomics, not file management. Codex proposed a versioned asset store (`~/.claude/assets/<skill>/<version>/templates/`) as the "coolest version" but acknowledged the boring core (copy + manifest + checksum) is the right weekend build. Both models converged on: extend `skills-deploy` with template copy, SHA256 checksums in manifest, conflict detection, shared-ownership cleanup, and doctor/relink support.

## Approaches Considered

### Approach A: Minimal — Copy + manifest tracking
Extend `do_install` to copy per-skill templates to `~/.claude/templates/`, track ownership in the manifest, clean up on remove. ~50 lines. Solves the core problem but no drift detection. Completeness: 6/10.

### Approach B: Full — Copy + checksum + doctor + relink
Everything in A, plus SHA256 checksums in manifest for each template. Doctor reports template health (missing, drifted, orphaned). Relink repairs templates too. ~100 lines. Completeness: 9/10.

### Approach C: Symlink templates
Symlink templates like SKILL.md instead of copying. Auto-updates but fragile: breaks if source repo moves, user can't customize templates without breaking the link. Completeness: 5/10.

## Recommended Approach

**Approach B: Full — Copy + checksum + doctor + relink**

Chosen because the marginal cost of checksums + doctor + relink is ~50 extra lines over the minimal approach, and it makes the template system actually debuggable. The catalog already declares per-skill templates, the manifest already tracks installed skills. This extends both naturally.

### Implementation plan

**1. New constants and helpers in `scripts/skills-deploy`**

```bash
TEMPLATES_SRC="$REPO_ROOT/templates"
TEMPLATES_TARGET="${HOME}/.claude/templates"
```

Add helper:
```bash
# Read templates array for a skill from catalog
skill_templates() {
  local name="$1"
  [ -f "$CATALOG" ] || return 0
  jq -r --arg n "$name" '.[] | select(.name == $n) | .templates // [] | .[]' "$CATALOG" 2>/dev/null || true
}

# SHA256 of a file (portable: macOS shasum, Linux sha256sum)
file_checksum() {
  (shasum -a 256 "$1" 2>/dev/null || sha256sum "$1" 2>/dev/null) | awk '{print $1}'
}
```

**2. Extend `do_install` — after skill symlinking loop**

For each installed skill, read its `templates` array from the catalog. For each template:
- If source template file doesn't exist in `$TEMPLATES_SRC`: WARN and skip that template (don't abort the install)
- If target exists and checksum matches source: skip the copy, but still append the current skill to `owners` if not already present (ensures shared-ownership tracking is accurate even when the file was already deployed by another skill)
- If target exists and checksum differs: warn and skip (user-modified or from another source). With `--force`, overwrite and update checksum.
- If target doesn't exist: copy and record in manifest

Manifest schema extension for templates:
```json
{
  "skills": { ... },
  "templates": {
    "doc-PRD.md": {
      "owners": ["workflow"],
      "source_checksum": "abc123...",
      "installed_at": "2026-04-10T..."
    }
  }
}
```

The `owners` array tracks which skills claim each template. This enables safe removal.

**3. Extend `do_remove` — after skill removal loop**

For each removed skill, iterate its templates. Remove the skill from each template's `owners` array. If `owners` becomes empty, delete the template file from `~/.claude/templates/`. If other skills still own it, leave it in place.

**4. Extend `do_doctor` — new "Templates" section**

Check each template in the manifest:
- File exists at `~/.claude/templates/{name}`? If not: FAIL
- Current checksum matches `source_checksum`? If not: WARN (drifted)
- All `owners` still in `manifest.skills`? If not: WARN (orphaned ownership)
- For each installed skill, any template in its catalog `templates` array but absent from manifest? WARN (not deployed)

**5. Extend `do_relink` — template repair**

For each template in manifest where the file is missing or checksum has drifted:
- If source exists at `$TEMPLATES_SRC/{name}`: re-copy and update checksum in manifest
- If source is not found: emit WARN and leave the manifest entry intact (same pattern as install)

**6. Update `scripts/test.sh`**

Add template deployment tests:
- Install a skill, verify its templates appear in `~/.claude/templates/`
- Install two skills sharing a template, remove one, verify template persists
- Remove both, verify template is cleaned up
- Doctor reports missing/drifted templates correctly

### Edge cases

- **Template shared by multiple skills**: The `owners` array handles this. Currently no two skills share a template, but the mechanism is forward-compatible. If a future skill declares a template already owned by another skill, it gets appended to `owners` and the file persists until all owners are removed.
- **User customizes a deployed template**: Checksum drift detected by doctor. Install won't overwrite. User keeps their version.
- **Source repo moves**: Templates are copies, not symlinks. They survive source relocation. Only `relink` needs the source path (for re-copying).
- **Source template missing from repo**: Catalog declares `foo.md` but `templates/foo.md` doesn't exist. Install emits WARN and skips that template. Does not abort the rest of the install.
- **First install with no `~/.claude/templates/` dir**: `mkdir -p` before copying.
- **GENERATION-GUIDE.md and TRACKER-TEMPLATE.md**: These are meta-templates not referenced by any skill's `templates` array. They stay in the repo, don't get deployed. Correct behavior.

## Success Criteria

- `skills-deploy install workflow` copies workflow's templates (per catalog `templates` array) to `~/.claude/templates/`
- `skills-deploy install contracts` copies contracts' templates (no duplicates if shared)
- `skills-deploy remove workflow` only removes templates not also owned by contracts
- `skills-deploy doctor` reports template health alongside skill health
- A fresh clone of another repo with workflow installed can resolve templates via the fallback chain
- `scripts/test.sh` passes with template deployment tests

## Distribution Plan

This is an enhancement to an existing CLI (`scripts/skills-deploy`) in a git repo distributed via `git clone`. No new distribution channel needed. The deploy pipeline itself IS the distribution mechanism for templates.
