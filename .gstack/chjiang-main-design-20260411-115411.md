## Problem Statement

After real usage with gstack, the /workflow skill's 4-phase pipeline turns out to be largely redundant. Phase 3 (review) and Phase 4 (ship) explicitly delegate to gstack's /review and /ship skills — they're thin wrappers. Phase 2 (implement) has original logic (build-forward/debug-backward mode detection, hypothesis-driven RCA with 3-strike escalation), but this behavior is already covered by gstack's natural patterns: /investigate handles debug-backward RCA, and build-forward is just "read the docs, write code" which any dev does naturally. The /contracts skill is useful during scaffolding but unnecessary as a standalone enforcement gate. The unique value lives in the templates and the work-items/ directory pattern, not in the orchestration skills.

The 2026 landscape trends toward MORE orchestration (GitHub spec-kit, Kiro, harness engineering). This repo goes the opposite direction: less orchestration, stronger templates. The hypothesis is that CLAUDE.md rules + good templates are sufficient because Claude Code follows structured instructions reliably.

## What Makes This Cool

Deleting code is the hardest design decision. Most projects accumulate orchestration layers. This one is removing them after proving they're redundant through real usage. What remains is a clean template library with CLAUDE.md rules that any gstack user can adopt in 5 minutes. The "skill" becomes invisible — it's just how your repo works.

## Constraints

- Must preserve all unique logic from /workflow track.md (type-aware scaffolding, parent/child hierarchy, git-journal synthesis)
- Must preserve contract validation capability (frontmatter checks, required sections) without a standalone skill
- Templates stay in templates/ — no directory restructuring
- work-items/ pattern stays as-is
- Must work for solo devs using gstack (primary audience)
- artifact-manifests.json remains the source of truth for per-type required artifacts
- Existing work items (F000001_workflow_alpha and children) must still function

## Premises

1. /workflow's review and ship phases are explicit delegation to gstack (they invoke `Skill: review` and `Skill: ship`). implement.md has original logic (build/debug modes, hypothesis RCA, 3-strike escalation) but this is functionally covered by gstack's /investigate (debug-backward) and natural dev workflow (build-forward). Deleting loses the structured implement framework but not capability. Confirmed with caveat.
2. /workflow track (create, journal, milestones, list, close, child-items) IS unique value, but its logic can be expressed as CLAUDE.md rules + templates rather than a skill file. Confirmed.
3. /contracts enforcement is valuable during scaffolding (validate docs against templates) but not as a standalone skill. Its checks fold into CLAUDE.md rules. Confirmed.
4. The work-items/ directory pattern and template library are the real product. The skills were scaffolding for the scaffolding. Confirmed.
5. Solo gstack devs can follow CLAUDE.md rules reliably enough that a skill wrapper adds no safety. Confirmed.

## Approaches Considered

### Approach A: Minimal Delete
Delete skills, add 10-15 lines of generic CLAUDE.md rules. Fastest but loses type-aware logic, journal synthesis, and contract validation. Completeness 5/10.

### Approach B: Smart CLAUDE.md Rules (Chosen)
Delete skills, write comprehensive CLAUDE.md rules encoding type-aware artifacts, parent/child hierarchy, branch conventions, git-journal synthesis, and basic contract validation. Preserves track-phase and contract logic as plain-text rules. Explicitly drops implement-phase structured framework (covered by gstack /investigate and natural dev workflow) and hard enforcement gates (accepted tradeoff for solo devs). Completeness 7/10.

### Approach C: Convention-over-Configuration
Delete skills, make templates self-documenting via artifact-manifests.json. Agent-agnostic but less reliable — depends on AI inferring behavior from manifest. Completeness 6/10.

## Recommended Approach

**Approach B: Smart CLAUDE.md Rules.** The delta between A and B is ~5 minutes of CLAUDE.md writing, but B preserves the valuable unique logic from both skills. Type-aware scaffolding, journal synthesis, and contract validation survive as instructions. What's explicitly dropped: implement.md's structured build/debug framework (gstack's /investigate covers debug-backward, build-forward is natural dev workflow) and hard enforcement gates from review.md/ship.md (CLAUDE.md rules are advisory, not blocking — accepted tradeoff per Premise 5).

### What gets deleted

1. `skills/workflow/` — entire directory (SKILL.md, track.md, implement.md, review.md, ship.md, DESIGN.md, CHANGELOG.md)
2. `skills/contracts/` — entire directory (SKILL.md, DESIGN.md, CHANGELOG.md)
3. References in `skills-catalog.json` — remove workflow and contracts entries
4. CLAUDE.md skill routing entries for /workflow and /contracts

### What stays

1. `templates/` — all template files (tracker-*.md, doc-*.md, contract-*.md, *-GENERATION-GUIDE.md)
2. `work-items/` — existing work items (F000001_workflow_alpha and children)
3. `artifact-manifests.json` — the CLAUDE.md rules hardcode type-to-artifact mappings directly (no manifest lookup needed at runtime). The manifest at `~/.claude/artifact-manifests.json` becomes supplementary reference only. If a repo-root copy exists, it stays as documentation.
4. All scripts (validate.sh, test.sh, etc.) — may need minor updates to skip deleted skill references
5. `skills/docs/`, `skills/skill-author/`, `skills/system-health/` — docs needs reference updates (see step 6b below), others untouched

### What gets added to CLAUDE.md

A new `## Work Item Templates` section replacing the `## Skill routing` entries for workflow and contracts. The rules encode:

**Type-aware scaffolding:**
```
When creating work items, use templates from templates/ directory:
- feature: tracker-feature.md + doc-PRD.md + doc-ARCHITECTURE.md + doc-TEST-SPEC.md + doc-milestones.md
- defect: tracker-defect.md + doc-RCA.md + doc-test-plan.md
- task: tracker-task.md + doc-test-plan.md
- user-story: tracker-user-story.md + doc-PRD.md + doc-ARCHITECTURE.md + doc-TEST-SPEC.md + doc-milestones.md
```

**Branch conventions:**
```
Branch naming determines work item type:
- feature-* or feat-* or feat/* -> feature
- defect-* or fix-* or fix/* or bugfix-* -> defect
- task-* or chore-* or chore/* -> task
- story-* -> user-story
```

**Directory structure:**
```
work-items/{slug}/
  TRACKER.md (required, from tracker-{type}.md template)
  {artifact}.md (per type, from doc-{name}.md templates)
  {child-slug}/ (nested for parent/child: feature > user-story > task)
```

**Placeholder replacement:**
```
When scaffolding, replace in templates:
{ITEM_NAME}, {ITEM_ID}, {PARENT_ID}, {FEATURE_ID}, {YYYY-MM-DD}, {BRANCH_NAME}, {author}

ID generation: {TYPE_PREFIX}{NNNNNN} where prefix is F (feature), S (user-story),
T (task), D (defect). Number increments from the highest existing ID of that type
in work-items/. Example: if F000001 exists, next feature is F000002.
```

**Git-journal synthesis:**
```
When updating a work item, check git log for commits since merge-base with main/master:
  git merge-base main HEAD -> BASE, then git log --oneline $BASE..HEAD
Group commits: fix/debug/investigate -> finding, decide/choose/switch -> decision, other -> implementation.
Propose journal entries with commit SHAs. Ask before adding.
```

**Contract validation (lightweight):**
```
When scaffolding or reviewing work items, check:
- Each doc has YAML frontmatter with required fields per template
- Required sections (## headers) from template exist in instance
- For features/user-stories: PRD user stories have corresponding TEST-SPEC entries
Warn on missing sections. Never auto-fix without asking.
```

### Implementation steps

1. Create feature branch: `git checkout -b feat/slim-to-templates`
2. Create work items for this change (dogfooding the current system one last time):
   - Feature: `work-items/F000002_slim_to_templates/`
   - User story: `work-items/F000002_slim_to_templates/S000001_delete_skills/`
   - Task: `work-items/F000002_slim_to_templates/S000001_delete_skills/T000001_update_claude_md/`
3. Delete `skills/workflow/` directory
4. Delete `skills/contracts/` directory
5. Update `skills-catalog.json` — remove workflow and contracts entries
6. Update `CLAUDE.md`:
   - Remove /workflow and /contracts from skill routing
   - Add new `## Work Item Templates` section with smart rules
6b. Update `skills/docs/` references to /contracts:
   - `SKILL.md` line 53: remove parenthetical "(that's `/contracts`)"
   - `check.md` line 75: remove file path reference to `skills/contracts/SKILL.md`
   - `check.md` line 80: remove "match the `/contracts check` output style"
   - `DESIGN.md` lines 11, 29: rephrase /contracts references to note it was removed and validation moved to CLAUDE.md rules
7. Update scripts that reference deleted skills (audit each for workflow/contracts references):
   - `validate.sh` — remove skill-specific validation for workflow/contracts entries
   - `test.sh` — remove workflow/contracts test cases
   - `generate-readme.sh` — verify no workflow/contracts references (likely clean already)
   - `test-deploy.sh` — has 20+ references to workflow/contracts as test fixtures (install workflow, dependency resolution workflow->contracts, multi-owner scenarios). All test cases using these skills must be rewritten to use a surviving skill (e.g., `docs` or `system-health`)
   - `skills-deploy` — remove workflow/contracts from any hardcoded references; `skills-deploy remove workflow` and `skills-deploy remove contracts` should handle manifest cleanup for users who had them installed
   - Full audit: `grep -rl "workflow\|contracts" scripts/` to catch any others
8. Update `README.md` — reflect new repo identity (template library, not skill pipeline)
9. Run `./scripts/validate.sh` and `./scripts/test.sh`
10. Ship via gstack `/ship`

## Distribution Plan

Changes are to the repo structure and CLAUDE.md. Distribution via git pull. Users who had /workflow or /contracts installed via skills-deploy should run `skills-deploy remove workflow` and `skills-deploy remove contracts` to clean up symlinks and manifest entries. Note: `contract-*.md` templates remain in the repo's templates/ directory (they're documentation, not enforcement code). If they were deployed to `~/.claude/templates/` via skills-deploy, they become orphaned from the contracts skill but are still useful as reference. `skills-deploy doctor` will report them — users can keep or remove at their discretion. Users who customized `~/.claude/artifact-manifests.json` with additional artifact types should manually add corresponding entries to their CLAUDE.md rules, as the manifest is no longer read at runtime. Update README with migration instructions.

**Rollback:** If CLAUDE.md rules prove insufficient, `git revert` the merge commit to restore both skills. No data migration needed.
