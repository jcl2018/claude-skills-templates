## Problem Statement

Two readability problems in the workflow system:

1. `/docs check` emits `work-item-graph.json` (machine-readable) but no human-friendly output. The JSON has everything — nodes, badges, completeness — but nobody reads raw JSON. Users run the check and see terminal output, then have to mentally reconstruct the health picture.

2. Tracker lifecycle phases 1-3 are generic checklists ("Doc review completed", "Core changes committed") while Phase 4 names exact commands (`/ship`, `/land-and-deploy`). Someone looking at Phase 1 has no idea what to actually do. Phase 4 is the only phase that works as a runbook.

## What Makes This Cool

The report turns `/docs check` from "run it and read the terminal scroll" into "open a single markdown file and see everything." The runbook phases turn every tracker into a self-contained SOP. A new contributor opens a TRACKER.md and knows exactly what command to run at every stage. No tribal knowledge needed.

## Constraints

- Report must extend existing check.md pipeline (new Step 19, not a separate tool)
- Report is regenerated every `/docs check` run — never stale
- Runbook format: numbered procedural steps + checkboxes for completion gates
- Different types get different runbooks (4 types × 4 phases)
- All existing trackers get updated (not just templates for new items)
- Preserve existing checkbox states in migrated trackers
- No new abstractions (no compiler, no inheritance, no structured schemas)

## Premises

1. The human-readable report is `.docs/work-item-report.md`, written alongside `work-item-graph.json`. Regenerated each run.
2. Runbook updates BOTH tracker templates AND all existing TRACKER.md files. Existing checkbox states preserved.
3. Different work item types get different runbooks (user-story has `/office-hours` + doc triplet, task is simpler).
4. Numbered steps for procedures, checkboxes for completion gates. Separates "what to do" from "what to verify."

## Cross-Model Perspective

Codex proposed "phase-as-code": define phases as structured JSON schemas (id, command, expected_output, failure_modes, recovery, completion_gate), then compile into tracker sections, reports, and graph artifacts. The tracker becomes a resumable operator console.

Codex's key insight: "What excites them is not better documentation aesthetics. It is procedural exactness: turning vague phase labels into deterministic, repeatable operations."

Codex also flagged Runme (docs.runme.dev) as prior art — executable markdown runbooks. Different paradigm (VS Code notebooks vs Claude Code slash commands) but worth knowing.

We chose prose runbooks over the compiler because 4 templates × 4 phases = 16 sections is too small a surface to justify a compilation layer. The compiler makes sense at 15+ templates.

## Approaches Considered

### Approach A: Prose Runbooks (Chosen)
Hand-author runbook lifecycle sections in each template. 4 templates updated with numbered steps, commands, expected outputs, error handling. Migrate existing trackers by matching checkbox states. Report is Step 19 in check.md.
- Effort: M (CC: ~20min)
- Risk: Low
- Completeness: 8/10
- Pros: Simple, readable, no new abstractions, easy to iterate
- Cons: ~40 lines of duplication across types sharing steps

### Approach B: Shared Phases + Per-Type Overrides
Shared phase definitions in a reference doc. Templates inherit + override.
- Effort: L (CC: ~30min)
- Risk: Medium
- Completeness: 9/10
- Rejected: inheritance model adds complexity to validation for minimal DRY benefit at this scale

### Approach C: Phase-as-Code Compiler (Codex's Vision)
Structured JSON schemas compiled into markdown, reports, artifacts.
- Effort: XL (CC: ~2hrs)
- Risk: High
- Completeness: 10/10
- Rejected: more infrastructure than needed for 4 templates. Revisit at 15+ templates.

## Recommended Approach

Approach A: Prose Runbooks. Two deliverables:

### Deliverable 1: Human-Readable Report (.docs/work-item-report.md)

Add a new Step 19 to `skills/docs/check.md` AFTER Step 18 (structural completeness output). The report consumes all data collected by Steps 1-18, including structural findings from Step 15 and the tree from Step 16b. It must run last. Write `.docs/work-item-report.md` with this structure:

```markdown
# Work Item Health Report

Generated: {ISO-8601 timestamp}
Commit: {short SHA}
Repo: {repo name}

## Tree

{exact same tree visualization from Step 16b, in a fenced code block}
(The tree shows hierarchy with inline badges; the Badge Summary table below provides a flat, sortable view for quick scanning.)

## Badge Summary

| Item | Type | State | Template | Lifecycle | Traceability | Structure |
|------|------|-------|----------|-----------|--------------|-----------|
| F000001_workflow_alpha | feature | In Progress | PASS | PASS | PASS | PASS |
| ...rows for all items... |

## Findings

### Critical
- [INCOMPLETE] F000002_system_health_v1 — feature has 0 user-story children (minimum: 1)
- [LIFECYCLE_INCONSISTENT] F000002_system_health_v1 — "broken down" is checked but has 0 user-story children

### Warnings
- [INCOMPLETE] S000002_template_consolidation — user-story has 0 task children (minimum: 1)
- ... other WARN/DRIFT/EXTRA findings ...

### Advisory
- [INFO] ... P1/P2 traceability notes ...

(If no findings in a severity category, omit that subsection.)

## Structural Summary

- **Items:** {N} total ({N} features, {N} user-stories, {N} tasks, {N} defects)
- **Incomplete:** {N}
- **Misplaced:** {N}
- **Lifecycle issues:** {N}
- **Stray directories:** {N}
```

**Severity grouping:**

| Severity | Statuses | When |
|----------|----------|------|
| Critical | INCOMPLETE (root items), LIFECYCLE_INCONSISTENT, MISPLACED | Structural violations that block shipping |
| Warnings | INCOMPLETE (non-root), DRIFT, UNTESTED | Template/coverage issues needing attention |
| Advisory | INFO, EXTRA, WARN | Informational, no action required |

"Root items" = work items whose placement rule is `root` (features and defects by default, per artifact-manifests.json placement rules).

**Report path:** `.docs/work-item-report.md` (alongside `work-item-graph.json` and `claims.json`).

**Also update `/docs tree`:** `skills/docs/tree.md` should write a lightweight version to `.docs/work-item-tree.md` (structural badges only, no badge summary table, no findings grouping).

### Deliverable 2: Runbook Lifecycle Phases

Update all 4 tracker templates with runbook-style lifecycle phases. Each phase has:
1. **Numbered procedural steps** — what to do, in order, with exact commands
2. **Checkboxes as completion gates** — what to verify before moving to next phase

#### tracker-user-story.md Lifecycle

```markdown
## Lifecycle

### Phase 1: Track

1. Read parent tracker to understand scope
2. Run `/office-hours` with your idea
   → produces design doc in `~/.gstack/projects/`
3. Create working branch: `git checkout -b feat/{slug}`
4. Scaffold work item directory and TRACKER.md
5. Extract from design doc into doc triplet: requirements → `PRD.md`, architecture decisions → `ARCHITECTURE.md`, test scenarios → `TEST-SPEC.md`
   (use templates from `templates/doc-PRD.md`, `doc-ARCHITECTURE.md`, `doc-TEST-SPEC.md`)
6. Create milestones from PRD acceptance criteria
7. Break into child tasks if scope warrants decomposition

**Gates:**
- [ ] Acceptance criteria defined
- [ ] Working branch created (`branch` field populated)
- [ ] Doc triplet produced (PRD + ARCHITECTURE + TEST-SPEC)
- [ ] Milestones created
- [ ] Tasks broken down (if needed)

### Phase 2: Implement

1. Work from doc triplet + acceptance criteria (build-forward mode)
2. Commit changes incrementally with descriptive messages
3. Update Todos section — remove completed items, add new discoveries
4. Update Files section with all changed file paths

**Gates:**
- [ ] Implementation committed (>=1 commit SHA in Log)
- [ ] Acceptance criteria verified met
- [ ] Todos section reflects remaining work (no stale items)
- [ ] Files section updated with changed files

### Phase 3: Review

1. Run `/docs check` — verify all validation passes
   → should show PASS for template, lifecycle, traceability, structure badges
2. Run `/docs tree` — verify hierarchy and structural completeness
3. Run tests: `./scripts/test.sh`
4. Review TEST-SPEC alignment: do test cases cover all P0 acceptance criteria?
5. Run `/review` for code review (if PR exists)

❌ If `/docs check` finds issues: fix findings, re-run until clean

**Gates:**
- [ ] `/docs check` — validation passed
- [ ] `/docs tree` — structure verified
- [ ] Test verification passed
- [ ] Doc triplet alignment verified (TEST-SPEC covers P0 stories)

### Phase 4: Ship

1. Run `/ship` — creates PR, bumps version, updates changelog
2. Run `/land-and-deploy` — merges PR and verifies deployment

❌ If CI fails: fix, push, re-run `/ship`

**Gates:**
- [ ] `/ship` — PR created
- [ ] `/land-and-deploy` — merged and deployed
```

### Migration Strategy

**Key design shift for features:** Feature Phase 2 changes from hands-on implementation ("Core implementation committed") to child coordination ("All child stories/tasks have entered Phase 2+"). Feature-level implementation happens via children, not directly. This is intentional.

**Key design shift for defects:** "Root cause identified" moves from Phase 2 to Phase 1. The defect workflow now requires root cause analysis BEFORE implementation, not during.

For each existing TRACKER.md in `work-items/`:

1. Read the current lifecycle section
2. Note which checkboxes are checked (`- [x]`) and which are unchecked (`- [ ]`)
3. Replace the lifecycle section with the new runbook format for that type
4. Map old checkbox labels to new gate labels per type:

**Task mappings:**
- "Parent scope read" → "Parent scope read (parent tracker reviewed)" (same)
- "Working branch created" → "Working branch created (`branch` field populated)" (same)
- "Files section populated" → "Files section populated" (same)
- "Core changes committed" → "Core changes committed (>=1 commit SHA in Log)" (same)
- "Todos section reflects remaining work" → "Todos section reflects remaining work (no stale items)" (same)
- "Files section updated with changed files" → "Files section updated with changed files" (same)
- "Doc review completed" → "`/docs check` — validation passed" (upgrade)
- "Doc generation finalized" → removed as gate (covered by `/docs check`)
- "Test verification passed" → "Test verification passed" (same)

**User-story mappings:**
- "Acceptance criteria defined" → "Acceptance criteria defined" (same)
- "Working branch created" → "Working branch created (`branch` field populated)" (same)
- "Doc triplet produced" → "Doc triplet produced (PRD + ARCHITECTURE + TEST-SPEC)" (same)
- "Milestones created" → "Milestones created" (same)
- "Tasks broken down (if needed)" → "Tasks broken down (if needed)" (same)
- "Build-forward mode (from doc triplet + acceptance criteria)" → removed as gate (covered by procedural step)
- "Implementation committed (>=1 commit SHA in Log)" → "Implementation committed (>=1 commit SHA in Log)" (same)
- "Acceptance criteria verified met" → "Acceptance criteria verified met" (same)
- "Todos section reflects remaining work" → "Todos section reflects remaining work (no stale items)" (new gate, leave unchecked)
- "Files section updated with changed files" → "Files section updated with changed files" (new gate, leave unchecked)
- "Doc review completed" → "`/docs check` — validation passed" (upgrade)
- "Doc generation finalized" → removed as gate (covered by `/docs check`)
- "Doc triplet alignment check (TEST-SPEC)" → "Doc triplet alignment verified (TEST-SPEC covers P0 stories)" (same intent)

**Feature mappings:**
- "Acceptance criteria scoped" → "Acceptance criteria scoped" (same)
- "Working branch created" → "Working branch created (`branch` field populated)" (same)
- "Doc triplet produced" → "Doc triplet produced (PRD + ARCHITECTURE + TEST-SPEC)" (same)
- "Broken down into child tasks/stories" → "Broken down into child tasks/stories" (same)
- "Doc triplet read (build-forward mode)" → removed (feature coordinates via children, not hands-on)
- "Core implementation committed" → removed (implementation happens via children)
- "Child tasks completed or deferred" → "All child stories/tasks have entered Phase 2+" (broadened)
- "Files section updated" → removed as gate (feature-level files tracked informally; children track their own)
- "Doc review completed" → "`/docs check` — all children pass validation" (upgrade)
- "Doc generation finalized" → removed as gate (covered by `/docs check`)

**Defect mappings:**
- "Reproduction steps documented" → "Reproduction steps documented" (same)
- "Working branch created" → "Working branch created (`branch` field populated)" (same)
- "Initial symptom logged in Log" → removed as gate (covered by procedural step 4)
- "Root cause identified (RCA in Insights section)" → moves from Phase 2 to Phase 1: "Root cause identified (or best hypothesis logged)"
- "Hypothesis tested with evidence" → removed as gate (covered by `/investigate` procedural step and Insights section)
- "Fix committed (>=1 commit SHA in Log)" → "Fix committed with regression test" (strengthened)
- "Regression test added" → folded into "Fix committed with regression test" (merged)
- "Files section updated" → removed (defect fixes are typically small, Files section is informational)
- "Doc review completed" → "`/docs check` — validation passed" (upgrade)
- "Doc generation finalized" → removed as gate (covered by `/docs check`)

5. Preserve the checked/unchecked state for gates that map 1:1
6. When a gate moves between phases during migration, preserve its checked state in the new location regardless of which phase it was in before
7. For new gates that didn't exist before: leave unchecked
8. For removed gates: drop (the procedural steps cover the intent)
9. **Edge case — Closed items with unverifiable gates:** When a feature is Closed and has 0 children (e.g., F000002), preserve existing checked states for all mapped gates even if the new gate semantics are unverifiable. History trumps structural truth for closed items.
