## Problem Statement

The skill workbench has 9 skills, but no way for other Claude Code users to install them. Skills are currently deployed as flat file copies to `~/.claude/skills/`, which means updates require re-copying and there's no way to track what came from where. The `skill-author` skill isn't deployed at all. The repo needs a distribution pipeline that lets power users install, update, and remove skills with one command.

## What Makes This Cool

Zero-friction adoption for power users. Clone the repo, run one command, skills appear alongside gstack. `git pull` updates everything. A manifest tracks ownership so you never accidentally clobber gstack skills. And it's the foundation for a skill registry later.

## Constraints

- Must coexist with gstack skills in `~/.claude/skills/` without collisions
- macOS (darwin) primary. Linux nice-to-have. No Windows (symlink pain).
- No build step, no bun, no node. Pure POSIX-compatible bash.
- Must detect and refuse to overwrite skills owned by other tools (gstack, manual installs)
- Repo clones to a fixed location: `~/.claude/skills-templates/`
- Read the skill list from the repo's `skills/` directory (auto-discovery, not a hardcoded list)

## Premises

1. Follow gstack's exact pattern: real directory at `~/.claude/skills/{name}/`, with `SKILL.md` symlinked back to the cloned repo source.
2. Clone to a fixed, known location (`~/.claude/skills-templates/`) so symlinks resolve predictably.
3. Collision detection is required. Some skill names (system-health, work, etc.) could overlap with gstack or other tools.
4. Scope is a thin wrapper with a manifest: single CLI script + bootstrap script, ~150 lines total.
5. Manifest file at `~/.claude/.skills-templates.json` tracks installed skills, source repo, and commit hash.

## Approaches Considered

### Approach A: 4 Standalone Scripts (TODO.md as-written)

Exactly the scripts described in TODO.md: `setup.sh` (clone + deploy), `deploy-local.sh` (symlink from existing clone), `relink.sh` (repair), `uninstall.sh` (remove). Each script is self-contained. ~100 lines total.

**Pros:** Matches TODO spec, each script readable in isolation, easy to test individually.
**Cons:** Duplicated manifest/symlink logic across scripts, 4 entry points to document.

### Approach B: Single CLI with Subcommands

One script (`scripts/skills-deploy`) with subcommands: `install`, `remove`, `relink`, `doctor`. Shared functions eliminate duplication. A thin `setup.sh` bootstrap clones the repo and calls `skills-deploy install`. ~150 lines total.

**Pros:** Single entry point, no duplication, natural path to the registry CLI vision, `doctor` subcommand comes free with the manifest.
**Cons:** Slightly more complex structure, ~50 lines more than Approach A.

### Approach C: Makefile

Makefile with targets: `install`, `remove`, `relink`, `doctor`. Ultra-minimal (~60 lines). Declarative.

**Pros:** Tersest option, self-documenting targets.
**Cons:** Unconventional for skill repos, harder to do conditional logic, can't curl-pipe for first install.

## Recommended Approach

**Approach B: Single CLI with Subcommands.** The single entry point eliminates duplication, matches gstack's CLI patterns (gstack-config, gstack-relink are all single-script + subcommands), and is the natural stepping stone toward the registry CLI. Power users already expect this pattern.

### Architecture

```
scripts/
  setup.sh           # Bootstrap: clone repo + call skills-deploy install
  skills-deploy      # Main CLI: install | remove | relink | doctor
```

### `setup.sh` (bootstrap, ~30 lines)

Entry point for new machines. Can be curl-piped.

1. Check if `~/.claude/skills-templates/` already exists
   - If yes: `cd` into it, `git pull`, run `skills-deploy install`
   - If no: `git clone https://github.com/jcl2018/claude-skills-templates.git ~/.claude/skills-templates/`
2. Run `~/.claude/skills-templates/scripts/skills-deploy install`
3. Print summary: "Installed N skills. Run `skills-deploy doctor` to verify."

### `skills-deploy install [skill...]` (~40 lines)

1. Auto-discover skills from `skills/*/SKILL.md` in the repo
2. If specific skills named, filter to those. Otherwise install all.
3. For each skill:
   a. Check if `~/.claude/skills/{name}/` exists
   b. If exists and NOT in manifest as ours: **skip** with warning ("collision: {name} is owned by {owner}")
   c. If exists and IS ours: update symlink (idempotent)
   d. If doesn't exist: `mkdir -p ~/.claude/skills/{name}/` + `ln -snf` SKILL.md
   Note: Only `SKILL.md` is symlinked (not DESIGN.md, CHANGELOG.md, etc.). Claude Code only reads `SKILL.md` from skill directories. Supporting files remain in the source repo for development use.
4. Write manifest to `~/.claude/.skills-templates.json`:
   Note: `source` stores the resolved absolute path (tilde-expanded at install time). Skill `path` values are relative to `source` and resolved as `${source}/${path}` at runtime.
   ```json
   {
     "source": "/Users/you/.claude/skills-templates",
     "repo": "jcl2018/claude-skills-templates",
     "commit": "<current HEAD>",
     "installed_at": "<ISO timestamp>",
     "skills": {
       "skill-author": { "path": "skills/skill-author/SKILL.md" },
       "work-track": { "path": "skills/work-track/SKILL.md" }
     }
   }
   ```

### `skills-deploy remove [skill...]` (~20 lines)

1. Read manifest
2. For each skill (or all if none specified):
   a. Verify the symlink points back to our repo (safety check)
   b. Remove the symlink and the directory
   c. Remove from manifest
3. If no skills remain, remove the manifest file

### `skills-deploy relink` (~15 lines)

1. Read manifest
2. For each skill in manifest:
   a. Check if symlink exists and points to correct target
   b. If broken or missing: recreate
   c. If directory exists but no symlink: recreate symlink
3. Report: "Relinked N skills, M were already correct"

### `skills-deploy doctor` (~25 lines)

1. Read manifest (or report "no manifest found")
2. Check each installed skill:
   a. Symlink exists? Points to correct target? Target file exists?
   b. Skill directory in repo still exists? (catches deleted skills)
3. Check for orphans: skills in `~/.claude/skills/` that point into our repo but aren't in manifest
4. Check for collisions: skills in our repo that exist in `~/.claude/skills/` but are owned by something else
5. Report health summary

### Manifest Format

`~/.claude/.skills-templates.json`:
```json
{
  "source": "/Users/chjiang/.claude/skills-templates",
  "repo": "jcl2018/claude-skills-templates",
  "commit": "055ed8e",
  "installed_at": "2026-04-10T20:10:00Z",
  "skills": {
    "skill-author": {
      "path": "skills/skill-author/SKILL.md",
      "installed_at": "2026-04-10T20:10:00Z"
    },
    "work-track": {
      "path": "skills/work-track/SKILL.md",
      "installed_at": "2026-04-10T20:10:00Z"
    }
  }
}
```

### Collision Detection Logic

A skill at `~/.claude/skills/{name}/` is "owned by us" if:
1. Our manifest lists it, AND
2. The `SKILL.md` in that directory is a symlink pointing into `~/.claude/skills-templates/`

A skill is "owned by gstack" if:
1. The `SKILL.md` is a symlink pointing into `~/.claude/skills/gstack/`

A skill is "manually installed" if:
1. The `SKILL.md` is a regular file (not a symlink)

The installer never overwrites skills it doesn't own. It warns and skips.

## Open Questions

1. **Should `setup.sh` support a `curl | bash` one-liner?** e.g., `curl -fsSL https://raw.githubusercontent.com/jcl2018/claude-skills-templates/main/scripts/setup.sh | bash`. The pattern works but some users distrust piping to bash.
2. **Should existing copied skills be migrated?** The 8 skills currently deployed as flat copies (not symlinks) should probably be migrated to symlinks. `skills-deploy install` could detect this case and offer to convert.
3. **Should the manifest name be generic?** If other repos adopt this pattern, `.skills-templates.json` is too specific. Maybe `~/.claude/.skill-sources.json` with per-repo entries? Defer to registry design.

## Success Criteria

- `scripts/setup.sh` clones and deploys all 9 skills (8 existing + skill-author) in one command
- `skills-deploy install skill-author` deploys a single skill
- `skills-deploy remove skill-author` cleanly removes it
- `skills-deploy doctor` reports healthy with no orphans or collisions
- Existing gstack skills untouched after install
- `git pull && skills-deploy relink` updates all skills
- Manifest is accurate after every operation

## Distribution Plan

This is a set of bash scripts committed to the repo. Distribution is the repo itself.

- Users clone the repo to `~/.claude/skills-templates/`
- `setup.sh` can be curl-piped for first install
- Updates via `git pull` (symlinks auto-resolve)
- No CI/CD needed for the deploy scripts themselves
- The repo's existing GitHub Actions (validate, test) cover the skills
