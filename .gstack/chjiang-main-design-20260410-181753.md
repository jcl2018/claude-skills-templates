## Problem Statement

The skill workbench has 5 lifecycle scripts that work individually but require manual orchestration: skill-design.sh, create-skill.sh, skill-check.sh, skill-version.sh, skill-ship.sh (plus lint-skill.sh as advisory). Creating a new skill means remembering the order, running each script by hand, filling in scaffolded files, running validation, fixing errors, bumping versions, and tagging. There's no guided workflow, no auto-fill from prior design work, and no way to pause mid-pipeline and resume later. The gap between "I have a design doc" and "I have a shipped skill" is 6 manual steps across 3+ files.

## What Makes This Cool

One conversation takes you from a design doc to a shipped, versioned, tagged skill. The harness reads your /office-hours design doc, pre-fills DESIGN.md and SKILL.md with real content (not placeholders), walks you through the creative parts with targeted questions, runs every quality gate automatically, and produces a verified release. The demo: `/skill-author` and 10 minutes later you have a new skill in the catalog with a git tag. The harness is itself a skill built using its own pipeline.

## Constraints

- Must orchestrate existing scripts, not replace them. Scripts remain independently usable.
- No new runtime dependencies (shell scripts + jq, same as the rest of the workbench).
- The harness is a SKILL.md (Claude Code skill), not a shell script. It needs AI access for auto-fill and guided authoring.
- Checkpoint state must survive session boundaries (file-based, not in-memory).
- Must work without a design doc (manual input fallback), but the design doc path is the primary UX.
- The harness skill itself is created using its own pipeline (dogfood).

## Premises

1. The harness is a skill, not a script. Lives at `skills/skill-author/SKILL.md`, invoked via `/skill-author`. Runs inside Claude Code with AI access for auto-fill and guided authoring.
2. The harness orchestrates existing scripts, not replaces them. Calls skill-design.sh, create-skill.sh, skill-check.sh, skill-version.sh, skill-ship.sh under the hood (plus lint-skill.sh as advisory). Scripts remain independently usable.
3. The /office-hours design doc is the primary input. Reads the most recent design doc from `~/.gstack/projects/{slug}/` and pre-fills DESIGN.md and SKILL.md from its conclusions. Without a design doc, the harness still works but requires more manual input.
4. Resume state is file-based, not in-memory. A checkpoint file tracks which pipeline stage was completed, so you can stop mid-session and pick up later.
5. The harness is the first skill created using its own pipeline (dogfood).

## Cross-Model Perspective

Codex's cold read produced four key insights:

1. **Coolest version:** A "self-testing skill foundry" that installs the fresh skill in a scratch workspace, invokes it on benchmark prompts, and scores trigger accuracy. Evidence the skill works, not just that it passes linting.
2. **Revealing answer:** "Design doc -> shipped skill in one session" reveals the core obsession is compression of the idea-to-artifact loop, not scaffolding or release automation.
3. **50% tool:** Superpowers framework proves that a skill can orchestrate a staged workflow inside the agent. Missing 50%: design doc intake, human-in-the-loop authoring gates, checkpoint/resume, shell script orchestration, release semantics, dogfood verification.
4. **Weekend build:** 6 stages, one checkpoint file, parse design doc into manifest, feed to create-skill.sh, generate draft SKILL.md with TODO blocks, ask minimum questions to resolve TODOs, run skill-check in a fix loop, version/ship, invoke the new skill on one canned task. "Build the resumable state machine and the dogfood loop. That's the product."

The foundry concept (scratch workspace + benchmark scoring) is deferred. The verify stage (invoke on a canned prompt) is adopted as a lighter-weight version that proves the concept without uncharted complexity.

## Approaches Considered

### Approach A: Minimal Pipeline
6 stages: intake, scaffold, author, check, version, ship. One SKILL.md file. Reads design doc when available, runs scripts in sequence, asks questions at the author stage. No checkpoint, no resume. Single-session only.
- Effort: S | Risk: Low
- Pros: Ships today, proves the loop, simple SKILL.md
- Cons: No resume across sessions, no auto-fill intelligence, no dogfood verification
- Completeness: 5/10

### Approach B: Interactive Pipeline + Resume
7 stages: intake, design, scaffold, author, check, version+ship, verify. Checkpoint file tracks stage progress for resume. Design doc auto-fill extracts purpose, behavior, dependencies, security boundaries into DESIGN.md and SKILL.md drafts. Guided authoring with TODO blocks and minimum questions. Verify stage invokes the new skill on a canned prompt.
- Effort: M | Risk: Medium
- Pros: Full loop with proof, resume works, auto-fill cuts authoring time in half, verify catches broken skills before they ship
- Cons: More complex SKILL.md, checkpoint schema needs design, verify stage is novel
- Completeness: 8/10

### Approach C: Foundry (Codex-inspired)
Everything in B, plus: verify stage installs the skill in a scratch workspace, runs 2-3 benchmark prompts, scores trigger accuracy and instruction adherence. Produces a "skill report card" artifact.
- Effort: L | Risk: High
- Pros: Most impressive demo, shipped skills have proof they work, pushes toward behavioral eval harness
- Cons: Scratch workspace invocation is uncharted in Claude Code, benchmark prompt design is its own problem
- Completeness: 10/10

## Recommended Approach

**Approach B: Interactive Pipeline + Resume.** Delivers the full interactive pipeline with checkpoint/resume, design doc auto-fill, guided authoring, and a verify stage (Codex's best idea), without the uncharted scratch-workspace complexity of C. The foundry can be added later once verify proves the concept.

### 7 Pipeline Stages

```
Stage 1: INTAKE
  What: Detect skill name, find design doc, read codebase context
  Script: None (AI reads files)
  Gate: Skill name must be valid (kebab-case, no tag collisions)
  Input: User invokes /skill-author [name] or /skill-author (prompted)
  Output: Skill name + design doc contents loaded into context
  Checkpoint: { stage: "intake", skill_name, design_doc_path }

Stage 2: DESIGN
  What: Run skill-design.sh, then auto-fill DESIGN.md from design doc
  Script: ./scripts/skill-design.sh {name}
  Gate: DESIGN.md must exist after this stage
  Auto-fill: Extract Purpose from design doc "Problem Statement" or "What Makes This Cool"
             Extract Behavior from design doc "Recommended Approach" or "Next Steps"
             Extract Dependencies from design doc "Constraints" or "Dependencies"
             Extract Security Boundaries from design doc tool references
             Extract Test Criteria from design doc "Success Criteria"
  Human checkpoint: Show pre-filled DESIGN.md, ask user to review/edit
  Checkpoint: { stage: "design", design_filled: true }

Stage 3: SCAFFOLD
  What: Run create-skill.sh, which creates SKILL.md + CHANGELOG.md + catalog entry
  Script: ./scripts/create-skill.sh {name}
  Gate: SKILL.md, CHANGELOG.md must exist; catalog entry must be valid
  Checkpoint: { stage: "scaffold" }

Stage 4: AUTHOR
  What: Guide the user through writing SKILL.md content
  Script: None (AI writes the skill)
  Auto-fill: Generate draft SKILL.md body from design doc conclusions:
    - Skill description from design doc Purpose
    - Behavior instructions from design doc Behavior
    - Tool list from design doc Security Boundaries
    - Error handling from design doc constraints
  Guided authoring: For each section with TODO blocks, ask one targeted question:
    - "What triggers this skill?" (maps to routing rules / user intents)
    - "What does the skill produce?" (maps to output artifacts)
    - "What tools does it need?" (maps to allowed-tools frontmatter)
    - Skip questions where auto-fill already provided good content
  Human checkpoint: Show draft SKILL.md, ask user to review/edit
  Checkpoint: { stage: "author", sections_completed: [...] }

Stage 5: CHECK
  What: Run skill-check.sh in a fix loop
  Script: ./scripts/skill-check.sh {name}
  Gate: All checks must pass
  Fix loop: If checks fail, read the error output, fix the issue (edit files),
            re-run skill-check.sh. Max 3 iterations before escalating to user.
  Fix loop scope: Handles mechanical errors (missing frontmatter fields, malformed YAML,
            missing catalog fields). For subjective errors (lint warnings about content
            quality), presents the warning to the user and asks for guidance rather than
            attempting auto-fix.
  Also run: ./scripts/lint-skill.sh {name} (advisory, doesn't block)
  Checkpoint: { stage: "check", passed: true, lint_warnings: [...] }

Stage 6: VERSION + SHIP
  What: Bump version, fill changelog, commit, tag
  Script: ./scripts/skill-version.sh {name} patch
          (user prompted for bump type if not first version)
          ./scripts/skill-ship.sh {name}
  Gate: skill-check.sh passes (enforced by skill-version.sh)
  Human checkpoint: Show diff summary, confirm ship
  Checkpoint: { stage: "shipped", version, tag, commit_sha }

Stage 7: VERIFY
  What: Invoke the new skill on a canned prompt to prove it works
  Method: Use Claude Code's Skill tool to invoke the newly created skill
          with a simple test prompt derived from the skill's Purpose
  Gate: Skill tool invocation completes without throwing an error and the
        skill produces non-empty output. Content correctness is out of scope
        (deferred to behavioral eval).
  Fallback: If the Skill tool cannot discover the newly created skill in the
            current session, fall back to a syntax check: parse SKILL.md
            frontmatter, verify all required fields exist, confirm valid
            YAML+Markdown. Log a warning that live invocation was skipped.
  Output: Brief report of what the skill did when invoked
  Checkpoint: { stage: "verified", test_prompt, test_result, method: "live"|"syntax" }
```

### Checkpoint Schema

File: `skills/{name}/.skill-author-state.json`

```json
{
  "skill_name": "my-skill",
  "started_at": "2026-04-10T18:00:00Z",
  "updated_at": "2026-04-10T18:15:00Z",
  "design_doc_path": "~/.gstack/projects/slug/user-branch-design-datetime.md",
  "current_stage": "author",
  "completed_stages": ["intake", "design", "scaffold"],
  "stage_data": {
    "intake": { "design_doc_found": true },
    "design": { "auto_filled_sections": ["Purpose", "Behavior", "Dependencies"] },
    "scaffold": { "files_created": ["SKILL.md", "CHANGELOG.md"] },
    "author": { "sections_completed": ["description", "behavior"], "sections_remaining": ["error-handling"] }
  }
}
```

Co-located in the skill directory so it travels with the skill during development. Deleted after successful ship (clean artifact). If the user chooses "start fresh" during resume detection, the stale checkpoint file is deleted. The `doctor.sh` script can optionally warn about checkpoint files older than 7 days.

### Resume Flow

When `/skill-author` is invoked:
1. Check if `skills/*/.skill-author-state.json` exists for any skill
2. If multiple found, list all in-progress checkpoints and let the user pick one, or allow specifying: `/skill-author resume my-skill`
3. If one found, ask: "Found in-progress authoring for {name} at stage {stage}. Resume or start fresh?"
4. If resume: skip to the current stage, re-read all context from checkpoint
5. If start fresh: delete the checkpoint file and begin from intake

### Auto-Fill Intelligence

The harness reads the design doc and maps sections to skill artifacts:

| Design Doc Section | Maps To | Target File |
|---|---|---|
| Problem Statement / What Makes This Cool | Purpose section | DESIGN.md |
| Recommended Approach / Next Steps | Behavior section | DESIGN.md |
| Constraints / Dependencies | Dependencies section | DESIGN.md |
| Tool references in approach | Security Boundaries | DESIGN.md |
| Success Criteria | Test Criteria | DESIGN.md |
| Description from Purpose | description frontmatter | SKILL.md |
| Behavior from approach | Skill body instructions | SKILL.md |
| Tool list from security boundaries | allowed-tools frontmatter | SKILL.md |

This is AI-powered extraction, not string matching. The SKILL.md instructs the agent: "Read the design doc at {path}. For each target field in the mapping table, find the most relevant section in the design doc and generate a concise paragraph. If no matching content is found, insert a TODO marker and flag it for the user at the AUTHOR stage human checkpoint." The user reviews and edits, never writes from scratch.

### Verify Stage Details

The verify stage uses Claude Code's Skill tool to invoke the newly created skill:
1. Construct a test prompt from the skill's Purpose section (e.g., if the skill is "code-review", the test prompt is "review this code")
2. Invoke via the Skill tool
3. Check that the skill loads without errors
4. Report: "Skill invoked successfully. It [brief description of what happened]."

This is a smoke test, not a behavioral eval. It proves the skill is syntactically valid and invocable. Deeper behavioral testing is deferred to the P2 "Behavioral eval harness" TODO.

### Catalog Entry

The harness updates the catalog entry created by create-skill.sh:
- `description`: from design doc (replaces "TODO" placeholder)
- `depends.skills`: from design doc Dependencies section
- `depends.tools`: from design doc tool references
- `portability`: inferred (standalone if no skill deps, pipeline if has deps)
- `templates`: list of templates referenced in the skill

### Files Produced

After a complete run, the harness has created or modified:
- `skills/{name}/DESIGN.md` (filled, not template)
- `skills/{name}/SKILL.md` (written, not scaffold)
- `skills/{name}/CHANGELOG.md` (with real entry)
- `skills/{name}/.skill-author-state.json` (deleted after ship)
- `skills-catalog.json` (entry updated with real metadata)
- `README.md` (regenerated by skill-ship.sh)
- Git commit + tag: `{name}-v0.1.0`

### Error Handling Policy

If any script exits non-zero, the harness:
1. Captures stderr and reports the error to the user
2. Does NOT advance the checkpoint (stage remains at the failed step)
3. The user can fix the issue and re-run `/skill-author` to resume from the failed stage

The checkpoint file ensures no work is lost on script failure. Each stage is idempotent: re-running a failed stage from the checkpoint produces the same result as running it fresh.

### Rollback

If a shipped version needs correction after Stage 6 completes:
- Delete the tag: `git tag -d {name}-v{version}`
- Undo the commit: `git reset --soft HEAD~1`
- The harness does not automate rollback. This is a manual escape hatch for the rare case where a ship was premature.

## Open Questions

1. **Verify stage invocation:** Can the Skill tool invoke a skill that was just created in the same session? Needs testing. Fallback defined (syntax check) if discovery fails.
2. **Design doc selection:** If multiple design docs exist for the project, which one does the harness read? Current design: most recent by timestamp. May need user confirmation if multiple exist.
3. **SKILL.md complexity:** The harness SKILL.md will be the most complex in the repo (it orchestrates all lifecycle scripts plus lint-skill.sh as advisory, AI authoring, and checkpoint state). How do we keep it maintainable? Consider splitting into sections with clear stage boundaries.

## Success Criteria

- Running `/skill-author my-skill` with an existing design doc produces a shipped skill in one conversation session (under 15 minutes of human interaction)
- Running `/skill-author` without a design doc still works (manual input path, longer)
- Resuming a partially-completed authoring session works across conversation boundaries
- All quality gates pass automatically (no manual script running)
- The verify stage catches a broken skill (e.g., missing frontmatter field)
- The harness itself was created using skill-design.sh -> create-skill.sh -> skill-check.sh -> skill-version.sh -> skill-ship.sh
