## Problem Statement

The S000002_template_consolidation user story specifies exact lifecycle gates for 4 tracker types (feature, defect, task, user-story) plus removal of scrum, review type, and enterprise frontmatter fields. The S000002 TRACKER says "Implemented" but the actual templates on disk still have old enterprise gates ("reviewer noted", "Linux branch build", JIRA URLs, `workflow_type`). The work items in `.claude/worktrees/old-skills/work-items/` already use the new format, so there's a mismatch: work items are ahead of their own templates.

The parent feature F000001_workflow_alpha needs closing. Remaining gates: E2E test, /contracts check, ./scripts/test.sh, then review and ship phases.

## What Makes This Cool

This is the "eating your own cooking" moment. F000001 was the first feature tracked using the workflow skill itself. Closing it proves the 4-phase lifecycle works end-to-end. The template fix means every future work item scaffolds with the right gates from day one.

## Constraints

- Phase detection counts checkboxes, not gate text. Gate rewording is safe (confirmed in S000002 ARCHITECTURE).
- The S000002 PRD has 10 P0 user stories specifying exact gates per type. The PRD is the source of truth for gate text. The S000002 ARCHITECTURE has slightly different gate wording for Review phase ("Self-review completed" vs "Doc review completed"); the PRD wins, and the ARCHITECTURE should be updated to match.
- `tracker-review.md` must be removed (review type eliminated). `doc-review-notes.md`, `doc-scrum.md`, `TRACKER-TEMPLATE.md` also removed (all 3 still exist on disk).
- `artifact-manifests.json` at `~/.claude/artifact-manifests.json` must drop the review type entry. No repo-root copy exists.
- SKILL.md must remove `review-*` branch pattern (line 76), scrum routing (line 56), and "review" from template list (line 138).
- track.md must remove scrum subcommand (line 18, lines 125-126), remove "review" from valid types (line 24), and remove review artifact mapping (line 40).
- Structured ID generation (F/S/D/T prefix) is deferred. It's a feature addition, not a template text change. Manual ID assignment (as used in F000001 work items) works for now.

## Premises

1. Templates need updating to match S000002 PRD specifications. The on-disk templates still have old enterprise gates. Confirmed.
2. Existing work items in `.claude/worktrees/old-skills/work-items/` are already correct. They use the new-style gates. No "alignment fix" needed.
3. /contracts should NOT enforce tracker template alignment. Template text alignment is a migration concern (one-time), not an enforcement concern (ongoing). Work items diverge from templates intentionally as they're worked on.
4. Phase detection is safe because it counts checkboxes, not text. Old work items with old gate text still function correctly.
5. The E2E test for F000001 will be completed by executing this design's steps: updating templates, running validate/test scripts, and shipping. The existing work item hierarchy (F000001 > S000001 > T000001, F000001 > S000002) demonstrates the scaffolding works; closing the feature demonstrates the full lifecycle.

## Approaches Considered

### Approach A: Template Fix Only (Chosen)

Update 4 templates to match S000002 PRD, remove dead files, no enforcement layer.

- **Effort:** S (CC: ~5min)
- **Risk:** Low
- **Pros:** Smallest diff, ships today, addresses the actual problem
- **Cons:** No guard against future drift (accepted: drift is a migration concern, not enforcement)

### Approach B: Fix + /contracts Tracker Layer

Extend /contracts to validate tracker structure and child completeness.

- **Effort:** M (CC: ~20min)
- **Risk:** Medium (expands /contracts scope)
- **Pros:** Automated enforcement, catches drift
- **Cons:** Scope creep. Trackers intentionally differ from templates (checked boxes, filled values).

### Approach C: Fix + Lint Script

Template fix plus standalone lint-tracker.sh.

- **Effort:** S-M (CC: ~10min)
- **Risk:** Low
- **Pros:** Clean separation of concerns
- **Cons:** Not integrated into review gate, another script to maintain

## Recommended Approach

**Approach A: Template Fix Only.** The ongoing enforcement need isn't real. Work items scaffold from templates (start correct) and intentionally diverge (checkboxes, values, journal). Template text matching is the wrong check. If structural drift becomes a real problem later, Approach C (lint script) is the right next step.
