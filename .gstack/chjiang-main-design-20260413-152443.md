## Problem Statement

After shipping F000001 (workflow alpha), five inconsistencies accumulated across the rule/template/manifest system:

1. **Rule drift**: Global rules (`~/.claude/rules/work-items.md`) say features get full doc triplet (PRD + ARCHITECTURE + TEST-SPEC + milestones). Repo-local rules and `artifact-manifests.json` say features get tracker only. Doc triplet lives at user-story level. Every new feature in any repo gets scaffolded wrong.

2. **Dead files**: Both `F000001_workflow_alpha` and `F000002_system_health_v1` have PRD, ARCHITECTURE, TEST-SPEC, milestones at feature level, violating current manifest rules. F000001 is closed, F000002 is active (but flagged INCOMPLETE with 0 children).

3. **Phase redundancy**: 4-phase lifecycle (Track, Implement, Review, Ship) has `/review` in Phase 3 and `/ship` in Phase 4, but `/ship` already runs a full pre-landing review internally (checklist, specialists, adversarial, Greptile). Double review. Phase 3 also includes `/docs check` and `/docs tree` which `/ship` does NOT do.

4. **Stale examples**: `check.md` and `tree.md` reference deleted work items (S000002, S000003, T000002, T000003).

5. **Fallback chain inconsistency**: Three different descriptions exist (2-level vs 3-level) across CLAUDE.md, rules, PHILOSOPHY.md, check.md.

## What Makes This Cool

Compression. Five truth sources converge to one (`artifact-manifests.json`). Four lifecycle phases compress to three. The workflow becomes what it should have been from the start: Track what you're building, Implement it, Ship it.

## Constraints

- Must not break `/docs check` validation (existing checks 1-17 still pass)
- F000001 work items are closed history... they don't need to comply with new rules, but feature-level docs should go because they're misleading examples
- Global rules must work in repos that DON'T have this repo's templates (fallback chain matters)
- Changes must be backward-compatible: existing work items with 4-phase trackers should still be parseable

## Premises

1. Global rules should match `artifact-manifests.json` (features get tracker only, doc triplet at user-story level)
2. Feature-level docs (PRD, ARCHITECTURE, TEST-SPEC, milestones) should be deleted from both F000001 (closed) and F000002 (active but INCOMPLETE)
3. Phase 3 (Review) and Phase 4 (Ship) should be combined into a single "Ship" phase
4. Template fallback chain should be standardized to one description across all files
5. Stale examples in check.md and tree.md should be updated to match current hierarchy (1 story, 1 task)

## Cross-Model Perspective

Codex proposed a "protocol compiler" that generates all files from one canonical spec. The core insight: "The real excitement is compression: fewer phases, fewer truth sources, more opinionated behavior." Codex suggested Copier as a 50% solution and recommended collapsing to Track/Implement/Ship.

We adopt the 3-phase model but reject the protocol compiler as over-engineering for current scale (4 templates, 2 rules files). The canonical spec already exists (`artifact-manifests.json`). Align everything to it by hand.

## Approaches Considered

### Approach A: Focused Cleanup
Fix all 5 issues by hand. Sync rules, delete files, keep 4 phases, fix examples, standardize fallback.
Effort: S. Risk: Low.
Pros: Ships fast. No new abstractions.
Cons: Doesn't simplify the lifecycle model.

### Approach B: Cleanup + 3-Phase Model (CHOSEN)
Same as A, but tracker templates move from 4 phases to 3 (Track, Implement, Ship). Ship absorbs doc checks as pre-flight.
Effort: S-M. Risk: Low.
Pros: Fixes drift AND simplifies model. 3 phases matches reality.
Cons: Slightly larger diff. Historical F000001 trackers used 4 phases.

### Approach C: Protocol Compiler
Build a CLI that generates rules/templates/examples from artifact-manifests.json.
Effort: L-XL. Risk: High.
Pros: Drift impossible by construction.
Cons: Over-engineering for 5 files.

## Recommended Approach

Approach B. Fix all drift now. Simplify lifecycle to 3 phases. Design doc captures the "why" for future reference.

### The 3-Phase Lifecycle Model

**Phase 1: Track** (unchanged from current Phase 1)
- Run `/office-hours` to explore the problem space
- Create working branch
- Scaffold work item directory, TRACKER.md, and type-specific docs
- Define acceptance criteria, decompose into children

**Phase 2: Implement** (unchanged from current Phase 2)
- Work from design doc + acceptance criteria
- Commit changes incrementally
- Update Todos and Files sections

**Phase 3: Ship** (combines current Phase 3 + Phase 4)
1. Run `/docs check` to verify work item validation
2. Run `/docs tree` to verify structural completeness (feature/user-story only)
3. Verify tests pass
4. Verify child items have shipped (feature/user-story only)
5. Run `/ship` (includes pre-landing code review, creates PR)
6. Run `/land-and-deploy` to merge and verify

Gates vary by type:
- **Feature**: `/docs check` passes, `/docs tree` complete, all children shipped, `/ship` PR created, `/land-and-deploy` merged
- **User-story**: `/docs check` passes, `/docs tree` verified, TEST-SPEC covers P0 ACs, all children shipped, `/ship` PR created, `/land-and-deploy` merged
- **Task**: `/docs check` passes, test-plan verified, `/ship` PR created, `/land-and-deploy` merged
- **Defect**: `/docs check` passes, regression test passing, `/ship` PR created, `/land-and-deploy` merged

Key change: `/review` is dropped as a standalone gate because `/ship` runs it internally. The `/review` gate in the old Phase 3 was redundant.
