## Problem Statement

The workbench has accumulated 5 deprecated shims and a layered deprecation-infrastructure surface (catalog enum + `deprecated/` directory + `--include-deprecated` flag + audit conventions for retired skills + tombstone sections across 4 doc surfaces). All of it was originally justified for backward-compat with operators running in-flight pipelines — a real concern in a multi-operator world.

**This is a solo project.** There are no other operators. There are no in-flight pipelines stuck on the lowercase `/cj_goal_feature` or on `/CJ_goal_investigate`. The 5 shims are dead weight; the infrastructure that supports them is dead weight; the audit checks and convention sections that exist solely to enforce graceful deprecation are dead weight. CLAUDE.md's planned-but-deferred "v6.0.0 sunset wave" can execute now.

The user's framing — "since this is a solo project, let's delete the references for all deprecated features" — is the trigger. Approach B (the full nuke, chosen via AUQ) takes it further than the documented v6.0.0 plan: it also retires the deprecation INFRASTRUCTURE (status enum value, `deprecated/` dir, `--include-deprecated` flag, F000030 retired-skill drift convention). Future deprecations would re-introduce the infrastructure when needed, designed around whatever the next retirement actually requires.

## What Makes This Cool

This is the workbench's first deletion-only feature — every prior F-ID added a convention. F000035 removes them. The diff is large but every line is subtractive. The post-merge workbench is smaller, simpler, faster to read end-to-end, and the documented "v6.0.0 sunset" promise is kept.

The bonus dogfood: the deprecation pattern that ships *now* will land via the same `/CJ_goal_feature` pipeline that the deprecated shims point at — the canonical replacement path is the one routing the cleanup PR. The shims have effectively documented their own death.

## Open Questions

1. **Should we leave one tombstone breadcrumb anywhere — a CHANGELOG entry isn't enough?** No. CHANGELOG.md entry + git history (commit message + the F000027 + F000031 + T000035 + F000035 PR titles) form the audit trail. PHILOSOPHY.md `## Retired skills` was the per-deprecation tombstone home; with the deprecation pattern itself retired, the tombstones are retired too. Future archaeologists use `git log --grep deprecated` or `git log --all` over the deleted paths.
2. **What if a future skill legitimately needs to deprecate?** Re-introduce the pattern when that happens, designed around the actual retirement (not the F000027 one). The patterns we're removing — `deprecated/` dir convention, `--include-deprecated` flag, status enum value — are easy to re-add if a real need surfaces. Solo project = we control the schedule.
3. **Should the F000030 "Retired-skill drift check" convention stay in case PHILOSOPHY.md or ARCHITECTURE.md gain post-mortem tombstones in the future?** No. The check enumerates `status: deprecated` entries from the catalog. With zero such entries (and the enum value removed), the check has nothing to check. Removing the convention from CLAUDE.md is consistent with removing the substrate it operated on.
4. **Should the `skills/CJ_goal_run/` and `skills/CJ_goal_auto/` shims have moved to `deprecated/` before this PR (the F000031 convention)?** They didn't, because the convention shipped via F000031 but the F000027 shims predated it and the deferral note (F000031 Open Q #6) said "move at v6.0.0 sunset PR." This PR is that PR; the moves happen via the deletion, not via relocation.
5. **Does removing the `--include-deprecated` flag affect any external caller (cron job, scheduled task)?** Solo project. Operator has no scheduled tasks calling `skills-deploy --include-deprecated`. The flag was for in-flight pipelines; sunset = no in-flight pipelines.
6. **Should `tests/eval/CJ_goal_run/` deletion include the `tests/eval/` parent if it's empty?** Inspect after deletion. If only `tests/eval/CJ_goal_run/` was there, the parent goes too. If other eval dirs exist (`CJ_goal_feature`, `CJ_goal_defect`?), leave the parent. Eval-hardening (D000023) is deferred per memory `project_eval_hardening_deferred` — eval dir scope minimization is a follow-up if needed.
