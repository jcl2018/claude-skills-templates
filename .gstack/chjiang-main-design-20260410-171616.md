## Problem Statement

The skill workbench has tooling (scaffold, validate, test, lint, distribute) but no process. There's no design phase before writing a skill, no per-skill version tracking, and no enforced review/ship workflow. When the repo scales to 30+ skills, there's no way to know what changed in a specific skill, why design decisions were made, or whether it's safe to update a skill that other skills depend on. The routine that protects future-you from past-you is missing.

## What Makes This Cool

Skills become first-class software artifacts with a full lifecycle. Every skill ships with evidence: a design doc that captures why, a changelog that captures what changed, version numbers that communicate impact, and quality gates that can't be skipped. The pipeline IS the routine — following it produces the artifacts automatically. No discipline required, just the process.

## Constraints

- Must work with the existing tooling stack (shell scripts, YAML frontmatter, JSON catalog)
- No new runtime dependencies (no Node.js, no Python, just POSIX + grep/sed/awk + jq for JSON)
- Frontmatter `sed`-modified fields (`name`, `description`, `version`) must be single-line `key: value`. Other fields (e.g., `allowed-tools`) may use YAML list syntax and are not modified by `sed`.
- Must be usable by one person (the author) before it's useful to a community
- Skills are Markdown + shell scripts — not compiled packages, no registry, distribution is `git pull`
- Must be lightweight enough that small skills don't feel over-documented
- Must integrate with the existing pre-commit hook infrastructure (setup-hooks.sh)

## Premises

1. Skills need a dedicated lifecycle pipeline, separate from the feature work pipeline (work-track → work-ship), because skills are reusable components with different versioning, review, and maintenance concerns.
2. Each skill gets its own design doc (a skill-specific variant of the triplet), created before implementation, that captures the "why" and survives as living documentation.
3. Version control is per-skill (version in frontmatter, per-skill changelog), not per-repo, because skills ship independently and consumers need to know what changed in the skill they use.
4. Quality gates are enforced by scripts/hooks, not just documented — because documented-but-optional gates always get skipped under time pressure.
5. Everything built with existing tooling (shell scripts, YAML frontmatter, JSON catalog).

## Cross-Model Perspective

Codex's cold read produced four key insights:

1. **Coolest version:** A skill registry with machine-readable contracts (inputs, outputs, trust boundaries, compatibility guarantees). CI computes semantic diffs on every change. "A time machine, not just a checklist."
2. **Key insight:** "You're not excited by documentation. You're excited by enforcement." The automated quality gates answer reveals the real motivation.
3. **50% tool:** Changesets (the monorepo release tool) solves per-item versioning, changelog generation, and CI-enforceable release metadata. The missing 50%: skill-native artifacts (design docs), frontmatter-aware diffing, golden tests, per-skill policy rules.
4. **Weekend build:** One gatekeeper command (`skill-check --changed`) that fails if required artifacts are missing, plus `skill-init` to scaffold them. Wire into CI immediately.

The contract-driven approach (machine-readable contracts, semantic diff) is deferred to a future iteration. The patterns from Changesets (intent files, per-item changelogs, CI status checks) are adopted but implemented natively in shell, not via the Node.js tool.

## Approaches Considered

### Approach A: Gatekeeper-First (Minimal Viable)
One script (`skill-check.sh`) that validates changed skills against lifecycle requirements, plus a skill-specific design doc template. Enforce via pre-commit hook.
- Effort: S | Risk: Low
- Pros: Ships fast, builds on existing scripts, enforcement teaches the workflow
- Cons: No per-skill changelog, no version bump automation, design doc status untracked
- Completeness: 5/10

### Approach B: Full Lifecycle Pipeline (Ideal Architecture)
Complete pipeline: `skill-design.sh` (scaffold design doc) → `create-skill.sh` (scaffold SKILL.md) → implement → `skill-check.sh` (gate) → `skill-version.sh` (bump + changelog) → `skill-ship.sh` (review + tag). Each stage produces an artifact. Pre-commit enforces.
- Effort: M | Risk: Medium
- Pros: Every stage gated, per-skill CHANGELOG.md, design doc captures decisions, skill-check ties it all together
- Cons: More scripts to maintain, need to define "ship" semantics, design doc might feel heavy for trivial skills
- Completeness: 8/10

### Approach C: Contract-Driven (Codex-Inspired Lateral)
Machine-readable contracts in SKILL.md frontmatter (inputs, outputs, trust-boundary, compat-version). `skill-check.sh` validates contracts + semantic diff. Golden test fixtures per skill.
- Effort: L | Risk: High
- Pros: Most future-proof, semantic diff catches behavior changes, golden tests catch regressions
- Cons: Contract format needs design upfront, golden tests for Markdown skills are novel, possibly over-engineering for 8 skills
- Completeness: 10/10

## Recommended Approach

**Approach B: Full Lifecycle Pipeline.** Solves all three gaps (design, versioning, review) with enforced gates, without over-engineering. Can evolve toward Approach C's contracts once the lifecycle is running and real-world usage reveals what contracts should look like.

### Pipeline Stages

```
1. DESIGN    → skill-design.sh {name}
               Creates: skills/{name}/DESIGN.md (from template)
               Gate: Design doc must exist before SKILL.md can be created

2. SCAFFOLD  → create-skill.sh {name}  (existing, enhanced)
               Creates: skills/{name}/SKILL.md, CHANGELOG.md
               Gate: Checks for DESIGN.md first
               Enhancement: Also scaffolds CHANGELOG.md with initial entry

3. IMPLEMENT → (manual — author writes the SKILL.md)
               No script. This is the creative work.

4. VALIDATE  → skill-check.sh {name}  (new)
               Checks: DESIGN.md exists, SKILL.md frontmatter valid, 
               CHANGELOG.md has entry for current version, version in 
               frontmatter matches catalog version, lint passes
               Gate: Must pass before version bump

5. VERSION   → skill-version.sh {name} {major|minor|patch}  (new)
               Updates: version in SKILL.md frontmatter, version in 
               skills-catalog.json, adds version header to CHANGELOG.md
               Gate: skill-check.sh must pass first

6. SHIP      → skill-ship.sh {name}  (new)
               Runs: full validation, shows diff summary, commits with 
               conventional message (feat/fix/chore: skill-name description),
               tags: {name}-v{version}
               Gate: skill-check.sh must pass, version must be bumped
```

### Artifacts Per Stage

| Stage | Input | Output Artifact | Enforced By |
|-------|-------|----------------|-------------|
| Design | Idea | `skills/{name}/DESIGN.md` | skill-check.sh (existence check) |
| Scaffold | Design doc | `SKILL.md` + `CHANGELOG.md` | create-skill.sh (checks DESIGN.md) |
| Implement | Scaffold | Written SKILL.md | Manual |
| Validate | Implementation | Pass/fail report | skill-check.sh |
| Version | Validated skill | Bumped version + changelog entry | skill-version.sh (runs skill-check first) |
| Ship | Versioned skill | Git commit + tag | skill-ship.sh (runs skill-check first) |

### Design Doc Template: `templates/doc-SKILL-DESIGN.md`

A skill-specific design doc template, lighter than the feature triplet (PRD + ARCH + TEST-SPEC). Skills are components, not features — they need different documentation.

```markdown
---
skill-name: {name}
version: 0.1.0
status: DRAFT | APPROVED | IMPLEMENTED | DEPRECATED
created: {date}
last-updated: {date}
---

# Skill Design: {name}

## Purpose
What problem does this skill solve? Who uses it and when?

## Behavior
What does the skill do, step by step? What tools does it use?
What are the inputs (user prompts that trigger it) and outputs (artifacts it produces)?

## Design Decisions
Why this approach over alternatives? What tradeoffs were made?

## Dependencies
Which other skills does this depend on? What external tools are required?

## Security Boundaries
What tools should be allowed? What should be restricted?
(Maps to allowed-tools in SKILL.md frontmatter)

## Test Criteria
How do you know this skill works correctly? What would a smoke test look like?
```

### Per-Skill CHANGELOG.md

Lives at `skills/{name}/CHANGELOG.md`. Format:

```markdown
# Changelog: {name}

## [0.1.0] - {date}
### Added
- Initial implementation
- {description of what the skill does}
```

Follows Keep a Changelog format (keepachangelog.com). Each version bump via `skill-version.sh` adds a new header. The author fills in the changes.

### Gate Architecture

`skill-check.sh` is the **single source of all validation**. No other script performs ad-hoc checks. Other scripts (`create-skill.sh`, `skill-version.sh`, `skill-ship.sh`) call `skill-check.sh` as a prerequisite and only perform their own argument validation (e.g., "is this a valid skill name?"). This means validation rules are defined in one place and enforced everywhere.

### Pre-Commit Integration

The existing `setup-hooks.sh` installs a pre-commit hook that runs `validate.sh`. This gets extended:

1. Detect which skills have changed files (`git diff --cached --name-only | grep '^skills/'`)
2. For each changed skill, run `skill-check.sh {name}`
3. Block commit if any check fails

Per-skill checks are lightweight (file existence, grep, jq queries). At 30 skills with a worst-case commit touching all of them, execution is under 5 seconds.

This is the enforcement layer. You can't commit a skill change without passing the gates.

### Frontmatter Version Field

SKILL.md frontmatter gains a new `version` field. Note: `skills-catalog.json` already has a `version` field for each skill — that is the existing source of truth. The new frontmatter `version` is the co-located addition. `skill-check.sh` validates that these two values match.

```yaml
---
name: work-track
description: Context-aware work item management
version: 1.2.0
allowed-tools: Read, Edit, Glob, Grep, Write, Bash
---
```

Fields modified by `sed` (`name`, `description`, `version`) must be single-line `key: value`. Other fields like `allowed-tools` may use YAML list syntax — `sed` does not touch them.

### Catalog Sync

`skill-version.sh` updates both the SKILL.md frontmatter (via `sed`) and the `skills-catalog.json` entry (via `jq`) sequentially. This is a single-author repo, so no locking or true atomicity is needed — the updates happen in one script invocation and both complete before the script exits. No manual catalog editing for version bumps.

### Lightweight Escape Hatch

Any skill's DESIGN.md can be minimal — just Purpose and Behavior sections filled in. The other sections (Design Decisions, Dependencies, Security Boundaries, Test Criteria) can contain "N/A" for simple skills. This is the author's judgment call, not enforced by line count. `skill-check.sh` validates that at least Purpose and Behavior are non-empty.

### Skill Name Validation

Skill names must not match the pattern `*-v[0-9]*` to avoid collisions with the git tag namespace (`{name}-v{version}`). `skill-check.sh` and `create-skill.sh` both reject invalid names.

### DESIGN.md Location

DESIGN.md lives inside the skill directory at `skills/{name}/DESIGN.md`, co-located with SKILL.md and CHANGELOG.md. This differs from the feature doc triplet pattern (which lives in `docs/{name}/`). The rationale: skills are self-contained components. Co-locating all skill artifacts in one directory makes the skill portable — you can copy the entire `skills/{name}/` directory and get everything. The feature triplet pattern at `docs/` is for cross-cutting features that span multiple skills.

### Migration Plan for Existing Skills

The 8 existing skills lack DESIGN.md and CHANGELOG.md files. Migration approach:

1. `scripts/skill-migrate.sh` creates minimal DESIGN.md and CHANGELOG.md stubs for each existing skill
2. DESIGN.md is seeded from the existing SKILL.md: Purpose extracted from `description` frontmatter, Behavior extracted from the first section of the skill body. Other sections set to "N/A — retroactively documented."
3. CHANGELOG.md initialized with a single `[0.1.0]` entry: "Initial version (retroactively documented)"
4. `version: 0.1.0` added to each SKILL.md frontmatter
5. Migration is a hard prerequisite — once the pipeline ships, `skill-check.sh` applies to ALL skills, not just new ones. Run `skill-migrate.sh` once, commit the results, then the gates apply uniformly.

## Open Questions

1. **What does "ship" mean?** Is it a git tag per skill? A catalog status change? Both? The design assumes git tags (`{name}-v{version}`) but this needs validation with real usage.
2. **Breaking change detection:** How do you know a change is breaking? For compiled code, API signatures change. For skills, the "API" is the prompt surface and the artifacts produced. This is hard to automate without Approach C's contracts.
3. **Multi-skill commits:** When a single change touches multiple skills (e.g., updating a shared pattern), should each skill get its own version bump, or should there be a batch mode?

## Success Criteria

- Every new skill follows the pipeline: DESIGN.md → SKILL.md + CHANGELOG.md → validate → version → ship
- Pre-commit hook blocks commits with invalid skill state
- `skill-check.sh` catches: missing design doc, missing changelog entry, version mismatch between frontmatter and catalog, lint failures
- Any skill's full history is reconstructable: design decisions (DESIGN.md), version history (CHANGELOG.md), code changes (git log)
- Adding a new skill from scratch takes under 5 minutes of setup (scripts handle scaffolding)

## Distribution Plan

This is a Claude Code plugin distributed via git clone/pull. The lifecycle pipeline scripts are part of the repo and available to anyone who clones it. No separate distribution needed — the scripts ARE the deliverable.

CI/CD: The pre-commit hook is the primary enforcement. GitHub Actions can optionally run `skill-check.sh` on PRs for remote enforcement.
