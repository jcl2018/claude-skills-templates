## Problem Statement

The /docs skill (v0.1.0) has two subcommands: init (generate narrative docs) and check (staleness detection via claims sidecar). Work item validation lives entirely in CLAUDE.md rules (rules/work-items.md) — passive instructions that Claude follows during scaffolding but never enforces after the fact. Nobody checks whether existing work items still match their templates, whether required sections are present, whether frontmatter fields are correct, or whether parent/child lifecycle states are consistent (parent closed + children open).

The existing work items in work-items/F000001_workflow_alpha/ already have real mismatches: tracker-user-story.md says `type: user-story` but S000001_TRACKER.md uses `type: userstory`, the manifest expects filenames like `PRD.md` but actual files are ID-prefixed like `S000002_PRD.md`, and S000001 has no `parent` field despite the template requiring one.

## What Makes This Cool

A unified "truth checker" for the entire repo. /docs check already knows how to flag stale narrative docs. Adding work item validation means one command tells you everything that's drifted, broken, or inconsistent. The normalization layer (canonical models from manifests + templates + work-items/) makes this robust — it handles the naming quirks and type spelling mismatches that would break a naive string-comparison approach.

## Constraints

- Must extend existing check.md, not replace it
- Must read artifact-manifests.json for per-type required artifacts (no hardcoded template lists — prior learning: track-validation-hardcoded-templates)
- Must handle ID-prefixed filenames (S000002_PRD.md matches PRD.md artifact)
- Must normalize type spelling (user-story vs userstory — prior learning: userstory-vs-user-story-inconsistency)
- Must work in foreign repos where work-items/ may not exist (graceful skip)
- Must gracefully skip if artifact-manifests.json is missing or malformed (same pattern as claims.json: warn and skip, don't crash)
- Output format must be consistent with existing check.md staleness report
- No scaffolding — validation only. Scaffolding stays in CLAUDE.md rules.

## Premises

1. /docs check expands to cover work items using the same compare-against-baseline, flag-drift pattern as narrative doc staleness.
2. Scaffolding stays in CLAUDE.md rules. This feature is validation only.
3. Two engines in one check: claims-based staleness (existing, for narrative docs) + rule-based lifecycle validation (new, for work items).
4. Doc writer is phase 2. This design covers validation only.

## Cross-Model Perspective

Codex (independent cold read) proposed a versioned contract system — compile templates into contracts with hashes, stamp work items at scaffold time, report three states (invalid/behind/graph-blocked) with a migration advisor. The versioned contracts vision is compelling at scale but overkill for the current state (1 feature, 5 work items). The normalization layer from Approach B captures the structural foundation that versioned contracts would build on later.

Key Codex insight: "Until that canonical model exists, every higher-level validator will be brittle." This confirmed the normalization-first approach.

Codex also found real mismatches in existing data (type spelling, missing parent fields, ID-prefixed filenames) — the same issues captured in prior learnings.

## Approaches Considered

### Approach A: Minimal Check Extension
Add work-item scanning to check.md. Hardcode known quirks. Ships in ~15 min.
- Effort: S
- Risk: Low
- Completeness: 5/10
- Rejected: too brittle, doesn't handle naming quirks cleanly

### Approach B: Normalization Layer + Structured Validation (Chosen)
Build canonical models from manifests/templates, crawl work-items/ into a second model, validate with three checks. Handles ID-prefix and type normalization.
- Effort: M (CC: ~30 min)
- Risk: Low-Med
- Completeness: 8/10
- Chosen: robust, extensible, catches real mismatches immediately

### Approach C: Versioned Contracts
Template hashing, three-state validation, migration advisor. Full Codex vision.
- Effort: L (CC: ~60 min)
- Risk: Med
- Completeness: 10/10
- Deferred: earns its keep at scale, not with 5 work items

## Recommended Approach

Approach B: Normalization Layer + Structured Validation

### Architecture

The check subcommand gains a new section. After the existing staleness check (claims.json-based) and coherence check (broken links, version conflicts), add:

**Work Item Validation** with three check types:

#### Check 1: Template Compliance

For each work item in work-items/:
1. Parse TRACKER.md frontmatter to determine type (feature/defect/task/user-story)
2. Normalize type spelling (strip hyphens for comparison: "user-story" and "userstory" both match)
3. Look up required artifacts in artifact-manifests.json for that type
4. For each required artifact:
   - Find the matching file (handle ID-prefix: `{ID}_{filename}` or just `{filename}`)
   - If missing: flag as MISSING
   - If found: parse YAML frontmatter, check required fields (derived from the Expected Model — see Normalization Layer below)
   - Compare `##` and `###` section headers against the template's required sections (also derived from the Expected Model)
   - Flag DRIFT for missing sections, EXTRA for unexpected sections (warn only)

#### Check 2: Lifecycle Consistency

Walk the work-items/ tree (max depth 3 per rules):
1. Parse each TRACKER.md for lifecycle state (look for checkbox patterns: `- [x]` = done, `- [ ]` = pending)
2. Determine work item status from lifecycle gates:
   - All gates checked = Closed
   - Some gates checked = In Progress
   - No gates checked = Open
3. Validate parent/child relationships:
   - If parent is Closed, all children must be Closed (flag: LIFECYCLE_INCONSISTENT)
   - If child is Closed but parent is Open, warn (may be intentional — advisory only)
4. Validate nesting depth (max 3: feature > user-story > task)

#### Check 3: Cross-Reference Traceability

For features and user-stories:
1. Parse PRD.md for user story entries in the P0/P1/P2 tables (rows with `| # | Tag | ...`)
2. Parse PRD.md acceptance criteria sections (`### Story #{n}: {title} [{tag}]`)
3. Parse TEST-SPEC.md test matrix for `AC-{n}` references in the AC column
4. Check that each PRD acceptance criterion has at least one TEST-SPEC row referencing it
5. Flag UNTESTED for PRD stories with no test coverage

**Concrete matching example:**

PRD.md has:
```
### P0 (Must-Have)
| # | Tag  | What it asks        | As a... | I want to... | So that... |
| 1 | core | Does type normalize | dev     | type match   | no drift   |

### Story #1: Type normalization [core]
GIVEN a tracker with type "userstory"
WHEN  /docs check runs
THEN  it normalizes to "user-story" for comparison
```

TEST-SPEC.md should have a row referencing AC-1:
```
| # | Tag  | Test Case           | AC   | ... |
| 1 | core | type normalization  | AC-1 | ... |
```

Match heuristic: PRD story `#N` corresponds to TEST-SPEC rows with `AC-N` in the AC column. If no TEST-SPEC row references `AC-1`, flag story #1 as UNTESTED.

Parsing patterns are derived from the actual templates:
- PRD structure from `doc-PRD.md`: user stories in `### P0/P1/P2` tables, acceptance criteria in `### Story #{n}` sections
- TEST-SPEC structure from `doc-TEST-SPEC.md`: test matrix table with `AC-{n}` in the AC column
If templates change their section structure, the parsing heuristic should be updated to match.

### Normalization Layer

Before running checks, build two canonical models:

**Expected Model** (derived by parsing manifests + templates at check time, not hardcoded):
```json
{
  "feature": {
    "required_artifacts": ["TRACKER.md", "PRD.md", "ARCHITECTURE.md", "TEST-SPEC.md", "milestones.md"],
    "tracker_required_fields": ["name", "type", "id", "status", "created", "updated", "repo", "branch", "blocked_by"],
    "tracker_required_sections": ["## Lifecycle", "### Phase 1: Track", "### Phase 2: Implement", "### Phase 3: Review", "### Phase 4: Ship"]
  }
}
```

Note: `tracker_required_fields` and `tracker_required_sections` are derived by parsing the actual template file (e.g., `tracker-feature.md`), not hardcoded. If a template adds or removes fields, the check picks up the change automatically.

**Actual Model** (from crawling work-items/):
```json
{
  "F000001_workflow_alpha": {
    "type": "feature",
    "tracker_fields": {"type": "feature", "status": "...", "id": "F000001"},
    "artifacts_found": ["TRACKER.md", "PRD.md", "ARCHITECTURE.md", "TEST-SPEC.md", "milestones.md"],
    "children": ["S000001_four_phase", "S000002_template_consolidation"]
  }
}
```

Normalization rules:
- Type spelling: normalize to hyphenated form (user-story) for comparison
- Filenames: strip `{ID}_` prefix when matching against manifest `filename` field
- Paths: all relative to work-items/

### Output Format

Extends the existing check.md report:

```
=== /docs check ===
Claims: .docs/claims.json (generated TIMESTAMP, commit SHA)

STALENESS CHECK:
  [existing narrative doc staleness output]

COHERENCE CHECK:
  [existing coherence output]

WORK ITEM VALIDATION:
  F000001_workflow_alpha (feature):
    TEMPLATE:
      [PASS]  TRACKER.md — all required fields present
      [PASS]  PRD.md — all required sections present
      [DRIFT] ARCHITECTURE.md — missing "## Security Considerations"
      [PASS]  TEST-SPEC.md — all required sections present
      [PASS]  milestones.md — all required sections present
    LIFECYCLE:
      [PASS]  All children consistent with parent state
    TRACEABILITY:
      [WARN]  PRD US-3 has no corresponding TEST-SPEC entry

  S000001_four_phase (user-story):
    TEMPLATE:
      [DRIFT] TRACKER.md — type field says "userstory", expected "user-story"
      [DRIFT] TRACKER.md — missing required field "parent"
      [MISSING] PRD.md — required artifact not found
      [MISSING] ARCHITECTURE.md — required artifact not found
    LIFECYCLE:
      [PASS]  No children

SUMMARY: 2 items checked, 4 drift issues, 1 traceability warning, 0 lifecycle issues
```

## Open Questions

1. Should template compliance check against `contract-*.md` (enforcement reference templates) or `doc-*.md` (scaffolding templates)? Contract templates define "what good looks like" which seems right for validation. Scaffolding templates have placeholders.
2. When a work item intentionally diverges from its template (e.g., extra sections added during development), should EXTRA sections be flagged? Currently proposed as warn-only.
3. Should the check auto-discover work-items/ location or read it from artifact-manifests.json `work_items_dir`?

## Success Criteria

- `/docs check` in a repo with work-items/ produces the extended report including template compliance, lifecycle consistency, and traceability checks
- `/docs check` in a repo without work-items/ gracefully skips the work item section
- Real mismatches in F000001_workflow_alpha are caught (type spelling, missing parent field, missing artifacts for S000001)
- Output format is consistent with existing staleness/coherence report sections
