## Problem Statement

The `/docs` skill is a grab-bag: narrative doc generation (init), work item validation
(check), and hierarchy view (tree). It depends on repo-root `artifact-manifests.json`
and repo-root `templates/*.md`, making it entangled with the workbench's global state.
Meanwhile, `/company-workflow` is fully self-contained: own templates, own manifest,
own contract, own examples. The two skills should be structurally independent and easy
to compare side-by-side.

## What Makes This Cool

Two self-contained workflow skills, side by side. Same directory structure. You can
instantly see what's shared (contract.json, manifest, templates, fixtures) and what's
different (personal has hierarchy enforcement + cross-refs + graph artifacts + reports;
company has 4-phase lifecycle + review type + examples). One `ls` of each skill dir
tells the whole story.

## Constraints

- Zero shared dependencies between personal-workflow and company-workflow
- Same directory structure as company-workflow for side-by-side comparison
- Preserve all work item validation functionality from check.md (hierarchy, cross-refs, graph, report). Staleness detection (old Steps 1-5) is killed by design.
- /docs init (narrative doc generation, claims sidecar, staleness detection) is killed entirely
- Personal lifecycle is 3-phase (Track/Implement/Ship), company is 4-phase

## Premises

1. `/personal-workflow` is fully self-contained: own templates, manifest, contract, fixtures
2. Kill `/docs` entirely (including init.md, claims sidecar, staleness detection)
3. Repo-root templates move to `templates/personal-workflow/`
4. `artifact-manifests.json` at repo root moves into the skill dir as `personal-artifact-manifests.json`
5. `check` and `tree` subcommands keep their current behavior but rewritten around contract.json pattern
6. `/personal-workflow` gets its own WORKFLOW.md (scaffolding conventions from rules/work-items.md)

## Approaches Considered

### Approach A: Mirror + Migrate
Move files, update paths, mirror directory structure. No logic changes beyond path
references. Fast but leaves check.md structurally different from company-workflow.

### Approach B: Mirror + Rewrite (CHOSEN)
Same structure as A, but rewrite check.md to use company-workflow's validation approach
(contract.json + manifest) as the foundation. Layer hierarchy, cross-refs, graph, and
report as clearly labeled extensions. Makes the shared DNA obvious.

### Approach C: Mirror + Thin SKILL.md
Same as A but rename `check` to `validate` for naming consistency. Risk of confusion
since personal `validate` would behave differently from company `validate`.

## Recommended Approach

Approach B: Mirror + Rewrite. Deeper structural alignment between the two skills.
Check.md gets rewritten with clear separation between "foundation" (same as company)
and "extensions" (personal-workflow only).

## contract.json (personal)

```json
{
  "version": 1,
  "frontmatter": {
    "required": ["name", "type", "status", "created", "updated"],
    "recommended": ["repo", "branch"]
  },
  "sections": {
    "required": ["Lifecycle", "Todos", "Log", "PRs", "Files", "Journal"],
    "optional": ["Acceptance Criteria", "Reproduction Steps", "Insights"],
    "expected_order": ["Lifecycle", "Acceptance Criteria", "Reproduction Steps", "Todos", "Log", "PRs", "Files", "Insights", "Journal"],
    "order_skip_absent": true,
    "type_specific_optional": {
      "feature": ["Acceptance Criteria"],
      "user-story": ["Acceptance Criteria"],
      "defect": ["Reproduction Steps"],
      "task": []
    }
  },
  "lifecycle": {
    "min_checkboxes": 3,
    "phases": ["Track", "Implement", "Ship"]
  }
}
```

Key differences from company contract.json:
- 3 phases vs 4 (no Review phase)
- 3 min checkboxes vs 4 (one per phase minimum)
- "Acceptance Criteria" and "Reproduction Steps" in optional (type-specific sections)
- `order_skip_absent: true` — order validation skips optional sections not present in the file.
  Algorithm: filter `expected_order` to only sections actually present in the file,
  then assert the filtered list matches the file's section order. Example: if
  `expected_order` is `[Lifecycle, AC, Todos, Log]` and file has `[Lifecycle, Todos, Log]`,
  filter produces `[Lifecycle, Todos, Log]` which matches. PASS.
  Note: this is a personal-workflow divergence from company-workflow. Company's contract
  has no `order_skip_absent` field and its optional sections appear after all required
  sections, avoiding the issue. Tier 1 is "foundation-inspired," not identical.
- No "Meetings" optional section
- No overdue/deadline fields

## check.md Rewrite Structure

The rewritten check.md is organized in two tiers:

### Tier 1: Foundation (matches company-workflow validate)

Maps to company-workflow SKILL.md sections "Command: validate" through "Report".

**File Mode** (Steps 1-7):
1. Path resolution (2-level fallback)
2. Read contract.json
3. Read target file, parse YAML frontmatter
4. Check required frontmatter fields from contract
5. Check required sections, verify expected_order (skip absent optional sections)
6. Check lifecycle structure (3 phases, min 3 checkboxes)
7. Report: VALID or list of VIOLATION lines

**Directory Mode** (Steps 7-13):
7. Locate TRACKER.md (with or without ID prefix)
8. Read type from frontmatter, normalize spelling
9. Load personal-artifact-manifests.json, find type entry
10. Check artifact completeness (filename matching rule: strip ID prefix)
11. Template frontmatter comparison (2-level fallback, placeholder detection)
12. Check lifecycle in TRACKER.md (phases, checkboxes, required sections)
13. Report: structured output with [PASS], [MISSING], [DRIFT] per artifact

### Tier 2: Extensions (personal-workflow only)

These steps have no equivalent in company-workflow. They require walking the
entire work-items/ tree, not just validating one file or directory.

**Recursive walk** (Steps 14-15):
14. Build Actual Model: walk work-items/ recursively (max depth 4), parse every
    TRACKER.md, determine parent/child relationships and state
15. Build Expected Model: for each type in manifest, resolve templates, extract
    required frontmatter fields and section headers

**Cross-checks** (Steps 16-19):
16. Template compliance: compare each work item against Expected Model
17. Lifecycle consistency: parent/child state checks, "broken down" cross-ref
18. Cross-reference traceability: PRD P0 stories mapped to TEST-SPEC matrix
19. Structural completeness: hierarchy enforcement, placement, orphan detection

**Outputs** (Steps 20-23):
20. Badge taxonomy: map all statuses to 4 badge categories with severity ordering
21. Tree report: unified tree view with per-node badges
22. Graph artifact: .docs/work-item-graph.json
23. Human-readable report: .docs/work-item-report.md

**Invocation model:** When invoked on a file path, only Tier 1 File Mode runs.
When invoked on a directory path, Tier 1 Directory Mode runs, then Tier 2 always
follows. When invoked with no path argument (`/personal-workflow check`), both tiers
run on the entire `work-items/` directory. If `work-items/` does not exist, Tier 2
is skipped with: `INFO: no work-items/ directory found. Skipping hierarchy checks,
graph artifact, and report generation.`

Each tier is clearly commented so future readers can see which parts match
company-workflow and which parts extend it.

## Path Resolution

2-level fallback chain (same pattern as company-workflow):

```
Level 1: $REPO_ROOT/skills/personal-workflow/     (workbench)
Level 2: ~/.claude/skills/personal-workflow/       (deployed)

Templates:
Level 1: $REPO_ROOT/templates/personal-workflow/
Level 2: ~/.claude/templates/personal-workflow/
```

The `~/.claude/spec/templates/` middle level from the old 3-level chain is dropped.
It was a legacy indirection that is no longer needed.

## SKILL.md Command Routing

SKILL.md is a thin router. It detects the subcommand from the user's input and
dispatches to the corresponding .md file:

- `/personal-workflow check` or `/personal-workflow check <path>` -> read check.md, follow it
- `/personal-workflow tree` -> read tree.md, follow it
- `/personal-workflow` with no subcommand -> show usage (list available subcommands)

This matches the pattern in the old `/docs` SKILL.md. SKILL.md contains only
preamble (path resolution, git repo check) and routing logic. All validation
logic lives in check.md and tree.md.

## Migration for skills-deploy Users

Users who previously ran `skills-deploy install` will have `/docs` deployed at
`~/.claude/skills/docs/` and templates at `~/.claude/templates/*.md`. After this change:

- `skills-deploy install` deploys personal-workflow to `~/.claude/skills/personal-workflow/`
  and templates to `~/.claude/templates/personal-workflow/`
- `skills-deploy remove` (run first) cleans up the old `/docs` deployment
- The old flat templates at `~/.claude/templates/*.md` become orphans.
  `skills-deploy doctor` will flag them. Manual cleanup: `rm ~/.claude/templates/tracker-*.md ~/.claude/templates/doc-*.md`

**Breaking change: scaffolding rules are now skill-invocation-scoped.**
Previously, `rules/work-items.md` was deployed to `~/.claude/rules/` and loaded into
every Claude Code session automatically. After this change, scaffolding conventions
live in WORKFLOW.md inside the skill. They are only read when `/personal-workflow` is
invoked, not globally available. Users must invoke the skill to access scaffolding
conventions. This is intentional: the skill should be self-contained, and global rules
create invisible coupling.
