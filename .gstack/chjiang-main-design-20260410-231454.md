## Problem Statement

This repo contains 4 custom skills and 20+ templates distributed as an open source collection. Each skill has its own version (system-health 0.3.0, workflow 0.1.0, etc.) tracked in SKILL.md frontmatter and skills-catalog.json. But there is no repo-level version. Templates have zero versioning. A consumer who clones this repo cannot say "I'm on v1.2.0" and know what that means. An author who ships a skill change has no way to signal to consumers that the collection has advanced. The repo looks like a loose bag of files, not a versioned product.

## What Makes This Cool

Versioning as trust. Not just a VERSION file and a tag, but a system where `skills-deploy` can tell a consumer: "You're on collection 1.2.0, locally drifted on 2 templates, and contracts changed incompatibly since 1.1.0." The manifest turns the repo from a clone-and-pray experience into something reproducible and supportable.

## Constraints

- Pure bash. No Node, no Python, no external tooling dependencies. Changesets is conceptually relevant but architecturally wrong for this repo.
- Must compose with existing scripts: skill-version.sh, skill-ship.sh, validate.sh, skills-deploy, lib.sh.
- Cannot break existing per-skill versioning. Skills keep their own versions for internal tracking.
- Templates are covered by the collection version. The TODOS.md "Template version tracking (P3, S)" item becomes unnecessary.
- Consumers interact via git clone/pull + skills-deploy install. No package manager, no registry.

## Premises

1. The repo needs a single VERSION file at the root that tracks the collection version. This is what consumers reference.
2. Per-skill versions remain for internal tracking but are NOT what consumers pin to.
3. skill-ship.sh auto-bumps the collection patch version on every skill ship. The collection version always advances when any skill changes.
4. Templates are covered by the collection version. No separate template versioning needed.
5. validate.sh checks VERSION consistency (file matches latest git tag, catalog state is coherent).
6. skills-deploy records which collection version it installed from, so consumers can answer "what version am I on?" after install.

## Cross-Model Perspective

Codex (independent cold read) proposed enriching the approach beyond a simple VERSION file:

- **Coolest version:** Auto-generate a `collection-manifest.json` at each tag containing collection version, git commit, all skill versions, all templates, dependency graph, and checksums. This turns versioning from bookkeeping into trust, reproducibility, and supportability.
- **Key insight:** The user is excited about making the repo "feel like a real product people can talk about, install, report bugs against, and share."
- **50% tool:** Changesets conceptually maps (fixed groups, internal dependency updates) but is architecturally wrong (Node dependency for a bash repo).
- **Weekend build order:** VERSION file, collection-version.sh, patch skill-ship.sh, extend validate.sh, extend skills-deploy manifest, CHANGELOG.md last.

Claude agreed on build order and manifest enrichment. Disagreed on Changesets (not worth the dependency). Refined: manifest data should integrate into the existing skills-deploy manifest rather than creating a separate artifact.

## Approaches Considered

### Approach A: Minimal Viable
- VERSION file at root (start at 1.0.0)
- Repo-level CHANGELOG.md
- Patch skill-ship.sh to auto-bump collection patch + create v-tag
- Add collection-version.sh for manual minor/major bumps
- Effort: S | Risk: Low | Completeness: 5/10
- Pros: Ships in an hour. Consumers immediately get version + tags.
- Cons: No validation, no deploy tracking, no drift detection.

### Approach B: Ideal Architecture
- Everything in A, plus:
- validate.sh checks VERSION matches latest v-tag, catalog coherence
- skills-deploy records collection_version + commit SHA in manifest
- skills-deploy doctor reports version status
- Template changes also bump collection patch
- Effort: M | Risk: Low | Completeness: 9/10
- Pros: Full consumer story. Install, check version, validate, detect basic drift.
- Cons: ~3 scripts to modify. Template-bump trigger needs design.

### Approach C: Manifest-First (chosen)
- Everything in B, plus:
- Auto-generate collection-manifest.json at each v-tag
- Manifest contains: collection version, git commit, skill versions, template list with SHA256 checksums, dependency graph
- skills-deploy can diff local state against manifest for drift detection
- skills-deploy doctor reports drift (modified templates, version mismatches)
- Effort: L | Risk: Med | Completeness: 10/10
- Pros: Reproducibility, trust, drift detection. Versioning earns consumer confidence.
- Cons: New artifact. Some overlap with skills-catalog.json (but manifest is a snapshot, catalog is source of truth).

## Recommended Approach

**Approach C: Manifest-First.** The full version spine with drift detection.

### New files

| File | Purpose |
|------|---------|
| `VERSION` | Single line: `1.0.0`. The collection version. Source of truth. |
| `CHANGELOG.md` (repo root) | Collection-level changelog. Entries added on each collection version bump. |
| `scripts/collection-version.sh` | `get` / `bump patch\|minor\|major` / `manifest` for the collection version. |

### New script: collection-version.sh

- `get`: reads VERSION file, prints current version.
- `bump patch|minor|major`: increments VERSION, adds placeholder to CHANGELOG.md, prints new version.
- `manifest`: generates `collection-manifest.json` from current committed state (reads skills-catalog.json, checksums all templates, records git commit SHA). The manifest is regenerable on-demand from any commit or tag. It is NOT checked into git and NOT stored anywhere permanently. Any consumer or script can regenerate it by running `collection-version.sh manifest`.
- Validates semver format. Sources lib.sh but calls individual helpers selectively (`require_jq`, `validate_version_string`) rather than `init` (which pulls in catalog validation unnecessary for `get`/`bump`).

### Modified scripts

**scripts/skill-ship.sh:**
- After successful skill commit + skill tag, auto-bump collection patch version via collection-version.sh.
- Stage VERSION, commit the version bump, create collection v-tag (e.g., `v1.0.1`).
- The existing per-skill tag (e.g., `system-health-v0.4.0`) is still created. The collection v-tag is an additional tag on the version-bump commit.
- Two commits per ship:
  1. Skill commit: `feat(skill): ship system-health v0.4.0` (same as current format, includes skill files + catalog)
  2. Collection version bump commit: `chore: bump collection version to v1.0.1` (includes only VERSION + CHANGELOG.md)
- Each `skill-ship.sh` invocation is atomic. Ship one skill at a time. For multi-skill releases, ship each sequentially; the collection version increments once per ship.

**scripts/validate.sh:**
- New check: VERSION file exists and contains valid semver.
- New check: If latest git tag exists, VERSION >= latest tag version (no regression).
- New check: All skill versions in catalog are valid semver.
- Does NOT require VERSION to exactly match a tag (there may be unreleased commits).

**scripts/skills-deploy:**
- `install` subcommand: records `collection_version` and `collection_commit` in `~/.claude/.skills-templates.json` manifest.
- `doctor` subcommand: compares installed collection_version against current VERSION. Reports if consumer is behind. For drift detection, generates the manifest on-the-fly via `collection-version.sh manifest` and diffs template checksums against the installed state. No pre-existing manifest file required.

### collection-manifest.json schema

```json
{
  "collection_version": "1.2.0",
  "commit": "abc1234",
  "date": "2026-04-10T23:14:54Z",
  "skills": [
    {
      "name": "system-health",
      "version": "0.3.0",
      "files": ["skills/system-health/SKILL.md"]
    }
  ],
  "templates": [
    {
      "name": "doc-PRD.md",
      "sha256": "a1b2c3..."
    }
  ],
  "skill_dependencies": {
    "workflow": ["contracts"],
    "system-health": [],
    "skill-author": [],
    "contracts": []
  }
}
```

### Version flow

```
Author changes skill code
  -> runs skill-version.sh (bumps skill version)
  -> runs skill-ship.sh (commits, tags skill, auto-bumps collection patch)
     -> collection-version.sh bump patch
     -> git commit (VERSION bump)
     -> git tag v1.0.1 (collection tag, in addition to skill tag)
     -> commit message includes both versions

Consumer:
  -> git pull
  -> cat VERSION (see current collection version)
  -> skills-deploy install (records collection version in manifest)
  -> skills-deploy doctor (regenerates manifest on-the-fly, compares local state)
  -> validate.sh (checks VERSION consistency)
```

### Interaction with TODOS.md

- "Template version tracking (P3, S)" — **RETIRE.** Collection version covers templates. Checksums in manifest provide per-template tracking if needed.
- "Batch version mode for multi-skill commits (P3, S)" — **SIMPLIFIED.** The collection version already batches. A multi-skill commit just bumps collection patch once.

### Starting version

Start at `1.0.0`. The repo has multiple active skills, a working deploy pipeline, and established patterns. It's past 0.x territory.

### Decided

- **Template-only changes:** Require a manual `collection-version.sh bump patch`. Automated detection (template-ship.sh or skill-ship.sh detecting template changes) is deferred until it becomes friction.
- **collection-manifest.json:** Regenerable on-demand, never stored. Attach to GitHub Releases when/if that feature is added later.
- **CHANGELOG.md format:** Keep-a-changelog (## [version] - date, ### Added/Changed/Fixed).
- **Manifest dependency scope:** `skill_dependencies` in the manifest tracks skill-to-skill dependencies only. Tool dependencies (jq, git, waza) are tracked in skills-catalog.json and are not duplicated in the manifest.

## Open Questions

1. None. All questions resolved during design.

## Success Criteria

- A consumer can run `cat VERSION` and get the collection version.
- A consumer can run `skills-deploy doctor` and see their installed version vs. current.
- An author can run `skill-ship.sh <name>` and the collection version auto-bumps.
- `validate.sh` catches VERSION file issues (missing, invalid, regression).
- `collection-version.sh manifest` can regenerate the manifest for any commit with accurate checksums.
- Templates are tracked by checksum in the manifest, enabling future drift detection.

## Distribution Plan

Git clone/pull remains the distribution channel. No package manager needed.
skills-deploy install is the install command. It records collection_version in its manifest.
No CI/CD changes needed for this feature. Existing pre-commit hooks via setup-hooks.sh continue to work.
