## Problem Statement

S000004 shipped the `AI_KNOWLEDGE_DIR` env-var resolution block. `/company-workflow`
now knows **where** the engineer's knowledge folder lives. It does nothing with the
contents.

S000005 (reframed) ships **always-on** loading only: categories marked
`surface: always` have every `.md` file injected into Claude's context on every
skill invocation. Used for house-style guidance (e.g. cpp error-handling
conventions) the engineer wants applied by default, without copy-paste.

**On-demand** surfacing — categories with a `triggers` list loading when the user's
latest message mentions a trigger — is **deferred to a new follow-up story**
(S000006 once renamed; see §Deferred to Follow-Up). Deferral rationale:

- /autoplan CEO dual-voice review (2026-04-21) returned 5/6 CONFIRMED-NO on the
  combined plan. Both voices independently recommended shipping always-on first.
- No documented user incident shows on-demand trigger matching was needed. The
  always-on use case has a clear scenario (house-style guidance); on-demand is
  anticipatory.
- On-demand carries 60% of the implementation complexity (trigger parser,
  tokenization rules, phrase semantics, case-insensitivity, match log,
  Coverage-Gap Unicode edge cases, 50KB threshold) for speculative value.
- Anthropic may ship native knowledge-base support; a custom on-demand matcher
  risks becoming migration-hostile dead code.

This design doc captures **how** to execute the reduced T000006 scope — always-on
loading + per-repo opt-in gate — the sequencing, the commit boundaries, and the
tripwires that prove the contract is real.

## What Makes This Cool

**House-style knowledge shows up without the engineer copy-pasting.** Two unlocks
in one ship:

1. An engineer drops `coding/.knowledge.yml { surface: always }` + a cpp style guide
   under `$AI_KNOWLEDGE_DIR`. From that moment on, every `/company-workflow`
   invocation in an opted-in repo includes that style in Claude's context. Style
   violations drop to near-zero without the engineer ever asking.
2. The per-repo opt-in marker (`.claude/knowledge-enabled`) means the engineer's
   global `$AI_KNOWLEDGE_DIR` pointed at Company A's knowledge folder doesn't leak
   into Company B's repo. Cross-context contamination is architecturally impossible,
   not just policy.

The implementation move: bash does discovery + enumeration (deterministic,
testable); skill emits paths under `## Always-On Knowledge`; Claude Reads them
before answering. No prompt-length blowup from `cat`-ing files into the preamble.
Same helper layer (yml parser + category enumerator + md file lister) is
intentionally general enough to host the deferred on-demand path later with zero
refactoring.

## Constraints

- **S000004 must stay on `main`.** Resolution block is the input. It's there (PR #38).
  No retouching in this PR.
- **Zero regression.** `/company-workflow validate` on existing `fixtures/valid-feature-dir/`
  must produce byte-identical output with env unset vs set-to-empty-dir.
- **No new system deps.** Bash + coreutils (`find`, `sort`, `grep`) only. No `yq`.
- **No committed fixtures under `skills/`.** The knowledge dir is user-owned
  (external by design). Test fixtures materialize in `mktemp -d` via a shared
  helper at `scripts/test-helpers/knowledge.sh`.
- **Per-repo opt-in gate is P0.** The marker (`.claude/knowledge-enabled` in repo
  root) must be present for ANY loading to activate. This is a security control,
  not a preference — Codex outside-voice F2 from 2026-04-18.

## Premises (reduced scope)

Inherited from S000005_ARCHITECTURE.md, trimmed to what's needed for always-on
only. Dropped premises (on-demand matching, phrase semantics, trigger tokenization,
50KB threshold, match log format) move with c3 to the follow-up story.

1. **Inline bash `grep` yml parser, not `yq`.** For the reduced scope, the
   supported subset is only `surface: always | on-demand`. No `triggers` parsing
   needed in v1 — the field is IGNORED (categories with `surface: on-demand`
   are skipped entirely, same as categories with no yml). Out-of-subset keys →
   malformed-yml warning + skip. WORKFLOW.md documents the subset explicitly.
2. **Skill emits paths; Claude Reads them.** Not `cat`-inline into the preamble.
   Keeps preamble small; uses Claude's native Read tool; lets Claude paginate huge
   files if needed. The contract is the `## Always-On Knowledge` section. E2E
   canary tests prove Claude performs the Reads.
3. **`.claude/knowledge-enabled` marker gates always-on loading.** Missing marker
   → no `## Always-On Knowledge` emitted. Gate check runs first in the Knowledge
   Loading section. Prevents cross-context contamination.
4. **Missing `.knowledge.yml` is silent skip.** A category with no yml is "in
   progress" from the user's POV. Warnings are for broken states, not unconfigured
   ones.
5. **Malformed `.knowledge.yml` is loud skip.** One stderr warning line naming
   the file; sibling categories unaffected; exit 0.
6. **Load order is deterministic.** Categories lex-sorted; files within a category
   lex-sorted by relative path. Re-runs produce byte-identical output given the
   same filesystem state.
7. **`surface: on-demand` is recognized but inert in v1.** Categories with
   `surface: on-demand` parse cleanly (no malformed warning), but emit no content.
   This preserves forward compatibility: users can author on-demand yml files
   today, and when the c3 follow-up ships, those files activate automatically
   without edits.

Deferred premises (move to c3 follow-up):
- "Claude does on-demand matching" — full rule set (case-insensitive whole-word,
  phrase at token boundaries, load-all-matched)
- Match log format (`[knowledge] matched: <cat> via <trigger>`)
- 50KB soft-warn threshold — not calibrated against real data; CEO review flagged
  as theater
- "Latest user message only" scope decision
- On-demand opt-in gate contract details

## Approaches Considered (reduced scope)

The original approach analysis compared 3 execution sequencings for the combined
c1+c2+c3 plan. After the CEO-phase reframe, the decision surface is narrower —
c3 is deferred, so the execution question is simply "how to sequence c1+c2."

### Approach A: Single commit (helpers + always-on + gate + docs + tests all in one)

Ship all of c1+c2 in a single commit inside one PR.

- Effort: ~200 lines bash + tests
- Risk: Low
- Pros: Simplest review narrative; smaller PR so single-commit is fine
- Cons: No bisectable boundary between "helpers" and "loading"; drift risk
  between the helpers section and the loading section is harder to pin down in
  review

### Approach B: Two bisectable commits (c1 then c2) — chosen

Same scope as A, sequenced as **2 bisectable commits inside one PR**:

- **c1: helpers** — `scripts/test-helpers/knowledge.sh` + SKILL.md
  `## Knowledge Helpers` section. No user-visible behavior; just infra.
- **c2: always-on + opt-in gate + docs + tests** — SKILL.md `## Knowledge Loading`
  (with opt-in marker check inline) + WORKFLOW.md `## Knowledge Configuration`
  (always-on schema + security callout + opt-in rationale) + tests (A1–A10
  + E1–E4 + G1 + partial G3 from TEST-SPEC).

- Effort: same as A (~200 lines) + ~3 min of commit-message work
- Risk: Low
- Pros: Reviewer can verify "helpers pass tests as infra" before seeing them
  consumed; post-merge `git bisect` can isolate breakage to helpers vs loading
  logic; if the reviewer wants to stop at c1 and merge it alone (infrastructure
  commit, very hard to misreview), they can
- Cons: Marginal discipline overhead

### Approach C: Canary-driven TDD (dropped)

Same as the original Approach B — write failing tests first, then impl. Previously
considered for the full 3-commit plan. At reduced scope (~200 lines), the TDD
overhead isn't worth the discipline boost. TEST-SPEC rows A1–A10 + E1–E4 are
mechanical enough that writing them after impl is fine.

## Recommended Approach

**Approach B (2 commits).** At reduced scope, two commits is the right discipline
level — preserves a bisectable boundary between infrastructure and behavior
without splitting so finely that each commit reads as incomplete. Ship as one
atomic PR so the reviewer can verify docs + tests cover the shipped behavior.

### c2 — always-on loading + opt-in gate + docs + tests

1. Add `## Knowledge Loading` section to SKILL.md after `## Knowledge Helpers`.

     activation in a monorepo subdir of an un-opted-in parent.

     `[ -f "$_REPO_ROOT/.claude/knowledge-enabled" ]` — regular file only.
     Fail closed on symlinks, directories, and anything else. If absent, emit
     nothing and exit the section.
   - **Env var fail-closed** (eng-review HIGH): if `$_KNOWLEDGE_DIR` is empty
     (S000004 already set it that way for unset/invalid/non-dir), emit nothing
     and exit. No additional warnings beyond S000004's.
   - **One-shot disable** (dx-review CRITICAL — escape hatch):
     if `$AI_KNOWLEDGE_DISABLE` is set (any non-empty value), emit nothing and
     exit. Takes precedence over marker state. Lets developers debug bad
     knowledge files without `rm`-ing the committed marker.
   - **Helpful-silence-no-more** (dx-review HIGH): when `$_KNOWLEDGE_DIR` is
     valid AND at least one category has `surface: always` yml AND
     `.claude/knowledge-enabled` is missing, emit exactly one stderr line:

     silent fail trains users to distrust the feature.
