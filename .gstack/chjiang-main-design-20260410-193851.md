## Problem Statement

This repo has 9 skills (7 in scope for consolidation, 2 unchanged) and 16+ templates that work together as a dev workflow pipeline, but they're scattered across 7 skill directories with duplicated templates, no composition story, and no install mechanism for other repos. A team lead who wants to standardize their team's dev workflow has to understand 7 separate skills, figure out which order they run in, and manually reconcile templates that exist in two places (`templates/` and inside `skills/align-feature-contract/`). The goal: consolidate the 7 workflow + contract skills into 2 clean skills that any repo can install as a Claude Code plugin with one command.

## What Makes This Cool

GitHub templates suggest. This enforces. A team installs one plugin, types `/workflow track`, and gets a perfectly scaffolded feature with PRD + ARCHITECTURE + TEST-SPEC, all generated from canonical templates. When they type `/workflow review`, Claude checks the diff against the doc contracts automatically. When they type `/workflow ship`, it refuses unless test evidence is attached. The pipeline IS the workflow. Templates are the source of truth. Claude is the enforcer. And because templates are pure Markdown, the whole system is portable to other AI tools later.

## Constraints

- Must work as a Claude Code plugin (`.claude-plugin/plugin.json` with `skills/` auto-discovery)
- Templates stay pure Markdown (no Claude-specific syntax) for future multi-tool portability
- SKILL.md files are Claude-native (allowed-tools, frontmatter) — export to other tools is a future concern
- No external dependencies beyond POSIX + grep/sed/awk + jq
- Git clone is the install/update mechanism
- Must be usable by one person before it's useful to a team
- skill-author and system-health are OUT OF SCOPE — they stay as-is

## Premises

1. The core value is the PIPELINE (track -> implement -> review -> ship), not individual skills. A team installs the pipeline as a unit.
2. Templates live in ONE canonical location (`templates/`). Skills reference templates, they don't carry their own copies. The duplicated templates inside `skills/align-feature-contract/` get removed.
3. align-feature-contract and test-align-contract fold into a single `/contracts` skill. It runs as quality gates within the pipeline AND as standalone validation.
4. Distribution via Claude Code plugin for now. Templates stay pure Markdown for future portability to Copilot, Cursor, etc.
5. Documentation (README with workflow examples) is the shipping gate, not code completeness.

## Cross-Model Perspective

Codex proposed collapsing all 9 skills into 3 and identified the key insight: the user's real excitement is "closing the loop from specification to enforcement to executable confidence" (the auto-generated tests vision), not packaging skills. Codex suggested Copier for distribution (disagreed — Claude plugin format is simpler for the audience) and proposed an install/run/validate command structure. The "observable and replayable" idea — each phase emitting structured artifacts — is worth carrying forward as it becomes the bridge to multi-tool support.

## Approaches Considered

### Approach A: Single Workflow Skill
Collapse all 7 into one `/workflow` skill with subcommands (track, implement, review, ship). Doc contract enforcement runs as quality gates within review and ship phases.
- Effort: M | Risk: Low | Completeness: 8/10
- Pro: Simplest mental model, one skill to learn
- Con: Large SKILL.md, can't run contract checks independently

### Approach B: Pipeline + Contracts
Collapse 7 into 2 skills: `/workflow` (track/implement/review/ship) and `/contracts` (merged align + test). Contracts runs as quality gates within the pipeline AND as standalone validation.
- Effort: M | Risk: Low | Completeness: 9/10
- Pro: Contracts usable independently, clean separation, incremental adoption
- Con: Two skills instead of one

### Approach C: Stateful Pipeline
Collapse 7 into 1 skill with hard phase gates. Each phase writes state artifacts. State machine enforced.
- Effort: L | Risk: Medium | Completeness: 9/10
- Pro: Most opinionated, best multi-tool story, observable
- Con: State machine is new complexity, harder to adopt incrementally

## Recommended Approach

**Approach B: Pipeline + Contracts.** Clean separation of concerns. Teams can adopt contracts first (just template enforcement) and add the full pipeline later. The contracts skill is useful standalone for any repo that already has doc triplets.

### Skill Architecture

**Skill 1: `/workflow`**
- Replaces: work, work-track, work-implement, work-review, work-ship (5 skills)
- Subcommands: `track`, `implement`, `review`, `ship`
- allowed-tools: `Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `AskUserQuestion`, `Agent`
- The main SKILL.md handles routing (branch detection, subcommand dispatch) and shared conventions. Each subcommand's full logic lives in a supporting file within the skill directory: `track.md`, `implement.md`, `review.md`, `ship.md`. SKILL.md references these with instructions like "Read skills/workflow/track.md for the track phase workflow." This follows the same pattern as align-feature-contract (which already has 8 files in its directory).
- The router logic (current `/work` skill) becomes the entry point that detects the current branch and suggests the right subcommand
- Templates are referenced from the canonical `templates/` directory, not embedded

**Track subcommand mapping** (work-track has the most subcommands):

| User types | Maps to | Source |
|---|---|---|
| `/workflow track` | Evidence synthesis for current work item | work-track default |
| `/workflow track create` | Scaffold new work item + doc triplet | work-track create |
| `/workflow track journal` | Add manual journal entry | work-track journal |
| `/workflow track milestones` | CRUD milestone entries | work-track milestones |
| `/workflow track list` | List all work items with status | work-track list |
| `/workflow track close` | Close a work item | work-track close |
| `/workflow track scrum` | Generate scrum notes | work-track scrum |
| `/workflow track child-items` | Create sub-tasks under feature | work-track child-items |

The implement, review, and ship subcommands have no sub-subcommands (they each run a single workflow).

**Skill 2: `/contracts`**
- Replaces: align-feature-contract, test-align-contract (2 skills)
- Subcommands: `check` (template alignment + cross-doc traceability), `test` (Tier 1 smoke + Tier 2 e2e)
- allowed-tools: `Bash`, `Read`, `Glob`, `Grep`, `AskUserQuestion`
- The 3 enforcement template files and 3 generation guides move to `templates/` (renamed: `PRD-TEMPLATE.md` -> `contract-PRD.md`, etc.). Legacy template name resolution (`PRD-TEMPLATE.md` fallback) is dropped — the consolidated skill uses only the new `contract-*` names.
- Pipeline integration: `/workflow review` and `/workflow ship` invoke contracts checks as quality gates
- Standalone mode: `/contracts check docs/my-feature/` works without the pipeline

**Unchanged:**
- skill-author — standalone skill authoring tool
- system-health — standalone health dashboard

### Migration Path

Migration is done on a feature branch with intermediate commits. The final merged state must pass all validation.

1. Create `skills/workflow/SKILL.md` (+ track.md, implement.md, review.md, ship.md) by merging content from work, work-track, work-implement, work-review, work-ship
2. Create `skills/workflow/DESIGN.md` (reference this design doc) and `skills/workflow/CHANGELOG.md` (initial 0.1.0 entry)
3. Create `skills/contracts/SKILL.md` by merging align-feature-contract and test-align-contract
4. Create `skills/contracts/DESIGN.md` and `skills/contracts/CHANGELOG.md` (initial 0.1.0 entry)
5. Move template files from `skills/align-feature-contract/` to `templates/` (renaming PRD-TEMPLATE.md -> contract-PRD.md, etc.)
6. Update all internal references (template paths, skill names in routing)
7. Update validation scripts:
   - `validate.sh`: remove the "Template sync drift" warning check, update orphan-directory check to expect new skill dirs
   - `test.sh`: update deps.sh expected output to reflect new skill names
   - `deps.sh`: update dependency references
8. Update `skills-catalog.json` to contain only 4 entries (workflow, contracts, skill-author, system-health)
9. Remove old skill directories (work, work-track, work-implement, work-review, work-ship, align-feature-contract, test-align-contract)
10. Update CLAUDE.md routing rules
11. Rewrite README.md with the "one-command install" story and workflow examples
12. **Final gate: run `./scripts/validate.sh` && `./scripts/test.sh` — full suite must pass before merging.**

### Template Reference Convention

Skills reference templates by relative path from repo root:
```
Read templates/doc-PRD.md for the PRD scaffolding template.
Read templates/contract-PRD.md for the canonical PRD enforcement format.
Read templates/prd-GENERATION-GUIDE.md for generation rules.
```

This eliminates the duplication problem. One source of truth, multiple consumers.

### Pipeline Integration Points

The `/workflow` skill invokes `/contracts` at two points:
- **During review:** `contracts check` validates doc triplet alignment before code review proceeds
- **During ship:** `contracts check` + `contracts test` must pass before shipping is allowed

**Invocation mechanism:** The workflow's review.md and ship.md files include inline instructions that tell Claude to read the contracts skill and execute checks. Concrete example from review.md:

```markdown
## Quality Gate: Doc Contract Check

Before proceeding with code review, validate the doc triplet:

1. Read `skills/contracts/SKILL.md` for the contract check workflow.
2. Run the "check" subcommand against the current work item's doc directory.
3. If all checks pass, proceed to code review.
4. If checks fail, follow the soft gate behavior below.
```

The contracts skill runs under the **workflow skill's** allowed-tools (not its own frontmatter). This is because Claude reads the file as reference material within the workflow context, not as a separate skill invocation. If the contracts SKILL.md changes section headers, the workflow still works because it reads the entire file, not specific anchors.

**Failure modes (two distinct cases):**

- **(a) Contracts skill not installed** (file missing or unreadable): Claude warns "Contracts skill not found at skills/contracts/SKILL.md, skipping quality gate" and proceeds without prompting. No override needed.
- **(b) Contracts check ran and found failures:** Claude outputs "WARNING: Doc contracts failed: [specific failures listed]. These issues should be fixed before shipping. Override and ship anyway?" If the user says yes, shipping continues and the override is logged as a journal entry in the work item tracker: `[OVERRIDE] Shipped with contract failures: [list]`. This preserves an audit trail for the team.

### Install Story (for consuming repos)

**Primary method (git clone):**
```bash
cd my-project
git clone https://github.com/jcl2018/claude-skills-templates.git .claude/plugins/workflow-pipeline
```
This places the plugin under `.claude/plugins/` where Claude Code auto-discovers plugin manifests. The `.claude-plugin/plugin.json` in the cloned repo tells Claude where to find the `skills/` directory.

**Alternative (if/when Claude Code plugin marketplace supports it):**
```bash
claude /plugin add github:jcl2018/claude-skills-templates
```
Note: The `/plugin add` command may not be available in all Claude Code versions. Git clone is the guaranteed install path.

**Post-install:** The team adds routing rules to their project's CLAUDE.md (manually or via a setup script in `scripts/`). Example routing rules are provided in the README.

## Resolved Design Decisions

1. **Template naming:** `doc-*` files are scaffolding templates (used by `/workflow track` to create new documents). `contract-*` files are enforcement templates (used by `/contracts` to validate existing documents). Previously `PRD-TEMPLATE.md`, now renamed `contract-PRD.md` for clarity. Two different purposes, two clear prefixes.
2. **SKILL.md splitting:** The workflow skill uses supporting .md files in its directory: `SKILL.md` (routing + shared conventions), `track.md`, `implement.md`, `review.md`, `ship.md`. This follows the existing pattern in align-feature-contract (8 files in one skill dir). Claude Code reads all .md files in a skill directory.
3. **Soft gates in v1:** Contract checks warn and offer override with audit trail logging. Hard gates deferred to v2 based on team feedback.

## Reviewer Concerns

These were flagged across 3 rounds of adversarial review and could not be fully resolved at design time. They need to be validated during implementation.

1. **Plugin auto-discovery mechanism:** The install story assumes `git clone` into `.claude/plugins/` and that Claude Code auto-discovers plugin manifests there. The current `plugin.json` is a bare metadata file with no `skills` path configuration. This needs to be tested against actual Claude Code behavior during implementation. If `.claude/plugins/` is not a supported auto-discovery path, the install story must change to direct `skills/` directory installation.

2. **Cross-skill invocation pattern:** The design uses a "read and execute" pattern (workflow reads contracts SKILL.md as reference material) rather than the `Skill` tool invocation pattern (which work-ship currently uses for gstack's `/ship`). The "read" pattern was chosen because `/contracts` is a local skill in the same plugin, not an externally installed gstack skill. However, this means contracts runs under the workflow's allowed-tools, not its own. This is intentional (simpler, no tool dispatch overhead) but should be validated that it produces correct behavior during implementation.

## Open Questions

1. **Backward compatibility:** Teams already using the current 7-skill structure. Should the old skill names (/work-track, /work-review, etc.) continue to work as aliases, or is a clean break acceptable? This repo is pre-v1 with no known external consumers, so clean break is likely fine.

## Success Criteria

- 7 skills collapsed into 2 with no loss of workflow functionality
- Templates in one canonical `templates/` directory with zero duplication
- A team can install the plugin into a fresh repo and run `/workflow track` to create their first feature with scaffolded docs
- README shows the complete track -> implement -> review -> ship workflow with examples
- All existing validation scripts pass after consolidation
- `skills-catalog.json` reflects the new 4-skill structure
