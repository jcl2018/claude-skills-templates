## Problem Statement

`/company-workflow` will soon honor an external knowledge folder (`AI_KNOWLEDGE_DIR`)
via two bash blocks in its SKILL.md: Knowledge Resolution (shipped in S000004, PR #38)
and Knowledge Loading (in-flight as S000005). `/personal-workflow` — the sibling
validation skill — has no equivalent. Engineers who run both skills across work +
personal repos hit an asymmetry cliff: the same env var and the same `.knowledge.yml`
schema produces rich context in one skill and silently does nothing in the other.

S000006 closes that gap via a mechanical parity port. This design doc captures **how
to execute that port** — the sequencing, the file changes, the test strategy, the
drift containment, and what's deliberately deferred.

## What Makes This Cool

Zero new design surface. No new env var, no new schema, no new opt-in marker, no new
concepts. All the hard decisions (sanitization contract, two-tier surfacing, per-repo
opt-in marker, trigger semantics, deterministic load order, malformed-yml resilience)
were fought through in S000004 + S000005 and survived `/plan-eng-review` and the
Codex outside voice. S000006 gets to inherit every one of those decisions for free.

The interesting part isn't *what* gets copied — it's the one thing this design adds
on top of copy-paste: a CI-level drift tripwire that treats the two skills' duplicated
bash as a single logical artifact. Duplicate code stops being a maintenance tax the
moment divergence is mechanically impossible.

## Constraints

- **S000005 must land on `main` first.** The Loading block's shape (parser, opt-in
  marker filename, diagnostic format) is still being finalized. Porting against a
  moving target guarantees rework. Hard blocker on T000007's Phase 2 entry.
- **Zero regression for existing `/personal-workflow check`.** Proven style: byte-identical
  stdout when `AI_KNOWLEDGE_DIR` is unset, matching S000004's Case 11 assertion.
- **Port is additive.** `skills/company-workflow/*` must be untouched by this PR.
  Confirmed by a Tier-1 git-diff assertion.
- **Single env var + single opt-in marker.** Per F000004 design, knowledge is a
  user-workspace-level resource, not a skill-level one. Both skills read the same
  `$AI_KNOWLEDGE_DIR` and honor the same `.claude/knowledge-enabled` marker.
- **Deployment paths stay symmetric.** `skills-deploy install` already supports both
  `skills/company-workflow/` and `skills/personal-workflow/` with identical structure;
  the port doesn't require deploy-tool changes.

## Premises

The port proceeds on these agreed premises (all challenged during /office-hours):

1. **Single atomic port beats a two-PR slice.** Resolution alone delivers no user-visible
   value (it just warns); shipping Resolution+Loading together is the smallest unit of
   real user benefit. Confirmed.
2. **v1 accepts bash duplication on disk; the drift tripwire caps the risk.** Extraction
   to a shared helper is the right endgame once a Loading bug actually needs a two-skill
   fix, but not before. Meanwhile, a CI diff-check makes divergence a hard failure with
   ~20 lines of bash.
3. **Single `.claude/knowledge-enabled` marker semantics are correct.** The marker
   answers "does this repo want knowledge injected?" — a repo-level property. Users who
   want per-skill differentiation can shape it via `.knowledge.yml` `surface` modes.
4. **Block position in personal-workflow/SKILL.md: after Stale Rules Detection, before
   Overview.** Stale Rules Detection is skill-own-config hygiene (must run during
   bootstrap); Knowledge Resolution is user-workspace discovery (comes after the skill
   is known-good). Overview is documentation (comes after infrastructure resolves).
   **Note on asymmetry with company-workflow:** company-workflow places Knowledge
   Resolution directly after Path Resolution because it has no Stale Rules Detection
   section. Personal-workflow's order is: Preamble → Path Resolution → Stale Rules
   Detection → Knowledge Resolution → Knowledge Loading → Overview → Usage. The
   *logical* ordering principle ("infrastructure resolution runs before documentation")
   is identical; the *structural* neighbor of the Knowledge blocks differs because
   personal-workflow has an extra infrastructure section. The drift tripwire (Q1)
   compares the Knowledge blocks themselves, not their surrounding neighbors, so this
   asymmetry does not produce false positives.

## Approaches Considered

### Approach A: Lift-and-shift + drift tripwire (chosen)

Copy Resolution + Loading blocks from company-workflow/SKILL.md verbatim into
personal-workflow/SKILL.md. Adjust only skill-name strings. Mirror WORKFLOW.md
§Knowledge Configuration. Add T000007 test block mirroring T000003's 11 cases against
personal-workflow's SKILL.md. **Added**: a CI assertion that diffs the two skills'
Resolution + Loading blocks (modulo skill-name regex) — drift becomes a hard CI failure.

- Effort: S (~200 lines copied + ~80 lines tests + ~20 lines diff-check)
- Risk: Low
- Pros: Atomic PR; trivial review; drift mechanically blocked; no new file-layout
  concepts; SKILL.md files stay self-contained per the current skill pattern
- Cons: Bash genuinely duplicated on disk; both SKILL.md files grow in parallel
  (mitigated by the tripwire)
- Reuses: S000004's Resolution block, S000005's Loading block, S000005's
  `scripts/test-helpers/knowledge.sh` fixture builder, T000003's assertion pattern

### Approach B: Extract `scripts/knowledge.sh` now

Create a shared helper file both skills source. Each SKILL.md gets a small stub
(`source "$_REPO_ROOT/scripts/knowledge.sh" || source "$HOME/.claude/scripts/knowledge.sh"`).
WORKFLOW.md sections stay per-skill. T000003 refactors to test the helper; per-skill
tests verify presence + section emission.

- Effort: M (re-touches company-workflow/SKILL.md; no longer additive)
- Risk: Med — re-opens shipped code for review; introduces a sourcing concept the
  other skills don't use; `skills-deploy install` has to learn about a new file
- Pros: Single source of truth from day one; zero drift surface; future Layer-N
  features ship to both skills automatically
- Cons: Expands scope beyond "parity port"; asymmetric deployment (two files to ship
  where today there's one per skill); the current skill convention is self-contained
  SKILL.md — breaking that is a precedent worth making consciously, not as a side
  effect of a port
- Reuses: all of S000004 + S000005

### Approach C: Split port by layer (resolution now, loading later)

Ship resolution-only port in PR 1 (unblocked today, S000004 is frozen). Ship loading
port in PR 2 once S000005 lands.

- Effort: M (two PRs)
- Risk: Low-Med — intermediate state where `/personal-workflow` resolves env var but
  loads nothing produces support questions ("why warn and not use it?")
- Pros: Incremental; first PR is genuinely frozen code
- Cons: Two separate reviews for essentially the same bash shape; context loss
  between landings; half-ported state needs documentation; no real user value until
  PR 2 lands anyway
- Reuses: S000004 only in PR 1; S000004 + S000005 in PR 2

## Recommended Approach

**Approach A + drift tripwire.** Three reasons:

1. **Single review surface.** Reviewing the Loading block once is work; reviewing it
   twice (or worse, reviewing a near-copy in PR 2 after the first landed in PR 1)
   dilutes attention. One atomic PR keeps the review focused.
2. **Drift is the only real cost of duplication, and it's mechanically cappable.**
   A ~20-line CI diff-check turns duplication from "slow-burn maintenance tax" into
   "hard CI failure the moment someone edits one and not the other." Approach B
   eliminates drift at the cost of re-opening shipped code and breaking the
   self-contained-SKILL.md convention — that trade only makes sense once a Loading
   bug actually has to be fixed in both skills.
3. **Matches how the rest of F000004 shipped.** S000004 was one PR. S000005 will be
   one PR. S000006 being one PR makes the F000004 feature boundary clean.

## Open Questions

- **Q1: What does the drift tripwire actually check?** Proposal: a bash helper that
  extracts the **full contiguous `Knowledge-*` block** from each SKILL.md (every
  section from `## Knowledge Resolution` up to but not including the next non-Knowledge
  `##` header — this captures whatever sibling sections S000005 introduces:
  `## Knowledge Helpers`, `## Knowledge Loading`, `## On-Demand Matching`, etc.,
  without enumerating them), applies the full skill-name regex substitution set
  (see Step #6), and `diff`s the two normalized blocks. Non-zero exit = drift.
  Part of T000007.
- **Q2: Does the tripwire apply to the WORKFLOW.md §Knowledge Configuration section
  too?** Yes. Same extraction rule (full contiguous section from
  `## Knowledge Configuration` to the next `## ` header), same substitution set,
  same `diff`. Added in the same tripwire helper.
- **Q3: What happens to the tripwire when Approach B becomes justified later (say,
  F000005 or F000006)?** It gets deleted as part of the extraction PR. It's scaffolding,
  not infrastructure.
- **Q4: Does personal-workflow's `check.md` (the validation logic file) need any
  knowledge-aware changes?** Current answer: no. Knowledge is a preamble concern
  (SKILL.md), not a validation concern. Verified by a grep assertion in Step #13 —
  `check.md` must contain zero references to `AI_KNOWLEDGE_DIR` or `_KNOWLEDGE_DIR`.
- **Q5: SKILL.md version bump for personal-workflow?** 2.0.0 → 2.1.0 (semver-minor
  for additive feature — no breaking change, no new required config). Applied in
  Step #7.5.
- **Q6: Catalog / manifest updates?** `skills-catalog.json` has a `files` list and
  a `templates` list per skill entry; WORKFLOW.md content changes do not alter
  either. If a downstream hook regenerates README.md from the catalog, run
  `./scripts/generate-readme.sh` once after the edits land. Applied in Step #10.5.
