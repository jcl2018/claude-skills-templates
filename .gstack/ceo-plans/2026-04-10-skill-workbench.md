# CEO Plan: Skill Workbench

Generated by /plan-ceo-review on 2026-04-10

Repo: jcl2018/claude-skills-templates

## Vision

### 10x Check
The 10x version is a **skill development platform** where every skill ships with
evidence, not just prompts. `cargo new` for Claude skills: scaffold, write, test,
lint, publish. The skill becomes a testable artifact with structural assertions,
content quality checks, dependency graphs, and auto-generated documentation.

### Platonic Ideal
A new engineer clones this repo, runs `./scripts/create-skill.sh code-review`,
gets a full skeleton with SKILL.md, doc triplet, and catalog entry. They write
the skill, run `./scripts/test.sh`, see structural checks pass, then run
`./scripts/lint-skill.sh` to catch content issues. They run `./scripts/doctor.sh`
for a health check. They push, CI validates everything, the PR gets a green check.
Another repo installs the plugin via `claude plugin install`. The structural
evidence (valid frontmatter, passing validation, clean lint) travels with the
skill because the plugin system auto-discovers `skills/` and the catalog metadata
lives in the repo. The catalog is not consumed by the plugin system directly, but
it ensures the repo's internal quality bar is met before distribution.

## Accepted Scope — Deliverables with Acceptance Criteria

### Base scope (Full Workbench)

**1. `.claude-plugin/plugin.json`** — Static plugin manifest.
```json
{
  "name": "claude-skills-templates",
  "version": "0.1.0",
  "description": "Work lifecycle pipeline, doc contract enforcement, and skill authoring workbench",
  "author": { "name": "jcl2018" },
  "repository": "https://github.com/jcl2018/claude-skills-templates"
}
```
Acceptance: file exists, valid JSON, `name` field matches repo name. Skills in `skills/`
are auto-discovered by the plugin system (no explicit listing needed).

**2. `skills-catalog.json`** — Validation-only metadata registry (NOT distribution).
Each skill entry declares: name, version, description, source (local/upstream),
depends (object: `skills` array of skill names, `tools` array of external tools like jq),
portability (standalone/pipeline), files (paths owned by this skill), templates
(template file names in `templates/` this skill uses), status (active/experimental/deprecated).
Top-level structure: a bare JSON array of skill objects (not wrapped in an outer object).
Example entry:
```json
[
  {
    "name": "work-track",
    "version": "0.1.0",
    "description": "Work item creation and scaffolding",
    "source": "local",
    "depends": { "skills": [], "tools": ["jq"] },
    "portability": "standalone",
    "files": ["skills/work-track/SKILL.md"],
    "templates": ["doc-PRD.md", "doc-ARCHITECTURE.md", "doc-TEST-SPEC.md"],
    "status": "active"
  }
]
```
The `files` field is informational (documents which paths a skill owns). validate.sh
does not enforce uniqueness across skills — it only checks that listed files exist.
Acceptance: valid JSON, one entry per skill in `skills/`, all entries have required fields.

**3. `scripts/create-skill.sh`** — Scaffold new skill + doc triplet + catalog entry.
Usage: `./scripts/create-skill.sh my-new-skill`
Creates: `skills/my-new-skill/SKILL.md` (skeleton with frontmatter),
`docs/my-new-skill/PRD.md`, `docs/my-new-skill/ARCHITECTURE.md`,
`docs/my-new-skill/TEST-SPEC.md` (from templates), appends catalog entry.
Guards: exits 1 if skill name already exists in catalog or as directory.
Acceptance: running on a new name produces all files, catalog entry is valid JSON,
running on existing name exits 1 with error message.

**4. `scripts/validate.sh`** — Catalog-to-filesystem validation. Pure shell + jq.
Error checks (exit 1 on failure):
- Every catalog entry has a SKILL.md on disk
- Every SKILL.md has required frontmatter (name, description)
- Every skill's `templates` list has those files in `templates/`
- No orphan skill dirs (dirs under `skills/` with no catalog entry)
- Doc triplets have all three files and each has `type` frontmatter
- All `depends.skills` entries exist in the catalog
Warning checks (printed, don't affect exit code):
- Template sync drift (align-feature-contract only): sha256 checksum comparison between
  `skills/align-feature-contract/PRD-TEMPLATE.md` and `templates/doc-PRD.md` (and same
  for ARCHITECTURE and TEST-SPEC). This is specific to align-feature-contract because
  it's the only skill with internal template copies (legacy naming, kept for its
  resolution fallback logic). Not a generic check.
- Orphan doc directories under `docs/` with no matching skill
- Orphan template files in `templates/` not referenced by any catalog entry
Acceptance: exits 0 on current repo state after consolidation. Each check prints
PASS, FAIL, or WARNING.

**5. `scripts/test.sh`** — Smoke tests, superset of validate.sh.
Runs validate.sh first, then adds:
- No duplicate skill names in catalog
- All SKILL.md files have parseable frontmatter (text between `---` lines has name: and description:)
- Doc triplets have required sections: PRD.md has `## Problem Statement`,
  ARCHITECTURE.md has `## Overview` or `## Architecture`, TEST-SPEC.md has `## Test Matrix`
- `create-skill.sh` integration test: scaffold a temp skill named `__test-scaffold__`
  into `skills/`, run validate.sh on it, clean up (rm skill dir + docs dir + catalog entry).
  Cleanup runs in a `trap EXIT` to handle failures.
Acceptance: exits 0 on clean repo. Output: one line per test, OK or FAIL.

**6. `CLAUDE.md`** — Repo conventions and skill routing.
Required sections:
- Repo purpose (1 paragraph: what this repo is, who it's for)
- Quick start (`git clone`, `./scripts/validate.sh`, `./scripts/create-skill.sh test-skill`)
- Skill routing rules (map user intents to skills, e.g. "scaffold work item" -> /work-track)
- Conventions (skill directory structure, template naming with `doc-` prefix, catalog format,
  frontmatter requirements, personal-native pattern)
- Scripts reference (one-liner per script: what it does, when to run it)
Acceptance: file exists, contains all required sections, a developer can read it
and create a new skill without asking questions.

**7. `.github/workflows/validate.yml`** — CI for PRs.
Triggers on: pull_request to main.
Runner: ubuntu-latest.
Dependencies: jq (install via apt-get), bash (built-in).
Steps: checkout, install jq, run `./scripts/validate.sh`, run `./scripts/test.sh`.
Note: `lint-skill.sh` and `doctor.sh` are advisory tools (exit 0 always). They are
NOT run in CI — they are developer aids, not gates. The PR checklist in CONTRIBUTING.md
should say "lint clean (recommended)" not "lint clean (required)".
Acceptance: validates cleanly on a green repo, fails on a broken repo.

**8. `CONTRIBUTING.md`** — Authoring guide.
Required sections:
- How to create a new skill (step-by-step using create-skill.sh)
- Skill directory structure and naming conventions
- Template usage (which templates exist, when to use each)
- Running validation locally before pushing
- PR checklist (validate passes, test passes, lint clean)
Acceptance: a new contributor can follow the guide end-to-end.

**9. `scripts/sync-upstream.sh`** — Pull gstack skill updates. **Local-only developer tool.**
Mechanism: reads catalog entries with `source: "upstream"`, compares the skill's
SKILL.md against the source at `~/.claude/skills/gstack/{skill-name}/SKILL.md`,
reports diff. Does NOT auto-overwrite. Prints which upstream skills have changed
and offers a `--apply` flag to copy the updated versions.
If the upstream path does not exist (e.g., gstack not installed), prints a warning
and skips that skill. Does not exit 1 for missing upstream.
**Not run in CI** — requires local gstack installation.
Acceptance: detects when an upstream skill has changed, shows the diff,
applies changes only with explicit `--apply` flag. Gracefully skips missing upstreams.

**10. Pre-commit hook + `scripts/setup-hooks.sh`** — Plain shell, no external tooling.
`setup-hooks.sh` copies a pre-commit script to `.git/hooks/pre-commit` and makes it
executable. The pre-commit script runs `./scripts/validate.sh`.
Usage: `./scripts/setup-hooks.sh`
Acceptance: after running setup-hooks.sh, `.git/hooks/pre-commit` exists and is
executable. Committing with a broken catalog entry is blocked.

**11. Template consolidation** — Move `spec/templates/` content to `templates/`.
File mapping:
- `spec/templates/doc-PRD.md` -> `templates/doc-PRD.md`
- `spec/templates/doc-ARCHITECTURE.md` -> `templates/doc-ARCHITECTURE.md`
- `spec/templates/doc-TEST-SPEC.md` -> `templates/doc-TEST-SPEC.md`
- `spec/templates/doc-RCA.md` -> `templates/doc-RCA.md`
- `spec/templates/doc-milestones.md` -> `templates/doc-milestones.md`
- `spec/templates/doc-review-notes.md` -> `templates/doc-review-notes.md`
- `spec/templates/doc-scrum.md` -> `templates/doc-scrum.md`
- `spec/templates/doc-test-plan.md` -> `templates/doc-test-plan.md`
- `spec/templates/tracker-defect.md` -> `templates/tracker-defect.md`
- `spec/templates/tracker-feature.md` -> `templates/tracker-feature.md`
- `spec/templates/tracker-review.md` -> `templates/tracker-review.md`
- `spec/templates/tracker-task.md` -> `templates/tracker-task.md`
- `spec/templates/tracker-user-story.md` -> `templates/tracker-user-story.md`
Existing files in `templates/` with old naming (PRD-TEMPLATE.md, etc.): keep the
`spec/templates/doc-*.md` version as canonical. Remove old-named files from `templates/`
(they are superseded). If content differs, the `spec/templates/` version wins.
`spec/templates/` directory is removed after migration.
Acceptance: `templates/` has all template files with `doc-` or `tracker-` prefix.
`spec/templates/` no longer exists. validate.sh passes.

### Expansions

**12. `scripts/doctor.sh`** — Skill health diagnostics.
Checks: SKILL.md descriptions longer than 1 line (won't render well in /help),
skills missing allowed-tools frontmatter (security risk), template references
to nonexistent files, catalog entries where version is 0.1.0 but SKILL.md has
been modified since last tag/commit (uses `git log --format=%H -- skills/{name}/SKILL.md`;
skipped if not in a git repo).
Acceptance: runs on repo, reports issues with severity (warning/error), exits 0.

**13. `scripts/lint-skill.sh`** — Content-level skill linting.
Checks: missing recommended sections (## Error Handling, ## Usage), overly vague
descriptions (< 10 words or generic phrases), SKILL.md over 500 lines, bash code
blocks longer than 5 lines that lack `set -e` or `|| exit` (simple heuristic,
may have false positives on example snippets), cross-skill references to names
not in catalog.
Accepts optional skill name argument to lint one skill, or lints all if no arg.
Acceptance: lints all skills, reports issues per skill, exits 0 (lint is advisory).

**14. `scripts/deps.sh`** — Dependency graph visualization.
Reads `depends.skills` field from skills-catalog.json (already defined in catalog
schema above). Prints ASCII tree showing skill dependency chains. Detects and
reports circular dependencies. Shows standalone vs pipeline classification.
Example output:
```
work (pipeline)
  -> work-track (pipeline)
  -> work-implement (pipeline)
  -> work-review (pipeline)
  -> work-ship (pipeline)
align-feature-contract (standalone)
system-health (standalone)
test-align-contract (standalone)
```
Cycle detection: best-effort via depth limit (warn if dependency chain exceeds 10 levels).
Full topological sort in bash is complex; depth-limit heuristic catches real cycles
without requiring DFS implementation. Upgrade to proper topo sort later if needed.
Acceptance: produces readable tree, correctly identifies standalone vs pipeline,
warns on suspected circular dependencies (depth > 10). Note: short cycles (A->B->A)
may not be caught by depth heuristic alone. Upgrade to proper DFS if this becomes
a real problem.

**15. `scripts/generate-readme.sh`** — Auto-generate README.md from catalog.
Reads skills-catalog.json, produces README.md with:
- Repo title and description
- Skills table (name, description, status, portability, version)
- Quick-start section (clone, validate, create-skill)
- Installation instructions (plugin install + git clone)
- Link to CONTRIBUTING.md
Do not include timestamps or generation metadata in output (breaks idempotency).
Acceptance: generates valid markdown, table includes all catalog skills,
re-running produces identical output (idempotent).

## Key Corrections from Design Doc
- **plugin.json location:** `.claude-plugin/plugin.json` (not root level)
- **plugin.json format:** `skills` field is a directory path, not a list. Default
  `skills/` is auto-discovered. No need for explicit listing.
- **generate-plugin.sh:** ELIMINATED. Plugin manifest is static, not generated.
- **Catalog role:** Validation metadata only, not distribution infrastructure.
  The plugin system consumes `skills/` directly. The catalog ensures internal
  quality (valid metadata, dependency tracking, template references) but is not
  read by the plugin system itself.

## Script Exit Code Policy
Scripts exit 1 only when the repo is structurally invalid (validate.sh, test.sh).
Content-quality and advisory tools (doctor.sh, lint-skill.sh, deps.sh) always exit 0.
generate-readme.sh exits 1 only on write failure.

## Content Ownership
- **README.md** is auto-generated by `generate-readme.sh` — canonical source for
  quick-start and installation instructions. Never hand-edit.
- **CLAUDE.md** is hand-maintained — canonical source for repo conventions, skill
  routing, and developer guidance. Links to README for quick-start.
- **CONTRIBUTING.md** is hand-maintained — canonical source for authoring workflow.
  Links to README for installation.

## Reviewer Concerns (unresolved after 3 iterations)
1. **test.sh catalog cleanup** requires a jq filter to delete an array element by name.
   Non-trivial but achievable: `jq '[.[] | select(.name != "__test-scaffold__")]'`.
2. **sync-upstream.sh path not configurable** — `~/.claude/skills/gstack/` is hardcoded.
   Acceptable for v1. Add GSTACK_SKILLS_PATH env var override if a second user needs it.
3. **CONTRIBUTING.md PR checklist** should explicitly say "lint clean (recommended, not required)"
   to match the CI policy that lint is advisory-only.
4. **CLAUDE.md skill routing rules** — derive the full mapping from the current skill set at
   implementation time. Don't attempt to enumerate all intents in this planning document.
5. **"Personal-native pattern"** means: no $AI_CONTENT_DIR indirection, work items at
   ./work-items/ per repo, templates at ~/.claude/templates/, upstream skills via git pull.
   Define this inline in CLAUDE.md when writing it.
6. **generate-readme.sh exit on malformed catalog** — should exit 1, add to exit code policy.

## CI Prerequisites
All scripts use: bash (4.0+), jq (for JSON parsing), grep, sed, awk (POSIX).
GitHub Actions runner: ubuntu-latest (has all of these). No Node.js, Python, or
other runtimes required. Scripts must be self-contained with no npm/pip deps.
