## Vision

### 10x Check
A **work item compiler**, not just a validation tool. `/docs check` compiles the entire `work-items/` tree into a navigable model: structural completeness, completeness percentages per feature, configurable hierarchy rules, and a machine-readable graph artifact that any downstream tool can consume. The tree report becomes the visual interface, the graph JSON becomes the data model.

### Platonic Ideal
You open your repo. The graph artifact knows every story, every task, every dependency. When you change a PRD user story, your editor highlights which TEST-SPEC rows need updating. When you close a task, the feature's completeness meter ticks up. When all tasks under all stories are closed, the feature auto-transitions to "Ready for Review." The work item system becomes an executable project plan. The graph artifact is the data model. An LSP extension would be the interface.

## Implementation Details

### Hierarchy schema in artifact-manifests.json

```json
"hierarchy": {
  "feature": {"required_child": "user-story", "min": 1},
  "user-story": {"required_child": "task", "min": 1}
}
```

Types not listed in `hierarchy` (task, defect) have no structural requirements.
If `hierarchy` field is missing from manifest: warn and skip structural checks (required field, no fallback).
If `hierarchy` field is malformed: warn and skip structural checks.

### Placement rules (from outside voice review)

- **feature**: root-level only (direct child of work-items/)
- **defect**: root-level only (direct child of work-items/)
- **user-story**: inside a feature only
- **task**: inside a user-story only

Items at wrong placement level are flagged as MISPLACED.

### Badge taxonomy mapping (from outside voice review)

Existing check statuses map to badge categories:

| Badge category | Status values (severity order) |
|---------------|-------------------------------|
| template | PASS < WARN (EXTRA) < DRIFT (missing field/section) < INCOMPLETE (MISSING artifact) |
| lifecycle | PASS < WARN < LIFECYCLE_INCONSISTENT |
| traceability | PASS < INFO (P1/P2 untested) < UNTESTED (P0 untested) |
| structure | PASS < INCOMPLETE < MISPLACED |

Each badge shows the worst (highest severity) status from its category for that node.
For /docs tree (structural badges only): template/lifecycle/traceability badges show "—".

### Claims.json gate fix (from outside voice review)

check.md Steps 1-5 (staleness check) require claims.json and stop if missing.
Steps 6+ (work item validation, structural checks) do NOT need claims.json.
Fix: If claims.json is missing, skip Steps 1-5 with a note ("No claims.json found, skipping staleness check") and continue to Step 6.

### Lifecycle cross-reference (from outside voice review)

If a TRACKER.md has "Broken down into child tasks/stories" checked but the item has 0 children per structural check, flag as LIFECYCLE_INCONSISTENT in Check 2's output. This prevents contradictory signals (lifecycle says "done breaking down" but structure says "no children").

### Step mapping

- **Step 15:** Structural Completeness + Orphan Detection (read hierarchy rules from manifest, count children per type, flag INCOMPLETE and MISPLACED items)
- **Step 16:** Completeness Percentage + Tree Report (calculate percentages, render depth-first tree with all 4 badges)
- **Step 17:** Graph Artifact (serialize nodes with badges + completeness to `.docs/work-item-graph.json`)

### Completeness percentage formula

`completeness = (count of children matching required_child type) / (expected minimum)` expressed as a fraction.
- Feature with 2 user-story children and min=1: "2/1 (200%, exceeds minimum)"
- Feature with 0 user-story children and min=1: "0/1 (0%)"
- For tree report, show as "2 user-story children" (count + type). For graph JSON, include count and min.

Simpler: just report the count vs minimum. The percentage is implicit.

### Orphan detection criteria

A work item is **orphaned/misplaced** if:
- Its type is not an allowed child type of its parent per the hierarchy rules
- Example: a task (T000001) directly inside a feature directory is MISPLACED because `hierarchy.feature.required_child` is `user-story`, not `task`

A directory is **stray** if:
- It exists under `work-items/` and contains `.md` files but no TRACKER.md (not a work item at all)

### Badge accumulation across checks

Steps 11-14 (existing checks 1-3) produce per-item findings. The implementer should accumulate a badges object per node as checks run. Step 17 reads this accumulated structure when writing graph JSON. This is a natural extension of the Actual Model from Step 10.
