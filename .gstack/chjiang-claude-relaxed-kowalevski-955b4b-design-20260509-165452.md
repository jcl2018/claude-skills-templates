## Problem Statement

gstack writes per-project state to `~/.gstack/projects/<slug>/` on the local machine — design docs, CEO plans, eng-review reports, /context-save checkpoints, learnings, session timelines, analytics, mockup PNGs. The user wants this output to live inside the repo so:

- Reviewers see the design intent that produced the code (PR review context)
- Plans survive machine wipes via `git pull` on a fresh clone
- Multiple machines see the same artifact history
- Design history travels with code history in `git log`

Today this isn't possible without a workaround: gstack has two relevant env vars (`GSTACK_HOME` for state, `GSTACK_PLAN_DIR` for /context-save plans), but ~30+ skill bash blocks hardcode `~/.gstack/projects/$SLUG/...` directly (e.g., /office-hours SKILL.md lines 1360, 1555, 1563), bypassing the env var. So setting `GSTACK_HOME` only partially redirects output — broken state.

## What Makes This Cool

A one-line symlink fixes a problem that would otherwise need an upstream PR to fix cleanly. Works with every gstack skill today, including the ones that hardcode `~/.gstack/...`. Reversible. Zero gstack changes. The kind of fix you're slightly annoyed you didn't think of sooner.

## Constraints

- Must work today, no upstream gstack changes required
- Must cover ALL gstack output for this project (full `~/.gstack/projects/<slug>/` tree), not just /office-hours design docs
- Operating gstack as a USER, not a maintainer — solutions requiring gstack source patches are out of scope for v1
- Existing `~/.gstack/projects/jcl2018-claude-skills-templates/` has 138 entries (~2.6 MB) of historical design docs, CEO plans, eng-review reports — must migrate cleanly without losing history
- Repo uses git worktrees — main checkout at `/Users/chjiang/Documents/projects/claude-skills-templates`, worktrees under `.claude/worktrees/<name>/`. The symlink at `~/.gstack/projects/<slug>/` can only point to ONE physical location, so the target must be the MAIN repo's `.gstack/`, not a worktree-local one

## Premises

1. Plans should be first-class repo artifacts, committed to git — confirmed (D1)
2. Scope is the full `~/.gstack/projects/<slug>/` tree (everything gstack writes for this project), not a curated subset — confirmed (D2)
3. Symlink approach beats waiting for upstream gstack patches — confirmed (D3)
4. This repo's existing `work-items/` structure (from /personal-workflow) is parallel to gstack output, not integrated. gstack output lives under `<repo>/.gstack/`; work items live under `<repo>/work-items/` — flagged for awareness, not blocking

## Approaches Considered

### Approach A: Symlink swap (chosen)

Replace `~/.gstack/projects/<slug>/` with a symlink pointing to `<main-repo>/.gstack/`. gstack continues to write to its hardcoded path; the symlink redirects writes into the repo. Add `<repo>/.gstack/` to git.

- Effort: S (~10 min one-time setup)
- Risk: Low (reversible)
- Pros: works with ALL gstack skills today; trivially reversible (`rm` symlink, `mv` files back); covers env-respecting and hardcoded paths uniformly
- Cons: fresh clone on another machine needs setup script re-run; symlinks can confuse `find` without `-L` and a few other tools
- Reuses: existing gstack-slug binary; standard rsync/ln; no new gstack code

### Approach B: Stop-hook auto-sync

Add a Claude Code Stop hook in `~/.claude/settings.json` that rsyncs `~/.gstack/projects/<slug>/` → `<main-repo>/.gstack/` after every Claude session. gstack writes to its native location; the hook mirrors after each session.

- Effort: M (hook + per-repo slug→path mapping)
- Risk: Med (silent failures, drift between two copies)
- Pros: no symlinks; gstack writes wherever it wants; sync is deterministic per session-end
- Cons: two physical copies that drift if files are edited in one location and not the other; hook can silently fail; mapping logic to know which slug → which repo
- Reuses: Claude Code hooks system; rsync

### Approach C: Upstream patch + .envrc + direnv

Submit gstack PR to honor `GSTACK_HOME` everywhere (fix all hardcoded `~/.gstack/projects/$SLUG/...` references to use `${GSTACK_HOME:-$HOME/.gstack}`). Add a `.envrc` to the repo: `export GSTACK_HOME=$(git rev-parse --show-toplevel)/.gstack`. Use direnv to auto-load.

- Effort: L (PR + review + merge cycle for ~30 SKILL.md files)
- Risk: Med (blocked on upstream merge; partial behavior in interim is broken)
- Pros: cleanest end-state, no symlinks; works on any clone with direnv; upstream fix helps every gstack user
- Cons: blocked on upstream merge — until then, output is split between repo and home; requires direnv installed per developer
- Reuses: direnv (standard tool); upstream gstack codebase

## Recommended Approach

**A: Symlink swap.** Smallest move that delivers the user's stated outcome (everything from `~/.gstack/projects/<slug>/` committed to repo) and works today across all gstack skills.

C is the right long-term answer; A is the right answer right now. Once C lands upstream, A can be removed (just rm the symlink and let `GSTACK_HOME` env var take over) — they're not mutually exclusive over time.

### Setup script

Write to `<repo>/scripts/setup-gstack-symlink.sh`. Run from the repo root: `cd <repo> && ./scripts/setup-gstack-symlink.sh`.

```bash
#!/usr/bin/env bash
# Redirects ~/.gstack/projects/<slug>/ into this repo's .gstack/ via symlink.
# Run once per repo per machine, from the repo root. Idempotent.
# Pass --force to re-point a symlink that currently targets a different repo.
set -euo pipefail

FORCE=0
[ "${1:-}" = "--force" ] && FORCE=1

# Resolve MAIN repo (not worktree). Worktrees share the same slug, so the
# symlink can only point to one physical location — main is the right home.
# --path-format=absolute returns an absolute path whether run from main or worktree.
COMMON_DIR=$(git rev-parse --path-format=absolute --git-common-dir)
MAIN_REPO=$(dirname "$COMMON_DIR")  # strips trailing /.git

# Don't swallow gstack-slug stderr — if the binary isn't installed, you want to see why.
eval "$(~/.claude/skills/gstack/bin/gstack-slug)"
: "${SLUG:?gstack-slug ran but emitted no SLUG; check ~/.claude/skills/gstack install}"

SRC="$HOME/.gstack/projects/$SLUG"
DEST="$MAIN_REPO/.gstack"

# DEST already exists with content: warn before merging
if [ -d "$DEST" ] && [ -n "$(ls -A "$DEST" 2>/dev/null)" ]; then
  if [ -d "$SRC" ] && [ ! -L "$SRC" ]; then
    echo "WARN: $DEST is non-empty AND $SRC has content to migrate."
    echo "  Files with same name in both will be OVERWRITTEN by SRC versions."
    echo "  Existing $DEST contents:"
    ls -la "$DEST" | head -20
    [ "$FORCE" = "1" ] || { echo "Re-run with --force to proceed."; exit 1; }
  fi
fi

mkdir -p "$DEST"

# Migrate existing project state into repo (if SRC is a real dir, not already symlinked)
if [ -d "$SRC" ] && [ ! -L "$SRC" ]; then
  echo "Migrating $SRC → $DEST/"
  rsync -a "$SRC/" "$DEST/"
  # Backup the original before removing, in case the user wants to revert
  mv "$SRC" "$SRC.bak.$(date +%s)"
  echo "Original backed up to $SRC.bak.<timestamp> — delete after verifying"
fi

# Idempotent symlink creation
if [ -L "$SRC" ]; then
  CURRENT=$(readlink "$SRC")
  if [ "$CURRENT" = "$DEST" ]; then
    echo "Symlink already correct: $SRC → $DEST"
    exit 0
  fi
  echo "WARN: $SRC is already a symlink to a DIFFERENT location: $CURRENT"
  echo "  Re-pointing will redirect gstack output AWAY from that other location."
  [ "$FORCE" = "1" ] || { echo "Re-run with --force to re-point."; exit 1; }
  rm "$SRC"
fi

ln -s "$DEST" "$SRC"
echo "Done. ls $SRC now shows $DEST contents."
```

### Reversal

Write to `<repo>/scripts/teardown-gstack-symlink.sh`. Run from the repo root: `cd <repo> && ./scripts/teardown-gstack-symlink.sh`.

```bash
#!/usr/bin/env bash
# Reverses setup-gstack-symlink.sh: restores ~/.gstack/projects/<slug>/ to a real directory.
# Same MAIN-repo resolution as setup, so teardown works from main or any worktree.
set -euo pipefail

COMMON_DIR=$(git rev-parse --path-format=absolute --git-common-dir)
MAIN_REPO=$(dirname "$COMMON_DIR")

eval "$(~/.claude/skills/gstack/bin/gstack-slug)"
: "${SLUG:?gstack-slug ran but emitted no SLUG}"

SRC="$HOME/.gstack/projects/$SLUG"
DEST="$MAIN_REPO/.gstack"

[ -L "$SRC" ] || { echo "$SRC is not a symlink — nothing to revert"; exit 0; }
CURRENT=$(readlink "$SRC")
[ "$CURRENT" = "$DEST" ] || { echo "ERR: $SRC points to $CURRENT, not $DEST — refusing to revert blindly"; exit 1; }

rm "$SRC"
rsync -a "$DEST/" "$SRC/"
echo "Restored $SRC from $DEST. The repo's $DEST is unchanged — delete it manually if you want to remove all traces."
```

## Open Questions

1. **Should `learnings.jsonl` be committed (shared insights for this project) or gitignored (machine-local pattern memory)?** The gstack learnings skill writes user preferences and observed patterns, which can leak machine-local context (filenames, tool paths). Recommendation: gitignore for v1; revisit if learnings prove team-shareable. The setup above gitignores them.
2. **Should historical 138 design docs be migrated, or start fresh?** Migrating commits ~2.6 MB of mostly-abandoned design exploration in the initial commit. Recommendation: migrate (full history is valuable; one-time storage cost is fine; you already have it).
3. **Will the symlink break gstack's brain-sync writes?** Brain-sync (`gstack-brain-sync`) treats `~/.gstack/.git/` as the marker for "this is a state-sync repo." That marker is at `~/.gstack/.git/`, not `~/.gstack/projects/<slug>/.git/`, so the symlink at `~/.gstack/projects/<slug>` doesn't interact. Safe. Verified by reading the preamble's brain-sync logic. (Also: `artifacts_sync_mode: off` for this user, so brain-sync isn't even active.)
4. **Parallel-design conflict with `work-items/`?** This repo already has structured design intent under `work-items/features/`, `work-items/defects/`, `work-items/user-stories/`, etc. gstack output lands at `<repo>/.gstack/` in parallel — different surface, different lifecycle. Recommendation: keep them parallel, document the split in CLAUDE.md ("`.gstack/` = lateral/exploratory; `work-items/` = structured per-feature"). Don't try to merge them; they serve different stages.
5. **Worktree behavior:** Symlink target is `<main-repo>/.gstack/`. Worktrees writing to gstack will land in main's working tree, which means `git status` from the worktree won't show the new files. Always commit gstack output from main (or `git -C <main-repo> add .gstack/`). Acceptable tradeoff — the alternative (per-worktree gstack dirs) would require switching the symlink each time you cd, which is fragile.
