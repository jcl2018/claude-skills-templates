## Problem Statement

The personal-workflow user-story test spec (`templates/personal-workflow/doc-TEST-SPEC.md`) has three overlapping test sections — `## Test Matrix` (h2), `## Test Tiers` (h2) wrapping `### Tier 1: Smoke Tests` (h3) and `### Tier 2: E2E Tests` (h3), and `## Coverage Gaps` (h2) — that say similar things in different shapes. In practice, this produces test specs that are hard for a human to scan and verify (S000012's TEST-SPEC has 12 Test Matrix rows + 5 smoke + 6 E2E = 23 entries to cross-reference). The user wants a minimum useful surface: smoke = automated regression that is never edited after writing, E2E = manual user-scenario verification done after implementation and before ship, with a soft cap of 5 rows per tier per user-story (validator advisory, not a violation — the engineer can exceed when justified).

Secondary issues this restructure must address:

- `skills/personal-workflow/examples/example-doc-TEST-SPEC.md` is already drifted from its template — it shows `## Test Strategy` + `## Test Matrix` + `## Edge Cases` instead of the current Test Matrix + Test Tiers + Coverage Gaps shape. Re-sync needed.
- `skills/personal-workflow/examples/example-doc-test-plan.md` is also drifted from its template — re-sync needed (template itself unchanged).
- `skills/personal-workflow/check.md` has TWO surfaces that depend on the section shape:
  - **Step 16 (Tier-2 Template Compliance)** derives `expected_sections` from the template and flags `[DRIFT] missing section` for any required template section absent from the instance, plus `[EXTRA]` for unexpected sections.
  - **Step 18 (Cross-Reference Traceability)** is hardwired to scan `## Test Matrix` for `AC-{n}` values to verify P0 PRD coverage.
  Both must follow the new shape, otherwise every historical TEST-SPEC.md file emits drift findings on every `/personal-workflow check` run.

## What Makes This Cool

The thing the user is trading is "complete-looking spec" for "spec a human can actually read in one screen." The 5-row soft cap is a forcing function: it makes you pick the tests that prove the story works, not the tests that demonstrate completeness. The split — automated smoke that lives forever in CI vs. manual E2E run once before ship — also collapses two cognitive modes into the right shape: you stop touching one, you sit down and run the other. The validator change keeps PRD↔test traceability automatic without the Test Matrix middle table.

## Constraints

- **Personal-workflow only.** Company-workflow has parallel templates (`templates/company-workflow/doc-TEST-SPEC.md`, `doc-test-plan.md`) that are NOT in scope here. Mirroring is a follow-up work item (see Open Questions).
- **doc-test-plan.md is OUT of scope.** The user request was about TEST-SPEC ("5 tests per story max"). The test-plan template (used by tasks and defects) stays as-is. Only the example file (`example-doc-test-plan.md`) re-syncs to its existing template, since that example is drifted.
- **No transitional drift state.** All 9 historical TEST-SPEC.md files get swept to the new shape in the same PR. The validator becomes properly strict on the new shape — no permanent "legacy mode" in `check.md`. This is the only clean-cut option that doesn't leave `[DRIFT]` findings on every `/personal-workflow check` run.
- **Soft cap, not hard.** The 5-row cap is enforced via template comment + validator advisory `[INFO]`, not a `[VIOLATION]`. The engineer retains the escape hatch when a story genuinely needs a 6th test. (See Decisions Already Made #2 — earlier draft used inconsistent "hard 10-row ceiling" language; that was wrong, soft is the right call.)
- **Workbench scope.** No downstream-consumer changes (e.g., portfolio repo).

## Premises (confirmed)

1. Test Matrix is bloat — it duplicates whatever lands in Smoke + E2E. Drop from new template.
2. Cap = 5 rows per tier, per user-story, **soft** (template comment + validator advisory `[INFO]`). 6 rows is allowed but flagged.
3. Smoke = automated regression, runnable from a CI script, never edited after writing. E2E = manual user-scenario verification, done after implementing and before `/ship`.
4. Sweep all 9 historical TEST-SPEC.md files in the same PR. Each Test Matrix row maps to Smoke or E2E based on its `Type` column (Unit/Integration → Smoke; E2E → E2E). Tag and AC values carry across. Where the row count exceeds 5+5, an editorial pass consolidates to fit.
5. Stale example files (`example-doc-TEST-SPEC.md`, `example-doc-test-plan.md`) get re-synced. The first re-syncs to the new template. The second re-syncs to the (unchanged) existing test-plan template.
6. Validator (`check.md`) gets stricter, not laxer: Step 16's required-sections list updates to match the new template; Step 18 scans `## Smoke Tests` + `## E2E Tests` for AC values; new Step 18.5 emits soft-cap advisory. No legacy fallback — the sweep makes it unnecessary.

## Approaches Considered

### Approach A: Two compact tables, Smoke (5 max) + E2E (5 max), AC column folded into both — drop Test Matrix entirely. Sweep all historical files.

`doc-TEST-SPEC.md` becomes three sections only: `## Smoke Tests`, `## E2E Tests`, `## Coverage Gaps`. Both tier tables include an AC column so PRD↔test traceability lives directly in the rows. `doc-test-plan.md` is unchanged. Validator Step 16 picks up the new required-sections list automatically (template-driven). Validator Step 18 scans `## Smoke Tests` + `## E2E Tests` for AC values. New Step 18.5 emits a soft-cap advisory. All 9 historical TEST-SPEC.md files are swept to the new shape in the same PR (mechanical Test Matrix → Smoke|E2E remap based on `Type` column). Stale `scripts/test.sh:77-83` `## Test Matrix` grep is also updated to match the new shape.

- Effort: L
- Risk: Low
- Pros: Direct match to the user's stated split (visible at a glance, no filtering); soft 5+5 cap forces consolidation without losing escape hatch; AC column preserves traceability without a third table; sweep + strict validator means no permanent legacy mode in `check.md`; no transitional `[DRIFT]` noise on `/personal-workflow check`.
- Cons: Sweep is ~9 files of editorial work, not pure mechanics — some files will need a row consolidated to fit 5+5 (S000012 currently has 6 E2E rows; others need verifying). Each row's Smoke/E2E classification is a small judgment call when `Type` is ambiguous (e.g., a "Smoke" labelled row that's really an integration test).
- Reuses: existing template-derived validation pipeline (Step 4 derives required sections from the template; Step 16 picks up new sections automatically; Step 18 only needs new parsing logic).

### Approach B: Single unified Tests table with Tier column

`doc-TEST-SPEC.md` has one `## Tests` table with columns `# | Tier (Smoke|E2E) | AC | Test Case | Steps | Expected | Priority`. Soft cap = 5 rows per Tier value (10 total). Validator Step 18 scans the Tests table.

- Effort: M
- Risk: Low
- Pros: One table to scan; closest structurally to today's Test Matrix; minimal validator-parsing change. Sweep is more mechanical (just renaming columns and filtering rows by Tier).
- Cons: Smoke vs. E2E becomes a column to filter on rather than a section header — flattens the cognitive split the user explicitly named in the request.
- Reuses: same template-derived pipeline.

### Approach C: Smoke-only TEST-SPEC; E2E moves to TRACKER Phase 3 gates

`doc-TEST-SPEC.md` becomes a pure automated-regression spec (Smoke only, 5 max). E2E becomes a Pre-Ship Verification checklist inside `tracker-user-story.md`'s Phase 3 gates.

- Effort: M-L
- Risk: Medium
- Pros: TEST-SPEC becomes a clean "never touched" automated spec; E2E lives where it actually runs (right before `/ship`).
- Cons: Bigger blast radius — `tracker-user-story.md` changes, Phase 3 gate spec changes, E2E↔AC traceability gets weaker (checklist items aren't a structured table the validator can scan).
- Reuses: existing tracker template structure.

## Recommended Approach

**Approach A.** Picked by the user. It directly executes the stated intent — Smoke and E2E as separately visible tables, AC column folded into each so traceability stays automatic, soft 5+5 cap as a forcing function with an escape hatch, smallest viable surface area in the validator. Sweeping all 9 historical files in the same PR keeps the validator strict on the new shape — no permanent legacy mode, no transitional drift noise.

## Detailed Design

### File-by-file changes

**1. `templates/personal-workflow/doc-TEST-SPEC.md` — restructure**

New `##` section list (in order): `## Smoke Tests`, `## E2E Tests`, `## Coverage Gaps`. Frontmatter unchanged. The h2 `## Test Tiers` wrapper goes away — Smoke and E2E become top-level sections instead of h3 children.

```markdown
## Smoke Tests

<!-- Automated regression. Fast, deterministic, runnable from a script or CI.
     Once written, you should not need to edit these. Soft cap: 5 rows
     (validator emits [INFO] advisory if exceeded; exceed only when justified).
     Pick the structural checks that catch real regressions, not all checks
     that could exist. AC column maps each row to a PRD acceptance criterion. -->

| # | Tag | AC | Check | What It Validates | Script/Command |
|---|-----|-----|-------|-------------------|----------------|
| S1 | {tag} | AC-{n} | {description} | {what passes/fails} | `{command}` |

## E2E Tests

<!-- Manual end-to-end verification, run after implementing and before /ship.
     You drive the feature as a real user would and observe the outcome.
     Soft cap: 5 rows (validator emits [INFO] advisory if exceeded). Each row
     should be one user-visible scenario, not one branch in the code. AC column
     maps each row to a PRD acceptance criterion. -->

| # | Tag | AC | Scenario | Steps (as a real user would) | Expected Outcome | Rubric |
|---|-----|-----|----------|------------------------------|------------------|--------|
| E1 | {tag} | AC-{n} | {description} | {step-by-step what user does} | {what user sees} | {pass/fail criteria} |

## Coverage Gaps

<!-- What is explicitly NOT tested and why. Honesty beats false confidence. -->

| Gap | Why Not Tested | Risk Accepted |
|-----|----------------|---------------|
| {untested scenario} | {reason} | {what could go wrong} |
```

Tag vocabulary stays the same: `core`, `resilience`, `observability`, `usability`, `security`, `integration`.

**5. `skills/personal-workflow/check.md` — Step 16 picks up new shape automatically**

No code change in Step 16 itself. Step 4 derives `expected_sections` from the template (`## ` headers in document order). Once `doc-TEST-SPEC.md` ships with the new headers, Step 16's required-section check uses `["Smoke Tests", "E2E Tests", "Coverage Gaps"]` automatically. The same Step 4 derivation also populates Step 12's frontmatter check.

This is why the sweep is non-negotiable: with no code change in Step 16, every legacy TEST-SPEC.md file would emit `[DRIFT] missing section "Smoke Tests"` and `[EXTRA] unexpected section "Test Matrix"`. The sweep makes those findings disappear at the source.

**6. `skills/personal-workflow/check.md` — Step 18 (Cross-Reference Traceability) — replace Test Matrix scan**

Current Step 18 (`check.md:312-321`):
```
3. Parse TEST-SPEC.md test matrix:
   - Find the ## Test Matrix table
   - Extract all AC-{n} values from the AC column
4. For each P0 story number: ...
5. For each P1/P2 story number: ...
```

New Step 18:
```
3. Parse TEST-SPEC.md for AC values (single ac_set, used by both P0 and P1/P2 loops):
   raw_acs = AC-{n} values in the AC column of ## Smoke Tests
           ∪ AC-{n} values in the AC column of ## E2E Tests

   ac_set = raw_acs minus any value matching the literal placeholder pattern
            `AC-\{n\}` or `AC-\{[a-zA-Z_]+\}` (template scaffolding, not real AC).

   No fallback. After the sweep (this PR), all TEST-SPEC.md files use the new
   shape. A file that has `## Test Matrix` instead of Smoke + E2E will fail
   Step 16's section check (correct behavior — file is non-compliant).

4. For each P0 story number from PRD.md:
   - If story_number not in ac_set:
     flag [UNTESTED] P0 story #{n} has no TEST-SPEC coverage

5. For each P1/P2 story number from PRD.md:
   - If story_number not in ac_set:
     flag [INFO] P1/P2 story #{n} has no TEST-SPEC coverage (advisory)

6. If all P0 stories have coverage: [PASS] All P0 stories have TEST-SPEC coverage
```

Edge cases stated explicitly:
- **Smoke section present, E2E section absent:** `ac_set` = AC values from Smoke only. Step 16 will already flag `[DRIFT] missing section "E2E Tests"`. Step 18 doesn't double-flag.
- **Both sections present but empty (template placeholders only):** `ac_set` = empty after the placeholder filter. All P0 stories flag `[UNTESTED]` (correct — file is unfilled).
- **Freshly-scaffolded file with template placeholder rows like `| S1 | core | AC-{n} | ... |`:** the `AC-\{n\}` placeholder is filtered out, so `ac_set` does NOT include `AC-{n}` as a literal value. Without the filter, every P0 story would falsely match the placeholder and the freshly-scaffolded file would silently pass — the filter prevents that false-positive. After scaffolding, the engineer is expected to replace `AC-{n}` with real AC numbers; until they do, all P0 stories will (correctly) flag `[UNTESTED]`.
- **One row's AC cell is `-` or blank:** that row contributes nothing to `ac_set`; not an error per Step 18 (the row may be a smoke check that doesn't map to an AC).

**7. `skills/personal-workflow/check.md` — NEW Step 18.5 (Test Cap Advisory)**

A separate step, not a sub-step of Step 18, because cap-counting is structural compliance (logically a Check 1 cousin), not traceability. Position: between Step 18 and Step 19. Output flows into the existing `[INFO]` advisory category in the Findings list (Step 23).

```
## Step 18.5: Check 3.5 — Test Cap Advisory

For each TEST-SPEC.md file in the Actual Model:

1. Find the `## Smoke Tests` heading. Count lines that match the regex
   `^\s*\|.*\|\s*$` between that heading and the next `^## ` heading or EOF.
   Subtract 2 (header row + `|---|` separator). → smoke_row_count
2. Same for `## E2E Tests` → e2e_row_count
3. If smoke_row_count > 5:
   flag [INFO] {path} — Smoke Tests has {N} rows, soft cap is 5 (per template guidance)
4. If e2e_row_count > 5:
   flag [INFO] {path} — E2E Tests has {N} rows, soft cap is 5 (per template guidance)

Files without Smoke or E2E sections (already flagged by Step 16): skip silently.
```

`[INFO]` is non-blocking: it appears in the Findings → Advisory subsection (Step 23) and does NOT affect exit code.

Badge category assignment: cap-advisory `[INFO]` is structural compliance (about file shape, not PRD↔test traceability), so it surfaces under the **template** badge category (Step 20, Check 1), NOT the traceability badge. This requires extending Step 20's template badge severity ladder to include `[INFO]`: `template: PASS < INFO (cap-advisory) < WARN (EXTRA sections) < DRIFT (missing field/section) < MISSING (required artifact absent)`. Existing template badge consumers tolerate this — `[INFO]` is below WARN so any existing WARN-or-worse takes precedence in the worst-of fold. The traceability badge ladder stays unchanged (`PASS < INFO (P1/P2 untested) < UNTESTED (P0 untested)`).

The user reading the badge sees `template: INFO` and knows: "this file's shape is fine, just slightly over the soft row cap" — clean separation from traceability. The badge taxonomy table at Step 20 (`check.md:347-352`) needs the template row updated; that's part of the Step 20 edit in the Next Steps below.

**8. Sweep all 9 historical TEST-SPEC.md files**

Per-file mechanical pass: each Test Matrix row maps to either Smoke or E2E based on its current `Type` column:

| Current `Type` value | New tier |
|-|-|
| `Unit` | Smoke |
| `Integration` | Smoke |
| `Smoke` (from current Tier 1 table) | Smoke |
| `E2E` | E2E |
| `Manual` | E2E |

For each file:
1. Take rows from `## Test Matrix` and rewrite under `## Smoke Tests` or `## E2E Tests` per the table above.
2. Re-shape columns: Smoke gets `# | Tag | AC | Check | What It Validates | Script/Command`. E2E gets `# | Tag | AC | Scenario | Steps | Expected Outcome | Rubric`.
3. If the Test Matrix already had Tag and AC columns: carry them over.
4. If the existing `## Test Tiers` h2 has Tier 1 / Tier 2 sub-tables: merge those rows in too (deduping against Test Matrix rows that overlap).
5. If the resulting Smoke or E2E count > 5: consolidate. Drop low-priority rows or merge two narrow tests into one broader one. Don't widen scope just to fit.
6. **Delete the original headings:** remove `## Test Matrix` (h2 + table), and remove `## Test Tiers` (h2 wrapper + `### Tier 1` / `### Tier 2` h3 children + their tables). The final file has exactly three `##` headers: `## Smoke Tests`, `## E2E Tests`, `## Coverage Gaps`. (Without this step, Step 16 will flag `[EXTRA] unexpected section "Test Matrix"`.)
7. Rename `## Coverage Notes` → `## Coverage Gaps` (only S000001 has this drift today). This is tangential drift cleanup that surfaces during the sweep — included because the section name needs to match the new template either way; not an unrelated scope addition.
