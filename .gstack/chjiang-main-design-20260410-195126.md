## Problem Statement

The current system-health skill (v0.1.0) is a 9-layer dashboard designed for the home-setup repo. It references scripts that don't exist in this repo (health-checks.sh, deploy.sh, validate-skill-contracts.sh), conflates config correctness with filesystem health, and doesn't declare its waza dependency. Meanwhile, nobody checks the physical health of ~/.claude/ — disk growth, stale sessions, orphan skills, broken symlinks, dead references between skills. Power users with 40+ skills installed have no way to answer: "what depends on what?" or "what's broken in my Claude setup?"

## What Makes This Cool

The dependency graph. Not just "is your setup healthy?" but "here's the map of every skill, rule, hook, MCP server, and template in ~/.claude/, how they connect, and what's broken." The "npm ls" for your Claude Code environment. Nobody has built this. Waza checks if your config is correct. Gstack /health checks if your code is healthy. This checks if your ~/.claude/ filesystem is healthy and shows you the dependency topology.

## Constraints

- Must work as a Claude Code skill (SKILL.md with frontmatter)
- Self-contained within ~/.claude/ — no scripts from home-setup or other repos. Waza (installed in ~/.claude/) is the sole external dependency
- Waza is a runtime dependency (invokes collect-data.sh), not just a catalog reference
- Read-only by default. Report findings, don't auto-fix
- Output is terminal text, not a web dashboard
- Must work on macOS (darwin). Linux compatibility is nice-to-have

## Premises

1. system-health owns ~/.claude/ FILESYSTEM health only. Config correctness stays with waza. Code quality stays with gstack /health.
2. system-health INVOKES waza's health check as part of its run. Waza is a runtime dependency. The skill calls `bash ~/.claude/skills/waza/health/scripts/collect-data.sh` and includes config findings as a separate section.
3. Self-contained: Bash/Glob/Grep against ~/.claude/ directly. No external scripts from home-setup.
4. Old layers 7 (governance) and 8 (doc quality) from the original design are dropped. They belonged to the home-setup context, not ~/.claude/ health.
5. The dependency graph (skill-to-skill deps, orphan detection, dead references, circular deps) is the differentiating feature worth building.

**Summary:** The dependency graph IS the skill. Five steps: scan, analyze graph, filesystem health, waza integration, score + trend.
**Effort:** M
**Risk:** Medium — graph extraction via grep is heuristic, may miss some references or produce false positives
**Pros:**

- Clean separation: waza owns config, this owns filesystem + topology
**Cons:**
- Reference extraction is imperfect (grep-based, not AST-parsed)

**Reuses:** Waza's collect-data.sh for config layer, gstack's health-history.jsonl pattern for trend tracking

**Effort:** S

## Recommended Approach

**Approach A: Graph-First.** It builds the unique feature (dependency graph) without overscoping into DOT files and impact simulation. The graph analysis is done by Claude (reading the scan output and reasoning about dependencies), not by a separate binary. This keeps it self-contained as a SKILL.md.

## Architecture — 5-Step Skill

### Step 1: Scan ~/.claude/

**Performance optimization:** To avoid 80-100+ sequential tool calls, Step 1 uses a single consolidated bash command that walks all of `~/.claude/` and produces structured output. Claude then analyzes the consolidated output rather than reading each file individually.

```bash
# Single bash command that collects all skill metadata and cross-references
for d in ~/.claude/skills/*/; do
  name=$(basename "$d")
  echo "SKILL:$name"
  # Frontmatter extraction
  [ -f "$d/SKILL.md" ] && head -20 "$d/SKILL.md" | grep -E '^(name|version|description):' || echo "  NO_FRONTMATTER"
  # Symlink check
  [ -L "$d" ] && echo "  SYMLINK:$(readlink "$d")" && ([ -e "$d" ] && echo "  SYMLINK_OK" || echo "  SYMLINK_BROKEN")
  # Cross-references: grep for skill names, file paths, invoke patterns
  grep -rh 'skills/[a-z0-9][-a-z0-9_]*' "$d" 2>/dev/null | grep -oE 'skills/[a-z0-9][-a-z0-9_]*' | sort -u | sed 's/^/  REF:/'
done
# Settings: hooks and MCP
[ -f ~/.claude/settings.json ] && echo "SETTINGS:settings.json" && cat ~/.claude/settings.json
[ -f ~/.claude/settings.local.json ] && echo "SETTINGS:settings.local.json" && cat ~/.claude/settings.local.json
# Rules
[ -d ~/.claude/rules ] && find ~/.claude/rules -name '*.md' -exec echo "RULE:{}" \;
# Templates
[ -d ~/.claude/templates ] && find ~/.claude/templates -type f -exec echo "TEMPLATE:{}" \;
```

Walk `~/.claude/skills/`, `~/.claude/rules/` (if exists), `~/.claude/templates/`, `~/.claude/settings.json`, `~/.claude/settings.local.json`. For each:
- Classify into node type: skill, rule, hook, mcp_server, template, settings, binary
- For skills: read SKILL.md frontmatter (name, description, allowed-tools, version)
- For skills: grep for references to other skills, file paths, hook commands, MCP tools
- For settings: parse hooks and MCP server configurations
- For templates: link to skills via grep (which skills reference this template filename?)
- For rules: check if rule file paths match actual files
- Record: symlink status (size and mtime are collected in Step 3 via `du` and `find`, not in the scan step)

Output: structured inventory of all nodes and their references. Path-reference false positives (matching `skills/X` in non-skill contexts) are filtered in Step 2 by checking whether the referenced skill directory actually exists.

### Step 2: Graph Analysis

Build adjacency list from Step 1 references. 

**Edge types** (confidence levels for false-positive handling):
| Edge Type | Source | Confidence | Example |
|-----------|--------|------------|---------|
| `declared` | SKILL.md frontmatter `depends` field | High | skill A declares dependency on skill B |
| `path-reference` | grep-discovered `~/.claude/skills/X` path | Medium | SKILL.md body references another skill's path |
| `hook-invocation` | settings.json hook command | High | hook runs a script from a skill dir |
| `mcp-reference` | settings.json MCP server config | High | MCP server points to a skill binary |
| `template-usage` | skill body references a template filename | Low | may be coincidental string match |

**Hub threshold:** Any node with in-degree > 5 (referenced by 5+ other nodes) is flagged as HIGH FRAGILITY. Top 3 hubs by in-degree are always listed.

Detect:
- **Orphan skills:** installed but never referenced by any other skill, hook, or settings
- **Circular dependencies:** skill A references skill B references skill A (best-effort detection via reference chain traversal, may miss deep cycles in v0.2.0)
- **Dead references:** skill references a path/skill/tool that doesn't exist
- **Broken symlinks:** symlink targets that don't resolve
- **Hub nodes:** skills/configs with in-degree > 5 (high fragility if changed)

### Step 3: Filesystem Health

- `du -sh ~/.claude/` and per-subdirectory breakdown
- `~/.claude/history.jsonl` size and line count
- Stale sessions: `~/.claude/sessions/` files with mtime older than 24h
- Temp files: `.tmp`, `.bak`, `.pending-*` accumulation
- `~/.claude/file-history/` entry count and growth
- `~/.claude/plans/` count
- Empty directories

### Step 4: Waza Integration

```bash
bash ~/.claude/skills/waza/health/scripts/collect-data.sh
```

Include waza's config findings as a **separate section** in the report. Do NOT re-score waza's findings — waza owns config correctness. Just surface the output so the user gets both filesystem health and config health in one run.

If waza's script is missing: report "Waza not installed — config health checks skipped. Install: see ~/.claude/skills/waza/" and continue with filesystem-only results.

### Step 5: Score + Trend

Score across 5 buckets (each 0-10):
| Bucket | What it measures | Weight | Calibration |
|--------|-----------------|--------|-------------|
| Structure | Proper skill/rule/template organization | 20% | 10 = all skills have name+version+description frontmatter; 7 = 1-3 skills missing version; 4 = 5+ skills missing frontmatter |
| References | No dead refs, no orphans, no circulars | 30% | 10 = zero dead refs, zero orphans; 7 = 1-2 orphans only; 4 = 3+ dead refs or any circular dep |
| Integrity | No broken symlinks, no corrupt files | 20% | 10 = zero broken symlinks; 7 = 1 broken symlink; 4 = 3+ broken or any unreadable SKILL.md |
| Hygiene | No stale sessions, temp files, bloat | 15% | 10 = zero stale sessions, history.jsonl < 1MB; 7 = some stale sessions; 4 = history > 10MB or 50+ stale sessions |
| Runtime | Waza config health (pass-through) | 15% | Pass-through of waza's assessment. 10 = all checks pass; 7 = structural findings only; 4 = critical findings. If waza unavailable, score is N/A and weight redistributed |

Save snapshot to `~/.gstack/health/claude-home-health-history.jsonl` (fixed path, not repo-slug-based, since this is a global health check). Show delta from previous run if history exists.
