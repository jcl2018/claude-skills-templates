## Problem Statement

F000028 (v5.0.8) + F000029 (v5.0.9) wired the doc-sync loop. F000028's post-merge git hook drops a marker at `~/.gstack/doc-sync-pending/<slug>.json` when main moves non-trivially; F000029's `skills-doc-sync-check` script + 3 cj_goal preamble snippets surface a `DOC_SYNC_PENDING` AUQ on next session asking whether to run `/document-release`, snooze, or skip. D000026 (v5.0.14) made the AUQ recommendation branch-aware (B-snooze on main, A-run on a feature branch).

That loop works, but it has two gaps for the cj_goal orchestrator family (`/CJ_goal_feature`, `/CJ_goal_defect`, `/CJ_goal_todo_fix`):

1. **Drift window.** F000029's AUQ only fires on the NEXT `/CJ_goal_*` session. If the operator's next cj_goal invocation is hours or days away, doc drift sits on main until then. The marker lifetime decouples doc-sync from the shipping cycle that produced it.
2. **No per-doc parameterization.** F000029's marker-AUQ runs `/document-release` with no args — the operator can't say "this PR only touched README; skip the ARCHITECTURE audit." It's all-or-nothing. For small surgical PRs (TODO drains, single-doc edits), this means either the full audit runs (overhead) or the operator declines/snoozes and drift accumulates.

There's a better-fit shape for the cj_goal pipeline specifically: run `/document-release` BEFORE `/ship` on the feature branch, after QA passes. Doc updates auto-commit on the same branch and fold into the same code PR. Drift closes to ~minutes (atomic with the code PR, no second auto-deploy cycle). And the new wrapper skill `CJ_document-release` exposes a `--docs` flag for per-invocation subset control — useful for both orchestrator-driven calls (in v1 they pass no flag, defaulting to full audit) and manual operator runs (`/CJ_document-release --docs README` from a feature branch).

F000029's marker-AUQ stays as a fallback for paths that bypass the orchestrator (raw `git push`, manual `/ship` from outside the cj_goal pipeline, etc.) — the two mechanisms layer, they don't fight.

## What Makes This Cool

This closes the doc-sync loop into the shipping cycle that originated it. Instead of "main moved → marker dropped → next session surfaces an AUQ → operator decides whether to do doc work," we get "QA passed → docs audited and committed → /ship opens ONE PR with code + docs → /land-and-deploy merges everything atomically." Code and docs ship together because they were CHANGED together. The two-PR follow-up pattern was an artifact of the marker mechanism, not a desired property of the workflow; the same-PR shape removes the artifact.

The `--docs` flag is the second win: per-invocation control over which audit categories fire. The default (when omitted) is `/document-release`'s full audit set, which is what the orchestrator wiring uses in v1. But operators can manually run `/CJ_document-release --docs README,CHANGELOG` from a feature branch to do a narrow audit without the SKILL-CATALOG or ARCHITECTURE drift checks. F000029's marker-AUQ has no such surface — it's a single button labeled "run /document-release" with no knobs.

The third (subtle) win: this is the workbench's first skill with the EXPLICIT shape of "thin wrapper around an upstream gstack skill." `/CJ_document-release` calls `/document-release` via the Skill tool, adding workbench-specific concerns (per-doc filtering, halt taxonomy, auto-commit of doc-only changes) without touching upstream. If we later want similar wrappers for other upstream skills (`/CJ_ship`? `/CJ_review`?), this is the template.

## Constraints

- **Workbench-only scope.** No upstream `/document-release` modification (per memory `feedback_workbench_scope` + `project_workbench_auto_deploy_unsafe`). `CJ_document-release` invokes upstream `/document-release` via the Skill tool with project-context priming; the filter/halt/auto-commit logic lives in the workbench skill, not upstream.
- **Auto-deploy unsafe.** Per memory `project_workbench_auto_deploy_unsafe`: `cj-handoff-gate.sh`'s denylist blocks `skills-catalog.json`, `tests/`, `validate.sh`, `test.sh`, and skill dirs — exactly the surfaces every feature touches. Same-PR shape sidesteps the auto-deploy doubling that a second-PR (after `/land-and-deploy`) pattern would introduce. The doc updates fold into the existing code PR's review surface — operator reviews once, lands once.
- **ERROR-strict validation.** Per memory `feedback_skill_contracts_strict` + the F000030/F000032/F000033/F000034 precedent: WARN gets ignored; ERROR-with-cheap-override is the load-bearing pattern. `CJ_document-release` returning non-green produces a `[doc-sync-red]` halt marker that the orchestrator treats as a build failure (not a soft warning the operator can dismiss).
- **F000032+F000033+F000034 compliance.** The skill ships with: per-skill `USAGE.md` (F000032), USAGE.md drift detection works against it (F000033), a section in `doc/SKILL-CATALOG.md` with a `(phase-step in /CJ_goal_feature chain)` tag (F000034). Validate.sh checks 13-15 already cover this shape — no new checks needed.
- **F000029 BD#1 explicit supersession.** F000029_DESIGN.md Big Decision #1 explicitly rejected "Approach B: new /CJ_doc_sync skill" for two reasons: (a) "adds skill to catalog forever," (b) "extra invocation hop." This PR knowingly reopens that decision because the new skill has STRICTLY MORE capability than the rejected detection-only shape: (a) per-doc `--docs` flag parameterization, (b) halt-on-red contract, (c) auto-commit-doc-only-changes behavior, (d) operator-callable `/-command` surface for manual narrow audits. The catalog cost is justified by the capability surface; the design doc must call this out so the supersession is auditable. F000029_DESIGN.md BD#1 row gets a brief "**SUPERSEDED BY F000035**" annotation in the same PR.
- **No /CJ_goal_feature auto-deploy expansion.** `/CJ_goal_feature` PR-stops by design. `CJ_document-release` runs BEFORE `/ship` on the feature branch, so it works with PR-stop semantics without expanding `/CJ_goal_feature` to auto-deploy. The wiring is uniform across all 3 orchestrators (feature, defect, todo_fix) precisely because the same-PR shape doesn't depend on `/land-and-deploy` running.
- **Atomic commit ordering.** The pre-commit hook runs `validate.sh`. Stage everything together: skill files + catalog entry + doc/SKILL-CATALOG.md section + 3 pipeline.md updates + 3 SKILL.md halt-taxonomy updates + tests + CHANGELOG + VERSION + the F000029_DESIGN.md BD#1 annotation. One commit, one push.
- **CLAUDE.md tracked-doc/ manifest unchanged.** `CJ_document-release` doesn't add a new `doc/*.md` file; it reads the existing manifest as project context to know which doc/ files to audit when the orchestrator passes no `--docs` flag.

## Premises

(confirmed via D1+D2+D3)

1. **Trigger**: `CJ_document-release` runs BEFORE `/ship` on the feature branch, after QA passes; doc updates auto-commit on the same branch and fold into the code PR.
2. **Surface**: All 3 cj_goal orchestrators (feature, defect, todo_fix) get the wiring — uniform behavior across the family.
3. **Skill**: `CJ_document-release` IS a new skill in `skills/` with SKILL.md, USAGE.md, catalog entry, doc/SKILL-CATALOG.md section, validate.sh-compliant.
4. **Flag**: Accepts `--docs <subset>` flag for per-invocation doc subset (comma-separated). Default (omitted): full `/document-release` audit.
5. **F000029 coexistence**: F000029's marker-AUQ stays as fallback for non-orchestrator paths (raw `git push`, manual `/ship` from outside the cj_goal family, etc.). Not deprecated.
6. **F000029 BD#1 supersession**: F000029 BD#1 ("rejected new /CJ_doc_sync skill in catalog") is explicitly superseded; `CJ_document-release` earns its catalog cost via parameterization + halt-on-red + auto-commit that the F000029 detection-only script can't expose.
7. **Halt-on-red**: If `/document-release` (invoked via Skill from inside `CJ_document-release`) returns non-green, the orchestrator HALTs with marker `[doc-sync-red]` — doc-sync failure is a build failure.
8. **Workbench-only**: No upstream `/document-release` modification — `CJ_document-release` wraps it via Skill tool with project-context priming.

## Approaches Considered

(see Approach A/B/C in D4 AUQ; Approach A chosen)

### Approach A: New SKILL.md skill (CHOSEN)

`skills/CJ_document-release/SKILL.md` + `USAGE.md` + catalog entry + `doc/SKILL-CATALOG.md` section. Skill reads `--docs` arg, builds a project-context block, invokes `/document-release` via Skill tool. Each orchestrator's `pipeline.md` gets a new Step 5.5 between QA and `/ship`. Halt marker `[doc-sync-red]`.

**Pros**: Clean abstraction — `--docs` flag, halt taxonomy, auto-commit behavior, USAGE.md all live in one skill folder; future per-verb extensions are single-file changes. Matches the workbench's just-shipped F000032+F000033+F000034 pattern exactly (per-skill USAGE.md + doc/SKILL-CATALOG.md section + validate.sh-compliant). Operator gets a real `/CJ_document-release` callable manually from any feature branch with `--docs` subset.

**Cons**: New entry in `skills-catalog.json` forever; F000029 BD#1 explicitly rejected this shape — design doc must document the supersession.

### Approach B: Hybrid (SKILL.md + scripts/cj-doc-release-prep.sh helper)

Thin SKILL.md; `scripts/cj-doc-release-prep.sh` does the heavy lifting (reads CLAUDE.md tracked-doc/ manifest, parses `--docs`, prints filtered set + halt verdict). Mirrors F000029's `scripts/skills-doc-sync-check` pattern.

**Pros**: Mirrors F000029's proven shape (script + thin SKILL.md). Filter logic in bash is single-evolution surface.

**Cons**: Two surfaces (skill + script) for a problem that fits in one. `--docs` flag handling split between script `--help` and SKILL.md instructions — discoverability fragmentation. Marginal complexity over A.

### Approach C: Inline triplicate (no new skill)

Each cj_goal `pipeline.md` gets a Step 5.5 with inline bash (filter manifest by `--docs`) + inline Skill(`/document-release`) invocation. No new SKILL.md, no new script.

**Pros**: Smallest diff. Zero catalog/USAGE.md/doc/SKILL-CATALOG.md churn. No F000029 BD#1 supersession needed.

**Cons**: 3 places to edit on logic change — EXACTLY the anti-pattern F000029 BD#1 itself flagged when it rejected its own internal "Approach A" (triplicate 10-line block in 3 SKILL.md preambles) for the same reason. No `/-command` surface — operator can't manually run `/CJ_document-release --docs README` from any branch; parameterization only fires inside an orchestrator pipeline.

## Recommended Approach

**Approach A**, new SKILL.md skill. Detailed plan:

### Step 1: Skill skeleton

`skills/CJ_document-release/SKILL.md`:

```yaml
---
name: CJ_document-release
description: "Wraps /document-release for cj_goal orchestrators with per-invocation --docs filter, halt-on-red contract, and auto-commit of doc-only changes. Folds doc updates into the same code PR by running BEFORE /ship on a feature branch."
version: 0.1.0
allowed-tools:
  - Bash
  - Read
  - Glob
  - Grep
  - Skill
---
```

SKILL.md body covers: preamble (skills-update-check), arg parsing (`--docs`), clean-tree precondition check, project-context block construction, Skill(`/document-release`) invocation, halt-on-red emit, doc-only-changes auto-commit, success summary.

`skills/CJ_document-release/USAGE.md`: 5 required sections per the F000032 convention:
- **When to use** — auto-invoked between QA and `/ship` by cj_goal orchestrators; manually invoke `/CJ_document-release [--docs <subset>]` from a feature branch when you want a narrow doc audit before `/ship`.
- **When NOT to use** — on main (refuses; `/document-release` hard-aborts on the base branch); when the working tree is dirty with non-doc changes (refuses; cleanliness precondition).
- **Mental model** — thin wrapper over upstream `/document-release` that adds workbench-specific filter, halt, and commit behavior. Doesn't replace F000029's marker-AUQ; layers with it.
- **Common pitfalls** — passing `--docs FOO` where FOO doesn't match any doc category (skill warns + no-ops, doesn't halt); expecting it to work on main (it refuses, mirroring upstream Step 1's hard-abort).
- **Related skills** — `/document-release` (the upstream skill wrapped), `/CJ_goal_feature`, `/CJ_goal_defect`, `/CJ_goal_todo_fix` (the orchestrators that auto-invoke it), `scripts/skills-doc-sync-check` (F000029's fallback path).

### Step 2: Skill behavior

The skill prose in SKILL.md walks the orchestrator through these steps in order:

1. **Preamble**: standard `skills-update-check` (and optionally `skills-doc-sync-check` — but skip; this skill is itself part of the doc-sync surface, AUQ-from-self would be recursive).
2. **Arg parse**: extract `--docs <comma-separated-list>` from `$ARGUMENTS`. Valid values (case-insensitive): `README`, `CHANGELOG`, `CLAUDE`, `ARCHITECTURE`, `PHILOSOPHY`, `SKILL-CATALOG`, `all`. Unknown values warn + skip (don't halt — operator may have typo'd a doc name, loud warn is enough). If no `--docs` arg, default = full audit (no filtering).
3. **Branch + clean-tree gate**: `git symbolic-ref --short HEAD` — if `main`/`master`/the base branch, refuse with message "CJ_document-release refuses on the base branch (upstream /document-release aborts there). Run from a feature branch." Then `git status --porcelain | grep -v -E '^.. (README\.md|CHANGELOG\.md|CLAUDE\.md|ARCHITECTURE\.md|doc/.*\.md)$'` — if non-empty (non-doc changes are uncommitted), refuse with message "Working tree has uncommitted non-doc changes. Commit them first." The clean-tree gate is critical: `/document-release` reads the working tree as context; uncommitted code changes would pollute its audit.
4. **Project context block**: build a multi-line string for `/document-release`:
   ```
   CJ_document-release-context:
     - mode: workbench-orchestrator-wrap
     - filter_docs: <subset_or_full>
     - caller: <orchestrator_name_if_known_else_manual>
     - skip_categories: <inverse_of_subset_when_filtered>
   ```
   This block is documentation-only — `/document-release` doesn't parse it programmatically (upstream skill, we don't modify it). The skill instructions tell `/document-release` "this run is filtered to <subset>; audit ONLY those categories and skip the rest." If `/document-release` honors the request, filtering works; if it audits everything anyway, the skill still auto-commits whatever `/document-release` produced. Best-effort filter, not enforced filter.
5. **Skill(`/document-release`) invocation**: via Skill tool, passing `args=<the context block above>`. Wait for completion. Read result (success/failure indicator).
6. **Halt-on-red**: if `/document-release` returned non-green (errored mid-write, failed audit checks, etc.), emit `[doc-sync-red]` halt marker + journal entry:
   ```
   [doc-sync-red]
   next_action=inspect /document-release output; fix doc errors; re-run /CJ_document-release
   resume_cmd=/CJ_document-release [--docs <same-subset>]
   pr_url=N/A
   raw_output_path=<path-from-document-release-output-if-any>
   ```
   Return RESULT=red so caller orchestrator HALTs.
7. **Auto-commit doc-only changes**: `git status --porcelain` — for each line, check if path matches `^(README|CHANGELOG|CLAUDE|ARCHITECTURE)\.md$` or `^doc/.+\.md$`. If matches, `git add <file>`. If ANY non-doc file is dirty at this point (`/document-release` modified something outside the whitelist), HALT with `[doc-sync-non-doc-write]` — this is a new halt class for the upstream-skill-misbehaved case. On clean doc-only diff, `git commit -m "docs: post-build sync via CJ_document-release"`. If `git status --porcelain` is empty after `/document-release` ran (no doc changes needed), print "Docs already current — no commit" and return RESULT=green-noop.
8. **Success summary**: print:
   ```
   CJ_document-release: green
     /document-release: green
     commit: <sha-or-noop>
     filtered: <subset-or-full>
   ```
   Return RESULT=green.

### Step 3: Orchestrator wiring

Each of `/CJ_goal_feature/pipeline.md`, `/CJ_goal_defect/pipeline.md`, `/CJ_goal_todo_fix/pipeline.md` gets a new "Step 5.5: Doc-sync (CJ_document-release)" subsection inserted between QA pass and `/ship`. The block is identical across all 3 (modulo orchestrator-name in the resume_cmd):

```markdown
### Step 5.5: Doc-sync (inline)

After QA passes green and BEFORE `/ship`:

1. Invoke `Skill(skill="CJ_document-release")` (no `--docs` flag in v1 orchestrator wiring; default full audit).
2. If RESULT=red: HALT with marker `[doc-sync-red]`. Journal entry:
   - `next_action=inspect /document-release output; fix; re-run /CJ_goal_<verb>`
   - `resume_cmd=/CJ_goal_<verb> [args]`
3. If RESULT=green: continue to Step 6 (`/ship`). The doc commit (if any) is now on the feature branch; `/ship` will include it in the code PR diff.
4. If RESULT=green-noop: continue to Step 6 — no doc commit was needed.
```

Each orchestrator's `SKILL.md` halt taxonomy table also gets a new row:

```markdown
| `halted_at_doc_sync` | `[doc-sync-red]` | CJ_document-release returned red (doc-sync failure is a build failure) |
| `halted_at_doc_sync_non_doc_write` | `[doc-sync-non-doc-write]` | /document-release modified files outside the doc-only whitelist (upstream-skill misbehavior) |
```
