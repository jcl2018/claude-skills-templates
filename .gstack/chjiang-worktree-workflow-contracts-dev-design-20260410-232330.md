## Problem Statement

The workflow skill (track/implement/review/ship) has never been tested end-to-end in a foreign repo. It was developed and mentally traced inside the skills-templates workbench. A UX audit for the "small team lead" persona (2-5 devs wanting consistency) revealed 2 real bugs that break core functionality and 3 UX gaps that leave new users stranded.

The skill works in the workbench repo because templates live at `$REPO_ROOT/templates/` and the contracts skill lives at `$REPO_ROOT/skills/contracts/`. In any other repo, both paths fail silently.

## What Makes This Cool

Fixing these issues turns the workflow skill from "works for the author" to "works for anyone who installs it." The template fallback chain and contracts detection, once fixed, make the skill genuinely portable. The UX polish (diagnostics, hints, first-run guidance) means a team lead can self-serve without reading the source.

## Constraints

- gstack is an intentional dependency. Review (Phase 3) and Ship (Phase 4) delegate to gstack `/review` and `/ship`. This is by design, not a bug.
- Templates are deployed to `~/.claude/templates/` via `skills-deploy install`. The 3-level fallback chain is documented in SKILL.md but only 2 levels are implemented in track.md.
- The contracts skill at `contracts/SKILL.md:41-43` already has the correct 3-level fallback. The fix reuses that pattern.
- Changes are limited to skill markdown files (SKILL.md, track.md, review.md, ship.md). No script changes.

## Premises

1. The contracts path check in review.md and ship.md (`$REPO_ROOT/skills/contracts/SKILL.md`) will always fail in a foreign repo. The skill is installed at `~/.claude/skills/contracts/SKILL.md`. Confirmed by the user.
2. The template fallback chain in track.md misses `~/.claude/templates/` (level 3). Scaffold will fail with "MISSING" errors. The contracts skill already implements the correct 3-level chain.
3. gstack dependency is intentional. Not in scope to decouple.
4. The skill has never been tested in a foreign repo. Mental tracing missed these issues. Confirmed by the user.

## Recommended Approach

**Approach B: Bug Fix + UX Polish.** The bugs alone don't make the skill usable for a new team lead. The UX improvements (template diagnostics, first-run guidance, contracts install hint) are the difference between "it fails" and "it tells you what to do."

### Change 1: Fix contracts path check (review.md + ship.md)

**Current (review.md:23-25, ship.md:37-39):**
```bash
SKILL_DIR=$(git rev-parse --show-toplevel 2>/dev/null)/skills/contracts
[ -f "$SKILL_DIR/SKILL.md" ] && echo "CONTRACTS_AVAILABLE" || echo "CONTRACTS_MISSING"
```

**Fixed:**
```bash
CONTRACTS_FOUND="no"
for dir in "$(git rev-parse --show-toplevel 2>/dev/null)/skills/contracts" "$HOME/.claude/skills/contracts"; do
  [ -f "$dir/SKILL.md" ] && CONTRACTS_FOUND="yes" && break
done
[ "$CONTRACTS_FOUND" = "yes" ] && echo "CONTRACTS_AVAILABLE" || echo "CONTRACTS_MISSING"
```

Checks repo-local first (for the workbench), then `~/.claude/skills/` (for installed skills).

### Change 2: Fix template fallback chain (track.md)

**Current (track.md:45-46):**
```bash
TEMPLATE_DIR="$REPO_ROOT/templates"
[ -d "$TEMPLATE_DIR" ] || TEMPLATE_DIR=~/.claude/spec/templates
```

**Fixed (matches contracts/SKILL.md:41-43 pattern):**
```bash
TEMPLATE_DIR=""
for dir in "$REPO_ROOT/templates" "$HOME/.claude/spec/templates" "$HOME/.claude/templates"; do
  [ -d "$dir" ] && TEMPLATE_DIR="$dir" && break
done
```

If no directory is found, TEMPLATE_DIR remains empty and the diagnostic in Change 3 fires.

### Change 3: Template diagnostics (track.md)

After template validation, if any are MISSING, add a diagnostic message:

```
If required templates are missing:
"Templates not found in:
  - $REPO_ROOT/templates/ (repo-local)
  - ~/.claude/spec/templates/ (user spec)
  - ~/.claude/templates/ (skills-deploy)
Run `skills-deploy install` from your claude-skills-templates clone,
or copy templates/ into your repo root."
```

## Open Questions

1. Should `work-items/` be committed to git by default? The design says yes (team visibility), but some teams may prefer gitignoring it. Consider adding a note in the first-run guidance.
2. Should `track.md` also check for `artifact-manifests.json` at `~/.claude/` in addition to `$REPO_ROOT/`? The current code has this fallback (track.md:30-33) but it's not tested.
3. Should there be an end-to-end test script that installs the skills to a temp repo and runs through the lifecycle? (Deferred to Approach C / future work.)

## Success Criteria

- A team lead can install skills via `skills-deploy install`, go to their own repo, and run `/workflow track create --type feature` without hitting silent failures.
- Contract quality gates fire correctly during `/workflow review` and `/workflow ship` when contracts is installed at `~/.claude/skills/`.
- When something is missing (templates, contracts), the error message tells the user exactly what to do.

## Distribution Plan

Changes are to skill markdown files only. Distribution via the existing `skills-deploy install` pipeline (symlinks SKILL.md, copies templates). No new distribution mechanism needed.
