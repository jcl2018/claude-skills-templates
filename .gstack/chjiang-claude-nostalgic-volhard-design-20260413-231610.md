## Problem Statement

F000003's PRD describes "3 capabilities in one skill" (registry + enforcement + scaffolding) but doesn't articulate the real product: a standalone company template system. The system should be a portable Claude Code skill that packages an entire company's work item spec. Install it in any repo. No gstack dependencies. No /review, /ship, /docs check. It just works.

Template registration is done (templates copied, registry created, validate subcommand shipped). The remaining work is enforcement (`check`) and scaffolding (`create`), both built standalone.

## What Makes This Cool

One skill, one install, complete company spec enforcement. A repo that has never heard of gstack can run `company-workflow check` and get artifact completeness validation. The skill is the spec and the enforcement in one package.

## Lifecycle Boundary

The company-workflow skill enforces the company's 4-phase lifecycle (Track, Implement, Review, Ship) for company-type work items. The workbench uses 3-phase (Track, Implement, Ship) for its own work items. This is intentional divergence: the company spec IS 4-phase, and the skill preserves the spec exactly. F000003's own tracker uses 3-phase because it's a workbench work item tracked by workbench rules.

## Subcommand Boundaries

- **validate**: checks a SINGLE FILE against contract.json structural rules (required fields, required sections, section order, lifecycle phases). Exit 0/1.
- **check**: checks a DIRECTORY for artifact completeness (are all required files present for the type? do they have correct frontmatter type fields?). Reports [PASS]/[MISSING]/[DRIFT].
- **create**: scaffolds a new work item directory from templates, fills placeholders, runs validate on the result.

Each subcommand is independent. No cross-command data flow. Validate does not call check. Check does not call validate. Create calls validate at the end as a post-scaffold verification step.

## Constraints

- Zero gstack dependencies (no analytics logging, no /review, no /docs check references)
- Preserves company spec conventions (userstory spelling, workflow_type, 4-phase lifecycle)
- Lives in this repo but installable in any repo via skills-deploy
- Owns its own data: contract.json, company-artifact-manifests.json, templates
- Template registration (Task 1) is DONE and should not be re-specified

## Premises

1. The company-workflow skill is standalone. Zero gstack dependencies.
2. Three subcommands: `validate` (done), `check` (next), `create` (after check). Each independent.
3. The skill owns its own manifest, contract, and templates. Reads ONLY its own files.
4. Template registration is shipped. PRD acknowledges this as done.
5. Company conventions preserved even where they differ from workbench conventions.
6. Designed for installation in any repo via skills-deploy.

### PRD Content Guide

The new PRD should have this structure:

**Problem Statement:** The company follows a formal work item spec (contract.json, 5 tracker types, 8 doc types, reference guides, philosophy docs). Today it lives at ~/Downloads/spec/ with no enforcement. The company-workflow skill packages this spec as a portable, self-contained Claude Code skill.

**Mental Model:**
```
company-workflow skill (self-contained)
    |
    +-- validate <path>   (done: contract.json structural rules)
    +-- check <path>      (next: artifact completeness per type)
    +-- create --type --name  (last: scaffold new work items)
    |
    reads: company-artifact-manifests.json, contract.json, templates/company-workflow/
    writes: new work item directories + files
    depends on: nothing external
```

**User Stories:** Organized by subcommand:
- validate stories (done, shipped)
- check stories (P0: missing detection, drift detection, 5-type coverage, independence)
- create stories (P0: scaffold all 5 types, placeholder substitution, post-scaffold validation)

**ACs:** GWTB (Given/When/Then/But) blocks per subcommand
