## Problem Statement

The `/company-workflow` skill has three contract/template drift defects that surface every time it's deployed and exercised in a consumer repo (ai-content, master). Trackers freshly scaffolded from the skill's own templates fail the skill's own validator, and the manifest forces a 5-artifact set at both feature scope and nested user-story scope (verified in `ai-content/F973012/S1441024-hfss-integration/`, where the same PRD/ARCH/TEST-SPEC content lives in two places).

The defect tracker [D000003_company_workflow_contract_template_drift](work-items/defects/D000003_company_workflow_contract_template_drift/D000003_TRACKER.md) captures the symptoms; this design specifies the fix.

## What Makes This Cool

The fix is a **round-trip invariant** test. For each tracker type, scaffold from the template into a temp dir and run the validator against it. The fixture must pass with zero violations. This single test catches all three defect classes — frontmatter-required mismatches, section-order drift, and any future template/contract divergence. Once it's in `scripts/test.sh`, this kind of bug stops being possible.

The other cool part: the reference implementation **already exists in the same repo**. `personal-workflow` has solved Issues 2 and 3 cleanly via `type_specific_optional` sections and a split feature-vs-user-story artifact set. We don't have to invent the schema; we just align the divergent twin.

## Constraints

- **Don't break deployed validators.** The ai-content repo runs `/company-workflow check` against existing trackers. The fix must accept legacy trackers (those still carrying PRD/ARCH/TEST-SPEC at feature scope) as valid, then provide a one-shot migration path.
- **Don't fight the templates.** Per Issue 1's resolution, `workflow_type` and `url` stay in templates because they're meaningful (Jira/TFS integration). The contract has to learn about them, not the other way around.
- **Keep skills standalone.** The two skills (personal-workflow, company-workflow) must not develop a runtime dependency on each other. Shared schema is documentation/convention, not code.
- **Preserve `tracker-task.md` and `tracker-review.md`.** Both templates already match the contract today (neither emits Acceptance Criteria or Reproduction Steps). Issue 3 affects only feature/user-story/defect templates. The fix must not regress task or review.

## Premises

1. **All three issues are confirmed drift, not perception.** Verified by reading `skills/company-workflow/contract.json`, `company-artifact-manifests.json`, `SKILL.md` (lines 126, 145), and all four `templates/company-workflow/tracker-*.md` files side by side.
2. **Single root cause: contract.json and templates/ are independent sources of truth, with no round-trip test.** The fixtures dir has happy-path and explicit-bad-case fixtures but no "scaffold-then-validate" invariant.
3. **The reference implementation already exists in this repo: `personal-workflow`.** Its `contract.json` has `optional`, `order_skip_absent: true`, and `type_specific_optional`; its manifest splits feature (2 artifacts) from user-story (4 artifacts). Both Issue 2's recommended fix and Issue 3's fix are already shipping there.
4. **Issue 1 direction (keep `workflow_type`/`url`, fix contract) is correct because the validator already enforces these fields.** Removing them from templates breaks deployed `/company-workflow check` runs.
5. **Issue 2 direction (feature drops PRD/ARCH/TEST-SPEC, gains a new `feature-summary.md`, keeps milestones) preserves high-level design intent at feature scope while eliminating the duplication with nested user-stories.** The `feature-summary.md` artifact is lightweight (one-pager: scope, success criteria, links to constituent user-stories) and gives the feature its own roll-up identity beyond just milestones. **Scope note:** strictly, D000003's symptom would be resolved by mirroring personal-workflow exactly (feature = `tracker + milestones`, no new artifact). Adding `feature-summary.md` is a deliberate scope expansion taken to preserve feature-level design identity in the company-workflow context (Jira/TFS-backed work where features have stakeholders distinct from individual stories). Owned, not accidental.

## Approaches Considered

### Approach A: Mechanical Patch
Edit `contract.json` + `manifest.json` + add `feature-summary.md` template + mirror personal-workflow's section pattern + write one round-trip fixture script. Single PR. ~3-4 hr human / ~30 min CC. Ships fast; doesn't extract any reusable invariant.

### Approach B: Shared Pattern + Round-Trip Invariant *(Recommended)*
Same surface fixes as A, plus:
- Document the contract schema convention in `skills/_shared/contract-schema.md` (a markdown reference, not code) so the next skill author has a clear model.
- Add a generic round-trip fixture runner to `scripts/test.sh` that both `personal-workflow` and `company-workflow` consume. The runner: for each tracker type in each skill's manifest, scaffold from template into a temp dir, then exec the skill's `check` subcommand against it. Zero violations or test fails.
- Two coordinated PRs: PR 1 adds the round-trip runner and proves both skills pass today; PR 2 lands the company-workflow contract/manifest/template fixes.

~1 day human / ~1-2 hr CC.

### Approach C: Contract-First Generation
Templates become derived artifacts. `contract.json` + `manifest.json` declare fields/sections; a generator scaffolds templates at deploy time. Eliminates drift architecturally. ~3-5 days human / ~3-4 hr CC. Loses template ergonomics (humans prefer reading markdown templates over JSON schemas) and reshapes skills-deploy. YAGNI.

## Recommended Approach

**Approach B** — Shared Pattern + Round-Trip Invariant.

The marginal cost from A → B is one fixture runner (~50 lines of bash) plus one convention doc (~one page). In exchange, the two sibling skills stop drifting from each other permanently, and any third skill following the pattern inherits the invariant.

### Implementation order

**PR 1 — Round-trip invariant (lands first, proves the gap):**
1. Add `scripts/test-roundtrip.sh` (~50-80 lines). Implementation contract pinned by /plan-eng-review:
   - **Source `lib.sh`; call `init`.** Use `$REPO_ROOT`, `$SKILLS_DIR`, `$TEMPLATES_DIR`. Match existing scripts/test.sh pattern with `pass()`/`fail_test()` helpers and `ERRORS` tally.
   - **Skill discovery: file-presence opt-in.** Walk `skills/*/`. Process only skills that have BOTH `contract.json` AND `*-artifact-manifests.json`. Skills missing either (e.g., `system-health`) are silently skipped.
   - **Subcommand mapping: hardcoded inside the runner.** `personal-workflow` → `check`, `company-workflow` → `validate`. (Manifest-field approach is over-engineering for two skills.)
   - **Placeholder substitution: inline sed.** ~10-line substituter inside the runner. Fixed deterministic test values: `FEATURE_ID=F999999`, `STORY_ID=S999999`, `TASK_ID=T999999`, `DEFECT_ID=D999999`, `*_NAME=Test {Type}`, `YYYY-MM-DD=2026-01-01`, `BRANCH_NAME=test/branch`, `REPO_PATH=test/repo`, `JIRA_OR_TFS_URL=https://example.com/X-1`, `PARENT_ID=F999999`, `author=test`. List explicit substitutions in the script header.
   - **Failure differentiation.** Two distinct failure classes with distinct messages:
     - `TEMPLATE_BUG: skills/{X}/templates/{Y}.md — {reason}` for scaffold/template failures (template missing, malformed YAML, unknown placeholder).
     - `DRIFT: skills/{X} — {N} violations on type {Y}` for validator violations.
     - Both produce non-zero exit; CI logs distinguish them.
   - **Cleanup.** `mktemp -d` + `trap '... rm -rf ...' EXIT` matching scripts/test.sh:137 pattern. Per-skill, per-type temp dirs cleaned even on failure.
2. Wire it into `scripts/test.sh` as a new test block.
3. Run it against the current state. **Scope of detection:** the round-trip runner catches Issue 1 (frontmatter required-field mismatch will surface as VIOLATION on `workflow_type`/`url` once contract is updated; today it passes only because the validator's company-specific check is the source of truth) and Issue 3 (section-order failure on freshly scaffolded feature/user-story/defect trackers). It does **not** catch Issue 2 — manifest-level artifact duplication is a design choice, not a scaffold-validate failure. Personal-workflow should pass. Company-workflow should fail on Issues 1 and 3 — this becomes the failing test that PR 2 fixes.
4. **Self-tests** (per /plan-eng-review section 3A — 2 self-tests + 2 regression tests):
   - **S1 — DRIFT detection:** add fixture `skills/zzz-test-roundtrip-bad/` with intentionally drifting template (e.g., section in wrong order). Add an assertion in scripts/test.sh: runner reports DRIFT non-zero exit with "DRIFT" in stderr.
   - **S2 — TEMPLATE_BUG detection:** add fixture `skills/zzz-test-roundtrip-malformed/` with malformed template (e.g., unclosed YAML frontmatter). Assertion: runner reports TEMPLATE_BUG non-zero exit with "TEMPLATE_BUG" in stderr.
   - **R1 — personal-workflow CLEAN regression:** runner against real `skills/personal-workflow/` exits 0.
   - **R2 — company-workflow CLEAN regression after PR 2:** before PR 2 the runner against real `skills/company-workflow/` MUST FAIL on Issues 1 and 3 (this is the failing test that justifies PR 2). After PR 2 lands, exit 0.
5. Document the runner's contract: a skill is "round-trip clean" if its templates scaffold into validator-passing files.

**PR 2 — Company-workflow fixes (turns the failing test green):**
1. **Issue 1:**
   - **Pre-step:** Grep ai-content's existing trackers for `workflow_type` to confirm it's universally present before promoting to `required`. Command: `grep -L "^workflow_type:" path/to/ai-content/work-items/**/*_TRACKER.md`. Any file in the output is a legacy tracker missing the field — if any exist, demote `workflow_type` to `recommended` instead of `required` (same grandfather policy as `url`). If none exist (expected case, since templates have always emitted it), `required` is safe.
   - Add `workflow_type` to `contract.frontmatter.required` (or `recommended` per the pre-step result).
   - Add `url` to `contract.frontmatter.recommended` (NOT required) to grandfather any legacy ai-content trackers that pre-date the field. This is the explicit policy for the "don't break deployed validators" constraint.
   - Update `SKILL.md` line 126 to reference `contract.json` instead of inlining the field list. (Line 145 is an example error string in a code fence — leave it alone.)
2. **Issue 2:**
   - Update `company-artifact-manifests.json`: feature requires `tracker + feature-summary + milestones` (3 artifacts); user-story keeps `tracker + PRD + ARCH + TEST-SPEC + milestones`.
   - Add `templates/company-workflow/doc-feature-summary.md`. **Sections** (resolves Open Question 1): YAML frontmatter (`type: feature-summary`, `parent: {FEATURE_ID}`, `title`, `date`, `author`, `status`); `## Scope` (one paragraph: what this feature delivers); `## Success Criteria` (bulleted measurable outcomes); `## Constituent User-Stories` (markdown links to nested user-story TRACKERs); `## Out-of-Scope` (explicit non-goals).
   - Update `WORKFLOW.md` summary table.
   - Add `feature-summary.md` row to the per-skill template manifest consumed by `skills-deploy install` (per CLAUDE.md's "Template deployment" section). Verify `skills-deploy doctor` reports the new template.
   - Provide a migration note in CHANGELOG: existing feature-scope `PRD.md`/`ARCHITECTURE.md`/`TEST-SPEC.md` files become legacy artifacts (validator no longer requires them at feature scope) but are not flagged. Author should migrate canonical content to the nested user-story directory.
3. **Issue 3:** Update `contract.json`:
   - `sections.optional`: add `"Acceptance Criteria"` and `"Reproduction Steps"`.
   - `sections.expected_order`: insert both between `"Lifecycle"` and `"Todos"`.
   - Add `"order_skip_absent": true`. **Semantic:** when this flag is true, the validator filters `expected_order` to sections actually present in the file before checking order — so an optional section that's absent never causes a position mismatch downstream. (Note: company-workflow's `tracker-defect.md` currently passes section-order only because the contract has no explicit position for `Reproduction Steps` — it's an accidental pass via undefined behavior. Adding `order_skip_absent: true` plus the position entry makes the pass intentional.)
   - Add `"type_specific_optional"` mirroring personal-workflow's pattern. `tracker-task.md` (no optional sections) and `tracker-review.md` (no optional sections) are both fine — `type_specific_optional` will list them with empty arrays.
4. Add `skills/_shared/contract-schema.md` (the convention doc — not code).
5. Re-run `scripts/test-roundtrip.sh`. Company-workflow now passes.
6. Bump skill version per `scripts/collection-version.sh`. Update CHANGELOG.
7. Update `D000003_TRACKER.md`: append commit SHAs for both PRs to the Log section, check the corresponding Lifecycle gates, add the modified files to the Files section.

## Failure modes

| Codepath | Realistic failure | Test? | Error handling? | User-visible? |
|----------|-------------------|-------|-----------------|---------------|
| Round-trip runner: skill discovery | Skill has contract.json but no manifest (or vice versa) | YES (file-presence skip is itself a tested behavior) | Skip silently | No surface — that's correct |
| Round-trip runner: scaffold | Template malformed YAML | YES (S2 self-test) | TEMPLATE_BUG with template path | Clear stderr line in CI log |
| Round-trip runner: scaffold | Unknown placeholder in template | YES (covered by S2 — fixture skill triggers it) | TEMPLATE_BUG with placeholder name | Clear stderr line |
| Round-trip runner: validator exec | Validator command not on PATH / not executable | NO test | Fails with TEMPLATE_BUG | Stderr line with error |
| Round-trip runner: cleanup | Process killed mid-run | Implicit (trap) | trap fires on EXIT including signals | Temp dir removed |
| Company-workflow validator | Legacy ai-content tracker without `url` | YES (regression R2 + post-deploy check) | `url` is recommended, not required — passes | None |
| Company-workflow validator | Legacy ai-content feature with PRD/ARCH/TEST-SPEC at feature scope | YES (regression R2 + post-deploy check) | Validator only requires the new 3-artifact set; extras are tolerated | None |
| Company-workflow validator | Defect tracker with Reproduction Steps section | YES (R2) | Now intentionally accepted via `type_specific_optional` + `order_skip_absent` | None (was accidentally passing; now passes intentionally) |

**Critical gap check:** No path has all three of (no test AND no error handling AND silent failure). The runner's "validator binary missing" path has no dedicated test but has explicit error handling and is loud (TEMPLATE_BUG message). Acceptable.
