## Problem Statement

F000004 work-copilot is the Copilot-native port of F000003 company-workflow,
intended to give the Windows + VS Code + Copilot work box the same workflow
tooling Claude Code provides at home. F000004 v1 (shipped v0.14.0, PR #43)
covers the validator core — templates + manifest + `validate.prompt.md` +
`copilot-instructions.md` + `scripts/copilot-deploy.py` — but stops short of
full bundle parity. The bundle is missing:

- `WORKFLOW.md` — 21 KB procedural backbone of the skill (phase guidance,
  scaffolding rules); the bundle has no analog
- `reference/` — 7 guides (`guide-architecture.md`, `guide-general.md`,
  `guide-prd.md`, `guide-rca.md`, `guide-review-notes.md`, `guide-task.md`,
  `guide-test-spec.md`)
- `philosophy/` — 3 rationale notes (`rationale-ARCHITECTURE.md`,
  `rationale-PRD.md`, `rationale-TEST-SPEC.md`)
- `examples/` — 14 example artifacts (5 trackers + 9 doc types: ARCHITECTURE,
  PRD, RCA, TEST-SPEC, feature-summary, milestones, review-notes, scrum,
  test-plan)
- Test fixtures, 5 changes total (verified by `cmp -s` in plan-eng-review):
  3 missing flat files (`invalid-bad-frontmatter.md`,
  `invalid-missing-lifecycle.md`, `invalid-wrong-order.md`); 1 missing
  nested file (`valid-feature-dir/DESIGN.md`); 1 currently-drifted file
  (`valid-feature-dir/TRACKER.md`, bundle has older content than source).
  The other 3 nested fixture files (`invalid-missing-artifact-dir/TRACKER.md`,
  `valid-feature-dir/feature-summary.md`, `valid-feature-dir/milestones.md`)
  already match.
- The entire knowledge-integration subsystem — `$AI_KNOWLEDGE_DIR` env-var
  seam, two-tier surfacing (always/on-demand), `bin/knowledge-helpers.sh`,
  `knowledge-doctor` diagnostic. (`bin/` itself is shell-only and does not
  port to Copilot regardless of the knowledge-integration redesign — it
  goes away when the redesigned subsystem ships, replaced by instructions
  + static indexes.)

Work-box users get a narrower workflow than home-box users, and the
F000004_DESIGN.md (v1, Status: Draft, dated 2026-04-22) does not document
the gap or the path to closing it.

This realignment plan closes the easy gaps — bundle artifact completeness —
and explicitly defers knowledge integration to a follow-up feature where it
can get a real Copilot-native design pass (no shell execution, no env-var
resolution at prompt time, helpers redesigned as instructions or static
indexes).

## What Makes This Cool

After realignment, the same `python scripts/copilot-deploy.py install
<target>` drop into `<target>/.github/` gives a Copilot user the same
surface area a Claude Code user gets on their home box: scaffolding cues
from examples, design rationale from philosophy notes, deeper how-to from
reference guides, and full fixture coverage for validator self-testing.
One bundle, two runtimes, zero functional asymmetry on the
artifact-completeness axis. The work-box workflow stops being the
"second-class" experience.

## Constraints

- **No shell execution at prompt time.** Copilot has no bash. All bundle
  behavior must be expressible as instructions the model follows, plus
  static files it reads. As a direct consequence, `skills/company-workflow/bin/`
  (currently `knowledge-helpers.sh`) does not get a Copilot mirror — shell
  helpers are useless on the work box.
- **Templates already byte-identical.** `validate.sh` Error check 10
  (file lines ~165–190) enforces sync between
  `templates/company-workflow/*.md` and `work-copilot/templates/*.md`. Do
  not modify templates; the realignment extends Error check 10 to iterate
  a config-driven list of mirror dirs (single check, multiple dirs) so
  `WORKFLOW.md`, `reference/`, `philosophy/`, `examples/`, and missing
  fixtures all get the same pattern.
- **Windows + corporate proxies.** Installer is `python3` stdlib only (no
  pip), no bash branching. Already true; preserve it.
- **`copilot-instructions.md` ≤ 8 KB budget.** Current size: 5158 bytes
  (verified `wc -c`). Headroom for pointer paragraphs to `WORKFLOW.md`,
  `reference/`, `philosophy/`, `examples/` is comfortable
  (~3 KB available). Where the budget is *enforced* in CI is an open
  question — settle in S000010 PRD by either pointing at the existing
  check or adding one.
- **F000004 is in-flight, not closed.** v0.14.0 shipped two of three child
  stories (S000007, S000008); S000009 has 1 outstanding AC (live E2E on
  Windows box). Realignment expands the feature in-place; it does not
  spawn a new feature ID. The S000009 Windows-E2E AC is independent of
  this realignment and keeps progressing in parallel.
- **Existing F000004 milestones are stale.** `F000004_milestones.md` shows
  #1–#8 all "Not Started" as of 2026-04-23, but v0.14.0 already shipped
  per the F000004_TRACKER Journal entry of 2026-04-25. Reconcile in the
  Next-Steps milestone update — mark #1–#6 Done before adding the
  realignment milestones.
- **Knowledge integration deferred.** Out of scope for this realignment per
  D1. Becomes its own follow-up feature when it gets a real Copilot-native
  design pass. `bin/knowledge-helpers.sh` is part of that follow-up's
  redesign — it has no port path that lives outside knowledge integration.
- **Personal-workflow validation governs new scaffolding.** F000004's own
  Phase 3 invokes `/personal-workflow check`. Therefore S000010 and
  T000XXX use `templates/personal-workflow/` templates
  (`tracker-user-story.md`, `tracker-task.md`, `doc-PRD.md`,
  `doc-ARCHITECTURE.md`, `doc-TEST-SPEC.md`), not
  `templates/company-workflow/`.

## Premises

(Confirmed in D2.)

1. `skills/company-workflow/` is the source of truth for what "the
   workflow does." The work-copilot bundle should mirror its scope,
   adapting only the runtime surfaces that don't exist in Copilot
   (`.prompt.md` files instead of slash commands, `.github/` install path
   instead of `~/.claude/`, instructions-as-prose instead of bash blocks).
2. Knowledge integration is out of scope for this realignment. It needs
   its own Copilot-native design pass and ships as a follow-up feature.
3. Templates stay byte-identical. We add new mirror artifacts
   (`WORKFLOW.md`, `reference/`, `philosophy/`, `examples/`, missing
   fixtures) and extend `validate.sh` Error check 10 to iterate a
   config-driven list of mirror dirs (single composite check, not four
   sibling checks).
4. The deliverable for this `/office-hours` run is this realignment
   design doc. The user transcribes/promotes it into a v2 of
   `F000004_DESIGN.md` afterward.
5. Realignment expands F000004 in-place. New child stories/tasks attach
   to F000004's tracker; the feature ID stays the same.

## Approaches Considered

### Approach A: Decompose by artifact type — 4 stories + 1 task

Maximum tracking granularity. Each artifact category (reference,
philosophy, examples, fixtures) gets its own story with PRD +
ARCHITECTURE + TEST-SPEC. Sync-check extension is a separate task.
Stories can be picked up in parallel; bisection is granular.

Trade-off: 4 near-identical PRDs for what is structurally one mirror
operation (copy from `skills/company-workflow/X/` to `work-copilot/X/`,
register in sync check). Design surface multiplied without signal.

### Approach B: Decompose by concern — 1 story + 1 task (chosen)

Bundle artifact completeness is one story with one PRD/ARCHITECTURE/
TEST-SPEC covering all mirror artifacts (because the design IS the same
for all of them — copy + register in sync check). Sync-check extension
is one task. Mirrors F000003's actual decomposition shape — F000003 has
2 stories total (`S000003_company_workflow_implementation` +
`S000004_knowledge_integration`), not one per template directory.

The sync-check task is mostly mechanical — extending an existing pattern,
not designing a new one. It exists for trackability and a clean PR
boundary (story = bundle expansion, task = CI enforcement), not because
it carries non-trivial design surface.

### Approach C: Single unit — 1 task only

No story-level scaffolding. Treats the whole realignment as one
bundle-completion task. Lightest scaffolding, smallest PR.

Trade-off: tasks don't carry PRD/ARCHITECTURE in the personal-workflow
manifest. The bundle loses the structural "why these dirs exist" docs
Copilot reads on the work box, defeating part of the realignment's
purpose.

#### T000XXX_validate_sync_check_extension (task)

Extends `scripts/validate.sh` Error check 10 to a config-driven
`MIRROR_SPECS` array (locked in plan-eng-review D3) so that
`work-copilot/WORKFLOW.md` and
`work-copilot/{templates,reference,philosophy,examples,fixtures}/*`
(recursive) are byte-identical to the corresponding paths under
`skills/company-workflow/` (templates source is `templates/company-workflow/`).

Pattern (locked):

```bash
# Each entry is "src-path-or-pattern:dst-path-or-pattern".
# - A path with no glob = single-file mirror.
# - SRC/*.md:DST/*.md  = flat directory, *.md only.
# - SRC/**/*.md:DST/**/*.md = recursive directory, *.md only (handles
#   nested fixture dirs like valid-feature-dir/).
# Explicit file-name filtering (locked in plan-eng-review C1) keeps OS
# junk out of the check (.DS_Store on macOS, Thumbs.db on Windows).
MIRROR_SPECS=(
  "templates/company-workflow/*.md:work-copilot/templates/*.md"
  "skills/company-workflow/WORKFLOW.md:work-copilot/WORKFLOW.md"
  "skills/company-workflow/reference/*.md:work-copilot/reference/*.md"
  "skills/company-workflow/philosophy/*.md:work-copilot/philosophy/*.md"
  "skills/company-workflow/examples/*.md:work-copilot/examples/*.md"
  "skills/company-workflow/fixtures/**/*.md:work-copilot/fixtures/**/*.md"
  "skills/company-workflow/company-artifact-manifests.json:work-copilot/copilot-artifact-manifests.json"
)
```

Manifest pair sync (locked in plan-eng-review D4): both manifests get the
same `description` field naming both audiences ("Read by company-workflow
validate (Claude Code) and validate.prompt.md (Copilot). Mirrored, sync
enforced by scripts/validate.sh."). Filenames stay different
(`company-artifact-manifests.json` vs `copilot-artifact-manifests.json`)
because the file paths are part of each runtime's contract; the content
is byte-identical after the description unification.

The check loops the array, branches on the `/*` suffix, and runs
`cmp -s` per file. Same binary comparison the existing check uses
(re: D000005 — CRLF on Windows would otherwise make checksums flap).
The existing counterpart-warning loop runs the same way (find files in
bundle that have no upstream — warns rather than fails).

Future mirror dirs (e.g., when knowledge integration ships) get added
to the bundle as one new line in `MIRROR_SPECS`, no other changes.

Required scaffolding (per `templates/personal-workflow/`):

- `T000XXX_TRACKER.md` (from `tracker-task.md`)
- `T000XXX_PRD.md` (short — a few sentences plus the new check's match
  pattern). Plus: T000XXX_TEST-SPEC.md (locked in plan-eng-review D5):
  **9 negative-path synthetic cases** in `scripts/test.sh`. Build a
  tmpdir with a stripped Error check 10 callable, then for each of 3
  shapes (single-file, flat-glob, recursive-glob) × 3 failure modes
  (drift / missing-file / orphan), assert the check exits non-zero with
  the correct error string. Plus one happy-path tmpdir case (all in
  sync → exits 0). Total ~150 lines of test bash; reuses the existing
  `mktemp -d -t ...` + post-test cleanup pattern (see lines 1454-1510
  for the round-trip test scaffolding to mimic).

Task ID is TBD — assigned at scaffold time based on next-available
T-number in the repo.
