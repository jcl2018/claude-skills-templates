## Problem Statement

After F000030 (PR #180) added `doc/PHILOSOPHY.md` + `doc/ARCHITECTURE.md`, F000032 (PR #186) added per-skill `USAGE.md` (operator/agent best-practice), and F000033 (PR #188) added Check 14 (USAGE.md content freshness vs SKILL.md), there is still no consolidated workbench-level surface where a reader can scan ALL skills at a glance: name, status, when-to-invoke, and a visual workflow chart. The decision tree in PHILOSOPHY.md is a one-line-per-skill routing index — useful for "which skill?" but not "how does this skill flow?"

The user's framing is two-part:
1. Each skill gets a section in a consolidated `doc/` file with an ASCII workflow chart (the "verdict" the user sees at a glance).
2. All files in `doc/` are tracked + enforced re-update on `/document-release` (extending F000030's audit-conventions pattern).

Both parts are natural extensions of the convention-first pattern this workbench keeps reinforcing: hand-written, ERROR-strict validate.sh-enforced, audited at commit time + on `/document-release`.

## What Makes This Cool

This is the workbench's first **catalog**-shaped doc — not "principles" (PHILOSOPHY), not "mechanisms" (ARCHITECTURE), not "how to use" (per-skill USAGE.md). It is "scan-everything-at-once." The ASCII workflow chart per skill is what makes it concrete: a reader who has never seen `/CJ_goal_feature` can see the topic → office-hours → silent-build → /ship → PR chain in 10 lines and know what they're about to invoke. The chart already exists informally in each orchestrator's SKILL.md `## Overview` section — this PR consolidates and standardizes it.

The bonus: the tracked-doc/ manifest closes the F000030 doc-audit story. Currently F000030 names two doc/ files explicitly (PHILOSOPHY, ARCHITECTURE) but provides no convention for what happens when a third file is added. This PR establishes the pattern: any new file in `doc/` must be registered in the manifest with a drift criterion. /document-release reads the manifest and audits accordingly. Future doc/ additions just append one entry.

## Constraints

- **Workbench-only scope.** Per memory `feedback_workbench_scope`, no surfacing of downstream-consumer (work-copilot/, portfolio) loose ends.
- **Hand-written, NOT auto-generated.** Auto-generating from SKILL.md/USAGE.md is brittle (parsing prose, ASCII charts not derivable). Approach C in the alternatives AUQ was rejected for the same reason F000032 rejected the auto-gen path. Operator writes each section; Check 15 enforces structure + completeness.
- **Audit predicate matches F000032/F000033.** `jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name'` — routable non-deprecated skills only. 11 skills (3 active + 8 experimental). Re-use the predicate; do not fork.
- **No upstream `/document-release` modification.** Per memory `project_workbench_auto_deploy_unsafe`, upstream skills are not ours to edit. The integration is via extending the CLAUDE.md `/document-release workbench audit conventions` section, which `/document-release` reads as project context (per F000030's established pattern).
- **ERROR severity, not WARN.** Per memory `feedback_skill_contracts_strict` + the F000030/F000032/F000033 precedent: WARN gets ignored; ERROR-with-cheap-override (or ERROR-on-missing-section) is the load-bearing pattern in this workbench.
- **ASCII chart is mandatory for orchestrators, optional for single-step skills.** The four CJ_goal_* + CJ_personal-pipeline are orchestrators with natural chains (already documented in SKILL.md ## Overview). Single-step skills (CJ_personal-workflow validator; CJ_system-health/CJ_suggest/CJ_improve-queue utilities; CJ_scaffold/CJ_implement/CJ_qa phase-step skills) tag themselves with `(single-step utility)` or `(phase-step in /CJ_goal_feature chain)` in lieu of a chart. Check 15 enforces presence of EITHER a chart OR the explicit-no-chart tag — never silent omission.
- **Manifest format is YAML in a fenced block inside CLAUDE.md.** Same shape pattern as F000030's `EXPECTED_BUNDLE_FILES` array for work-copilot, but inline-parsed by validate.sh. Each entry: `path`, `audit_class` (one of `skill-routing-drift` / `skill-catalog-completeness` / `static-reference` / `auto-generated`), `owner` (free-text who maintains it). Three known entries land in v1: PHILOSOPHY.md (`skill-routing-drift`, F000030), ARCHITECTURE.md (`skill-routing-drift`, F000030), SKILL-CATALOG.md (`skill-catalog-completeness`, this PR).
- **No backfill for `auto-generated` audit_class in v1.** The audit class is reserved for future docs that a script regenerates; v1 has none. Listed in the enum to leave the door open.
- **Tracked-doc/ orphan check.** Check 15's second half: every `doc/*.md` file on disk must be registered in the manifest. New files added to doc/ without a manifest entry ERROR out, forcing the author to declare an audit class. This is the load-bearing extensibility primitive — future doc surfaces can't slip in unannotated.

## Recommended Approach

**Approach A**, hand-written catalog + tracked-doc/ manifest. Detailed plan:

### Step 1: Create the section template

Write `templates/doc-SKILL-CATALOG-section.md` with the per-skill section structure:

```markdown
### {name}

**Status:** {active | experimental} ({short rationale})
**Source:** `skills/{name}/SKILL.md` · `skills/{name}/USAGE.md`

**Invoke when:** {one-to-two-line distilled invocation pattern, copied from USAGE.md ## When to use first paragraph}

**Workflow:**

```
{ASCII chart for orchestrators / phase-step skills}
```

OR (for single-step skills, mandatory replacement for the chart):

`{single-step utility | validator | phase-step in /CJ_goal_feature chain}` — {one-line behavioral summary}
```

Each instruction is inline so the author knows what to write.

### Step 2: Backfill 11 skill sections

Write `doc/SKILL-CATALOG.md` with:

- Header: title + one-paragraph description ("This is the consolidated index of every routable non-deprecated skill in this workbench. Sections are hand-written and audited by `scripts/validate.sh` Check 15.")
- One section per routable skill, grouped by role:
  - **Orchestrators (ASCII chart mandatory):** CJ_goal_feature, CJ_goal_defect, CJ_goal_todo_fix, CJ_personal-pipeline
  - **Phase-step skills (tag mandatory):** CJ_scaffold-work-item, CJ_implement-from-spec, CJ_qa-work-item
  - **Validators / utilities (tag mandatory):** CJ_personal-workflow, CJ_system-health, CJ_suggest, CJ_improve-queue
- Footer: link to PHILOSOPHY.md ## Decision tree (for the per-skill routing rules) + USAGE.md per skill (for operator best-practice).

Each section is 25-40 lines.

ASCII charts extracted/distilled from each orchestrator's existing SKILL.md ## Overview section (already documented for CJ_goal_feature, CJ_goal_defect, CJ_personal-pipeline). For CJ_goal_todo_fix the chart is the same shape as CJ_goal_feature with TODOS:NNN as the input.

### Step 3: Extend CLAUDE.md `/document-release workbench audit conventions` section

Append a new subsection **after** the `### New-skills check` and **before** the `### Reporting` subsection:

```markdown
### Tracked doc/ files manifest

Every `*.md` file under `doc/` MUST be registered in this manifest with an `audit_class`. `validate.sh` Check 15 fires ERROR for any orphan file (in `doc/` but not in the manifest) and for any manifest entry pointing to a missing file. Adding a new doc/ file is intentional; declaring its audit class is the cost of admission.

```yaml
- path: doc/PHILOSOPHY.md
  audit_class: skill-routing-drift
  owner: F000030 — workbench-level overview + decision tree
- path: doc/ARCHITECTURE.md
  audit_class: skill-routing-drift
  owner: F000030 — mechanism reference
- path: doc/SKILL-CATALOG.md
  audit_class: skill-catalog-completeness
  owner: F000034 — consolidated per-skill catalog with ASCII workflow charts
```

`audit_class` enum (closed):

- `skill-routing-drift` — F000030 retired-skill + new-skills check (already applied to PHILOSOPHY.md + ARCHITECTURE.md)
- `skill-catalog-completeness` — every active routable skill has a section AND every section names a real skill (Check 15)
- `static-reference` — file is hand-written reference content; audit only checks the file exists. Reserved for future docs whose drift criteria the author hasn't worked out yet.
- `auto-generated` — file is regenerated by a script; audit checks `script-output == on-disk content`. Reserved; v1 has no entries.

`/document-release` reads this manifest as project context (the existing F000030 pattern) and surfaces drift findings in the PR body's `## Documentation` section under a new `### Doc/ manifest drift` subheading, alongside the existing `### Skill-routing drift` subheading.
```

The `### Reporting` subsection (existing) gets one additional line: "Doc/ manifest drift findings (Check 15) appear under `### Doc/ manifest drift`. One finding per line; positive `Doc/ manifest drift: none` if clean."

### Step 4: Add Check 15 to scripts/validate.sh

Insert after Check 14 (currently the last check at line ~600 post-F000033):

```bash
# Check 15: doc/ manifest + SKILL-CATALOG.md completeness
# F000034: every doc/*.md must be registered in the CLAUDE.md manifest; the
# SKILL-CATALOG.md entry must list every active routable skill with at least
# one of (a) a fenced ASCII workflow chart, or (b) an explicit tag line
# matching one of `(single-step utility)` / `(validator)` / `(phase-step in
# /CJ_goal_feature chain)`. No silent omission.
echo ""
echo "=== Check 15: doc/ manifest + SKILL-CATALOG.md completeness ==="

# 15a: enumerate manifest entries (parse the YAML block under
# `### Tracked doc/ files manifest` in CLAUDE.md)
MANIFEST_PATHS=$(awk '/^### Tracked doc\/ files manifest$/,/^### /{if($0 ~ /^- path:/) print $3}' CLAUDE.md)
DOC_FILES_ON_DISK=$(find doc -maxdepth 1 -type f -name '*.md' 2>/dev/null | sort)

# 15a-orphan: doc/ files on disk that aren't in the manifest
for f in $DOC_FILES_ON_DISK; do
  if ! echo "$MANIFEST_PATHS" | grep -qFx "$f"; then
    echo "  ERROR: $f is in doc/ but not registered in CLAUDE.md tracked-doc/ manifest"
    ERRORS=$((ERRORS+1))
  fi
done

# 15a-missing: manifest entries pointing to missing files
for p in $MANIFEST_PATHS; do
  if [ ! -f "$p" ]; then
    echo "  ERROR: $p is in CLAUDE.md manifest but missing from disk"
    ERRORS=$((ERRORS+1))
  fi
done

# 15b: SKILL-CATALOG.md per-skill completeness (only if catalog file exists)
CATALOG_FILE="doc/SKILL-CATALOG.md"
if [ -f "$CATALOG_FILE" ]; then
  TAG_RE='^\((single-step utility|validator|phase-step in /CJ_goal_feature chain)\)'
  while IFS= read -r SKILL_NAME; do
    # Section heading is `### <name>` line-anchored
    if ! grep -qE "^### ${SKILL_NAME}$" "$CATALOG_FILE"; then
      echo "  ERROR: $CATALOG_FILE missing section: ### $SKILL_NAME"
      ERRORS=$((ERRORS+1))
      continue
    fi
    # Extract the section body (between this ### and the next ###)
    SECTION=$(awk "/^### ${SKILL_NAME}$/,/^### /" "$CATALOG_FILE" | sed '/^### /d')
    # Section must have EITHER a fenced ``` block (ASCII chart) OR a tag line
    HAS_CHART=$(echo "$SECTION" | grep -cE '^```' || true)
    HAS_TAG=$(echo "$SECTION"   | grep -cE "$TAG_RE" || true)
    if [ "$HAS_CHART" -lt 2 ] && [ "$HAS_TAG" -lt 1 ]; then
      echo "  ERROR: $CATALOG_FILE section '$SKILL_NAME' has neither ASCII chart (fenced block) nor a tag line; one is required"
      ERRORS=$((ERRORS+1))
    else
      echo "  PASS: $CATALOG_FILE has section for $SKILL_NAME"
    fi
  done < <(jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json)
fi
```

Notes:
- The manifest parser uses awk-range matching the YAML's H3 heading; that's good enough for a small inline manifest (no real YAML parser needed; this is hand-written content).
- The catalog-completeness check is scoped under `if [ -f "$CATALOG_FILE" ]` so the check doesn't fire ERROR before the file lands (defensive against test-mode runs that might not have the catalog yet — though for this PR, the file lands in the same atomic commit as the check).
- HAS_CHART threshold is 2 (one opening ` ``` ` and one closing ` ``` `).
- HAS_TAG threshold is 1 (one tag line).

## Open Questions

1. **Should phase-step skills (CJ_scaffold-work-item, CJ_implement-from-spec, CJ_qa-work-item) get full ASCII charts or just the tag?** Resolution: tag-only, since they're called transitively by orchestrators and their "chart" is just one rectangle. The tag `(phase-step in /CJ_goal_feature chain)` is more useful than a one-box chart.
2. **Should the manifest also enforce update-on-/document-release dates?** No. F000033's drift check pattern (mtime-equivalent via git log -1) is per-skill; expanding to per-doc would require a per-file drift criterion which is brittle. The audit_class enum already separates static-reference (no drift check) from skill-catalog-completeness (structural drift only); content freshness across the doc as a whole is genuinely hard to define. Defer indefinitely.
3. **Should SKILL-CATALOG.md replace PHILOSOPHY.md ## Decision tree?** No. They serve different readers: decision tree = routing decision (which skill?); catalog = workflow understanding (what does the skill do?). Both stay. PHILOSOPHY.md gets a one-line reference to SKILL-CATALOG.md in the Decision tree intro.
4. **Should the manifest live in a separate file (e.g., `skills-catalog.json` analog: `doc-catalog.json`) instead of inline in CLAUDE.md?** Considered. CLAUDE.md inline is simpler for v1 (one file to read; awk-parseable). Separate JSON would be cleaner for tool consumption but adds a file. Defer — if a future tool needs structured parsing, hoist then.
5. **What about ARCHITECTURE.md's per-skill mentions? Are they audited by Check 15?** No. Check 15 audits SKILL-CATALOG.md's per-skill completeness (specifically). ARCHITECTURE.md's audit is the existing F000030 skill-routing-drift check (retired-skill mentions outside the right section). The two audits are orthogonal.
6. **Backwards-compat: do deprecated shims need catalog sections?** No. The predicate `status != "deprecated"` excludes them, same as F000032 + F000033. The 4 deprecated shims (`CJ_goal_run`, `CJ_goal_auto`, `cj_goal_feature`, `cj_goal_defect`) + `CJ_goal_investigate` are not in the audit set.
