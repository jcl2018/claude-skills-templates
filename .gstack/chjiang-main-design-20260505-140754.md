## Problem Statement

The personal-workflow skill's tracker templates (`templates/personal-workflow/tracker-*.md`) and artifact set (`personal-artifact-manifests.json`) drifted away from the user's actual day-to-day workflow. The user's stated workflow is:

1. `/office-hours` → produces a design plan in `~/.gstack/projects/`
2. `/personal-workflow` templates → scaffold structure docs (the locked spec)
3. Implement
4. Tests: smoke (automated, never re-edited) + E2E (manual, walked once before ship)
5. `/ship` → creates PR; bundled internal pre-landing review
6. `/land-and-deploy` → merges + verifies

Three concrete gaps in the current trackers:

1. **Mental-model misordering of `/office-hours`.** `tracker-feature.md:20` and `tracker-user-story.md:19` list `/office-hours` as step 1 *inside* Phase 1 — implying the work-item TRACKER exists before `/office-hours` runs. In practice, `/office-hours` is the entry point and the design doc precedes the work item.

2. **Smoke vs. E2E not surfaced as separate gates.** Today's TEST-SPEC restructure (shipped on `claude/cool-payne-300ef9`) split `## Smoke Tests` and `## E2E Tests` into two distinct cognitive modes (CI-resident automation vs. manual pre-ship walkthrough). Phase 3 of the trackers still says `[ ] TEST-SPEC verified`, which collapses both modes back into one gate and loses the "you stop touching one, you sit down and run the other" affordance.

3. **Artifact redundancy and over-decomposition.**
   - `feature-summary.md` (scope, success criteria, constituent stories, non-goals) overlaps with `TRACKER.md` fields (`name`, Acceptance Criteria, Todos with child links).
   - `milestones.md` is a standalone timeline doc that often sits empty or duplicates info in DESIGN.md / feature-summary.md.
   - `PRD.md` and `ARCHITECTURE.md` for user-stories are written and edited together as one logical "lock the spec" act; the user's phrasing "lock *the* structure doc" (singular) reflects that.

## What Makes This Cool

The re-cut produces an artifact set where **every persistent doc maps 1:1 to a workflow step that produces it**. You stop thinking "which template does this artifact come from?" and start thinking "which skill produces this doc?":

- `DESIGN.md` ← `/office-hours` (condensed local distillation of the `~/.gstack/projects/` transcript)
- `SPEC.md` ← scaffold step (the locked structure doc — requirements + architecture in one place)
- `TEST-SPEC.md` ← scaffold step (smoke + E2E sections — today's shape)
- `ROADMAP.md` ← scaffold step (feature-level decomposition + delivery timeline)
- `RCA.md` ← `/investigate` or hand-written (defects only)
- `test-plan.md` ← scaffold step (lighter than TEST-SPEC, for tasks/defects where 5 smoke + 5 E2E rows would be theater)

The forcing function: if a workflow step doesn't produce a doc, no template exists for it. `/review` becomes invisible at the artifact layer (bundled into `/ship`'s internal diff review per the user's premise call), so no template exists for it. Implementation produces code, not docs, so no template. The artifact set becomes a literal map of "the docs we keep."

A second affordance: `feature` and `user-story` both gain a `DESIGN.md` requirement (currently only `feature` has it). This makes the `/office-hours → distill into DESIGN.md` ritual explicit at the user-story level too, where today it's implicit.

## Constraints

- **Personal-workflow only.** Company-workflow lives at `deprecated/company-workflow/` and is the byte-mirror source for `work-copilot/` (see [validate.sh](scripts/validate.sh) Error check 10's `MIRROR_SPECS`). This re-cut does NOT touch company-workflow templates, manifests, or `work-copilot/`. Mirroring the new shape into a future personal-copilot bundle is out of scope.
- **No backward-compat / legacy mode in `check.md`.** Per today's TEST-SPEC pattern: validator becomes properly strict on the new shape; all 13 historical work items migrate in the same PR.
- **Trackers are LLM-driven, not bash.** Per `validators-may-be-llm-not-bash` learning (10/10, Apr 16): `/personal-workflow` is a SKILL.md instruction file read by Claude Code. Template content rules are derived from templates at runtime by the LLM, not by a bash parser. No new executable scripts needed.
- **Templates remain the single source of truth.** Per `personal-workflow-restructure` learning (10/10, Apr 16): no separate `contract.json`. Every structural rule (frontmatter fields, required sections, lifecycle phases, minimum checkbox counts) is derived from the matching template.
- **Single sweep PR.** No two-phase split, no permanent legacy tolerance.

## Premises (all confirmed)

1. **Phase mapping** — Track = `/office-hours` + scaffold + lock SPEC; Implement = code + smoke (CI) + E2E (manual); Ship = `/ship` (with bundled pre-landing review) + `/land-and-deploy`.
2. **`SPEC.md` replaces `PRD.md` + `ARCHITECTURE.md`** for user-stories. One doc holds: requirements, acceptance criteria, architecture decisions, tradeoffs.
3. **`ROADMAP.md` replaces `feature-summary.md` + `milestones.md`** for features. One doc holds: scope/non-goals, story decomposition, delivery timeline.
4. **`DESIGN.md` is a condensed local distillation** of the `/office-hours` output. The full transcript stays in `~/.gstack/projects/`. Required for feature + user-story; absent for task and defect.
5. **`/review` is bundled into `/ship`'s internal diff review** — not a separate Phase 3 gate. Phase 3 has 4 gates: smoke pass, E2E walked, `/ship`, `/land-and-deploy`.
6. **Sweep migration** — all 13 historical work items migrated in same PR. No legacy mode in `check.md`. Same scorched-earth approach as today's TEST-SPEC restructure.

## Approaches Considered

### Approach A: Single sweep PR (recommended; chosen)
Everything in one PR — new templates, deleted templates, manifest update, `check.md` updates, `WORKFLOW.md`, all 13 historical migrations, fixtures, examples. `/personal-workflow check` flips from old-strict to new-strict in one commit. No broken intermediate state, no legacy code in `check.md`.

### Approach B: Two-phase (foundation, then migration)
PR 1 lands templates + manifest + `check.md` updates; validator becomes strict on new shape, so all historical work items show DRIFT/MISSING. PR 2 migrates the 13 historical items. Smaller per-PR review surface but the repo is "broken" between the two PRs — anyone running `/personal-workflow check` sees walls of red.

### Approach C: Foundation + permanent legacy tolerance
One PR like A, but `check.md` keeps treating old-shape work items as `[LEGACY]` WARN (not VIOLATION) indefinitely. Historical items migrate lazily as touched. Smallest PR but introduces permanent dual-shape complexity in `check.md` (which the user explicitly rejected for TEST-SPEC today).
