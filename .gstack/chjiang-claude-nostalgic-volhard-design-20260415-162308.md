## Problem Statement

The company-workflow skill was designed with 3 subcommands: `validate` (shipped), `check` (T000005, not implemented), `create` (T000006, not implemented). The `check`/`create` approach is over-engineered. The skill should be a delivery package: templates + reference guides + philosophy docs + a single `validate` command that handles both file-level and directory-level validation. No YAML-based check/create subcommands.

## What Makes This Cool

One skill, one command, complete company spec enforcement. `validate` a single tracker file or an entire work item directory. The templates ARE the spec. The guides ARE the reference. The manifest IS the truth. Everything else is validation.

- Zero gstack dependencies (no analytics, no /review, no /docs check references)

## Premises

1. Company-workflow has ONE subcommand: `validate`. File path = structural check (contract.json). Directory path = whole work item check (artifact completeness, frontmatter, lifecycle). No separate check/create.
2. Validation reads company-specific config: company-artifact-manifests.json (NEW), contract.json (exists), 4-phase lifecycle, 5 types (feature, defect, task, userstory, review).
3. The skill is a delivery package: templates + guides + philosophy + validation spec + config files.
4. Delivery means updating SKILL.md, CLAUDE.md, skills-catalog.json, template-registry.json — all must be consistent.
5. T000005 and T000006 work items are killed (deleted or closed-as-won't-do).

## Validate Command Spec (Directory Mode)

When `validate` receives a directory path, it validates the **immediate directory only** (no recursive descent into child directories). To validate a feature and its children, run validate on each directory separately.

### Filename Matching Rule

Strip the leading ID prefix (regex `^[A-Z]\d+_`) to get the canonical filename, then compare against the manifest's `filename` field. Examples:
- `S000003_PRD.md` -> strip `S000003_` -> `PRD.md` (matches manifest)
- `F000003_TRACKER.md` -> strip `F000003_` -> `TRACKER.md` (matches manifest)
- `T000002_test-plan.md` -> strip `T000002_` -> `test-plan.md` (matches manifest)

### Template Frontmatter Comparison

To validate an artifact's frontmatter against its template:
1. Resolve the template file from `$_TMPL_DIR/{template}` (using the 2-level fallback chain)
2. Parse the template's YAML frontmatter to extract its key names
3. Parse the artifact's YAML frontmatter
4. For each key present in the template's frontmatter: check that the same key exists in the artifact. Comparison is key-presence only (values contain placeholders in templates).

### Directory Mode Error Handling

- No TRACKER.md found: `"Error: no TRACKER.md found in {directory}. Not a work item directory."`
- company-artifact-manifests.json missing: `"Error: company-artifact-manifests.json not found at {path}. Run skills-deploy install or check skill structure."`
- Template file missing during frontmatter comparison: `"Warning: template {filename} not found at {path}. Skipping frontmatter validation for {artifact}."`

### Steps

Path Resolution runs first (same as file mode). `$_SKILL_DIR` and `$_TMPL_DIR` are available for config and template lookup throughout directory mode.

1. **Locate TRACKER.md** — Find `*_TRACKER.md` or `TRACKER.md` in the directory
2. **Read type** — Parse frontmatter `type` field, normalize spelling (userstory/user-story both accepted). Verify type is one of the 5 known types (feature, defect, task, userstory, review). If unknown: `[WARN] — type "{value}" not recognized`
3. **Load manifest** — Read company-artifact-manifests.json, find the type entry
4. **Check artifact completeness** — For each required artifact in the manifest:
   - Match files in directory using the Filename Matching Rule above
   - If missing: `[MISSING] {artifact} — required artifact not found`
   - If found: validate frontmatter using the Template Frontmatter Comparison above
   - Missing key: `[DRIFT] {artifact} — missing required field "{field}"`
5. **Check lifecycle** — Read TRACKER.md lifecycle section:
   - Verify all 4 phases exist (Track, Implement, Review, Ship)
   - Verify min 4 checkboxes total (across all phases) per contract.json
   - Verify required sections exist per contract.json
   - Note: section ordering is checked in file mode (per contract.json expected_order), not in directory mode
6. **Report** — Emit structured output:
   ```
   COMPANY-WORKFLOW VALIDATE: {directory}
     Type: {type}
     ARTIFACTS:
       [PASS]    TRACKER.md — all required fields present
       [PASS]    PRD.md — all required fields and sections present
       [MISSING] test-plan.md — required artifact not found
       [DRIFT]   ARCHITECTURE.md — missing required field "repo"
     LIFECYCLE:
       [PASS]    4 phases present, 12 checkboxes
     SUMMARY: 4 artifacts checked, 1 missing, 1 drift
   ```

## Resolved Questions

1. **Recursion:** Directory mode validates the immediate directory only. No recursive descent. Run validate separately on each directory. This keeps the command simple and predictable.
2. **Hierarchy rules:** company-artifact-manifests.json does NOT include hierarchy or placement rules. Validate does not check parent-child nesting. That's /docs check territory. Keep the manifest minimal: type-to-artifact mapping only.
3. **allowed-tools:** Read-only (Bash, Read, Glob, Grep). Validate never modifies files.

## Known Gaps

1. **Handoff section:** contract.json lists `Handoff` as a required section, but existing company tracker templates do not include a `## Handoff` section.
