## Problem Statement

The workbench has two doc surfaces today: `doc/PHILOSOPHY.md` (workbench-level overview, design principles, decision tree, retired skills) and `doc/ARCHITECTURE.md` (mechanism reference), both shipped 8 days ago in F000030 (PR #180, v5.0.11). What is missing is a per-skill best-practice / "how to think about this skill" surface.

Today the per-skill doc surface is `skills/{name}/DESIGN.md`, declared **optional** by CLAUDE.md and supported by `templates/doc-SKILL-DESIGN.md` (Purpose / Behavior / Design Decisions / Dependencies / Security Boundaries / Test Criteria). Only `skills/CJ_system-health/DESIGN.md` exists; the other 10 routable skills have none. DESIGN.md is design-rationale shaped — useful for developers extending the skill — but it does not answer "when do I invoke this?" "when do I NOT?" "what is the mental model?" That gap is exactly what the user wants closed.

The user framing — "philosophy as an overview logic/best practice" + "create doc for each skill" — names two things: keep PHILOSOPHY.md as the workbench overview, and add a per-skill best-practice doc that mirrors that shape at the leaf level.

## What Makes This Cool

This is the smallest possible self-documentation feature that survives 12 months. It ships one new file class (USAGE.md), one new template, one new validate.sh check, and 11 backfill files (one per routable non-deprecated skill). Eight days from now there is no per-skill best-practice surface; thirty days from now there is one for every routable skill, audited at commit time. The skill's own SKILL.md (which the agent loads on every invocation) stays exactly as it is — the operator-facing best-practice prose lands in a sibling file that does not pay a per-turn token cost.

The bonus dogfood: USAGE.md is the doc the agent itself would want to read to know whether to suggest a skill. F000030's `/document-release` audit already validates that every active skill appears in `doc/PHILOSOPHY.md ## Decision tree` — USAGE.md fills out the "why" behind each decision-tree entry.

## Constraints

- **Workbench-only scope.** Per memory `feedback_workbench_scope`: do not surface downstream-consumer work (work-copilot/, portfolio repo) as loose ends. The audit runs on `skills/` only, not on `work-copilot/` or `deprecated/`.
- **Routable non-deprecated skills only.** Audit set = `jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json`. This is NOT the same predicate as F000030's "## New-skills check" (which uses `status=="active"`). Reason for the difference: F000030's predicate gates `doc/PHILOSOPHY.md ## Decision tree` placement, which is reserved for ship-stable (`status: active`) skills only. USAGE.md audits routability — operators and the agent route to `experimental` skills today (`CJ_goal_feature`, `CJ_goal_defect`, `CJ_scaffold-work-item`, etc.), so they need USAGE.md too. The 5 deprecated shims (`CJ_goal_run`, `CJ_goal_auto`, `CJ_goal_investigate`, `cj_goal_feature`, `cj_goal_defect`) and the tooling-only `templates` catalog entry (`files: []`) are excluded automatically by the predicate. Audit set = 11 skills (3 active + 8 experimental). Re-verified against catalog HEAD on 2026-06-01.
- **DESIGN.md stays optional.** USAGE.md and DESIGN.md are different doc shapes serving different readers: USAGE.md = operator/agent "when + why" (required), DESIGN.md = developer "how was this built and why" (optional, for skills complex enough to warrant the rationale doc). The 1 existing DESIGN.md (CJ_system-health) is unchanged by this PR.
- **PHILOSOPHY.md decision tree links must update.** Each active skill's decision-tree entry in `doc/PHILOSOPHY.md` gains a one-line link to its `skills/{name}/USAGE.md` so a reader who clicks through the tree lands on the per-skill best-practice. PHILOSOPHY.md's other sections (Why this repo exists, Design principles, Retired skills, etc.) are unchanged.
- **No new sunset wave.** USAGE.md is additive; no deprecation, no shims, no version coupling. The 4 deprecated shims (`CJ_goal_run`, `CJ_goal_auto`, `cj_goal_feature`, `cj_goal_defect`) and `CJ_goal_investigate` retired shim do not need USAGE.md — the audit predicate excludes them.
- **Audit is one check, not a check class.** validate.sh gets ONE new check: "every active routable skill has a sibling USAGE.md with the required H2 sections." Not a check per section. Implementation grep matches `^## When to use`, `^## When NOT to use`, `^## Mental model`, `^## Common pitfalls`, `^## Related skills` — five greps in a small loop. If any required H2 is missing, ERROR.
- **Template lives next to the existing one.** `templates/doc-SKILL-USAGE.md` lives sibling to `templates/doc-SKILL-DESIGN.md`. Both are top-level templates (not under `templates/CJ_personal-workflow/`) because they are workbench-wide skill-authoring templates, not per-workflow work-item templates.
- **No deploy surface.** Per-skill USAGE.md files live in `skills/{name}/USAGE.md` (in-repo). They are NOT deployed to `~/.claude/skills/{name}/USAGE.md` by `skills-deploy install` — the deployed path already has SKILL.md (the agent-facing file) and the agent does not need USAGE.md at runtime. USAGE.md is a human-reading file kept next to the skill source. The template `templates/doc-SKILL-USAGE.md` likewise stays in-repo only (not in `skills-catalog.json`'s `templates` entry's `files` array). Less to deploy = less drift.
- **No README.md changes for the audit logic.** `scripts/generate-readme.sh` auto-generates README.md from `skills-catalog.json`; the script does not need to know about USAGE.md. If the user later wants README.md to link each skill row to its USAGE.md, that is a follow-up TODOS row.

## Premises

(stated in chat above + confirmed via Approach B selection — re-listed for the record)

1. Scope is workbench-self-documentation (you, future-you, the agent), not external-user docs.
2. `doc/PHILOSOPHY.md` keeps its workbench-overview role. The new doc lives at the per-skill layer.
3. Per-skill docs must be enforced via validate.sh or they decay (F000030 added the template; 1/13 adoption proves this).
4. Audit scope is `status: active` skills with non-empty `files`. Deprecated shims and tooling-only catalog entries out of scope.
5. USAGE.md is a NEW file class, distinct from DESIGN.md. DESIGN.md stays optional.

## Approaches Considered

### Approach A: Lift existing DESIGN.md from optional to required

Use the existing `templates/doc-SKILL-DESIGN.md` (Purpose / Behavior / Design Decisions / Dependencies / Security Boundaries / Test Criteria). Backfill the 12 missing DESIGN.md files, change CLAUDE.md from "optional" to "required", add validate.sh audit. PHILOSOPHY.md stays as overview.

**Pros**: Reuses existing template (zero new convention); audit on existing file class.

**Cons**: DESIGN.md is design-rationale shaped, not "how to use" shaped — weaker semantic match to the user's phrasing ("philosophy as overview / best-practice"). Forces 12 backfill writes of a doc shape the user didn't ask for. Doesn't create the per-skill best-practice surface the user wants — just makes the existing developer-rationale doc mandatory.

### Approach B: New USAGE.md per skill (chosen)

Add a new required file `skills/{name}/USAGE.md` per active routable skill. Shape: When to use / When NOT to use / Mental model / Common pitfalls / Related skills. PHILOSOPHY.md's decision tree gains a link to each USAGE.md. DESIGN.md stays optional for design-rationale.

**Pros**: Direct semantic match to the user's "best-practice per skill" framing; clean shape split between USAGE.md (operator + agent) and DESIGN.md (developer); PHILOSOPHY.md decision tree becomes an actual index that resolves through to per-skill best-practice instead of dead-ending at a one-line description.

**Cons**: 13 new files to backfill (one per active routable skill); new template and new convention to document in CLAUDE.md; adds one validate.sh check.

### Approach C: Required `## Philosophy` section inside each SKILL.md

No new file. Add a required H2 section (`## Philosophy` or `## When to use`) to every active SKILL.md, ~10-20 lines. Audit checks the section exists. PHILOSOPHY.md keeps overview role.

**Pros**: Lowest blast radius (no new file class); philosophy lives next to execution instructions and cannot drift.

**Cons**: Every SKILL.md grows; SKILL.md is loaded into agent context on every skill invocation, so the operator-facing best-practice prose pays a per-turn token cost on every run; mixing operator-readable best-practice with agent-readable execution instructions in the same file blurs the audience.

## Recommended Approach

**Approach B**, new USAGE.md per skill. Detailed plan:

### Step 1: Create the template

Write `templates/doc-SKILL-USAGE.md` with five H2 sections (When to use / When NOT to use / Mental model / Common pitfalls / Related skills) plus minimal frontmatter (skill-name, version, last-updated). Frontmatter mirrors `templates/doc-SKILL-DESIGN.md` for visual consistency. Each section has a 2-3 line prompt explaining what belongs there.

### Step 2: Backfill 11 USAGE.md files

One USAGE.md per routable non-deprecated skill. Catalog-verified list (2026-06-01):

**Active (3):**
1. `CJ_system-health` (standalone utility — already has DESIGN.md, gets USAGE.md as a peer)
2. `CJ_personal-workflow` (workflow validator)
3. `CJ_goal_todo_fix` (top-level orchestrator)

**Experimental (8):**
4. `CJ_scaffold-work-item` (internal phase-step)
5. `CJ_qa-work-item` (internal phase-step)
6. `CJ_implement-from-spec` (internal phase-step)
7. `CJ_personal-pipeline` (internal orchestrator)
8. `CJ_suggest` (standalone utility)
9. `CJ_improve-queue` (standalone utility)
10. `CJ_goal_feature` (top-level orchestrator)
11. `CJ_goal_defect` (top-level orchestrator)

**Excluded by predicate** (do NOT get USAGE.md): `CJ_goal_run` + `CJ_goal_auto` + `CJ_goal_investigate` + `cj_goal_feature` + `cj_goal_defect` (all `status: deprecated`); `templates` (`files: []`, tooling-only).

Each file gets concrete content pulled from the skill's existing SKILL.md description + the routing rules in `rules/skill-routing.md`. Each file is short (40-80 lines; no hard cap enforced).

### Step 3: validate.sh audit check

Add **Check 13** to `scripts/validate.sh` (current highest is Check 12: pipeline.md Step 6 guard, verified at scripts/validate.sh:518):

```bash
# Check 13: USAGE.md present + has required H2 sections (every routable non-deprecated skill)
echo "=== Check 13: per-skill USAGE.md present + required sections ==="
REQUIRED_H2=("## When to use" "## When NOT to use" "## Mental model" "## Common pitfalls" "## Related skills")
while IFS= read -r SKILL_NAME; do
  USAGE_PATH="skills/$SKILL_NAME/USAGE.md"
  if [ ! -f "$USAGE_PATH" ]; then
    echo "  ERROR: $USAGE_PATH missing"
    ERRORS=$((ERRORS+1))
    continue
  fi
  MISSING=0
  for H2 in "${REQUIRED_H2[@]}"; do
    # Line-anchored heading match — reject substring matches inside code fences.
    if ! grep -qE "^${H2}[[:space:]]*\$" "$USAGE_PATH"; then
      echo "  ERROR: $USAGE_PATH missing section heading: $H2"
      ERRORS=$((ERRORS+1))
      MISSING=$((MISSING+1))
    fi
  done
  [ "$MISSING" -eq 0 ] && echo "  PASS: $USAGE_PATH has all required sections"
done < <(jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json)
```

Two notes:
- The predicate is `status != "deprecated"`, NOT `status == "active"` — see Constraints for the rationale.
- The grep is line-anchored (`^${H2}[[:space:]]*$`), not substring (`grep -F`), so a USAGE.md body that quotes `"## When to use"` inside a code fence will NOT falsely pass.

### Step 4: Update CLAUDE.md

Two edits:
1. "Skill directory structure" section: add `USAGE.md       # required, has When-to-use / When-NOT / Mental model / Pitfalls / Related skills` between `SKILL.md` and `*.md  # optional supporting files`.
2. "Creating a new skill" section: add a new step 5 — "Create `skills/{name}/USAGE.md` using `templates/doc-SKILL-USAGE.md` and fill in all five required H2 sections." Renumber the existing steps 5-6 to 6-7. Update step 4 to clarify DESIGN.md is still optional ("Optionally create `skills/{name}/DESIGN.md` for design rationale if the skill is complex enough to warrant a developer-facing doc").

### Step 5: Update doc/PHILOSOPHY.md decision tree

For each active routable skill that has a decision-tree entry under `## Decision tree`, append a link to its USAGE.md. Format: existing one-line description, then `[USAGE](../skills/{name}/USAGE.md)` at end. The decision tree is a markdown list; adding a trailing link per entry is a one-line edit per entry, ~11 lines total.

### Step 6: Add a new PHILOSOPHY.md `## Documentation surfaces` section

Add a NEW top-level section to `doc/PHILOSOPHY.md` named `## Documentation surfaces`, placed between `## Key patterns and conventions` and `## Decision tree`. Explains the three-doc-per-skill model:

- `SKILL.md` (required) — agent execution instructions
- `USAGE.md` (required for active routable skills) — operator + agent best-practice: when to use, when not, mental model, pitfalls, related skills
- `DESIGN.md` (optional) — developer-facing design rationale

This subsection is also where the "audit fires on missing USAGE.md or missing required H2" rule is documented so the user reads it the next time they look at PHILOSOPHY.md.

### Step 7: Atomic-commit ordering

The validate.sh pre-commit hook (installed via `setup-hooks.sh`) runs Check 13. If Check 13 lands as a separate commit from the 11 USAGE.md files, the hook BLOCKS commits in the intermediate state. Mitigation: stage the validate.sh change + all 11 USAGE.md files + CLAUDE.md + PHILOSOPHY.md edits as ONE commit. Implementing phase writes all files first, runs `./scripts/validate.sh` to confirm green, THEN commits. /ship handles this naturally because it commits once at the end of the workflow; the only failure mode is an operator manually running `git commit` between stages.

### Step 8: Bump VERSION + CHANGELOG entry

Standard `/ship` flow. CHANGELOG entry voice: "feat: F000NNN per-skill USAGE.md convention + validate.sh audit — every active routable skill now has a USAGE.md best-practice doc next to its SKILL.md; PHILOSOPHY.md decision tree links each entry to the new USAGE.md."

## Open Questions

1. **Should USAGE.md be deployed to ~/.claude/skills/{name}/USAGE.md?** No. Constraint #6 covers this — USAGE.md is human-reading, not agent-runtime. `skills-deploy install` already deploys SKILL.md (agent-facing). USAGE.md stays in-repo. If a future need arises (e.g. an in-repo-aware `/skill-help` skill that wants to display USAGE.md inline), deployment can be added then without breaking changes.
2. **Should the audit be ERROR or WARN?** ERROR. Predicate F000030 already establishes that missing PHILOSOPHY.md decision-tree entries are ERROR. USAGE.md matches that severity — both are mandatory per-skill documentation surfaces. A WARN would replay the F000030 1/13 adoption story.
3. **Should `CJ_goal_investigate` (deprecated shim, under `deprecated/`) get a USAGE.md?** No. The audit predicate `status=="active"` excludes it. The shim's purpose is one-line banner-then-route; SKILL.md alone is sufficient. The retired-skill tombstone in PHILOSOPHY.md `## Retired skills` already documents why it exists.
4. **README.md per-skill USAGE.md links?** Deferred. `scripts/generate-readme.sh` regenerates README.md from skills-catalog.json. Adding a USAGE.md column would require a catalog field + script change + regeneration. Out of scope for v1; tracked as a follow-up TODOS row tagged P3/S if the user finds README.md insufficient after v1 ships.
5. **work-copilot/ analog?** Deferred. work-copilot/ is a self-contained bundle for non-Claude target repos; it has its own WORKFLOW.md + templates. Whether work-copilot ships its own USAGE.md analog is a separate decision; this PR is workbench-only per Constraint #1.
6. **USAGE.md frontmatter shape — copy DESIGN.md's exactly, or simplify?** Copy DESIGN.md's exactly (skill-name, version, status, created, last-updated) for visual consistency. **Audit does NOT validate frontmatter at all** — not its presence, not field shape. Frontmatter is recommended (the template ships with it) but not enforced. Rationale: over-constraining frontmatter is a separate, lower-value concern; the audit's job is to guarantee the five operator-facing sections exist, nothing more.

## Success Criteria

1. `templates/doc-SKILL-USAGE.md` exists with five required H2 sections (When to use, When NOT to use, Mental model, Common pitfalls, Related skills) and DESIGN.md-shaped frontmatter.
2. `skills/{name}/USAGE.md` exists for all 11 active routable skills (the 13 - 2 deprecated shims). Each file has all five required H2 sections filled with content (not empty placeholders).
3. `scripts/validate.sh` has a new check that fires ERROR on a missing USAGE.md or a missing required H2 section for any active routable skill. `./scripts/validate.sh` passes on this PR's HEAD.
4. `doc/PHILOSOPHY.md ## Decision tree` entries each link to their USAGE.md (markdown link at end of each list item).
5. `doc/PHILOSOPHY.md` has a new top-level `## Documentation surfaces` section (placed between `## Key patterns and conventions` and `## Decision tree`) documenting the three-doc-per-skill model (SKILL.md + USAGE.md + optional DESIGN.md) and the validate.sh audit rule.
6. `CLAUDE.md` "Skill directory structure" lists USAGE.md as required; "Creating a new skill" instructs new-skill authors to create USAGE.md from the template.
7. `scripts/test.sh` runs green (full suite, not just validate).
8. CHANGELOG.md entry for the next free version slot (queried via `./scripts/check-version-queue.sh`) describes the feature in user-forward voice.
