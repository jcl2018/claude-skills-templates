## Problem Statement

The F000010 pipeline (`/scaffold-work-item` → `/implement-from-spec` → `/qa-work-item`) has two known gaps surfaced during the F000011 dogfood on 2026-05-08:

1. **Pipeline coverage is partial.** `/scaffold-work-item` handles all 4 work-item types (feature / user-story / task / defect), but `/implement-from-spec` (S000018 SPEC AC-7) and `/qa-work-item` (S000019 SPEC AC-7) explicitly reject anything that's not a user-story. A defect or task tracker that's been scaffolded cannot flow through the pipeline — the user has to drop back to manual implementation.
2. **`/personal-workflow check` Step 18 traceability parser misses multi-AC cells.** The prose at `skills/personal-workflow/check.md:339-371` says "extract all values from the `AC` column" but doesn't specify how to handle cells like `AC-1, AC-2, AC-3`. Real TEST-SPEC files contain them: `S000018_TEST-SPEC.md:24` (`AC-1, AC-2, AC-3`), `S000018_TEST-SPEC.md:26` (`AC-5, AC-6`), `S000019_TEST-SPEC.md:32` (`AC-2, AC-4`). A field-by-field exact-match implementation produces false `[UNTESTED]` findings on every multi-AC P0 story.

Both issues live in the validator + pipeline surface and were originally tracked as separate P3 TODOs (#5, #6 in `TODOS.md`). The user opted to bundle them as a single feature — the bug fix can ride through the new defect path as the integration test for the per-type generalization.

## Premises

1. **Per-type pipeline parity is the right long-term shape.** Of the 4 options listed in the original TODO, option 1 (generalize per-type) is the cleanest — fewer skill names to remember, type-detection plumbing already exists in `/scaffold-work-item`. Options 2 and 3 multiply skills or add indirection; option 4 (accept) loses pipeline value for non-user-story work.
2. **Step 18 is a documentation bug, not a code bug.** `check.md` is a skill prose spec, not executable code. The fix is tightening the prose so any LLM following Step 18 splits AC cells on comma before set-membership check.
3. **The two fixes share a verification surface.** Once the per-type pipeline accepts defects, the Step 18 fix can be implemented and QA'd through that same pipeline as a defect — closing the loop and proving the per-type path works.
4. **Workbench-only scope.** No downstream-consumer (e.g. portfolio repo) work is in scope; this is workbench-internal pipeline polish.
5. **`E2E walked manually` Phase 3 gates are still human-driven.** This feature does not change Phase 3 gate semantics; `/qa-work-item` extension preserves the existing E2E-vs-smoke distinction per work-item type.

## Recommended Approach

Two user-stories under one feature, sequenced so story 1 unblocks story 2's verification path:

### Story 1 — Generalize `/implement-from-spec` and `/qa-work-item` per work-item type

Both skills currently hard-fail on non-user-story work-items. Extend each to branch on the `type` field in `_TRACKER.md` frontmatter and read the type-appropriate input artifacts:

| Type | `/implement-from-spec` reads | `/qa-work-item` reads |
|---|---|---|
| user-story (today) | `SPEC.md` + `DESIGN.md` | `TEST-SPEC.md` |
| defect (new) | `RCA.md` + `test-plan.md` | `test-plan.md` |
| task (new) | `task-spec.md` | `task-spec.md` (test rows) |
| feature (new) | AskUserQuestion to pick a child user-story (delegates) | AskUserQuestion to pick a child user-story (delegates) |

Updates required in each skill's SKILL.md and routing file (implement.md, qa.md):
- Type detection from tracker frontmatter (parse `type:` field).
- Per-type input-artifact resolution (which files to read for the SPEC equivalent).
- Per-type boundary check: still calls `/personal-workflow check`, but interprets results per `personal-artifact-manifests.json` for that type.
- Per-type idempotency check: same shape (Phase 2 implementer-owned gates + journal `[impl-pass]` entry), but gate labels differ per template.
- Error-handling table updates: remove "Wrong type" hard-fails, replace with type-specific guidance.
- Per-type sensitive-surface AUQ rules carry over unchanged (catalog/manifest/validator are still the trip-wires regardless of type).

### Story 2 — Step 18 traceability parser comma-split fix

Tighten Step 18 prose in `skills/personal-workflow/check.md:339-371` (the current ambiguous "extract all values from the `AC` column"):

- Add explicit instruction: "For each cell in the AC column, split on comma and trim whitespace. Each resulting token (matching `^AC-[0-9]+$` after the existing placeholder filter) contributes one value to `ac_set`."
- Add a worked example showing `| S2 | core | AC-1, AC-2, AC-3 | ...` → `{AC-1, AC-2, AC-3}` set membership.
- Keep the existing placeholder filter (`^AC-\{[a-zA-Z_]+\}$`) — comma-split happens BEFORE the filter so a cell like `AC-{n}, AC-1` correctly contributes `AC-1` and drops the placeholder.
- No tracker-template changes; no manifest changes.

This story scaffolds as a user-story under this feature (per the user's "use one feature" decision in scoping). The Step 18 fix is small but ships through the same pipeline as story 1, with its own SPEC + TEST-SPEC describing the expected post-fix behavior on multi-AC TEST-SPEC fixtures.

## Approaches Considered

### Approach A — Generalize per-type (RECOMMENDED)
Both skills branch on the `type` field and read per-type input artifacts. Single entry point per phase regardless of type.
**Pros:** uniform pipeline; one skill name to remember per phase; type-detection plumbing already exists in `/scaffold-work-item` to model on.
**Cons:** each skill grows ~30%; new code paths (defect, task, feature-delegate) need their own tests.
**Completeness: 9/10.**

### Approach B — Per-type sibling skills
Add `/implement-from-rca`, `/qa-from-test-plan`, etc.
**Pros:** each skill stays focused on one input shape.
**Cons:** skill explosion (2 phases × 3 new types = 6 new skills); user has to remember which to invoke for which type.
**Completeness: 6/10.**

### Approach C — Smart `/personal-workflow` dispatcher
Add `/personal-workflow implement` and `/personal-workflow qa` that read the type and delegate to the appropriate underlying skill.
**Pros:** one entry point per phase regardless of type.
**Cons:** extra indirection layer; doesn't avoid the per-type code in the underlying skills, just hides it; two ways to invoke the same operation (the dispatcher and the underlying skill).
**Completeness: 7/10.**

### Approach D — Accept the constraint
Defects/tasks/features stay manual; only user-stories use the pipeline.
**Pros:** zero new code.
**Cons:** pipeline coverage stays partial; user keeps switching workflows depending on type. The Step 18 bug still has to be fixed but loses the dogfood path.
**Completeness: 3/10.**

## Recommended Approach

Approach A. The per-type code complexity is bounded (4 types, all with existing manifests). Pipeline uniformity is the long-term win. Approach D was the v1 fallback if no demand surfaced for non-user-story pipeline use; the demand has now surfaced (this very work item — the Step 18 fix is naturally a defect, not a user-story).

## Constraints

- **Must not break existing user-story flows.** Per-type branching adds a `type == "user-story"` case that preserves today's exact behavior.
- **Idempotent** — re-running `/implement-from-spec` or `/qa-work-item` on a completed work-item of any type must be NO-OP (existing Premise 1.1 carries over per-type).
- **Boundary-safe** — `/personal-workflow check` still gates input drift and output compliance per work-item type (uses existing per-type entries in `personal-artifact-manifests.json`).
- **No template churn** — tracker / SPEC / DESIGN / TEST-SPEC templates stay as-is. The change is in skill prose only.
- **Step 18 fix preserves placeholder filter** — comma-split happens before the `^AC-\{[a-zA-Z_]+\}$` filter, so freshly-scaffolded TEST-SPECs with `AC-{n}` placeholders still flag correctly.

## Open Questions

- **Defect implement reads RCA + test-plan, but RCA is "what went wrong" not "what to build."** Is the implementer reading RCA for context only and treating `test-plan.md` as the de-facto SPEC? Decision for v1: yes — the test-plan defines the desired post-fix behavior; RCA is the why-and-history pointer the implementer reads to understand what NOT to revert. Document this in the per-type prose.
- **Feature-level implement: AUQ to pick a child user-story, or auto-pick the next un-shipped child?** v1: AUQ — explicit is better than clever, and the user can scaffold + ship one story at a time without surprise.
- **Defect QA: `test-plan.md` doesn't have the smoke / E2E split that TEST-SPEC has.** Decision for v1: treat all `test-plan.md` rows as smoke-equivalent; no E2E subagent dispatch for defects. Future work if defect QA needs deeper coverage.
