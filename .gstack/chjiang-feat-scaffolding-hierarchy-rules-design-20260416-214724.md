## Problem Statement

Both `/company-workflow` and `/personal-workflow` skills scaffold doc-driven work items (features → user-stories → tasks; defects standalone), but enforcement of structural rules between docs is incomplete:

1. **No init-time enforcement.** Scaffolding is AI-driven via WORKFLOW.md instructions. Nothing prevents an AI from creating an orphan feature with zero user-story children, a user-story misplaced outside a feature dir, or a directory named without the `{ID}_{slug}` prefix.
2. **Asymmetric post-hoc validation.** `personal-workflow/check.md:282-320` enforces hierarchy + placement at validation time. `company-workflow` has no equivalent.
3. **No directory-naming validation.** Neither skill validates that work-item directories match the expected regex `^[FSTD]\d{6}_[a-z0-9_-]+$`, that the type prefix letter matches the type, or that the embedded ID matches the TRACKER's frontmatter `id` field.

The user wants strict, atomic enforcement at scaffold time so that orphaned/misplaced/misnamed work items become structurally impossible to create, not just catchable after the fact.

## Constraints

- **D000007 just merged (v0.9.0).** `contract.json` is gone from both skills. Templates are the single source of truth for IN-doc structure (frontmatter, sections, lifecycle phases). Hierarchy rules live in `*-artifact-manifests.json` because they describe BETWEEN-doc structure, which templates can't express.
- **Existing directory convention preserved.** `work-items/{type-subfolder}/{ID}_{slug}/` with nested children up to 3 levels deep (feature → user-story → task). All file names prefixed `{ID}_*.md`.
- **PR #33 was a prior attempt at this work and was closed.** The user confirmed it was a "different approach" — its scope (personal-workflow only, spec-only enforcement via WORKFLOW.md) is not what we want. We're starting fresh with broader scope and stronger enforcement.

## Premises

1. **Hierarchy rules belong in the artifact manifests**, not a new file and not embedded in templates. Personal-workflow already does this; company-workflow needs to mirror. Single conceptual location: "what is a feature?" = artifacts + hierarchy + placement, all in one file per skill.
2. **Strict enforcement is the right default for both skills.** No carve-outs for personal-workflow being "lighter" — same hierarchy rules, both skills, no exceptions. Defects and tasks still allowed standalone via the existing `placement` rules (defect → `defects/`, task → can be standalone or under a parent — to be confirmed in the manifest).
3. **Init-time enforcement requires deterministic tooling, not AI obedience.** A scaffolder script that refuses to create orphans is the only way "must have ≥1 user-story" becomes impossible to violate. Validator-only enforcement is reactive; a scaffolder is preventive.
4. **Same scaffolder script serves both skills.** Both skills share the same hierarchy schema (per #1 + #2), so one scaffolder reading the per-skill manifest works for both.

## Approaches Considered

### Approach A: Manifest extension + WORKFLOW.md spec rule + post-hoc validator only

Mirror personal-workflow's manifest hierarchy/placement to company-workflow. Add "recursive scaffolding required" rule to both WORKFLOW.md files. Add `[MISFORMATTED]` rule to both validators. AI follows the spec; validator catches violations after.

- **Effort:** S (~150 LoC: manifest edits + WORKFLOW.md prose + check.md rule + fixtures)
- **Risk:** Medium — spec-only enforcement was attempted in closed PR #33; relies on AI obedience.
- **Pros:** Smallest diff. No new code paths. Mirrors PR #33's structure (proven to author quickly).
- **Cons:** Doesn't actually prevent orphan creation at init time — only catches it post-hoc. Violates the user's explicit requirement of "during doc initialization."
- **Reuses:** PR #33's WORKFLOW.md prose, fixtures, and `[MISFORMATTED]` regex.

### Approach B: Manifest extension + scaffolder script + WORKFLOW.md spec rule + validator (RECOMMENDED)

Same as A, plus a deterministic `scaffold` script per skill (or shared) that:
- Takes type + name + optional parent path
- Reads the manifest's `hierarchy` + `placement` rules
- Refuses to create a parent type without simultaneously creating ≥`min` of each `required_child` (atomic — either everything is created or nothing is)
- Auto-assigns next available `{ID}` per type
- Creates `{ID}_{slug}/` dir at the correct placement
- Copies templates with placeholder substitution ({ITEM_ID}, {ITEM_NAME}, {YYYY-MM-DD}, {author}, {SLUG}, {PARENT_ID}, {FEATURE_ID})
- WORKFLOW.md instructs AI to invoke the scaffolder rather than writing files free-form
- Validator still runs and catches violations from manually-edited (non-scaffolder) work items

- **Effort:** M (~400-600 LoC: scaffolder bash/awk + manifest edits + WORKFLOW.md + check.md rule + fixtures + tests)
- **Risk:** Low-Medium — adds new code paths but each is isolated and testable.
- **Pros:** Init-time enforcement actually means enforced. Atomic creation. Self-documenting via `--help`. Validator becomes a backstop, not the primary gate. Bootstrap-friendly: future AI sessions don't need to reason about layout.
- **Cons:** New script to maintain. AI must learn to invoke it (mitigated: WORKFLOW.md spells out the command pattern). Slight tension with "templates as SoT" theme of D000007 since scaffolder needs runtime config beyond templates — but manifest is already that runtime config.
- **Reuses:** PR #33's `[MISFORMATTED]` regex + fixtures. Personal-workflow's existing `hierarchy`/`placement` schema. Existing template files unchanged.

### Approach C: Skip the scaffolder, build a JSON-Schema-style hierarchy declaration + AI gets stricter spec

Same as A but with a more rigid schema in the manifest that downstream tools (validator, future scaffolder) consume. Defer scaffolder script entirely.

- **Effort:** S
- **Risk:** Same as A — still spec-only enforcement.
- **Pros:** Sets up the schema cleanly for a future scaffolder.
- **Cons:** Doesn't deliver init-time enforcement. Defers the harder problem. User explicitly picked Fork 4B (scaffolder).

## Recommended Approach

**Approach B.** The user's explicit Fork 4B pick (scaffolder script) makes A and C non-starters. The user wants enforcement at init time, and a script is the only mechanism that actually delivers that — AI obedience to spec is unreliable enough that PR #33's spec-only attempt was closed.

Concrete deliverables:

### 1. Manifest changes

**company-workflow** (`skills/company-workflow/company-artifact-manifests.json`):
Add `hierarchy` and `placement` blocks mirroring personal-workflow's existing schema.

```json
{
  "version": "3.1.0",
  "types": { ... existing ... },
  "hierarchy": {
    "feature": {"required_child": "user-story", "min": 1},
    "user-story": {"required_child": "task", "min": 1}
  },
  "placement": {
    "feature": "features",
    "defect": "defects",
    "review": "reviews",
    "user-story": "feature",
    "task": "user-story"
  }
}
```

Note: `defect` and `review` go into top-level subfolders (`defects/`, `reviews/`) — they're standalone types with no required parent.

Confirmed: `task` placement is `user-story` (same as personal), `review` placement is `reviews` (standalone subfolder).

**personal-workflow** (`skills/personal-workflow/personal-artifact-manifests.json`):
No schema change — already has hierarchy + placement. May need to bump min counts or add review type if missing.

### 2. Scaffolder script

Location: `scripts/scaffold-work-item` (top-level, shared across skills via `--skill {personal|company}` flag). Migration is a subcommand: `scripts/scaffold-work-item migrate --skill {personal|company} [--apply]`.

**Two input modes** (resolved C2):

- **CLI flags** for the simple case (zero-child types, single-level scaffold):
  ```bash
  scaffold-work-item --skill personal --type defect --name login-bug
  scaffold-work-item --skill company --type review --name q1-review
  ```
- **`--plan plan.yaml`** for trees (recommended for AI callers):
  ```yaml
  type: feature
  name: my-feature
  children:
    - type: user-story
      name: story-one
      children:
        - type: task
          name: task-a
        - type: task
          name: task-b
  ```
  Then: `scaffold-work-item --skill company --plan plan.yaml`

The plan file is committable — AI writes it, the diff shows intent before any work-item dirs exist.

**Atomic creation pattern** (resolved A2): build the entire tree under `work-items/.tmp-scaffold-$$/`, then `mv` the top-level dir into place. Single atomic rename on POSIX same-filesystem. If anything fails before the `mv`, `rm -rf` the tempdir — no partial state ever visible.

**ID assignment lock** (resolved A1): wrap the ID-compute → write transaction in `flock work-items/.scaffold.lock`. Three lines. Prevents collision under concurrent invocations.

**Placeholder substitution** (resolved C3): strict allow-list. Script substitutes only these exact names: `{ITEM_ID}, {ITEM_NAME}, {PARENT_ID}, {FEATURE_ID}, {YYYY-MM-DD}, {author}, {SLUG}, {BRANCH_NAME}`. Anything else passes through untouched. Document the allow-list at the top of the script. If a template author needs a literal `{foo}` in content, they pick a name not in the allow-list and it's safe.

**Logic:**
1. Acquire `flock work-items/.scaffold.lock`
2. Load `skills/{skill}-workflow/{skill}-artifact-manifests.json` and run schema validation (resolved A4: assert every `hierarchy.*.required_child` and `hierarchy.*.min` exists; hard fail with clear error on typos)
3. Parse input (CLI flags or `--plan` file). For `--plan`, validate against type names from the manifest.
4. Validate slug regex `^[a-z0-9_-]+$` for every name in the input
5. Walk the `hierarchy` from the requested type. For each level requiring `min ≥ 1` children, verify input provides that many. Refuse with clear error if not.
6. Compute next available IDs per type by scanning existing `work-items/`
7. Build the entire tree in `work-items/.tmp-scaffold-$$/`
8. Copy templates with placeholder substitution (allow-list)
9. Run validator on the tempdir tree to confirm it passes
10. `mv` the top-level dir into its placement target
11. Release lock

**Migration subcommand** (resolved A3): `scaffold-work-item migrate --skill {personal|company} [--apply] [--force]`.

Edge-case behavior (default = skip + log + non-zero exit; `--force` to apply anyway):

| Condition | Default behavior | --force behavior |
|---|---|---|
| Bare-slug dir, no TRACKER.md | Skip, log warning | Skip (can't infer ID) |
| Bare-slug dir, TRACKER missing `id` field | Skip, log warning | Skip (can't infer ID) |
| Bare-slug dir, TRACKER malformed YAML frontmatter | Skip, log warning | Skip |
| Inferred ID collides with existing properly-named dir | Skip, log conflict | Skip (still unsafe) |
| Two bare-slug dirs with same TRACKER `id` (data corruption) | Fail loudly, exit 2 | Fail loudly, exit 2 |
| Bare-slug parent with properly-named children | Rename parent only, preserve relative paths | Same |

`--dry-run` is the default (no `--apply` = print planned moves, don't execute). Idempotent by construction: properly-named dirs are skipped silently.

Error format examples:
```
ERROR: feature requires ≥1 user-story child. Pass --child-stories "<name1>[,<name2>...]"
ERROR: user-story 'story-one' requires ≥1 task child. Pass --child-tasks-of-story-one "<name1>[,<name2>...]"
ERROR: type prefix 'F' for feature collides with existing F000007 — next available is F000008 (used automatically)
```

## Resolved Questions

1. **company-workflow `task` placement**: `user-story` only. (Mirrors personal-workflow.)
2. **`review` type placement**: standalone in `work-items/reviews/`.
3. **Migration for existing work items**: auto-rename script (`scripts/migrate-work-items` or one-shot run inside `scaffold-work-item migrate`). ## Schema Sharing (one ruleset, two skills)

**One algorithm, two manifest files.** Both skills' validators and the scaffolder share the same enforcement logic, parameterized by `--skill {personal|company}`. The manifests differ in DATA (which types exist, which placements apply) but not in SCHEMA (the shape of `hierarchy` and `placement` blocks is identical).

This is consistent with the user's Fork 2A pick: same rules, different declarations.

If a future need arises for skill-specific constraints (e.g., approval workflows in company that don't exist in personal), this design doesn't preclude that — those would be additive new fields in the manifest, with the validator/scaffolder ignoring them when not present.

## Failure Modes (per Section 1 Architecture)

For each new codepath, the realistic production failure + whether it's handled:

| Codepath | Failure mode | Handled? | User experience |
|---|---|---|---|
| Manifest load | Malformed JSON | Yes (jq fails, script exits) | Clear error pointing at file + line |
| Manifest schema check (A4) | `requried_child` typo | Yes (schema check rejects) | Clear error naming the bad field |
| ID compute | Concurrent invocations colliding | Yes (flock A1) | Second invocation waits, no collision |
| Atomic write (A2) | Mid-build template substitution failure | Yes (tempdir + cleanup on trap) | No partial state visible; tempdir removed |
| Atomic write | Process killed mid-build (SIGKILL — no trap) | **No** | Stale tempdir at `work-items/.tmp-scaffold-$$/`, requires manual cleanup |
| Migration | Bare-slug dir, no TRACKER.md | Yes (skip + log) | Clear log line; non-zero exit signals issue |
| Migration | Two bare-slug dirs with same TRACKER id (corruption) | Yes (fail loudly, exit 2) | Loud error pointing at both paths |
| Plan parsing | Circular parent reference in `--plan` YAML | Yes (validate before write) | Clear error before any work-items touched |
| Validator | Fixture passes invalid manifest | Yes (schema check at load) | Validator refuses to run on bad manifest |

**Critical gap**: SIGKILL mid-build leaves a stale tempdir. Logged in NOT in scope; doctor.sh follow-up.

## Worktree Parallelization

Sequential implementation. The 8 steps form a chain (manifest → validator → fixtures+tests → scaffolder → migrate → docs → catalog), and each builds on the prior. Splitting across worktrees would create merge friction without speed gain. Single-lane execution.
