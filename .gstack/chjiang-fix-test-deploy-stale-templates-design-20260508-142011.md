## Problem Statement

`scripts/test-deploy.sh` is the integration test suite for the `skills-deploy` script. Tests T2, T4, T5, T6, T7 reference `doc-RCA.md` at the **top-level** templates path. That template moved to `templates/personal-workflow/doc-RCA.md` in v1.3.x. The 5 test cases now fail when `test-deploy.sh` is run end-to-end.

Compounding the bug: `scripts/test.sh` (CI's entry point) does **not** invoke `scripts/test-deploy.sh`. It only greps `test-deploy.sh` for the `jq()` CRLF-stripping wrapper. So the broken integration tests are invisible to CI — they fail silently, and the failure has been latent since v1.3.x.

This is documented as two linked TODOs in `TODOS.md`:
- "Pre-existing template-ownership test failures in test-deploy.sh (P2, S)"
- "Wire test-deploy.sh into CI / test.sh (P3, S)" — depends on the first

This design closes both in one PR.

## What Makes This Cool

The fix isn't just re-pointing 5 references — it's elevating `test-deploy.sh` from a manually-runnable suite to a real CI-gated suite. Once wired in, future template moves will fail loudly in CI instead of rotting silently.

It's also a small, contained showcase of "boil the lake": the literal TODO described option A (re-point only) and option B (retire). The lake-boil option (C — re-point AND wire-into-CI) closes the underlying gap that produced the bug in the first place. Future regressions of this exact shape become impossible.

## Constraints

- **Workbench-only scope.** No changes to consumer-facing skills, work-copilot bundle, or downstream repos.
- **No flaky test tolerance.** If test-deploy.sh is wired into CI, every test in it must be reliable — flaky tests would block every PR.
- **Pre-existing wrapper check in test.sh stays.** The grep-based check that confirms `test-deploy.sh` defines the `jq()` wrapper is independent and should remain (it's a structural assertion, not a runtime test).
- **No new external dependencies.** This stays in pure bash + jq + git, matching the existing repo idiom.
- **Idempotency / re-runnability.** test-deploy.sh already creates and tears down a temp dir per test. Wiring it into CI doesn't change that contract.

## Premises

1. **Right problem to solve.** Silent test failures hide regressions in `skills-deploy` template ownership semantics. The tests measure something real (template lifecycle: install → ownership tracking → drift → cleanup). Letting them rot is technical debt with a real cost: the next template move will break ownership semantics and we won't know.
2. **Existing code reused.** `test-deploy.sh` itself stays — only the 5 references to `doc-RCA.md` change. `scripts/test.sh` gains one new line invoking `test-deploy.sh` near the end of its sequence. No new test infrastructure.
3. **Distribution.** N/A — this is a test fix, no new artifacts.

## Approaches Considered

### Approach A: Re-point references only
Replace the 5 `doc-RCA.md` references in `scripts/test-deploy.sh` with `doc-SKILL-DESIGN.md` (the only remaining flat-path template). Tests pass when manually run. Effort: human ~30 min / CC ~5 min. Completeness 6/10 — fixes the symptom, not the cause. Next time a template moves, the tests will silently fail again until someone runs `test-deploy.sh` manually.

### Approach B: Retire T2, T4-T7
Delete the 5 failing test cases entirely. Removes the symptom by removing coverage for `skills-deploy` template-ownership semantics. Loses a real test surface. Effort: human ~10 min / CC ~2 min. Completeness 3/10. Not recommended — the tests measure real behavior; killing them is degrading regression coverage to make red go away.

### Approach C: Re-point + wire test-deploy.sh into CI (chosen)
Replace the 5 references AND modify `scripts/test.sh` to invoke `scripts/test-deploy.sh` end-to-end as its final step. CI now catches future template-move regressions automatically. Closes both linked TODOs in one PR. Effort: human ~1–2h / CC ~15–20 min. Completeness 9/10.

## Recommended Approach

**Approach C: Re-point + wire-into-CI.**
