## Problem Statement

Consumers who install claude-skills-templates skills on their machine via `git clone` + `./scripts/skills-deploy install` have no way to learn that a new version of the collection has shipped. They only find out by happening to `git pull`. The repo ships periodic releases (v1.5.2 today, on a steady cadence) and the `~/.claude/skills/{name}/SKILL.md` they're using can drift arbitrarily far from upstream main.

Goal: a passive-then-actionable nudge, modeled after gstack's `gstack-update-check`, that fires when the user invokes one of this repo's installed skills and a newer collection version exists on `origin/main`.

## What Makes This Cool

- The user's first signal that "something changed upstream" arrives *exactly when they're already in a skill workflow* — no separate `git pull` muscle memory required.
- Symmetric with how gstack does it (the user's chosen mental model). Less cognitive load: same `UPGRADE_AVAILABLE` shape, same snooze/skip semantics, same upgrade-on-confirm.
- The cache + 24h TTL means `git fetch` runs at most once per day per machine — invisible cost, real benefit.

## Constraints

- Claude Code only for v1. The work-copilot/ bundle (GitHub Copilot consumers) is out of scope; that path needs its own mechanism since it has no skill preamble surface to instrument.
- No GitHub API, no auth tokens, no rate-limit handling. Source of truth is the user's local clone, refreshed via `git fetch`.
- Must not interfere with `--offline` use: `git fetch` failures fall back to stale cache silently.
- Total new code budget: small. ~50 LOC bash for the check script, ~4-line snippet in each of 3 skill preambles, one new field in the existing manifest.

## Premises

1. Check fires from each skill's preamble (Claude Code only). Confirmed.
2. Source of truth = user's local git clone, refreshed via `git fetch origin`. Confirmed.
3. Result cached for 24h in `~/.claude/.skills-templates-update.json`. Confirmed.
4. Comparison is collection-level (`VERSION` file), not per-skill versions. `skills-deploy install` reinstalls everything anyway. Confirmed.
5. Upgrade action runs `git pull && ./scripts/skills-deploy install` in the user's clone, with explicit confirmation. Confirmed.
6. Clone path tracked in the existing `~/.claude/.skills-templates.json` manifest at install time. Confirmed.

## Approaches Considered

### Approach A: Shared bin script + per-skill preamble line (CHOSEN)

New `scripts/skills-update-check` (~50 LOC). `skills-deploy install` copies it to `~/.claude/bin/skills-update-check` and writes `repo_root` into the existing manifest. Each skill's preamble runs the script and reacts to `SKILLS_UPGRADE_AVAILABLE` / `JUST_UPGRADED` markers. Mirrors gstack one-to-one.

### Approach B: Settings.json SessionStart hook

Single hook in `~/.claude/settings.json`. One nudge per Claude Code session. Rejected: settings.json mutation has wider blast radius, harder to undo cleanly across multiple installs, and isn't symmetric with gstack's per-skill model that the user named as the reference.

### Approach C: `skills-deploy doctor` flag only

Update status surfaces only when the user runs `skills-deploy doctor`. Rejected: doesn't match the "nudge" intent — the user has to remember to check.

## Recommended Approach

**Approach A.** Three pieces.

### Piece 1: `scripts/skills-update-check` (new, ~80 LOC bash)

Single script with a default action (the check) plus three subcommands (`--snooze`, `--skip`, `--prompted`) so the model never writes JSON itself — it calls subcommands. All times are stored as **epoch seconds** (portable across BSD/GNU `date`), not ISO strings.

**Default action (no args):**
1. `manifest=$HOME/.claude/.skills-templates.json`. If missing or `jq -r '.repo_root // empty' "$manifest"` is empty, exit silent.
2. `repo_root=$(jq -r '.repo_root' "$manifest")`. If `[ ! -d "$repo_root/.git" ]`, exit silent.
3. `cache=$HOME/.claude/.skills-templates-update.json`. Shape:
   ```json
   {
     "checked_at": 1778217204,
     "local_version": "1.5.2",
     "remote_version": "1.6.0",
     "snooze_until": null,
     "skip_version": null,
     "prompted_session": null,
     "prompted_at": null
   }
   ```
4. `now=$(date +%s)`. If `snooze_until != null && now < snooze_until` → exit silent.
5. If `checked_at != null && (now - checked_at) < 86400`, skip the fetch and reuse cached values.
6. Else: `git -C "$repo_root" fetch --quiet origin main 2>/dev/null || { use_stale_cache; exit 0; }`. Read local: `cat "$repo_root/VERSION"`. Read remote: `git -C "$repo_root" show origin/main:VERSION 2>/dev/null`. If either read fails, fall back to stale cache (no banner if cache is also empty). On success, update cache atomically (see write rules below).
7. Semver compare via the standard idiom: `[ "$(printf '%s\n%s\n' "$local_version" "$remote_version" | sort -V | tail -1)" = "$remote_version" ] && [ "$local_version" != "$remote_version" ]`. If true (remote is strictly newer) AND `remote_version != skip_version`: print `SKILLS_UPGRADE_AVAILABLE <local_version> <remote_version>`.
8. Read marker `$HOME/.claude/.skills-templates-just-upgraded`. Format: single line, two space-separated fields: `<from_version> <to_version>` (e.g. `1.5.2 1.6.0`). Order: read into a var → `rm -f` the marker → print `SKILLS_JUST_UPGRADED <from> <to>`. Single-shot: avoids "read after another process unlinked" by tolerating `cat` failure (treat as no marker).
9. Exit 0 always. All non-fatal errors silenced via `2>/dev/null || true`.

**`--snooze [hours]` subcommand** (default 24): writes `snooze_until = $(date +%s) + (hours * 3600)` to cache. Does not touch `prompted_session` — that's the body block's responsibility via `--prompted`, which keeps session debouncing separate from snooze.

**`--skip <version>` subcommand**: writes `skip_version = <version>` to cache.

**`--prompted <session_id>` subcommand**: writes `prompted_session = <session_id>` and `prompted_at = $(date +%s)` to cache. Used by skills to debounce same-session double-prompts (see Piece 3).

**`--should-prompt <session_id>` subcommand**: follows bash convention — exit 0 ("yes, prompt the user") if no prompt has been shown in this session, exit 1 ("no, suppress") if cache's `prompted_session == <session_id>` AND `(now - prompted_at) < 600` (10 min). Used by skills to know whether to surface the AskUserQuestion. Reads cleanly: `if ~/.claude/bin/skills-update-check --should-prompt "$session"; then ask_user; fi`.

**Atomic cache writes:** all subcommands write via `mktemp "$HOME/.claude/.skills-templates-update.tmp.XXXXXX"` (same filesystem as target — `mv` is atomic), then `mv` over the cache. Pattern matches `manifest_write()` in the existing `scripts/skills-deploy`.

**No timeout flag.** `git fetch` against an unreachable remote typically resolves in 2-3s on macOS; longer-tail cases are rare. We don't rely on the GNU `timeout` utility (not present on stock macOS without coreutils). Worst-case latency on a flaky network: one `git fetch` failure → fall back to stale cache.

### Piece 2: `scripts/skills-deploy` modifications

- **`install`:**
  - Compute `REPO_ROOT` (already done) and write into manifest: `manifest_read | jq --arg r "$REPO_ROOT" '. + {repo_root: $r}' | manifest_write`.
  - `mkdir -p "$HOME/.claude/bin"` then `cp scripts/skills-update-check "$HOME/.claude/bin/skills-update-check"` and `chmod +x` it.
  - On the upgrade path (`install` invoked after `git pull` from a known prior version): write `$HOME/.claude/.skills-templates-just-upgraded` containing `<old_version> <new_version>` (single line, space-separated). The upgrade caller passes `--from-upgrade <old>` to signal this; without the flag, the marker is not written.
  - Print a one-line first-run hint at the end of the install summary: "Update checks installed. You'll see a banner the next time a new collection version is available."

- **`remove`:** when no skills owned by this manifest remain, also `rm -f "$HOME/.claude/bin/skills-update-check" "$HOME/.claude/.skills-templates-update.json"`. Don't touch the manifest's `repo_root` field — it's harmless after the script is gone.

- **`doctor`:** add a section that reads `$HOME/.claude/.skills-templates-update.json` and prints `last_check`, `local_version`, `remote_version`, `snooze_until`, `skip_version`. If the cache file is missing, prints `(never checked)`. This is small additional logic — call it out in the diff, don't claim it's free.

**Migration:** users with an existing install (pre-feature) have a manifest without `repo_root`. They must run `skills-deploy install` once to bootstrap. The first-run hint above doubles as the migration nudge — the next release-of-this-feature's CHANGELOG should call this out explicitly.

**Deprecated skills (rule, not waffle):** deprecated skills (catalog `status: deprecated`) **do not** get the preamble snippet. They're skipped by default `skills-deploy install` and only land via `--include-deprecated`. Adding the snippet to a skill that may not be installed is dead code and asymmetric with the existing deprecation gate.

### Piece 3: Skill preamble snippet + body instruction block

The snippet is bash inside each active skill's SKILL.md preamble (single line, executed as part of the existing preamble bash block):

```bash
~/.claude/bin/skills-update-check 2>/dev/null || true
```

No `_SKILLS_UPD=` variable — the model reads the echoed banner from preamble output, not from a bash var. The variable assignment was dead code.

The skill body, immediately after the preamble bash block, gets this instruction block (for the model to read):

> **If preamble output contained `SKILLS_UPGRADE_AVAILABLE <old> <new>`:**
> 1. Parse the two version tokens from the banner line. Capture them as shell vars for use below: `read -r _ OLD_VERSION NEW_VERSION <<< "$banner_line"`. (The banner is whitespace-separated: marker, old, new.)
> 2. Resolve session id with fallback: `SESSION="${CLAUDE_SESSION_ID:-$PPID}"`.
> 3. Run `~/.claude/bin/skills-update-check --should-prompt "$SESSION"`. If it exits non-zero (already prompted in this session), suppress the prompt and continue with the skill workflow.
> 4. Read `repo_root` from manifest: `repo_root=$(jq -r '.repo_root' "$HOME/.claude/.skills-templates.json")`.
> 5. Precondition check: `git -C "$repo_root" diff --quiet && git -C "$repo_root" diff --cached --quiet` — if either fails, the working tree is dirty. Print: "Skipping upgrade — uncommitted changes in $repo_root. Stash or commit first." Then call `~/.claude/bin/skills-update-check --snooze 1` to suppress for an hour, and continue with the skill workflow.
> 6. Otherwise, call AskUserQuestion with three options: Upgrade now (recommended) / Snooze 24h / Skip this version.
> 7. Mark this session as prompted regardless of choice: `~/.claude/bin/skills-update-check --prompted "$SESSION"`.
> 8. **Upgrade now:** `git -C "$repo_root" pull --ff-only origin main && "$repo_root/scripts/skills-deploy" install --from-upgrade "$OLD_VERSION"`. The `--from-upgrade` flag tells skills-deploy to write the JUST_UPGRADED marker with the parsed `$OLD_VERSION` and the new `$repo_root/VERSION` value.
> 9. **Snooze 24h:** `~/.claude/bin/skills-update-check --snooze 24`.
> 10. **Skip this version:** `~/.claude/bin/skills-update-check --skip "$NEW_VERSION"`.
>
> **If preamble output contained `SKILLS_JUST_UPGRADED <from> <to>`:** print "claude-skills-templates upgraded to v\<to\> (was v\<from\>)" and continue with the skill workflow. The marker file has already been removed by `skills-update-check` itself.

## Edge Cases (must handle in v1)

- **`repo_root` path missing/moved.** User deleted or moved their clone. Check exits silent. `skills-deploy doctor` surfaces this with a "(repo_root missing — re-run `skills-deploy install` from your clone)" line.
- **`git fetch` fails (offline / no remote / auth issue / non-standard refspec).** Fall back to stale cache; no banner if cache is also empty. Never error. Same fallback if `git show origin/main:VERSION` fails (e.g. sparse-checkout missing the file, weird refspec).
- **User is on a non-`main` branch in the clone.** Always check `origin/main` for the version comparison — releases ship there. Working-tree branch doesn't matter.
- **VERSION file format edge cases.** Use `sort -V` semver comparison via the pinned idiom in Piece 1, not string compare. `1.10.0` > `1.5.2`.
- **Multiple consumers (catalog has more skills than installed).** No effect — check is collection-level, only nudges if `VERSION` advanced.
- **Snooze + skip both set.** `skip_version` wins. If user later upgrades and `remote_version` advances past `skip_version`, the skip clears naturally on next check.
- **Cache file corrupted.** Treat as empty cache; re-fetch.
- **Two skills invoked in same session.** Cache TTL absorbs the second call (no double `git fetch`). The banner echo is idempotent and prints both times — fine, it's just text. **The AskUserQuestion is debounced via `--should-prompt "$SESSION"`** (10-min window, where `$SESSION` is `${CLAUDE_SESSION_ID:-$PPID}`). Without this debounce, the user would see the upgrade prompt twice in one session.
- **Concurrent skill invocations writing the cache.** Write via `mktemp` (in `$HOME/.claude/`, same filesystem) + atomic `mv` to avoid torn writes. Same pattern `skills-deploy` already uses for its manifest.
- **Dirty working tree at upgrade time.** The body instruction block precondition-checks `git diff --quiet && git diff --cached --quiet`. If dirty, skip the upgrade with a clear message and snooze for 1h (don't re-prompt every skill call until the user resolves it).
- **JUST_UPGRADED marker race.** Two skills running near-simultaneously could both try to read+unlink the marker. Tolerate `cat` failure as "marker already consumed." Idempotent: marker is best-effort, not transactional.
- **Pre-existing install without `repo_root` field.** Migration path: user runs `skills-deploy install` once; the install writes the field. Until they do, the check exits silent (no harm, no nudge). The first-run hint printed by `skills-deploy install` calls this out for the migration release.

## Open Questions

- Should the upgrade flow run `./scripts/test.sh` after `git pull` and bail if tests fail? Argument for: protects users from a broken HEAD. Argument against: latency on the upgrade prompt; users on machines without bun/jq/etc may not have a working test env. Default: skip tests in v1, document the recommendation to upgrade only when in a known-good shell.
- Where exactly does the preamble snippet sit relative to existing preamble logic? Each SKILL.md today has skill-specific setup. The update-check echo should run *first* (before any other preamble work) so the banner shows even if the rest of the preamble fails. Confirm during implementation.
- Session-id source: `$CLAUDE_SESSION_ID` is the assumed env var name for the debounce. If Claude Code doesn't surface a stable session id, fall back to `$PPID` for the marker. Either way the debounce window is short (10 min) so a missed match is low-cost (one redundant prompt at worst).
- The first-run hint printed by `install` is a fixed decision (resolved: do it).

## Success Criteria

- On a fresh clone with no upgrade available, running any installed skill prints no banner. (Silent path.)
- When `origin/main` advances and 24h has passed, running any installed skill prints `SKILLS_UPGRADE_AVAILABLE 1.5.2 1.6.0`.
- AskUserQuestion offers Upgrade / Snooze / Skip. Upgrade option runs `git pull && skills-deploy install`, writes marker, succeeds within 30s.
- Snooze suppresses for exactly 24h. Skip suppresses until `remote_version` advances past the skipped version.
- `git fetch` failures (offline) do not break or delay the skill workflow.
- `skills-deploy doctor` shows last-check timestamp + remote/local versions.
- `scripts/test.sh` exercises the check script with a fake repo_root pointing at a temp git repo whose VERSION file is bumped.

## Distribution Plan
