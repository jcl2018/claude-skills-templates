## Problem Statement

/system-health reports on ~/.claude/ filesystem health: skill inventory, dependency
graph, broken symlinks, hygiene. But it's blind to which skills actually get used.
The user has ~400+ usage entries across 15+ skills spanning 22 days in
~/.gstack/analytics/skill-usage.jsonl. The data exists. Nobody's surfacing it.

The gap: you can see what's installed and whether it's structurally healthy, but you
can't see what you actually use, how often, for how long, or whether your workflow
is evolving.

## What Makes This Cool

Behavioral topology overlay. Take the static dependency graph (already computed in
Step 2) and overlay actual usage data on top. The dashboard can now say: "gstack is
your most-referenced hub skill, but you've never directly invoked browse or canary.
office-hours accounts for 39% of all skill usage. Your average ship session takes
5 minutes. You stopped using review 2 weeks ago."

It turns system-health from a structural audit into a personal workflow MRI.

## Constraints

- All computation in bash/jq. Claude interprets structured output, does NOT compute stats.
- Data source: ~/.gstack/analytics/skill-usage.jsonl exclusively. No new instrumentation.
- Usage section is UNSCORED (same pattern as waza appendix). Does not affect the
  4-bucket composite score. Rationale: data lives in ~/.gstack/, not ~/.claude/.
- Must handle three JSONL schemas plus filter out non-run events:
  - Simple: {"skill":"X","ts":"...","repo":"Y"} — no duration or outcome
  - Intermediate: {"skill":"X","ts":"...","duration_s":N,"outcome":"...","browse":"...","session":"..."} — has duration/outcome but no `v` field, no `_repo_slug`, no repo field
  - v1: {"v":1,"skill":"X","ts":"...","duration_s":N,"outcome":"...","_repo_slug":"...",etc} — note: v1 entries may have `event_type` values other than `skill_run` (e.g. `upgrade_prompted`). Only `event_type: "skill_run"` or absent `event_type` are actual skill runs.
  - Event entries (e.g. {"event":"resources_shown",...}) are NOT skill runs. Filter out any entry where: `event` field is present, OR `event_type` is present and not equal to `skill_run`, OR `skill` is empty/missing. This catches both simple-schema event entries and v1-schema non-run entries like `event_type: "upgrade_prompted"`.
- Repo field normalization: Simple uses `repo`, v1 uses `_repo_slug` (with owner prefix like "jcl2018-portfolio"). Intermediate has no repo field. Normalize: `repo = .repo // (._repo_slug | sub("^[^-]+-"; "")) // "unknown"`.
- Integrated into existing terminal ASCII dashboard. No separate flags or modes.
- Graceful degradation: if skill-usage.jsonl doesn't exist or is empty, skip with message.

## Premises

1. Usage analytics belongs in /system-health as a new section, not a separate skill.
2. The usage section is unscored (like waza) since it reads from ~/.gstack/.
3. All computation happens in bash/jq, Claude only interprets.
4. skill-usage.jsonl is the sole data source. No new instrumentation needed.
5. "Smart insights" use Claude's interpretation of bash-computed stats.

## Cross-Model Perspective

Codex proposed a "behavioral topology overlay" — overlaying actual usage on the
installed dependency graph to reveal installed-vs-used gaps, repeated skill chains,
"dead weight" skills, and "habit regressions." This concept was adopted as the
core differentiator.

Codex also proposed building the "deterministic usage reducer" first — normalize
both JSONL formats, emit structured summary lines, then render. "The reducer is
the product." This matches our approach exactly.

Codex suggested three starter insights: stopped-using, long-and-failing,
discovery-gap. Adopted as the v0.3.0 insight scope.

Codex referenced GoAccess as the closest existing tool (50% of the way there).

## Approaches Considered

### Approach A: Simple Usage Table
Sorted list of skills with total runs and last-used date. Ships in ~15 min.
Completeness 4/10 — misses trends, durations, insights entirely.

### Approach B: Full Observability Dashboard (CHOSEN)
Comprehensive bash/jq reducer with 5 dashboard panels, installed-vs-used overlay,
3 rule-based insights, time comparisons. Estimated ~45-60 min with CC.
Completeness 9/10.

### Approach C: Layered Build
Ship A as v0.3.0, add B as v0.4.0. Lower risk per PR but slower to full value.
Two version bumps, intermediate incomplete state.

## Recommended Approach

Approach B: Full Observability Dashboard.

### Architecture

New **Step 4.5: Usage Trends (unscored)** inserted between Step 4 (Waza) and
Step 5 (Score + Trend). Steps are NOT renumbered — 4.5 is a literal label to
minimize diff against the existing SKILL.md. One bash/jq script that:

1. **Filters** out non-run entries (where `event` field is present) and **normalizes**
   all three JSONL schemas into a common shape:
   `ts, day, hour, repo, skill, duration_s, outcome, schema_version`
   Repo normalization: `.repo // (._repo_slug | sub("^[^-]+-"; "")) // "unknown"`
   Schema detection: `v` field present = v1, `duration_s` present but no `v` = intermediate, otherwise = simple

2. **Emits structured lines** that Claude interprets:

   ```
   USAGE_OVERVIEW: total_runs=N, unique_skills=N, date_range=Nd, unique_repos=N
   USAGE_COMPARE: last_7d=87, prev_7d=102, delta=-14.7%
   USAGE_SKILL: skill=office-hours, runs=N, pct=N%, last=DATE, avg_dur=Ns, min_dur=Ns, max_dur=Ns, success=N%, error=N%, abort=N%
   USAGE_SKILL: skill=ship, runs=N, pct=N%, ...
   USAGE_REPO: repo=portfolio, runs=180, top_skill=office-hours, pct=42%
   USAGE_REPO: repo=exploration, runs=45, ...
   USAGE_HOUR: hour=16, runs=52
   USAGE_HOUR: hour=21, runs=48
   USAGE_INSTALLED_VS_USED: installed=N, ever_used=N, never_used=N
   USAGE_NEVER_USED: skill=canary,benchmark,design-html,freeze,...
   USAGE_ANOMALY: type=stopped-using, skill=review, last_seen=2026-03-28, days_ago=13
   USAGE_ANOMALY: type=long-and-failing, skill=autoplan, avg_dur=2400s, success=60%
   USAGE_ANOMALY: type=discovery-gap, repo=exploration, missing=review,health
   ```

3. **Installed-vs-used overlay**: Cross-references the skill inventory from Step 1
   with usage data to identify:
   - Skills installed but never invoked (dead weight)
   - Skills invoked but with high error rates (unhealthy usage)
   - Hub skills (high dependency centrality) that are rarely directly used

### Three Rule-Based Insights (v0.3.0 scope)

1. **stopped-using**: Skill was active in prior 14 days, zero usage in last 14 days.
   Threshold: at least 3 runs in prior period.

2. **long-and-failing**: Skill has above-median duration AND below 80% success rate.
   Median computed across all skills that have duration_s data (both intermediate
   and v1 schemas). Only fires for skills with 5+ entries that include duration/outcome.

3. **discovery-gap**: Repo has 10+ total skill runs but zero runs of review, health,
   or investigate skills. These are the v0.3.0 defaults — a reasonable heuristic for
   "skills most repos should use." Could be made configurable in a future version.

### Graceful Degradation

- No skill-usage.jsonl: "No usage data found. Run skills with gstack to start collecting."
- Empty file: same as above.
- No v1 entries (duration/outcome missing): skip duration and success columns, show
  "N/A" for those fields. Insights 1 (stopped-using) still works. Insights 2-3 skip.
- jq unavailable: skip usage trends section entirely with message "jq required
  for usage trends. Install jq to enable." (jq is already required by existing
  Steps 1 and 5, so this is a pre-existing dependency, not a new one.)

## Open Questions

1. Should "never used" list exclude gstack internal skills (like the gstack meta-skill)?
   Probably yes — filter out skills that are frameworks, not user-invokable.
2. Should peak hours use local time or UTC? The JSONL stores UTC. Local time is more
   useful but requires timezone detection.
3. Future: should the timeline.jsonl (richer session data) also feed into this? v0.3.0
   scope says no (skill-usage.jsonl only), but it's worth considering for v0.4.0.

## Success Criteria

- Running /system-health shows the USAGE TRENDS section with all 5 panels
- Never-used skills are correctly identified by cross-referencing installed skills
- At least one insight fires when applicable data exists
- Graceful degradation when data is missing or sparse
- No new dependencies (bash + jq only, jq already required by existing steps)
- SKILL.md version bumped to 0.3.0
