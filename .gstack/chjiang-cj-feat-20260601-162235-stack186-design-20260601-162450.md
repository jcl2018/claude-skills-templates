## Problem Statement

F000032 (PR #186) makes per-skill `USAGE.md` mandatory and audits its *structural existence* via `scripts/validate.sh` Check 13 (the five required H2 sections must be present). What Check 13 does NOT catch: USAGE.md going stale when its sibling SKILL.md changes. A SKILL.md edit that adds new behavior, renames a flag, or removes a feature can leave USAGE.md silently out of date, and Check 13 still passes.

The user's reasonable next ask in conversation: "will /document-release recognize these and update?" Honest answer: no. `/document-release` is upstream gstack — it doesn't know USAGE.md as a doc class. The workbench has to add the drift signal itself.

This feature closes that gap with a cheap, deterministic, repo-internal check.

## What Makes This Cool

This is the smallest possible drift-detection surface that survives the same 12-month horizon as F000032. It ships ONE new validate.sh check (~20 lines), one CLAUDE.md paragraph documenting the operator override, one PHILOSOPHY.md sentence extending the `## Documentation surfaces` section, and one test in `scripts/test.sh` to prove it actually fires. No new file class, no new scripts, no upstream coupling. The check uses git's commit history (not filesystem mtimes, which are unreliable across worktrees), so the answer is the same on every machine.

The bonus: the check is symmetric with F000032's audit. F000032 says "USAGE.md must exist with these headings." This feature says "and its content must be at least as recent as the SKILL.md it documents." Together, the two checks cover (presence + structure + freshness). Nothing more.

## Constraints

- **Workbench-only scope.** Per memory `feedback_workbench_scope` — drift check runs on `skills/` only, never on `work-copilot/`, `deprecated/`, or downstream consumers.
- **Audit predicate matches F000032.** Same `jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json`. Different predicate would let the two checks fall out of sync. Re-use, don't fork.
- **Use commit timestamps, NOT filesystem mtimes.** `git log -1 --format=%ct -- <path>` gives the committer Unix timestamp of the most recent commit that touched the file. Filesystem mtimes are not portable across worktrees, fresh clones, or CI runners; commit timestamps are deterministic and reproducible. The check MUST use commit timestamps.
- **ERROR severity, with a cheap operator override.** Per memory `feedback_skill_contracts_strict`: WARN gets ignored (F000030's 1/13 DESIGN.md adoption rate is the proof). ERROR with a documented override is the load-bearing pattern in this workbench. The override is intentional: if SKILL.md changed but USAGE.md genuinely doesn't need updating (cosmetic typo, version bump, etc.), the operator runs a documented one-liner that re-stamps USAGE.md as "verified current."
- **Operator override mechanism: bump `last-updated:` frontmatter.** If the operator confirms USAGE.md is still accurate after a SKILL.md edit, they bump the `last-updated:` YAML field in USAGE.md's frontmatter to today's date, then commit. This produces a real content change (so `git log -1 -- <path>` picks up the new commit and `%ct` advances) AND leaves a human-readable audit trail (the date field). Empty commits via `git commit --allow-empty` do NOT work — `git log -1 -- <path>` only returns commits that touched the path, and empty commits touch no paths. The `last-updated:` field is already in F000032's template frontmatter, so every existing USAGE.md has it.
- **No upstream `/document-release` coupling.** Upstream skills are not ours to modify. The drift check is repo-internal. If future-us wants to surface drift via the F000028 doc-sync-pending mechanism, that's a separate follow-up (Open Question 5).
- **Same-commit edits are not drift.** When F000032 first committed all 11 USAGE.md alongside the SKILL.md updates, every `%ct` matched. The check uses `>` not `>=`, so equal timestamps are not drift. This is the correct behavior — both files updated in the same atomic PR is exactly the convention F000032 establishes.
- **Brand-new skills are not drift.** Per CLAUDE.md "Creating a new skill," step 1 creates SKILL.md and step 5 creates USAGE.md, both shipped in one PR. They land in the same commit, same `%ct`, no fire on first ship.
- **Stacking-on-#186 acknowledgment.** This PR's branch is `cj-feat-20260601-162235-stack186`, branched off `cj-feat-20260601-152835-3769` (PR #186), NOT off `main`. The validate.sh Check 14 logic depends on USAGE.md files existing, which they do only in PR #186's branch. Merge order: PR #186 first, then this PR. If PR #186 is closed without merging, this PR rebases onto main and naturally fails to apply (because skills/*/USAGE.md don't exist on main) — fail-loud, not silent-pass.

## Premises

(confirmed via Approach A selection)

1. The gap is content-staleness, not structural-presence (F000032 already covers presence + 5 H2 sections).
2. Workbench-internal solution: extend validate.sh, no upstream skill modification.
3. ERROR severity, not WARN. WARN gets ignored; ERROR + documented override is the workbench-load-bearing pattern.
4. Git commit timestamps, not filesystem mtimes (deterministic across machines).
5. Stacks on PR #186; merge order matters; failure mode on rebase-onto-main is fail-loud (Check 14 references USAGE.md paths that wouldn't exist).

## Approaches Considered

(reproduced from the AUQ for the design doc record)

### Approach A: Lightweight mtime check + ERROR (chosen)

Add Check 14 to validate.sh: for every routable non-deprecated skill, compare `git log -1 --format=%ct -- skills/{name}/SKILL.md` vs the same query on USAGE.md. If SKILL.md `%ct` > USAGE.md `%ct`, ERROR with the diff range. ~20 lines of bash, one new check. Drift surfaces at every commit. Operator override: `git commit --allow-empty` after touching USAGE.md.

**Pros**: Cheap (no heuristic tuning, no new scripts); deterministic (commit timestamps); symmetric with Check 13 predicate; ERROR severity enforces consistency.

**Cons**: Fires on cosmetic SKILL.md edits (typo, version bump). Mitigation is the documented override — but the override is friction the operator hits on every cosmetic edit. Acceptable cost for the drift signal.

### Approach B: Content-aware diff (validate-time)

Add `scripts/lint-usage.sh` comparing SKILL.md `description:` against USAGE.md `## When to use` via similarity heuristic.

**Pros**: Higher signal; cosmetic edits don't fire.

**Cons**: ~5x complexity, heuristic to tune, false positives/negatives erode trust. Overkill for the drift signal needed.

### Approach C: Same as A but WARN, not ERROR

**Pros**: Never blocks legitimate PRs.

**Cons**: WARN gets ignored. F000030's 1/13 DESIGN.md adoption is the proof. The workbench enforces via ERROR-with-override; this PR follows that pattern.

### Approach D: Hook into /document-release post-merge AUQ

**Pros**: Surfaces drift at the natural refresh moment, not pre-commit.

**Cons**: Doesn't protect against in-branch drift; depends on F000028 hook surface; more moving parts.

## Recommended Approach

**Approach A**, lightweight git-log-based check + ERROR. Detailed plan:

### Step 1: Add Check 14 to scripts/validate.sh

Insert after Check 13. Same predicate (`status != "deprecated"` + non-empty `files`). Loop:

```bash
# Check 14: USAGE.md content freshness (git-log %ct comparison)
echo "=== Check 14: USAGE.md content freshness ==="
while IFS= read -r SKILL_NAME; do
  SKILL_PATH="skills/$SKILL_NAME/SKILL.md"
  USAGE_PATH="skills/$SKILL_NAME/USAGE.md"
  if [ ! -f "$SKILL_PATH" ] || [ ! -f "$USAGE_PATH" ]; then
    # Check 13 already errored on missing USAGE.md; SKILL.md missing is a different bug.
    continue
  fi
  SKILL_CT=$(git log -1 --format=%ct -- "$SKILL_PATH" 2>/dev/null)
  USAGE_CT=$(git log -1 --format=%ct -- "$USAGE_PATH" 2>/dev/null)
  if [ -z "$SKILL_CT" ] || [ -z "$USAGE_CT" ]; then
    # Untracked-or-staged-only: git log returns empty. Skip drift check (Check 13 covers presence).
    echo "  SKIP: $USAGE_PATH (file untracked or staged-only; drift check requires committed history)"
    continue
  fi
  if [ "$SKILL_CT" -gt "$USAGE_CT" ]; then
    SKILL_SHA=$(git log -1 --format=%h -- "$SKILL_PATH")
    USAGE_SHA=$(git log -1 --format=%h -- "$USAGE_PATH")
    echo "  ERROR: $USAGE_PATH is stale (SKILL.md last updated at $SKILL_SHA, USAGE.md last updated at $USAGE_SHA)."
    echo "         If USAGE.md is still accurate, bump its last-updated frontmatter field and commit:"
    echo "           sed -i.bak 's/^last-updated:.*/last-updated: \"'\"\$(date +%Y-%m-%d)\"'\"/' $USAGE_PATH && rm ${USAGE_PATH}.bak"
    echo "           git add $USAGE_PATH && git commit -m \"docs: verify USAGE.md current for $SKILL_NAME\""
    ERRORS=$((ERRORS+1))
  else
    echo "  PASS: $USAGE_PATH is current (SKILL.md $SKILL_CT <= USAGE.md $USAGE_CT)"
  fi
done < <(jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json)
```

Three implementation notes:
- The check uses `git log -1 --format=%ct` (committer Unix timestamp), not author timestamp. Committer time is what `git log` orders by and matches operator intuition.
- If `git log` returns empty (file untracked, never committed, or only staged), the check SKIPs with a one-line note. Check 13 already enforces presence; this is a deliberate handoff.
- The error message embeds the exact override command. Operator copy-pastes when they're confident USAGE.md is still accurate.

Add a paragraph to the `## Conventions` section, right after the "Skill directory structure" sub-section:

```markdown
### USAGE.md drift detection

`scripts/validate.sh` Check 14 detects USAGE.md content drift: if SKILL.md has a more
recent commit than USAGE.md, the audit flags USAGE.md as stale. The check uses git
commit timestamps (`git log -1 --format=%ct`), not filesystem mtimes — deterministic
across worktrees, fresh clones, and CI runners.

When the drift signal is real, update USAGE.md to match the new SKILL.md behavior
(this is the normal path). When SKILL.md changed cosmetically (typo, version bump,
comment edit) and USAGE.md is still accurate, bump the `last-updated:` field in
USAGE.md's frontmatter and commit:

\`\`\`bash
sed -i.bak 's/^last-updated:.*/last-updated: "'"$(date +%Y-%m-%d)"'"/' skills/{name}/USAGE.md && rm skills/{name}/USAGE.md.bak
git add skills/{name}/USAGE.md && git commit -m "docs: verify USAGE.md current for {name}"
\`\`\`

This is a real content change (one frontmatter line bumped), so `git log -1` picks up
the new commit and USAGE.md's `%ct` advances past SKILL.md's. The `last-updated:` field

that touched the path, and empty commits touch no paths, so they don't advance `%ct`.
```

## Open Questions

1. **Should the override commit message follow a parseable convention?** No — the documented format (`docs: verify USAGE.md current for <name>`) is recommendation, not enforcement. Future-us could parse it if a `/document-release` audit hook ever wants to surface "how many verifies in the last 30 days?" but v1 doesn't need it.
2. **Should Check 14 SKIP or ERROR when USAGE.md is untracked / staged-only?** SKIP. Check 13 already covers presence; Check 14 is freshness, and "freshness" of an uncommitted file is meaningless. The skip emits a visible one-line note so the operator knows it didn't run, not a silent pass.
3. **Should the check distinguish "real" SKILL.md changes from cosmetic ones?** No. Distinguishing is heuristic (see Approach B), and the documented override is cheap (one `touch + commit`). Forcing the operator to acknowledge cosmetic edits via the override IS the audit trail.
4. **How does the override actually advance USAGE.md's `%ct`?** Initial draft used `touch + commit` and then `git commit --allow-empty`. Both fail: `touch` advances filesystem mtime but doesn't change content, so `git add` finds nothing; `--allow-empty` creates a commit but `git log -1 -- <path>` only returns commits that *touched* the path, so empty commits don't advance `%ct`. **Resolution**: bump the `last-updated:` frontmatter field in USAGE.md via `sed -i.bak` (macOS BSD sed compatible). The field is already present in F000032's template + the 11 backfill files. One-line content change → real commit → `git log -1` returns the new commit → `%ct` advances. Audit trail is the date. Steps 1, 2, 3 all updated.
5. **Should the F000028 post-merge AUQ hook surface this drift signal?** Deferred — Approach D's premise. Out of scope for v1; tracked as a TODOS row if the user finds the validate-time signal insufficient.
6. **Should brand-new skill creation be tested too?** The existing `manual-skill-creation` test (scripts/test.sh:170-220) already exercises the "fresh SKILL.md + USAGE.md committed atomically" path and proves Check 14 doesn't fire on it. Coverage is sufficient; no new test needed for that path.
