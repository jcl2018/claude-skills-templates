## Subagent Invocation Pattern (QA engineer)

```
parent skill (e.g., /qa-work-item)
    │
    │ Agent tool (subagent_type, fresh context)
    │ prompt: "you are a QA engineer, read TEST-SPEC at <path>,
    │          verify acceptance criteria, report findings"
    ▼
fresh-context subagent
    ├── reads TEST-SPEC.md          (Read)
    ├── runs smoke scripts          (Bash)
    ├── reads implementation files  (Read)
    └── diffs outputs               (Bash)
    │
    │ return: 1-2 sentences + file pointers
    │ "smoke green; E2E criterion 1 passed,
    │  criterion 2 ambiguous (reason: ...),
    │  criterion 3 failed (reason: ...).
    │  Wrote findings to {tracker-path}."
    ▼
parent skill receives short report
    ├── writes findings to tracker journal
    └── AskUserQuestion if any red/ambiguous
```

## Problem Statement

The May 5 design defined the user's day-to-day workflow as 6 steps:

1. `/office-hours` → design plan in `~/.gstack/projects/`
2. **Convert design doc → work-item structure (manual)**
3. **Implement (manual)**
4. **Tests: smoke + E2E (manual)**
5. `/ship` → PR + bundled internal review
6. `/land-and-deploy` → merges + verifies

Steps 1, 5, 6 are skill-automated. Steps 2, 3, 4 are not — `WORKFLOW.md` describes the rules in prose and tells the AI to read them and follow them, but no skill *executes* them. Today the user does these three steps by directing Claude inline: "scaffold a feature for this design," "now implement it per SPEC," "now run the tests."

This means three things suffer:

- **No structural enforcement at scaffold time.** `/personal-workflow check` validates *after* the AI scaffolds; it cannot prevent a malformed scaffold from happening, only catch it post hoc.
- **No isolation across phases.** Scaffolding context (templates, manifests, design doc) bleeds into implementation context (codebase, prior commits, test infra) bleeds into QA context. The user wants subagent isolation per phase so each agent's prompt is small and well-scoped.
- **No standard handoff shape.** Today, scaffold→implement→test handoff is conversational. A standardized skill triplet that reads/writes only handoff docs (TRACKER.md, DESIGN.md, SPEC.md, TEST-SPEC.md) would make the workflow reproducible across sessions and machines.

## What Makes This Cool

The three-skill pipeline gives every workflow step its own skill, completing the May 5 artifact-to-skill mapping:

| Step | Doc produced | Skill that produces / consumes it |
|---|---|---|
| 1. Office hours | `~/.gstack/projects/...-design-*.md` | `/office-hours` (existing) |
| 2. Scaffold | `work-items/{type}s/{ID}_{slug}/{TRACKER, DESIGN, SPEC, ROADMAP, TEST-SPEC, test-plan}` | `/scaffold-work-item` (NEW) |
| 3. Implement | code + tracker journal entries | `/implement-from-spec` (NEW) |
| 4. Test | smoke + E2E results in tracker | `/qa-work-item` (NEW) |
| 5. Ship | commit history + PR | `/ship` (existing, unchanged) |
| 6. Deploy | merge + post-deploy verification | `/land-and-deploy` (existing, unchanged) |

Forcing function: every step now has a named skill. The personal-workflow surface becomes "six skills, three artifacts each step writes, two existing artifacts each step reads." Anyone joining the workbench can read the artifact set + skill list and understand the whole loop without reading conversation transcripts.

A second affordance: each new skill can internally delegate to a fresh-context subagent with the relevant handoff docs as the only input. The QA skill in particular gets to define a "QA engineer" subagent prompt that reads `TEST-SPEC.md` and verifies acceptance criteria — script-driven where possible, agent-judgment where not. This generalizes the workbench's testing model beyond bash scripts.

## Constraints

- **Personal-workflow only.** Company-workflow at `deprecated/company-workflow/` and the `work-copilot/` byte-mirror are out of scope. These three skills do not need a Copilot equivalent in v1.
- **No orchestrator in v1.** The user explicitly chose three independent skills over a single `/personal-pipeline` orchestrator (Approach A over B). The orchestrator is a follow-up decision after these three skills prove the handoff-doc pattern.
- **Each skill takes a path argument.** `/scaffold-work-item <design-doc-path>`, `/implement-from-spec <work-item-dir>`, `/qa-work-item <work-item-dir>`. No magic inference from branch or "most recent thing." Explicit > implicit for skills the user composes manually.
- **Skills follow the personal-workflow + workbench conventions.** Frontmatter with `name`, `description`, `version`, `allowed-tools`. Cataloged in `skills-catalog.json`. Validated by `validate.sh`. Deployed by `skills-deploy install`.
- **Templates remain the single source of truth.** The scaffold skill *reads* templates from `templates/personal-workflow/` (or `~/.claude/templates/personal-workflow/`) per the existing 2-level fallback chain. It does not embed template content. Edit a template, scaffold output changes.
- **Trackers are LLM-driven, not bash.** Per the workbench's existing `validators-may-be-llm-not-bash` learning. Each skill is a SKILL.md instruction file Claude reads and follows, not a bash script that does the work.

## Premises (all confirmed in office-hours session)

1. **Token savings from subagent isolation are real but conditional.** Savings only materialize if (a) the orchestrator/parent skill's prompt to each subagent is tiny (a few hundred tokens, mostly file paths), and (b) the subagent's return report is short (1–2 sentences plus pointers to files it wrote). If a subagent returns 2 KB of "here's what I did," savings collapse. The handoff docs (DESIGN.md, SPEC.md, TEST-SPEC.md, smoke output, E2E output) carry the actual context.
2. **E2E becomes agent-driven (supersedes May 5).** May 5 said "E2E walked once before ship" was a deliberate human checkpoint. v3 (this design) says a "QA engineer" subagent runs E2E using the prompt pattern: *"You are a QA engineer. Read `TEST-SPEC.md`. Verify the smoke and E2E acceptance criteria for this work item. Use whatever tools you need — run scripts, read code, diff outputs. Report findings: green / red / questions."* Generalizes beyond pre-scripted test harnesses.
3. **Pipeline scope: three skills, narrow.** `/scaffold-work-item`, `/implement-from-spec`, `/qa-work-item`. `/ship` and `/land-and-deploy` stay outside the pipeline and are invoked separately by the user. `/office-hours` also stays outside (inherently human-in-loop dialogue).
4. **Human-in-loop only at decision gates** (autoplan-style). Within each skill: scaffold approval (does the work-item shape match intent?), taste decisions during implementation (when subagents disagree on architecture), QA result review (only if QA agent flags issues), final go/no-go on shippability. Otherwise fully autonomous within each skill's run.
5. **Subagent infra: Claude Code's `Agent` tool with `subagent_type`.** Each skill's internal delegation uses Agent tool calls. Worktree isolation (`isolation: "worktree"`) available where filesystem isolation matters (e.g., implementer that touches code while QA is reading the prior state). Process-level isolation (separate `claude -p` subprocesses) explicitly rejected as too much new infra for marginal gains.

## Approaches Considered

### Approach A: Three independent skills, no orchestrator (recommended; chosen)
Build `/scaffold-work-item`, `/implement-from-spec`, `/qa-work-item` as standalone skills. User invokes each in sequence with explicit path arguments. Each skill reads handoff docs, internally delegates to subagents where useful, returns control to the user. Smaller per-skill scope, ships incrementally. Three keystrokes per pipeline run instead of one. Loses the autoplan-style "one keystroke" affordance but preserves all the agent-isolation and handoff-doc benefits within each skill.

### Approach B: Single `/personal-pipeline` orchestrator with three subagent phases
One skill orchestrates the three phases. Orchestrator passes only file paths to each subagent. Subagents return 1–2 sentence reports. Human gates at scaffold approval, taste decisions, QA review, ship-readiness. One keystroke per pipeline run. ~600–800 lines of skill markdown, debugging requires phase-level observability. Matches the user's original stated vision but trades simplicity for keystroke-economy.

### Approach C: Multi-session shell harness (`scripts/run-pipeline.sh`)
Process-level isolation via `claude -p` headless invocations. True context isolation by construction. Loses interactive AskUserQuestion gates across sessions; would require inventing a "wait for confirmation" file-based mechanism. Most architecturally pure, most new infra, worst failure modes.

### Approach D: Hybrid — A first, then B on top (offered but not chosen)
Ship A as standalone skills, validate the handoff-doc thesis on real work items, then build B as a thin orchestrator wrapping the three. Two phases of work, ~3 weekends total. The user did not pick this option, but Approach A naturally evolves toward Hybrid if the orchestrator turns out to be worth building later — the three skills' interfaces are already orchestrator-friendly.

## Recommended Approach

**Approach A: Three independent skills, no orchestrator in v1.**

### Skill 1: `/scaffold-work-item`

**Input:** `design-doc-path` (path to a `~/.gstack/projects/{slug}/...-design-*.md` file)

**Behavior:**

1. Read the design doc. Extract: title, mode, problem statement, recommended approach, success criteria.
2. Determine work-item type from current branch name (feature/feat → feature, story → user-story, task/chore → task, fix/defect/bugfix → defect). Confirm with the user via AskUserQuestion.
3. Generate ID: scan `work-items/` for highest existing ID of that type, increment.
4. Read `personal-artifact-manifests.json` to determine required artifacts for the type.
5. Read each required template from the 2-level fallback chain (`$REPO_ROOT/templates/personal-workflow/` then `~/.claude/templates/personal-workflow/`).
6. For each artifact, fill placeholders (`{ITEM_NAME}`, `{ITEM_ID}`, `{YYYY-MM-DD}`, `{BRANCH_NAME}`, `{author}`) and replace template prose with content distilled from the design doc.
7. Write the work-item directory: `work-items/{type}s/{slug}/{ID}_{slug}/` with all required artifacts. For features and user-stories, write `DESIGN.md` from the design doc's relevant sections. For features, scaffold required user-story children alongside per WORKFLOW.md.
8. Run `/personal-workflow check` on the new directory. If violations, surface them and ask the user to confirm before proceeding.

**Subagent use:** A "scaffold validator" subagent that takes the freshly-written directory and the relevant template, returns "shape green / shape red + specific deltas." Optional in v1 if `/personal-workflow check` already covers this.

**Decision gate:** AskUserQuestion to confirm the proposed work-item ID, slug, and directory path before writing files.

### Skill 2: `/implement-from-spec`

**Input:** `work-item-dir` (path to a scaffolded work item, e.g. `work-items/features/personal-workflow/F000010_pipeline_skills/`)

**Behavior:**

1. Read TRACKER.md, DESIGN.md, SPEC.md, ROADMAP.md (if present, for features). Validate that all required handoff docs exist (otherwise: "Run `/scaffold-work-item` first or fix the directory shape").
2. Determine the implementation surface: SPEC's `## Architecture` + `## Acceptance Criteria` sections name the files and changes.
3. Implement: write/edit code per the SPEC. Use Read, Edit, Write, Bash as needed.
4. Update tracker: append journal entries grouped by category (decision / finding / implementation), reference commit SHAs once committed.
5. Move tracker through Phase 1 → Phase 2 lifecycle gates as their preconditions are met.

**Subagent use:** A fresh-context "code reviewer" subagent for taste decisions where the implementation has multiple plausible architectural choices and the parent agent wants a second opinion. Worktree isolation if the implementation is large.

**Decision gate:** AskUserQuestion when subagent disagrees with the parent on architecture, when SPEC has gaps requiring user input, or before committing if changes touch sensitive surfaces (skill catalog, manifests, validators).

### Skill 3: `/qa-work-item`

**Input:** `work-item-dir` (path to a work item with implementation done)

**Behavior:**

1. Read TRACKER.md (verify Phase 2 gates are checked) and TEST-SPEC.md.
2. Run smoke tests. For this workbench, smoke = `scripts/test.sh` or `scripts/validate.sh`. For other consumers, the TEST-SPEC's `## Smoke Tests` section names the commands.
3. Run E2E. Delegate to a "QA engineer" subagent with this prompt shape:
   > "You are a QA engineer. Read TEST-SPEC.md at {path}. Verify the E2E acceptance criteria. You may run scripts, read code, diff outputs, or test fixtures — whatever the criteria require. The implementation is at {work-item-dir}. Report findings: which criteria passed, which failed, which are ambiguous. Be specific — name the criterion, name the evidence."
4. Write smoke + E2E results into the tracker as journal entries (or a dedicated `## QA Run` section).
5. Update tracker: check Phase 2's `[ ] Smoke pass` and `[ ] E2E walked` gates if both green.

**Subagent use:** The QA engineer subagent (always). Optionally a second QA subagent with a different framing ("you are an adversarial QA engineer — try to break this") for high-stakes work items.

**Decision gate:** AskUserQuestion only if the QA engineer flags red or ambiguous results. Green path is silent and writes the tracker update directly.

### Composition

User runs:
```
/scaffold-work-item ~/.gstack/projects/jcl2018-claude-skills-templates/chjiang-main-design-20260508-102829.md
# inspects the new work-items/.../F000010_pipeline_skills/
/implement-from-spec work-items/features/personal-workflow/F000010_pipeline_skills/F000010_pipeline_skills/

/qa-work-item work-items/features/personal-workflow/F000010_pipeline_skills/F000010_pipeline_skills/

```

Three new skill invocations + two existing skills = full workflow. The three new skills internally use subagents for isolation and for the QA-engineer pattern.

## Open Questions

1. **Where does the QA engineer prompt template live?** Options: (a) hardcoded in `skills/qa-work-item/SKILL.md`, (b) a separate prompt file under the skill directory (`prompts/qa-engineer.md`), (c) a frontmatter field in TEST-SPEC.md so each work item can customize. Recommendation: (a) for v1, with (b) as an extension if reuse demands it.
2. **How does `/implement-from-spec` decide between propose-and-confirm vs. write-directly?** A small change (one file, no architecture choice) should write directly. A large change with architecture forks should propose and confirm. Heuristic needed. Recommendation: propose-and-confirm by default; the user can toggle "just do it" via a skill argument.
3. **Does the scaffold skill update the parent design doc at `~/.gstack/projects/...`?** E.g., adding a "Status: SCAFFOLDED → work-items/.../F000010" line. Recommendation: yes, append a small footer; lets the user trace which design doc became which work item without grep.
4. ~~Failure-state contract~~ → RESOLVED by Eng-Review Revision 1.1 (idempotency premise).
5. ~~Relationship to /personal-workflow check~~ → RESOLVED by Eng-Review Revision 1.3 (boundary validation premise).
6. **Build order:** ship `/scaffold-work-item` first, validate on a real work item (probably this very design's work item), then `/qa-work-item`, then `/implement-from-spec`. ## Success Criteria

- All three skills exist with valid frontmatter, cataloged in `skills-catalog.json`, validated by `validate.sh`, deployed by `skills-deploy install`.
- A new work item can be created end-to-end via `/scaffold-work-item` → `/implement-from-spec` → `/qa-work-item` with no inline scaffolding instructions to Claude.
- `/scaffold-work-item` produces a directory that passes `/personal-workflow check` on first try.
- `/qa-work-item` runs smoke + E2E on the workbench's own validate.sh / test.sh and writes a structured tracker update.
- Each skill's invocation prompt to its subagents is < 500 tokens (orchestrator's contribution to the subagent context window). Subagent return reports are < 200 tokens. Verified by inspection on a real run.
- The three skills' SKILL.md files together total < 1500 lines. Smaller than `autoplan/SKILL.md` (1700+ lines) — proves the per-skill simplicity claim.
