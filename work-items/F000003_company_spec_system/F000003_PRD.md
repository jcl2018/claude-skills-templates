## Problem Statement

The company follows a formal work item specification that defines structural
validation rules (contract.json), tracker templates with Jira/TFS integration,
reference guides for AI doc generation, philosophy docs explaining lifecycle
rationale, and validation test fixtures. Today this spec lives at
`~/Downloads/spec/` with no deployment or enforcement mechanism. Engineers
must follow it manually.

## Mental Model

A new skill (`company-workflow`) packages the company spec as a deployable,
validatable template set inside this workbench repo:

```
templates/company-workflow/        (13 company spec templates)
skills/company-workflow/           (skill: enforcement, validation, guides)
template-registry.json             (declares all template sets, extensible)
```

Registry set names = skill or catalog entry names. The registry supports N sets.

## Acceptance Criteria

### Story #1: Company-compliant work items [core]

```
GIVEN the company spec templates exist at templates/company-workflow/
WHEN  I scaffold a new work item using the company skill
THEN  the tracker has all company-required frontmatter fields per contract.json
AND   the tracker has all required sections in expected order per contract.json
AND   the lifecycle has >= 4 checkboxes across 4 phases (Track, Implement, Review, Ship)
```

### Story #2: Template registry [core]

```
GIVEN template-registry.json exists at repo root
WHEN  I read the registry
THEN  it declares a "company-workflow" set pointing to templates/company-workflow/
AND   the set references contract path and guides path
AND   the registry schema supports adding additional sets without modifying existing entries
```

### Story #3: Validation entry point [core]

```
GIVEN a work item exists
WHEN  I run company-workflow validate on that path
THEN  exit code 0 if it passes all contract.json rules
AND   exit code 1 with stderr listing each specific violation
```

### Story #4: Zero regression [resilience]

```
GIVEN existing files in the repo
WHEN  company templates and skill are added
THEN  no existing files are modified
AND   ./scripts/validate.sh passes
AND   ./scripts/test.sh passes
```

### Story #5: Template format preservation [core]

```
GIVEN the 13 company templates are copied from ~/Downloads/spec/templates/
WHEN  I compare each file in templates/company-workflow/ to its spec source
THEN  every file is byte-identical to its source
```

### Story #6: Artifact completeness enforcement [core]

```
GIVEN a company work item exists at work-items/{slug}/
WHEN  I run company-workflow check on that path
THEN  for type=feature: verifies tracker + PRD + ARCHITECTURE + TEST-SPEC + milestones exist
AND   for type=defect: verifies tracker + RCA + test-plan exist
AND   for type=task: verifies tracker + test-plan exist
AND   for type=userstory: verifies tracker + PRD + ARCHITECTURE + TEST-SPEC + milestones exist
AND   for type=review: verifies tracker + review-notes exist
AND   each artifact's frontmatter has required fields per its template
AND   each artifact's sections match the template's required sections
AND   missing artifacts are flagged as [MISSING], field drift as [DRIFT]
```

## Out of Scope

- Automated sync from company spec upstream (one-time manual import)
- Converting existing work items (F000001, F000002) to company format
- Building a template switching CLI

## Assumptions

- The company spec at `~/Downloads/spec/` is the authoritative version
- `skills-deploy` can handle subfolder templates, or T000003 will patch it
