## Problem Statement

The company spec defines required companion artifacts per work item type: features
need a doc triplet + milestones, defects need RCA + test-plan, reviews need
review-notes, etc. Without enforcement, engineers can create a tracker with all
checkboxes checked but no PRD, and nothing catches it. The company skill needs
its own enforcement that validates artifact completeness, independent from /docs check.

## Mental Model

```
company-workflow check <path>
  │
  ├── reads company-artifact-manifests.json (type → required artifacts)
  ├── reads templates/company-workflow/ (expected fields + sections per artifact)
  ├── walks the work item directory
  └── reports: [MISSING], [DRIFT], [PASS] per artifact
```

## Company Artifact Manifest

| Type | Required artifacts | Count |
|------|-------------------|-------|
| feature | tracker (tracker-feature.md), PRD (doc-PRD.md), ARCHITECTURE (doc-ARCHITECTURE.md), TEST-SPEC (doc-TEST-SPEC.md), milestones (doc-milestones.md) | 5 |
| defect | tracker (tracker-defect.md), RCA (doc-RCA.md), test-plan (doc-test-plan.md) | 3 |
| task | tracker (tracker-task.md), test-plan (doc-test-plan.md) | 2 |
| userstory | tracker (tracker-user-story.md), PRD (doc-PRD.md), ARCHITECTURE (doc-ARCHITECTURE.md), TEST-SPEC (doc-TEST-SPEC.md), milestones (doc-milestones.md) | 5 |
| review | tracker (tracker-review.md), review-notes (doc-review-notes.md) | 2 |

Each artifact entry maps to a template in `templates/company-workflow/`. The template
defines expected frontmatter fields and `##` sections. Enforcement reads the template
to build the expected model, then compares against the actual work item files.

## Acceptance Criteria

### Story #1: Missing artifact detection [core]

```
GIVEN a feature work item with tracker but no PRD
WHEN  I run company-workflow check on that path
THEN  output includes [MISSING] PRD.md
AND   output includes [MISSING] for each other absent required artifact
```

### Story #2: Frontmatter drift detection [core]

```
GIVEN a company tracker missing the workflow_type field
WHEN  I run company-workflow check
THEN  output includes [DRIFT] TRACKER.md — missing required field "workflow_type"
```

### Story #3: Section drift detection [core]

```
GIVEN a company tracker with Lifecycle section but missing Journal section
WHEN  I run company-workflow check
THEN  output includes [DRIFT] TRACKER.md — missing section "Journal"
```

### Story #4: All 5 types enforced [core]

```
GIVEN one complete work item per type (feature, defect, task, userstory, review)
WHEN  I run company-workflow check on each
THEN  feature requires 5 artifacts, defect requires 3, task requires 2, userstory requires 5, review requires 2
AND   each complete work item gets [PASS] for all artifacts
```

### Story #5: Independent from /docs check [resilience]

```
GIVEN company-workflow check and /docs check both exist
WHEN  I run /docs check
THEN  it does NOT read company-artifact-manifests.json
AND   it does NOT validate against templates/company-workflow/
WHEN  I run company-workflow check
THEN  it does NOT read artifact-manifests.json
AND   it does NOT validate against templates/*.md (root level)
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Types covered | 5/5 | Enforcement has rules for feature, defect, task, userstory, review |
| Missing artifact detection | catches all | Scaffold each type incomplete, verify [MISSING] flags |
| Drift detection | catches field + section | Remove a field, remove a section, verify [DRIFT] flags |
| No /docs check interference | 0 side effects | Run /docs check before and after, output identical |

## Out of Scope

- Lifecycle consistency checks (parent/child state validation) — defer to later
- Cross-reference traceability (PRD stories → TEST-SPEC coverage) — defer to later
- Auto-fix for missing artifacts — enforcement reports, does not scaffold

## Assumptions

- The company-artifact-manifests.json format mirrors artifact-manifests.json structure
- Templates in templates/company-workflow/ are the source of truth for expected fields and sections
