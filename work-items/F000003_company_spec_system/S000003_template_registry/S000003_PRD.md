## Problem Statement

The company spec has 13 template files that need to live in this repo without
naming conflicts. The repo needs a namespace mechanism (subfolder) and a registry
(template-registry.json) that makes template set boundaries explicit and extensible.

## Mental Model

```
template-registry.json
  ├── "workbench"         → templates/                  (this repo's dev workflow)
  ├── "company-workflow"  → templates/company-workflow/  (company spec skill)
  └── "{future-skill}"   → templates/{future-skill}/   (extensible)
```

Registry set names identify the workflow they serve. Each name should be specific
enough that no set looks like "the default."

### Format preservation rule

All 13 files are copied **verbatim** from `~/Downloads/spec/templates/`. This is a
one-time manual import. After commit, the repo is the source of truth. Acceptance
criterion: `diff` each file against its source shows zero differences.

## Acceptance Criteria

### Story #1: Registry exists [core]

```
GIVEN the repo root is checked
WHEN  I look for template-registry.json
THEN  it exists and parses as valid JSON
AND   it has a "version" field (integer) and a "sets" object
```

### Story #2: Company-workflow set declared [core]

```
GIVEN template-registry.json is parsed
WHEN  I read the "company-workflow" set
THEN  path is "templates/company-workflow/"
AND   types include feature, defect, task, userstory, review
AND   contract points to skills/company-workflow/contract.json
AND   guides points to skills/company-workflow/reference/
```

### Story #3: All 13 templates present [resilience]

```
GIVEN templates/company-workflow/ directory exists
WHEN  I list its contents
THEN  it contains exactly 13 files:
      tracker-feature.md, tracker-defect.md, tracker-task.md,
      tracker-user-story.md, tracker-review.md,
      doc-ARCHITECTURE.md, doc-PRD.md, doc-RCA.md, doc-TEST-SPEC.md,
      doc-milestones.md, doc-review-notes.md, doc-scrum.md, doc-test-plan.md
```

### Story #4: Byte-identical to spec [core]

```
GIVEN the 13 files are copied from ~/Downloads/spec/templates/
WHEN  I diff each templates/company-workflow/ file against its source
THEN  every file shows zero differences
```

### Story #5: Tracker frontmatter complete [core]

```
GIVEN each tracker template in templates/company-workflow/
WHEN  I parse its YAML frontmatter
THEN  tracker-feature.md has 11 fields including workflow_type and url
AND   tracker-defect.md has 11 fields including workflow_type and url
AND   tracker-task.md has 12 fields including workflow_type, url, and parent
AND   tracker-user-story.md has 11 fields including workflow_type and url, with type: userstory
AND   tracker-review.md has 12 fields including workflow_type, url, and deadline
```

### Story #6: Section ordering preserved [core]

```
GIVEN each tracker template in templates/company-workflow/
WHEN  I extract ## headings
THEN  they appear in contract.json expected_order:
      Lifecycle, Todos, Log, PRs, Files, [Meetings], Insights, Journal, [Handoff]
      (Meetings and Handoff only in tracker-review.md)
```

### Story #7: Registry extensibility [usability]

```
GIVEN template-registry.json has entries
WHEN  a contributor adds a new entry to the "sets" object
THEN  no existing entries need modification
AND   the new entry follows the same schema (name, path, types, contract)
```

## Out of Scope

- Validation of the registry schema itself (the skill validates work items, not the registry)
- Migration tooling for converting existing work items to company format

## Assumptions

- Registry set names must match a skill name or catalog entry name (convention, not enforced)
- `skills-deploy` can handle subfolder templates, or T000003 will patch it
