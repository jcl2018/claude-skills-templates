## Problem Statement

The company follows a formal work item specification that defines structural
validation rules (contract.json), tracker templates with Jira/TFS integration,
reference guides for AI doc generation, philosophy docs explaining lifecycle
rationale, and validation test fixtures. Today this spec lives at
`~/Downloads/spec/` with no deployment or enforcement mechanism. Engineers
must follow it manually, and there is no way to scaffold new work items or
validate existing ones against the spec.

## Mental Model

Three capabilities in one skill (`company-workflow`):

```
template-registry.json (declares template sets)
    |
    v
templates/company-workflow/          (13 company spec templates)
    |
    +--> company-workflow validate   (contract.json structural rules)
    +--> company-workflow check      (artifact completeness per type)
    +--> company-workflow create     (scaffold new work items)
```

1. **Template Registry**: Where templates live. Subfolder namespacing under `templates/company-workflow/`, declared in `template-registry.json` alongside the existing workbench set. Two systems coexist without naming conflicts.
2. **Artifact Enforcement**: Are work items complete? Each type has required companion artifacts (feature=5, defect=3, task=2, userstory=5, review=2). The `check` subcommand validates completeness, independent from `/docs check`.
3. **Scaffolding**: Create new work items from templates. The `create` subcommand scaffolds the correct artifacts for a type, fills placeholders, and validates the result.

## Acceptance Criteria

### Stories #1-6: Template Registry [core + resilience]

```
GIVEN template-registry.json exists at repo root
WHEN  I read the registry
THEN  it declares a "company-workflow" set pointing to templates/company-workflow/
AND   the set references contract path and guides path
AND   the registry schema supports adding additional sets

GIVEN templates/company-workflow/ directory exists
WHEN  I list its contents
THEN  it contains exactly 13 files (5 trackers + 8 docs)
AND   each file is byte-identical to ~/Downloads/spec/templates/ source

GIVEN company templates are added
WHEN  I check existing files
THEN  templates/*.md are unchanged
AND   ./scripts/validate.sh passes
AND   ./scripts/test.sh passes
```

### Stories #7-10: Artifact Enforcement [core]

```
GIVEN a feature work item with tracker but no PRD
WHEN  I run company-workflow check on that path
THEN  output includes [MISSING] PRD.md

GIVEN a company tracker missing the workflow_type field
WHEN  I run company-workflow check
THEN  output includes [DRIFT] TRACKER.md — missing required field "workflow_type"

GIVEN one complete work item per type
WHEN  I run company-workflow check on each
THEN  feature requires 5, defect requires 3, task requires 2, userstory requires 5, review requires 2
AND   each complete work item gets [PASS] for all artifacts
```

### Stories #11-12: Scaffolding [core]

```
GIVEN the company templates exist
WHEN  I run company-workflow create --type TYPE --name NAME
THEN  a new directory is created with correct slug and ID
AND   all required artifacts for the type are created
AND   placeholders are filled (name, id, date, branch, repo)
AND   company-workflow validate exits 0 on the new tracker
```

## Out of Scope

- Automated sync from company spec upstream (one-time manual import)
- Converting existing work items (F000001, F000002) to company format
- Building a template switching CLI
- Lifecycle consistency checks (parent/child state validation)
- Cross-reference traceability (PRD stories -> TEST-SPEC coverage)

## Assumptions

- The company spec at `~/Downloads/spec/` is the authoritative version
- `skills-deploy` can handle subfolder templates, or a follow-up task will patch it
- Registry set names match skill or catalog entry names (convention, not enforced)
