## Problem Statement

Claude Code power users install 40+ skills, configure hooks, MCP servers, templates,
and rules across ~/.claude/. There is no way to answer basic health questions: "What
depends on what?", "Are any symlinks broken?", "Which skills do I actually use?", "Is
my ~/.claude/ growing out of control?"

Waza checks if your project config is correct. Gstack /health checks if your code is
healthy. Nothing checks if your ~/.claude/ filesystem is healthy or shows the dependency
topology between installed skills.

## Mental Model

system-health is a 5-step pipeline with a 4-bucket scoring model:

```
Scan -> Graph -> Filesystem -> [Waza] -> Score+Trend
                                  |
                            (optional, unscored)
```

Steps 1-3 are always scored. Step 4 (waza) is an unscored appendix. Step 4.5 (usage
trends) is also unscored. Step 5 computes the composite and tracks trends over time.

## User Stories

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | Can I see all installed skills and their metadata? | CC user | scan ~/.claude/skills/ and extract frontmatter | I know what's installed and at what version |
| 2 | core | Can I see how skills depend on each other? | CC user | see a dependency graph with hubs, orphans, and dead refs | I know what breaks if I remove a skill |
| 3 | core | Is my ~/.claude/ filesystem healthy? | CC user | check disk usage, stale sessions, temp files, history size | I catch bloat and corruption early |
| 4 | core | Can I get a single health score? | CC user | see a scored composite across 4 buckets | I can track health over time |
| 5 | observability | Can I see which skills I actually use? | CC user | see usage analytics from skill-usage.jsonl | I know which skills are valuable and which are dead weight |

| 7 | observability | Can I see usage anomalies? | CC user | detect stopped-using, long-and-failing, and discovery-gap patterns | I notice workflow problems before they compound |

## Acceptance Criteria

### Story #1: Skill scan [core]

```
GIVEN ~/.claude/skills/ contains installed skills
WHEN  /system-health is invoked
THEN  each skill directory is listed with frontmatter fields (name, version, description)
AND   symlinks are identified with target and health status
AND   cross-references between skills are extracted from *.md and *.json files
```

### Story #2: Dependency graph [core]

```
GIVEN the skill scan has completed
WHEN  graph analysis runs
THEN  an adjacency list is produced showing skill -> dependency edges
AND   in-degree is computed for each skill
AND   orphan skills (zero in-degree) are identified
AND   broken symlinks are listed
AND   dead references (to non-existent skills) are listed
AND   circular dependencies are flagged (best-effort)
```

### Story #3: Filesystem health [core]

```
GIVEN ~/.claude/ exists
WHEN  filesystem checks run
THEN  disk usage is reported per subdirectory
AND   stale sessions (>24h) are counted
AND   temp files (.tmp, .bak, .pending-*) are counted
AND   history.jsonl size and line count are reported
AND   empty directories are counted
AND   settings.json structural keys are extracted (hooks, MCP, permissions)
```

### Story #4: Scored composite [core]

```
GIVEN scan, graph, and filesystem data are collected
WHEN  scoring runs
THEN  4 buckets are scored 0-10: Structure (25%), References (35%), Integrity (25%), Hygiene (15%)
AND   composite score is computed as weighted sum
AND   snapshot is appended to ~/.gstack/health/claude-home-health-history.jsonl
AND   delta from previous run is shown (or "first run")
```

### Story #5: Usage trends [observability]

```
GIVEN ~/.gstack/analytics/skill-usage.jsonl exists
WHEN  usage analysis runs
THEN  per-skill breakdown shows runs, percentage, last used, avg/min/max duration, success rate
AND   per-repo breakdown shows runs and top skill
AND   peak hours are displayed as a bar chart
AND   installed-vs-used overlay shows never-used skills
AND   anomalies (stopped-using, long-and-failing, discovery-gap) are detected
```

### Story #6: Waza integration [integration]

```
GIVEN waza is installed at ~/.claude/skills/waza/
WHEN  /system-health is invoked (not --quick)
THEN  waza's collect-data.sh is executed
AND   output appears as an unscored appendix
AND   waza findings do NOT affect the scored composite
```

```
GIVEN waza is NOT installed
WHEN  /system-health is invoked
THEN  a message says "Waza not installed. Config hygiene checks skipped."
AND   the skill completes without error
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| All 5 steps produce output | 100% | E2E invocation |
| No crashes when waza absent | 100% | E2E without waza |
| No crashes when jq absent | 100% | E2E without jq |
| Score trend tracks over runs | Delta shown | Compare 2 sequential runs |

## Out of Scope

- Auto-fixing any issues (read-only skill)
- Per-repo code quality checks (that's gstack /health)
- Per-project config audits (that's waza /health)
- Web dashboard output (terminal text only)
- Linux compatibility testing (macOS primary, Linux best-effort)
- MCP server health checks
- Cross-machine ~/.claude/ sync

## Assumptions

- Users have ~/.claude/skills/ with at least some skills installed
- jq is available for most operations (graceful fallback if missing)
- ~/.gstack/analytics/skill-usage.jsonl uses one of 3 known schemas
- Waza install path is ~/.claude/skills/waza/ with health/scripts/collect-data.sh
