## Symptom

Two related coverage gaps from prior manifest changes (D000009, v0.14.2, and the original company-workflow PR-DESCRIPTION addition):

**A.** `skills/{personal,company}-workflow/WORKFLOW.md` type-to-artifact tables and prose understate the manifest's required artifact count in 4 places. The doc that AI-driven scaffolding reads is wrong about how many artifacts each type needs. Concrete drift:

| Workflow | Type | WORKFLOW.md says | Manifest says | Missing |
|---|---|---|---|---|
| personal | feature | TRACKER + milestones (2) | TRACKER + feature-summary + DESIGN + milestones (4) | feature-summary, DESIGN |
| company | feature | TRACKER + feature-summary + milestones (3) | TRACKER + feature-summary + DESIGN + milestones (4) | DESIGN |
| company | defect | TRACKER + RCA + test-plan (3) | TRACKER + RCA + test-plan + PR-DESCRIPTION (4) | PR-DESCRIPTION |
| company | task | TRACKER + test-plan (2) | TRACKER + test-plan + PR-DESCRIPTION (3) | PR-DESCRIPTION |

**B.** D000012's drift block (`scripts/test.sh`) iterates `for _src in $WORKBENCH/*.md` and verifies each has a byte-matching deployed copy. Deployed-extras — files in `~/.claude/templates/{workflow}/` that are *not* in the workbench — slip through. If a future PR removes a template from workbench, the old copy lingers in the deployed dir and the regression block stays green.

## Reproduction Steps

**Drift A:**
1. `jq -r '.types.feature.required | length' skills/personal-workflow/personal-artifact-manifests.json` → `4`
2. `grep -E "^\| feature \|" skills/personal-workflow/WORKFLOW.md | awk -F'|' '{print $4}' | tr -d ' '` → `2`
3. Mismatch: 4 vs 2.
4. **Observe:** WORKFLOW.md says feature requires 2 artifacts; manifest says 4. AI scaffolding from this WORKFLOW.md will produce work items missing two required artifacts that the validator (D000012's manifest reconciliation, post-D000009) will flag.

**Drift B:**
1. `touch ~/.claude/templates/personal-workflow/orphan.md` (simulates a stale template).
2. `./scripts/test.sh` → reports `OK: deployed templates/personal-workflow/ matches workbench source`.
3. **Observe:** the orphan template is undetected. D000012's loop checks `workbench → deployed` only; the reverse direction is silent.

**Environment:** macOS Darwin 25.3.0, workbench at v1.1.2 (`da3daa5`).

## Root Cause

**Root cause A (WORKFLOW.md drift):** The manifest-changing PRs (D000009 added DESIGN to feature, v0.14.2 added feature-summary, the original company-workflow scope added PR-DESCRIPTION to defect/task) updated the manifest and the templates dir but not WORKFLOW.md. WORKFLOW.md's type-to-artifact tables and prose are semantically derived from the manifest but stored as static text. No regression check ties them together; drift is silent.

**Root cause B (deployed-extra blind spot):** D000012's drift detection was scoped to the immediate failure mode — "workbench has new files that aren't in deployed." The symmetric failure mode — "deployed has files that aren't in workbench" — was never wired in. With D000013's auto-sync hook in place, the gap matters more: auto-sync only adds, so removed-from-workbench templates would accumulate undetected.

## Fix Description

Three concrete edits:

1. **WORKFLOW.md text edits** in personal-workflow + company-workflow: update the bulleted prose AND the markdown table to match the manifest's required-artifact count and content. No structural changes to either file beyond the affected rows/lines.

2. **`scripts/test.sh` D000012 block extension** — add a reverse loop after the existing forward loop:

```bash
for _dst in "$_D12_DEPLOYED"/*.md; do
  [ -f "$_dst" ] || continue
  _name=$(basename "$_dst")
  _src="$_D12_SRC/$_name"
  if [ ! -f "$_src" ]; then
    fail_test "deployed template not in workbench: $_wf/$_name (manually rm; D000014 guard)"
    _D12_DRIFT=$((_D12_DRIFT + 1))
  fi
done
```

The shared `_D12_DRIFT` counter keeps the existing OK/FAIL summary semantics: any drift in either direction → no OK line.

3. **`scripts/test.sh` new D000014 regression block** — for each workflow, parse `manifest.types.<type>.required | length` and grep WORKFLOW.md's `| <type> |` table row for the count column. Mismatch → fail. Source-of-truth one-way: manifest is authoritative; WORKFLOW.md must match. Future manifest changes will fail this check until WORKFLOW.md is updated.

The check uses count-only comparison rather than artifact-name comparison because: (a) count drift is the most important signal — it's what determines whether AI scaffolding creates the right number of files; (b) name comparison would be brittle against legitimate prose edits (e.g., "TRACKER" vs "tracker" in different rows); (c) the manifest already enforces names via D000012's regression block on the templates side.

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Existing AI-driven scaffolding | Low | Updating WORKFLOW.md to match manifest is a corrective change; AIs reading the new text will produce more-complete work items, not fewer | Validator (D000012 manifest reconciliation) catches incomplete work items either way |
| Deployed-extra reverse loop | Low | New loop runs only when `~/.claude/templates/{workflow}/` exists; same SKIP path as forward loop in CI | If the deployed dir has unexpected files (hand-edited), test fails with a clear message naming the file |
| WORKFLOW.md count check | Low | Count-only comparison is robust against prose edits; only fires on actual count mismatch | If someone legitimately changes the manifest, they MUST also update WORKFLOW.md — that's the intended behavior |
| `skills-deploy install` not pruning extras | Medium | Extras get reported but not auto-removed; user must `rm` manually | Out of scope; revisit if extras become common in practice |
