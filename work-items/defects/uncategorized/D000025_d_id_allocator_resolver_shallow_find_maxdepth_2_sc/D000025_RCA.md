## Symptom

`skills/CJ_goal_investigate/pipeline.md`'s D-ID allocator (Step 7.4) and the
exact-D-ID + BASENAME_HITS resolvers (Step 2) used `find "$DEFECTS_ROOT"
-maxdepth 2`, which only reached `work-items/defects/<domain>/D######_*`
(depth 2). Nested 2-segment domains (`ops/skills-deploy/`, `ops/ship/`,
`ops/workflow/` — depth 3) were invisible to the scan: the allocator
under-counted the highest existing D-ID and re-minted a colliding one, and
defects in nested domains were unresolvable by exact or fuzzy D-ID. Separately,
the highest-N allocator scanned only the filesystem, ignoring D-IDs recorded
solely in `git log` subjects or `TODOS.md`. Reproduction frequency: latent —
manifests whenever the highest allocated D-ID lives only in a nested-domain
directory or only in git/TODOS (always reproduces under those conditions).

## Reproduction Steps

1. From repo root: `find work-items/defects -maxdepth 2 -type d -name 'D[0-9][0-9][0-9][0-9][0-9][0-9]_*' | wc -l` → returns far fewer dirs than the unbounded scan (misses every depth-3 dir under `ops/skills-deploy/`, `ops/ship/`, `ops/workflow/`, including `ops/skills-deploy/D000022_setup_hooks_blind_clobber`).
2. Run `/CJ_goal_investigate` with a zero-match fragment so it drafts → investigates → promotes (Step 7.4 allocator runs) at a time when the highest D-ID lives only in a nested domain.
3. **Observe:** the allocator mints an already-used D-ID (the real incident minted D000022 over the existing depth-3 `ops/skills-deploy/D000022_*`; it had to be renumbered to D000024 mid-ship — PR #161 / commit dc7a46f, v4.6.12). A D-ID present only in `git log`/`TODOS.md` with no directory (e.g. deferred D000023, commit b3a67e3) is likewise silently re-mintable.

**Environment:** macOS (BSD `find`/`sed`); stock bash. claude-skills-templates workbench, branch `claude/friendly-cartwright-8d0f52`. Latent in the current tree because `uncategorized/D000024_*` happens to sit at depth 2 and mask the under-count.

## Root Cause

**Root cause:** Two distinct root causes.

1. The `-maxdepth 2` cap on all three `find "$DEFECTS_ROOT"` sites encoded a
   structural assumption — true when written, false now — that every defect
   domain is exactly one path segment under `work-items/defects/`
   (`work-items/defects/<domain>/D######_*`, depth 2). The repo organically
   grew nested 2-segment domains (`ops/ship/`, `ops/skills-deploy/`,
   `ops/workflow/`), placing those defect dirs at depth 3, outside the
   depth-2 horizon. Because the `D[0-9]{6}_` basename is globally
   unambiguous, the depth cap was never required for correctness — it was an
   invalidated structural shortcut that made the allocator under-count
   (re-minting a colliding D-ID) and made nested-domain defects unresolvable
   by exact/fuzzy D-ID.
2. The highest-N allocator scanned only the filesystem, but a D-ID is a
   durable identifier recorded independently in the directory tree, git
   commit subjects, and `TODOS.md`. A shipped-then-relocated defect, or a
   deferred/freestanding defect tracked only in git+TODOS with no directory
   (e.g. D000023), is invisible to a filesystem-only max and can be silently
   re-minted; allocation must take the max over the union of all three
   durable sources.

**Location:** `skills/CJ_goal_investigate/pipeline.md` — Step 2 exact-D-ID
`find` (~line 79), Step 2 BASENAME_HITS `find` (~line 94), Step 7.4 highest-N
allocator (~line 670, now a fs+git+TODOS union); corrected prose at
pipeline.md Step 2 docstring and `skills/CJ_goal_investigate/SKILL.md`
resolver note.

## Fix Description

Removed `-maxdepth 2` from all three `find "$DEFECTS_ROOT"` sites (exact-D-ID
resolver, BASENAME_HITS fuzzy matcher, highest-N allocator) — the scans are
now unbounded-depth, anchored on the globally-unambiguous `D[0-9]{6}_`
basename. Rewrote the Step 7.4 allocator so the next D-ID is
`max(union(filesystem D-IDs, `git -C "$_REPO_ROOT" log --all --format='%s'`
subject D-IDs, `TODOS.md` D-IDs)) + 1`, keeping it POSIX/BSD-portable (no
GNU-only flags). Every other semantic is preserved: glob-escaping of the
fuzzy fragment, the `grep -E '/D[0-9]{6}_'` post-filter on BASENAME_HITS,
dedup, case handling, the `-F`/`--` literal/option safety, and the
mkdir-lock. Corrected the now-false prose in pipeline.md Step 2 and SKILL.md's
resolver note to document multi-segment domains and the git/TODOS union. Added
a fully isolated regression test (`tests/cj-goal-investigate-did-allocator.test.sh`)
with a depth-3 nested fixture, git/TODOS-only stubs, a negative control
reproducing the old bug, and a guard preventing `-maxdepth 2` from
reappearing; wired it into `scripts/test.sh`.
