## Symptom

Merged FEATURE-level ID claims are never reaped from `cj-id-claims/` under
`$(git rev-parse --git-common-dir)`. Stale claim dirs accumulate, so the next
scaffold's same-branch-reuse / next-ID computation can hand back an already-used
F/S ID. Hit live twice on 2026-07-03: (1) an F000077 scaffold first returned
already-taken F000075/S000125; (2) two parallel Conductor worktrees both minted
F000076/S000126/v6.0.108 for different features (PR #314 vs PR #315), so #315
came back CONFLICTING at `/land-and-deploy` and needed a manual re-ID + version
bump. Frequency: deterministic whenever a merged feature claim precedes the next
scaffold (the parallel-worktree case surfaces it fastest).

## Reproduction Steps

1. In a git sandbox, push a slug-less feature tracker
   `work-items/features/ops/F000048_TRACKER.md` to `origin/main`.
2. Remove it locally, then pre-create a stale claim dir `cj-id-claims/F000048`.
3. **Observe:** run `cj-id-claim.sh --prefix F --floor 47`. BROKEN behavior:
   `id_on_origin`'s required-slug regex `(^|/)F000048_[^/]*_TRACKER\.md$` does NOT
   match `F000048_TRACKER.md`, so the stale claim is NOT reaped, it keeps counting
   toward the live max, and the mint ADVANCES to F000049 instead of reclaiming the
   reaped F000048 slot. `id_has_workitem_dir`'s `find -name F000048_*_TRACKER.md`
   glob has the symmetric miss.

**Environment:** POSIX shell / Git Bash on Windows (and macOS/Linux); the matcher
bug is platform-independent.

## Root Cause

**Root cause:** Both work-item-tracker matchers in `scripts/cj-id-claim.sh` require
a slug segment. `id_on_origin()` uses `grep -E "(^|/)${1}_[^/]*_TRACKER\.md$"` and
`id_has_workitem_dir()` uses `find -name "${1}_*_TRACKER.md"`. The mandatory
`_[^/]*_` / `_*_` between the ID and `_TRACKER.md` never matches a slug-less FEATURE
tracker named `{ID}_TRACKER.md`. Consequently the on-origin reap arm never fires for
a merged feature claim, stale feature claim dirs accumulate under `cj-id-claims/`,
they keep counting toward `max_live_claim`, and both the reuse gate and the atomic
mint can return / advance from an already-used F/S ID — producing the observed
parallel-build ID + VERSION collisions.

**Location:** `scripts/cj-id-claim.sh:177` (id_on_origin regex) and
`scripts/cj-id-claim.sh:189` (id_has_workitem_dir find glob).

## Fix Description

Made the slug **optional** in both matchers so they accept the slug-less feature
`{ID}_TRACKER.md` AND the slug-bearing `{ID}_{slug}_TRACKER.md`:

- `id_on_origin` regex → `(^|/)${1}(_[^/]*)?_TRACKER\.md$`. The `$` anchor + literal
  `_TRACKER.md` keep it from over-matching a longer sibling ID (`F000053` never
  matches `F000530`).
- `id_has_workitem_dir` → two `-name` terms:
  `\( -name "${1}_TRACKER.md" -o -name "${1}_*_TRACKER.md" \)`.

POSIX + LF preserved; shellcheck clean. Added regression Cases 8a (a merged
slug-less feature claim on origin/main IS reaped → floor 47 re-mints F000048) and
8b (once `{ID}_TRACKER.md` materializes locally, same-branch reuse advances to
F000091 instead of re-handing F000090). Updated `scripts/test.sh`'s registration
comment + case-count summary.
