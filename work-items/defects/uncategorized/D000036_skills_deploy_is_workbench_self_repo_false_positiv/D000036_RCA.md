# D000036 — Root Cause Analysis

## Symptom

On a real consumer repo with a hand-authored `spec/doc-spec-custom.md` and NO `skills-catalog.json`, `skills-deploy install` printed "Contracts — skipped (workbench self-repo; authored contracts left untouched)" and skipped the ENTIRE consumer adoption block (seeding, `complete_consumer_adoption` surface-refresh + auto-declare, AND the `cj-contract-gate` pre-commit hook install) — install did nothing.

## Reproduction Steps

In a git repo with `spec/doc-spec-custom.md` (no `auto-generated by skills-deploy` marker) and no `skills-catalog.json` at root, run `skills-deploy install` (or `seed-contracts`). It self-skips. Confirmed live on `/Users/chjiang/Documents/projects/knowledge-base`.

## Investigation Trail

- 2026-06-29: Confirmed the root cause by reading `is_workbench_self_repo()` in `scripts/skills-deploy`: signal (2) returns "yes" for any repo with a non-auto-generated `spec/doc-spec-custom.md`; signal (3) returns "yes" for any repo with `spec/test-spec-custom.md` — both too broad, since a consumer legitimately hand-authors those overlays.
- 2026-06-29: Verified signal (1) (manifest source/bundle_path match, worktree-aware via git-common-dir parent) is the correct, primary workbench identifier and is unaffected.
- 2026-06-29: Checked the existing self-repo drills in `tests/seed-contracts.test.sh` Case B and in `scripts/test-deploy.sh` (S000116a, S000117a) — all the test-deploy.sh self-repo drills use the manifest-source match (signal 1), so they are unaffected by gating signals (2)/(3); only `tests/seed-contracts.test.sh` Case B's overlay-only sub-drill needed a `skills-catalog.json` added to remain a valid workbench-self drill.

## Root Cause

**Root cause:** `is_workbench_self_repo()` OR'd two overly-broad secondary signals: presence of a non-auto-generated `spec/doc-spec-custom.md` (2) and presence of `spec/test-spec-custom.md` (3). A consumer that hand-authored either overlay (the documented way to declare repo-specific docs/units) tripped them and was misclassified as the workbench self-repo, so the data-loss guard skipped all of adoption.

**Location:** `scripts/skills-deploy` — `is_workbench_self_repo()` (the gated signal-(2)/(3) blocks).

## Affected Components

| File | Change type |
|------|-------------|
| scripts/skills-deploy | fix — gate signals (2)/(3) behind a root `skills-catalog.json` check |
| tests/seed-contracts.test.sh | regression test — Case B2 + Case B drill update |

## Fix Description

Wrapped the existing signal-(2) (`doc-spec-custom.md`, marker-excluded) and signal-(3) (`test-spec-custom.md`) blocks in a single `if [ -f "$target/skills-catalog.json" ]; then ... fi` gate, so they only fire when the unmistakable workbench marker (root `skills-catalog.json`, which a consumer never ships) is present. Signal (1) (manifest source/bundle_path match) is unchanged and remains the primary identifier. The real workbench still returns "yes" (it carries `skills-catalog.json` + authored overlays — data-loss protection preserved); a consumer with a curated overlay but no catalog now returns "no" (adoption + gate-hook install proceed). Used real `if`/`then` (no `A && B || C`) — shellcheck-clean (no SC2015).

## Regression Risk

Regression test added: `tests/seed-contracts.test.sh` — Case B2 (a consumer sandbox with a hand-authored `spec/doc-spec-custom.md` and NO `skills-catalog.json` must NOT print "workbench self-repo" and MUST seed the contracts); Case B's overlay sub-drill updated to also write a `skills-catalog.json` so it stays a valid workbench-self drill.
