## Symptom

`doc/SKILL-CATALOG.md` (the workbench's consolidated index of operator-facing surfaces) had zero mentions of the `work-copilot/` bundle or `scripts/copilot-deploy.py`, even though the bundle is a real workbench surface that the operator manages and deploys to non-Claude target repos. The operator's mental model is that the catalog should cover every named workbench surface they interact with, but the catalog was scoped narrowly to Claude skills only.

## Reproduction Steps

1. Run `grep -n 'copilot\|work-copilot\|Copilot\|copilot-deploy' doc/SKILL-CATALOG.md` on `main` at `c3a85f0` — returns zero matches.
2. Read `doc/SKILL-CATALOG.md` preamble — it explicitly says "every routable non-deprecated skill in this workbench", deliberately excluding non-skill surfaces.
3. **Observe:** the operator looking up "where is the Copilot bundle documented in the catalog?" finds nothing, even though `work-copilot/` is a first-class workbench surface (deployed via `scripts/copilot-deploy.py`, byte-mirrored from the workbench, validated by `scripts/validate.sh` Error check 10).

**Environment:** `claude-skills-templates` workbench at `c3a85f075f33bd0c28264b73bcda045244101a3e` (v5.0.19).

## Root Cause

**Root cause:** `doc/SKILL-CATALOG.md` was scoped narrowly to active routable Claude skills enforced by `scripts/validate.sh` Check 15b, which iterates `skills-catalog.json` and verifies each named skill has a `###` section with chart-or-tag. The `work-copilot/` bundle is a real workbench surface but is intentionally outside `skills-catalog.json` (it's not a Claude skill), so it was never added to the catalog. The catalog has a documented-but-undesired coverage gap between "Claude skills" and "workbench surfaces the operator manages": the operator-facing catalog mental model is broader (every named workbench surface) than the mechanically-enforced scope (Claude skills only).

**Location:** `doc/SKILL-CATALOG.md:1` (preamble scope) and `doc/SKILL-CATALOG.md:212` (missing `Companion surfaces` section).

## Fix Description

(1) Relaxed `doc/SKILL-CATALOG.md` preamble (line 3) to acknowledge the catalog now also indexes non-skill companion surfaces, while explicitly stating Check 15 enforcement only applies to the skill sections.

(2) Added a new `## Companion surfaces (non-skill)` top-level section with a brief explainer about what belongs there and why those sections are NOT Check-15-enforced.

(3) Added a `### work-copilot` subsection with Status, Source (`work-copilot/README.md` + `WORKFLOW.md` + `scripts/copilot-deploy.py`), Invoke-when guidance, and a new `(non-skill bundle)` tag line summarizing the bundle contents and the deploy CLI.

NO `scripts/validate.sh` changes — Check 15b's forward-only enforcement (catalog → file) already permits additive non-skill sections without modification.
