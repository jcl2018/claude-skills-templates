## Symptom

On Windows Git Bash with a CRLF-emitting jq build (jq 1.7.1 PE32+ at
`/c/Users/chang/bin/jq.exe`, first in PATH), `scripts/workflow-spec.sh
--list-orchestrators` false-halts
`[workflow-spec-no-config] registry-completeness (no-vanish): routable CJ_goal_*
skill 'CJ_goal_todo_fix' has NO entry in spec/workflow-spec.md` (rc=1) even
though the entry exists at `spec/workflow-spec.md:386`. The failure cascades
into `scripts/test-spec.sh --validate` / `--list-units` / `--render-docs` /
`--check-workflow-coverage` (`[test-spec-no-config]`) and turns `validate.sh`
Checks 24, 26, 27, 28 red — so the pre-commit hook blocks every commit on the
machine. Always reproducible on affected hosts; Linux CI cannot see it (Linux
jq emits LF).

## Reproduction Steps

1. On Windows Git Bash with jq 1.7.1 (Windows console binary) first in PATH,
   run `bash scripts/workflow-spec.sh --list-orchestrators`
2. **Observe:** `[workflow-spec-no-config] registry-completeness ... 'CJ_goal_todo_fix' has NO entry` with rc=1
3. CRLF proof: `jq -r '.[].name' skills-catalog.json | od -c` shows `\r\n` on every line

**Environment:** Windows 10 Home 10.0.19045, Git Bash (MINGW64), jq-1.7.1 PE32+ at `/c/Users/chang/bin/jq.exe`, repo v6.0.101.

## Investigation Trail

- Verified jq CRLF: `jq -r '.[].name' skills-catalog.json | od -c` shows `\r\n` on every line (jq-1.7.1, PE32+ Windows console binary, first in PATH)
- Reproduced the halt: `workflow-spec.sh --list-orchestrators` rc=1 naming CJ_goal_todo_fix; `test-spec.sh --validate` rc=1 with the cascaded `[test-spec-no-config]`; `workflow-spec.sh --render-docs --check` (Check 27 engine) rc=1; `test-spec.sh --render-docs --check` (Check 26 engine) rc=1; `test-spec.sh --check-workflow-coverage` (Check 28 engine) halts — full blast radius confirmed
- Grepped all three standalone spec engines for jq consumption: `workflow-spec.sh` has exactly one raw call site (line 242, the registry-completeness heredoc `$(jq -r ...)`); `test-spec.sh` and `doc-spec.sh` have ZERO jq call sites — test-spec is a pure cascade victim via its shell-out to `workflow-spec.sh --list-orchestrators` (test-spec.sh:173); `doc-spec.sh --validate` passes (unaffected)
- Explained the asymmetry: `validate.sh`/`test.sh` inherit `scripts/lib.sh`'s `jq() { command jq "$@" | tr -d '\r'; }` wrapper (lib.sh:24), but the standalone engines deliberately never source lib.sh, so they consume raw CRLF jq; engines strip CR from awk output via `_strip_cr` but never from jq output
- Confirmed the hypothesis with a controlled experiment: a PATH-prepended jq shim that pipes the real jq through `tr -d '\r'` makes the UNMODIFIED engine pass (rc=0, all 4 orchestrator names) — the jq CR is the sole cause (`read -r` splits on `\n` leaving `\r` on `_gskill`, so `_section_kind 'CJ_goal_todo_fix\r'` returns empty and `emit_halt` fires at line 238)
- Red-proof of the regression test: the pre-fix engine (`git show HEAD:scripts/workflow-spec.sh`) under the T7-style CR-APPENDING jq shim false-halts rc=1 — the drill genuinely catches this bug class

## Root Cause

**Root cause:** `scripts/workflow-spec.sh`'s registry-completeness loop feeds raw `$(jq -r ... skills-catalog.json)` into a `while IFS= read -r` heredoc; a Windows jq build emits CRLF, `read -r` splits on `\n` leaving a trailing `\r` on each catalog name, so `_section_kind 'CJ_goal_todo_fix\r'` finds no matching `## CJ_goal_todo_fix` section and the check false-halts. The engine strips CR from its awk pipelines (`_strip_cr`) but never from jq output, and unlike `validate.sh`/`test.sh` it does not source `scripts/lib.sh` whose `jq()` wrapper already solves this — the standalone-engine asymmetry.

**Location:** scripts/workflow-spec.sh:242

## Fix Description

`workflow-spec.sh`: added the scripts/lib.sh-precedent local wrapper
`jq() { command jq "$@" | tr -d '\r'; }` right after `_strip_cr` (now lines
72–77), covering every current and future jq call site while preserving the
engine's no-lib.sh standalone posture (`test-spec.sh` and `doc-spec.sh` need
no change — zero jq call sites, verified). `tests/workflow-spec-render.test.sh`:
added T7, the CRLF-jq drill — a PATH-prepended jq shim appending `\r` to every
real-jq output line (portable: simulates Windows jq on Linux CI; on Windows
yields `\r\r\n`, still fully stripped) then asserts `--list-orchestrators`
exits 0 with CR-free output and no `[workflow-spec-no-config]`, and
`--validate` exits 0, against the LIVE repo; suite header updated.
`scripts/test.sh`: updated the suite's comment block + one-line ok message to
enumerate the new drill. Extending the already-registered suite
(`spec/test-spec-custom.md` anchor `tests/workflow-spec-render.test.sh`, wired
at scripts/test.sh:2195) required no new registration.
