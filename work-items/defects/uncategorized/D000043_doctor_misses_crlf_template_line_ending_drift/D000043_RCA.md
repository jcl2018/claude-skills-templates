## Symptom

On Windows Git Bash (`core.autocrlf=true`), a deployed template
(`~/.claude/templates/*.md`) can be CRLF while the committed content is LF
(`.gitattributes eol=lf`). `skills-deploy doctor` reports the template `OK` (no
drift), yet `scripts/test.sh`'s D000012 check byte-`cmp`s source-vs-deployed and
fails with "deployed template differs from workbench" when run from an LF
checkout. This is bucket (c) of the Windows `test.sh` P0.

## Reproduction Steps

1. On Windows Git Bash, `tr -cd '\r' < ~/.claude/templates/CJ_personal-workflow/tracker-defect.md | wc -c` → 101 (CRLF); the source blob is LF (`git cat-file blob HEAD:… | tr -cd '\r' | wc -c` → 0).
2. `skills-deploy doctor` → `OK: CJ_personal-workflow/tracker-defect.md` (no drift reported).
3. `bash scripts/test.sh` (from an LF checkout) → 9× `FAIL: deployed template differs from workbench`.

**Environment:** Windows 10 Git Bash (MINGW64), `core.autocrlf=true`, repo v6.0.119.

## Investigation Trail

- Confirmed the whole Windows P0's current state by reproduction: a full `bash scripts/test.sh` passed EVERY drill-harness check (Check 17/19/24-29, S000094/S000096, F000060) with 0 failures — bucket (b) is already resolved by F000081's targeted-engine rework of the negative tests + D000040/PR #328's `_VALIDATE_OUT` SIGPIPE here-string fix. The only remaining failures are the 9 template-`cmp` fails (bucket c).
- Measured the drift directly: 9/10 deployed CJ_personal-workflow templates + `doc-SKILL-DESIGN.md` are CRLF (deployed ~76-100 bytes larger than source per template = +1 byte/line CR); the source working tree + the committed blob are LF.
- Traced why `doctor` misses it: `do_doctor`'s template health compares the deployed file's checksum against the manifest's recorded `source_checksum` (skills-deploy:1964-1966). A CRLF `skills-deploy install` copies the CRLF source AND records the CRLF `source_checksum`, so deployed-CRLF == manifest-CRLF → checksum matches → `OK`. The comparison is deployed-vs-manifest, never deployed-vs-committed-LF-intent, so a uniformly-CRLF template is invisible.
- Traced the drift's origin: `core.autocrlf=true` left the ROOT checkout's templates CRLF (an old checkout predating/ignoring `eol=lf`); `install` from that root deployed CRLF. A freshly `git reset --hard`'d worktree respects `eol=lf` and is LF — hence the source-vs-deployed mismatch a from-worktree `test.sh` sees.
- Verified the fix both directions under a fixture install: a CRLF-drifted deployed template makes `doctor` emit the new CR-probe WARN; a clean LF install does NOT (no false positive).

## Root Cause

**Root cause:** `skills-deploy doctor`'s template-drift detection compares the
deployed copy's checksum only against the manifest-recorded `source_checksum`.
When a Windows (`core.autocrlf=true`) install deploys a CRLF template, it records
the CRLF checksum too, so the comparison passes — the deployed copy is CRLF
(diverging from the committed LF content) but doctor reports `OK`. There is no
line-ending-aware probe, so CRLF drift is checksum-invisible and only surfaces as
a downstream `test.sh` D000012 byte-`cmp` failure from an LF checkout.

**Location:** scripts/skills-deploy:1962-1972 (do_doctor template health loop).

## Fix Description

Added an independent CR probe to `do_doctor`'s per-template health block: after the
existing checksum comparison, if the deployed template contains any CR byte
(`tr -cd '\r'` non-empty), emit a WARN — "deployed copy has CRLF line endings
(Windows drift; re-run 'skills-deploy install --overwrite' from an LF checkout)".
It is checked independently of the checksum (which a CRLF install can make match),
so a uniformly-CRLF template is caught. The committed content is LF
(`.gitattributes eol=lf`), so a CR byte in any deployed template is unambiguous
drift. `test-deploy.sh` Test 8c installs, drifts one deployed template to CRLF
(`sed 's/$/\r/'`), and asserts the WARN fires (and does not false-fire on a clean
LF install).
