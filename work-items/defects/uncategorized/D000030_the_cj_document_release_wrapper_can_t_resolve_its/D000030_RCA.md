## Symptom

`/CJ_document-release` HALTs with a spurious `[doc-sync-no-config]` in every repo
*except* `claude-skills-templates`, even when the consumer repo has a valid
`cj-document-release.json` at its root. Because the three `cj_goal_*`
orchestrators invoke `/CJ_document-release` inline at Step 5.5, doc-sync — and
therefore the whole orchestrated build — can only complete in the workbench
repo. A `cj_goal_*` run in a consumer repo (e.g. `cjiang327-exploration`,
`jcl2018-portfolio`) cannot fold doc updates into its PR. Reproduction
frequency: always, in any repo whose cwd is not the workbench.

## Reproduction Steps

1. Adopt `/CJ_document-release` in a consumer repo: scaffold a valid
   `cj-document-release.json` (+ `CJ-DOC-RELEASE.md`) at its root via
   `/CJ_repo-init`. The repo has NO `scripts/cj-document-release-config.sh`
   (that helper lives only in the workbench).
2. From that repo, invoke `/CJ_document-release` (or any `cj_goal_*` run that
   reaches Step 5.5 doc-sync).
3. **Observe:** Step 0.5 runs `bash scripts/cj-document-release-config.sh
   --validate`. The bare relative path does not exist in the consumer repo's
   cwd, so bash exits 127; `CONFIG_RC -ne 0` fires and the wrapper HALTs with
   `RESULT: red; HALT_MARKER=[doc-sync-no-config]` — despite the config being
   present and valid.

**Environment:** macOS (workbench-only); `/CJ_document-release` v0.1.0; helper
installed only under the workbench clone resolved by
`~/.claude/.skills-templates.json` `.source`.

## Root Cause

**Root cause:** `skills/CJ_document-release/SKILL.md` invokes the config-parser
helper via a **bare relative path** (`bash scripts/cj-document-release-config.sh`),
which resolves against the current working directory. The helper is a
workbench-global tool that ships only in `claude-skills-templates`, so the
invocation resolves only when cwd is the workbench. From any consumer repo the
call fails (rc=127) and the wrapper HALTs with a spurious `[doc-sync-no-config]`.
The wrapper never performs the `.source` reach-back that other workbench
scripts use to stay reachable from any repo.

**Location:** `skills/CJ_document-release/SKILL.md:129` (+ 234, 277, 339)

| Component | File/Module | Impact |
|-----------|------------|--------|
| Doc-sync wrapper | `skills/CJ_document-release/SKILL.md` | 4 executable helper invocations + 3 prose refs used a non-portable bare relative path |

| Skill usage doc | `skills/CJ_document-release/USAGE.md` | Did not document the (now portable) helper-resolution behavior |

## Fix Description

Resolve the helper **repo-local-first** (`$REPO_ROOT/scripts/cj-document-release-config.sh`,
for workbench self-development or a repo that vendors `scripts/`), then fall
back to the manifest **`.source`** path (`~/.claude/.skills-templates.json`),
mirroring `post-land-sync.sh` / `skills-update-check`. Because the skill's bash
fences run as separate shell invocations (shell vars do not persist), the
resolution is repeated in each of the 4 blocks: Step 0.5 carries the full
resolution plus an unreachable-`[doc-sync-no-config]` HALT; Steps 2/3/6 use a
compact 3-line re-resolution. All 4 executable call sites and the 3 prose
references now use `bash "$_CFG_HELPER"`. The fix is safe because the helper
reads its config from the cwd's git toplevel, so a `.source`-resolved helper
still parses the consumer repo's own `cj-document-release.json`. `USAGE.md`
documents the behavior (+ `last-updated` bump for Check 14); the regression
test gains 3 assertions, including a functional case that runs the real helper
from a temp repo with no `scripts/` dir and asserts it parses that repo's config.

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Workbench self-development doc-sync | Low | Repo-local-first keeps the in-repo helper as the primary resolution; behavior unchanged when cwd IS the workbench | `validate.sh` PASS; existing test rows + the new resolved-form assertion |
