## Symptom

The auto-installed **post-merge** git hook (generated from `scripts/setup-hooks.sh`'s
heredoc, "Section 2: F000011 Phase 3 lifecycle-gate auto-update") auto-edits
work-item `*_TRACKER.md` files on `main` after a pull. Two failures: (1) it resolves
the shipping PR via `gh pr list --search <id>` which matches the work-item ID in
title OR body, **mis-linking** during ID collisions (it wrote a `PR #202` link into a
tracker that shipped via #203); (2) it leaves the tracker edit **uncommitted**,
dirtying `main` on every main-moving pull — which breaks `post-land-sync.sh`'s
`git pull --ff-only`, re-arms `cj-worktree-init.sh`'s dirty-checkout guard, and
pollutes the next worktree's branch. Observed repeatedly: this exact bug dirtied
`S000077_TRACKER.md` and forced a manual rebase during the D000028 land.

## Reproduction Steps

1. On `main`, pull a merge that touches a `work-items/**/*_TRACKER.md` whose Phase-2
   `[x]`-count increased (a normal cj_goal land).
2. The post-merge hook fires Section 2 → runs `scripts/check-gates-update.sh` against
   the tracker dir → writes a `[x]` / PR-link edit into the tracked file.
3. **Observe:** `git status` is dirty on `main` with no commit; a subsequent
   `post-land-sync.sh` fails its `pull --ff-only` guard on the dirty tracked tree.

**Environment:** macOS, this workbench; the live hook at `.git/hooks/post-merge`.

## Root Cause

**Root cause:** A git post-merge hook cannot cleanly mutate a tracked file on `main` —
any edit either dirties the working tree (the observed failure) or, if committed,
creates a local-ahead commit that diverges `main` (breaking `pull --ff-only` and
polluting the next worktree branch). The F000011 Section-2 block does exactly this
uncommitted edit, and additionally mis-resolves the PR via title/body grep.

**Location:** `scripts/setup-hooks.sh:118-160` (the removed Section 2 block).

## Fix Description

Approach A (operator-chosen): **disable the auto-tick.** Removed the entire Section 2
Phase-3 block (the `BRANCH` guard + `TOUCHED_TRACKERS` loop + `check-gates-update.sh`
invocation) from `setup-hooks.sh`'s post-merge heredoc; kept Section 1 (the D000013
skills/templates re-deploy) + the `exit 0` tail; updated the file comment + install
echo to drop the Phase-3 claim. `check-gates-update.sh` is kept as a **manual operator
tool** (with a header note surfacing the `gh pr list --search` mis-link caveat) — its
PR-detection is deliberately NOT repaired here (out of scope; the hook no longer calls
it automatically). The cosmetic auto-tick of the "merged & deployed" gate is gone; the
merged PR remains the source of truth.

## Regression Risk

Regression test: `tests/setup-hooks.test.sh` — Smoke 0 inverted to assert the
Phase-3/`check-gates` auto-tick is **absent** from the generated post-merge hook;
new Smoke 1 installs into a throwaway temp git repo and asserts the hook has Section 1

test was **previously unregistered in `scripts/test.sh`** (so it never ran in CI) — now
wired in.
