## Symptom

On every consumer machine `scripts/skills-update-check` computes `remote(1.1.0) <
local(6.0.119)` and takes the no-downgrade-nudge branch, so operators are NEVER told
an upgrade is available even though `VERSION` has advanced. The feature (F000081's
`git ls-remote` version-notification) ships green — its hermetic + agentic tests stub
a *tagged* upstream — but does nothing in the field.

## Reproduction Steps

```
$ git ls-remote --tags https://github.com/jcl2018/claude-skills-templates 'v*'
… refs/tags/v1.1.0     # the ONLY v-tag on origin
$ cat VERSION
6.0.119
```
`remote_max_tag()` yields `1.1.0`; `version_gte(remote, local)` is false; the
`SKILLS_UPGRADE_AVAILABLE` banner is never emitted.

**Environment:** any consumer machine (macOS / Linux / Windows Git-Bash); no special
configuration — the inertness is structural (origin's newest v-tag is permanently
below `VERSION`).

## Root Cause

**Root cause:** The release/land flow never publishes a `v<VERSION>` git tag to origin.
The version-notification's staleness signal is derived entirely from the newest
published `v<X.Y.Z>` tag, so with origin's newest tag frozen at `v1.1.0` the
remote-vs-local comparison is permanently `remote < local` and the upgrade nudge can
never fire.

**Location:** producer gap in the land flow (`scripts/post-land-sync.sh`); consumer
read at `scripts/skills-update-check` `remote_max_tag()` (≈:147) + `cmd_default` (≈:374).

## Fix Description

Add the missing producer: at LAND, `scripts/tag-release.sh` reads `VERSION`, checks
origin via `git ls-remote --tags`, and if `v<VERSION>` is absent creates an annotated
tag and `git push origin v<VERSION>`. Idempotent (no-op if the tag already exists),
fail-soft by default (a create/push failure WARNs and exits 0 so it never halts a
land; `--strict` makes failures non-zero for the test). Wired fail-soft into
`post-land-sync.sh` after `skills-deploy install`. Post-land / advisory only — no
per-PR gate (honoring the timing constraint that VERSION is bumped in the PR before
any tag exists).

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Land flow halting | Low | `tag-release.sh` is fail-soft by default — a create/push failure WARNs + exits 0 | `tests/tag-release.test.sh` asserts #8/#9: `--strict` → rc 2, default → WARN + exit 0 (never halts a land) |
| Duplicate / force-push of an existing tag | Low | Second run could re-push or clobber the tag | Idempotent no-op verified (assert #4: second run exit 0, origin tag unchanged; assert #5: fresh clone with tag already on origin → clean no-op) |
| Bad `VERSION` value | Low | A non-semver VERSION could push a garbage tag | Guarded (assert #7): non-semver VERSION → exit 1, nothing pushed |
| Real origin never gets the current tag | Med | The tag now rides every land automatically, but origin is currently frozen at `v1.1.0` | One-time OPERATIONAL backfill (operator-gated, out of scope for the code fix): push a real `v6.0.119` tag so the nudge works today; from the next land onward `tag-release.sh` keeps it current |
| Future re-inertness undetected | Med (accepted) | No live `git ls-remote` check catches a future producer regression per-PR | Deferred recurrence-guard (F000082 follow-up (c)): a live `git ls-remote` portability smoke that would catch re-inertness nightly/on-demand (never per-PR) |
