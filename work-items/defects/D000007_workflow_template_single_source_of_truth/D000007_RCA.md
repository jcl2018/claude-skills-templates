## Symptom

Validator-vs-template **drift defects** keep recurring across both workflow skills. Each one is a real bug — fresh-scaffolded trackers fail the skill's own validator, or hand-edited trackers silently pass when they shouldn't — but every fix is local: patch the contract, patch the template, ship. The underlying drift mechanism stays in place, so the next template change reopens the same bug class.

Recurrence to date:

- **D000004** (BLOCKED): `workflow_type` / `url` / section-order drift between `skills/company-workflow/{SKILL.md,contract.json}` and `templates/company-workflow/tracker-*.md`.
- **D000006** partial (deferred): Phase 2 test-verification gates added to all 4 company-workflow trackers; `contract.json` not updated to require them. Author-asserted only.
- **Audit 2026-04-17** (4 new findings): missing `type_specific_optional` in company contract, structural outliers in 3 templates.

Frequency: **always**, every time a template or contract is edited without the matching update on the other side.

## Reproduction Steps

1. Edit `templates/company-workflow/tracker-defect.md` to add a new `## Some Section` between `## Lifecycle` and `## Todos`
2. Do not update `skills/company-workflow/contract.json`
3. Scaffold a fresh defect tracker from the (now-updated) template
4. Run `company-workflow validate` against the freshly scaffolded directory
5. **Observe:** the validator reports a section-order violation: "Some Section" appears in unexpected position. The contract's `expected_order` was never updated, so the new section breaks ordering.

Symmetric reverse case:

1. Edit `skills/company-workflow/contract.json` to add `Some New Section` to `sections.required`
2. Do not update `templates/company-workflow/tracker-defect.md`
3. Scaffold any fresh defect tracker
4. Run validator
5. **Observe:** validator reports `VIOLATION: missing required section "Some New Section"` against every newly scaffolded tracker

**Environment:** macOS Darwin 25.3.0; both workflow skills as deployed in this repo on branch `claude/nostalgic-volhard`. The bug is architectural, not platform-specific.

## Root Cause

**Root cause:** Both workflow skills maintain **two parallel sources of truth** for structural validation rules — the templates themselves (which authors edit) and `contract.json` (which the validator reads). Every edit to one requires a matching edit to the other, but the skill provides no automated way to detect when they diverge. Authors fix the local symptom (add the gate, ship), the drift gets ratcheted in, and the next edit replays the bug.

**Location:**
- `skills/company-workflow/contract.json` — entire file is the second source
- `skills/company-workflow/SKILL.md` lines that read contract.json (the validator's `## Step 4-6` and the company-specific `workflow_type`/`url` checks at lines 126/145)
- `skills/personal-workflow/contract.json` — same shape, same problem
- `skills/personal-workflow/check.md` lines that read contract.json (Steps 2, 4, 5, 6 of Tier 1; partially Step 15-16 of Tier 2)

The recurring defects are not the bug. The bug is the architecture that lets them recur.

## Fix Description

**Approach:** Eliminate `contract.json` from both skills. Validator derives every structural rule from the template at runtime.

### Rule derivation contract (the new spec, in prose)

For a tracker file under validation:

1. Read its `type` from frontmatter.
2. Resolve the matching template at `templates/{skill}/tracker-{type}.md` via the existing 2-level fallback chain (`$REPO_ROOT/templates/...` → `~/.claude/templates/...`).
3. Parse the template's:
   - **Frontmatter keys** → these are the required fields (instance must contain all of them; extras are allowed).
   - **`##` section headers in document order** → these are the required sections, AND the expected order. (Sections present in some per-type templates but not others become per-type-structural automatically: a feature template has `## Acceptance Criteria`, a defect template doesn't, so a feature instance requires it and a defect instance doesn't even consider it.)
   - **`### Phase N:` headers** → these are the required lifecycle phases (Track/Implement/Ship for personal, Track/Implement/Review/Ship for company — each skill's templates declare their own phase model).
   - **Count of `- [ ]` checkboxes inside the Lifecycle section** → minimum checkbox count.
4. Compare the instance against the derived rules. Same violation messages as today.

For doc artifacts (PRD, ARCHITECTURE, RCA, test-plan, etc.): same flow but parses the corresponding `templates/{skill}/doc-{name}.md` template.

### Net behavior change

- Validator's *output* (PASS / VIOLATION messages) stays virtually identical for any tracker that's compliant today and stays compliant tomorrow.
- Edge cases that change:
  - "Recommended but not required" frontmatter (today's `repo`, `branch`): no longer expressible. Either promote to required or drop the distinction. Since today's behavior is "advisory only, never enforced," dropping the distinction is the zero-impact choice.
  - "This section is optional everywhere except for type X" (today's `type_specific_optional`): no longer needs declaration. The per-type template either includes the section or omits it; the validator infers from the template.

**Why not alternative fixes:**

- **Options A, E (round-trip validators)** — preserve the two-source-of-truth pattern; add tooling to keep them aligned. More moving parts to maintain. Drift still possible during the window between template edit and validator-tooling re-run.
- **Option B (pre-rendered fixtures + git-diff)** — catches template drift but not contract.json drift directly; fixtures themselves become a third source of truth that can drift.
- **Option C (drop the runner; CI lint only)** — smaller surface, less safety net; still preserves two sources.
- **Option D (self-test inside the skill)** — loses CI gate; relies on the skill being invoked to catch drift.
- **Option F (THIS DEFECT — eliminate contract.json)** — single source. Drift mechanism removed entirely. The validator's runtime cost goes up by one template-read (negligible) and the maintenance cost goes down to zero.

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Existing work items under `work-items/` | Very Low | Templates haven't changed; validator's derived rules match today's contract for every section/phase that's currently in sync | Run `/personal-workflow check` against entire `work-items/` tree post-rewrite; confirm 0 new violations |
| Existing trackers in consumer repos (e.g., ai-content) | Low | Same reasoning; their trackers were authored against templates, not against contract.json directly | Smoke-test by deploying the new skills via `skills-deploy install --overwrite` and running validate against a known-good tracker |
| Author-asserted gates (D000006's Phase 2 test-verification text) | Positive | The new validator counts `- [ ]` boxes in the template, so missing checkboxes in an instance now actually trip a violation. Today they're text-only. | Acceptable — strictly stronger enforcement |
| `frontmatter.recommended` advisory removal | Very Low | Today nothing enforces "recommended"; removing the concept changes nothing observable | None needed |
| `type_specific_optional` removal | Very Low | The per-type template is the new source; behavior matches today's intent for every type-section pair currently in `contract.json` | Cross-check the new derived rules against today's contract for both skills as part of test-plan verification |
| Fixtures testing contract-specific behavior | Med | A fixture like `invalid-bad-frontmatter.md` may rely on contract.json declaring `recommended` fields; needs to be rewritten against the new model | Audit each fixture; rewrite or retire |
| Downstream tooling reading contract.json | Low | Grep `scripts/`, `~/.claude/`, and known consumer repos for `contract.json`; nothing in this repo's `scripts/` references it; consumer repos should not either | Grep audit before merging |
| Catalog validation | Low | `scripts/validate.sh` walks `skills-catalog.json` files arrays; removing contract.json from both arrays is a coordinated change | Update `skills-catalog.json` in the same commit as the contract.json deletions |
| New skill author confusion | Med | Future skills might still scaffold with a contract.json by habit (or AI guess) | Update CLAUDE.md "Creating a new skill" section to clarify: templates are the spec; no contract.json |
| Performance | Negligible | Validator reads one extra file per artifact (the template). On a typical work-items tree of ~10 artifacts, that's ~10 extra small file reads per `check` run | None needed |
