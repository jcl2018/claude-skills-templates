## Symptom

Every `/ship` or `/land-and-deploy` run in this repo silently fails to enable auto-merge on the PR, then falls back to a direct squash-merge with a noisy stderr from a broken local-cleanup step. Two recurrences in this session (PR #32, PR #35), same exact pattern. The fix lands eventually (via fall-back) so PRs do merge, but every release burns ~30 seconds on the wrong-then-right command sequence and produces a confusing log trail.

Symptom A (primary):

1. LLM runs `gh pr merge --auto --delete-branch` per /ship's Step 4.
2. gh CLI prints its help text to stdout and exits 0.
3. LLM checks `gh pr view --json state` → still `OPEN`. No merge happened.
4. LLM falls back to `gh pr merge --squash --delete-branch` (which works).

Symptom B (follow-on, worktree-specific):

1. The fall-back `--delete-branch` triggers a local `git checkout {base-branch}` before deleting the local tracking ref.
2. In a git worktree where `main` is checked out in the parent repo, this errors: `'main' is already checked out at '/Users/.../parent-repo-path'`.
3. The remote merge succeeds via the GitHub API; only the local-cleanup step fails. Exit code is 0, but the remote branch is NOT deleted.
4. LLM works around with `gh api -X DELETE repos/{owner}/{repo}/git/refs/heads/{branch}` to delete the remote branch.

Frequency: **always** in this repo. Confirmed on PR #32 (D000006 ship) and PR #35 (D000007 ship), same session.

## Reproduction Steps

### Symptom A

1. From a feature branch in this repo with an open PR:
   ```bash
   gh pr merge --auto --delete-branch
   ```
2. **Observe:** stdout begins with the gh help text (`-m, --merge ... -r, --rebase ... -s, --squash ...`). Exit code is 0.
3. `gh pr view --json state -q .state` → `OPEN`. No auto-merge enabled.
4. With explicit method:
   ```bash
   gh pr merge --auto --squash --delete-branch
   ```
   → succeeds (auto-merge enabled, immediately merges if checks already pass).

### Symptom B

1. From a worktree (e.g., `.claude/worktrees/{name}`) where the parent repo has `main` checked out:
   ```bash
   gh pr merge {N} --squash --delete-branch
   ```
2. **Observe:** Remote merge succeeds (PR state → `MERGED`), but stderr contains:
   ```
   failed to run git: fatal: 'main' is already checked out at '/Users/.../parent-repo-path'
   ```
3. `git ls-remote origin {branch}` → still returns the branch SHA (not deleted on remote).
4. Local branch tracking ref still points to the old SHA (not pruned).

**Environment:** macOS Darwin 25.3.0; gh CLI from Homebrew (any recent version); GitHub repo without an org-level default merge method; the worktree-specific symptom requires running /ship from inside a `git worktree` whose parent has `main` checked out.

## Root Cause

**Root cause (Symptom A):** gh CLI's `--auto` flag enables auto-merge but does NOT imply a merge method. When invoked without `--merge`/`--squash`/`--rebase` AND the repo has no org-level default merge method, gh prints help and exits 0 (silent failure). The /ship and /land-and-deploy skill workflows in gstack instruct the wrong invocation: they pass `--auto --delete-branch` without a method.

**Root cause (Symptom B):** `gh pr merge --delete-branch` performs the post-merge local cleanup by checking out the base branch and deleting the now-merged local feature ref. In a git worktree where the parent repo already has `main` checked out, the local `git checkout main` fails. gh swallows the error and exits 0 (the remote merge already succeeded), but the side effects (local + remote branch cleanup) silently don't happen.

**Locations:**
- `~/.claude/skills/gstack/ship/SKILL.md` Step 4 — line containing `gh pr merge --auto --delete-branch` (the wrong-method-less invocation)
- `~/.claude/skills/gstack/land-and-deploy/SKILL.md` Step 4 — same line
- `CLAUDE.md` (this repo) — does NOT yet have a CI/CD merge-convention section that would guard against this. This is the gap to fill locally.

The bug class is **upstream skill-workflow doc bug + downstream missing local guard.** Fix is two-layer: upstream PR to gstack, plus a local CLAUDE.md note as defense-in-depth.

## Fix Description

**Two-layer fix.**

### Local (this repo, ships in this defect)

Add a `## CI/CD merge convention` section to `CLAUDE.md` documenting:

1. This repo uses **squash merges**.
2. When running `/ship` or `/land-and-deploy` here, ALWAYS use `gh pr merge --auto --squash --delete-branch` (combine `--auto` AND `--squash`). The `--auto` flag alone is not sufficient — gh requires an explicit method.
3. When running inside a git worktree where the parent has `main` checked out, the `--delete-branch` flag's local cleanup will fail (cosmetic). Delete the remote branch with `gh api -X DELETE repos/jcl2018/claude-skills-templates/git/refs/heads/{branch}` after the merge.

Add a regression test in `scripts/test.sh` ("Regression test (D000008)") that asserts `CLAUDE.md` contains the merge-convention section so future CLAUDE.md edits don't drop the guard.

### Upstream (gstack, file separately)

File a gstack issue or PR against `~/.claude/skills/gstack/{ship,land-and-deploy}/SKILL.md` Step 4. Recommended change:

```diff
- gh pr merge --auto --delete-branch
+ gh pr merge --auto --squash --delete-branch
```

(Or `--merge` / `--rebase` if the project prefers a different default — but the repo's preference should be detected at preamble time, not assumed.) Add a comment line explaining gh CLI requires the method when `--auto` is used.

For the worktree-specific `--delete-branch` failure, suggest adding to the SKILL.md a one-line note: "If running inside a git worktree, prefer `gh api -X DELETE refs/heads/{branch}` for remote-branch cleanup since `--delete-branch` does a local `git checkout` that fails in worktrees."

**Why not alternative fixes:**
- **Configure org-level default merge method** — would fix Symptom A globally for any consumer of the org. Out of scope for this defect; also doesn't help repos in other orgs.
- **Wrap gh in a local helper script** — heavier; CLAUDE.md note is the lightest-touch fix that gets the LLM to the right invocation.
- **Drop `--auto` entirely; always use direct merge** — loses the merge-queue benefit for repos that have one. The intent of `--auto` (queue for after CI passes) is correct.
