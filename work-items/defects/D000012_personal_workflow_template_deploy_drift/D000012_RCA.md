## Symptom

The deployed templates directory at `~/.claude/templates/personal-workflow/` is missing two template files that the workbench manifest now requires for the `feature` type: `doc-DESIGN.md` and `doc-feature-summary.md`. Additionally, `tracker-feature.md` in the deployed copy is older than the workbench source (Apr 16 vs Apr 25; 2728 bytes vs 2855 bytes).

Consequence: any repo on this machine that uses personal-workflow but is not the workbench itself (e.g., `/Users/chjiang/Documents/projects/portfolio`) sees the new manifest requirement but cannot resolve the templates. The validator's 2-level resolution falls to the deployed copy and either skips frontmatter validation (warning only) or reports `[MISSING]` for the artifact files themselves.

## Reproduction Steps

1. `ls -la ~/.claude/templates/personal-workflow/` — confirm `doc-DESIGN.md` and `doc-feature-summary.md` are absent
2. `ls -la /Users/chjiang/Documents/projects/claude-skills-templates/templates/personal-workflow/` — confirm both files exist in workbench source
3. `jq '.types.feature.required' ~/.claude/skills/personal-workflow/personal-artifact-manifests.json` — returns 4 entries including `feature-summary` and `design`
4. From `/Users/chjiang/Documents/projects/portfolio`, run `/personal-workflow check work-items/features/discord-v1`
5. **Observe:** `[MISSING] feature-summary — required artifact not found` and `[MISSING] design — required artifact not found`. Note that `cold-start` (which has `feature-summary.md` present) cannot have its frontmatter validated because `doc-feature-summary.md` cannot be resolved at Level 2.

**Environment:** macOS Darwin 25.3.0, workbench at commit d00ff78 (v1.1.0). Deployed templates last touched 2026-04-16 per filesystem mtime. `~/.claude/skills/personal-workflow/` is symlinked to the workbench (so manifest is fresh), but `~/.claude/templates/personal-workflow/` contains real files that were copied during the original `skills-deploy install` and have not been refreshed.

## Root Cause

**Root cause:** D000009 (v0.13.1) and v0.14.2 each shipped manifest + template additions to the workbench, but neither change triggered a re-deploy of `~/.claude/templates/`. `skills-deploy install` is a manual one-off; nothing on the workbench side enforces a re-deploy when templates change. Skill assets under `~/.claude/skills/` are symlinked into the workbench so they stay current automatically, but the `templates/` directory was copied as files at original install time and has no refresh trigger. The drift was not limited to D000009/v0.14.2's new files — the regression block (Phase 2) surfaced 7 deltas total across both workflows, including stale `tracker-feature.md`, `tracker-user-story.md`, `doc-milestones.md` files that had been edited in workbench post-deploy. D000009/v0.14.2 are the visible-by-manifest cases; the systemic mechanism affects every template edit.

**Location:** the gap is between `$REPO_ROOT/templates/{workflow}/` (workbench source, byte-current) and `~/.claude/templates/{workflow}/` (deployed copy, frozen at last `skills-deploy install`). No mechanism enforces sync. Both `personal-workflow` and `company-workflow` are affected — the drift is not personal-workflow-specific.

## Fix Description

Three options, narrow to wide:

**Option A (one-off):** Re-run `scripts/skills-deploy.py install --overwrite` from the workbench. Refreshes `~/.claude/templates/personal-workflow/` with the missing templates and the updated `tracker-feature.md`. Does not prevent recurrence next time templates change.

**Option B (regression check):** Add a `doctor` / `test.sh` check that diffs workbench `templates/personal-workflow/*.md` against `~/.claude/templates/personal-workflow/*.md` and flags drift (missing files, mtime/size delta). Fast, catches the next occurrence, but still requires manual remediation.

**Option C (auto-refresh — pick a sub-path):** The current asymmetry deserves an explicit design call. `~/.claude/skills/personal-workflow/` is symlinked into the workbench (so manifest, SKILL.md, check.md stay current automatically). `~/.claude/templates/personal-workflow/` was copied as real files at install time and has no refresh trigger — which is exactly what caused this defect. Two sub-paths:

  - **C1. Symlink the templates dir too.** One-line change to `scripts/skills-deploy.py`. Makes templates auto-current the same way skills are. Trade-off: requires the workbench checkout to exist on every machine using personal-workflow. Today portfolio has no workbench checkout — it relies on `~/.claude/templates/` having real copies. On a machine where the user uninstalled the workbench, validation would break instead of falling back to a frozen-in-time copy.
  - **C2. Post-merge hook on the workbench `main`.** A `git post-merge` hook in the workbench calls `skills-deploy install --overwrite`. Templates stay copies (so they survive workbench uninstall) but get refreshed on every `git pull` of the workbench. Trade-off: runs on every pull, which is more side effect than most users expect from `git pull`.

  Either sub-path eliminates drift; pick based on how you treat the workbench checkout. C1 matches the skills behavior exactly; C2 preserves the standalone-templates contract.

**Behavior change to expect from Option A.** Restoring the templates immediately surfaces `[MISSING] feature-summary` and `[MISSING] design` on portfolio work items (notably `discord-v1`) that the validator was silently skipping due to template-not-found. This is intentional — the validator *should* flag missing artifacts, not silently skip — but the change in validator output is a hard fact to plan around. Portfolio backfill becomes the next obvious thread to pull after this defect ships.

Recommended: **A + B in this defect** (immediate fix + regression gate). **C deserves a design call** — file a follow-up to pick C1 vs C2 based on whether `~/.claude/templates/` should track the workbench live (C1) or stay independent (C2).

**What shipped in this defect.** A + B, both generalized to cover personal-workflow AND company-workflow (the original scope was personal-workflow only; the regression block surfaced that company-workflow had the same drift, including its own missing `doc-DESIGN.md`). Concretely:

- **A.** `scripts/skills-deploy install --overwrite` run from the workbench. Output: `Templates — Installed: 7` (3 newly installed: `personal-workflow/doc-DESIGN.md`, `personal-workflow/doc-feature-summary.md`, `company-workflow/doc-DESIGN.md`; 4 overwritten on checksum mismatch: `personal-workflow/tracker-feature.md`, `personal-workflow/tracker-user-story.md`, `company-workflow/tracker-feature.md`, `company-workflow/doc-milestones.md`).
- **B.** New regression block in `scripts/test.sh` after the existing D000009 block. Verifies (a) `skills-catalog.json` declares `doc-DESIGN.md` and `doc-feature-summary.md` for both workflows (4 PASS lines), and (b) when `~/.claude/templates/{workflow}/` exists, every workbench template is byte-identical in the deployed copy (uses `cmp -s`). Skips gracefully with an INFO line on hosts where `skills-deploy` hasn't run, so CI doesn't false-fail.
