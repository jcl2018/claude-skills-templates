## Symptom

`/CJ_suggest` exits 1 with `(eval):33: read-only variable: status` and prints
no top-5 ranked table when invoked on a machine whose Claude Code Bash tool
dispatches commands through zsh. Reproduction is **deterministic** — every
invocation hits the same line and fails the same way.

## Reproduction Steps

<!-- Minimal steps to reproduce the issue. -->

1. On a machine with `Shell: zsh` in the Claude Code env block.
2. Invoke `/CJ_suggest` from any branch in the workbench repo so the harness
   pastes the SKILL.md bash heredoc body into the Bash tool.
3. The block executes through zsh; at line 33 it runs
   `status=$(echo "$row" | awk -F'\t' '{print $2}')`.
4. **Observe:** zsh aborts: `(eval):33: read-only variable: status`. Skill
   exits 1. No table.

**Environment:** macOS (Darwin 25.3.0), Claude Code with `Shell: zsh`. Workbench
repo at `claude-skills-templates` HEAD `83028d7` (or later) where
`skills/CJ_suggest/SKILL.md` contains the inline bash body.

## Root Cause

<!-- One clear statement of WHY the bug happens. Not what the fix is, but what
     is wrong in the existing code/logic/data. -->

**Root cause:** The `/CJ_suggest` skill ships a bash-shaped script as a heredoc
inside `SKILL.md` and instructs the agent to "Run the bash block below." The
agent dispatches that block through the Claude Code Bash tool, which on this
machine is **zsh**. zsh treats `status` as a read-only special parameter (alias
of `$?`); the skill's tracker-join code at line 33 assigns to `status=`, which
zsh refuses with `(eval):33: read-only variable: status`. The script never gets
to print its table. Nothing is wrong with the script's logic — the harness
shell is wrong for an unannotated bash-shaped block.

**Location:** `skills/CJ_suggest/SKILL.md` heredoc bash body, line 33
(`status=$(echo "$row" | awk -F'\t' '{print $2}')`).

## Fix Description

<!-- What was changed to fix the root cause. Not a diff, but a human-readable
     explanation of the approach. -->

Approach B from the design doc — **extract the bash body to a sibling file**:

1. Create `skills/CJ_suggest/scripts/suggest.sh` containing the existing bash
   body verbatim, prefixed with `#!/usr/bin/env bash` and `set -euo pipefail`
   (slight hardening over today's `set -u`).
2. Replace the `skills/CJ_suggest/SKILL.md` routing's heredoc body with a
   one-liner: `bash "$(git rev-parse --show-toplevel)/skills/CJ_suggest/scripts/suggest.sh"`.
   Trim the now-obsolete "Single-file by design" caveat in Notes.
3. Add `skills/CJ_suggest/scripts/suggest.sh` to the `CJ_suggest` entry's
   `files` array in `skills-catalog.json` so `skills-deploy` and
   `scripts/validate.sh` track it.

The shebang pins execution to bash regardless of which shell the harness
dispatches through. Approach A (inline rename `status` → `wi_status`) only
patches the one variable that crashed today; future zsh reserved words in the
skill silently re-crash. Approach C (wrap inline body in `bash -c <<'BASH'`)
keeps single-file but introduces double-escape fragility on routine edits.
