## Symptom

Running the documented update path (`scripts/setup.sh`) **silently and
irrecoverably destroys** any operator- or tooling-customized
`.git/hooks/pre-commit` or `.git/hooks/post-merge` (Husky, lefthook, a local
debug hook, etc.). `setup-hooks.sh` does `cat > "$HOOK_DIR/pre-commit"` and
`cat > "$HOOK_DIR/post-merge"` — an unconditional overwrite with no existence
check, no ownership check, and no backup. Since D000021 / PR #150 wired
`setup-hooks.sh` into `setup.sh`, and `setup.sh` takes its update branch on
**every** re-invocation (it is the documented repeated "sync drift" path, not
first-run only), the destruction happens on the next `setup.sh` for every such
user. Always reproduces (deterministic). Disclosed and consciously deferred in
PR #150's v4.6.5 CHANGELOG to hold that PR's minimal scope; this is the tracked
discharge of that carry-forward.

## Reproduction Steps

1. `printf '#!/bin/sh\necho custom\n' > .git/hooks/pre-commit && chmod +x .git/hooks/pre-commit`
   (or install Husky/lefthook, which own `.git/hooks/pre-commit`).
2. `./scripts/setup.sh`   (the documented update path; calls `setup-hooks.sh`).
3. **Observe:** `.git/hooks/pre-commit` is now the workbench hook; the custom
   one is gone and there is **no `.git/hooks/pre-commit.bak`**. Unrecoverable.
   `post-merge` behaves identically.

**Environment:** workbench `main` ≥ v4.6.5 (D000021 landed). `setup.sh` runs
under `set -euo pipefail`; the `setup-hooks.sh` call is guarded by
`|| echo WARN >&2` (setup.sh:37). macOS workbench + Linux CI.

## Root Cause

**Root cause:** `scripts/setup-hooks.sh` installs both git hooks with an
unconditional `cat > "$HOOK_DIR/<hook>"` — no check for an existing file, no
check of whether that file is workbench-owned (despite the workbench's own hook
bodies already carrying a sentinel for exactly this purpose), and no backup.
Combined with D000021 wiring `setup-hooks.sh` into the always-on `setup.sh`
update path, the documented "keep my workbench current" action destroys
operator/tooling-owned hooks with no recovery path. Secondarily, the write is
non-atomic: `cat >` truncates the target immediately, so an interrupted write
(or a `chmod` failure) leaves a corrupt/half-installed hook **present**, which
`setup.sh`'s `|| echo WARN >&2` guard then masks.

**Location:** `scripts/setup-hooks.sh:23` and `:36` (the two unguarded
`cat > "$HOOK_DIR/<hook>"` writes). The sentinel that makes ownership-detection
trivial already exists in the hook bodies (`:25`, `:38`) — this is purely a
missing guard + non-atomic write, not a defect in the hook logic.

## Affected Components

| Component | File/Module | Impact |
|-----------|------------|--------|
| Hook installer | `scripts/setup-hooks.sh` | Unconditional no-backup clobber; non-atomic write |
| Update path | `scripts/setup.sh` | Runs the clobber on every re-invocation; its `\|\| echo WARN >&2` guard masks partial-write failure |
| Operator/tooling hooks | `.git/hooks/{pre-commit,post-merge}` | Custom hooks (Husky, lefthook, local) silently destroyed, unrecoverable |
| Regression coverage | `scripts/test.sh` | D000013 guard anchored on the `cat >` token the fix removes — must be re-anchored or it false-fails; no assertion that clobber-safety exists |

## Fix Description

Replace the two unguarded `cat > "$HOOK_DIR/<hook>"` blocks in
`scripts/setup-hooks.sh` with a single `install_hook <name>` helper (hook
bodies passed unchanged via the existing heredocs, so they stay byte-identical
and the sentinel is preserved). For each hook the helper:

1. **Writes to a temp file first.** `mktemp "$HOOK_DIR/.<name>.XXXXXX"` — same
   directory as the target ⇒ same filesystem ⇒ the later `mv` is an atomic
   `rename(2)`; the dotfile name means git never executes the temp even if it
   is left behind on failure. `chmod +x` the temp **before** any `mv`. If
   `mktemp`/`cat`/`chmod` fails, remove the temp and return non-zero **without
   touching the real hook**. This closes the partial-write finding (a failure
   now leaves the prior hook intact, or nothing — never a truncated/non-exec
   file) and incidentally discharges D000021's parked low-severity
   "non-executable hook if chmod fails" note.
2. **Sentinel-guards the overwrite.** If the target exists **and** does not
   contain `# Auto-installed by scripts/setup-hooks.sh` (`grep -qF`), it is
   operator/tooling-owned: `cp -p` it to `<hook>.bak` (or
   `<hook>.bak.<YYYYMMDDHHMMSS>` if `.bak` already exists, so a second run
   against a since-changed custom hook does not overwrite the first backup),
   then emit a `WARN:` to stderr naming the backup. **If the backup `cp`
   fails, abort without clobbering** (return non-zero) — the one outcome that
   must never happen is destroying an unbacked custom hook. If the target is
   absent, or already carries the sentinel (our own prior install), no backup
   is written and the install is a quiet refresh — repeated `setup.sh` runs
   stay a NO-OP with zero `.bak` litter.
3. **Atomically installs.** `mv` temp → target. Print the existing
   success `echo`s only on the success path. Propagate a non-zero exit from
   `setup-hooks.sh` (`exit $rc`) so `setup.sh`'s `|| echo WARN >&2` guard
   surfaces a failure instead of the deploy proceeding silently.

Chosen approach (a) backup-then-install over (b) skip-and-warn: explicit task
guidance ("(a) is probably friendlier") and it matches this repo's documented
"atomic mv; backup rotation" idiom.

`scripts/test.sh`: re-anchor the D000013 post-merge guard from
`grep -q 'cat > "$HOOK_DIR/post-merge"'` to `grep -qE 'install_hook[[:space:]]+post-merge'`
(identical regression intent — "setup-hooks.sh still writes a post-merge
hook" — new code shape; required so the correct refactor does not turn
`test.sh` RED), update the D000021-guard comment example + SC2016 rationale
that cite the removed token, and add two D000022 source-level assertions in the
same D000013 block: (a) the sentinel string is grep'd before clobber; (b) the
write goes through `mktemp` + `mv` with a `.bak` backup. Static greps only — no
`git init`, no hook execution (same CI-safety rationale the D000013 block
already documents).

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Own hooks recognized as foreign (spurious `.bak` every run) | Low | If the sentinel substring did not match our own bodies, every `setup.sh` would back up our own hook | Sentinel `# Auto-installed by scripts/setup-hooks.sh` is grep'd `-F` as a substring; verified to match BOTH bodies (pre-commit exact; post-merge has only a trailing `.`). Idempotent-rerun regression case in the test-plan. |
| `mktemp` on a different filesystem ⇒ non-atomic `mv` | Low | A cross-FS `mv` is copy+unlink, not atomic | Temp is created **inside `$HOOK_DIR`**, guaranteeing same filesystem as the target ⇒ `mv` is `rename(2)` |
| D000013 regression guard false-fails | Low (mitigated) | The fix removes the `cat > "$HOOK_DIR/post-merge"` token that guard greps | Guard re-anchored to `install_hook post-merge`, same intent; D000013's body-content guards untouched (hook bodies byte-identical) |
| Backup accumulation | Low (accepted) | Timestamped `.bak`s could pile up across many runs against an ever-changing custom hook | Backups only written when a NON-sentinel hook is found; our own re-runs write none ⇒ zero steady-state litter. GC policy explicitly out of scope; revisit only if real accumulation observed |
| `setup.sh` deploy path | Low | Helper returns non-zero on failure; `setup.sh`'s `\|\| echo WARN >&2` still makes hook install best-effort and the final `exec skills-deploy install` still runs | Unchanged from D000021; new `exit $rc` only makes existing failures *visible*, it does not abort the deploy |
| macOS vs Linux | Low | `mktemp`/`mv`/`cp -p`/`grep -F` differ subtly across BSD/GNU | All four used with portable POSIX-safe invocations (`mktemp DIR/.nameXXXXXX`, `cp -p`, `grep -qF`); workbench is macOS, CI is Linux — both exercised |
