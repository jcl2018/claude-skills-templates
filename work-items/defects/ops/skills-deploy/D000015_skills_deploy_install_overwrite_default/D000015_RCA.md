## Symptom

`scripts/skills-deploy install` does not update deployed templates or rules when their content differs from the workbench source. By default it logs `WARN: {file} — exists with different content (use --overwrite to replace)` and exits leaving the deployed copy stale. The user has to remember to pass `--overwrite` for the deploy to actually deploy. Reproduction frequency: every time, on any post-edit-and-commit cycle.

## Reproduction Steps

1. Ensure `~/.claude/templates/personal-workflow/` already exists from a prior `skills-deploy install`.
2. From the workbench, modify a template (e.g. `echo '<!-- test -->' >> templates/personal-workflow/tracker-defect.md`).
3. Run `scripts/skills-deploy install`.
4. **Observe:** stderr/stdout shows `WARN: tracker-defect.md — exists with different content (use --overwrite to replace)`. The deployed file is unchanged.

**Environment:** workbench at v1.5.1 (`main`). `~/.claude/` deployed by a prior `skills-deploy install` on the same machine. macOS 25.3.0 (Darwin), zsh.

## Root Cause

**Root cause:** `skills-deploy install` defaults to safe-mode (`overwrite=false`) at [scripts/skills-deploy:149](scripts/skills-deploy:149), but the tool's purpose is "deploy = mirror workbench source to `~/.claude/`." The defensive default protects the rare case (intentional local edits to deployed files) at the cost of the common case (every routine deploy after a workbench change). The result is the user types `skills-deploy install`, sees `WARN: ...`, retypes with `--overwrite`, and over time the flag becomes muscle memory that should never have been required.

**Location:** [scripts/skills-deploy:149](scripts/skills-deploy:149) (default value), with downstream effects at lines 153 (flag parsing), 379, 386, 416, 421 (install branches), 664 (doctor hint), 881 (help text).

## Fix Description

Two design decisions, settled during implementation:

1. **Escape-hatch shape.** Chose **A** (add `--no-overwrite`) over **B** (drop the flag). Reasoning: D000013's post-merge hook in `scripts/setup-hooks.sh:43` passes `--overwrite` unconditionally. Dropping the flag would break the hook on every machine mid-pull. Keeping `--overwrite` as a tolerated no-op preserves backwards compat with zero coupling between D000015 and D000013's tests; the cost is one extra line in the case statement.

2. **Logging.** Chose **A** (keep per-file logging), but reworded the messages:
   - `OVERWRITE: {file} (checksum differs, --overwrite used)` → `UPDATE: {file} (checksum differs)` (the `--overwrite used` suffix is meaningless post-fix)
   - `WARN: {file} — exists with different content (use --overwrite to replace)` → `PRESERVE: {file} — exists with different content (--no-overwrite, keeping deployed copy)` (only fires under the new opt-out)

   Rejected B (silent on no-op, summary at end) — the per-file line is genuinely useful when the post-merge hook fires from a pull and the user wants to see what changed. The signal-to-noise ratio is fine because the hook only runs when files actually changed.

The fix itself: flipped [scripts/skills-deploy:149](scripts/skills-deploy:149) (`overwrite=true`), added `--no-overwrite` parsing at line 154, renamed log lines, updated the doctor reset hint at line 664, expanded the help text at line 881, rewrote the CLAUDE.md "Template deployment" bullet. Live smoke-tested all three paths (default → UPDATE; `--no-overwrite` → PRESERVE; legacy `--overwrite` → UPDATE).
