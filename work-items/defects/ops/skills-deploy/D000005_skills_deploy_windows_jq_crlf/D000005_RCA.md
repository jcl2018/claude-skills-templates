## Symptom

`scripts/skills-deploy install` fails on Windows with two distinct errors caused by the same upstream data corruption:

1. **Template name validation error.** Template names read from the catalog come back as `company-workflow/tracker-feature.md\r` (trailing CR). The validator's `\.md$` regex rejects the name because `\r` is not whitespace to the regex engine.
2. **Integer comparison error.** `files | length` returns the string `0\r` instead of `0`. Bash then errors with `[: : integer expression expected` when evaluating `[ "$count" -eq 0 ]` (or similar test expressions).

Frequency: **always** on Windows hosts. **Never** on macOS or Linux.

Reported by a user deploying `/company-workflow` to a Windows host. The bug is in `scripts/skills-deploy` itself — any skill install that shells through `skills-deploy` is affected on Windows. Reporter already applied the fix locally; this RCA exists to land it in the repo with a regression test.

## Reproduction Steps

1. On a Windows host with Git Bash (or MSYS / WSL with Windows `jq.exe` in `$PATH`), clone this repo
2. Run `scripts/skills-deploy install`
3. **Observe (Symptom A):** error referencing an invalid template name like `company-workflow/tracker-feature.md\r` — regex check against `\.md$` fails
4. **Observe (Symptom B):** bash error `[: : integer expression expected` during jq-driven length comparisons (e.g. `[ "$(jq '.files | length' ...)" -eq 0 ]`)
5. On macOS or Linux the same command completes cleanly — jq emits bare `\n` line endings there

**Environment:** Windows (Git Bash, MSYS, or WSL invoking Windows `jq.exe`). Root cause is `jq.exe` writing output in text mode, which is standard for Windows binaries built against the C runtime default.

## Root Cause

**Root cause:** Windows builds of `jq` (`jq.exe`) write output in text mode by default, which translates every `\n` the program writes into `\r\n` at the OS level. Bash scripts consuming that output get trailing `\r` bytes that (a) break regex anchors like `\.md$` and (b) make numeric strings unparseable by bash's arithmetic tests. macOS and Linux jq builds write in binary mode; the bug is Windows-exclusive.

**Location:** `scripts/skills-deploy` — every call to `jq` throughout the script is affected. The remediation point is a single shell-function wrapper inserted after the existing `require_jq()` function, which sits at `scripts/skills-deploy:19-21`.

## Fix Description

Insert a shell-function wrapper immediately after `require_jq()` in `scripts/skills-deploy` (line 22, the blank line between `require_jq()` and `manifest_read()`):

```bash
jq() { command jq "$@" | tr -d '\r'; }
```

How it works:
- Bash resolves `jq` to the shell function before consulting `$PATH`, so every existing `jq ...` call site in `skills-deploy` is transparently routed through the wrapper.
- `command jq` bypasses the function in the wrapper body itself, invoking the real binary.
- `tr -d '\r'` is a no-op on macOS/Linux (input has no `\r` to strip) and a fix on Windows (strips the CRLF back to LF).
- No call-site edits. Zero behavior change on Unix. One line total.

Follow-ups worth doing in the same PR:
1. Grep `scripts/` for other `jq ` invocations and apply the same wrapper pattern where they are reachable during Windows install.
2. Add a regression test that greps for raw `\r` bytes in the output of an end-to-end install flow (can mock Windows behavior by piping known CRLF input through the relevant script fragment, or by running in a Windows CI matrix).

**Why not alternative fixes:**
- **Per-call `| tr -d '\r'`** — works but is easy to miss at future call sites.
- **`jq --raw-output0` / `-j`** — changes output semantics; doesn't fix line endings.
- **`dos2unix` piping** — extra external dependency.
- **`winpty`** — heavier, not always installed.

The shell-function override is the minimum-intrusion fix that also future-proofs new call sites.
