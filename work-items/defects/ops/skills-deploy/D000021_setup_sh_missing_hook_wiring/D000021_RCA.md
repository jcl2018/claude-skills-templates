## Symptom

On a fresh clone, running only the documented first-time bootstrap
(`scripts/setup.sh`) does **not** install the git hooks. Specifically
`.git/hooks/post-merge` is absent after `setup.sh` completes. The D000013
post-merge auto-resync hook — the mechanism that makes "git pull keeps my
skills current" true — is silently inert until the user separately runs
`scripts/setup-hooks.sh` by hand (documented only as a manual "Once per clone"
step in CLAUDE.md). Always reproduces (deterministic; not intermittent).

## Reproduction Steps

1. `git clone https://github.com/jcl2018/claude-skills-templates.git /tmp/cst-test`
2. `/tmp/cst-test/scripts/setup.sh`   (the only documented first-time bootstrap)
3. **Observe:** `ls -l /tmp/cst-test/.git/hooks/post-merge` → No such file. The
   post-merge auto-sync hook was never installed; `git pull` will not re-sync
   `~/.claude/` until the user manually runs `scripts/setup-hooks.sh`.

**Environment:** workbench at current `main` (D000013 closed; the post-merge
hook body exists inside `setup-hooks.sh`). `setup.sh` runs under
`set -euo pipefail` (setup.sh:12). `setup-hooks.sh` resolves `HOOK_DIR` and can
`exit 1` on failure (setup-hooks.sh:18–21).

## Root Cause

**Root cause:** `scripts/setup.sh` — the documented first-time bootstrap —
terminates with `exec "$CLONE_DIR/scripts/skills-deploy" install` and never
invokes `scripts/setup-hooks.sh`. Git-hook installation is factored into a
*separate* script that is documented only as a manual step. So the D000013
post-merge auto-resync hook is never installed on the documented bootstrap
path; the workbench's "git pull keeps you synced" guarantee silently degrades
to manual behavior on every fresh machine until the user runs a second script
by hand. The mechanism that keeps you synced is not bootstrapped by the thing
that sets you up.

**Location:** `scripts/setup.sh:33` (the final `exec` line; the missing
`setup-hooks.sh` call belongs immediately before it). Hook bodies already
exist correctly in `scripts/setup-hooks.sh` (D000013) — this is purely a
*wiring* gap in `setup.sh`, not a defect in the hook logic.

## Fix Description

Two one-line changes, scope settled at autoplan GATE #1:

1. **`scripts/setup.sh`** — insert exactly one line immediately *before*
   setup.sh:33, leaving `exec` as the unchanged last line:

   ```sh
   "$CLONE_DIR/scripts/setup-hooks.sh" || echo "WARN: hook install failed (run scripts/setup-hooks.sh manually)" >&2
   ```

   The `|| echo ... >&2` guard is **mandatory and load-bearing**: `setup.sh`
   runs under `set -euo pipefail`, and `setup-hooks.sh` can `exit 1`. A bare
   call would abort `setup.sh` before the deploy `exec` — a regression of the
   working install path. The guard makes hook install best-effort while the
   deploy still runs. `CLONE_DIR` is resolved at setup.sh:14–21 (before the
   insertion point) so the line is correct in both documented invocation modes.
   `setup-hooks.sh` is already idempotent and worktree-safe — reuse it, do not
   reimplement.

2. **`scripts/test.sh`** — add ONE `if grep` source-level assertion inside the
   existing D000013 regression block (test.sh:724–749), matching that block's
   exact `if grep -q ... ; then ok "..." ; else fail_test "..." ; fi` idiom:

   ```sh
   if grep -q 'setup-hooks.sh' "$REPO_ROOT/scripts/setup.sh"; then
     ok "setup.sh bootstrap invokes setup-hooks.sh (post-merge hook auto-installed on fresh clone)"
   else
     fail_test "setup.sh does not invoke setup-hooks.sh — fresh-clone bootstrap leaves auto-sync hook uninstalled (D000013 bootstrap-wiring guard)"
   fi
   ```

   Static assertion only — no `git init`, no fixture, no `setup.sh` execution
   (its clone path hits the network; the existing D000013 block deliberately
   avoids firing hooks in CI for the same reason). Catches the silent
   regression (someone removes the wiring line) in the file that already owns
   D000013 hook-wiring guards, at zero new maintenance surface.

**Mandatory disclosure (no code change):** the `/ship` CHANGELOG entry and PR
body MUST state that `setup.sh` now also installs the commit-blocking
`pre-commit` hook (alongside the intended `post-merge` auto-sync hook).
`setup.sh` takes its update branch on every re-invocation and `setup-hooks.sh`
unconditionally rewrites both hooks, so this opts any prior `setup-hooks.sh`
holdout into commit-blocking pre-commit validation on their next `setup.sh`
run. Conscious distribution decision (autoplan Eng+DX SHIP-WITH-DISCLOSURE),
not a silent surprise.
