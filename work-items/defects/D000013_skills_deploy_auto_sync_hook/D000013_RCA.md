## Symptom

After `git pull` brings new templates into the workbench source, `~/.claude/templates/` does not update until someone manually runs `scripts/skills-deploy install --overwrite`. D000012 added a regression check that catches the drift (and ships v1.1.1 with the deployed copy refreshed) but the underlying mechanism — manual deploy — remains. Every workbench pull is a fresh opportunity for drift until the next `skills-deploy install` runs.

This is not a bug per se, it's a workflow gap. The fix is to make pull and deploy a single act on this machine.

## Reproduction Steps

1. On the workbench, edit `templates/personal-workflow/tracker-feature.md`, commit, merge to `main`.
2. From a separate clone (or a CI worker), `git pull origin main`.
3. **Observe:** `templates/personal-workflow/tracker-feature.md` is current in the working tree, but `~/.claude/templates/personal-workflow/tracker-feature.md` is whatever the last `skills-deploy install` left there. They diverge until the next manual deploy.
4. `./scripts/test.sh` correctly fails on the D000012 regression block (introduced in v1.1.1) — that's how the drift gets caught today.
5. **Post-fix expectation:** step 2 ends with `~/.claude/templates/` already byte-current. Step 4 stays green without anyone running `skills-deploy` by hand.

**Environment:** macOS Darwin 25.3.0, workbench at v1.1.1 (commit `5b421a4`). `scripts/setup-hooks.sh` currently installs only the pre-commit hook (validate.sh + per-skill checks).

## Root Cause

**Root cause:** No automatic trigger ties workbench pulls to `skills-deploy install`. The deploy mechanism is correct; the activation pattern is manual-only. Until v1.1.1, the workbench had no way to even detect drift (D000012 fixed detection); now we close the loop by removing the activation gap.

**Location:** `scripts/setup-hooks.sh` only installs `pre-commit`; there is no `post-merge` hook on a fresh clone. `~/.claude/templates/` only refreshes when the dev remembers to run `skills-deploy install --overwrite`.

## Fix Description

Extend `scripts/setup-hooks.sh` to write a second hook at `.git/hooks/post-merge`:

```bash
#!/usr/bin/env bash
# Auto-installed by scripts/setup-hooks.sh
# Re-deploys skills + templates after every git merge/pull that touches them.
# Closes D000012 Option C2: deploy is the per-machine sync-up.

REPO_ROOT="$(git rev-parse --show-toplevel 2>/dev/null)"
[ -z "$REPO_ROOT" ] && exit 0

CHANGED=$(git diff-tree -r --name-only --no-commit-id ORIG_HEAD HEAD 2>/dev/null \
  | grep -E '^(templates/|skills/|skills-catalog\.json|rules/)' || true)
[ -z "$CHANGED" ] && exit 0

echo "[skills-deploy] templates/skills/catalog/rules changed — re-deploying..."
"$REPO_ROOT/scripts/skills-deploy" install --overwrite
```

Filter rationale:
- `templates/` — the original D000012 case
- `skills/` — symlinks are already auto-current, but `--overwrite` is idempotent on symlinks (re-creates them, fast). Triggering on skill changes also catches new skill additions.
- `skills-catalog.json` — controls what `skills-deploy` knows to copy. Drift here is silent and has the same downstream blast radius as template drift.
- `rules/` — deployed by `skills-deploy` to `~/.claude/rules/`, same gap as templates.

The hook is silent on no-op (most pulls don't touch deploy-relevant files). When it does run, it prints one line so the dev knows their `~/.claude/` just got synced.
