## Symptom

The v1.1.0 install workflow for `work-copilot` on a Windows work box requires
cloning the entire `claude-skills-templates` repository (~350 files,
multi-MB), even though the Copilot user only needs `work-copilot/` (~50
files) plus `scripts/copilot-deploy.py` (1 file). Files pulled but not
needed on a Copilot-only machine include:

- `.claude/` — Claude Code workspace settings
- `skills/personal-workflow/` and `skills/system-health/` — Claude-only skills
- `skills/company-workflow/` — the upstream the bundle mirrors (duplicated
  content vs the byte-identical mirror under `work-copilot/`)
- `templates/personal-workflow/` — Claude-only templates
- `work-items/` — the workbench's own work-tracking history (irrelevant to
  end users, exposes internal feature/defect history)
- `docs/`, `.docs/`, root tooling configs (`package.json`, etc.)

Reproduction frequency: **always** — it's a property of the documented install
workflow, not a sometimes-bug.

## Reproduction Steps

1. On a fresh Windows work box with VS Code + Copilot but no Claude Code:
   ```sh
   git clone https://github.com/jcl2018/claude-skills-templates.git
   cd claude-skills-templates
   git checkout feat/v1-cut    # or main once landed
   ```
2. **Observe:** the local checkout contains:
   - `.claude/` — irrelevant to Copilot
   - `skills/personal-workflow/` + `skills/system-health/` — Claude-only
   - `skills/company-workflow/` — entire upstream (the mirror copy under
     `work-copilot/` is what the installer reads)
   - `work-items/`, `docs/`, `.docs/`, `bun.lockb`, `package.json`, etc.
3. Run `python scripts/copilot-deploy.py install <work-target>` — works
   correctly. Only the bundle ends up under `<work-target>/.github/`.
4. **The bug is in step 1**, not step 3. The installer is correct; the
   distribution path is the problem.

**Environment:** Any OS — the issue is in the documented install workflow
itself, not platform-specific. Surfaced on Windows because that's the
target user's machine.

## Root Cause

**Root cause:** The F000004 feature design assumed Copilot users would
clone the full `claude-skills-templates` workbench to install the bundle.
That assumption is wrong for the actual target user (Windows work box,
no Claude Code, no maintainer history needed). No distribution-path
work item ever filed; the implicit assumption was never challenged.

**Location:** Architectural — no single file owns the bug. The
`copilot-deploy.py` installer is correct; `work-copilot/` is correctly
self-contained; the docs in `work-copilot/README.md` are correct *given
the cloning assumption*. The bug lives in the **mismatch between
"bundle is self-contained inside `work-copilot/`" (already true)** and
**"distribution path delivers only the bundle" (not true).**

## Fix Description

Decision (leaning per Investigation Trail): **publish a release tarball.**

### Mechanism

On every workbench release tag (e.g., `v1.1.0`), publish a release asset
`work-copilot-bundle-vX.Y.Z.tar.gz` containing exactly:

```
work-copilot/                          # entire bundle
scripts/copilot-deploy.py              # installer (renamed to ./copilot-deploy.py inside tarball, or kept under scripts/)
README.md                              # short, bundle-specific (not the workbench README)
```

End-user install on Windows work box:

```sh
# One-time:
curl -L https://github.com/jcl2018/claude-skills-templates/releases/download/v1.1.0/work-copilot-bundle-v1.1.0.tar.gz | tar xz
# Or with gh:
gh release download v1.1.0 -R jcl2018/claude-skills-templates -p 'work-copilot-bundle-*.tar.gz'

# Install into work repo:
python copilot-deploy.py install C:\path\to\work-target
```

### Implementation work units

1. **`scripts/release-bundle.sh`** — packs `work-copilot/` +
   `scripts/copilot-deploy.py` + a bundle-specific README into a tarball,
   names it `work-copilot-bundle-vX.Y.Z.tar.gz` per the VERSION file.
   Verifies tarball contents match expected manifest before publishing.
2. **`/ship` integration** — after tag/push, call
   `gh release create vX.Y.Z work-copilot-bundle-vX.Y.Z.tar.gz` (or extend
   existing release-creation step). Tarball publishing becomes part of
   the release ritual.
3. **`work-copilot/README.md` install section rewrite** — replace the
   current "clone the workbench" path with the tarball install path.
4. **Workbench root `README.md` update** — add a "For Copilot users (work
   box)" section pointing at the tarball. Keep the "For workbench
   maintainers" section (full clone) for contributors.
5. **Optional: `--from-url` flag on `copilot-deploy.py`** — fetch the
   tarball directly. Defer; `curl + tar` is fine for v1 of the fix.

### Why release tarball over alternatives

- **vs sparse checkout:** end-user friction is lower. Tarball needs `curl
  + tar` (universal); sparse checkout needs ~6 git commands a non-git-native
  user has to remember.
- **vs dedicated bundle repo:** no second repo to keep in sync. Sync drift
  becomes a new defect class with the dedicated-repo approach. Tarball is
  generated from the workbench's own bundle dir at release time — single
  source of truth preserved.
- **vs `gh release download` only:** the tarball approach supports both
  `curl` and `gh` users. `gh release download` is the recommended path
  in the docs but `curl` is the fallback for users who don't have `gh`.

### Out of scope for this fix

- Symlink mode (F000004 milestone #7) — different audience (workbench
  contributors), already separately scheduled.
- `pip install`-style packaging — the bundle isn't a Python package and
  shouldn't pretend to be.
- Auto-update mechanism — out of scope; users re-run the curl command on
  new releases.

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Workbench `/ship` flow | Low | Adds one step (publish release asset). If the publish fails, the rest of `/ship` already succeeded; manual `gh release create` recovers | Smoke test in `scripts/test.sh` that runs the bundle pack script and asserts the tarball contains exactly the expected files |
| Existing v0.14.0 / v1.1.0 install behavior on workbench-cloners | None | Cloning the workbench still works for contributors; the new tarball is an additional path, not a replacement | n/a |
| Bundle drift between tarball and source | Low | Tarball is generated from `work-copilot/` at release time; if `validate.sh` Error check 10 is green, the tarball is in sync by construction | Pack script verifies `cmp -s` against source before tarballing |
| Stale tarballs on old releases (v0.14.0, etc.) | Low | Pre-fix releases never had a tarball; users who upgrade install from the new release tag | Document in CHANGELOG: "v1.1.x release adds work-copilot-bundle release asset; pre-1.1 releases require workbench clone" |
| Path inside tarball (`./copilot-deploy.py` vs `./scripts/copilot-deploy.py`) | Med | Decision affects user-facing install command. Pick one and document explicitly | Default to keeping `scripts/copilot-deploy.py` (matches workbench) so the same `python scripts/copilot-deploy.py install` muscle memory works in both paths |
