## Summary

`scripts/copilot-deploy.py` (264 LoC, shipped in v0.14.0 via F000005) has two
related classes of file-system boundary violations:

1. **Trusts `install-manifest.json` `dest` values verbatim** — a manifest whose
   `dest` points outside the target repo (`../../../etc/hosts`, absolute paths)
   will be read by `doctor` and unlinked by `remove`.
2. **Follows symlinks in both source and destination trees** — a symlink in the
   source bundle gets copied through (exfil); a symlinked destination path gets
   written through (targeted clobber with `--overwrite`).

Neither is reachable in the current single-user self-install threat model, but
both would be live vulnerabilities if `copilot-deploy.py` is published for
other users to run against repos they don't fully control.

## Symptoms (what the user sees)

None in the current threat model. Both issues are latent: they require an
attacker-controlled manifest or symlink to trigger, and in self-install the
user controls both.

## Root Cause

### Category 1: no path validation in cmd_doctor / cmd_remove

`cmd_doctor` (lines 159-203) and `cmd_remove` (lines 206-235) iterate over
`manifest["files"]` and use `entry["dest"]` directly as a path relative to
`target`. There is no check that the resolved absolute path stays under
`target.resolve()`. Any attacker able to write the manifest controls both
reads and deletes.

### Category 2: symlink-following in copy + walk

- `Path.rglob("*")` in `build_file_map` (line 51) walks through directory
  symlinks.
- `shutil.copy2` (line 123) follows file symlinks by default.
- No `is_symlink()` check anywhere in the source or destination paths.

## 5 Whys

1. Why does `doctor` trust the manifest? → It was written as a convenience
   tool for the author, not a hardened public installer.
2. Why is that tradeoff wrong? → Because `copilot-deploy.py` is committed to
   the public repo and will naturally get copied and used by others.
3. Why did the initial review miss this? → The adversarial review wasn't run
   against the Python code specifically; the template-sync tripwire
   (validate.sh Error check 10) was the main guardrail thought through.
4. Why is symlink-following the default in Python's shutil? → Historical
   Unix semantics; `follow_symlinks=False` is opt-in.
5. Why was this not caught by tests? → No tests existed for
   `copilot-deploy.py` until the smoke test added in this same ship
   (PR #43 hotfix); the smoke test covers the happy path only.

## Fix Direction

One helper centralizes the fix:

```python
def safe_dest(target: Path, rel_posix: str) -> Path:
    """Resolve rel_posix under target and refuse anything that escapes."""
    if rel_posix.startswith("/") or "\\" in rel_posix:
        raise ValueError(f"absolute or windows-style path in manifest: {rel_posix}")
    candidate = (target / rel_posix).resolve(strict=False)
    if target.resolve() not in (candidate, *candidate.parents):
        raise ValueError(f"manifest dest escapes target: {rel_posix}")
    return candidate
```

Apply at every point that consumes a manifest `dest` (cmd_doctor, cmd_remove,
install's existing-manifest diff). For symlinks: reject in `build_file_map`
by checking `p.is_symlink()` (and walk parents up to `bundle_dir`) and skip
any dest whose chain contains a symlink.

## Regression Tests to Add

1. Poisoned `install-manifest.json` with `dest: "../../etc/hosts"` → doctor
   exits non-zero with a "manifest dest escapes target" error; no read attempt.
2. Poisoned manifest with absolute `dest: "/tmp/attacker"` → remove refuses.
3. Symlinked source file (`work-copilot/instructions/evil -> /etc/passwd`) →
   install skips the entry with a clear warning.
4. Symlinked dest dir (target `.github` itself is a symlink) → install refuses
   to write.
