## Symptom

`scripts/test-deploy.sh` test cases T2, T4, T5, T6, T7 fail when run end-to-end. Each failure is related to template ownership lifecycle: install → ownership tracking → drift → cleanup. The tests reference `doc-RCA.md` at the top-level templates path (e.g., `$REPO_ROOT/templates/doc-RCA.md`), but that file no longer exists at that path.

**Reproduction frequency:** Always (deterministic — file does not exist at that path).

**Severity:** P2 — real coverage loss for skills-deploy template-ownership semantics, but invisible to CI so no in-tree feedback loop. The compounding latent risk (next template move silently breaks tests too) is what elevates this from P3.

## Reproduction Steps

1. Clone the repo at any commit since v1.3.x (when `doc-RCA.md` was subfoldered).
2. From repo root: `./scripts/test-deploy.sh`
3. **Observe:** tests T2, T4, T5, T6, T7 fail. Errors include "no such file or directory" for `templates/doc-RCA.md` references and assertion failures inside the temp install dirs that the tests construct.

**Environment:** Any POSIX shell (bash/zsh) with `jq` and `git`. Tested on macOS 25.3.0 (Darwin) but not platform-specific.

## Root Cause

**Root cause:** Two coupled defects, one immediate and one latent:

1. **Immediate (symptom):** `scripts/test-deploy.sh` references `doc-RCA.md` at the pre-v1.3.x flat path (`templates/doc-RCA.md`). When that template was subfoldered to `templates/personal-workflow/doc-RCA.md` in v1.3.x, the test references were never updated. Tests now fail end-to-end because the referenced file does not exist.

2. **Latent (meta-bug):** `scripts/test.sh` (CI's entry point) invokes `validate.sh` and other suites but does NOT invoke `scripts/test-deploy.sh` end-to-end. It only `grep`s test-deploy.sh for the `jq()` wrapper definition (a structural assertion, not a runtime check). So the failing T2/T4-T7 tests have been silently broken since v1.3.x without any CI signal. This is what allowed the immediate defect to persist undetected.

**Location:**
- Immediate: `scripts/test-deploy.sh` lines containing `doc-RCA.md` references (lines 717, 718, 720, 722, 723, 725, 727, 729, 730, 732, 759, 761, 772, 774, 787, 794, 801, 808, 816, 831, 835, 850 per grep)
- Latent: `scripts/test.sh` — missing invocation of `scripts/test-deploy.sh`

## Affected Components

| Component | File/Module | Impact |
|-----------|------------|--------|
| skills-deploy template-ownership tests | `scripts/test-deploy.sh` (T2, T4, T5, T6, T7) | These tests verify install / ownership-tracking / cleanup semantics for skills-deploy. With them failing, regressions in those semantics would not be caught locally or in CI. |
| CI test runner | `scripts/test.sh` | The wire-grep is too weak — it only confirms the `jq()` wrapper exists in test-deploy.sh as a string, not that test-deploy.sh actually runs and passes. Real test-deploy.sh failures slip through. |
| skills-deploy itself | `scripts/skills-deploy` | Indirect — the tests measure its template-ownership semantics. If the tests rot, regressions in skills-deploy go uncaught. |

## Fix Description

**Two-part fix (matches the two-part root cause):**

1. **Re-point references.** In `scripts/test-deploy.sh`, replace every reference to `doc-RCA.md` (flat top-level path) with `doc-SKILL-DESIGN.md` (the only remaining flat-path template). The tests treat the referenced template as an opaque asset for ownership semantics, so any flat-path template works as the test fixture. doc-SKILL-DESIGN.md has stable ownership (the `templates` catalog entry) and is unlikely to be subfoldered.

2. **Wire test-deploy.sh into CI.** In `scripts/test.sh`, add an invocation of `scripts/test-deploy.sh` after the existing wrapper-grep pre-flight check. Capture the exit code; if non-zero, fail the test.sh suite. This causes any future template-move regression in test-deploy.sh to fail in CI loudly, instead of rotting silently.

The wrapper-grep check stays in test.sh as-is — it's a structural pre-flight (asserts the `jq()` CRLF-stripping wrapper is defined), not a runtime test, and remains independently useful.

**Approach trade-offs (from /office-hours design doc):**
- Approach A (re-point only): fixes the symptom, leaves the latent CI gap. Rejected.
- Approach B (retire failing tests): removes coverage to make red go away. Rejected.
- Approach C (re-point + wire-into-CI, chosen): closes both defects; ~1.5h human effort.

## Regression Risk

| Area | Risk Level | Why | Mitigation |
|------|-----------|-----|------------|
| Existing test.sh suites | Low | New invocation is appended after existing checks; no existing logic changes | Run `./scripts/test.sh` end-to-end before merging; confirm all prior phases still pass and the new test-deploy.sh phase is reached |
| CI wall-clock time | Low | test-deploy.sh adds wall time to CI runs (~30s based on local runs) | Acceptable — current CI ran in 31s for /ship's PR #65; adding ~30s keeps total under 2 min |
| test-deploy.sh own flakiness | Medium | If test-deploy.sh has any flaky tests beyond T2/T4-T7, wiring it into CI will surface them as new noise | Run test-deploy.sh end-to-end 3 times in succession on the fix branch before merging; if any test other than T2/T4-T7 is unstable, fix or quarantine it before wire-up lands |
| Future template moves | Negative risk (this fix prevents a class of bug) | After wire-up, the next template move that misses test-deploy.sh references will fail in CI immediately | This IS the mitigation — it converts a latent bug class into a loud one |
| doc-SKILL-DESIGN.md being subfoldered later | Low | If `templates/doc-SKILL-DESIGN.md` ever moves, this fix re-introduces the same problem | Easy follow-up: introduce a test-only flat template (e.g., `templates/test-fixture.md`) and re-point test-deploy.sh at it. Out of scope for this PR. |
