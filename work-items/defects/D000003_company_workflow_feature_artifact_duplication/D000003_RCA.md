## Symptom

`/company-workflow`'s manifest declares the same 5-artifact set (TRACKER + PRD + ARCHITECTURE + TEST-SPEC + milestones) for both `feature` and `user-story`. When a feature contains nested user-stories — the documented hierarchy — the parent feature dir and each child user-story dir each carry the same five artifacts with overlapping content scopes. Verified concretely in `ai-content/work-items/F973012/` containing `S1441024-hfss-integration/`: the feature dir has `F973012_PRD.md` (148 lines), `F973012_ARCHITECTURE.md` (173 lines), `F973012_TEST-SPEC.md` (76 lines), and the nested user-story dir has the same three artifacts covering the same scope.

Frequency: **always**, on any feature with nested user-stories.

This is the second of three drift symptoms originally filed under D000003 on 2026-04-16. The other two (`workflow_type` frontmatter drift, section-order drift) were spun out to D000004 because they require resolving a separate architectural question about how to enforce template-vs-contract round-trip in CI.

## Reproduction Steps

1. Read `skills/company-workflow/company-artifact-manifests.json`. Confirm both `feature` and `user-story` declare the same 5 required artifacts.
2. Read `skills/company-workflow/WORKFLOW.md`. Confirm the type-to-artifact summary table shows the same 5 for both types.
3. In a consumer repo (ai-content @ master), open `work-items/F973012/`. Confirm 5 artifacts at feature scope.
4. Open the nested `work-items/F973012/S1441024-hfss-integration/`. Confirm the same 5 artifacts at user-story scope.
5. Diff `F973012_PRD.md` against `S1441024-hfss-integration/PRD.md`. **Observe:** content scopes overlap; the feature-level PRD describes the feature, and the user-story-level PRD describes the same scope from a story angle, with no contract-level distinction.

**Environment:** macOS Darwin 25.3.0; consumer repo ai-content @ master; `/company-workflow` deployed via `skills-deploy install`.

## Root Cause

**Root cause:** `skills/company-workflow/company-artifact-manifests.json` was authored without a contract-level distinction between feature-scope and user-story-scope artifacts. The same 5 artifacts were declared for both types as a placeholder, and the resulting duplication was not caught because:
- the validator only checks artifact *presence*, not *content overlap* with parent/child trackers, and
- the personal-workflow split (feature = roll-up; user-story = story-scope) already shipped but was not propagated into company-workflow.

**Location:**
- `skills/company-workflow/company-artifact-manifests.json` (the manifest itself)
- `skills/company-workflow/WORKFLOW.md` (summary table mirroring the manifest)

| Component | File/Module | Impact |
|-----------|-------------|--------|
| Type → artifact map | `skills/company-workflow/company-artifact-manifests.json` | Requires the same 5 artifacts at both feature and user-story scope, causing content duplication |
| Workflow doc | `skills/company-workflow/WORKFLOW.md` | Summary table mirrors the manifest; needs the same update |
| Templates | `templates/company-workflow/` | New `doc-feature-summary.md` template needed for the new artifact |
| Catalog | `skills-catalog.json` | New template entry for `feature-summary.md` so `skills-deploy install` picks it up |
| Consumer repo trackers | `ai-content/work-items/F973012/` | Holds duplicated PRD/ARCH/TEST-SPEC content; will be tolerated as legacy after fix (validator no longer requires them at feature scope, doesn't reject them either) |

## Fix Description

Three coordinated edits in one PR:

1. **Manifest split.** Update `company-artifact-manifests.json` so feature requires `tracker + feature-summary + milestones` (3 artifacts); user-story unchanged at 5.
2. **New template.** Add `templates/company-workflow/doc-feature-summary.md` — a one-pager with: YAML frontmatter (`type: feature-summary`, `parent: {FEATURE_ID}`, `title`, `date`, `author`, `status`), `## Scope` (one paragraph: what this feature delivers), `## Success Criteria` (bulleted measurable outcomes), `## Constituent User-Stories` (markdown links to nested user-story TRACKERs), `## Out-of-Scope` (explicit non-goals).
3. **Catalog + WORKFLOW.md + CHANGELOG.** Register the new template in `skills-catalog.json` so `skills-deploy install` picks it up; update the WORKFLOW summary table; add a CHANGELOG entry with a migration note for legacy ai-content trackers.

Bump skill version per `scripts/collection-version.sh`.

**No round-trip runner needed for this defect.** The duplication symptom is visible by reading the manifest, not by running a validator on a scaffolded file. The validator can't even detect this as a violation. Issue 2 ships independently of the architectural rethink that D000004 carries.
