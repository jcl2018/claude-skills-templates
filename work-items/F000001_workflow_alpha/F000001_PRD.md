## Problem Statement

Claude Code skill developers need a structured way to manage dev work from scoping through shipping. Without it, work items are ad-hoc, there's no audit trail of decisions, and phase transitions happen without gates. The target user is a developer using Claude Code in a repo who wants lightweight but rigorous lifecycle management baked into their AI workflow.

## Mental Model

A single `/workflow` skill acts as a branch-aware router dispatching to four lifecycle phases:

```
Branch (feat/*, fix/*, task/*) → Work Item Resolution → Phase Detection
                                                            │
                    ┌──────────┬──────────┬─────────────────┘
                    ▼          ▼          ▼          ▼
                  Track    Implement   Review      Ship
```

Each phase has sub-gates tracked in TRACKER.md. The branch name is the key — it determines which work item is active and what type it is.

## User Stories

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | Can I scaffold a new work item from a branch? | developer | run `/workflow track create` on a feature branch and get a TRACKER.md + doc artifacts | I have structured docs from day one |
| 2 | core | Can I track lifecycle phases? | developer | see which phase I'm in and what gates remain | I know exactly what's left before shipping |
| 3 | core | Does implement read my specs? | developer | run `/workflow implement` and have it read PRD/ARCHITECTURE/TEST-SPEC | implementation is guided by the specs I wrote |
| 4 | core | Are contracts enforced before shipping? | developer | have review and ship phases invoke `/contracts check` | I can't ship structurally broken docs |
| 5 | core | Can I create different work item types? | developer | create features, defects, tasks, and user-stories | each type gets the right artifacts (defects get RCA, not PRD) |
| 6 | core | Does the router resolve context automatically? | developer | just type `/workflow` and have it detect my branch, find my work item, show my phase | I never manually specify which work item I'm working on |

## Acceptance Criteria

### Story #1: Scaffold work item [core]

```
GIVEN a branch matching feat/* pattern
WHEN  I run /workflow track create --type feature
THEN  work-items/{slug}/ is created with TRACKER.md, PRD.md, ARCHITECTURE.md, TEST-SPEC.md, milestones.md
AND   all placeholders are replaced with real values
AND   TRACKER.md has 4-phase lifecycle with unchecked gates
```

### Story #2: Lifecycle phase tracking [core]

```
GIVEN an active work item with TRACKER.md
WHEN  I run /workflow (no subcommand)
THEN  I see work item name, type, status, branch, current phase (1-4), per-phase completion
AND   options for the next action based on current phase
```

### Story #3: Spec-guided implementation [core]

```
GIVEN a feature work item with PRD.md and ARCHITECTURE.md
WHEN  I run /workflow implement
THEN  it reads the doc triplet and drafts an implementation plan before executing
```

### Story #4: Contract quality gates [core]

```
GIVEN a work item with a doc triplet
WHEN  I run /workflow review or /workflow ship
THEN  /contracts check is invoked
AND   if failures exist, I'm warned and prompted to override or fix
```

### Story #5: Type-specific artifacts [core]

```
GIVEN artifact-manifests.json defines required artifacts per type
WHEN  I create a defect work item
THEN  I get TRACKER.md + RCA.md + test-plan.md (not PRD/ARCHITECTURE/TEST-SPEC)
```

### Story #6: Automatic context resolution [core]

```
GIVEN I'm on branch feat/workflow-alpha
WHEN  I run /workflow
THEN  it detects type=feature, slug=workflow-alpha, finds work-items/workflow-alpha/TRACKER.md, and shows current phase
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Work item scaffolding creates correct artifacts per type | 5/5 types produce manifest-specified files | Manual test per type |
| Phase gates tracked accurately | Status menu reflects actual TRACKER.md state | `/workflow` output matches checked/unchecked gates |
| Contract gates fire at review + ship | Both phases invoke /contracts before proceeding | Run `/workflow review` and `/workflow ship`, observe contract invocation |

## Out of Scope

- Multi-repo work items (work-items/ is per-repo only)
- CI/CD integration (no pipeline hooks)
- Real-time collaboration (single-user, no locking)
- Custom work item types beyond the 4 defined
- The /contracts skill itself (separate feature)
- The /system-health skill (separate feature)
- The /skill-author skill (separate feature)

## Assumptions

- Git is available and branches follow naming conventions (feat/*, fix/*, task/*, etc.)
- Templates are available via the 3-level fallback chain (repo → spec → legacy)
- artifact-manifests.json exists at repo root or ~/.claude/
- gstack `/review` and `/ship` skills are available for delegation at review/ship phases
