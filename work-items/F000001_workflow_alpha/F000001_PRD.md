## Problem Statement

Claude Code skill developers need a structured way to manage dev work from scoping through shipping. Without it, work items are ad-hoc, tracker templates assume multi-person teams (reviewers, Linux builds, JIRA URLs), and there's no validation that work item hierarchies are structurally complete. The target user is a solo developer using Claude Code who wants lightweight but rigorous lifecycle management.

## Mental Model

Three layers of capability:

1. **4-phase lifecycle pattern** (Track -> Implement -> Review -> Ship): Encoded in tracker templates with gates. Branch naming conventions determine work item type. Scaffolding rules in CLAUDE.md (via rules/work-items.md).
2. **Type-specific artifact system**: `artifact-manifests.json` declares what docs each work item type needs. Features get PRD + ARCHITECTURE + TEST-SPEC + milestones. Defects get RCA + test-plan. Tasks get test-plan.
3. **Structural completeness validation**: `/docs check` validates that features have stories and stories have tasks. `/docs tree` renders the hierarchy visually. Graph artifact for downstream tooling.

## Acceptance Criteria

### Story #1-3: Work item scaffolding and artifacts [core]

- AC-1: Scaffolding creates type-specific artifacts per artifact-manifests.json
- AC-2: Features get TRACKER (doc triplet lives at user-story level)
- AC-3: Defects get TRACKER + RCA + test-plan
- AC-4: Tasks get TRACKER + test-plan
- AC-5: Branch naming conventions map to work item types (feat/* -> feature, fix/* -> defect, etc.)

### Story #4: Solo-dev templates [core]

- AC-6: No "reviewer noted", "Linux branch build", JIRA/TFS URL, or workflow_type in templates
- AC-7: Lifecycle gates meaningful for solo developer + Claude Code
- AC-8: Task tracker lighter than feature tracker (fewer gates)
- AC-9: Review work item type removed
- AC-10: Structured IDs: F/S/D/T prefix with 6-digit number

### Story #5-6: Structural completeness [core]

- AC-11: Feature with 0 user-story children flagged INCOMPLETE
- AC-12: Tree report with per-node badges (template, lifecycle, traceability, structure)
- AC-13: work-item-graph.json emitted to .docs/
- AC-14: Hierarchy rules in artifact-manifests.json, not hard-coded

### Story #7: Misplaced detection [core]

- AC-15: Item whose type is not an allowed child of its parent flagged MISPLACED

### Story #8: Claims.json gate [resilience]

- AC-16: Steps 1-5 skip if claims.json missing; Steps 6+ run regardless

### Story #9: /docs tree [usability]

- AC-17: /docs tree renders tree with structural badges only; others show "-"
