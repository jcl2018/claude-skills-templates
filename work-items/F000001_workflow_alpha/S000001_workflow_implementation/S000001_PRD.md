## Problem Statement

Claude Code skill development lacks a structured lifecycle. Work items are created ad-hoc, tracker templates assume multi-person teams (reviewers, Linux builds, JIRA URLs), and there is no validation that work item hierarchies are structurally complete. A developer working solo with Claude Code has no quality gates that match their actual workflow, no type-specific artifact sets, and no way to verify that features are decomposed into actionable work.

## Mental Model

Three layers of capability, each building on the previous:

1. **Solo-dev templates**: 4 tracker templates rewritten with a 4-phase lifecycle (Track -> Implement -> Review -> Ship) and gates meaningful for a single developer + Claude Code. Type-specific artifact scaffolding via artifact-manifests.json.
2. **Structural completeness** (Check 4 in /docs check): Does the hierarchy exist? Feature -> stories -> tasks. Validates placement rules, detects orphans and misplaced items.
3. **Visualization and artifacts**: Tree report with per-node badges, machine-readable graph artifact for downstream tooling, human-readable report.

## Acceptance Criteria

### Story #1-3: Solo-dev templates and scaffolding [core]

- AC-1: No "reviewer noted", "Linux branch build", JIRA/TFS URL, or workflow_type in templates
- AC-2: Lifecycle gates meaningful for single developer + Claude Code
- AC-3: Task tracker lighter than feature tracker (fewer gates)
- AC-4: All 4 tracker templates updated consistently
- AC-5: Structured IDs: F/S/D/T prefix with 6-digit number + keywords
- AC-6: Review work item type removed (tracker-review.md deleted)
- AC-7: Scrum removed (subcommand, template, notes generation)
- AC-8: Phase 1 lists required artifacts per type (feature: PRD+ARCH+TEST-SPEC+milestones, task: test-plan, defect: RCA+test-plan)

### Story #4: Incomplete feature/story detection [core]

- AC-9: Feature with 0 user-story children flagged INCOMPLETE
- AC-10: Feature with >= 1 user-story children shows PASS
- AC-11: User-story with 0 task children flagged INCOMPLETE

### Story #5: Tree report [core]

- AC-12: Tree renders depth-first, sorted alphabetically by slug
- AC-13: Each node shows 4 badges (template, lifecycle, traceability, structure)
- AC-14: Badge shows worst severity from its check category

### Story #6: Graph artifact [core]

- AC-15: work-item-graph.json written to .docs/ with version, nodes, edges, structural_rules
- AC-16: Each node includes id, slug, type, state, path, parent, children, badges, completeness

### Story #7: Configurable hierarchy [core]

- AC-17: Hierarchy rules read from artifact-manifests.json hierarchy field
- AC-18: Missing hierarchy field warns and skips structural checks

### Story #8: /docs tree subcommand [usability]

- AC-19: /docs tree renders tree with structural badges only; other badges show "-"

### Story #9: Misplaced detection [core]

- AC-20: Item whose type is not an allowed child of its parent flagged MISPLACED
- AC-21: Placement rules: feature/defect root-level, story inside feature, task inside story

### Story #10: Lifecycle cross-reference [observability]

- AC-22: "Broken down" checked + 0 children = LIFECYCLE_INCONSISTENT

### Story #11: Claims.json gate [resilience]

- AC-23: Steps 1-5 skip with note if claims.json missing; Steps 6+ run regardless

## Success Metrics

- Test suite passes with 0 failures after all changes
- /docs check flags F000002 as INCOMPLETE on first run
- /docs tree renders in under 5 seconds on repos with < 50 work items
- Zero false positives on correctly structured hierarchies (F000001)

## Out of Scope

- Dependency edges (blocked-by, depends-on) in graph artifact — deferred to v2
- validate.sh integration with graph artifact — deferred until schema stable
- Auto-scaffolding of missing children — validation only, no enforcement

## Assumptions

- Work items follow the directory nesting convention from rules/work-items.md
- artifact-manifests.json is the single source of truth for hierarchy rules and artifact sets
- Solo developer + Claude Code is the primary workflow
