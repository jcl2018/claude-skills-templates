## Problem Statement

The /docs check skill validates work items for template compliance, lifecycle consistency, and PRD-to-TEST-SPEC traceability. But it does NOT check structural completeness. F000002_system_health_v1 has all its doc artifacts (TRACKER, PRD, ARCHITECTURE, TEST-SPEC, milestones) but zero child user stories or tasks. It passes all three existing checks. Nobody flags that a feature without decomposition is structurally incomplete.

The user who runs `/docs check` expects a complete picture of work item health. Right now they get a false "all good" for a feature that has never been decomposed into actionable work.

## Mental Model

Three layers of validation, building on each other:

1. **Structural completeness** (Check 4): Does the hierarchy exist? Feature → stories → tasks.
2. **Tree report**: Visual hierarchy showing all nodes with per-category badges.
3. **Graph artifact**: Machine-readable JSON representation of the work item graph for downstream tooling.

## Acceptance Criteria

### Story #1: Incomplete feature detection [core]

- AC-1: Feature with 0 user-story children is flagged `[INCOMPLETE]`
- AC-2: Feature with >= 1 user-story children shows `[PASS]`

### Story #2: Incomplete story detection [core]

- AC-3: User-story with 0 task children is flagged `[INCOMPLETE]`
- AC-4: User-story with >= 1 task children shows `[PASS]`

### Story #3: Tree report [core]

- AC-5: Tree renders depth-first, sorted alphabetically by slug
- AC-6: Each node shows 4 badges (template, lifecycle, traceability, structure)
- AC-7: Badge shows worst severity from its check category

### Story #4: Graph artifact [core]

- AC-8: `work-item-graph.json` written to `.docs/` with version, nodes, edges, structural_rules
- AC-9: Each node includes id, slug, type, state, path, parent, children, badges, completeness

### Story #5: Configurable hierarchy [core]

- AC-10: Hierarchy rules read from `artifact-manifests.json` `hierarchy` field
- AC-11: Missing `hierarchy` field warns and skips structural checks

### Story #6: Claims.json gate [resilience]

- AC-12: Steps 1-5 skip with note if claims.json missing; Steps 6+ run regardless

### Story #7: /docs tree subcommand [usability]

- AC-13: `/docs tree` renders tree with structural badges only; other badges show "—"

### Story #8: Orphan/misplaced detection [core]

- AC-14: Item whose type is not an allowed child of its parent is flagged MISPLACED
- AC-15: Placement rules: feature/defect root-level, story inside feature, task inside story

### Story #9: Lifecycle cross-reference [observability]

- AC-16: "Broken down" checked + 0 children = LIFECYCLE_INCONSISTENT

### Story #10: Completeness counts [usability]

- AC-17: Tree report shows child count vs minimum per node

## Success Metrics

- `/docs check` flags F000002 as INCOMPLETE on first run
- `/docs tree` renders in under 5 seconds on repos with < 50 work items
- Zero false positives on correctly structured hierarchies (F000001)

## Out of Scope

- Dependency edges (blocked-by, depends-on) — deferred to graph artifact v2
- Impact analysis (file change → affected nodes) — north star
- validate.sh integration — deferred until graph schema stable
- Auto-scaffolding of missing children — validation only, no enforcement

## Assumptions

- Work items follow the directory nesting convention from rules/work-items.md
- artifact-manifests.json is the single source of truth for hierarchy rules
- The `hierarchy` field is required — repos without it skip structural checks entirely
