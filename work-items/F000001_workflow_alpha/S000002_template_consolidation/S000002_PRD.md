## Problem Statement

The 5 tracker templates (feature, defect, task, user-story, review) were written with a team/enterprise workflow in mind. Gates like "Code review requested (reviewer noted)", "Linux branch build passes", and `url: "{JIRA_OR_TFS_URL}"` don't apply to a solo developer working with Claude Code. This creates friction: the developer checks off gates that don't mean anything, or leaves them unchecked and the phase detection thinks work is incomplete.

## Mental Model

Three changes layered on top of the existing 4-phase lifecycle:

1. **Gate language** — rewrite Phase 3 (Review) and Phase 4 (Ship) gates to reflect what a solo dev actually does
2. **Template weight** — make the task tracker lighter than the feature tracker
3. **Frontmatter cleanup** — remove fields that only exist for enterprise integrations

## User Stories

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | Do work items have structured IDs? | solo developer | have every work item get an ID in the format `{letter}{6-digit}_{keywords}` (F=feature, S=user-story, D=defect, T=task), folder names use this ID format, and doc filenames are prefixed with the ID (e.g. `S000001_PRD.md`) | I can reference items unambiguously everywhere — ID, folder, and filenames |
| 2 | core | Is scrum removed? | solo developer | have all scrum-related functionality removed (scrum subcommand, scrum template, scrum notes generation) | the workflow is streamlined for solo dev without team ceremony artifacts |
| 3 | core | Is the review work item type removed? | solo developer | not have a review work item type — no tracker-review.md, no review-* branch pattern, no doc-review-notes.md | there are only 4 types: feature, defect, task, user-story |
| 4 | core | Does the feature tracker enforce the right gates? | solo developer | have feature Track gates: scope AC, create branch, produce doc triplet, break down into child tasks/stories. Implement gates: read doc triplet (build-forward), commit core implementation, complete child tasks, update Files section | the lifecycle matches how I actually work on features |
| 5 | core | Does the defect tracker enforce the right gates? | solo developer | have defect Track gates: document reproduction steps, create branch, log initial symptom. Implement gates: debug-backward mode (symptoms → hypotheses → root cause), 3-strike escalation if 3 hypotheses fail, commit fix + regression test, write RCA to Insights | defects follow a systematic investigation path |
| 6 | core | Does the user-story tracker enforce the right gates? | solo developer | have user-story Track gates: define AC, create branch, produce doc triplet, create milestones, break down tasks if needed. Implement gates: build-forward mode (from doc triplet + AC), commit implementation, verify AC met | user stories follow a medium-weight path between feature and task |
| 7 | core | Does the task tracker enforce the right gates? | solo developer | have task Track gates: read parent scope, create branch, populate Files. Implement gates: commit core changes, keep Todos current, update Files with changed files | tasks are the lightest path — no doc triplet, no AC, just do the work |
| 8 | core | Is the Ship phase simplified to two steps? | solo developer | have Ship phase gates be just: `/ship` (create PR) and `/land-and-deploy` (merge + deploy) across all tracker types | no redundant gates — ship is just two commands |
| 9 | core | Is the Review phase just doc review and generation? | solo developer | have Review phase gates be: doc review and doc generation for features. Task and defect add test verification. User-story adds doc triplet alignment check (TEST-SPEC) | each type gets the right level of review |
| 10 | core | Are child item and doc completeness rules enforced? | solo developer | have features require at least one user story, user stories require at least one task, and all items must have their required docs (per artifact-manifests.json) verified in Phase 3 Review | nothing ships without proper decomposition and documentation |
