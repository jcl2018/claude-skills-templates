## Problem Statement

The cj_goal orchestrator framework compacts BETWEEN phases (the silent build dispatches scaffold / implement / QA as depth-≤2 leaf subagents that return ≤200-token summaries) but never WITHIN a long inline phase. `/office-hours` runs inline at the top level (subagents have no AskUserQuestion tool), so its full transcript stays resident in the orchestrator's context window through the rest of the build — wasted context pressure (GAP C / P1). The user here is the orchestrator operator running a long `/CJ_goal_feature` build: the office-hours transcript they no longer need crowds the window that the autonomous build budget depends on.

## Mental Model

<!-- If the story has layers, phases, or modes, add a diagram or short
     explanation here so readers understand the structure before reading
     individual requirements. For simple stories, a one-sentence summary suffices. -->

The fix is narrow: after the office-hours inline phase, write ONE compact phase receipt to `.cj-goal-feature/`, then have the orchestrator's post-phase steps continue from that receipt file rather than from the resident transcript. The existing resume state file (`.cj-goal-feature/${branch}.state`, flat key=value, atomic mktemp+mv) is already a proto-receipt; generalize it into a per-phase receipt chain that PRESERVES the atomic-write + the ancestor-SHA validate-before-skip contract. The receipt schema is the SAME one S000093 (Trajectory QA) establishes — one schema, not two. Scope is the known long inline phases (office-hours) only; this is deliberately NOT a generic "compact everything" framework.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group.

### Story #1: Receipt written at the office-hours boundary [core]

```
GIVEN  a /CJ_goal_feature build that has just completed the office-hours inline phase
WHEN   the orchestrator crosses the office-hours boundary
THEN   a compact phase receipt is written to `.cj-goal-feature/` via the existing atomic mktemp+mv path
```

### Story #2: Post-phase steps read the receipt, not the transcript [core]

```
GIVEN  the office-hours receipt has been written to `$RECEIPT_PATH`
WHEN   the orchestrator runs its post-office-hours steps (the design-summary digest)
THEN   those steps READ `$RECEIPT_PATH` and the design-summary digest is sourced from the receipt file, not regenerated from conversation context
```

### Story #3: Scoped to known long inline phases [core]

```
GIVEN  the within-phase receipt mechanism
WHEN   it is implemented
THEN   it is scoped to the known long inline phases (office-hours) and introduces no generic "compact everything" compaction framework
```

### Story #4: Receipt reuses the shared schema [integration]

```
GIVEN  S000093 (Trajectory QA) establishes a receipt schema
WHEN   this story writes the office-hours receipt
THEN   the receipt reuses that same shared schema (one format, set by whichever story ships first)
```

### Data Flow

<!-- How does data move through the system for the primary use case?
     Step-by-step, component to component. -->

1. The orchestrator runs the office-hours inline phase, producing an APPROVED design doc and a full resident transcript.
2. At the office-hours boundary, the orchestrator writes a compact phase receipt (shared S000093 schema) to `.cj-goal-feature/` via atomic mktemp+mv, recording `$RECEIPT_PATH`.
3. The post-office-hours steps read `$RECEIPT_PATH`; the design-summary digest is sourced from the receipt file rather than regenerated from the resident transcript.
4. On resume, the per-phase receipt chain is read with the existing ancestor-SHA validate-before-skip contract preserved.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale.
     Future readers need to know why this path over the others. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Receipt home | Generalize the existing `.cj-goal-feature/${branch}.state` into the receipt chain | Add a sibling per-phase receipt file | Avoids a second state surface; reuses the proven atomic write + validate-before-skip contract |
| Scope | Known long inline phases (office-hours) only | A generic "compact everything" framework | Lowest marginal value + highest over-build risk; overlaps Claude Code's built-in auto-compaction (sequenced last for this reason) |
| Receipt schema | Reuse S000093's schema | Define a second schema for this story | One schema, not two — whichever story ships first sets it |
