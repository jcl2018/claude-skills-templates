## Problem Statement

The cj_goal user-story/feature QA path can report GREEN without actually re-executing on a same-SHA resume — so a human who trusts a resumed run is trusting a verification that may never have re-run. The operator (and the orchestrator that consumes the QA verdict) is the user with this problem.

Two distinct resume mechanisms skip a genuine re-run (GAP A of F000053):

1. **`qa.md` Step 3 NO-OP short-circuit** (`qa.md:144-146`): user-story QA is short-circuited as a NO-OP when BOTH QA-owned Phase-2 gates are checked AND a `[qa-pass]` journal entry is **dated-today OR matches HEAD**. The HEAD-match branch is sound; the **date-only branch is the hole** — a same-day earlier-commit `[qa-pass]` marker satisfies it even though behavior may have changed since that commit.
2. **`CJ_goal_feature/pipeline.md` phase-skip** (`pipeline.md:502`): the feature orchestrator's resume is phase-granular and **skips the whole QA phase** when `LAST_PHASE ∈ {qa, ship}` on a still-valid SHA. It never dispatches QA, so a same-SHA resume where untracked / generated / fixture / environment state changed reports ready without re-verifying.

Boundary checks (`/CJ_personal-workflow check`) validate artifact SHAPE, not behavior — they cannot catch this. Defect/task QA already re-runs unconditionally (`CJ_goal_defect/pipeline.md:857`) and a stale SHA already demotes qa→impl, so this is the user-story/feature path specifically, not all three verbs.

## Mental Model

QA has three layers on a resume, cheapest-first:

```
RESUME → QA
  │
  ├─ [1] re-validate (CHEAP): re-run smoke rows + read the execution receipt
  │        ├─ receipt present + complete + SHA-matches HEAD ──► GREEN (no E2E re-run)
  │        └─ receipt missing / incomplete / stale-SHA ───────► [2]
  │
  ├─ [2] re-run E2E subagent (EXPENSIVE, ~5 min) — only when [1] fell through
  │
  └─ verdict: FAIL CLOSED
           every AC has a passing row  ──► GREEN
           any AC uncovered, OR no receipt (artifacts-only) ──► RED
```

WRITES are idempotent across re-execution (reuse the Step 6.5 run-start marker): no duplicate Phase-2 gate transitions, no journal thrash.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group.

### Story #1: Resume re-validates; E2E re-runs only when receipt insufficient [core]

```
GIVEN a user-story whose two QA-owned Phase-2 gates are checked AND a `[qa-pass]`
      journal entry exists that is dated-today but from an EARLIER commit than HEAD
WHEN  QA runs on a same-SHA resume
THEN  the `qa.md` Step 3 NO-OP no longer short-circuits on the date-only branch:
      QA re-validates (re-runs the smoke rows + checks the execution receipt),
      AND re-runs the expensive E2E subagent ONLY when the receipt is missing,
      incomplete, or stale-SHA (it does not unconditionally re-pay the ~5-min
      E2E budget when a complete HEAD-matching receipt is present)
```

### Story #2: QA writes a structured execution receipt [observability]

```
GIVEN QA executes (a first run or a re-validate)
WHEN  it finishes running the test rows
THEN  it writes an execution receipt capturing {test commands run, timestamp
      (completed_at), commit SHA, one pass/fail row per acceptance-criterion /
      test-row}, matching the work-copilot `receipts.qa` schema:
      completed_at, test_rows_run, ac_ids_covered, ac_ids_uncovered,
      diff_audit.changed_files_without_tests, ready_for_ship
```

### Story #3: QA fails closed on a missing/incomplete receipt [resilience]

```
GIVEN QA is asked for a verdict
WHEN  the execution receipt is missing OR incomplete (an AC has no passing row)
THEN  QA reads RED; it reads GREEN only when every acceptance-criterion has a
      passing row in the receipt (fail closed — never green-by-absence)
```

### Story #4: An artifacts-only "edited but never executed" item is flagged RED [resilience]

```
GIVEN a work-item with its artifacts present (SPEC/DESIGN/TEST-SPEC + edited code)
      but NO execution receipt (it was edited but never actually run)
WHEN  QA evaluates it
THEN  it is flagged RED, not silently passed
```

### Story #5: Re-execution is idempotent in its writes [core]

```
GIVEN QA has already run once and transitioned the Phase-2 QA-owned gates
WHEN  QA re-executes on a resume (reusing the Step 6.5 run-start marker)
THEN  it adds no duplicate Phase-2 gate transition and no journal thrash
      (the writes are idempotent; the run-start marker dedupes them)
```

### Data Flow

<!-- How does data move through the system for the primary use case?
     Step-by-step, component to component. -->

1. A resume re-enters `CJ_goal_feature/pipeline.md`. With `LAST_PHASE ∈ {qa, ship}` and a still-valid SHA, the orchestrator re-dispatches QA (NEW) rather than phase-skipping.
2. `qa.md` Step 6.5 writes/reuses a run-start marker keyed to the current SHA (dedup anchor for idempotent writes).
3. `qa.md` Step 3 evaluates the NO-OP: it short-circuits ONLY when a complete `[qa-pass]`/receipt matches HEAD — the date-only branch no longer qualifies.
4. QA re-validates: re-runs the smoke rows and reads the execution receipt.
5. If the receipt is missing/incomplete/stale-SHA, QA re-runs the expensive E2E subagent; otherwise it skips the E2E re-run.
6. QA writes/updates the execution receipt (`receipts.qa` schema) with the commands run, `completed_at`, SHA, per-AC pass/fail rows, `diff_audit.changed_files_without_tests`, and `ready_for_ship`.
7. QA emits the fail-closed verdict: GREEN only if every AC has a passing row and the receipt is complete + HEAD-matching; RED if the receipt is missing/incomplete (including artifacts-only).
8. QA transitions the Phase-2 QA-owned gates idempotently (the Step 6.5 marker prevents a duplicate transition / journal thrash on a second run).

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale.
     Future readers need to know why this path over the others. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Receipt schema | Adopt work-copilot's `receipts.qa` schema verbatim | Invent a new receipt shape for cj_goal | Premise 4 "do not reinvent"; `receipts.qa` is a near-exact prototype and Story 3 of F000053 reuses the SAME schema (one schema, not two). |
| E2E re-run policy on resume | Gate the ~5-min E2E re-run on a missing/incomplete/stale receipt | Re-run E2E unconditionally on every resume | Cost-curation (P1) — receipt-validate is cheap; re-paying the full E2E budget every resume is wasteful. A complete HEAD-matching receipt already vouches for behavior. |
| Number of fixes | Fix BOTH skip paths (`qa.md` marker AND `pipeline.md` phase-skip) | Fix only the `qa.md` marker | They are independent holes in two files; closing only one leaves the orchestrator phase-skip hole open, so a same-SHA resume still never reaches QA. |
| Write-idempotency mechanism | Reuse the existing `qa.md` Step 6.5 run-start marker | Add a new dedup mechanism | The Step 6.5 marker already exists for run-start; reusing it keeps the surface small and avoids a second state surface. |
