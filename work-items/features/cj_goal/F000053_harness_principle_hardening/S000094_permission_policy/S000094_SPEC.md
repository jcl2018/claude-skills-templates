## Problem Statement

cj_goal permission (GAP B / P5) is implicit and scattered across three unconnected places, only two of them live:

- `allowed-tools` frontmatter — **allow**, live.
- the sensitive-surface AskUserQuestion in leaf-skill code — **ask**, live.
- `cj-handoff-gate.sh`'s denylist — **deny**, **DORMANT**: its consumers `/CJ_goal_auto` and `/CJ_goal_run` are deleted, so no current orchestrator invokes the gate; the live orchestrators cite it only rhetorically to justify PR-stop.

The riskiest verbs — git push to main, gh pr merge, rm, network — are not enumerated as explicit deny anywhere. A deny-list that exists but nothing honors is "polish on an unsafe control plane": the permission rule cannot be audited as one contract. The user here is the cj_goal framework maintainer (and any reviewer of a resumed run) who needs to answer "what is this orchestrator allowed to do?" from one declared artifact, not by reading three half-connected surfaces.

## Mental Model

One declared policy, three modes, two live consumers + one dormant deriver:

```
                       ┌──────────────────────────────┐
                       │  permission policy artifact   │
                       │  (doc-spec.md-style: prose +   │
                       │   fenced machine-readable block)│
                       │  rows: {verb, mode, scope}     │
                       │  mode ∈ allow | ask | deny     │
                       │  absent verb  ⇒ deny           │
                       └──────────────┬────────────────┘
                                      │ parsed by scripts/ helper
              ┌───────────────────────┼───────────────────────────┐
              │ references (live)     │ references (live)          │ derives (dormant)
              ▼                       ▼                            ▼
      allowed-tools           sensitive-surface AUQ        cj-handoff-gate.sh
      frontmatter (allow)     in leaf-skill code (ask)     denylist (deny, no live consumer)
              └───────────────────────┴───────────────────────────┘
                                      ▲
                       advisory validate.sh check flags drift
                       (+ parallel test.sh zzz-test-scaffold fixture)
```

### Story #1: One declared allow/ask/deny policy artifact [security]

```
GIVEN the cj_goal framework needs one declared permission contract
WHEN  the policy artifact is authored
THEN  it exists as a doc-spec.md-style file (prose + a fenced machine-readable
      block, rows {verb, mode ∈ allow|ask|deny, scope}) enumerating in-scope edits
      (allow), sensitive surfaces (ask), and the risky verbs git-push-to-main /
      gh-pr-merge / rm / network (deny or ask)
```

### Story #2: Live points reference the policy; dormant denylist derives from it [integration]

```
GIVEN the policy artifact and its scripts/ parser exist
WHEN  the enforcement points are wired
THEN  the two LIVE enforcement points (allowed-tools frontmatter, sensitive-surface
      AUQ) reference the policy AND the dormant cj-handoff-gate.sh denylist is
      derived from the policy's deny set (forward-looking, not a live-enforcement
      claim)
```

### Story #3: An absent verb resolves to deny [security]

```
GIVEN the policy and its scripts/ parser
WHEN  a verb that is NOT enumerated in the policy is resolved
THEN  the parser resolves it to deny (design permission before capability — fail
      closed)
```

### Story #4: Advisory drift check flags policy↔enforcement drift [observability]

```
GIVEN drift is injected between the policy and an enforcement point
WHEN  scripts/validate.sh runs
THEN  an advisory check FIRES on the drift (advisory-first like portability Check
      18 — exit 0, never halts; a follow-up flips it strict once reconciled), and
      the parallel scripts/test.sh zzz-test-scaffold integration fixture ships in
      the SAME PR. REQUIRED, not stretch.
```

### Data Flow

<!-- How does data move through the system for the primary use case?
     Step-by-step, component to component. -->

1. The policy file declares each cj_goal verb as a row `{verb, mode, scope}` in its fenced machine-readable block.
2. The `scripts/` helper parses the block and exposes a resolve(verb) → `allow | ask | deny`; a verb absent from the policy resolves to `deny`.
3. The two live enforcement points reference the policy: `allowed-tools` frontmatter for the allow set, the sensitive-surface AUQ for the ask set.
4. `cj-handoff-gate.sh`'s denylist is derived from the policy's deny set (computed, not hand-maintained) — dormant, correctness-if-reactivated.
5. `scripts/validate.sh` runs the advisory drift check: it compares the policy against the live enforcement points and flags divergence (exit 0 — advisory).
6. `scripts/test.sh`'s zzz-test-scaffold fixture exercises the parser + the advisory check on injected drift, in the same PR.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale.
     Future readers need to know why this path over the others. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Policy artifact shape | doc-spec.md-style file (prose + fenced machine-readable block) parsed by a small `scripts/` helper | A bespoke JSON/YAML config with no prose; or hardcoding the policy inside each skill | Mirrors a pattern the repo already trusts (doc-spec.md); prose explains, the fenced block is the source of truth; one parser, one artifact. |
| Default for an unenumerated verb | deny (fail closed) | allow / pass-through | "Design permission before capability" (P5) — an unknown verb must not be implicitly permitted. |
| Handoff-gate denylist relationship | denylist DERIVES from the policy (forward-looking) | claim cj-handoff-gate.sh as a live enforcement point | The gate is dormant (no live consumer — `/CJ_goal_auto` + `/CJ_goal_run` are deleted); deriving keeps it correct-if-reactivated without overclaiming live enforcement. |
| Drift-check strictness | advisory-first (exit 0), like portability Check 18 | hard-fail strict from day one | Established workbench pattern; lets the policy reconcile before a follow-up PR flips it strict. Still REQUIRED — it is what makes the single source of truth enforceable. |
