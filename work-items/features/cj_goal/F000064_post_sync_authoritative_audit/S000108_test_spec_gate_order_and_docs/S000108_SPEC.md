## Problem Statement

The reorder makes doc-sync precede the qa-audit gate. The `spec/test-spec` registry (the
canonical gate sequence, enforced by `validate.sh` Check 24), the Check-15b-enforced
`docs/workflow.md` ASCII charts, the SKILL.md Overview chains, the catalog descriptions, and
three named tests all describe the old QA → checkpoint → doc-sync order and must be brought
into line, or Check 24 / Check 15b / `tests/cj-goal-doc-sync-wiring.test.sh` will fail.

## Mental Model

Two layers move together: (1) the SOURCE OF TRUTH — the `qa-audit`/`doc-sync` gate `order:`
values in `spec/test-spec-custom.md` plus the `qa-audit` backing prose; (2) the DERIVED
descriptions — CLAUDE.md prose, the workflow.md charts, the SKILL.md chains, the catalog
descriptions, and the named tests that assert the ordering.

## Acceptance Criteria

### Story #1: Gate order swapped [core]

```
GIVEN spec/test-spec-custom.md with qa-audit order=45 and doc-sync order=50
WHEN  the swap is applied
THEN  doc-sync's order is less than qa-audit's order (doc-sync precedes qa-audit), and test-spec.sh --validate passes
```

### Story #2: qa-audit backing prose updated [core]

```
GIVEN the moved audit no longer runs inside QA on orchestrator paths
WHEN  the qa-audit gate's backing:/checks: prose is read
THEN  it names the orchestrator-level post-sync /CJ_doc_audit + /CJ_test_audit + the checkpoint AUQ, not "the QA Step 8.6 audit block"
```

### Story #3: Check-15b charts updated [core]

```
GIVEN the four per-CJ_goal_* ASCII charts in docs/workflow.md
WHEN  validate.sh Check 15b runs
THEN  every chart shows the reordered doc-sync → audit → checkpoint sequence and Check 15b is green
```

### Story #4: Derived prose consistent [usability]

```
GIVEN root CLAUDE.md, the four SKILL.md Overview chains, and the catalog descriptions
WHEN  they are read
THEN  none claims the old QA → checkpoint → doc-sync order; all reflect doc-sync → audit → checkpoint
```

### Story #5: Named tests updated [integration]

```
GIVEN the reorder
WHEN  the test suite runs
THEN  the zzz-test-scaffold fixture, cj-goal-doc-sync-wiring.test.sh (the ORDERING assertion), and the per-pipeline halt-marker tests all assert the new sequence and pass
```

### Story #6: Full suite green [integration]

```
GIVEN all edits applied
WHEN  ./scripts/test.sh runs
THEN  validate.sh Check 24 + Check 15b + every test pass (exit 0)
```

## Architecture

```
SOURCE OF TRUTH
  spec/test-spec-custom.md
    ├── gate qa-audit  order: 45 → 50
    └── gate doc-sync  order: 50 → 45   (doc-sync now precedes qa-audit)
    └── qa-audit backing: → "orchestrator-level post-sync /CJ_doc_audit + /CJ_test_audit + checkpoint AUQ"
         │
         ▼  (must stay consistent — enforced by validate.sh Check 24 / Check 15b)
DERIVED
  CLAUDE.md prose · docs/workflow.md ASCII charts (×4) · SKILL.md Overview chains (×4) · catalog descriptions
  tests: scripts/test.sh zzz-test-scaffold fixture · tests/cj-goal-doc-sync-wiring.test.sh · per-pipeline halt-marker tests
```

### Data Flow

1. Edit `spec/test-spec-custom.md` (swap `order:`, rewrite qa-audit backing) → `test-spec.sh --validate` passes.
2. Edit the derived docs (CLAUDE.md, workflow.md charts, SKILL.md chains, catalog) to match.
3. Update the three named tests to assert the new ordering.
4. Run `./scripts/test.sh`; Check 24 + Check 15b + every test green.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Gate sequence source | The `order:` integers in the test-spec registry | Encoding the order only in prose | The registry is the machine-checked single source of truth (Check 24) |
| Test update strategy | Update all three named tests explicitly | Rely on the implement subagent to find them | Known blind spot — the zzz-test-scaffold fixture + halt-marker tests are systematically forgotten |
