## Problem Statement

The workbench has a machine-readable test contract (`spec/test-spec.md` general +
`spec/test-spec-custom.md` units overlay, ~71 verification units) but no
human-browsable rendering of "what tests exist and what each one proves." A
maintainer or reviewer must read raw YAML. The workflow surface already solved the
symmetric problem with a generated-or-hand-authored two-level human doc set
(`docs/workflow.md` + `docs/workflows/*.md`); the test side has nothing. Hand-
authoring a test catalog would create a second hand-maintained copy of registry-
derivable content that rots and violates the contract's own `single-owner` rule.

The fix: a GENERATED, freshness-gated human catalog rendered from the merged
test-spec registry, enforced both in `validate.sh` (workbench) and in
`/CJ_test_audit` Stage 1 (portable — any consumer repo). This is the second
instance of the proven primitive `README.md` ↔ `generate-readme.sh` ↔
`validate.sh` Check 25.

## Mental Model

One primitive, applied to the test surface:

```
spec registry (source of truth)
   ──(engine --render-docs)──> generated docs/ surface (committed)
   ──(engine --render-docs --check)──> freshness verdict (regenerate→diff→exit code)
        ↑ called by BOTH validate.sh Check 26 AND /CJ_test_audit Stage 1
```

The renderer is PURE: registry in → deterministic docs out. The `--check` mode is
the single freshness owner. Rendered fields only (`label`, `purpose`, `layer`,
`disposition`, `trigger`); the `anchor` is shown as a code reference, never a
claim — so the output is work-item-ID-free by construction and passes Check 19.

## Acceptance Criteria

### Story #1: Render the catalog [core]

```
GIVEN the merged test-spec registry (spec/test-spec.md + spec/test-spec-custom.md) with units rows
WHEN  I run `scripts/test-spec.sh --render-docs`
THEN  one docs/tests/<family>.md is written per unit family present in the registry
AND   docs/test-catalog.md is written as an index grouped by family with per-family counts
AND   every rendered cell comes only from rendered fields (label, purpose, layer, disposition, trigger), with the anchor shown as a code reference
```

### Story #2: Deterministic + ID-free output [resilience]

```
GIVEN a fixed registry
WHEN  I run `--render-docs` twice in a row
THEN  the two outputs are byte-identical (stable sort, fixed headers, single emit order)
AND   no rendered file contains a work-item ID matching [FSTD][0-9]{6}
```

### Story #3: Self-check freshness [observability]

```
GIVEN the on-disk docs/tests/*.md + docs/test-catalog.md match a fresh render
WHEN  I run `scripts/test-spec.sh --render-docs --check`
THEN  it renders to a temp dir, diffs vs on-disk, and exits 0
GIVEN a hand-edited (or missing) catalog file
WHEN  I run `--render-docs --check`
THEN  it exits non-zero and names the mismatched/missing file
```

### Story #4: Hard workbench gate + its test fixture [integration]

```
GIVEN the generated catalog is on disk
WHEN  `scripts/validate.sh` runs
THEN  Check 26 regenerates to a temp dir, diffs vs on-disk, and ERRORs on any mismatch/missing file (mirror of Check 25)
AND   `scripts/test.sh` contains the parallel integration-fixture assertion exercising Check 26 (e.g. in the zzz-test-scaffold block — a stale catalog makes validate.sh fail there)
```

### Story #5: Committed + declared in registries [integration]

```
GIVEN docs/tests/*.md and docs/test-catalog.md are generated
WHEN  they are committed
THEN  spec/doc-spec-custom.md declares each as a generated human-doc (no work-item IDs)
AND   spec/test-spec-custom.md has units rows for the new test(s)
AND   doc-spec.sh --check-on-disk reports no orphan and no human-doc-id finding for them
AND   test-spec.sh --check-coverage reverse-sweep resolves the new test(s) to exactly one registry row
```

### Story #6: Audit owns freshness end-to-end [integration]

```
GIVEN /CJ_test_audit runs in any repo carrying the renderer
WHEN  Stage 1 executes
THEN  it runs `test-spec.sh --render-docs --check` and reports a freshness finding if the catalog is stale
AND   Stage 3 ground-truth enumeration recognizes docs/tests/ as a generated surface (no false orphan/uncontemplated finding)
```

### Story #7: Hermetic test [core]

```
GIVEN a temp-dir registry fixture
WHEN  tests/test-spec-render.test.sh runs
THEN  it asserts render→render byte-stability, ID-freeness of the output, --check exit 0 on a fresh render, and --check non-zero after a hand-edit
```

## Architecture

```
                      spec/test-spec.md  (general rules — rendered fields source)
                      spec/test-spec-custom.md (units overlay — per-unit rendered fields)
                                   │  merge (existing test-spec.sh parser)
                                   ▼
        ┌──────────────────────────────────────────────────────────┐
        │  scripts/test-spec.sh                                      │
        │    --render-docs        → write docs/tests/<family>.md     │
        │                           + docs/test-catalog.md (index)   │
        │    --render-docs --check→ render to $TMPDIR, diff on-disk, │
        │                           exit 0 fresh / non-zero stale    │
        └──────────────────────────────────────────────────────────┘
            │ (writes)                         ▲ (--check called by)
            ▼                                  │
   docs/tests/validate.md, test.md, ci.md,     ├── scripts/validate.sh Check 26
   hook.md, windows-smoke.md, test-deploy.md,  │      (regenerate→temp→diff→ERROR)
   eval.md  +  docs/test-catalog.md            │
            │                                  └── skills/CJ_test_audit/SKILL.md
            ▼                                        Stage 1 (freshness) + Stage 3
   committed; declared in spec/doc-spec-custom.md          (generated-surface aware)
   as generated human-docs
```

### Data Flow

1. The maintainer (or a generator step) runs `scripts/test-spec.sh --render-docs`.
2. The engine merges `spec/test-spec.md` + `spec/test-spec-custom.md` (existing parser), reads each unit's rendered fields, and groups units by `family`.
3. For each family it writes `docs/tests/<family>.md` (a table of label / purpose / layer / disposition / trigger + the anchor as a code reference) with a stable sort + fixed headers.
4. It writes `docs/test-catalog.md` — the index grouped by family with per-family counts, linking each family page.
5. The maintainer commits the generated docs.
6. `scripts/validate.sh` Check 26 (and `/CJ_test_audit` Stage 1) call `--render-docs --check`: render to a temp dir, diff vs on-disk; non-zero exit → ERROR / finding.
7. `spec/doc-spec-custom.md` declares the docs (no orphan, no IDs); `spec/test-spec-custom.md` declares the new test units (reverse-sweep resolves them).

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Source of the catalog | Generated from the merged registry | Hand-authored `docs/test-catalog.md` | A hand-maintained copy rots and violates the `single-owner` rule; generation keeps `spec/` the single source of truth |
| Fields rendered | Rendered fields only; anchor as a code reference | Render all fields incl. work-item-tagged ones | Rendered fields are already ID-free by the existing lint, so the human-docs pass Check 19 by construction; rendering the anchor as a claim risks ID leakage |
| Freshness owner | One `--render-docs --check` entry point shared by Check 26 + audit Stage 1 | Separate diff logic in validate.sh and in the audit | A single owner means the workbench gate and the portable standalone audit can't disagree |
| Test fixture timing | Add the `scripts/test.sh` fixture in THIS story | Defer the fixture to a follow-up | A new validate check without its parallel test.sh fixture is the recurring implement-subagent blind spot (F000032/34/35); pinning it here closes it |
| `tests-subfolder` presence check | P2 / deferrable | Force it into this story | It is symmetric-nice but widens scope; the core freshness primitive is the story's spine, the subfolder check can follow |
