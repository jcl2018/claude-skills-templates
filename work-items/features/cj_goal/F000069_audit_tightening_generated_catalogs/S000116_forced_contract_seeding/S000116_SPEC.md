## Problem Statement

In a consumer repo that adopts this workbench's audit skills, contract seeding —
the act of delivering `spec/doc-spec.md`, `spec/test-spec.md`, and (since Story 2)
`spec/workflow-spec.md` when they are absent — is neither forced nor reliable. The
operator who runs `/CJ_doc_audit` / `/CJ_test_audit` in their own repo sees the
seeding fail to fire. Two root causes:

1. **Stale-engine shadow (the actual bug).** The audits resolve their engine
   repo-local `scripts/<engine>.sh` → `_cj-shared` with NO staleness check. A
   consumer repo that vendored an OLD `scripts/doc-spec.sh` (or `test-spec.sh` /
   `workflow-spec.sh`) has that stale copy WIN over the current `_cj-shared` one.
   The stale engine lacks the current `--classify`/`--seed`/`--render-docs`
   surface, so the audit's seed step silently no-ops.
2. **Lazy + incomplete seeding.** Even when it fires, seeding is lazy (only on
   the first audit run) and each audit seeds only its OWN contract; the new
   `workflow-spec` is seeded by no audit, and there is no single "force-seed
   everything at adoption" entry point.

This story fixes the stale-engine shadow so the existing seeding actually fires,
and force-generates ALL THREE contracts through every adoption path. The hard
constraint: seeding must NEVER fire on the workbench's own checkout (it AUTHORS
the contracts — re-seeding with skeletons is data loss).

## Mental Model

Two halves — a probe (the bug fix) and three triggers calling one shared routine:

```
PART 1 (the bug fix) — capability probe in BOTH audits' Step-1 engine resolution
   pick repo-local scripts/<engine>.sh
   probe it: `<engine> --classify`  (side-effect-free; current engines emit GENERATION=)
   GENERATION= present  → use repo-local
   GENERATION= ABSENT   → STALE: fall back to _cj-shared + emit stage1/engine-stale
   both stale + _cj-shared unusable → existing stage1/engine unreachable finding

PART 2 (forced seeding) — one routine, three triggers
   do_seed_contracts(target = cwd | --repo):
     workbench self-repo?  → skip (data-loss guard)
     for contract in {doc-spec, test-spec, workflow-spec}:
       resolve engine (repo-local → STALE-PROBE → _cj-shared)
       spec/<contract>.md ABSENT? → engine --seed > $TMP ;
                                     require non-empty AND --validate-clean ;
                                     mv $TMP spec/<contract>.md     (seeded)
       present? → skip                                              (present)
       guard/validate fail? →                                      (seed-failed)
   trigger (a): `skills-deploy seed-contracts [--repo P]`        (explicit)
   trigger (b): `skills-deploy install` from a consumer git repo (forced, no flag)
   trigger (c): `/CJ_doc_audit` Step 2 lazily seeds workflow-spec; test-spec stays
```

The probe makes the lazy path reliable; the shared routine makes all three
triggers consistent; the self-repo skip is the data-loss guard. The seed is
corruption-guarded so a botched `--seed` is a `seed-failed` report, never a
corrupt registry.

## Acceptance Criteria

### Story #1: Stale-engine capability probe (the bug fix) [core]

```
GIVEN a consumer repo carries a stale repo-local scripts/<engine>.sh (lacks --classify)
WHEN  /CJ_doc_audit (doc-spec + workflow-spec engines) or /CJ_test_audit (test-spec engine)
      resolves its Step-1 engine and probes the picked repo-local engine with --classify
THEN  the probe sees no GENERATION= output, treats the repo-local engine as STALE,
      falls back to the _cj-shared engine,
AND   emits `FINDING: stage1/engine-stale — repo-local <engine> is stale (missing --classify);
      using _cj-shared. Remedy: update/remove the vendored scripts/<engine>.sh or re-run
      'skills-deploy install.'`
GIVEN BOTH the repo-local(stale) AND the _cj-shared engine are unusable
WHEN  resolution runs
THEN  the existing stage1/engine unreachable finding is emitted (not a silent no-op)
AND   each skill's error-path grammar / Step 6 report shape documents stage1/engine-stale
```

### Story #2: Shared `do_seed_contracts` routine [core]

```
GIVEN a target repo (cwd or --repo <path>) missing one or more of spec/{doc,test,workflow}-spec.md
WHEN  do_seed_contracts runs
THEN  for each contract it resolves the engine (repo-local → stale-probe → _cj-shared),
AND   for each ABSENT contract it runs the engine's --seed to a temp file, requires the
      output non-empty AND --validate-clean, then mv's it into spec/<contract>.md
AND   each PRESENT contract is skipped (idempotent)
AND   it reports a per-contract line (seeded / present / seed-failed)
GIVEN a --seed that yields empty or --validate-dirty output
WHEN  the corruption guard runs
THEN  nothing is mv'd into spec/ and the contract is reported seed-failed
```

### Story #3: Workbench self-repo skip (data-loss guard) [security]

```
GIVEN the target repo is the workbench self-repo (toplevel == manifest source/bundle_path,
      OR it already carries the canonical non-skeleton contracts)
WHEN  do_seed_contracts runs
THEN  it skips seeding entirely with a note
AND   the workbench's real spec/*.md are left untouched (no skeleton overwrite)
```

### Story #4: `seed-contracts` subcommand [usability]

```
GIVEN the skills-deploy CLI
WHEN  I run `skills-deploy seed-contracts` (or `seed-contracts --repo <path>`)
THEN  it dispatches `shift; do_seed_contracts "$@"`, operates on cwd (or --repo),
      and reports the per-contract result + a summary
AND   a usage line documents the subcommand
```

### Story #5: install always-seeds the consumer path [integration]

```
GIVEN cwd is a consumer git repo that is NOT the workbench self-repo
WHEN  skills-deploy install (or bundle install) completes the skills install
THEN  do_seed_contracts is called on cwd (idempotent, forced — no flag) and a visible
      one-line note reports whether seeding ran
GIVEN cwd is the workbench self-checkout (install==clone, source == bundle_path == cwd)
WHEN  install runs
THEN  seeding is skipped (self-repo) and the contracts are untouched
GIVEN install is run from a non-git directory
WHEN  the seeding hook reaches the git-repo guard
THEN  it is a clean no-op
```

### Story #6: audits seed-all-3 lazily [integration]

```
GIVEN a consumer repo missing spec/workflow-spec.md
WHEN  /CJ_doc_audit Step 2 runs
THEN  it lazily seeds workflow-spec (in addition to doc-spec) using the corruption-guarded
      temp→validate→mv shape, made reliable by the Part-1 stale probe
GIVEN /CJ_test_audit Step 2
WHEN  it runs
THEN  it still seeds test-spec when absent (unchanged behavior)
```

### Story #7: test-deploy coverage + hermetic test [integration]

```
GIVEN the new seeding behavior
WHEN  scripts/test-deploy.sh runs
THEN  it covers `seed-contracts` + the install always-seeds-consumer path (mirroring existing cases)
GIVEN a temp consumer repo
WHEN  tests/<name>.test.sh runs
THEN  (a) `skills-deploy seed-contracts` creates valid spec/{doc,test,workflow}-spec.md,
      a re-run reports all present (no-op), and a workbench-like temp repo is SKIPPED
AND   (b) with a planted stale repo-local engine stub (lacks --classify), the relevant
      resolution falls back to _cj-shared and emits the engine-stale finding
```

### Story #8: declared in the test contract [integration]

```
GIVEN the new test(s) exist
WHEN  spec/test-spec-custom.md is updated with units row(s) for them
THEN  test-spec.sh --validate passes
AND   test-spec.sh --check-coverage reverse-sweep resolves the new test(s) (Check 24 green)
```

### Data Flow

1. Adoption fires via any of three triggers: `skills-deploy seed-contracts`, a consumer `skills-deploy install`, or the first `/CJ_doc_audit` / `/CJ_test_audit` run.
2. The trigger calls `do_seed_contracts(target)` (or the audit Step-2 lazy seed); the routine first checks the workbench self-repo guard and skips entirely if the target is the workbench checkout.
3. For each contract (doc-spec, test-spec, workflow-spec) the routine resolves the engine repo-local → stale-probe (`--classify` → `GENERATION=`) → `_cj-shared`.
4. If `spec/<contract>.md` is absent, the resolved engine's `--seed` writes to a temp file; the routine requires the output non-empty AND `--validate`-clean, then `mv`s it into `spec/<contract>.md` (`seeded`); a present contract is skipped (`present`); a guard/validate failure leaves `spec/` untouched (`seed-failed`).
5. The same engine-resolve + stale-probe runs inside the audits' Step-1 resolution; a stale repo-local engine emits `stage1/engine-stale` and the audit proceeds on `_cj-shared`.
6. `scripts/test-deploy.sh` + `tests/<name>.test.sh` exercise the triggers, the workbench-self skip, and the stale-engine probe; `spec/test-spec-custom.md` units rows make Check 24 reverse-sweep resolve the new test(s).

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| The bug fix | Capability probe (`--classify` → `GENERATION=`) with fallback to `_cj-shared` + an advisory finding | Always prefer `_cj-shared` (drop repo-local resolution); or version-pin the engine | The probe keeps the repo-local-first resolution (a deliberately vendorable engine) while detecting the stale case; an advisory finding names the remedy without halting the audit |
| Proactive seeding scope | Three triggers (explicit subcommand + consumer install + lazy audit), one shared routine | One trigger (only the explicit `seed-contracts`) | The operator chose the maximal/forced combination so every adoption path force-generates the seeding; a single routine keeps them consistent |
| Probe subcommand | `--classify` (side-effect-free) | `--seed` or `--render-docs` | `--classify` is read-only and every current engine emits `GENERATION=`; a mutating probe could write before the staleness verdict |
| Seed safety | Corruption-guarded temp → non-empty + `--validate`-clean → mv | Seed directly into `spec/` | A half-written or invalid `--seed` must never land as a corrupt registry; validate-before-mv makes a botched seed a `seed-failed` report |
| Workbench self skip | Manifest `source == toplevel` AND/OR canonical-contract presence, plus idempotent present-skip + corruption guard as defense-in-depth | Trust idempotent present-skip alone | A false negative would re-seed the workbench's real contracts with skeletons (data loss); the explicit self-repo detection is the primary guard, the others are backstops |
| Lazy workflow-spec owner | `/CJ_doc_audit` Step 2 | `/CJ_test_audit` Step 2, or both | The doc audit's Stage 1 freshness-checks the `docs/workflows/` surface, so it is the natural owner of the `workflow-spec` lazy seed |
