## Problem Statement

After Stories 1-3 a consumer repo HAS the contracts (seeded, reliable — S000116)
and the audits RUN on demand (`/CJ_doc_audit` / `/CJ_test_audit`) — but nothing
FORCES the check. A developer can commit a contract-violating change (a stale
generated catalog, an unregistered `tests/*.test.sh`, a malformed registry) and
not know until someone manually runs an audit. The workbench enforces its own
contract via a `validate.sh` pre-commit hook; a consumer has no equivalent.

This story delivers the "enforced in each repo" half of the original ask: a
deterministic **`scripts/cj-contract-gate.sh`** — the engine-only (Stage-1)
checks, runnable with NO agent — installable as a consumer **pre-commit hook**
(auto on consumer adoption, guarded) plus a documented **CI snippet**, so a
contract violation fails the commit / the PR automatically. It is the FINAL story
of the F000069 epic; on its land the epic is COMPLETE (all 4 stories).

The hard constraints: the gate must be the deterministic engine subset only (a
git hook can't run an agent, so Stage 2/3 are out); `declared-exists` must be SOFT
(a freshly-seeded consumer has the contracts but not yet the declared docs, so
blocking on a missing declared doc would brick adoption); the auto-install must be
guarded (back up a non-workbench hook, skip a custom `core.hooksPath`/husky, skip
the workbench self-repo, no-op on a non-git cwd); and the new test must register
in the test contract (units row + `scripts/test.sh` runner block) or Check 24
hard-fails.

## Mental Model

Two parts — the gate (a pure deterministic script) and its install surfaces
(consumer pre-commit hook, auto + standalone, + a doc-only CI snippet):

```
PART 1 — scripts/cj-contract-gate.sh (the gate; engine-only Stage-1 subset of validate.sh)
   for each engine in {doc-spec, test-spec, workflow-spec}:
       resolve engine  (repo-local → STALE-PROBE `--classify`/GENERATION= → _cj-shared)
   run the DETERMINISTIC checks against cwd (REGISTRY-GATED — absent ⇒ clean SKIP, exit 0):
       doc-spec.sh --check-on-disk             HARD, EXCEPT declared-exists → SOFT REMEDIATION
                                               (note: run /CJ_document-release — NOT a block)
       test-spec.sh --validate                 HARD
       test-spec.sh --check-coverage           HARD (rules-only ⇒ "inactive", not a finding)
       workflow-spec.sh --validate             HARD (no-vanish / registry-completeness)
       test-spec.sh --render-docs --check      HARD freshness WHEN a generated surface exists
       workflow-spec.sh --render-docs --check  HARD freshness WHEN a generated surface exists
   compact per-check summary: PASS / FINDING / REMEDIATION / SKIP   (`--quiet` ⇒ summary only on failure)
   EXIT non-zero IFF any HARD check finds a violation

PART 2 — install surfaces (consumer pre-commit hook + standalone command + CI snippet)
   ship the gate:   add cj-contract-gate.sh to the shared-scripts deploy set → _cj-shared
   auto-install:    do_install consumer-path block ALSO installs the gate pre-commit hook
                    (reuse setup-hooks.sh install_hook safety: sentinel-aware idempotent;
                     back up a non-workbench hook + WARN; SKIP custom core.hooksPath/husky;
                     workbench-self SKIP — it runs validate.sh; non-git cwd no-op)
   standalone:      skills-deploy install-contract-gate   (install on cwd)
                    skills-deploy install-contract-gate --remove   (uninstall the sentinel hook)
   CI (doc-only):   a GitHub Actions copy-paste snippet that runs the gate on PRs
```

The gate is pure + deterministic (an agent-free engine subset); the hook body
resolves it from `_cj-shared` at commit time and blocks on non-zero; the install
is guarded so it never bricks a fresh adopter or clobbers a custom hook.

## Acceptance Criteria

### Story #1: Deterministic engine-only gate [core]

```
GIVEN a repo with the contracts present (doc-spec / test-spec / workflow-spec)
WHEN  scripts/cj-contract-gate.sh runs against cwd
THEN  it resolves each engine (repo-local → stale-probe `--classify` → _cj-shared)
AND   runs the deterministic checks: doc-spec.sh --check-on-disk, test-spec.sh
      --validate + --check-coverage, workflow-spec.sh --validate, and the two
      --render-docs --check freshness checks
AND   prints a compact per-check summary (PASS / FINDING / REMEDIATION / SKIP)
AND   runs ONLY the engine-only (deterministic) checks — NOT the agent-judged Stage 2/3
AND   `--quiet` reduces the output to a summary-only-on-failure form for hook use
```

### Story #2: HARD/SOFT/SKIP disposition + registry-gating [core]

```
GIVEN cj-contract-gate.sh runs against a repo
WHEN  a declared doc is missing (doc-spec.sh --check-on-disk declared-exists)
THEN  the gate prints a SOFT REMEDIATION note pointing at /CJ_document-release and
      does NOT block (exit 0 for that disposition)
WHEN  any OTHER doc-spec check fails, or test-spec.sh --validate/--check-coverage
      finds a violation, or workflow-spec.sh --validate finds a no-vanish gap, or a
      --render-docs --check freshness check fails (a generated surface exists)
THEN  that is a HARD finding and the gate exits non-zero
WHEN  test-spec is rules-only (no units rows)
THEN  --check-coverage reports "inactive", which is NOT a finding
WHEN  a contract is absent (REGISTRY=absent)
THEN  that check is a clean SKIP (exit 0 for that check) — registry-gated throughout
AND   the gate exits non-zero IFF at least one HARD check found a violation
```

### Story #3: install ships the gate to `_cj-shared` [integration]

```
GIVEN scripts/cj-contract-gate.sh is added to the shared-scripts deploy set
WHEN  skills-deploy install runs
THEN  cj-contract-gate.sh is deployed into _cj-shared/scripts/ (alongside the engines)
AND   the consumer pre-commit hook body resolves it from
      ${CJ_SHARED_SCRIPTS:-$HOME/.claude/_cj-shared/scripts}/cj-contract-gate.sh
```

### Story #4: consumer install auto-installs the gate hook, guarded [integration]

```
GIVEN cwd is a consumer git repo that is NOT the workbench self-repo
WHEN  skills-deploy install completes (the consumer-path block, alongside the
      Story-3 seeding hook)
THEN  it installs a pre-commit hook (carrying the setup-hooks SENTINEL) whose body
      resolves cj-contract-gate.sh from _cj-shared and runs it; non-zero blocks the commit
AND   re-running install is idempotent (sentinel-aware — no duplicate install)
GIVEN cwd already has a NON-workbench pre-commit hook
WHEN  the auto-install runs
THEN  it backs the existing hook up and emits a WARN — it does NOT clobber it
GIVEN cwd has a custom core.hooksPath set (husky-like)
WHEN  the auto-install runs
THEN  it SKIPS installing with a printed note (does not fight the custom hooks path)
GIVEN cwd is the workbench self-repo (is_workbench_self_repo)
WHEN  install runs
THEN  the gate-hook install is SKIPPED (the workbench runs validate.sh — no double-enforcement)
GIVEN install is run from a non-git directory
WHEN  the auto-install reaches the git-repo guard
THEN  it is a clean no-op
```

### Story #5: standalone install-contract-gate command [usability]

```
GIVEN the skills-deploy CLI
WHEN  I run `skills-deploy install-contract-gate`
THEN  it installs the gate pre-commit hook on cwd (same guarded install_hook safety)
WHEN  I run `skills-deploy install-contract-gate --remove`
```

```
THEN  it covers: consumer install installs the sentinel gate hook; husky/custom

GIVEN a temp consumer repo
WHEN  tests/<name>.test.sh runs
THEN  (a) cj-contract-gate.sh PASSes on a clean contract, hard-FAILS on a planted
      violation (stale generated catalog / malformed registry / unregistered
      tests/*.test.sh), treats a missing declared doc as a SOFT remediation (exit 0,
      not a block), and SKIPS a registry-absent contract
AND   (b) the consumer auto-install installs a sentinel pre-commit hook in a temp git
```

### Data Flow

1. `skills-deploy install` deploys `cj-contract-gate.sh` into `_cj-shared/scripts/` (it is in the shared-scripts deploy set).
2. On a consumer install (cwd is a git repo AND not the workbench self-repo), `do_install`'s consumer-path block — alongside the Story-3 seeding — installs a pre-commit hook (carrying the `setup-hooks` SENTINEL) whose body resolves the gate from `_cj-shared` and runs it; the install is guarded (back up a non-workbench hook + WARN; SKIP a custom `core.hooksPath`/husky with a note; workbench-self SKIP; non-git no-op).
3. At `git commit` time the hook runs `cj-contract-gate.sh`: it resolves each engine (repo-local → stale-`--classify`-probe → `_cj-shared`) and runs the deterministic checks against cwd. `declared-exists` is a SOFT remediation; an absent contract is a clean SKIP; every other check is HARD. The gate exits non-zero iff any HARD check finds a violation — blocking the commit.
4. A standalone `skills-deploy install-contract-gate` (install) / `--remove` (uninstall) lets a consumer explicitly opt in/out.
5. A documented CI snippet runs the same gate on PRs (server-side equivalent; doc-only, no workflow file shipped).
6. `scripts/test-deploy.sh` + `tests/cj-contract-gate.test.sh` exercise the gate's dispositions, the guarded install, the skips, and the remove path; `spec/test-spec-custom.md` units row + the `scripts/test.sh` runner block make Check 24 resolve the new test.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Enforcement layer | Engine-only deterministic gate (Stage-1 subset of `validate.sh`) installable as a pre-commit hook / CI step | Run the full audit (Stage 1+2+3) in the hook | A git hook / CI step can't run an agent; the deterministic engine checks are the enforceable subset. Stage 2/3 stay on-demand via `/CJ_doc_audit` / `/CJ_test_audit`. |
| `declared-exists` disposition | SOFT — a REMEDIATION note pointing at `/CJ_document-release`, not a block | HARD block on a missing declared doc | A freshly-seeded consumer (Story 3) has the contracts but not yet the declared docs; a hard block would brick its next commit. Soft keeps adoption unbricked while still surfacing the gap. |
| Install trigger | Auto on consumer install (guarded) + a standalone `install-contract-gate` command | Standalone-only (no auto-install) | Symmetric with Story 3's forced seeding — every adoption path enforces. The guards (back-up, husky-skip, self-skip, non-git no-op, `--remove`) make the auto-install safe. |
| Engine resolution | Repo-local → Story-3 stale-`--classify`-probe → `_cj-shared` | Repo-local → `_cj-shared` (no stale check) | A consumer with a stale vendored engine would otherwise mis-gate; reusing the proven Story-3 probe makes resolution reliable. |
| Hook-install safety | Reuse `setup-hooks.sh`'s `install_hook` (sentinel-aware, back-up, husky-skip), shared helper preferred | Write a fresh hook installer in `skills-deploy` | One safe implementation beats two drifting copies; if extraction is risky, mirror the exact behavior + a parity test. |
| CI surface | A documented copy-paste GitHub Actions snippet | Ship a `.github/workflows/*.yml` into consumers | A shipped workflow file is intrusive + repo-specific; a snippet lets a team opt in on their own terms. |
