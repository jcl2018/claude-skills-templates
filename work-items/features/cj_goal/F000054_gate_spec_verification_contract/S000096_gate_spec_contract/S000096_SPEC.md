## Problem Statement

The workbench verifies a cj_goal change at four layers that grew up independently
— (1) the local pre-commit hook (`validate.sh`, hard-fail), (2) GitHub Actions CI
(`validate.yml` / `windows.yml` gating, `eval-nightly.yml` non-gating), (3)
in-orchestrator pipeline gates (isolation / design-summary / QA / doc-sync /
portability / ship), and (4) regression ratchets (Check 8 VERSION-never-regresses,
the portability FINDINGS=0 baseline, Check 14 USAGE.md freshness). The user — the
cj_goal framework maintainer, or any reviewer of a resumed run — cannot answer one
basic question from a single place: **"What stops a broken cj_goal change from
landing, and at which layer?"**

Three concrete symptoms: the verification story is scattered across three docs
(`philosophy.md §4` = the why, `architecture.md` = a section mislabeled "The CI
gate" that actually only covers the doc-spec checks, `workflow.md` = individual
gate scripts); the word "gate" is overloaded (an inline orchestrator halt AND
`validate.sh`-as-a-whole); and the contract is duplicated per orchestrator with
ad-hoc per-story `test.sh` regression guards, with nothing keeping the four
pipelines in sync.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group.
     Each #n header should match the # column in the Requirements tables. -->

### Story #1: One human-readable verification map [usability]

```
GIVEN the workbench verifies a cj_goal change at four independent layers with no single map
WHEN  gate-spec.md is authored at repo root
THEN  it is a doc-spec.md-style file (prose intro + a summary table of the four
      layers local-hook|ci|pipeline-gate|ratchet + an ASCII diagram + a
      division-of-labor assigning each guarantee to exactly ONE owning layer +
      a fenced `yaml` registry), and a reader who has NOT seen the design can
      answer "what stops a broken cj_goal change from landing, and at which
      layer?" in under a minute reading only that file
```

### Story #2: Machine-readable registry + a doc-spec.sh-style reader [core]

```
GIVEN gate-spec.md carries a fenced `yaml` registry (schema_version: 1; layers
      with id/name/trigger/disposition; gates with id/layer/order/markers/
      disposition/backing; markers a per-mode map whose values are a "[...]"
      literal or {enforced_by: subagent|auq})
WHEN  scripts/gate-spec.sh runs
THEN  --validate exits 0 + `OK schema_version=<n>` on the committed registry (and
      exit 1 + `[gate-spec-no-config] <reason>` on a malformed one), --list-layers
      emits every declared layer id, and --list-gates emits every gate id (sorted,
      unique); it reads gate-spec.md via `git rev-parse --show-toplevel`
```

### Story #3: Advisory cross-orchestrator conformance check [observability]

```
GIVEN the registry and the four live CJ_goal_* pipelines
WHEN  scripts/validate.sh runs Check 22 (structurally a clone of Check 21)
THEN  it asserts gate-spec.sh --validate exits 0 AND, for every gate, for every
      mode key in its `markers` map: a literal "[marker]" must appear in at least
      one of that mode's files (skills/CJ_goal_<mode>/pipeline.md OR SKILL.md;
      todo dir = CJ_goal_todo_fix), while an {enforced_by: ...} value is skipped;
      it is GREEN on the clean tree and REPORTS a finding when a declared literal
      marker is removed from the registry or from both of its mode's files. It is
      ADVISORY in v1 (a finding prints but exit code stays 0 — exactly like Check
      21), and the parallel scripts/test.sh zzz-test-scaffold fixture ships in the
      SAME PR
```

### Story #4: "Gate" disambiguated + docs wired coherently [usability]

```
GIVEN the verification story is scattered and "gate" is overloaded
WHEN  the docs are wired to gate-spec.md
THEN  architecture.md gains a "The gate-spec.md contract" section AND the
      mislabeled "The CI gate (scripts/validate.sh)" heading is renamed/reframed
      to name what it actually covers; philosophy.md §4 gains a one-line pointer;
      the four skills/CJ_goal_* pipeline/SKILL files + CLAUDE.md gain a one-line
      canonical-gate-sequence reference; and gate-spec.md is registered in
      doc-spec.md (section: custom, audit_class: operational) so Check 17 passes
```

### Data Flow

<!-- How does data move through the system for the primary use case?
     Step-by-step, component to component. -->

1. `gate-spec.md` declares the four layers and each gate as a registry row `{id, layer, order, markers (per-mode map), disposition, backing, checks}` in its fenced `yaml` block.
2. `scripts/gate-spec.sh` parses the block (located via `git rev-parse --show-toplevel`) and exposes `--validate` (shape-checks schema_version + every gate's required keys + closed enums + each `markers` value is a `"[...]"` literal or `{enforced_by: subagent|auq}`), `--list-layers`, and `--list-gates`.
3. The four `CJ_goal_*` pipeline/SKILL files carry a one-line reference to `gate-spec.md` as the canonical sequence (the de-duplication — local prose stays, the source of truth is named once).
4. `scripts/validate.sh` Check 22 runs the conformance guard: it confirms `gate-spec.sh --validate` exits 0, then for every gate × every mode key in `markers` greps each literal `"[marker]"` in `skills/CJ_goal_<mode>/{pipeline.md,SKILL.md}` (either file), skipping `{enforced_by}` values; a missing declared literal marker is a finding. Advisory — exit 0.
5. `scripts/test.sh`'s zzz-test-scaffold fixture exercises the parser + the advisory check (including the negative drift path) in the same PR.
6. `architecture.md` / `philosophy.md §4` / `doc-spec.md` / `CLAUDE.md` map the human reader to the one contract and disambiguate "gate."

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale.
     Future readers need to know why this path over the others. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Artifact shape | doc-spec.md-style file (prose map + fenced `yaml` registry) + a `scripts/` reader + an advisory `validate.sh` check | A registry embedded in `cj-goal-common.sh` (`--list-gates`); or grep-only conformance with no reader script | Truest mirror of the in-repo idiom (doc-spec.md, permission-policy.md) — legibility-by-symmetry. Bash-embedded data reads worse as "the single map"; grep-only loses the programmatically-consumable reader. |
| Marker schema | per-mode `markers` map + an `{enforced_by: subagent\|auq}` escape hatch | A flat `applies_to: [...]` + a single `halt_marker` string | The real markers are irregular: isolation has three markers + todo has none; only `[portability-red]` + `[doc-sync-red]` are universal. A flat schema cannot model that; the map makes "in sync" honestly enforceable, and the escape hatch keeps the baseline clean. |
| Check disposition | advisory-first (exit 0), mirroring Check 21 | hard-ERROR from day one | The immediately-preceding sibling check shipped advisory; consistency is a legibility win, and the honest registry is GREEN today, so flip-to-strict is a one-line follow-up (Check 21 ratchet precedent). A hard error on a hand-authored registry's first cut is risky. |
| Reader API surface | `--validate` / `--list-layers` / `--list-gates` only | also ship `--seed` + `--list-for <mode>` | The review flagged `--seed` + `--list-for` as speculative API with no v1 caller (the check computes the per-mode subset internally; no skill recreates a missing gate-spec.md). Add either when a real consumer appears. |
