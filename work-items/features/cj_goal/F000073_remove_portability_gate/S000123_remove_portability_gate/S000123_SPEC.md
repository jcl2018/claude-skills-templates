## Problem Statement

The four `CJ_goal_*` orchestrators (feature, task, defect, todo_fix) run a pre-ship
portability gate — `cj-goal-common.sh --phase portability-audit` (F000051 /
S000091) — that runs `scripts/cj-portability-audit.sh` STRICT and HALTs
(`[portability-red]` / `halted_at_portability`) on a dishonest `portability`
declaration. This is workbench-only logic (it audits `skills-catalog.json`, which
only this repo ships) living inside otherwise-portable orchestrators. The operator
(a workbench maintainer) wants the gate OUT of the cj_goal build path, leaving
portability enforced solely by the separate test (Check 18 + `/CJ_portability-audit`).

## Mental Model

Portability enforcement has two layers today: (1) the SEPARATE test — `validate.sh`
Check 18 (strict-by-default global ratchet, T000054) + the standalone
`/CJ_portability-audit` skill + `scripts/cj-portability-audit.sh` engine — and (2)
the redundant cj_goal build-path GATE removed by this story. Removing layer (2)
leaves layer (1) fully intact; a cj_goal build commits ≥ twice (pre-doc-sync +
`/ship`), each firing the pre-commit hook → Check 18, so the guarantee holds.

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. -->

### Story #1: Script mechanism removed [core]

```
GIVEN scripts/cj-goal-common.sh currently has the --phase portability-audit block,
      the resolve_portability_engine() helper, and portability-audit in the phase
      enum + usage string + header/phase-list comments
WHEN  the change is applied
THEN  none of those remain, and `grep -n "portability" scripts/cj-goal-common.sh`
      returns nothing gate-related
```

### Story #2: Orchestrators carry zero portability wiring [core]

```
GIVEN the four CJ_goal_* orchestrators call the Step 5.7 (defect: same-numbered)
      portability gate, declare [portability-red]/halted_at_portability, surface a
      ### Portability PR-body line, write .cj-goal-feature/portability-verdict.md,
      and show a portability node in their overview chains
WHEN  the change is applied
THEN  `grep -rn "phase portability-audit\|portability-red\|halted_at_portability"`
      over the four skills/CJ_goal_*/ dirs returns nothing, and /ship follows the
      QA-audit checkpoint directly (the registered-doc verdict splice is KEPT,
      only the ### Portability line drops)
```

### Story #3: Contract registries surgically updated [integration]

```
GIVEN spec/test-spec-custom.md has the portability gates: row (order 60), the
      cj_goal-gate units: rows, and the ratchet row; spec/workflow-spec.md has the
      portability node + verdict wording in all four charts/Touches and
      portability-audit in the cj-goal-common phase list
WHEN  the change is applied
THEN  those rows/nodes are removed, the goal-common phase-integration unit row no
      longer enumerates portability-audit, the Check 18 unit row + engine unit row
      are KEPT, `bash scripts/workflow-spec.sh --render-docs` regenerates
      docs/workflow.md + docs/workflows/*.md, and validate.sh Checks 24 + 27 pass
```

### Story #4: Tests updated without losing enum coverage [resilience]

```
GIVEN tests/cj-goal-common-portability.test.sh exists; scripts/test.sh has the
      Step 6b (F000051/S000091) integration block (~822-848), the deleted-test

      --phase portability-audit --mode task --dry-run; the F000047/S000083 engine
      fixture block (~2762) exists

THEN  the gate test is deleted, the integration block + runner line are removed,
      the task-enum probe is repointed to a surviving mode-agnostic phase
      (--phase recap --mode task OR --phase sync --mode task --dry-run), any
      enumerated-phase assertions in tests/test-spec.test.sh /
      tests/cj-goal-common-sync.test.sh are updated, the engine fixture block is
```

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Extent of removal | A: Full extraction (delete the mechanism) | B: Minimal unwire (keep phase dormant) | B leaves dead code + workbench-specific logic in `cj-goal-common.sh`; only half-honors "not inside that skill." |
| Extent of removal | A: Full extraction | C: Soft-remove (advisory, never halts) | C still RUNS inside the orchestrators, retains workbench-specific logic, and adds behavior instead of removing it — worst fit for the goal. |
| `task`-enum probe in `test.sh` | Repoint to a surviving mode-agnostic phase | Delete the probe outright | The probe proves the `task` enum is accepted; deleting it loses that coverage. |
