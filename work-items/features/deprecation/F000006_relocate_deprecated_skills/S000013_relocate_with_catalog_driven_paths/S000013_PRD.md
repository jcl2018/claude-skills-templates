## Problem Statement
<!-- What problem does this feature solve? Who has this problem?
     Be specific: name the user, their role, and the pain they experience today. -->

A workbench maintainer (chjiang) opens `skills/` and sees three subdirectories: `company-workflow/` (deprecated), `personal-workflow/` (active), `system-health/` (active). The deprecated entry is visually indistinguishable from the active ones — `status: deprecated` lives only in the catalog JSON, not on the filesystem. The folder tree reads as "deployable skills + one deprecated holdout" instead of "deployable skills only." This makes `skills/` a less useful catalog and creates a low-grade "did we finish the deprecation work?" feeling every time the directory is opened.

The fix moves `company-workflow/` source out of `skills/` into a new top-level `deprecated/` directory. The path itself names the lifecycle state. Consumer scripts (`skills-deploy`, `validate.sh`) are refactored to derive paths from `skills-catalog.json` instead of hardcoding `skills/{name}/`, so the move is a catalog change, not a script-rewrite.

## Acceptance Criteria

<!-- Given/When/Then format. One block per user story or logical group. -->

### Story #1: skills/ contains only deployable skills [usability]

```
GIVEN F000006 has shipped
WHEN  I run `ls skills/`
THEN  I see exactly two entries: `personal-workflow/` and `system-health/`
AND   no `company-workflow/` entry under skills/
```

### Story #2: Mirror invariant holds post-move [core]

```
GIVEN MIRROR_SPECS has been retargeted to deprecated/company-workflow/...
WHEN  I run `./scripts/validate.sh`
THEN  Error check 10 reports PASS for all 7 mirror entries
AND   each entry's source-vs-destination cmp -s is byte-identical
AND   no FAIL or WARN line appears for the mirror check
```

### Story #3: skills-deploy lifecycle unchanged [core]

```
GIVEN F000006 has shipped on a fresh SKILLS_DEPLOY_TARGET (no prior install)
WHEN  I run `scripts/skills-deploy install`
THEN  ~/.claude/skills/company-workflow/ is NOT created
AND   exactly one WARN line is printed for company-workflow

WHEN  I run `scripts/skills-deploy install --include-deprecated`
THEN  ~/.claude/skills/company-workflow/ is created
AND   the install manifest's path field reflects deprecated/company-workflow/SKILL.md (not skills/company-workflow/SKILL.md)

WHEN  I run `scripts/skills-deploy doctor`
THEN  company-workflow is reported under INFO (deprecated, not installed by default)
AND   no WARN line appears for it
```

### Story #4: Catalog-driven path resolution [core]

```
GIVEN scripts/skills-deploy and scripts/validate.sh have been refactored
WHEN  I grep for "skills/company-workflow" in scripts/skills-deploy
THEN  zero matches (paths now derived from skills-catalog.json files[])

WHEN  I grep for "skills/$name/SKILL.md" in scripts/validate.sh line 30
THEN  the hardcoded path has been replaced with a catalog lookup
```

### Story #5: Test suite passes with no behavioral changes [core]

```
GIVEN the relocation has been applied
WHEN  I run `./scripts/test.sh`

AND   Failures count is 0
AND   no test was skipped or modified to accommodate the move (only path constants introduced)
```

## Out of Scope

<!-- Explicitly list what this feature does NOT do. Prevents scope creep. -->

- Removing `skills/company-workflow/` entirely from the repo (deletion). The same byte-mirror constraint that drove F000005's deletion-out-of-scope still applies. We are *moving*, not deleting.
- Refactoring `personal-workflow/` or `system-health/` to use catalog-driven path lookup. Both are active and live at `skills/{name}/`. The catalog-driven refactor is principled (works for any path) but only `company-workflow` exercises the new behavior today.
- Eliminating the `templates/` top-level directory. `personal-workflow/` templates and `doc-SKILL-DESIGN.md` still live there.
- Changing SKILL.md frontmatter (description, allowed-tools, etc.). F000005's same out-of-scope applies — catalog `status: deprecated` is the canonical signal.
- Auto-uninstalling pre-existing `~/.claude/skills/company-workflow/` installs. Users own removal via `skills-deploy remove`.

## Assumptions

<!-- What this PRD takes for granted. If an assumption is wrong, revisit the PRD. -->

- `scripts/generate-readme.sh` and `scripts/copilot-deploy.py` are catalog-driven (preliminary grep showed zero hardcoded `skills/company-workflow` references). If implementation reveals hidden hardcoded paths in either, scope creeps slightly to cover them.
- `skills-catalog.json` `templates[]` entries are filesystem-relative paths under `templates/` (e.g., `"company-workflow/tracker-feature.md"` resolves to `templates/company-workflow/tracker-feature.md`). If they are catalog keys with a different resolver mechanism, ARCHITECTURE adjusts accordingly.
- `git mv` preserves blame history through the relocation. Verified during implementation.
- The work-copilot bundle's destination paths under `work-copilot/templates/`, `work-copilot/reference/`, etc. are stable. We retarget MIRROR_SPECS source paths only.
- No external consumer (outside this repo) references `skills/company-workflow/` paths. If a downstream user has hardcoded `~/.claude/skills/company-workflow/` references in their workflow, this feature does not break that — `--include-deprecated` still installs to the same destination path.
