## Problem Statement

A workbench maintainer cannot today tell whether a skill's declared
`portability` field is honest. Skills declared `standalone` reach for repo-local
artifacts (`scripts/test.sh`, `scripts/cj-goal-common.sh`, root config,
`CLAUDE.md`, the `.source` reach-back) that a fresh target repo will not have, so
they break at runtime in another repo despite being "installed." There is no
deterministic check that compares the declared field against actual dependencies.
This story gives the maintainer that check — a static lint engine surfaced both as
a `/CJ_portability-audit` skill report and an always-on `validate.sh` advisory
check — turning the honor-system label into a verified invariant.

## Mental Model

Two surfaces, one engine. The engine `scripts/cj-portability-audit.sh` is pure
static analysis: for each catalog skill it collects the skill's files, finds each
**executed** repo-local dependency, and classifies it against the skill's declared
`portability` tier using a strict ladder. The classification has three carve-outs
(bundled-own-script, scoped self-resolution-preamble, `portability_requires`
adjudication) and emits one of three verdicts per skill. Surface 1 = the skill
(rich table). Surface 2 = a `validate.sh` advisory check (exit 0). A separate
opt-in Layer-2 dynamic surface exists in v1 only as a single proof-of-life case.

```
                         scripts/cj-portability-audit.sh   (the one engine)
                                       │
        ┌──────────────────────────────┼───────────────────────────────┐
        ▼                               ▼                               ▼
 /CJ_portability-audit          validate.sh advisory            scripts/eval.sh --portability
   skill report                 check (exit 0 in v1)            (ONE CJ_suggest case, v1 proof-of-life)
```

## Acceptance Criteria

<!-- Given/When/Then. Each #n header matches the # column in Requirements. -->

### Story #1: Runtime-derived skill set [core]

```
GIVEN skills-catalog.json with N entries (some deprecated, one tooling-only `templates` with empty files)
WHEN  scripts/cj-portability-audit.sh runs
THEN  it audits exactly the entries matching `status != "deprecated"` AND non-empty `files`,
      with no hardcoded skill name or count in the engine source
```

### Story #2: EXECUTED-vs-documented precision [core]

```
GIVEN a SKILL.md that MENTIONS scripts/test.sh in a prose sentence/table AND a SKILL.md that EXECUTES it inside a ```bash fence (e.g. `bash "$ROOT/scripts/test.sh"`)
WHEN  the engine scans both
THEN  the prose mention is recorded as an informational note (NOT a finding),
      and only the executed reference is a candidate finding
```

### Story #3: Strict tier ladder classification [core]

```
GIVEN a skill declared `standalone` whose SKILL.md executes a ROOT scripts/cj-goal-common.sh
WHEN  the engine classifies the dependency
THEN  it emits FINDING "<skill> declared standalone but depends on scripts/cj-goal-common.sh (needs workbench)";
      AND a skill declared `workbench` referencing the same root script is within-tier → OK;
      AND a skill whose `portability` value is not in {standalone, local-only, workbench} is itself a finding
```

### Story #4: Carve-outs (bundled-own + scoped self-resolution preamble) [core]

```
GIVEN CJ_suggest references its OWN skills/CJ_suggest/scripts/suggest.sh,
  AND a `standalone`-declared orchestrator's SKILL.md contains the self-resolution preamble (`.source` reach-back to a ROOT script),
  AND a `workbench`-declared skill contains the same preamble
WHEN  the engine classifies each
THEN  the bundled-own script is OK (never a finding);
      the self-resolution preamble is a FINDING for the `standalone` orchestrator;
      the self-resolution preamble is OK-with-note for the `workbench` skill
```

### Story #5: portability_requires honoring [core]

```
GIVEN CJ_qa-work-item executes scripts/test.sh AND its catalog entry lists `portability_requires: ["scripts/test.sh"]`,
  AND a second skill lists a portability_requires entry it no longer references
WHEN  the engine runs
THEN  CJ_qa-work-item's scripts/test.sh dep is OK (adjudicated, not a finding);
      the stale entry produces an informational note ("portability_requires entry 'X' no longer referenced"), never a finding
```

### Story #6: Three-value verdict [observability]

```
GIVEN the audited skill set
WHEN  the engine finishes
THEN  each skill has exactly one verdict in {portable, portable-with-notes, findings:<list>},
      and every `findings:` entry names the skill + the executed repo-local dep + why (declared level vs actual)
```

### Story #7: validate.sh advisory check (exit 0) [integration]

```
GIVEN the workbench with the CJ_portability-audit catalog entry + pre-seeded portability_requires
WHEN  ./scripts/validate.sh runs (default, no PORTABILITY_STRICT)
THEN  the new check prints the portability findings AND validate.sh exits 0
      (the check is advisory; PORTABILITY_STRICT=1 is documented as the future hard-fail path)
```

### Story #8: test.sh integration fixture [resilience]

```
GIVEN the zzz-test-scaffold integration pattern in scripts/test.sh
WHEN  ./scripts/test.sh runs
THEN  a fixture exercises scripts/cj-portability-audit.sh (e.g. a synthetic catalog/skill where a standalone skill
      executes a root helper) and asserts the engine emits the expected FINDING — proving the check + the parallel test.sh edit exist
```

### Story #9: Registration + green-by-adjudication [integration]

```
GIVEN the new catalog entry (portability: workbench, status: experimental) and the pre-seeded portability_requires on the current mismatches
WHEN  ./scripts/validate.sh and ./scripts/test.sh run
THEN  both are green; the catalog shows each adjudicated dep explicitly; no new hard failure is introduced
```

### Story #10: Documentation to convention [usability]

```
GIVEN the new skill
WHEN  validate.sh runs its doc-presence + New-skills no-vanish checks
THEN  SKILL.md + USAGE.md (5 H2 sections) exist; the correct-behavior spec section is in doc/WORKFLOWS.md;
      a roster line is in doc/ARCHITECTURE.md; a decision-tree entry is in doc/PHILOSOPHY.md; all checks pass
```

### Story #11: Layer-2 proof-of-life (one CJ_suggest case) [integration]

```
GIVEN a stripped scratch repo (no workbench scripts/CLAUDE.md/config/work-items) + a scratch ~/.claude whose .skills-templates.json .source points at the stripped repo (or is unset)
WHEN  scripts/eval.sh --portability runs the single CJ_suggest case with CJ_suggest's own allowed-tools passed through
THEN  the case runs locally green (CJ_suggest degrades gracefully), proving the --portability mode + fixture-prep helper exist;
      broad coverage + nightly CI remain out of scope (Story 2)
```

### Story #12: Predicted first-run findings [observability]

```
GIVEN today's workbench BEFORE the portability_requires pre-seed
WHEN  scripts/cj-portability-audit.sh runs
THEN  CJ_goal_feature, CJ_goal_defect, CJ_goal_todo_fix, CJ_qa-work-item, CJ_implement-from-spec all appear with findings
      (declared standalone, reach root helpers / .source), demonstrating the audit is non-no-op
```

### Data Flow

<!-- How data moves through the system for the primary use case. -->

1. A surface (skill / validate.sh / eval.sh) invokes `scripts/cj-portability-audit.sh`.
2. The engine reads `skills-catalog.json` and selects the audit set via `status != "deprecated"` + non-empty `files` (jq).
3. For each selected skill it collects files: catalog `files` entries + the skill dir's `*.md` + any `scripts/*.sh` it references.
4. It greps for repo-local dependency signatures: root `scripts/*.sh` (basenames globbed at runtime — NOT hardcoded), root config files (`skills-catalog.json`/`cj-document-release.json`/`template-registry.json`/`VERSION`), `CLAUDE.md` reads, the GitHub slug, `work-items/` path assumptions.
5. Each hit is classified EXECUTED (runnable position) vs DOCUMENTED (prose/table/comment). DOCUMENTED → informational note.
6. Each EXECUTED hit is classified against the skill's declared tier; carve-outs applied (bundled-own → OK; self-resolution preamble → OK-with-note for workbench/local-only, FINDING for standalone); `portability_requires`-listed dep → OK; stale listed dep → note; dep exceeding tier → FINDING.
7. The engine emits a per-skill verdict (`portable` / `portable-with-notes` / `findings:<list>`).
8. The skill renders the table; validate.sh prints findings + exits 0; eval.sh's one case drives `CJ_suggest` against the stripped scratch repo.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Surfacing posture | Advisory (exit 0 in v1) | Hard `validate.sh` gate | The workbench HAS real mismatches today; a hard gate red-CIs the merge. Mirrors the registered-doc audit posture. Hard-gate is a documented Story-2 follow-up. |
| First-run noise control | `portability_requires` accepted-deps field shipped in v1 | Ship without it; tolerate an all-red first run | Without adjudication the table is unusable noise. The field is a v1 correctness prerequisite, not polish; mirrors `doc_requirement`. |
| Root-helper detection | Glob `scripts/*.sh` basenames at runtime | Hardcode a known-helper list | A hardcoded list is the exact baked-in-workbench-specifics rot this skill catches; the repo adds root helpers often. |
| Self-resolution preamble for `standalone` | Treat as a FINDING | Auto-OK-with-note for all skills (earlier draft) | The `.source` reach-back IS a workbench dependency — for a `standalone` skill it contradicts the claim. This is the single most valuable finding class (D4 correction). |
| Layer-2 scope in v1 | ONE `CJ_suggest` proof-of-life case | Broad coverage + nightly CI in v1 | Broad Layer-2 re-imports the parked-eval-harness cost/flake (D000023). v1 proves the mode exists; breadth is Story 2. |
| Engine placement | ROOT script resolved via `.source` | Bundle under `skills/CJ_portability-audit/scripts/` (deployed) | Bundling buys no portability — the catalog + skill source the engine reads aren't deployed either. Matches `cj-repo-init.sh`. |
