## Problem Statement

An operator deciding what to work on runs `/CJ_suggest` and gets a markdown
table (`Rank | Title | Pri | Size | Status | Why`). The title is terse, `Size`
is a bare `S/M/L` letter, and no row says in plain language what the work
*does*. The operator must open `TODOS.md` and read each body before choosing.
The data needed to fix this — body prose and the Size letter — is already in
hand at render time; only the presentation withholds it.

## Mental Model

`suggest.sh` runs: collect candidates → score into a tempfile (`$SCORED`,
which already carries the raw heading in col 7) → render top-N. This story
adds a single render fork at the last step:

```
                          ┌─ $FOR_SKILL empty ──► render_cards (NEW)
score rows → top-N → fork ┤
                          └─ $FOR_SKILL set ────► render_table (UNCHANGED)
```

Everything before the fork is untouched.

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. -->

### Story #1: Interactive card render [usability]

```
GIVEN TODOS.md has actionable rows
WHEN  the operator runs `bash skills/CJ_suggest/scripts/suggest.sh` with no flags
THEN  each ranked item prints as a card: header `N. [ID] Title   Pri · Effort`,
      a wrapped `What:` line from the body's first non-empty line, and a
      `Status:` line folding in the existing `Why` reasons
```

### Story #2: Byte-stable consumer table [integration]

```
GIVEN the same TODOS.md
WHEN  `bash skills/CJ_suggest/scripts/suggest.sh --for-skill cj-goal --limit 15` runs
THEN  the output is byte-identical to today's markdown table (title in column 2),
      so /CJ_goal_todo_fix still parses candidates
```

### Story #3: Effort labels [usability]

```
GIVEN a row whose Size letter is S, M, or L
WHEN  the card renders
THEN  the effort segment reads `quick (<1h)`, `~half-day`, or `large (1-2 days)` respectively
```

### Story #4: Edge cases preserved [resilience]

```
GIVEN an actionable row with an empty body, OR a missing TODOS.md, OR no actionable items
WHEN  suggest.sh runs (no flags)
THEN  the empty-body card shows `What: (no description)`;
      a missing TODOS.md exits 1;
      no actionable items prints `No actionable items.` and exits 0
```

## Architecture

```
suggest.sh
  collect candidates ─┐
  score → $SCORED ────┤  (col7 = raw heading; unchanged)
  pick top-N ─────────┤
  render fork ────────┴─► if [ -n "$FOR_SKILL" ]: render_table   (existing, verbatim)
                          else:                    render_cards   (NEW)
                                                     ├─ effort_label(Size)
                                                     ├─ extract_body → first non-empty line | "(no description)"
                                                     └─ Why → Status: line
```

### Data Flow

1. Candidates collected and scored into `$SCORED` (unchanged); col 7 holds the raw heading.
2. Top-N rows selected (unchanged).
3. Render fork reads `$FOR_SKILL`: empty → `render_cards`; non-empty → `render_table` (existing path, untouched).
4. `render_cards` for each row: derive ID+title from the heading, expand `Size` via `effort_label`, pull the first non-empty body line via `extract_body` (or `(no description)`), fold `Why` into the `Status:` line.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Render strategy | Card list interactive + keep table for consumers (A) | Add `What`/`Effort` columns to the one table (B) | A prose column wraps badly in a terminal — the exact scan problem, just wider |
| Blast radius | Fork on `$FOR_SKILL`, leave consumer code path verbatim | Cards for all invocations + rewrite todo_fix.sh parser & fixture (C) | C is a breaking change to a second skill for no interactive-UX gain |
| `What:` source | First non-empty body line via `extract_body` | Truncated multi-line summary | Deterministic, author-controlled; no mid-thought cut |
