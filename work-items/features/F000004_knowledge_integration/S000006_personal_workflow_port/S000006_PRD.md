## Problem Statement

F000004 gives `/company-workflow` an external knowledge folder: engineers export
`AI_KNOWLEDGE_DIR`, drop categorized markdown under it, declare `.knowledge.yml`
surfacing modes, and Claude picks up both always-on guidance and on-demand
domain content automatically. `/personal-workflow` — the parallel validation
skill for personal-dev work items — has none of it. Engineers who use both
skills (company day job + personal side project) hit a cliff: the same
`$AI_KNOWLEDGE_DIR` works in one repo and silently does nothing in another.

This story closes that gap via a mechanical parity port. No new knobs, no new
schema, no new design decisions. Everything F000004 shipped for company-workflow
also works for personal-workflow.

## Mental Model

```
Before this story:
  /company-workflow ──► reads $AI_KNOWLEDGE_DIR ──► injects always-on + on-demand
  /personal-workflow ──► (nothing — knowledge env var is invisible to this skill)

After this story:
  /company-workflow ──► reads $AI_KNOWLEDGE_DIR ──► injects always-on + on-demand
  /personal-workflow ──► reads $AI_KNOWLEDGE_DIR ──► injects always-on + on-demand
                                                    (same env var, same schema,
                                                     same opt-in marker, same
                                                     warnings, same shipping
                                                     test helper)
```

The port is three copy-and-adapt moves, no more:

1. **SKILL.md §Knowledge Resolution** — copy company-workflow's block; adjust the `/company-workflow` reference to `/personal-workflow` in docstrings.
2. **SKILL.md §Knowledge Loading** — copy company-workflow's block; reuse the shared `scripts/test-helpers/knowledge.sh` as-is.
3. **WORKFLOW.md §Knowledge Configuration** — copy company-workflow's section; update command names in the prose.

`scripts/test.sh` gains a mirror of T000003's 11-case assertion block, pointed at personal-workflow's SKILL.md.

## Acceptance Criteria

### Story #1: resolution parity [core]

```
GIVEN AI_KNOWLEDGE_DIR is set to a valid directory
AND   company-workflow's SKILL.md and personal-workflow's SKILL.md both exist (deployed or in-repo)
WHEN  each skill's Knowledge Resolution block executes
THEN  both set $_KNOWLEDGE_DIR to the same resolved path
AND   neither emits a warning
AND   both exit 0 with empty stdout
```

### Story #2: warning-output parity [core]

```
GIVEN AI_KNOWLEDGE_DIR is unset
WHEN  personal-workflow's Knowledge Resolution block runs
THEN  exactly one stderr line is emitted
AND   the line names the variable
AND   the line is the same shape as company-workflow's (allowing only the skill-name reference to differ)
AND   exit code is 0

AND
GIVEN AI_KNOWLEDGE_DIR is set to a path with control characters or >200 chars
WHEN  the resolution block runs
THEN  the warning is sanitized identically to company-workflow (control chars stripped, truncated at 200)
```

### Story #3: always-on parity [core]

```
GIVEN $_KNOWLEDGE_DIR/coding/.knowledge.yml declares surface: always
AND   $_KNOWLEDGE_DIR/coding/ contains one or more *.md files (nested or flat)
AND   the current repo has .claude/knowledge-enabled
WHEN  /personal-workflow runs any command
THEN  the emitted Always-On Knowledge section lists the same paths that /company-workflow would list for the same fixture
AND   load order is deterministic (lexical, matching /company-workflow)
```

### Story #4: on-demand parity [core]

```
GIVEN a category with .knowledge.yml { surface: on-demand, triggers: [pricing engine] }
AND   the current repo has .claude/knowledge-enabled
WHEN  the user's latest message mentions "pricing engine"
THEN  /personal-workflow emits the same On-Demand Knowledge Candidates block that /company-workflow would
AND   Claude Reads the same set of paths under that block

AND
WHEN  the user's latest message mentions neither "pricing" alone nor "engine" alone
THEN  no on-demand files from that category are surfaced
```

### Story #5: per-repo opt-in parity [security]

```
GIVEN AI_KNOWLEDGE_DIR resolves to a valid directory with always-on + on-demand categories
AND   the current repo has NO .claude/knowledge-enabled marker
WHEN  /personal-workflow runs
THEN  no Always-On Knowledge section is emitted
AND   no On-Demand Knowledge Candidates section is emitted
AND   S000004's resolution warning still fires if the env var is misconfigured (the opt-in gate does not suppress resolution warnings)
```

### Story #6: zero regression [resilience]

```
GIVEN a clean personal-workflow install with no AI_KNOWLEDGE_DIR and no .claude/knowledge-enabled
WHEN  /personal-workflow check is run on an existing work-items/ tree
THEN  stdout is byte-identical to a run against the pre-port SKILL.md
AND   stderr contains only the AI_KNOWLEDGE_DIR=unset warning (identical to /company-workflow's)
AND   exit code is unchanged
```

### Story #8: docs read correctly [usability]

```
GIVEN a reader who only uses /personal-workflow
WHEN  they read personal-workflow/WORKFLOW.md §Knowledge Configuration
THEN  every command reference is /personal-workflow (not /company-workflow)
AND   the F000004 backlink points at the same feature (both skills share it)
AND   no unexplained company-workflow concepts are mentioned
```

## Success Metrics

|--------|--------|-------------|
| Byte-identical resolution + loading bash | 100% (modulo the skill-name docstring) | `diff` the two blocks after the port commit |
| Test assertion parity | T000007 mirrors T000003's 11 cases against personal-workflow's SKILL.md | `./scripts/test.sh` grep for both test-block headers |
| Zero regression | `/personal-workflow check` output unchanged pre/post port when no knowledge configured | Checked-in baseline file diffed in CI |
| Zero drift across skills | No text difference in the `## Knowledge Loading` bodies except command-name strings | Tier-1 assertion added to `scripts/test.sh` |
| Opt-in gate honored | 100% — loading never activates without `.claude/knowledge-enabled` | E2E case: env var set, marker absent, expect empty loading sections |

## Out of Scope

- Any change to `AI_KNOWLEDGE_DIR` semantics, `.knowledge.yml` schema, or the opt-in marker
- New surfacing modes (anything beyond `always` and `on-demand`)
- Per-skill knowledge directories (single env var stays shared)
- Moving the shared bash into a sourced helper file — P2 Story #9, deferred unless a maintenance pain surfaces
- Changes to `personal-artifact-manifests.json` (knowledge is runtime, not scaffolding)
- Changes to `/company-workflow` (this port is additive)
- Port of any F000004 follow-ups not yet scoped into S000004/S000005 (e.g., soft-cap warnings for always-on bytes, if deferred)

## Assumptions

- S000005 has landed on `main` before Phase 2 starts. Without it, there's no Loading block to port.
- S000004's Resolution block is stable (it shipped in PR #38; no changes expected before this port begins).
- `scripts/test-helpers/knowledge.sh` exists and is reusable once S000005 lands.
- The deployed install paths for both skills (`~/.claude/skills/{company,personal}-workflow/`) are structurally symmetric — the port can copy bash verbatim without path-resolution rework.
- `.claude/knowledge-enabled` is the final opt-in marker name (decided during S000005's Track phase).
- Users who run both skills in the same repo will place the opt-in marker once and have both skills honor it.
