## Problem Statement
<!-- What problem does this feature solve? Who has this problem?
     Be specific: name the user, their role, and the pain they experience today. -->

An engineer using the `company-workflow` skill wants Claude to consult their house coding guidance (e.g. cpp style) and company-specific domain knowledge while working on work items. Today, no such discovery mechanism exists: the engineer has to copy-paste relevant docs into every prompt, or watch Claude make guesses that contradict house conventions. The knowledge already exists in their filesystem — the skill just can't find it.

This story delivers **only the discovery step**: teach the skill to resolve the knowledge folder location via a single environment variable (`AI_KNOWLEDGE_DIR`) and warn when it's not configured. No knowledge is loaded yet; that ships in follow-up stories (S00000X always-on, S00000X on-demand).

## Acceptance Criteria

<!-- Given/When/Then format. One block per user story or logical group. -->

### Story #1: env var resolution [core]

```
GIVEN the engineer has exported AI_KNOWLEDGE_DIR=/path/to/knowledge
WHEN  the company-workflow skill is invoked
THEN  the skill captures the resolved path in a skill-internal variable and proceeds to run the requested command
```

### Story #2: warn when unset [usability]

```
GIVEN AI_KNOWLEDGE_DIR is unset or empty
WHEN  any company-workflow skill command runs
THEN  the skill emits exactly one warning line on stderr naming the variable and pointing to setup docs
AND   exit code is unchanged (0 on success)
```

### Story #3: warn when path invalid [resilience]

```
GIVEN AI_KNOWLEDGE_DIR is set to a path that does not exist or is not a directory
WHEN  any company-workflow skill command runs
THEN  the skill emits a warning mentioning the configured path and the reason (not-found / not-a-directory)
AND   exit code is unchanged (0 on success)
```

### Story #4: diagnostic visibility [observability]

```
GIVEN the engineer wants to verify knowledge folder detection
WHEN  they run a diagnostic (TBD: new subcommand, or include in existing validate output)
THEN  the resolved knowledge dir path (or "unset") appears in the output
```

## Out of Scope

<!-- Explicitly list what this feature does NOT do. Prevents scope creep. -->

- Reading or parsing any file inside the knowledge folder (deferred to S00000X always-on)
- Matching user prompt against triggers (deferred to S00000X on-demand)
- Validating `.knowledge.yml` schema (deferred — not needed until Layer 2)
- Any UI or slash command for managing the knowledge folder (out of feature scope)
- Cross-skill sharing with `/personal-workflow` (deferred; tracked in feature TRACKER Todos)

## Assumptions

<!-- What this PRD takes for granted. If an assumption is wrong, revisit the PRD. -->

- The engineer's shell environment is inherited by the skill's bash blocks (standard Claude Code skill behavior)
- Stderr is visible to the user in their Claude Code session (true in all current clients)
- The engineer can set an environment variable in their shell profile without additional tooling
