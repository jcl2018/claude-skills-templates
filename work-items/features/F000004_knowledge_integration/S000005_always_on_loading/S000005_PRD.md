## Problem Statement
<!-- What problem does this feature solve? Who has this problem?
     Be specific: name the user, their role, and the pain they experience today. -->

After S000004 lands, the skill knows where the engineer's knowledge folder lives — but does nothing with it. This story closes the loop for the guidance the engineer wants Claude to apply *by default*: coding style, team conventions, house patterns. The engineer marks those categories `surface: always` in `.knowledge.yml`, and from then on every skill invocation automatically pulls the content into Claude's context. No per-prompt copy-paste, no remembering to ask for it.

This is Layer 2 of the 3-layer F000004 stack. It depends on S000004 (resolution) and is parallel to S000006 (on-demand matching).

## Mental Model
<!-- If the feature has layers, phases, or modes, add a diagram or short
     explanation here so readers understand the structure before reading
     individual stories. For simple features, a one-sentence summary suffices. -->

```
$_KNOWLEDGE_DIR (from S000004)
├── coding/                       ← category
│   ├── .knowledge.yml { surface: always, triggers: [] }
│   ├── cpp/
│   │   └── errors.md             ← loaded every invocation
│   └── style.md                  ← loaded every invocation
├── runbooks/                     ← category
│   ├── .knowledge.yml { surface: on-demand, triggers: [...] }  ← NOT this story
│   └── db-failover.md
└── notes/                        ← category, no .knowledge.yml
                                   ← treated as on-demand + empty triggers (not loaded)
```

The skill's job in this story: enumerate top-level categories, filter to `surface: always`, concatenate their `*.md` files (in deterministic order) into Claude's context.

## User Stories

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | Does the skill automatically include my coding guidance in every invocation? | engineer who configured a `surface: always` category | have Claude read my house style without my asking | style violations drop without friction |
| 2 | core | Does it handle nested directories within a category? | engineer who organizes by sub-topic | put `coding/cpp/errors.md` and `coding/python/typing.md` under one always-on category | I can structure knowledge the way I think about it |
| 3 | resilience | Does a bad `.knowledge.yml` in one category break loading of other categories? | engineer with one broken config file | see a warning and have other categories still load | a typo doesn't disable the whole feature |
| 4 | core | Does a category without `.knowledge.yml` get loaded? | engineer with a fresh, unconfigured category | have the skill ignore it (on-demand default) | I can stage content before declaring it always-on |

### P1 (Important)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 5 | observability | Can I see which always-on categories were loaded? | engineer debugging why Claude didn't apply a guide | see a list of loaded paths in diagnostic output | I can verify config vs. behavior |
| 6 | resilience | Is there a cap on total always-on bytes? | engineer who accidentally marks a huge folder as always-on | get a warning (or truncation) before context fills up | the feature doesn't blow up my effective context |

### P2 (Nice-to-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|

## Acceptance Criteria

### Story #1 + #2: always-on loading with nesting [core]

```
GIVEN $_KNOWLEDGE_DIR = /tmp/k
AND   /tmp/k/coding/.knowledge.yml contains "surface: always"
AND   /tmp/k/coding/ contains a.md and cpp/b.md
WHEN  the skill runs any command
THEN  Claude's context includes the contents of a.md and cpp/b.md
AND   load order is deterministic (a.md before cpp/b.md, lexical path order)
```

### Story #3: malformed yml resilience [resilience]

```
GIVEN two always-on categories, one with valid yml and one with broken yml
WHEN  the skill runs
THEN  the valid category loads fully
AND   a single warning names the broken yml file
AND   exit code is unchanged
```

### Story #4: missing yml = on-demand default [core]

```
GIVEN a category directory with no .knowledge.yml
WHEN  the skill runs
THEN  no files from that category are loaded
AND   no warning is emitted (missing yml is not an error)
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Always-on category content reaches Claude | 100% of declared always-on files | E2E test: inject unique string into a knowledge file, verify Claude can quote it back |
| Zero regression | `validate` output byte-identical w/ and w/o categories | Diff on existing fixtures |
| One-typo blast radius | A malformed yml in category A does not block category B | Integration test: mix valid + malformed |

## Out of Scope

- Matching user prompts against triggers (S000006)
- Editing or authoring `.knowledge.yml` (user responsibility)
- Hot-reload when knowledge files change mid-session (not relevant — each invocation reads fresh)
- Cross-category dependencies (a file in A requires a file in B) — treat categories as independent
- Format validation for the *markdown* content (only yml is validated)

## Assumptions

- S000004 has landed and `$_KNOWLEDGE_DIR` is populated before this story runs
- The engineer's knowledge folder has ≤ ~20 top-level categories (no pagination needed)
- The total size of always-on content is a user problem (they marked it always-on); story provides a warning for observability but does not enforce a hard cap
