## Problem Statement
<!-- What problem does this feature solve? Who has this problem?
     Be specific: name the user, their role, and the pain they experience today. -->

S000005 delivers always-on knowledge — the content that should apply every time. But not all knowledge is always-on. Domain deep-dives, runbooks, language references, team-specific jargon — these are only useful when the user is actually asking about them. Loading them on every invocation would flood Claude's context with irrelevant material.

This story closes that gap. The engineer declares `surface: on-demand` with a list of `triggers` in a category's `.knowledge.yml`. When the engineer's prompt mentions any trigger (case-insensitive, whole-word, quoted phrases matched as a unit), Claude loads that category's content — otherwise it doesn't. This is Layer 3 of the F000004 stack, the final step that makes the knowledge feature complete.

## Mental Model

```
User prompt: "help me debug the pricing engine's rounding error"
                        │
                        ▼
         ┌──────────────────────────────┐
         │  For each on-demand category │
         │  in $_KNOWLEDGE_DIR:         │
         │  - read triggers list         │
         │  - does any trigger match?   │
         └──────────────┬───────────────┘
                        │
                        ▼
      ┌─────────────────────────────────────┐
      │  Matches:                           │
      │    runbooks/      triggers: [pricing engine, PE]    ← MATCH
      │    cpp-style/     triggers: [cpp, c++]               ← no match
      │    security/      triggers: [auth, TLS, CVE]        ← no match
      └─────────────────────────────────────┘
                        │
                        ▼
         ┌──────────────────────────────┐
         │  Load all *.md under          │
         │  $_KNOWLEDGE_DIR/runbooks/    │
         └──────────────────────────────┘
```

Matching is done by Claude, not by bash: the prompt lives in Claude's context, and the skill's shell preamble can't see it. The skill emits a "candidate list" (on-demand categories + their triggers + their file paths) and instructs Claude to do the matching + Reading.

## User Stories

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | Does typing a trigger keyword pull in its category? | engineer asking about a specific domain | say "pricing engine" and have Claude reference my runbook | I don't have to copy-paste context every time |
| 2 | core | Does a multi-word phrase trigger work? | engineer whose domain has compound names | declare `"pricing engine"` as a single trigger | the trigger matches only the phrase, not "pricing" alone |
| 3 | resilience | Does absence of any trigger keyword leave on-demand categories alone? | engineer asking an unrelated question | have Claude NOT load domain knowledge my prompt didn't invoke | context stays focused |
| 4 | core | Does matching multiple categories load all of them? | engineer whose prompt mentions two domains | pull in both runbook and security guides | I get the union, not an arbitrary pick |

### P1 (Important)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 5 | observability | Can I see which categories were matched and why? | engineer tuning trigger lists | see a diagnostic line naming matched categories and matched triggers | I can narrow/broaden trigger coverage without guessing |
| 6 | usability | Is matching case-insensitive? | engineer who sometimes types `cpp` and sometimes `CPP` | have both match `triggers: [cpp]` | I don't need to tweak triggers per casing |

### P2 (Nice-to-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 7 | usability | Can the skill score matches by specificity (prefer more-specific triggers)? | engineer with overlapping triggers | have "pricing engine" win over "pricing" when both match | narrow matches surface first |
| 8 | usability | Can I declare synonyms compactly? | engineer with many aliases for one concept | use a trigger-set syntax like `triggers: {pricing engine: [PE, pricing, billing]}` | yml stays readable |

## Acceptance Criteria

### Story #1: single-word trigger match [core]

```
GIVEN category /k/runbooks/ with .knowledge.yml { surface: on-demand, triggers: [pricing] }
AND   /k/runbooks/overview.md contains unique canary CANARY_RB_1
WHEN  user asks "explain the pricing flow"
THEN  Claude Reads /k/runbooks/overview.md before answering
AND   its reply can cite CANARY_RB_1
```

### Story #2: multi-word phrase trigger [core]

```
GIVEN category with triggers: ["pricing engine"]
AND   a file under the category
WHEN  user asks "how does the pricing engine handle rounding?"
THEN  the category is loaded (phrase match)

AND
GIVEN same category
WHEN  user asks "what is pricing?"
THEN  the category is NOT loaded (the word "pricing" alone is not the phrase "pricing engine")
```

### Story #3: no match means no loading [resilience]

```
GIVEN any number of on-demand categories with any triggers
WHEN  the user's prompt contains none of those triggers
THEN  none of those categories' files are Read
AND   the request is answered from Claude's baseline context + always-on knowledge only
```

### Story #4: multi-match loads all [core]

```
GIVEN two on-demand categories: runbooks (triggers: [pricing]) and security (triggers: [auth])
WHEN  user asks about pricing auth
THEN  files from BOTH categories are Read
```

### Story #6: case-insensitivity [usability]

```
GIVEN trigger declared as "cpp"
WHEN  user writes "CPP", "Cpp", or "cpp"
THEN  the category is loaded in all three cases
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Trigger hit rate (intended matches found) | ≥ 95% across author-specified test cases | Tier 2 E2E canary tests |
| False positive rate (unwanted matches) | < 5% | Tier 2 suite: prompts that should NOT match |
| No regression on existing commands | `validate` byte-identical w/ and w/o on-demand categories | Diff against baseline |

## Out of Scope

- Fuzzy / semantic matching (embeddings, stemming, synonyms) — explicit triggers only
- Priority/scoring between matched categories (P2 Story #7)
- Trigger-set / synonym syntax in `.knowledge.yml` (P2 Story #8)
- Matching against prior conversation turns — latest user message only
- Editing or suggesting trigger lists (user-owned)

## Assumptions

- S000004 has landed and `$_KNOWLEDGE_DIR` is populated
- Claude can reliably tokenize the latest user message and match against a small keyword list (≤20 triggers per category × ≤20 categories)
- Users will iterate on their trigger lists after observing hit/miss behavior (hence P1 observability)
