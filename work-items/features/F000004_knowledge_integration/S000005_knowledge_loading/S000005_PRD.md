## Problem Statement
<!-- What problem does this feature solve? Who has this problem?
     Be specific: name the user, their role, and the pain they experience today. -->

After S000004 lands, the skill knows where the engineer's knowledge folder lives — but does nothing with it. This story closes the loop for both surfacing paths in one PR:

- **Always-on**: guidance the engineer wants Claude to apply *by default* — coding style, team conventions, house patterns. The engineer marks those categories `surface: always`, and from then on every skill invocation automatically pulls the content into Claude's context. No per-prompt copy-paste.
- **On-demand**: domain deep-dives, runbooks, language references, team-specific jargon — only useful when the user is actually asking about them. The engineer declares `surface: on-demand` with a `triggers` list. When the prompt mentions any trigger (case-insensitive, whole-word, quoted phrases as a unit), Claude loads that category — otherwise it doesn't.

The two paths share the category tree, the `.knowledge.yml` parser, the per-repo opt-in gate, and the test fixture builder. Splitting them across two stories was vertical-slicing bookkeeping; the implementation is one PR. This story replaces both former S000005 (always-on) and S000006 (on-demand).

## Mental Model
<!-- If the feature has layers, phases, or modes, add a diagram or short
     explanation here so readers understand the structure before reading
     individual stories. For simple features, a one-sentence summary suffices. -->

```
$_KNOWLEDGE_DIR (from S000004)
├── coding/                       ← category
│   ├── .knowledge.yml { surface: always, triggers: [] }
│   ├── cpp/
│   │   └── errors.md             ← loaded EVERY invocation (always-on)
│   └── style.md                  ← loaded EVERY invocation (always-on)
├── runbooks/                     ← category
│   ├── .knowledge.yml { surface: on-demand, triggers: ["pricing engine", PE] }
│   └── pricing.md                ← loaded ONLY if prompt matches a trigger
├── notes/                        ← category, no .knowledge.yml
│                                 ← treated as on-demand + empty triggers (never loads)
└── (repo root)/.claude/knowledge-enabled  ← REQUIRED opt-in marker
```

The skill's job in this story:
1. Check the per-repo opt-in marker — if absent, do nothing (S000004's resolution still ran, but no content loads).
2. Enumerate top-level categories. Parse each `.knowledge.yml`.
3. For `surface: always`: emit `## Always-On Knowledge` block listing absolute paths; instruct Claude to Read them all.
4. For `surface: on-demand` with non-empty triggers: emit `## On-Demand Knowledge Candidates` block (root + triggers + file paths); instruct Claude to tokenize the latest user message and Read paths from any matched category.

Bash handles discovery + file enumeration (deterministic, testable). Claude handles prompt matching for on-demand (the prompt is only visible to Claude). One shared `.knowledge.yml` parser, one shared file enumerator, one shared malformed-yml warning path.

## User Stories

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | always-on / core | Does the skill automatically include my coding guidance in every invocation? | engineer who configured a `surface: always` category | have Claude read my house style without my asking | style violations drop without friction |
| 2 | always-on / core | Does it handle nested directories within a category? | engineer who organizes by sub-topic | put `coding/cpp/errors.md` and `coding/python/typing.md` under one always-on category | I can structure knowledge the way I think about it |
| 3 | resilience | Does a bad `.knowledge.yml` in one category break loading of other categories? | engineer with one broken config file | see a warning and have other categories still load | a typo doesn't disable the whole feature |
| 4 | always-on / core | Does a category without `.knowledge.yml` get loaded? | engineer with a fresh, unconfigured category | have the skill ignore it (on-demand default) | I can stage content before declaring it always-on |
| 5 | on-demand / core | Does typing a trigger keyword pull in its category? | engineer asking about a specific domain | say "pricing engine" and have Claude reference my runbook | I don't have to copy-paste context every time |
| 6 | on-demand / core | Does a multi-word phrase trigger work? | engineer whose domain has compound names | declare `"pricing engine"` as a single trigger | the trigger matches only the phrase, not "pricing" alone |
| 7 | on-demand / resilience | Does absence of any trigger keyword leave on-demand categories alone? | engineer asking an unrelated question | have Claude NOT load domain knowledge my prompt didn't invoke | context stays focused |
| 8 | on-demand / core | Does matching multiple categories load all of them? | engineer whose prompt mentions two domains | pull in both runbook and security guides | I get the union, not an arbitrary pick |
| 9 | security / gating | Is knowledge loading gated per repo? | engineer with a global `$AI_KNOWLEDGE_DIR` pointing at Company A's folder, working in Company B's repo | have the skill NOT auto-inject Company A guidance in Company B's repo | I don't leak cross-context knowledge |

## Acceptance Criteria

### Stories #1 + #2: always-on loading with nesting [core]

```
GIVEN $_KNOWLEDGE_DIR = /tmp/k
AND   /tmp/k/coding/.knowledge.yml contains "surface: always"
AND   /tmp/k/coding/ contains a.md and cpp/b.md
AND   the per-repo opt-in marker is present
WHEN  the skill runs any command
THEN  Claude's context includes the contents of a.md and cpp/b.md
AND   load order is deterministic (a.md before cpp/b.md, lexical path order)
```

### Story #3: malformed yml resilience [resilience]

```
GIVEN two categories (one always-on, one on-demand), each with `.knowledge.yml`
AND   one of the yml files is broken
WHEN  the skill runs
THEN  the valid category loads (always-on) or emits as candidate (on-demand)
AND   a single warning names the broken yml file
AND   exit code is unchanged
```

### Story #4: missing yml = silent skip [core]

```
GIVEN a category directory with no .knowledge.yml
WHEN  the skill runs
THEN  no files from that category are loaded
AND   the category is not emitted as an on-demand candidate
AND   no warning is emitted (missing yml is not an error)
```

### Story #5: single-word on-demand trigger match [core]

```
GIVEN category /k/runbooks/ with .knowledge.yml { surface: on-demand, triggers: [pricing] }
AND   /k/runbooks/overview.md contains unique canary CANARY_RB_1
AND   the per-repo opt-in marker is present
WHEN  user asks "explain the pricing flow"
THEN  Claude Reads /k/runbooks/overview.md before answering
AND   its reply can cite CANARY_RB_1
```

### Story #6: multi-word phrase trigger [core]

```
GIVEN category with triggers: ["pricing engine"]
AND   a file under the category
WHEN  user asks "how does the pricing engine handle rounding?"
THEN  the category is loaded (phrase match)

AND
GIVEN same category
WHEN  user asks "what is pricing?"
THEN  the category is NOT loaded (the word "pricing" alone is not the phrase "pricing engine")
```

### Story #7: no match means no loading [resilience]

```
GIVEN any number of on-demand categories with any triggers
WHEN  the user's prompt contains none of those triggers
THEN  none of those categories' files are Read
AND   the request is answered from Claude's baseline context + always-on knowledge only
```

### Story #8: multi-match loads all [core]

```
GIVEN two on-demand categories: runbooks (triggers: [pricing]) and security (triggers: [auth])
WHEN  user asks about pricing auth
THEN  files from BOTH categories are Read
```

### Story #9: per-repo opt-in gate [security]

```
GIVEN $_KNOWLEDGE_DIR points at Company A's knowledge folder
AND   the current repo is Company B's repo (no `.claude/knowledge-enabled` marker)
WHEN  the skill runs
THEN  no Always-On Knowledge section is emitted
AND   no On-Demand Knowledge Candidates section is emitted
AND   no warning about unconfigured knowledge (S000004 still warned about resolution; this story is silent)

AND
GIVEN same setup but the marker IS present in the repo root
WHEN  the skill runs
THEN  loading + matching activate normally
```

### Story #13: case-insensitivity [usability]

```
GIVEN trigger declared as "cpp"
WHEN  user writes "CPP", "Cpp", or "cpp"
THEN  the category is loaded in all three cases
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Always-on category content reaches Claude | 100% of declared always-on files | E2E test: inject unique string into a knowledge file, verify Claude can quote it back |
| Trigger hit rate (intended on-demand matches found) | ≥ 95% across author-specified test cases | Tier 2 E2E canary tests |
| False positive rate (unwanted on-demand matches) | < 5% | Tier 2 suite: prompts that should NOT match |
| Zero regression | `validate` output byte-identical w/ and w/o categories | Diff on existing fixtures |
| One-typo blast radius | A malformed yml in category A does not block category B | Integration test: mix valid + malformed |
| Cross-context contamination prevention | 100% — no content loads without opt-in marker | E2E: marker absent + valid env → empty loading sections |

## Out of Scope

- Editing or authoring `.knowledge.yml` (user responsibility)
- Hot-reload when knowledge files change mid-session (each invocation reads fresh)
- Cross-category dependencies (a file in A requires a file in B) — categories are independent
- Format validation for the *markdown* content (only yml is validated)
- Fuzzy / semantic matching for on-demand (embeddings, stemming, synonyms) — explicit triggers only
- Priority/scoring between matched on-demand categories (P2 Story #15)
- Trigger-set / synonym syntax (P2 Story #16)
- Matching against prior conversation turns — latest user message only
- Per-file ignore syntax (P2 Story #14)

## Assumptions

- S000004 has landed and `$_KNOWLEDGE_DIR` is populated before this story runs
- The engineer's knowledge folder has ≤ ~20 top-level categories (no pagination needed)
- The total size of always-on content is a user problem (they marked it always-on); story provides a warning for observability but does not enforce a hard cap
- Claude can reliably tokenize the latest user message and match against a small keyword list (≤20 triggers per category × ≤20 categories)
- Users will iterate on their trigger lists after observing hit/miss behavior (hence P1 observability)
- The per-repo opt-in marker filename (`.claude/knowledge-enabled` is the leading candidate) is decided during T000006's design
