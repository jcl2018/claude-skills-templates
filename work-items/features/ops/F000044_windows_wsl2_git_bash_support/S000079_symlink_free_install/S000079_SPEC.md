## Problem Statement

`scripts/skills-deploy` installs skills by symlinking `*.md`/`*.json` into
`~/.claude/` (`ln -snf` at lines 275/305/667/681). `doctor` asserts each
installed file is a symlink (`[ -L ]` tests at ~830/851/870), and `remove`
(`-L` check ~545) and `relink` (~664) also assume symlinks. On Git Bash,
`ln -s` copies-by-default or requires Developer Mode/admin вЂ” so a maintainer or
operator on Windows Git Bash gets a degraded install and a `doctor` that reports
false failures for files that are actually present.

Critically, the runtime manifest (`~/.claude/.skills-templates.json`) stores
each skill record as `{path, installed_at}` ONLY вЂ” there is NO per-skill-file
checksum (`source_checksum` exists for templates/rules, not for skills). So
copy-mode has nothing to verify an installed copy against; a new manifest schema
is required before copy-mode `doctor` can detect drift. This schema dependency is
why this story is L-effort, not M.

## Mental Model

`skills-deploy` install runs: probe symlink capability в†’ pick install mode в†’
record mode (+ checksum for copies) in the manifest в†’ `doctor`/`remove`/`relink`
branch on the recorded mode.

```
install в”Ђв–є _can_symlink probe в”Ђв”¬в”Ђ capable в”Ђв”Ђв–є symlink-mode (ln -snf)
                               в””в”Ђ not в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–є copy-mode (cp + compute source_checksum)
                                                      в”‚
                                  write manifest {install_kind, source_checksum?}
                                                      в”‚
doctor в”Ђв–є read manifest в”Ђв–є branch on install_kind в”Ђв”¬в”Ђ copy   в–є re-hash file vs source_checksum
                                                   в””в”Ђ symlink в–є [ -L ] + link target
```

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. -->

### Story #1: Copy-mode install succeeds [core]

```
GIVEN Git Bash (no usable symlinks)
WHEN  `skills-deploy install` runs
THEN  skills install as regular file copies and the command exits 0
```

### Story #2: Mode chosen automatically [core]

```
GIVEN any platform
WHEN  install starts
THEN  `_can_symlink()` probes (creates a temp symlink, tests `[ -L ]`) and
      selects symlink-mode if real symlinks work, else copy-mode
```

### Story #3: Doctor verifies both modes [core]

```
GIVEN a copy install
WHEN  `doctor` runs
THEN  it re-hashes each installed file against the manifest `source_checksum`;
GIVEN a symlink install
THEN  doctor checks `[ -L ]` and link target
```

### Story #4: Remove + relink handle both modes [core]

```
GIVEN either install mode
WHEN  `remove`/`relink` run
THEN  they operate correctly on the recorded mode
```

### Story #5: Manifest records mode + checksum [resilience]

```
GIVEN a copy install
WHEN  the manifest is written
THEN  each skill record has `install_kind: copy` and a per-file `source_checksum`
```

### Story #6: Both-mode install is tested [observability]

```
GIVEN test-deploy.sh
WHEN  run
THEN  it exercises copy-mode install, symlink-mode install, and a doctor case for each
```

### Data Flow

1. install probes symlink capability
2. writes files (link or copy)
3. records install_kind (+ source_checksum for copies) in manifest
4. doctor re-reads manifest and verifies per recorded mode

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| install strategy | symlink on capable platforms, copy on Git Bash | unify on copy everywhere | keep the maintainer's instant-edit dev loop on macOS/WSL2 while making Git Bash work |
| verification source | new manifest install_kind + source_checksum | reuse existing checksum | skills currently store NO checksum (only templates/rules do), so there is nothing to reuse |
