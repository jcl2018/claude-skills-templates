## Problem Statement

Without Windows CI, the new Windows support rots silently. CI today is
`.github/workflows/validate.yml` + `.github/workflows/eval-nightly.yml`, both on
`ubuntu-latest` — no Windows coverage. There is also no "Running on Windows"
documentation. A maintainer has no signal when a change breaks Windows, and a
Windows user has no on-ramp. This story proves the support (a `windows-latest`
job running the relevant test subset) and documents it (README + CLAUDE.md).

## Mental Model

Add a Windows CI job + write the Windows docs. The CI side is a single new job
that runs a Windows-relevant test subset under Git Bash; the docs side is two
edits (README on-ramp + CLAUDE.md note):

```
PR push ──► windows-latest runner ──► Git Bash ──► run test subset ──► pass/fail gate
                                                    (portable-date +
                                                     copy-mode-install)
```

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. -->

### Story #1: Windows CI job [core]

```
GIVEN .github/workflows
WHEN  CI config is parsed
THEN  a windows-latest job exists that runs the portable-date + copy-mode-install
      test subset with shell bash (Git Bash)
```

### Story #2: Catch Windows regressions [observability]

```
GIVEN a PR that breaks Windows support
WHEN  CI runs
THEN  the windows-latest job fails
```

### Story #3: Windows user on-ramp [usability]

```
GIVEN README
WHEN  a Windows user reads it
THEN  a "Running on Windows" section explains WSL2 (recommended) + Git Bash
      (supported) + prereqs
```

### Story #4: CLAUDE.md note [usability]

```
GIVEN CLAUDE.md
WHEN  read
THEN  it notes Windows (WSL2 + Git Bash) support
```

## Architecture

```
.github/workflows/ (validate.yml OR new windows.yml)
  job: windows
    runs-on: windows-latest
    defaults.run.shell: bash        # Git Bash, built into windows-latest
    steps:
      - checkout
      - run portable-date tests      (from S000078)
      - run copy-mode-install tests  (from S000079)
    └─► job status gates the PR

docs:
  README.md   ── "Running on Windows" section (WSL2 recommended / Git Bash supported / prereqs)
  CLAUDE.md   ── Windows-support note
  scripts/skills-deploy (doctor) ── optional platform/capability summary line

flow:  PR push ─► windows-latest runner ─► Git Bash ─► run test subset ─► pass/fail gate
```

### Data Flow

1. PR push triggers workflow
2. windows-latest runner starts Git Bash
3. runs portable-date + copy-mode-install tests
4. job status gates the PR

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| job placement | new windows.yml | matrix in validate.yml | the Windows test subset differs from the full suite, so a dedicated job is cleaner |
| shell target | Git Bash on windows-latest | also a WSL2 job | Git Bash is built into windows-latest; WSL2 needs a marketplace action — defer |
