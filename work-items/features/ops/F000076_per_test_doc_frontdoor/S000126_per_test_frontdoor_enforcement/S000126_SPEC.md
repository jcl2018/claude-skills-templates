## Problem Statement

A maintainer opening `docs/tests/<category>/<name>.md` for a declared test gets a
thin stub (Name/Category/Command/Tier + skeleton), while the real detail lives in
a separate family table (`docs/tests/<family>.md`). Nothing requires the per-test
doc to actually state what the test is, how to run it, and why it exists, and
nothing keeps that content honest as the test changes. The person with this
problem is any operator or contributor (in this repo AND in any repo that adopts
the portable contract) trying to understand or run a single named test.

## Mental Model

Two doc surfaces, re-framed so they stop competing:

```
categories: axis ──> docs/tests/<category>/<name>.md   [AUTHORITATIVE front door]
                     (What it is / How to run / Explanation + link ↓)
                                                  │ cross-link
                                                  ▼
units: axis ──────> docs/tests/<family>.md             [units-detail drill-down]
        (GENERATED by --render-docs; KEPT unchanged; the per-unit table)
```

The front-door content is enforced by a GENERAL rule in the portable
`spec/test-spec.md` (seed), a deterministic `--check-structure` content check, and
a `/CJ_test_audit` Stage-2 truthfulness judgment. The family render and its
Check 26 freshness gate are untouched.

## Acceptance Criteria

<!-- Given/When/Then. Each #n header matches the # column in Requirements. -->

### Story #1: General seed rule via lockstep edit [core]

```
GIVEN the category-axis section already exists in both spec/test-spec.md and the scripts/test-spec.sh --seed heredoc
WHEN  the what/how/why requirement is added by EDITING that section in both files
THEN  spec/test-spec.md and the rendered --seed output are byte-identical (tests/test-spec.test.sh cmp -s passes) and the rule text states the three-section requirement
```

### Story #2: Enriched --seed-docs template [usability]

```
GIVEN a repo with a categories: axis and a declared test whose doc does not yet exist
WHEN  test-spec.sh --seed-docs runs
THEN  the seeded docs/tests/<category>/<name>.md contains the What it is / How to run / Explanation sections + a family-doc cross-link, and re-running --seed-docs skips the now-present doc (idempotent)
```

### Story #3: One-time fill of the 7 existing docs [usability]

```
GIVEN the 7 already-seeded category docs (validate, suite, test-deploy, windows, windows-deploy, goal-task-eval, e2e-local)
WHEN  their content is authored to the new template
THEN  each carries a filled What it is / How to run / Explanation and cross-links its relevant family doc(s)
```

### Story #4: --check-structure content check [core]

```
GIVEN a repo with a categories: axis
WHEN  test-spec.sh --check-structure runs
THEN  it reports a finding for any per-test doc missing a required section and passes when all three are present; and GIVEN a repo with NO categories: axis, WHEN it runs, THEN it reports the content check inactive (no crash, exit 0)
```

### Story #5: /CJ_test_audit Stage 1 + Stage 2 [observability]

```
GIVEN a per-test doc whose How-to-run no longer matches the category command
WHEN  /CJ_test_audit runs
THEN  Stage 1 runs the deterministic structure content check and Stage 2 flags the doc as not truthful (how-to-run mismatch / inaccurate what-why) with cited evidence
```

### Story #6: /CJ_test_run surfacing [usability]

```
GIVEN a declared test name
WHEN  /CJ_test_run <name> runs
THEN  it surfaces or links that test's docs/tests/<category>/<name>.md How-to-run so the executed command and the documented command agree
```

### Story #7: doc-spec registry consistency [integration]

```
GIVEN the category-doc rows (72-78) and family-doc rows (63-70) in spec/doc-spec-custom.md
WHEN  the category-doc Requirement cells are updated to include the what/how/why sections
THEN  the family rows and the docs/tests/index.md row are unchanged and validate.sh Checks 15/15a/16 stay green
```

### Story #8: skills + catalog docs [usability]

```
GIVEN the enforced per-test-doc-content model
WHEN  CJ_test_audit + CJ_test_run SKILL.md/USAGE.md + skills-catalog.json descriptions are updated (and CLAUDE.md via doc-sync)
THEN  the documented behavior matches the implementation and validate.sh Check 14 (USAGE freshness) + the registered-doc audit stay green
```

## Architecture

<!-- High-level design for this story. -->

```
spec/test-spec.md  <== cmp -s ==>  scripts/test-spec.sh --seed heredoc
   (general rule EDIT, lockstep)          │
                                          ├── --seed-docs  (enriched stub template; present ⇒ skip)
                                          ├── --check-structure  (adds content check (f), categories-gated)
                                          └── --list-categories  (name → category/command/doc; used by fill + runner)
                    │
   docs/tests/<category>/<name>.md  (7 filled) ──cross-link──> docs/tests/<family>.md (GENERATED, unchanged)
                    │
   /CJ_test_audit  Stage 1 (runs --check-structure) + Stage 2 (truthfulness judgment)
   /CJ_test_run <name>  (surfaces the doc's How-to-run)
   spec/doc-spec-custom.md  (category-doc rows: Requirement-cell update only)
```

### Data Flow

<!-- How data moves for the primary use case (a maintainer running the audit). -->

1. Operator runs `/CJ_test_audit`; Stage 1 invokes `test-spec.sh --check-structure`.
2. `--check-structure` reads the merged registry's `categories:` axis and, for each declared test, resolves `docs/tests/<category>/<name>.md`.
3. For each doc it checks the three required section headings are present; a missing section emits a finding; all-present passes; no `categories:` axis ⇒ content check reported inactive.
4. Stage 2 (agent-judged) reads each doc + its category `command` and judges truthfulness (how-to-run matches; what/why accurate), citing evidence.
5. Separately, `/CJ_test_run <name>` maps name → doc via `--list-categories` and surfaces the doc's How-to-run alongside the executed command.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Doc authority model | Category docs enforced as the front door; keep the family render | Retire the family render, generate one per-test doc (Approach A) | A needs a nonexistent categories→units join key, a home for orphan ci/hook units, and a 2-feature schema change (review dealbreaker). |
| How to add the rule | EDIT the existing category-axis section in both files | Append a new standalone rule | An append breaks the `spec/test-spec.md` ↔ `--seed` `cmp -s` byte-identity. |
| Upgrading the 7 stubs | One-time authored fill; template updated for future | Make `--seed-docs` overwrite existing docs | Overwriting would clobber any authored content and violate the idempotent present-⇒-skip contract. |
| Enrichment scope | Fill the stubs (Approach B) | Rule + check only, thin stubs (Approach C) | C leaves the docs thin; B is only marginally larger and delivers real front doors. |
