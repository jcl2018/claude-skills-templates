## Problem

`docs/tests/` carries two doc surfaces that don't line up. The **family docs**
(`docs/tests/<family>.md`) are GENERATED by `test-spec.sh --render-docs` from the
`units:` axis, grouped by family — a rich table of every atomic verification unit
(e.g. `validate.md` = ~30 `validate.sh` checks; ~82 units across 7 families): the
units-detail view. The **category-test docs** (`docs/tests/<category>/<name>.md`)
are SEEDED STUBS by `--seed-docs` from the `categories:` axis, one per declared
category test — the named-test view, but a thin stub (Name/Category/Command/Tier
+ a Purpose + a How-to-run skeleton).

The category docs are the right per-test surface but are near-empty and
unenforced: a reader gets a stub for "what is `validate`" and a 30-row table
elsewhere. There is no rule that a test's doc actually says what it is, how to run
it, and why. The goal is to make `docs/tests/<category>/<name>.md` the
**authoritative front door per test** — required What it is / How to run /
Explanation (why) — enforced as a GENERAL rule in the portable `spec/test-spec.md`
and checked by the two cj_test skills, while KEEPING the family docs as the linked
units-detail drill-down (re-framed as detail, not a competing authority).

## Shape of the solution

Enforce a three-section content template on each category-test doc as a GENERAL,
portable rule, and keep the family render intact as the linked detail. The seven
deliverables from the /office-hours design are one coherent, low-risk change with
no schema change, no categories→units join key, and no Check 26 rewrite — so they
land as a single user-story.

## Big decisions

<!-- Choices that shape the feature, with rationale. -->

| # | Decision | Why |
|---|----------|-----|
| 1 | Approach B: category docs are the enforced front door; KEEP the family render. | Approach A (retire the family render, generate one per-test doc) needs a categories→units join key that does not exist + a home for orphan ci/hook units + a 2-feature schema change (review dealbreaker, 5/10). B removes those by construction — a single low-risk feature. |
| 2 | The what/how/why requirement is a GENERAL rule in the portable `spec/test-spec.md`, not a repo-custom overlay rule. | Designing for every adopting repo — the rule ships in the seed and propagates to consumers on their next contract seed. |
| 3 | The rule is added by EDITING the existing `## The category axis` section (present in both `spec/test-spec.md` and the `scripts/test-spec.sh --seed` heredoc), in lockstep, NOT by appending a duplicate rule. | `tests/test-spec.test.sh` asserts the two copies are byte-identical (`cmp -s`); a duplicate/append would break that identity. |
| 4 | One-time authored content fill of the 7 existing category docs; `--seed-docs` template updated only for future tests. | `--seed-docs` is idempotent (present ⇒ skip) so it will NOT upgrade the already-seeded stubs — their content is authored, not generated. |
| 5 | `spec/doc-spec-custom.md` category-doc rows get requirement-cell UPDATES (not removal); family-doc rows stay unchanged. | Keeps doc-spec Checks 15/15a green with no row churn — the family render is kept, so no rows are added or removed. |
| 6 | Nothing is re-categorized; no test scripts move. | `eval`'s front door stays `docs/tests/workflow/goal-task-eval.md` (its declared category per F000075). Under B nothing moves — avoids the orphan/relocate risk of A. |

## Risks & open questions

<!-- What could go wrong, and what's still undecided. -->

| Risk / Question | Next check |
|-----------------|-----------|
| Exact section headings (`## What it is` / `## How to run` / `## Explanation`) — the content check keys off these. | Confirm wording at the start of implement (S000126 SPEC); the check is heading-anchored. |
| Editing the seed in two files could drift the `cmp -s` byte-identity. | `tests/test-spec.test.sh` `cmp -s` case run during QA; edit both copies in the same commit. |
| The 7 category docs' exact category subdirs (CI-push vs CI-nightly vs workflow) must match the live `categories:` axis after F000075. | Enumerate from `test-spec.sh --list-categories` at implement time before authoring; do not hardcode from memory. |
| POSIX + LF + portable-date + CR-stripped jq discipline in `scripts/test-spec.sh` edits. | shellcheck + Windows Git-Bash CI (`windows.yml`) gate; run `bash scripts/windows-smoke.sh` locally. |

## Definition of done

<!-- Objective, measurable criteria for "shipped." -->

- [ ] The general what/how/why rule is in `spec/test-spec.md` and the `--seed` heredoc (`cmp -s` green).
- [ ] All 7 category docs carry What it is / How to run / Explanation + a family-doc link.
- [ ] `test-spec.sh --check-structure` content check green, and inactive without a `categories:` axis.
- [ ] `/CJ_test_audit` Stage 1 (runs the check) + Stage 2 (judges truthfulness) green on the docs.
- [ ] `/CJ_test_run <name>` surfaces/links the per-test doc's How-to-run.
- [ ] `spec/doc-spec-custom.md` category-doc row requirements updated; family rows + `docs/tests/index.md` row unchanged.
- [ ] Both cj_test skills' docs + catalog describe the per-test-doc-content model.
- [ ] `validate.sh` green (Checks 15/15a, 24, 26 unaffected); full `test.sh` green (ubuntu CI authoritative); shellcheck clean.

## Not in scope

<!-- Explicit non-goals. -->

- Retiring or changing the family render (`--render-docs`) or its Check 26 freshness gate — the family docs are KEPT as linked units-detail.
- Any categories→units join key or schema change — deliberately avoided (Approach A's dealbreaker).
- Re-categorizing any test or moving any test script (e.g. moving `eval.md` into a "ci subdir") — nothing moves under Approach B.
- New taxonomy values — the category taxonomy from F000074/F000075 is unchanged.
- The `docs/tests/index.md` INDEX row content — unchanged.
