## Problem Statement

A user has a partially-completed work-item dir (scaffolded, maybe implemented,
maybe QA'd, maybe shipped). They want one command to resume from the correct
phase. Today they must read the TRACKER, figure out where they are, then pick
the right sub-skill manually (`/CJ_implement-from-spec`, `/CJ_qa-work-item`,
`/ship`, `/land-and-deploy`).

Branch(f) of `/CJ_run` accepts a work-item-dir, reads the TRACKER phase state,
and dispatches the right sub-pipeline. Branch(g) (S000038) calls into this same
logic after picking a candidate.

## Mental Model

```
   /CJ_run <work-item-dir>     OR    /CJ_run (no args) → Branch(g) picks
                                                          → calls Branch(f)
              │
              v
   ┌──────────────────────────────────┐
   │ Branch(f) — Phase Detection      │
   │                                  │
   │ Read TRACKER:                    │
   │   IMPL_GATE  ([x] or [ ])        │
   │   QA_GATE    ([x] or [ ])        │
   │   PR_URL     (frontmatter pr:)   │
   │                                  │
   │ Resolve MODE:                    │
   │   IMPL=0 → impl_qa_ship          │
   │   QA=0   → qa_ship               │
   │   PR= "" → ship                  │
   │   PR= URL → gh check state:      │
   │     MERGED → already_shipped     │
   │     OPEN   → open_pr             │
   │     else   → pr_unknown_state    │
   └──────────────────────────────────┘
              │
              v
       Dispatch by MODE
```

## Acceptance Criteria

### Story #1: Work-item-dir input detection [core]

```
GIVEN a directory at `work-items/features/ops/F000017_.../S000038_.../`
WHEN  I run `/CJ_run work-items/features/ops/F000017_.../S000038_.../`
THEN  Step 1 of run.md identifies it as work-item-dir mode (Branch f)
AND   reads the TRACKER for phase state
```

### Story #2: All 6 MODE values resolve correctly [core]

```
GIVEN 6 fixture work-items, each in one of {impl_qa_ship, qa_ship, ship, open_pr, already_shipped, pr_unknown_state}
WHEN  Branch(f) reads each
THEN  the resolved MODE matches the expected MODE for each fixture
```

### Story #3: impl_qa_ship dispatch [core]

```
GIVEN a work-item with Phase 1 done AND IMPL_GATE = [ ]
WHEN  Branch(f) dispatches
THEN  it invokes CJ_personal-pipeline --work-item-dir "$WORK_ITEM_DIR"
AND   follows up with /ship and /land-and-deploy after pipeline returns
```

### Story #4: qa_ship dispatch [core]

```
GIVEN a work-item with IMPL_GATE = [x] AND QA_GATE = [ ]
WHEN  Branch(f) dispatches
THEN  it invokes /CJ_qa-work-item "$WORK_ITEM_DIR"
AND   follows up with /ship + /land-and-deploy after QA passes
```

### Story #5: ship dispatch [core]

```
GIVEN a work-item with IMPL_GATE = [x] AND QA_GATE = [x] AND no PR URL in TRACKER
WHEN  Branch(f) dispatches
THEN  it invokes /ship
AND   follows up with /land-and-deploy
```

### Story #6: open_pr handling [core]

```
GIVEN a work-item with PR URL set AND `gh pr view` returns state=OPEN
WHEN  Branch(f) dispatches
THEN  it prints "PR already open at $PR_URL. Run /land-and-deploy to merge."
AND   exits 0 without invoking ship
```

### Story #7: already_shipped idempotency [core]

```
GIVEN a work-item with PR URL set AND `gh pr view` returns state=MERGED
WHEN  Branch(f) dispatches
THEN  it prints "Already shipped. Nothing to do."
AND   exits 0
```

### Story #8: gh-offline graceful degradation [resilience]

```
GIVEN a work-item with PR URL set AND `gh` is offline / unauthenticated
WHEN  Branch(f) tries to read PR state
THEN  PR_STATE_SCAN = "UNKNOWN"
AND   MODE = "pr_unknown_state"
AND   AUQ asks the user how to proceed
```

### Story #9: Branch(g) → Branch(f) integration [integration]

```
GIVEN Branch(g) has picked a candidate
WHEN  it calls into Branch(f) with the candidate path
THEN  phase detection runs as if the user passed the path directly
AND   dispatch behavior is identical to direct invocation
```

### Story #10: pr_unknown_state AUQ [core]

```
GIVEN gh returns state=CLOSED (without merge)
WHEN  Branch(f) hits the default case
THEN  AUQ presents options like {retry-ship, treat-as-merged, abort}
AND   no auto-dispatch happens without user confirmation
```

### Story #11: MODE logged to telemetry [observability]

```
GIVEN any Branch(f) invocation
WHEN  the dispatch completes (or aborts)
THEN  CJ_run.jsonl has an entry with `mode: <MODE>` and outcome
```

## Architecture

```
   /CJ_run <work-item-dir>
            │
            v
   ┌────────────────────────────────────────────┐
   │ Step 1.f of run.md: Branch(f) detection    │
   │                                            │
   │   Validate: dir exists, has *_TRACKER.md   │
   │   Locate TRACKER (first match)             │
   └────────────────────────────────────────────┘
            │
            v
   ┌────────────────────────────────────────────┐
   │ Step 1.f.1: Read phase state               │
   │                                            │
   │   IMPL_GATE = grep "Todos section..." | -c │
   │   QA_GATE   = grep "Acceptance criteria..."│
   │   PR_URL    = grep -E "^pr:|^PR:" | awk    │
   └────────────────────────────────────────────┘
            │
            v
   ┌────────────────────────────────────────────┐
   │ Step 1.f.2: Resolve MODE                   │
   │                                            │
   │   case ladder: impl/qa/ship/open/merged/?  │
   │   PR state via `gh pr view --json state`   │
   │   UNKNOWN → pr_unknown_state               │
   └────────────────────────────────────────────┘
            │
            v
   ┌────────────────────────────────────────────┐
   │ Step 1.f.3: Dispatch                       │
   │                                            │
   │   case "$MODE" in                          │
   │     impl_qa_ship)    CJ_personal-pipeline  │
   │                      --work-item-dir       │
   │                      then /ship, deploy    │
   │     qa_ship)         /CJ_qa-work-item      │
   │                      then /ship, deploy    │
   │     ship)            /ship; /land-deploy   │
   │     open_pr)         print, exit 0         │
   │     already_shipped) print, exit 0         │
   │     pr_unknown_state) AUQ                  │
   │   esac                                     │
   └────────────────────────────────────────────┘
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `skills/CJ_run/run.md` | this | Modified | New Step 1.f (Branch f); integrates with Step 1.g from S000038 |

### Data Flow

1. User invokes `/CJ_run work-items/features/ops/F000017_.../S000038_.../` OR Branch(g) selects this path.
2. Step 1.f validates: dir exists, contains `*_TRACKER.md`.
3. Locates TRACKER (first matching `*_TRACKER.md` in `WORK_ITEM_DIR`).
4. Reads phase state:
   - `IMPL_GATE`: count of `[x]` next to "Todos section reflects remaining work".
   - `QA_GATE`: count of `[x]` next to "Acceptance criteria verified met".
   - `PR_URL`: first match of `^pr:|^PR:` followed by URL.
5. Resolves MODE via the case ladder.
6. For PR-having states: calls `gh pr view "$PR_URL" --json state -q .state` with 2>/dev/null. Failure → "UNKNOWN".
7. Dispatches the matched sub-pipeline. Telemetry write at end (success or error).

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Gate-string matching | verbatim string match | regex/fuzzy match | Tracker template is canonical; exact match is safest contract |
| PR-state failure | fall to `pr_unknown_state` | abort | Offline use case is valid; let user decide what to do |
| MODE granularity | 6 distinct modes | 3 (in-progress/ready-to-ship/done) | Real states branch differently; coarse grouping hides bugs |
| Branch(f) ↔ Branch(g) | Branch(g) calls Branch(f) | duplicate phase logic | DRY; one source of truth for phase detection |
| Telemetry surface | log MODE + outcome | log only outcome | MODE is the most diagnostic dimension when triaging |

## Open Questions

| Question | Next check |
|----------|-----------|
| If `gh pr view` returns DRAFT, treat as OPEN or unknown? | Pick at impl time; lean toward OPEN with a print clarifier |
| Should `pr_unknown_state` AUQ default to "abort" or "retry-ship"? | Default to "abort" — safer; user-initiated retry is one extra step |
| Does Branch(f) need a `--force` flag for `already_shipped` re-ship? | Out of scope for v0.2; if needed, scaffold a separate story |
