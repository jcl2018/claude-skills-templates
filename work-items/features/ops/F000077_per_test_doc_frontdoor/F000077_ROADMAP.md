## Scope

Make `docs/tests/<category>/<name>.md` the authoritative What/How/Why front door
per test ‚Äî required What it is / How to run / Explanation (why), enforced as a
GENERAL rule in the portable `spec/test-spec.md` (byte-identical to the
`scripts/test-spec.sh --seed` heredoc), checked by `test-spec.sh --check-structure`
and `/CJ_test_audit`, and surfaced by `/CJ_test_run`. The flat family docs
(`docs/tests/<family>.md`, generated by `--render-docs`) are KEPT as the linked
units-detail drill-down. No schema change, no categories‚Üíunits join key, no
Check 26 rewrite ‚Äî a single, low-risk feature.

## Non-Goals

- Retiring or altering the family render or its Check 26 freshness gate ‚Äî kept as-is.
- A categories‚Üíunits join key or any test-spec schema change ‚Äî deliberately avoided.
- Re-categorizing any test or moving any test script ‚Äî nothing moves under Approach B.
- New taxonomy values or `docs/tests/index.md` INDEX row changes ‚Äî unchanged.

## Success Criteria

<!-- Bulleted, measurable outcomes observable from the outside. -->

- [ ] The general what/how/why rule is present in `spec/test-spec.md` AND the `--seed` heredoc, with `cmp -s` byte-identity green.
- [ ] All 7 category docs (`validate`, `suite`, `test-deploy`, `windows`, `windows-deploy`, `goal-task-eval`, `e2e-local`) carry the three sections + a family-doc link.
- [ ] `test-spec.sh --check-structure` content check is green, and reports inactive when no `categories:` axis exists.
- [ ] `/CJ_test_audit` Stage 1 + Stage 2 are green on the docs; `/CJ_test_run <name>` surfaces the per-test doc's How-to-run.
- [ ] `validate.sh` green (Checks 15/15a, 24, 26 unaffected); full `test.sh` green; shellcheck clean.
