## Problem Statement

The workbench maintainer pays a slow `windows-latest` `test-deploy.sh` run on
every PR, and the two skills that consume the category test contract
(`/CJ_test_audit`, `/CJ_test_run`) cannot express or select a push-vs-nightly
cadence because the V1 taxonomy `{workflow, CI}` has no cadence dimension. The
maintainer wants faster PRs without losing a Windows signal, and wants the
cadence to be a first-class, machine-checkable fact.

## Mental Model

The change threads ONE noun — the cadence — through the taxonomy by making it the
category name. V1 `{workflow, CI}` becomes V2 `{workflow, CI-push, CI-nightly}`:

```
        V1                         V2
  ┌───────────┐            ┌───────────────┐
  │ workflow  │            │ workflow      │  (unchanged: goal-task-eval, e2e-local)
  ├───────────┤            ├───────────────┤
  │ CI        │  ───────►  │ CI-push       │  (every push/PR: validate, suite,
  │           │            │               │   test-deploy, windows[-smoke])
  │           │            ├───────────────┤
  │           │            │ CI-nightly    │  (nightly: windows-deploy =
  └───────────┘            └───────────────┘   test-deploy.sh on windows-latest)
```

Because the category name IS the cadence, `--category CI-push` / `--category
CI-nightly` is the whole selection API — no new flag. The real CI config mirrors
the taxonomy: `windows.yml` = the `CI-push` Windows smoke, `windows-nightly.yml` =
the `CI-nightly` Windows deploy suite.

## Acceptance Criteria

<!-- Given/When/Then. Each #n header matches the # column in Requirements. -->

### Story #1: Portable taxonomy V2 in both copies [core]

```
GIVEN spec/test-spec.md and the scripts/test-spec.sh --seed heredoc
WHEN  the closed category set is bumped to {workflow, CI-push, CI-nightly} in both
THEN  test-spec.sh --validate passes AND `cmp -s <(test-spec.sh --seed) spec/test-spec.md` is byte-identical (tests/test-spec.test.sh green)
```

### Story #2: Every parser enum site recognizes V2 [core]

```
GIVEN scripts/test-spec.sh with a --validate enum and three --check-structure loops
WHEN  every category-enum site is updated to the V2 set
THEN  a V2 category (e.g. CI-nightly) is accepted by --validate and processed by --check-structure with no "outside taxonomy" rejection
```

### Story #3: --check-structure folders derive from declared categories [resilience]

```
GIVEN an overlay declaring only a subset of categories (e.g. workflow + CI-push)
WHEN  test-spec.sh --check-structure runs
THEN  it requires tests/<category>/ + docs/tests/<category>/<name>.md ONLY for the DISTINCT declared categories, not a hardcoded three-set (no forced empty tests/CI-nightly/)
```

### Story #4: Overlay rows re-keyed + windows-deploy added [core]

```
GIVEN spec/test-spec-custom.md categories: axis with CI rows
WHEN  the CI rows are re-keyed to CI-push and a windows-deploy CI-nightly row is added
THEN  test-spec.sh --list-categories shows validate/suite/test-deploy/windows under CI-push AND windows-deploy under CI-nightly
```

### Story #5: Folder/doc rename + doc-spec registry rewrite in the code commit [integration]

```
GIVEN tests/CI/ + docs/tests/CI/ + spec/doc-spec-custom.md rows for docs/tests/CI/*.md
WHEN  they are renamed to CI-push (+ new CI-nightly dirs/docs) AND the doc-spec rows are rewritten in the same implementation commit
THEN  validate.sh Check 15/15a passes (every declared doc exists AND every on-disk doc is declared) AND the lowercase docs/tests/ci.md family render is untouched
```

### Story #6: Runner selects V2 categories [usability]

```
GIVEN scripts/test-run.sh with a hardcoded {workflow, CI} enum + error string
WHEN  the enum + error string are updated to the V2 set
THEN  test-run.sh --category CI-push and --category CI-nightly each produce a correct --dry-run plan selecting that category's tests, honoring cost tiers
```

### Story #7: Real CI workflows implement the split [core]

```
GIVEN .github/workflows/windows.yml running windows-smoke.sh AND test-deploy.sh on PR+push
WHEN  windows.yml is trimmed to windows-smoke.sh only AND windows-nightly.yml is added (test-deploy.sh on windows-latest, on schedule + workflow_dispatch, permissions contents:read, concurrency guard, defaults shell:bash)
THEN  a PR triggers only the fast Windows smoke AND the slow deploy suite runs on the nightly schedule / on dispatch
```

### Story #8: Skills + convention docs document V2 [observability]

```
GIVEN the cj_test skills' SKILL.md + USAGE.md and CLAUDE.md describing V1 {workflow, CI}
WHEN  they are updated to V2 {workflow, CI-push, CI-nightly} + the cadence selection
THEN  CJ_test_audit docs describe the --check-structure category set + new folders, CJ_test_run docs describe --category CI-push|CI-nightly, and CLAUDE.md's taxonomy + Windows-CI references read V2
```

### Story #9: Tests cover V2 and the updated enum-rejection [resilience]

```
GIVEN tests/test-spec.test.sh and tests/test-run.test.sh on the V1 taxonomy
WHEN  they are updated to V2 (incl. derive-from-declared folder logic, CI-push/CI-nightly selection, the --seed byte-identity assertion, and the updated negative enum-rejection message test)
THEN  the full test.sh suite passes green with the V2 taxonomy
```

### Data Flow

<!-- How data moves through the system for the primary use case. -->

1. The V2 closed set is defined once in the portable seed (`spec/test-spec.md` == `scripts/test-spec.sh --seed`).
2. `scripts/test-spec.sh` parses the seed + the `spec/test-spec-custom.md` overlay; every enum site validates against the V2 set; `--check-structure` derives required `tests/<category>/` + `docs/tests/<category>/` folders from the DISTINCT declared `categories:` rows.
3. The overlay declares each test under its cadence category; `--list-categories` and `--render-docs` project that into `docs/tests/index.md` + per-test docs; `spec/doc-spec-custom.md` declares those doc paths so Check 15/15a passes.
4. `/CJ_test_run` (via `scripts/test-run.sh --category CI-push|CI-nightly`) maps the requested category → its tests' commands and runs them (cost-tiered), producing a `--dry-run` plan or a real run.
5. The real GitHub Actions workflows mirror the cadence: `windows.yml` fires the `CI-push` smoke on PR/push; `windows-nightly.yml` fires the `CI-nightly` deploy suite on schedule/dispatch.
6. `/CJ_test_audit --check-structure` cross-checks the declared cadence categories against the on-disk folders + docs; the convention docs + skill docs describe the model for humans.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Cadence model | Expand the taxonomy (`CI` → `CI-push`/`CI-nightly`) | An orthogonal additive `cadence: [push\|nightly]` field | Operator preference: cadence IS the category, so `--category` is the whole selection API — no new flag; the vocabulary matches the mental model. |
| `--check-structure` folders | Derive from the DISTINCT declared categories | Hardcode the three-set `{workflow, CI-push, CI-nightly}` | Hardcoding would force an empty `tests/CI-nightly/` on THIS repo + every consumer that upgrades the seed. |
| doc-spec rows timing | Rewrite in the implementation commit | Defer to Step 5.5 doc-sync | The rows are contract entries; deferring orphans declared rows + undeclares new docs → Check 15/15a hard fail. |
| `test-deploy` vs `windows-deploy` | Keep both as two rows (push + nightly) | Collapse into one row | Same script, two distinct CI contexts (platform + cadence); two rows read clearest and set up the deferred `platform:` field. |
| PR Windows signal | Keep the fast `windows-smoke.sh` on PR | Move ALL Windows work to nightly | Protects the per-PR Windows signal; only the slow deploy suite is deferred to nightly. |
| Multi-value cadence | Single category per test (name = cadence) | A single-value `cadence:` field (Approach C) | A single-value field can't cleanly represent a push+nightly test; the taxonomy split is unambiguous per test. |

## Open Questions

| Question | Next check |
|----------|-----------|
| Does `goal-task-eval` (the nightly eval) move to `CI-nightly`? | Recommend no — it is a `workflow`-kind test scheduled by `eval-nightly.yml` + its `paid` tier; revisit only if the operator wants cadence to subsume kind. |
| Backward-compat alias for `CI`? | Recommend no — the workbench is the only real consumer; flag before implement only if a consumer repo is known to declare `category: CI`. |
| Bare `push`/`nightly` names vs the `CI-` prefix? | Chose `CI-` to preserve the "deploy-gate" semantic and keep `workflow` distinct; revisit at approval only if bare names are preferred. |
| How do the two per-test docs explain the identical local command for `test-deploy` + `windows-deploy`? | Implement — each doc states the reason (platform + cadence differ; no `platform:` field yet), so the redundant local run is documented not accidental. |
