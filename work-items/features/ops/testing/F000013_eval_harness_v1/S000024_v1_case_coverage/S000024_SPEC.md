## Problem Statement

The V1 eval harness needs 6–10 cases across `personal-workflow` and `system-health` to satisfy the design's success criteria. Without these cases, the runner that S000023 ships is just plumbing. The maintainer (chjiang) needs cases that exercise: (a) the regression that motivated the harness (S000022 Step 18 traceability), (b) at least one valid-input baseline per skill (proves the skill correctly reports PASS), (c) at least one each of the most-common-failure modes per skill (frontmatter drift, lifecycle drift, etc).

## Mental Model

Each case is a directory under `tests/eval/<skill>/<case>/` containing:

- **`prompt.md`** — the prompt sent to Claude. Opens with the explicit `/skill-name ...` invocation; closes with the JSON contract instruction.
- **`fixture/`** — files seeded into the scratch tmpdir before invocation.
- **`expected.schema.json`** — JSON Schema asserting the shape of the model's `.result` output.

V1 case inventory:

| # | Case | Skill | Type | What it validates |
|---|------|-------|------|-------------------|
| 1 | check-step18-faithful-comma-split | personal-workflow | regression | S000022: Claude correctly comma-splits multi-AC traceability cells per the spec |
| 2 | check-passing-feature | personal-workflow | baseline | Canonical valid feature work-item produces overall=PASS, all checks=PASS |
| 3 | check-missing-frontmatter | personal-workflow | failure-detection | Malformed frontmatter is detected and surfaced in the output JSON |
| 4 | check-lifecycle-drift | personal-workflow | failure-detection | Tracker missing required lifecycle gates is detected |
| 5 | check-traceability-mismatch | personal-workflow | failure-detection | TEST-SPEC AC column referencing a SPEC story # that doesn't exist is surfaced (optional 5th case if drift pressure manageable) |
| 6 | report-clean-system | system-health | baseline | Healthy `~/.claude/skills/` state produces overall=PASS |
| 7 | report-with-issues | system-health | failure-detection | A deliberately drifted state surfaces specific issues in JSON |

## Acceptance Criteria

### Story #1: S000022 regression case exists and catches the bug [core]

```
GIVEN `tests/eval/personal-workflow/check-step18-faithful-comma-split/` exists
AND   the fixture has a tracker with multi-AC cells like `AC-1, AC-2, AC-3` and `AC-2, AC-4`
AND   `expected.schema.json` asserts ac_coverage shape with each AC mapped covered/uncovered
WHEN  the maintainer runs `bash scripts/eval.sh personal-workflow check-step18-faithful-comma-split` on a test branch with the S000022 fix reverted
THEN  the case FAILS — schema validation surfaces the comma-split miss
AND   on `main` (with fix in place), the case PASSES
```

### Story #2: Baseline case proves PASS handling [core]

```
GIVEN `tests/eval/personal-workflow/check-passing-feature/` exists with a canonical valid feature fixture
WHEN  the case runs
THEN  the model emits JSON with overall=PASS and all checks=PASS
AND   the schema validates and the case PASSES
```

### Story #3: Frontmatter drift case [core]

```
GIVEN `tests/eval/personal-workflow/check-missing-frontmatter/` exists with a fixture missing required frontmatter fields
WHEN  the case runs
THEN  the model emits JSON with overall=FAIL and the frontmatter check FAIL
AND   the schema validates and the case PASSES
```

### Story #4: Lifecycle drift case (4th personal-workflow case) [core]

```
GIVEN `tests/eval/personal-workflow/check-lifecycle-drift/` exists with a tracker missing required lifecycle gate rows
WHEN  the case runs
THEN  the model emits JSON identifying the missing gates
AND   the schema validates and the case PASSES
```

### Story #5: System-health baseline case [core]

```
GIVEN `tests/eval/system-health/report-clean-system/` exists with a fixture representing healthy `~/.claude/skills/` state
WHEN  the case runs
THEN  the model emits JSON with overall=PASS
AND   the schema validates and the case PASSES
```

### Story #6: System-health degraded case [core]

```
GIVEN `tests/eval/system-health/report-with-issues/` exists with a fixture representing a drifted state (missing skill, mismatched manifest, etc.)
WHEN  the case runs
THEN  the model emits JSON with overall=DEGRADED and at least one specific issue surfaced
AND   the schema validates and the case PASSES
```

### Story #7: Full suite green on main [core]

```
GIVEN all V1 cases authored
WHEN  `bash scripts/eval.sh` runs on a clean main checkout
THEN  exit code is 0
AND   stdout shows PASS for every case (count between 6–10)
```

### Story #8: S000022 caveat documented in case [usability]

```
GIVEN `tests/eval/personal-workflow/check-step18-faithful-comma-split/prompt.md` exists
WHEN  a future contributor reads the prompt
THEN  they encounter an in-line note: "This case tests that Claude faithfully executes the comma-split spec as written in `check.md`. It does NOT test the comma-split logic in isolation — that's covered (or scheduled) under V2's parser-extraction work."
```

### Data Flow

For each case authored:

1. Write the prompt.md mandating an explicit `/skill-name` invocation + JSON-only output instruction.
2. Construct the fixture/ directory with seeded files matching the case's failure-mode.
3. Hand-write expected.schema.json asserting the JSON shape Claude should emit.
4. Run `bash scripts/eval.sh <skill> <case>` locally.
5. Iterate prompt.md until the model reliably emits matching JSON.
6. For the S000022 case specifically: also verify on a test branch with the parser fix reverted that the case FAILS.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Number of personal-workflow cases | 4–5 | 2 (just S000022 + one baseline) | More cases = better coverage; soft cap of 10 total V1 cases gives room. |
| S000022 caveat placement | In-line in `prompt.md` | Only in design doc | Future contributors reading the case directly need to understand its coverage limit; design docs rot. |
| Schema reuse strategy | Conditional on drift pressure | Always use `$ref` from day one | V1 has only 6–10 cases; up-front consolidation is over-engineering. Lift only when 3+ cases share a fragment. |
| Fixture content for `check-passing-feature` | Existing `fixtures/valid-feature-dir/` if it exists, otherwise hand-author | Always hand-author | Reuse aligns with workbench fixture conventions; new authoring duplicates effort. |
