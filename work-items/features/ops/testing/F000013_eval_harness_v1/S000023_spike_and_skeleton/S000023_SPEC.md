## Problem Statement

The eval harness runner sketch in `F000013_DESIGN.md` is shaped pessimistically against three unverified CLI behaviors: `--plugin-dir` may not load workbench skills directly in `--bare --print` mode; `--json-schema` may not accept the inline JSON syntax shown in `claude --help`; `--json-schema` may not exit-fail on schema mismatch. Until those are empirically resolved, building the runner risks committing to fallback complexity unnecessarily. The maintainer (chjiang) needs Spike 0 results before the skeleton lands, and a first passing case to prove the pipeline works end-to-end.

## Mental Model

Three phases, in order:

1. **Spike 0** (30–60 min): three small `claude --bare -p ...` invocations against a known fixture, observing CLI behavior. Pick loader path (direct, plugin-manifest wrapper, or fake-`$HOME` fallback) and assertion path (exit-code-only or always-pipe-to-`ajv-cli`).
2. **Skeleton** (1–2 hours): write `scripts/eval.sh` + `tests/eval/lib/run-case.sh` + `tests/eval/lib/seed-fixture.sh` + `tests/eval/README.md` per spike outcome.
3. **First case** (1 hour): `tests/eval/personal-workflow/check-flags-missing-lifecycle/` — fixture is a malformed tracker missing lifecycle gates; eval prompt asks Claude to run validation and report the missing gates as JSON; schema asserts shape.

## Acceptance Criteria

### Story #1: Spike 0 findings recorded [core]

```
GIVEN the maintainer runs three small `claude --bare -p ...` invocations against a known-valid fixture
WHEN  Spike 0 completes
THEN  `tests/eval/README.md` contains a "Spike 0 findings" section recording: which loader path works (direct/plugin-manifest/fake-HOME), inline-vs-@file json-schema syntax, exit-fail vs warn-only behavior on schema mismatch
AND   the runner code in `lib/run-case.sh` is shaped to match the chosen paths
```

### Story #2: eval.sh accepts positional filter args [core]

```
GIVEN `scripts/eval.sh` exists
WHEN  the maintainer runs `bash scripts/eval.sh personal-workflow check-flags-missing-lifecycle`
THEN  only that one case runs (not all cases)
AND   `bash scripts/eval.sh personal-workflow` runs all personal-workflow cases
AND   `bash scripts/eval.sh` runs all cases across all skills
```

### Story #3: run-case.sh executes a case end-to-end [core]

```
GIVEN a case directory with prompt.md + fixture/ + expected.schema.json
WHEN  `lib/run-case.sh` is invoked on it
THEN  the fixture is copied to a fresh tmpdir
AND   the tmpdir is `git init`'d
AND   a fresh fake-`$HOME` is created with the skill symlinked into `~/.claude/skills/<skill>`
AND   `claude --bare -p --output-format json --json-schema --plugin-dir --add-dir --no-session-persistence --max-budget-usd 0.15` is invoked with the prompt
AND   the model's `.result` JSON is parsed via `jq -r '.result | fromjson'`
AND   the parsed JSON is validated against `expected.schema.json` (via `ajv-cli` if S0.3 is warn-only)
AND   the case prints `PASS:` or `FAIL:` to stdout, exits 0 or 1
```

### Story #4: xargs -P 4 concurrency is safe [core]

```
GIVEN two distinct cases authored
WHEN  `bash scripts/eval.sh` runs them under `xargs -P 4`
THEN  both pass (or both fail correctly), no fake-`$HOME` corruption
AND   running the same set 5× in a row produces consistent results
```

### Story #5: first case returns PASS [core]

```
GIVEN `tests/eval/personal-workflow/check-flags-missing-lifecycle/` with valid prompt.md + fixture/ + expected.schema.json
WHEN  `bash scripts/eval.sh personal-workflow check-flags-missing-lifecycle` runs
THEN  exit code is 0
AND   stdout contains `PASS: personal-workflow/check-flags-missing-lifecycle`
AND   the model's JSON output identifies the missing lifecycle gates correctly
```

### Story #6: ajv-cli fallback for schema validation [resilience]

```
GIVEN a known-bad case where the model's output violates the schema
WHEN  the runner invokes `claude` and parses `.result | fromjson`
THEN  if claude exit-fails on schema mismatch, the runner reports FAIL via exit code
AND   if claude warn-only's, the runner pipes through `ajv-cli validate -s schema -d output.json` and reports FAIL on validation failure
AND   either way, the case fails with a useful diagnostic on stderr
```

### Story #7: README documents case authoring [usability]

```
GIVEN `tests/eval/README.md` exists
WHEN  a new contributor reads it
THEN  they understand: where cases live, the prompt.md/fixture/expected.schema.json contract, how to run locally with filter args, how to debug a failing case
```

## Architecture

```
                                +----------------------+
                                |  scripts/eval.sh     |
                                |  (top-level runner)  |
                                +----------+-----------+
                                           |
                                           | discovers cases via filesystem
                                           | pre-warms ajv-cli
                                           | xargs -P 4 ...
                                           v
                          +---------------------------------+
                          |  tests/eval/lib/run-case.sh     |
                          |  (per-case execution)           |
                          +---+----------------+--------+---+
                              |                |        |
                              v                v        v
              +----------------+   +-----------+--+   +-+----------------------+
              | seed-fixture   |   | claude --bare|   | ajv-cli (fallback if   |
              | (cp -R + git   |   | --print etc. |   | --json-schema is       |
              |  init + fake-  |   | spawned in   |   | warn-only)             |
              |  $HOME setup)  |   | tmpdir       |   |                        |
              +----------------+   +--------------+   +------------------------+
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `scripts/eval.sh` | `claude-skills-templates` | New | Top-level runner. Discovers cases, pre-warms ajv-cli, runs cases via `xargs -P 4`, aggregates results, exits with status. |
| `tests/eval/lib/run-case.sh` | `claude-skills-templates` | New | Per-case execution. Seeds fixture, sets up fake `$HOME`, spawns `claude`, validates output. |
| `tests/eval/lib/seed-fixture.sh` | `claude-skills-templates` | New | Helper extracted from run-case.sh. Copies fixture dir + `git init`s tmpdir. |
| `tests/eval/README.md` | `claude-skills-templates` | New | Case authoring guide + Spike 0 findings + local invocation docs. |
| `tests/eval/personal-workflow/check-flags-missing-lifecycle/` | `claude-skills-templates` | New | First passing case. Tracker missing lifecycle gates; Claude detects + reports via JSON. |

### Data Flow

1. Maintainer runs `bash scripts/eval.sh [skill] [case]`.
2. `eval.sh` discovers cases (filtered by args) and pre-warms `ajv-cli` via `npx --prefer-offline`.
3. `eval.sh` pipes case list into `xargs -P 4 -L 1 -I '{}' bash run-case.sh '{}' ...`.
4. Each `run-case.sh` invocation: seeds fixture into `$tmpdir`, `git init`s it, creates fake `$HOME` with skill symlink.
5. Each invocation: spawns `HOME=$fake_home claude --bare -p ... --plugin-dir ... --add-dir $tmpdir ...` with the prompt.
6. Output captured; `jq -r '.result | fromjson' > $tmpdir/output.json` extracts the model's JSON.
7. `ajv-cli validate -s schema -d output.json` asserts shape (always, regardless of S0.3 outcome).
8. Per-case PASS/FAIL printed to stdout with case identifier.
9. After xargs completes, `eval.sh` aggregates PASS/FAIL counts, prints summary, exits with appropriate status.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| First-case selection | `check-flags-missing-lifecycle` (genuine reasoning) | `check-step18-faithful-comma-split` (S000022 regression) | Reasoning case proves Claude actually executes the skill; spec-execution case is weaker signal for first-case validation. S000022 lands in S000024. |
| `seed-fixture.sh` separation | Separate file from `run-case.sh` | Inline in `run-case.sh` | Testable in isolation; reused by future authoring helpers. |
| `--max-budget-usd` value | 0.15 per case | 0.50 per case | $0.50 × 10 cases = $5/run worst case — too high for nightly. $0.15 × 10 = $1.50 hits the design's success criterion. |
| `xargs -P 4` parallelism | In V1 from day one | Sequential in V1, parallel in V2 | Without parallelism, V1 wall-clock is 30 min — too slow. `xargs -P 4` is one-line cost in V1 if fake-`$HOME` is per-case. |
| ajv-cli source | `npx --prefer-offline --yes ajv-cli@5` | Globally installed `npm i -g ajv-cli` | npx avoids global install requirement; `--prefer-offline` after pre-warm uses cached version. |
