## Problem Statement

The eval harness only delivers regression-detection value if it runs without manual intervention. Nightly CI on `main` is the V1 mechanism. The first real CI run also closes a load-bearing risk: V1's success criteria (cost ≤ $1.50, wall-clock ≤ 12 min) are estimates derived from a 3-min/case worst case and `--max-budget-usd 0.15` per case. Real numbers may differ. The maintainer (chjiang) needs the workflow + the empirical first-run validation + the TODOS update before V1 can be marked shipped.

## Mental Model

Three deliverables, in order:

1. **Workflow file** (`.github/workflows/eval-nightly.yml`): cron + workflow_dispatch + claude install + run + summary.
2. **First run + observation**: `gh workflow run eval-nightly.yml` manually, observe cost + wall-clock, record in tracker journal.
3. **TODOS update**: mark "Behavioral eval harness (P1, M)" as DONE-V1 with link to F000013 and the V2 trajectory bullets.

Branching decision after step 2: if observed numbers ≤ V1 budget → ship as-is. If 50%+ over → open follow-up to cut cases / tighten prompts before V1 ship.

## Architecture

```
.github/workflows/
└── eval-nightly.yml                   (new)

Workflow job (one job, sequential steps):
  1. uses: actions/checkout@v4
  2. install claude CLI
     (TBD by Spike during S000025: try `npm i -g @anthropic-ai/claude-code`
      or curl install script — pick whichever works in actions/setup-node@v4)
  3. bash scripts/eval.sh
     env: ANTHROPIC_API_KEY=${{ secrets.ANTHROPIC_API_KEY }}
  4. write to GITHUB_STEP_SUMMARY
     (PASS=N FAIL=M + failed case list if any)
  5. (P2) upload-artifact: tests/eval/**/output.json on failure
```

### Data Flow

1. Cron fires (or maintainer triggers via `gh workflow run`).
2. Runner spins up: checkout main → install claude CLI → set `HOME=$RUNNER_TEMP/.../claude-home`.
3. Runner executes `bash scripts/eval.sh`. The script discovers cases under `tests/eval/`, pre-warms `ajv-cli`, runs cases via `xargs -P 4`, aggregates results.
4. Each case spawns its own `claude --bare -p ...` against a fresh tmpdir + fake `$HOME`. Cost capped at `--max-budget-usd 0.15` per case.
5. `eval.sh` exits with status 0 (all PASS) or 1 (one or more FAIL). Workflow status reflects exit code.
6. Workflow writes summary to `$GITHUB_STEP_SUMMARY` with PASS/FAIL counts and any failed case names.
7. (P2) On failure, upload artifacts (per-case `output.json` + claude stdout/stderr) for post-mortem.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Trigger cadence | Cron daily 09:00 UTC | Push to main on skill changes (`paths: skills/**`) | Push-trigger means every commit on a busy day runs evals; cost amplified. Daily catches regressions within 24h, which is acceptable for V1 workbench. Per-PR with paths filter is V2. |
| Timeout | 15 minutes | No timeout | Unbounded cost on stuck runs; 15 min is 25% headroom over success criterion. |
| Failure notification | GitHub Actions UI passive (red ✗) | Email/Discord notification | V1 accepts passive — workbench is single-maintainer; checking GitHub Actions UI is in the maintainer's daily flow. V2 wires email/Discord if appetite. |
| Claude install method | TBD during workflow authoring | Pre-decide | Several install paths (`npm`, curl script, possible action) have different reliability characteristics in GitHub runners; let the workflow author empirically pick. |
| Artifact upload | P2 (only on failure) | Always upload | Always-upload bloats artifact storage; failure-only keeps storage bounded while preserving post-mortem capability. |
