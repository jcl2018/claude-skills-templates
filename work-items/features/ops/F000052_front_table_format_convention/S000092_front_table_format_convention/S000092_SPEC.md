## Problem Statement

`docs/philosophy.md` (~149 lines: 3 `## Principle N` sections + a decision tree)
and `docs/workflow.md` (~636 lines: 3 orchestrators + machinery + utilities) are
long, section-structured human docs. A reader scanning for "which principle /
which workflow do I want" has to skim the whole file — there is no at-a-glance
index at the top. A maintainer adding a principle/workflow has no machine signal
forcing a matching index row, so any such index would silently rot.

This story adds a leading summary table to each of the two docs, registers the
convention in the `doc-spec.md` contract, and enforces it as a **hard,
registry-driven** `validate.sh` gate so the index cannot drift undetected.

## Mental Model

The convention is enforced as a one-way data flow, registry → parser → validator:

```
doc-spec.md (front_table: required field, workbench-local)
        │  read by
        ▼
scripts/doc-spec.sh --list-front-table-docs   (separate awk path)
        │  consumed by
        ▼
scripts/validate.sh Check 20  (assert leading table before first ^## )
        │  gates
        ▼
pre-commit hook + scripts/test.sh + windows-latest Git Bash CI
```

The new field is a workbench-LOCAL registry extension: it lives in the machine
registry OUTSIDE the `DOC-SPEC-COMMON:BEGIN/END` prose markers and is documented
in the Custom prose section, so `doc-spec.md` still satisfies its own "Common
section verbatim from the seed" requirement and `_emit_seed` stays byte-identical.

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. #n matches the Requirements # column. -->

### Story #1: Registry declares front-table-required docs [core]

```
GIVEN doc-spec.md's machine registry
WHEN  the docs/philosophy.md and docs/workflow.md entries gain `front_table: required`,
      their requirement strings are extended with the front-table clause, and a Custom-prose
      note documents the workbench-local field
THEN  the DOC-SPEC-COMMON block + scripts/doc-spec.sh _emit_seed remain byte-identical
      AND `scripts/doc-spec.sh --validate` prints `OK schema_version=1`
```

### Story #2: doc-spec.sh exposes the flag [core]

```
GIVEN the updated registry
WHEN  `scripts/doc-spec.sh --list-front-table-docs` runs
THEN  it prints exactly `docs/philosophy.md` and `docs/workflow.md`, one per line
      AND _parse_registry's 3-column TSV is unchanged (no 4th column)
      AND the existing config unit tests stay green
```

### Story #3: validate.sh Check 20 enforces the convention [core]

```
GIVEN a flagged doc with a leading summary table before its first `^## ` heading
WHEN  `scripts/validate.sh` runs
THEN  Check 20 passes
GIVEN a flagged doc whose leading table has been removed
WHEN  `scripts/validate.sh` runs
THEN  it exits non-zero and prints a `  ERROR:` Check-20 line (inline, two-space indent, not `FAIL:`)
```

### Story #4: both docs open with a leading table [usability]

```
GIVEN docs/philosophy.md and docs/workflow.md
WHEN  Check 20 + Check 15/15a/15b + Check 19 run
THEN  each doc has a leading summary table (a `^\|` row immediately followed by a
      `^\|[ :|+-]*-[ :|+-]*\|$` delimiter row) before the first `^## `
      AND neither table contains any `[FSTD][0-9]{6}` work-item ID
      AND all of Check 15/15a/15b/19 pass
```

### Story #5: tests prove the gate + subcommand [resilience]

```
GIVEN scripts/test.sh and tests/cj-document-release-config.test.sh
WHEN  the suite runs
THEN  the plant-and-restore Check-20 negative test strips a flagged doc's leading
      table, asserts a non-zero validate.sh exit with the `  ERROR:` Check-20 prefix,
      then restores the doc
      AND the unit test asserts a synthetic registry entry WITH `front_table: required`
      is emitted by --list-front-table-docs and one WITHOUT it is not
      AND scripts/test.sh is green end-to-end
```

### Story #6: dependent enumerations stay current [observability]

```
GIVEN the new subcommand and Check 20
WHEN  the PR is assembled
THEN  CLAUDE.md (check list 15/15a/15b/16/17/19 → +20; doc-spec.sh subcommand row),
      architecture.md (Check 20 note + subcommand list), and CJ_document-release
      SKILL.md (subcommand list) are updated
      AND skills/CJ_document-release/USAGE.md `last-updated` is bumped to an ISO-8601
      second-resolution timestamp so the SKILL.md edit doesn't leave USAGE.md flagged (Check 14)
```

### Data Flow

<!-- How data moves through the system for the primary use case. -->

1. A maintainer adds `front_table: required` to a doc's `doc-spec.md` registry entry.
2. `validate.sh` Check 20 calls `doc-spec.sh --list-front-table-docs`, which runs its own awk pass over the extracted yaml and prints each flagged `path`.
3. For each path, Check 20 walks the file from the top with awk, stopping at the first `^## ` heading, looking for a `^\|` line immediately followed by a `^\|[ :|+-]*-[ :|+-]*\|$` delimiter row.
4. If found before the first `^## ` → pass; if the first `^## ` (or EOF) is reached with no table → `  ERROR:` inline + `ERRORS=$((ERRORS+1))`.
5. The pre-commit hook, `scripts/test.sh`, and the `windows-latest` Git Bash CI job all run `validate.sh`, so the gate blocks any drift.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Gate posture | Enforced hard CI gate (Approach C) | Advisory-only requirement strings + tables (Approach A) | Operator escalated from advisory to a hard CI gate so the index cannot silently rot. |
| Gate source | Registry-driven (`front_table` field read by validate.sh) | Hardcode `philosophy.md`/`workflow.md` filenames in validate.sh | Honors Principle 3 (registry is the source of truth); flagging a third doc later is a one-line registry edit, no validator change. |
| Field home | Workbench-local registry extension (C2; outside COMMON markers) | Propagate via the `doc-spec.sh` `_emit_seed` heredoc (Approach B) | Keep the COMMON prose block + seed byte-identical so `doc-spec.md` still satisfies its "verbatim seed" requirement. |
| Parser change | Separate awk path for the new subcommand | Widen `_parse_registry` to a 4th TSV column | Its 3-var `read` consumer would mis-bind the 4th column onto `audit_class` and break the closed-enum gate for every entry. |
| Check error emit | `  ERROR:` inline (Check 15-19 style) | The `fail()` helper (`FAIL:` prefix) | The negative test greps a literal `  ERROR:` prefix. |
| Awk scan bound | Stop at the first `^## ` heading | Whole-file grep for any table | Both docs already contain tables LATER (philosophy.md:136, workflow.md:581); a whole-file grep would yield a false PASS. |
