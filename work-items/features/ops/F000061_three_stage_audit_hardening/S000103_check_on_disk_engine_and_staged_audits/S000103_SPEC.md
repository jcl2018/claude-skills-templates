## Problem Statement

The operator (chjiang, running `/CJ_doc_audit` / `/CJ_test_audit` standalone
or reading the QA checkpoint inside a cj_goal run) cannot trust the F000060
audit verdicts to be EARNED: Stage 1 is prose-described bash that every
executor re-derives (the same-day dogfood hit the zsh word-split gotcha —
two phantom findings and one silently VACUOUS check), Stage 2 verdicts lean
on the invoking session's own beliefs about the docs (resident-context
rubber-stamping, no forced evidence), and there is no Stage 3 at all —
nothing cross-walks a doc's CONTENT against the live repo state. The report
is flat, so even when findings exist the operator cannot separate which
verification layer produced them.

## Mental Model

Three stages, split exactly along the deterministic/judged boundary the
system already has, each with its own findings count and report section:

- **Stage 1 (deterministic — engine):** ONE tested engine call per audit.
  Doc audit: the NEW `doc-spec.sh --check-on-disk` (6 conformance checks
  against the MERGED registry). Test audit: the existing
  `test-spec.sh --validate` + `--check-coverage` (unchanged mechanics).
  Pre-stage failures (engine-unreachable, seed-failure, registry-invalid)
  are deterministic too — they count as STAGE1.
- **Stage 2 (requirement compliance — agent-judged, evidence-forced):**
  quote the declared requirement, decompose into clauses, check each clause
  against actual content, cite the decisive evidence per verdict.
- **Stage 3 (implementation drift — agent-judged, evidence-forced):**
  ground truth FIRST (enumerate the live repo state), judgment second
  (cross-walk each contract doc against it; name every delta).
- **Judge context:** standalone, Stages 2+3 run in ONE fresh-context
  subagent (REQUIRED — D10.2); inside QA (already a subagent) they run
  inline per the same protocols — the nested-subagent wall, documented
  honestly in both SKILL.mds.
- **Report:** per-stage counts (`STAGE1/2/3_FINDINGS=`) + three sections +
  grep-able `stageN/` finding prefixes — a pure EXTENSION of the F000060
  report contract (`DOC_AUDIT:`/`TEST_AUDIT:`, `FINDINGS=`,
  `DOCS_AUDITED=`/`UNITS_AUDITED=`, `seeded:` keep their meaning).

### Story #1: The six-check engine [core]

```
GIVEN the clean workbench after this story lands
WHEN  `bash scripts/doc-spec.sh --check-on-disk` runs
THEN  it prints one line per check (declared-exists, orphans, root-declared,
      human-doc-ids, front-table, views-render) all PASS, CHECKS_RUN=6,
      FINDINGS=0, and exits 0

GIVEN a temp fixture with one seeded violation per check id (missing declared
      doc; orphan in docs/; orphan in spec/ — including a non-self-declaring
      overlay file; undeclared root *.md; work-item ID in a human-doc;
      missing front table; view-table drift)
WHEN  --check-on-disk runs against each fixture
THEN  each violation flips EXACTLY its own `FINDING: stage1/<id>` line and
      exit code 1 — no cross-contamination between check ids

GIVEN a multi-line filename / multi-path result set in any check
WHEN  the engine iterates it
THEN  iteration is `while IFS= read -r` (no word-split re-derivation surface)
```

### Story #2: Engine output + error contract [core]

```
GIVEN a repo with NO doc-spec registry file
WHEN  --check-on-disk runs
THEN  output contains REGISTRY=absent and exit code is 0 (the probe runs
      BEFORE the parse gates — no [doc-sync-no-config] halt for absence)

GIVEN a present-but-invalid registry
WHEN  --check-on-disk runs
THEN  the existing [doc-sync-no-config] posture holds: marker printed, exit 1

GIVEN env overrides (DOC_SPEC_PATH, the REPO_ROOT-equivalent) pointing at a
      temp dir
WHEN  --check-on-disk runs
THEN  it operates entirely on the temp dir (hermetic — same idiom as the
      other subcommands)
```

```

WHEN  Stage 2 judges it
THEN  the requirement is QUOTED, decomposed into clauses, each clause checked
      against the doc's actual content, and the verdict line is exactly one of

      Stage 1 cite Stage 1's output instead of re-deriving
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. Operator invokes `/CJ_doc_audit` standalone (or QA Step 8.6c reads it inline).
2. The skill resolves its engine (repo-local `scripts/doc-spec.sh`, else `~/.claude/_cj-shared/scripts/`), seed-delivers a missing registry as before (`seeded: yes|no`).
3. **Stage 1:** the skill runs `bash "$_DA_ENGINE" --check-on-disk` — the engine probes registry existence itself (absent ⇒ `REGISTRY=absent` exit 0), parses the MERGED registry, runs the six checks, emits per-check lines + `CHECKS_RUN=`/`FINDINGS=`. The skill prints this output verbatim as the stage-1 section; any pre-stage failure becomes a `stage1/engine|seed|registry` finding.
4. **Dispatch:** standalone, the skill spawns ONE fresh-context subagent (Agent tool) with repo root + engine path + Stage-1 report + the stage protocols; in-QA the agent proceeds inline.
5. **Stage 2:** the judge quotes each declared doc's `requirement:`, decomposes clauses, checks content, emits evidence-cited verdicts (FINDING lines only count).
6. **Stage 3:** the judge enumerates ground truth (catalog skills, scripts, workflows, registry family, dirs), cross-walks each contract doc per the playbook, emits `no-drift`/`FINDING: stage3/` verdicts.
7. The skill assembles the per-stage report (headline counts + three sections); `DOC_AUDIT: ok` iff all three counts are 0. `/CJ_test_audit` follows the same flow with its own engines and `UNITS_AUDITED=`.
8. In-pipeline, qa.md Step 8.6 lifts both reports into the AUDIT_FINDINGS block (per-stage shape); the checkpoint AUQ prints it verbatim — zero pipeline edits.
