## Problem Statement

Operators and the agent need a per-skill best-practice doc — a first-stop file that answers "when do I invoke this skill?" "when do I NOT?" "what is the mental model?" "what are the common pitfalls?" "what related skills should I know about?" Today there is no such surface; `skills/{name}/SKILL.md` is agent-execution-instructions and `skills/{name}/DESIGN.md` is developer-rationale (optional, 1/13 adoption). This story builds the new surface: a required `USAGE.md` per routable non-deprecated skill, a new template, and a validate.sh audit that prevents drift.

## Mental Model

Three-doc-per-skill model:

```
skills/{name}/
  SKILL.md     (required, agent runtime)        — execution instructions, loaded into agent context every invocation
  USAGE.md     (required, operator + agent)     — when to invoke, when not, mental model, pitfalls, related skills
  DESIGN.md    (optional, developer)            — design rationale, behavior, decisions, dependencies, security, tests
```

Audit lives at `scripts/validate.sh` Check 13. Audit set derived at run time from `skills-catalog.json` via jq:

```
status != "deprecated" AND (files | length) > 0
```

= 11 routable non-deprecated skills as of 2026-06-01.

## Acceptance Criteria

### Story #1: template exists [core]

```
GIVEN the F000032 PR's HEAD
WHEN  I `ls templates/doc-SKILL-USAGE.md`
THEN  the file exists and contains exactly the five required H2 headings (`^## When to use$`, `^## When NOT to use$`, `^## Mental model$`, `^## Common pitfalls$`, `^## Related skills$`) and DESIGN.md-shaped frontmatter (skill-name, version, status, created, last-updated)
```

### Story #2: 11 USAGE.md files exist [core]

```
GIVEN the F000032 PR's HEAD
WHEN  I iterate `jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json`
THEN  for each of the 11 names, `skills/{name}/USAGE.md` exists
```

### Story #3: each USAGE.md has all five required H2 headings [core]

```
GIVEN the F000032 PR's HEAD
WHEN  I `grep -qE '^## When to use$' skills/{name}/USAGE.md` (and the other four headings) for each of the 11 skills
THEN  every grep matches exactly once per file
AND   each section contains substantive content distilled from SKILL.md + rules/skill-routing.md (not empty placeholders)
```

### Story #4: validate.sh Check 13 catches drift [observability]

```
GIVEN a temporarily-deleted skills/CJ_suggest/USAGE.md
WHEN  I run `./scripts/validate.sh`
THEN  the script exits non-zero with an ERROR line naming `skills/CJ_suggest/USAGE.md missing`
AND   restoring the file makes validate.sh exit 0 again
```

### Story #5: PHILOSOPHY.md documentation surfaces section [usability]

```
GIVEN the F000032 PR's HEAD

THEN  there is a `## Documentation surfaces` heading placed between `## Key patterns and conventions` and `## Decision tree`
```

### Story #6: decision-tree USAGE links [usability]

```
GIVEN the F000032 PR's HEAD
WHEN  I read each active-skill entry under `doc/PHILOSOPHY.md ## Decision tree`
```

### Story #7: CLAUDE.md updates [usability]

```
GIVEN the F000032 PR's HEAD
WHEN  I read `CLAUDE.md`
THEN  the Skill directory structure block lists `USAGE.md          # required, ...` between `SKILL.md` and `*.md`
AND   Creating a new skill has a step instructing authors to create USAGE.md from `templates/doc-SKILL-USAGE.md`
AND   the DESIGN.md step is rephrased to clarify DESIGN.md stays optional
```

### Story #8 + #9: validate + test green [resilience]

```
GIVEN the F000032 PR's HEAD
WHEN  I run `./scripts/validate.sh && ./scripts/test.sh`
THEN  both exit 0 (validate.sh with 0 errors / 0 warnings)
```

### Story #10: audit set computed, not hard-coded [core]

```
GIVEN I add a new test entry to `skills-catalog.json` with status "experimental" and a non-empty `files` array
WHEN  I run `./scripts/validate.sh`
THEN  Check 13 ERRORs that `skills/{new-name}/USAGE.md` is missing
AND   removing the new entry makes validate.sh exit 0 again
```

## Architecture

```
+--------------------------------------------------------------+
| scripts/validate.sh                                          |
|   ... existing checks 1-12 ...                               |
|   Check 13 (NEW):                                            |
|     audit_set := jq from skills-catalog.json                 |
|     for name in audit_set:                                   |
```

### Data Flow

1. Operator (or pre-commit hook, or CI) runs `./scripts/validate.sh`.
2. Check 13 invokes `jq -r '.[] | select(.status != "deprecated") | select((.files | length) > 0) | .name' skills-catalog.json` to derive the audit set.
3. For each name, Check 13 verifies `skills/{name}/USAGE.md` exists.
4. For each existing USAGE.md, Check 13 runs five line-anchored greps (`^## When to use$`, etc.). Any missing → ERROR.
5. ERRORS accumulate into validate.sh's count; non-zero exit on any ERROR.
6. Operator fixes the violations and re-runs.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| File class | New `USAGE.md` per skill (Approach B) | Lift DESIGN.md from optional to required (Approach A); required `## Philosophy` section inside each SKILL.md (Approach C) | A would force 12 backfills of a design-rationale doc shape the user didn't ask for. C grows every SKILL.md with operator-facing prose, paying a per-turn token cost on every agent invocation. B is the right semantic match. |
| Audit predicate | `status != "deprecated"` AND non-empty `files` | F000030's `status == "active"` | F000030 gates ship-stable decision-tree placement. F000032 audits routability — operators route to `experimental` skills today. |
| Deployment | In-repo only | Deploy USAGE.md to `~/.claude/skills/{name}/USAGE.md` via skills-deploy | USAGE.md is human-reading. Agent has SKILL.md at runtime; USAGE.md doesn't need delivery. Less to deploy = less drift. |
| Audit severity | ERROR | WARN | F000030 sets ERROR for missing decision-tree entries. WARN would replay F000030's 1/13 adoption decay. |
| Grep anchoring | Line-anchored (`^## When to use$`) | Substring (`grep -F "## When to use"`) | Substring would falsely pass on USAGE.md whose body quotes the required headings inside a code fence. |
| Frontmatter audit | NOT audited | Audit frontmatter shape too | Over-constrains a low-value surface. Template ships with frontmatter for visual consistency; that's enough. |
| Decomposition | Single user-story | Multiple stories (e.g., template / backfills / audit as separate) | Atomic-commit ordering: pre-commit hook blocks intermediate state where Check 13 exists but backfills don't. Splitting adds bookkeeping without splitting risk. |
