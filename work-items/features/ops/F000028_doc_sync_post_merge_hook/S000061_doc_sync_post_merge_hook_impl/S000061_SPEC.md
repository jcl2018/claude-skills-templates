## Problem Statement

Workbench developers (and the three cj_goal orchestrator skills) repeatedly ship code that lands on `main` without a corresponding update to README/ARCHITECTURE/CLAUDE.md/CONTRIBUTING.md. The docs drift silently. Existing solution surfaces (the three cj_goal skills, `/document-release`) are either misaligned (PR-stop vs post-deploy) or operator-initiated only. We need a passive trigger that fires whenever main moves and reminds the operator to run `/document-release` — without spawning Claude inside the user's git operation, and without touching the three cj_goal skill files.

## Mental Model

The hook lives at the **convergence point of all three pipelines**: the moment `main` moves in the operator's local checkout. Three pipelines, one trigger:

```
/cj_goal_defect  --> /land-and-deploy --> remote merge on GitHub
                                                |
/CJ_goal_investigate --> /land-and-deploy --> remote merge on GitHub
                                                |
/cj_goal_feature --> /ship (PR-stop) --> human review --> human merges on GitHub
                                                |
                                                v
                                        operator runs `git pull` on main
                                                |
                                                v
                                        post-merge hook fires
                                                |
                                                v
                                        triggers doc-sync trigger block
                                                |
                                                v
                                        writes marker -> next Claude session
                                          runs /document-release
```

Two hooks (`post-merge` + `post-rewrite`) share the same trigger block body. They cover the two main-syncing flows (`git pull` and `git pull --rebase`). `git reset --hard` is uncoverable; documented gap.

## Acceptance Criteria

### Story #1: setup-hooks.sh installs both hooks [core]

```
GIVEN a fresh clone of claude-skills-templates with no installed hooks
WHEN  the operator runs `./scripts/setup-hooks.sh`
THEN  the script creates `.git/hooks/post-merge` AND `.git/hooks/post-rewrite`
AND   both files contain the literal `# doc-sync trigger block` comment (greppable)
AND   both files contain the literal `# Auto-installed by scripts/setup-hooks.sh` sentinel
AND   the post-merge file also contains the existing D000013 skills-deploy section
AND   the post-merge file also contains the existing F000011 lifecycle-gate section
```

### Story #2: main-moving merge writes valid marker [core]

```
GIVEN a checkout on main with a previous HEAD recorded in `.doc-sync-last-head`
AND   a feature branch with non-doc changes (e.g., `scripts/foo.sh` modified)
WHEN  the operator runs `git merge --no-ff feat-branch` on main (simulating a `git pull` of a merged PR)
THEN  the hook fires
AND   writes `~/.gstack/doc-sync-pending/<repo-slug>.json` atomically (via `mktemp` + `mv`)
AND   the marker's `head_sha` equals `git rev-parse HEAD` (post-merge SHA)
AND   the marker's `diff_base` is a valid tree-ish (the previously recorded SHA, or `HEAD^`, or the empty-tree hash fallback)
AND   the marker's `repo` equals `basename $(git rev-parse --show-toplevel)`
AND   the marker's `main_moved_at` is ISO-8601 UTC (e.g., `2026-05-30T20:59:15Z`)
AND   the marker's `changed_files` is a positive integer
```

### Story #3: same-HEAD re-run is NO-OP [resilience]

```
GIVEN the hook just ran and `.doc-sync-last-head` matches `git rev-parse HEAD`
WHEN  the operator runs the hook again on the same HEAD (e.g., re-pulls a no-op `git pull`)
THEN  the hook exits 0 silently
AND   no new marker is written (mtime of existing marker unchanged)
AND   no stderr output is emitted
```

### Story #4: doc-only merge skips marker [resilience]

```
GIVEN a checkout on main and a feature branch whose ONLY changed files match the triviality regex (README.md, CHANGELOG.md, CLAUDE.md, CONTRIBUTING.md, ARCHITECTURE.md, or docs/*)
WHEN  the operator runs `git merge --no-ff feat-branch` without `DOC_SYNC_FORCE=1`
THEN  the hook fires
AND   detects only doc changes
AND   skips the marker write
AND   emits `[doc-sync] main moved but only docs changed; skipping /document-release.` to stderr
AND   still updates `.doc-sync-last-head` to the new HEAD (so subsequent runs are no-ops)
AND   exits 0
```

### Story #4b: DOC_SYNC_FORCE=1 overrides triviality filter [resilience]

```
GIVEN the same setup as Story #4
WHEN  the operator runs the merge with `DOC_SYNC_FORCE=1 git merge --no-ff feat-branch`
THEN  the hook fires
AND   writes the marker despite the doc-only diff
```

### Story #5: initial-commit fallback to empty-tree diff base [resilience]

```
GIVEN a single-commit repo with no `.doc-sync-last-head` file AND no `HEAD^` resolvable
AND   a feature branch with one commit (non-doc changes) merged into main
WHEN  the hook fires
THEN  `_DIFF_BASE` falls back to the empty-tree hash (`git hash-object -t tree /dev/null`)
AND   the marker is written with that fallback as `diff_base`
AND   the marker is otherwise valid (head_sha, repo, etc. correct)
```

### Story #6: post-rewrite covers rebase flow [integration]

```
GIVEN a checkout on main with non-doc changes pulled via rebase
WHEN  the operator runs `git pull --rebase origin main`
THEN  the post-rewrite hook fires (not post-merge)
AND   writes the same marker shape as post-merge would
AND   the marker contents are valid (head_sha, diff_base, repo, etc.)
```

### Story #7: D000013 + F000011 sections coexist [core]

```
GIVEN the new section 3 is appended to the post-merge hook body
WHEN  the operator runs the hook (via any main-moving merge)
THEN  section 1 (D000013 skills-deploy auto-sync) still runs to completion
AND   section 2 (F000011/S000020 lifecycle-gate auto-update) still runs to completion
AND   section 3 (doc-sync trigger) runs after sections 1+2
AND   any failure in section 3 (wrapped in `{ ... } || true`) does NOT prevent sections 1+2 from running on subsequent invocations
AND   the sentinel-aware re-install path in setup-hooks.sh does NOT backup-thrash on the combined body
```

## Architecture

<!-- High-level system design. -->

```
                          ┌────────────────────────────────────────────────────────┐
                          │ scripts/setup-hooks.sh (operator runs once per machine) │
                          └────────────────────────┬───────────────────────────────┘
                                                   │
                                  install_hook post-merge << 'HOOK' ...
                                  install_hook post-rewrite << 'HOOK' ...
                                                   │
                                                   v
        ┌──────────────────────────────┐       ┌──────────────────────────────┐
        │ .git/hooks/post-merge        │       │ .git/hooks/post-rewrite      │
        │ (extended, 3 sections)       │       │ (new, 1 section)             │
        ├──────────────────────────────┤       ├──────────────────────────────┤
        │ 1. D000013 skills-deploy     │       │ doc-sync trigger block       │
        │ 2. F000011 lifecycle-gate    │       │   (same body as post-merge   │
        │ 3. doc-sync trigger block    │       │    section 3)                │
        │ 4. exit 0                    │       └──────────────────────────────┘
        └──────────────────────────────┘
                                                   │
                  (both fire on main-moving syncs) │
                                                   v
                          ┌────────────────────────────────────────────────────────┐
                          │ doc-sync trigger block (shared between both hooks)     │
                          ├────────────────────────────────────────────────────────┤
                          │ 1. Verify HEAD is on main; else exit 0                 │
                          │ 2. Read .doc-sync-last-head; if == HEAD, exit 0       │
                          │ 3. Compute _DIFF_BASE (last-head, HEAD^, or empty-tree)│
                          │ 4. Triviality filter; if doc-only && !FORCE, exit 0   │
                          │ 5. mkdir -p ~/.gstack/doc-sync-pending                │
                          │ 6. Atomic write of marker JSON (mktemp + mv)          │
                          │ 7. Update .doc-sync-last-head                          │
                          │ 8. Echo [doc-sync] stderr note                         │
                          │ 9. exit 0                                              │
                          └────────────────────────┬───────────────────────────────┘
                                                   │
                                                   v
                          ┌────────────────────────────────────────────────────────┐
                          │ ~/.gstack/doc-sync-pending/<repo-slug>.json            │
                          │ { "repo": ..., "head_sha": ..., "main_moved_at": ...,  │
                          │   "diff_base": ..., "changed_files": ... }             │
                          └────────────────────────────────────────────────────────┘
                                                   │
                              (consumed by future cj_goal-marker-pickup AUQ — out of scope here)
                                                   v
                          [ next Claude session surfaces AUQ → runs /document-release ]
```

### Data Flow

1. Operator runs `./scripts/setup-hooks.sh` once per machine. The script edits `.git/hooks/post-merge` (atomic overwrite via mv) to install the combined 3-section body. It also writes `.git/hooks/post-rewrite` (new file, atomic via mv) with the doc-sync trigger block only.
2. Later, operator runs `git pull` (or equivalent) on `main`. Git invokes `.git/hooks/post-merge` (or `.git/hooks/post-rewrite` for rebase pulls).
3. Hook section 1 (D000013) runs first: re-deploys skills + templates if relevant files changed in the diff.
4. Hook section 2 (F000011) runs second: auto-updates Phase 3 lifecycle gates on touched work-item trackers.
5. Hook section 3 (doc-sync) runs third: branch=main? → idempotency check? → triviality filter? → write marker atomically.
6. Hook exits 0 (whether section 3 succeeded or hit a guard).
7. In the operator's next Claude session, a future cj_goal-marker-pickup AUQ (out of scope here) reads `~/.gstack/doc-sync-pending/<slug>.json` and offers to run `/document-release`. Currently in v1, the marker accumulates silently and the operator can grep `ls ~/.gstack/doc-sync-pending/` manually.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Hook source location | Heredoc body inside `scripts/setup-hooks.sh` | Separate `scripts/hooks/post-merge.sh` checked-in file that `install_hook` copies | Existing convention (D000013, F000011 sections) puts the body in the heredoc. Single source of truth; no copy step that could drift. |
| Marker invocation strategy | Drop marker (option a); next Claude session picks up | Spawn `claude --print /document-release` inline (option b) | Spawning Claude is ~30-60s synchronous inside `git pull` — too disruptive. |
| Marker invocation strategy | Drop marker (option a) | Read-only "doc debt detected" stderr message only (option c) | Too weak — no actual doc updates ever happen. |
| Rebase coverage | Install separate `post-rewrite` hook with same body | `post-merge` only; document rebase-flow as gap | Rebase users would silently miss every doc-sync prompt — too high a usability cost for a 5-line copy. |
| Idempotency state file location | `$(git rev-parse --git-common-dir)/.doc-sync-last-head` | `$(git rev-parse --git-dir)/.doc-sync-last-head` (worktree-local) | This repo's day-to-day work is in worktrees. `--git-common-dir` returns the shared parent `.git/`, so the marker is correctly cross-worktree. |
| Triviality regex anchoring | `^(README\.md|CHANGELOG\.md|...|docs/)` (anchored) | Unanchored alternation | Unanchored matches `READMEs.py` → silently skips a doc-sync that should fire. Anchoring is load-bearing. |
| Test file location | `tests/setup-hooks.test.sh` (flat) | `tests/setup-hooks/<test-name>.sh` (subdir) | Parent design Success Criteria explicitly call out the flat convention. `./scripts/test.sh` picks up `tests/*.test.sh` without configuration. |

| Should the marker JSON include the diff itself (so the consumer doesn't need a git operation)? | Defer to consumer-side design. v1 marker has `diff_base` + `head_sha` — consumer can re-compute the diff. Resolved when the marker-pickup AUQ is built. |
