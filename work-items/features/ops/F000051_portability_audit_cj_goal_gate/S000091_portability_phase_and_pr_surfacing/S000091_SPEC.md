## Problem Statement

cj_goal orchestrator runs (`/CJ_goal_feature`, `/CJ_goal_defect`,
`/CJ_goal_todo_fix`) execute `/CJ_portability-audit` on every commit via the
pre-commit `validate.sh` Check 18, but the check is advisory (exit 0), so a
portability finding never halts a run, and the verdict is buried in `validate.sh`
output instead of being surfaced like the registered-doc verdicts. The workbench
maintainer (the only user of these orchestrators) wants portability honestly
enforced: a dishonest declaration introduced mid-build should stop the run before a
PR is created, and a clean run's verdict should appear in the PR body. The catalog
baseline is clean (`FINDINGS=0`), so a strict halt-on-any-finding gate is green
today and is a free regression ratchet.

## Mental Model

Three layers, all driven by ONE shared phase:

1. **Phase** (`cj-goal-common.sh --phase portability-audit`) — runs the engine
   strict, classifies the result into `ok` / `findings` / `skipped`, emits a
   structured stdout block.
2. **Gate** (in each orchestrator, after Step 5.5 doc-sync, before `/ship`) —
   consumes `PHASE_RESULT`: continue on `ok`/`skipped`, HALT `[portability-red]` on
   `findings`.
3. **Surfacing** (the existing Step 4.6/9.5/5.6 PR-edit step) — splices the green
   `VERDICT_LINE` into the PR body alongside the registered-doc verdicts.

Data flow: implement (changes catalog/skills) → QA green → doc-sync → **portability
gate** (engine strict; clean → verdict scratch + continue; red → HALT) → /ship →
surfacing step splices the verdict into the PR body.

## Acceptance Criteria

<!-- Given/When/Then. Each #n header matches the # column in Requirements. -->

### Story #1: Shared strict-mode phase with structured output [core]

```
GIVEN the current clean catalog (FINDINGS=0)
WHEN  `cj-goal-common.sh --phase portability-audit --mode feature` runs
THEN  it exits 0 and stdout contains PHASE=portability-audit, MODE=feature,
      FINDINGS=0, SKILLS_AUDITED=<n>, a clean VERDICT_LINE, and PHASE_RESULT=ok
```

### Story #2: Fail-soft on engine-absent and dry-run [resilience]

```
GIVEN the portability engine cannot be resolved (sibling + .source both absent)
WHEN  the phase runs
THEN  it exits 0 with PHASE_RESULT=skipped (no halt)

GIVEN any catalog state
WHEN  the phase runs with --dry-run
THEN  it exits 0 with PHASE_RESULT=ok, empty FINDINGS=, and the engine is NOT run
```

### Story #3: Halt-on-red in all three orchestrators [core]

```
GIVEN a cj_goal run whose implement phase introduced a dishonest portability
      declaration (e.g. a `standalone` skill that added a root scripts/*.sh dep)
WHEN  the orchestrator reaches the gate (after Step 5.5 doc-sync, before /ship)
THEN  the phase returns PHASE_RESULT=findings (non-zero) and the orchestrator
      HALTs with [portability-red] / end_state halted_at_portability and a journal
      entry carrying next_action= / resume_cmd= / pr_url=N/A / raw_output_path=,
      and NO PR is created
```

### Story #4: PR-body verdict surfacing on green [observability]

```
GIVEN a clean cj_goal run that passes the gate
WHEN  the orchestrator continues to /ship and reaches the Step 4.6/9.5/5.6
      surfacing step
THEN  `.cj-goal-feature/portability-verdict.md` holds the clean VERDICT_LINE and
      the PR body shows a `### Portability` line alongside the registered-doc
      verdicts (best-effort; a surfacing failure never halts)
```

### Story #5: Resume-safe pure-read gate [resilience]

```
GIVEN a run halted at [portability-red]
WHEN  the orchestrator is resumed
THEN  it restarts at the gate (not at /ship), re-running the pure-read phase; no
      last_completed_phase value was recorded for the gate
```

### Story #6: Repo hard-gates stay green [integration]

```
GIVEN the implemented change
WHEN  `./scripts/validate.sh` and `./scripts/test.sh` run
THEN  both pass, including Check 15b (all 3 CJ_goal_* sections have an ASCII chart
      + 4-bullet Touches block) and the new tests/cj-goal-common-portability.test.sh
```

### Story #7: Dry-run preview mentions the gate [usability]

```
GIVEN an orchestrator invoked with --dry-run
WHEN  it prints its chain-plan preview
THEN  the preview includes a line describing the portability-audit gate
      (halt-on-red, before /ship) and the gate itself runs nothing
```

## Architecture

<!-- High-level design. ASCII diagram for the data flow. -->

```
                cj_goal orchestrator (feature | defect | todo)
                                  |
                  ... QA green -> Step 5.5 doc-sync ...
                                  |
            +---------------------v----------------------+
            |  GATE: cj-goal-common.sh                   |
            |        --phase portability-audit --mode m  |
            |                                            |
            |  resolve_portability_engine():             |
            |    (1) <scriptdir>/cj-portability-audit.sh |
            |    (2) <manifest .source>/scripts/...       |
            |                                            |
            |  PA_OUT=$(PORTABILITY_STRICT=1 bash ENGINE |
            |          2>&1) && PA_RC=0 || PA_RC=$?       |
            |  parse FINDINGS= / SKILLS_AUDITED=          |
            |  emit PHASE_RESULT + VERDICT_LINE           |
            +------+----------------+---------------+-----+
                   |                |               |
              PHASE_RESULT=ok   =skipped        =findings
                   |                |               |
        write verdict scratch   echo note       HALT [portability-red]
        continue -> /ship       continue        journal: next_action/
                   |             -> /ship        resume_cmd/pr_url=N/A/
                   |                              raw_output_path  (NO PR)
                   v
        Step 4.6/9.5/5.6 surfacing:
        read registered-doc-verdicts.md + portability-verdict.md
        -> gh pr edit: splice `### Portability` into PR body
```

### Data Flow

<!-- Step-by-step, component to component, for the primary use case. -->

1. Orchestrator finishes QA-green and the Step 5.5 doc-sync handler block.
2. Orchestrator calls `cj-goal-common.sh --phase portability-audit --mode <m>`.
3. The phase resolves the engine (sibling → `.source`), runs it under `PORTABILITY_STRICT=1`, captures the exit via `... && PA_RC=0 || PA_RC=$?`.
4. The phase parses `FINDINGS=` + `SKILLS_AUDITED=` from the engine output, composes `VERDICT_LINE`, and emits the structured block with `PHASE_RESULT=ok|findings|skipped`.
5. The orchestrator branches: `ok` → write `.cj-goal-feature/portability-verdict.md` + continue; `skipped` → echo note + continue; `findings` → HALT `[portability-red]` with the journal contract fields (no PR).
6. On green, `/ship` runs; the Step 4.6/9.5/5.6 surfacing step reads both verdict scratch files and `gh pr edit`s a `### Portability` line into the PR body (best-effort).

## Tradeoffs

<!-- Design decisions, rejected alternatives, rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Where the gate logic lives | Shared `--phase portability-audit` in `cj-goal-common.sh` | Inline per-orchestrator bash (Approach A) | Inline triples the drift surface + needs 3 halt-table edits — exactly the duplication the shared helper exists to prevent |
| How a red is enforced | Clean orchestrator halt on `PHASE_RESULT=findings` | Export `PORTABILITY_STRICT=1` into the pre-commit path (Approach C) | Env-var propagation across subagent commit shells is fragile; a red would surface as an opaque mid-subagent pre-commit failure, not a clean halt |
| Engine resolution idiom | sibling-then-`.source` (mirror `resolve_worktree_helper`) | `_cj-shared` / `CJ_SHARED_SCRIPTS` | `cj-goal-common.sh` has zero `_cj-shared` references; the engine resolves its own catalog via `git rev-parse`, so the sibling call from a worktree Just Works |
| Gate placement label | CONTENT anchor (after doc-sync, before `/ship`) | A new numbered step (e.g. "Step 5.7") | Orchestrator step numbers are non-monotonic; a number would be jarring + cross-wire-prone (todo surfacing is "Step 5.6") |
| Check 18 default | Stays advisory (global) | Flip to strict-by-default | Flipping would block manual commits repo-wide — a separate, broader decision |
