## Problem Statement

The workbench operator reads `philosophy.md` at repo root to understand "why this workbench exists + which skill does what." Today (2026-05-31) that doc has drifted across 5 weeks of shipping: it names three retired skills (`/workflow`, `/contracts`, `/docs`) as primary design-principle evidence, names two deprecated skills (`/CJ_goal_auto`, `/CJ_goal_run`) as primary entry points across the decision tree, and has zero mention of the actual current front doors `/cj_goal_feature` + `/cj_goal_defect` (which have shipped 10 + 4 telemetry entries this week). It also has zero mention of three mechanism layers landed in the last 14 days: `cj-goal-common.sh` shared helper (S000057), F000028 post-merge/post-rewrite doc-sync hooks, F000029 marker-pickup AUQ.

The story: relocate the doc to a real home (`doc/PHILOSOPHY.md`), rewrite it to current state, add a new `doc/ARCHITECTURE.md` for mechanism reference, and wire a workbench-specific `/document-release` audit convention via root `CLAUDE.md` that catches the drift class that just accumulated. The convention rides `/document-release`'s existing CLAUDE.md-reading behavior at Step 2 — no upstream skill modification.

Who: the workbench operator (single user: chjiang). Pain today: stale "why this exists" doc, no automated drift catch when routing changes, manual remember-to-update overhead. After this story: `doc/PHILOSOPHY.md` reflects current state, the named-doc audit catches future drift, the F000028+F000029 doc-sync loop closes on the content side.

## Mental Model

Four layers, in dependency order:

1. **Filesystem layout shift.** Create `doc/` at repo root; `git mv philosophy.md → doc/PHILOSOPHY.md`. Single command (case-insensitive APFS handles this in one shot because destination directory differs).
2. **Content rewrite at the new path.** `doc/PHILOSOPHY.md` is rewritten end-to-end. Drop retired-skill references except in a single `## Retired skills` subsection (tombstone-once rule). Replace `/CJ_goal_auto` + `/CJ_goal_run` references with `/cj_goal_feature` + `/cj_goal_defect`. Add `## Decision tree` heading + routing diagram covering active CJ_ skills. Preserve load-bearing sections with light edits only.
3. **New mechanism reference.** `doc/ARCHITECTURE.md` is created with exactly five `##` headings. Each section is content-gated (answers specific questions about S000057 helper, F000028 hooks, F000029 AUQ, decision-tree mirror, deprecation tombstones). Not word-count-gated.
4. **Wire the audit via CLAUDE.md.** Add `## /document-release workbench audit conventions` to root `CLAUDE.md`. Two literal jq commands (retired-skill drift + new-skills check) plus annotation suppression rules (mention inside `## Retired skills` OR inside `~~strikethrough~~` OR within 200 chars of `DEPRECATED`/`sunset`/`tombstone` is skipped). State that findings surface in PR-body `## Documentation` section under `### Skill-routing drift` subheading. Add discovery link via root `README.md ## Deeper reading`.

The CLAUDE.md convention is read by `/document-release` as project context during its Step 2 audit pass. Same pattern as the existing CI/CD merge convention section. Zero upstream skill modification.

### Data Flow

<!-- How does data move through the system for the primary use case?
     Step-by-step, component to component. -->

**Use case 1: Operator reads to understand the workbench.**

1. Operator lands on `README.md` (GitHub default or local view).
2. Reads `## Deeper reading` section; clicks `doc/PHILOSOPHY.md` link.
3. `doc/PHILOSOPHY.md` opens to design principles + `## Decision tree` + retired-skill tombstones.
4. Operator follows the routing diagram to pick the right CJ_ skill for their task.
5. (Optional) Operator clicks ARCHITECTURE link from PHILOSOPHY (or back via README); reads mechanism reference for cj-goal-common.sh helper, F000028 hooks, F000029 AUQ.

**Use case 2: `/document-release` runs on a feature branch in this workbench.**

1. Operator runs `/document-release` on a feature branch (Step 1 of the skill).
2. Skill reads project context including root `CLAUDE.md` (Step 2 audit pass).
3. Skill sees `## /document-release workbench audit conventions` section.
4. Skill executes the retired-skill drift check:
   a. `jq -r '.[] | select(.status=="deprecated") | .name' skills-catalog.json` extracts retired skill names.
   b. For each name, `grep -n` in `doc/PHILOSOPHY.md` + `doc/ARCHITECTURE.md`.
   c. For each match, apply annotation suppression: skip if inside `## Retired skills` subsection, inside `~~strikethrough~~` span, or within 200 chars of `DEPRECATED`/`sunset`/`tombstone` (case-insensitive).
   d. Remaining matches are drift findings.
5. Skill executes the new-skills check:
   a. `jq -r '.[] | select(.status=="active") | .name' skills-catalog.json` extracts active skill names.
   b. For each name, grep `doc/PHILOSOPHY.md` (in the `## Decision tree` section) for a case-sensitive literal match.
   c. Missing names are drift findings ("active skill not in decision tree: <name>").
6. Skill assembles findings into the PR body's `## Documentation` section under a `### Skill-routing drift` subheading.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale.
     Future readers need to know why this path over the others. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Where the audit logic lives | CLAUDE.md project context (`## /document-release workbench audit conventions` section) | Fork `/document-release` SKILL.md and edit it | `/document-release` is upstream gstack; forking adds maintenance burden. CLAUDE.md is read by `/document-release` as project context at Step 2; the workbench convention rides that existing behavior. Same pattern as the existing CI/CD merge convention section. |
| ARCHITECTURE section gating | Content-gated (specific entities each section must name) | Word-count gated (e.g., ≥200 words per section) | Content presence is the actual signal; a 200-word section that misses naming "post-rewrite" is broken. Word counts incentivize padding. QA validates with `grep -F`. |
| Retired-skill tombstone strategy | Single `## Retired skills` subsection in PHILOSOPHY, one paragraph per name | Inline tombstones (mention each retired skill where it was relevant, with strikethrough or annotation) | Inline tombstones smear the deprecation across the doc — every paragraph becomes a maintenance hazard ("did I remember to annotate all the inline mentions?"). Single subsection is one canonical location; the annotation suppression rule (proximity to `DEPRECATED`/`sunset`/`tombstone`) is the symmetric escape hatch for places that still need to name the retired entity (e.g., the audit convention itself). |
| `doc/` v1 contents | Two files (PHILOSOPHY + ARCHITECTURE), no `doc/README.md` index | Add `doc/README.md` listing the two files | YAGNI at 2 files. Discovery via root `README.md ## Deeper reading` + GitHub directory rendering is sufficient. Revisit if `doc/` grows past 4 files. |
| CLAUDE.md mechanism prose handling | Leave CLAUDE.md F000009/F000028/F000029/TODOS-hygiene sections in place; allow ~30% overlap with doc/ARCHITECTURE.md | Extract mechanism prose from CLAUDE.md into doc/ARCHITECTURE.md (Approach C) | Approach C is a real refactor — would need to verify nothing downstream greps CLAUDE.md for those exact section headings (manifest sync hooks, F000028's hook itself may grep). Rejected at scope AUQ; revisit if CLAUDE.md grows past ~500 lines. |
| jq command in CLAUDE.md convention | Literal copy-paste-runnable jq commands | Pseudocode or "the auditor should extract deprecated skill names" prose | Literal commands match the existing CI/CD merge convention's literal `gh pr merge` snippet pattern; reduces ambiguity at audit time; operator can copy-paste and run. |
