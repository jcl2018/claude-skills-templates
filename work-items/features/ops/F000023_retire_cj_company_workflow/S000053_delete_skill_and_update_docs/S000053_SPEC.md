## Problem Statement

After S000052 collapses `scripts/validate.sh` Error check 10, the deprecated skill source at `deprecated/CJ_company-workflow/` is orphaned but still on disk; the `skills-catalog.json` entry still references it; `scripts/test.sh` has ~35 assertions plus a T000011 sync-check synthetic block coupled to its paths; `CLAUDE.md` and `README.md` describe `work-copilot/` as a byte-mirror (no longer true); `template-registry.json` has a stale entry. The workbench operator (chjiang) needs all of this cleaned up so the retirement actually completes and the test suite reflects the post-retirement world.

## Mental Model

Two categories of work, run in any order after S000052 lands:

1. **Filesystem-level deletes:** `deprecated/CJ_company-workflow/` directory, `CJ_company-workflow` entry in `skills-catalog.json`, `CJ_company-workflow` entry in `template-registry.json` (or the file itself if confirmed orphaned).
2. **Code/doc updates:** prune `scripts/test.sh` of coupled assertions and the T000011 synthetic block (port structural assertions to `work-copilot/` equivalents); update `CLAUDE.md` 4 sections and `README.md` catalog table; verify `scripts/copilot-deploy.py` and `scripts/skills-deploy` need no changes; optionally polish textual references in three skill .md files.

Final gate: `./scripts/validate.sh && ./scripts/test.sh && ./scripts/copilot-deploy.py doctor <target_repo>` all PASS.

## Acceptance Criteria

<!-- Given/When/Then format. -->

### Story #1: deprecated/CJ_company-workflow/ deleted [core]

```
GIVEN  S000052 has merged and `./scripts/validate.sh` PASSes on origin/main
WHEN   the implementer runs `rm -rf deprecated/CJ_company-workflow/`
       and commits the deletion
THEN   the directory no longer exists
AND    `./scripts/validate.sh` continues to PASS (S000052's Error check 10 removal is what makes this safe)
```

### Story #2: Catalog entry removed [core]

```
GIVEN  skills-catalog.json contains a `CJ_company-workflow` entry
WHEN   the implementer removes the entry (JSON edit) and saves
THEN   `jq '.[] | select(.name == "CJ_company-workflow")' skills-catalog.json` returns empty
AND    `./scripts/validate.sh` PASSes
AND    `./scripts/skills-deploy doctor` returns no errors about a missing entry
```

### Story #3: test.sh blocks pruned [core]

```
GIVEN  scripts/test.sh contains ~35 assertions and a T000011 sync-check synthetic block referencing `deprecated/CJ_company-workflow/`
WHEN   the implementer deletes the coupled blocks and ports structural assertions to `work-copilot/` equivalents
       and saves
THEN   `grep -nc 'deprecated/CJ_company-workflow' scripts/test.sh` returns 0
AND    `./scripts/test.sh` PASSes
AND    structural coverage (template presence, manifest schema parity) is preserved against `work-copilot/`
```

### Story #4: CLAUDE.md and README.md updated [core]

```
GIVEN  CLAUDE.md describes `work-copilot/` in 4 sections as a byte-mirror
       and README.md catalog table has a CJ_company-workflow row
WHEN   the implementer rewrites the 4 sections per the design's migration steps 6+7
THEN   no occurrence of "byte-mirror" or "mirrored from `deprecated/CJ_company-workflow/`" remains in the new prose
AND    `work-copilot/` is described as self-contained
AND    README.md catalog table no longer lists CJ_company-workflow
```

### Story #5: template-registry.json cleaned [core]

```
GIVEN  template-registry.json contains a CJ_company-workflow entry
WHEN   the implementer either removes the entry OR deletes the file (after confirming no script readers)
THEN   either: the file no longer references CJ_company-workflow,
       OR:    the file does not exist
AND    no script in the repo errors on the change (verify via test.sh)
```

### Story #6: copilot-deploy.py doctor PASS [core]

```
GIVEN  all S000053 edits are complete
WHEN   the implementer runs `./scripts/copilot-deploy.py doctor <target_repo>`
THEN   the script exits 0 and prints no errors about missing source files
AND    the target repo's deployed bundle matches `work-copilot/` per the doctor's checks
```

### Story #7: skills-deploy graceful no-op on missing entry [resilience]

```
GIVEN  the CJ_company-workflow entry no longer exists in skills-catalog.json
WHEN   the implementer runs `./scripts/skills-deploy install --include-deprecated`
THEN   the script either silently no-ops on the missing entry or prints a clear "no deprecated entries to install" notice
AND    exits 0
```

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Single PR vs sub-split (catalog/dir/test.sh/docs) | Implementer's call at write time | Mandate sub-split | Both produce the same end state; sub-split is heavier for an internal cleanup, but acceptable if the diff feels large. |
| Port structural assertions vs simply delete them | Port the ones that still apply | Delete everything | Some assertions (template presence, manifest schema parity) still describe meaningful invariants in `work-copilot/`. Porting preserves coverage; deleting would silently reduce test power. |
| Edit template-registry.json vs delete it | Implementer's call after read | Always edit | If the file's only entry was CJ_company-workflow and no script reads it, the file is just dead weight. Delete is cleaner. If there are live entries, edit. |
| Polish textual references in check.md/implement.md/pipeline.md | Optional, in S000053 if convenient | Mandate or defer to later ticket | These are doc-only mentions; not blocking. Including them in S000053 is convenient; deferring is also fine. |
| Order of operations: catalog-edit first vs directory-delete first | Implementer's call | Mandate one order | They're independent — validate.sh has no enforcement after S000052, so either order produces the same end state. |
