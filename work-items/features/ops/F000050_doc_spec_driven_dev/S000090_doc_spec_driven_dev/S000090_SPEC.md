## Problem Statement

The workbench engineer (and any future adopter of the CJ_ skill family) has no
single, human-readable answer to "what docs does this repo carry, and what does each
mean." The contract is scattered across four surfaces — a `### Tracked doc/ files
manifest` + `### Tracked root docs allowlist` YAML buried in `CLAUDE.md`, a
`cj-document-release.json` sidecar, a `CJ-DOC-RELEASE.md` prose contract, and a
`/CJ_repo-init` bootstrap skill — and is workbench-coupled, so it cannot travel to
another repo without that repo replicating the workbench's `CLAUDE.md` structure.
Compounding this, the human-facing docs under `doc/` carry 41 internal work-item IDs
(21 in ARCHITECTURE, 18 in WORKFLOWS, 2 in PHILOSOPHY) — the opposite of
human-readable — and use capital filenames instead of the intended `docs/`.

## Mental Model

The change has four moving parts that must land together but in a strict order:

1. **The artifact** — a new root `doc-spec.md` (Common section + Custom section +
   one fenced ```yaml machine registry). It is BOTH the human answer and the machine
   source of truth.
2. **The doc migration** — `doc/` (3 capitalized files) → `docs/` (3 lowercase
   files, `workflow.md` singular), with all 41 work-item refs scrubbed and the
   content rewritten human-facing.
3. **The enforcement engine** — `/CJ_document-release` rewritten to read
   `doc-spec.md`, self-bootstrap it if missing, stub-scaffold missing declared docs,
   audit each against its `requirement` (+ a no-work-item-ref check for `human-doc`),
   and DERIVE the doc-only auto-commit whitelist from the registry.
4. **The CI gate + the retirements** — `validate.sh` Checks 15/15a/15b/16/17
   re-pointed + a NEW Check 19; `cj-document-release.json`, `CJ-DOC-RELEASE.md`, and
   `/CJ_repo-init` retired.

The ordering rule: author the artifact and migrate the docs BEFORE flipping the
`validate.sh` checks, so the new/changed checks (especially the hard no-ref Check 19)
land green rather than red mid-build.

### The artifact: `doc-spec.md`

A root, human-readable Markdown file with three parts:

1. **Common section** (portable — byte-identical across every adopting repo, seeded
   from a template). Declares the four required docs (`docs/philosophy.md`,
   `docs/workflow.md`, `docs/architecture.md`, `README.md`) + the human-doc rules.
2. **Custom section** (this repo only). Additional docs the repo carries
   (CHANGELOG.md, CLAUDE.md, CONTRIBUTING.md, TODOS.md, etc.) + repo notes.
3. **Machine registry** — a single fenced ```yaml block that `validate.sh` and
   `/CJ_document-release` parse. No table/YAML duplication: the YAML is the source,
   the prose explains it.

The registry schema (carried verbatim from the design doc):

```yaml
# doc-spec registry (parsed by validate.sh + /CJ_document-release)
schema_version: 1
docs:
  - path: docs/philosophy.md
    section: common
    audit_class: human-doc
    purpose: "Major design logic, one '## Principle N' section each."
    requirement: "Arranged by principle; states the repo's first principle(s); human-readable; no work-item IDs."
  - path: docs/workflow.md
    section: common
    audit_class: human-doc
    purpose: "The major workflows from a human's perspective; names the major entry points."
    requirement: "Lists every major workflow/entry point a human would invoke; ASCII flowcharts preferred; no work-item IDs."
  - path: docs/architecture.md
    section: common
    audit_class: human-doc
    purpose: "Meaningful infra under the hood, deeper than workflow.md."
    requirement: "Explains the load-bearing machinery deeper than workflow.md; ASCII diagrams preferred; no work-item IDs."
  - path: README.md
    section: common
    audit_class: human-doc
    purpose: "Repo landing page: folder structure + how to get started."
    requirement: "Has a folder-structure section and a getting-started section naming the major workflows; no work-item IDs."
  - path: CHANGELOG.md
    section: custom
    audit_class: operational
    purpose: "Release history (keep-a-changelog)."
    requirement: "Present; updated by /ship + /document-release."
  - path: CLAUDE.md
    section: custom
    audit_class: operational
    purpose: "Agent operating instructions (auto-loaded by Claude Code)."
    requirement: "Present; work-item references allowed (operational doc)."
  # ... remaining custom/operational docs (CONTRIBUTING.md, TODOS.md, etc.)
```

**`audit_class` enum (closed):**
- `human-doc` — human-facing. Enforced: must exist; **no work-item IDs**
  (`[FSTD]NNNNNN`); ASCII flowcharts preferred (advisory).
- `operational` — must exist; work-item refs allowed (CHANGELOG, CLAUDE.md, etc.).

**The doc-only auto-commit whitelist is DERIVED** from the registry (every declared
`path` + `doc-spec.md` itself + `docs/**/*.md`). `cj-document-release.json` is
retired; nothing hand-maintains a second whitelist.

### `/CJ_document-release` new behavior (replacing the `cj-document-release.json` read)

1. **Read `doc-spec.md`.** If it is **missing**, scaffold it from the portable
   Common seed (the self-bootstrap that replaces `/CJ_repo-init`'s doc duty), commit
   it, and continue.
2. **Enforce presence.** For each declared doc that is **missing**, scaffold a
   **stub** (title + the section skeleton its `audit_class`/role implies + a
   `<!-- TODO: fill in -->` marker), commit it, and record it in the audit as
   `stub — needs content`.
3. **Registered-doc audit (the existing Step 6.7 evolved).** Judge each declared doc
   against its `requirement`. For `human-doc` entries, also run the **no-work-item-ref
   check** (any `[FSTD][0-9]{6}` → `stale: contains work-item refs`). Emit verdicts
   as today (RESULT + scratch file + PR body).
4. **Whitelist** for the doc-only auto-commit gate is derived from the registry.

### `validate.sh` changes (the CI gate)

- **Check 15 / 15a** (tracked `doc/` manifest) → re-point to parse `doc-spec.md`'s
  YAML registry; assert every declared doc exists AND every `docs/*.md` on disk is
  declared (no orphans).
- **Check 15b** (WORKFLOWS orchestrator completeness) → re-point path
  `doc/WORKFLOWS.md` → `docs/workflow.md`; logic unchanged.
- **Check 16** (`cj-document-release.json` schema) → replace with a `doc-spec.md`
  schema check (file exists; YAML block parses; `schema_version: 1`; each entry has
  `path`/`section`/`audit_class`; `audit_class` in the closed enum).
- **Check 17** (root-docs allowlist in `CLAUDE.md`) → the allowlist migrates into
  `doc-spec.md` (custom/operational entries); re-point Check 17 to parse it from
  there.
- **NEW Check 19 — no-work-item-refs lint:** for every `audit_class: human-doc`
  declared doc, grep `[FSTD][0-9]{6}`; ERROR on any hit. (Hard, per the user's
  "no F-0000" requirement; the migration scrubs the docs first so it lands green.)
- **The known blind spot:** every `validate.sh` check change MUST get a parallel
  edit to `scripts/test.sh`'s `zzz-test-scaffold` integration fixture (forgotten on
  F000032/F000034/F000035 — pre-flighted here).

### Retirements (paired-layer, per the workbench deprecation convention)

- **`/CJ_repo-init`** → catalog `status` flip + skill-source relocation +
  work-item-history relocation + remove from routing (`rules/skill-routing.md`),
  decision tree (`docs/philosophy.md`), and `docs/workflow.md`. Non-doc duties:
  `work-items/` tree is lazy-created by `/CJ_scaffold-work-item`; `TODOS.md` is
  lazy-created (empty) by the skills that read it. Full new-repo non-doc bootstrap is
  a noted follow-up.
- **`cj-document-release.json`** → deleted; helper `cj-document-release-config.sh`
  re-pointed to parse `doc-spec.md` (or replaced by a `doc-spec.sh` parser).
- **`CJ-DOC-RELEASE.md`** → content absorbed into `docs/architecture.md` (mechanism
  reference) + `doc-spec.md` (the contract); file removed; its `/CJ_repo-init`
  presence-check is gone with repo-init.

### Data Flow

1. An engineer/agent edits or adds a doc; `doc-spec.md`'s registry declares the
   intended doc set.
2. `validate.sh` (CI gate / pre-commit) parses `doc-spec.md`: Check 15/15a assert
   declared ⇔ on-disk; Check 16 validates the schema; Check 17 reads the migrated
   root-docs allowlist; Check 19 greps each `human-doc` for `[FSTD][0-9]{6}` and
   ERRORs on any hit. `test.sh` re-runs the same checks against the `zzz-test-scaffold`
   fixture.
3. On a major `/cj_goal` run (Step 5.5) or a manual invocation, `/CJ_document-release`
   reads `doc-spec.md` (self-bootstrapping from the Common seed if absent),
   stub-scaffolds any missing declared doc, audits each declared doc against its
   `requirement` (+ the no-ref check for `human-doc`), derives the doc-only
   auto-commit whitelist from the registry, and emits verdicts (RESULT + scratch file
   + PR body).
4. The three retired surfaces (`cj-document-release.json`, `CJ-DOC-RELEASE.md`,
   `/CJ_repo-init`) are no longer read by any path; their duties are subsumed by the
   registry-derived whitelist, the absorbed docs, and lazy-create in consuming skills.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| v1 scope | Full migration in one PR (rename + scrub + README + manifests fold + no-ref lint) | Convention-only / Conv + front-docs (smaller v1, defer scrub + retirements) | User wants the complete vision in one PR |
| Doc registry | Consolidate into one `doc-spec.md` (human + machine) | Three registries side-by-side (doc-spec.md alongside the CLAUDE.md manifest + json) | Drift surface + blocks portability; consolidation is the portability unlock |
| Missing-doc behavior | Scaffold a stub (skeleton + TODO) | Auto-generate full doc prose | Slop risk in an autonomous build |
| `/CJ_repo-init` | Retire (fold into self-bootstrap + lazy-create) | Keep it as the bootstrap | Redundant under doc-spec.md self-heal |
| Check 19 severity | Hard ERROR on any `[FSTD][0-9]{6}` in a human-doc | Advisory warning | The user's "no F-0000 in human docs" requirement; the scrub precedes it so it lands green |
| Whitelist source | Derived from the registry | Hand-maintained second list (the old JSON) | Eliminates the drift surface and makes deleting the JSON safe |
