## Problem Statement

A workbench maintainer wants `/CJ_document-release` to be a standalone doc pass
they can trigger in **any** repo for peace of mind that docs are current — and to
name the general/custom two-tier doc model as a durable workbench principle.
Today the skill is framed as a workbench-internal wrapper: when
`skills-catalog.json` is absent (any non-workbench repo) the Step 6.7.2 skill-MD
audit half emits confusing `jq` stderr noise, silently produces a skill-MD-less
audit, and can leave a stray untracked `.cj-goal-feature/` artifact; and the
gstack-hard-require failure is not clearly messaged. The two-tier model and the
single canonical seed already exist — they need a name (a principle) and the cold
run needs to be clean.

## Mental Model

Three layers, in order of trigger:

```
Step 4  Skill(document-release)  ── resolution fails (gstack absent) ─┐
                                                                       ├─► Step 4→5 boundary: emit [doc-sync-red]
Step 5  upstream returned non-green ──────────────────────────────────┘   naming "gstack /document-release not installed"

Step 6.7 (advisory registered-doc audit)
  6.7.1 registry-doc audit          ── always runs (portable)
  6.7.2 skill-MD enumeration half    ── GUARDED: skip cleanly when skills-catalog.json absent
  6.7.3 human-doc no-work-item-ID    ── always runs (portable, doc-spec.sh --list-human-docs)
  6.7.4 .cj-goal-feature/ scratch    ── skip in non-workbench mode (no catalog)
```

The principle is documentation-only (workbench `philosophy.md` prose); the
mechanical cold guarantee is `doc-spec.sh --validate`. Only the general doc SET +
that schema check travel to a consumer repo; the prose + the declared⇔on-disk
loop (`validate.sh` Checks 15/15a) + `front_table` discipline are workbench-local.

### Story #1: General/custom two-tier principle [usability]

```
GIVEN docs/philosophy.md with its front-table and `## Topic: Deployment` section

      two-tier model + portable any-repo pass + wire-into-CI hook, plus a

      (front-table present before first ## heading) both pass green
```

### Story #2: Cold-run catalog guard [resilience]

```
GIVEN a repo with no skills-catalog.json running /CJ_document-release
WHEN  Step 6.7.2 reaches the skill-MD enumeration
THEN  it emits exactly one clean skip note (no `jq: Could not open file` stderr),
      skips the skill-MD audit half AND the .cj-goal-feature/ scratch write,
      while 6.7.1 (registry-doc audit) and 6.7.3 (human-doc no-work-item-ID lint)
      still run, and no new `set -e` abort is introduced
```

### Story #3: gstack-absent message at Step 4→5 [observability]

```
GIVEN gstack /document-release is not installed
WHEN  the Step 4 Skill(document-release) call fails to resolve OR Step 5 returns
      non-green
THEN  a [doc-sync-red] message surfaces at the Step 4→5 boundary naming
      "gstack /document-release not installed" as a possible cause, with no
      programmatic skill-presence probe added
```

### Story #4: Portability stays local-only [core]

```
GIVEN the Step 6.7.2 guard removes one unconditional repo-local dependency
WHEN  the Step 5.7 portability gate runs against CJ_document-release
THEN  the skill's declared portability remains `local-only` (NOT relabeled
      `workbench`), the gate passes with no [portability-red], and the USAGE.md
      Check-14 drift is resolved with the new behavior noted
```

### Story #5: Portable CI hook + cold-repo smoke [integration]

```
GIVEN docs/architecture.md's doc-spec section and tests/cj-document-release-config.test.sh
WHEN  the implementer documents the portable CI hook (doc-spec.sh --validate is
      portable; the declared⇔on-disk loop + front_table are workbench-local) and
      adds a cold-repo smoke row (synthetic temp repo with doc-spec.md but no
      skills-catalog.json)
THEN  the smoke row proves the Step 6.7.2 guard path runs with no `jq` error and
      no stray .cj-goal-feature/ artifact, and scripts/test.sh + scripts/validate.sh
      stay green
```

### Data Flow

<!-- How does data move through the system for the primary use case?
     Step-by-step, component to component. -->

1. Operator triggers `/CJ_document-release` in some repo.
2. The skill reads `doc-spec.md` (self-bootstraps from the seed if missing); the registry tags each doc `section: common` (general) or `section: custom`.
3. Step 4 invokes gstack `Skill(document-release)`; if it cannot resolve OR returns non-green, the Step 4→5 boundary emits `[doc-sync-red]` naming gstack-not-installed.
4. Step 6.7.2 checks for `skills-catalog.json`: absent ⇒ clean skip of the skill-MD audit half + the `.cj-goal-feature/` scratch write; present ⇒ existing skill-MD enumeration.
5. 6.7.1 (registry-doc audit) + 6.7.3 (human-doc no-work-item-ID lint) always run.
6. In CI, a repo can wire `doc-spec.sh --validate` as the portable schema gate; the cold-repo smoke row in `tests/cj-document-release-config.test.sh` mechanically proves the guard path.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale.
     Future readers need to know why this path over the others. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| How standalone | Decouple from workbench-repo-local files (guard the catalog read) | Native rebuild dropping gstack (Approach B) | Operator chose hard-require gstack; native audit + self-heal already exist — B is rework with regression risk |
| gstack-absent signal | Message-only at the Step 4→5 boundary | Programmatic skill-presence probe | A probe risks a new false-halt class (cf. TODO #251); the message covers resolution-failure AND non-green |
| Principle placement | New sibling principle + one new front-table row | Expand `### The doc contract is one file` in place (no new row) | The general/custom + portable-pass + CI framing is a distinct idea worth its own row |
| Portability label | Keep `local-only` | Relabel `workbench` | Guard removes a repo-local dep (trends MORE portable); still hard-deps gstack + `_cj-shared` + `doc-spec.sh` |
| `--check-on-disk` helper subcommand | Defer to TODOS | Build it now | Out of scope; consumer-repo CI gets schema validation now, full declared⇔on-disk later |
