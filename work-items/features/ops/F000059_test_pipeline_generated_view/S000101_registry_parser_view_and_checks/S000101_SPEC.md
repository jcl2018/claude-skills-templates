## Problem Statement

A maintainer (or a first-time reader) of this repo cannot survey its
verification surface without reading 1069 lines of validate.sh and 2773 lines
of test.sh: which checks exist (across two ID grammars), what each test family
covers, which CI gates fire when, and which regressions are ratcheted.
spec/gate-spec.md deliberately answers only the LAYER question. Worse, the
test surface itself has a live integrity gap: tests/*.test.sh files are
hand-registered in test.sh, and tests/cj-goal-feature-smoke.test.sh exists on
disk with zero runner references — it silently never runs, and nothing flags
that. This story builds the check-level human view as a generated doc whose
structure and coverage are enforced by construction.

## Mental Model

Two layers, one source of truth, two hard enforcement loops:

- **Machine layer**: spec/test-pipeline.md — one yaml registry row per
  verification unit (validate check, test sub-suite, inline test family,
  standalone suite, CI workflow, git hook), each with a literal `anchor`
  locating it in its `source` file.
- **Human layer**: docs/test-pipeline.md — GENERATED from the registry by
  scripts/test-pipeline.sh --render; humans read it, never edit it.
- **Loop 1 (view-sync, Check 23 extension)**: regenerate-and-diff — a
  hand-edit or stale render fails pre-commit/CI.
- **Loop 2 (coverage, new Check 24)**: forward (every anchor greps in its
  source — a removed/renamed unit orphans its row) + reverse (every live
  banner/comment/test-file/workflow/hook resolves to exactly one row — a new
  unit without a row fails) + floor-assert (≥ 20 reverse tokens so grammar rot
  can't make the check vacuously pass).

Semantic accuracy of each `purpose` one-liner stays with the advisory
registered-doc audit — we buy structural sync, not meaning sync.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group. -->

### Story #1: Machine registry of verification units [core]

```
GIVEN the live tree's verification surface (validate.sh checks in both ID
      namespaces + warning checks, registered tests/*.test.sh sub-suites,
      inline test.sh families, standalone suites, CI workflows, git hooks)
WHEN  spec/test-pipeline.md is authored from the design doc's Registry
      inventory appendix and re-verified against the live tree
THEN  it contains ONE fenced yaml registry (schema_version 1) of ~65 rows;
      every row has id/family/label/anchor/source/layer/disposition/trigger/
      purpose with closed enums honored; "Error check 11" and "Check 11" are
      distinct rows; Check 15 is one row; tests/*.test.sh rows use
      source: scripts/test.sh + the literal runner-path anchor; rendered
      fields (label, purpose) carry no work-item IDs
```

### Story #2: Parser (--validate / --list-units / --render) [core]

```
GIVEN spec/test-pipeline.md on disk
WHEN  scripts/test-pipeline.sh runs each subcommand
THEN  --validate exits 0 printing OK schema_version=1 (and exits non-zero on
      malformed fixtures, including a work-item ID slipped into a rendered
      field); --list-units enumerates every row; --render emits the full
      markdown view (AUTO-GENERATED header, per-family summary table first,
      per-family sections, single gate-spec pointer line) byte-identically
      across two consecutive runs; the registry resolves spec/-then-root
```

### Story #3: Generated human view [usability]

```
GIVEN scripts/generate-doc-views.sh wired with the third output
WHEN  the generator runs on the workbench tree
THEN  docs/test-pipeline.md is (re)written with zero diff on a fresh tree,
      opens with the summary table (family · units · hard/advisory split ·
      trigger) BEFORE the first ## heading, contains zero work-item IDs,
      links spec/gate-spec.md for the layer model, and enumerates every
      registry unit with label, purpose, disposition (+skip flag), trigger,
      and ratchet marking
```

### Story #4: Hard view-sync (Check 23 extension) [integration]

```
GIVEN a hand-edit to docs/test-pipeline.md OR a registry edit without
      re-rendering
WHEN  ./scripts/validate.sh runs (pre-commit hook or CI)
THEN  the extended Check 23 temp-regen+diff fails with "run
      scripts/generate-doc-views.sh"; AND on a repo where the parser or
      registry is absent, the third diff is skipped with a note while the
      existing two views are still checked
```

### Story #5: Hard coverage cross-check (Check 24) [resilience]

```
GIVEN the landed registry and a clean baseline (cj-goal-feature-smoke
      registered in test.sh or retired from disk)
WHEN  ./scripts/validate.sh runs
THEN  Check 24 passes: every anchor greps -F in its declared source
      (forward); every live validate banner/comment token, tests/*.test.sh
      path, workflow file, and install_hook name resolves to exactly one
      registry row in its namespace (reverse); reverse extraction yields
      >= 20 tokens (floor); AND when the registry is absent the check emits
      SKIP, never ERROR
```

### Story #6: Doc-contract registration + seed lockstep [integration]

```
GIVEN spec/doc-spec.md and the portable Common seed copies
WHEN  the registry entries and seed growth land
THEN  docs/test-pipeline.md is declared common/human-doc/front_table:required
      (front_table workbench-only) with the workbench requirement naming the
      generator + Check 23; spec/test-pipeline.md is declared
      custom/operational; the seed declares eleven general docs with a
      mechanism-neutral test-pipeline requirement string;
      templates/doc-spec-common.md and the scripts/doc-spec.sh heredoc are
      byte-identical (config-test 13 green); doc views regenerate with the
      new rows; validate Checks 15/15a/16/17/19/20/23 stay green
```

### Story #7: Self-tested, self-registered, self-described [core]

```
GIVEN tests/test-pipeline-spec.test.sh written with temp-dir-isolated
      fixtures
WHEN  ./scripts/test.sh runs
THEN  the new suite executes via the hand-wired runner registration and
      passes: parser round-trip green, malformed fixtures fail, and all four
      drift drills flag correctly ((a) fake banner → reverse, (b) broken
      anchor → forward, (c) hand-edited view → Check 23 ext, (d) removed
      runner block → orphaned row); the zzz-test-scaffold integration fixture
      is invariant; AND the registry carries self-inclusion rows for
      Check 24 and the new suite with the view re-rendered
```

### Story #8: Consumer-repo skip posture [resilience]

```
GIVEN a repo (or temp fixture) WITHOUT spec/test-pipeline.md and/or
      scripts/test-pipeline.sh
WHEN  generate-doc-views.sh and validate.sh run there
THEN  the generator's third output is skipped with a one-line note; the
      Check 23 extension's third diff is skipped with a note; Check 24 emits
      SKIP; nothing errors and nothing writes docs/test-pipeline.md
```

### Story #9: Secondary-doc sweep [usability]

```
GIVEN the landed mechanism
WHEN  CLAUDE.md and docs/architecture.md are swept
THEN  CLAUDE.md's scripts-reference table carries a test-pipeline.sh row and
      an updated generate-doc-views.sh description; the spec-registry family
      is described as 4 members in both docs; architecture.md's doc-contract
      section describes the test-pipeline mechanism; no decision-tree or
      workflow.md orchestrator edits exist (no new skill)
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. Implementer authors/edits a row in spec/test-pipeline.md's yaml registry (e.g. a new validate check gets a row with its banner string as `anchor`).
2. `scripts/test-pipeline.sh --validate` checks schema, closed enums, and the rendered-field work-item-ID lint — an ID slip fails at the registry, before the human-doc lint.
3. `scripts/generate-doc-views.sh` calls `test-pipeline.sh --render` and rewrites docs/test-pipeline.md (summary table + per-family sections + gate-spec pointer line).
4. At commit time the pre-commit hook runs validate.sh: extended Check 23 re-renders to a temp file and diffs (stale view → fail); Check 24 greps every anchor forward into its source and sweeps the live surface reverse into rows (orphaned row or unregistered unit → fail; < 20 reverse tokens → fail).
5. In CI (validate.yml), the same validate.sh runs plus scripts/test.sh, which executes the registered tests/test-pipeline-spec.test.sh — round-trip assertions and the four drift drills against temp copies, never the live tree.
6. In a consumer repo without the registry/parser, steps 3–4 degrade to one-line skips (Check 16/23 posture); the doc-spec seed still requires docs/test-pipeline.md, satisfied by a stub or hand-maintained copy (mechanism-neutral requirement).

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Drift defense | Generated view from a 4th registry + hard Check 23 ext + hard Check 24 (Approach C) | A: hand-written doc + advisory audit; B: hand-written doc + token-presence ratchet | A is mergeable-while-stale (the USAGE-freshness ratchet exists because judgment alone missed drift); B proves mention, not sync — prose between tokens still rots |
| Check 24 posture | HARD from day one (with registry-absent SKIP) | Advisory soak first (portability/policy-drift convention) | The soak convention covered checks inheriting baselines they didn't control; this PR authors the full registry, so the baseline is clean by construction — any finding is new by definition |
| Test-row anchoring | source = scripts/test.sh, anchor = literal runner path | source = the test file itself | Anchoring on the runner path makes the forward check prove the file is WIRED into the suite — closing the silent-skip gap is the point |
| Row granularity | Check 15 = one row (15a/15b in purpose); wrapper blocks share the standalone suite's row via multi-valued triggers | One row per sub-ID / per wrapper block | The live source has a single `=== Check 15` banner (15a/15b are bare comments); duplicate rows would double-count units the reverse sweep sees once |
| Registry classification | spec/test-pipeline.md = custom/operational; docs/test-pipeline.md = common/human-doc | Both common, or registry as human-doc | Anchors legitimately carry work-item IDs (operational docs allow them; human-docs hard-fail Check 19); only the VIEW is the portable universal doc kind |
| Seed requirement string | Mechanism-neutral ("check-level enumeration… kept matching the live surface; no work-item IDs") | Naming the generator + Check 23 in the seed | A hand-maintained consumer copy must satisfy the portable contract; only the workbench registry entry names the mechanism (existing generated-views precedent) |
| Naming | test-pipeline (registry/parser/view name-paired) | test-spec | Avoids collision with the per-work-item TEST-SPEC.md artifact that already has a distinct meaning in this repo |
