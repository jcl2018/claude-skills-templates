## Problem Statement

The workbench maintainer cannot tell at a glance which root `.md` files are machine
config and which are human docs. The 3 spec-registry files (`doc-spec.md`,
`gate-spec.md`, `permission-policy.md`) are machine config — one fenced `yaml` block each,
parsed by a sibling `.sh` helper — but they sit at the repo root next to `README.md`,
`CHANGELOG.md`, and `CLAUDE.md`, so they read as documentation. The maintainer wants the
structural signal of a dedicated `spec/` folder, WITHOUT breaking the one external
consumer (the knowledge-base repo, which has a root `doc-spec.md`) or the portable seed.

## Mental Model

Two layers of change, gated by ONE invariant: every consumer must keep resolving the
registry, whether it lives at `spec/<name>.md` (this repo, post-move) or root
`<name>.md` (the knowledge-base consumer, a fresh adopter, or any caller that hasn't
migrated).

```
              ┌──────────────── resolution order (every helper) ────────────────┐
  env override (outermost)  ->  $ROOT/spec/<name>.md (if exists)  ->  $ROOT/<name>.md
              └──────────────────────────────────────────────────────────────────┘

  Layer 1 (helpers)         doc-spec.sh / gate-spec.sh / permission-policy.sh
                            resolve via the order above.
  Layer 2 (literal-root     validate.sh Checks 16/19/20/21/22, test.sh S94/S96,
   gating callers)          CJ_document-release self-bootstrap guard + Step 6.7.1
                            parser — these gate on a literal root [ -f ] BEFORE
                            calling a helper, so EACH must learn spec/-then-root too.
                            The fallback alone does NOT cover them.
```

The seed / self-bootstrap WRITE path stays root-style (consumer convention, test #13
byte-identity); only READ/guard probes are taught spec/-then-root.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group.
     Each #n header matches the # column in the Requirements tables. -->

### Story #1: Files relocated with history preserved [core]

```
GIVEN doc-spec.md, gate-spec.md, permission-policy.md at the repo root
WHEN  the maintainer git-mv's all three into spec/ and creates spec/ (no other files)
THEN  spec/doc-spec.md, spec/gate-spec.md, spec/permission-policy.md exist,
      root no longer has any of the three,
      AND git log --follow tracks each file's history across the move
```

### Story #2: Helpers resolve spec/-first, root-second, env outermost [resilience]

```
GIVEN the 3 files now live under spec/
WHEN  doc-spec.sh --validate, gate-spec.sh --validate, permission-policy.sh --validate
      run from the repo
THEN  all three print their OK/validate-green line (resolving spec/<name>.md)
AND   GIVEN a simulated root-only repo (no spec/, file at root)
      WHEN the same helper runs
      THEN it resolves the root <name>.md via the fallback (knowledge-base unaffected)
AND   GIVEN PERMISSION_POLICY_PATH=/nonexistent (an env override to a missing file)
      WHEN permission-policy.sh --validate runs
      THEN it FAILS (the env override is outermost and wins over the spec/root fallback)
```

### Story #3: Registry self-declares spec/ paths [core]

```
GIVEN the 3 section: custom registry entries declared path: doc-spec.md etc.
WHEN  they are updated to path: spec/doc-spec.md / spec/gate-spec.md / spec/permission-policy.md
THEN  validate.sh Check 17's root-only slash-filter excludes them
      AND the declared-exists direction finds each under spec/
      AND root *.md on disk = {README, CHANGELOG, CLAUDE, CONTRIBUTING, TODOS}, all still declared
```

### Story #4: No silent-SKIP — validate.sh checks stay PASS [resilience]

```
GIVEN Checks 16/19/20/21/22 each gate on a literal root [ -f ] today
WHEN  each guard is taught to probe spec/<name>.md first then root, and validate.sh runs after the move
THEN  validate.sh prints PASS: (NOT SKIP:) for Checks 16, 19, 20, 21, and 22
      AND validate.sh exits 0/0 overall
```

### Story #5: No hard-FAIL — test.sh S94/S96 stay green [resilience]

```
GIVEN test.sh:88 (_S94_POLICY) and test.sh:125 (_S96_SPEC) point at root paths
WHEN  each is taught spec/-first and the full suite runs
THEN  scripts/test.sh PASSes, including the S94 (permission-policy) and S96 (gate-spec) cases
      AND seed test #13 (doc-spec seed byte-identity) is green (seed unchanged)
```

### Story #6: No duplicate-file bug in CJ_document-release [core]

```
GIVEN this repo now has spec/doc-spec.md and no root doc-spec.md
WHEN  CJ_document-release runs its self-bootstrap guard and Step 6.7.1 registry parser
THEN  the guard READs spec/-then-root, finds the existing spec/doc-spec.md, and does NOT
      re-create or commit a duplicate root doc-spec.md
      AND a genuinely-missing-everywhere bootstrap still WRITES root-style (consumer convention)
```

### Story #7: spec/ covered by declared⇔on-disk + test.sh mirror [observability]

```
GIVEN docs/*.md has an orphan scan today but spec/*.md does not
WHEN  a spec/*.md orphan scan is added to validate.sh AND mirrored into the test.sh
      zzz-scaffold integration fixture
THEN  a spec/ file not declared in the registry produces an ERROR
      AND a declared spec/ file that exists produces no false missing/orphan
      AND the test.sh zzz integration test exercises the new spec/ orphan scan
```

### Story #8: Whitelist + views updated [integration]

```
GIVEN --expand-whitelist emits the literal doc-spec.md and the views say "generated
      from the doc-spec.md registry"
WHEN  the literal is changed to spec/doc-spec.md, the generate-doc-views.sh header string
      is updated, and docs/doc-general.md + docs/doc-custom.md are regenerated
THEN  the doc-only auto-commit whitelist covers spec/doc-spec.md + the declared spec/ paths
      AND Check 23 (views in sync) is green (committed views match the regenerated output)
```

### Story #9: Prose/path sweep clean [usability]

```
GIVEN ~30/18/15 files mention doc-spec.md / gate-spec.md / permission-policy.md
WHEN  every stale PATH reference to the 3 files is updated to spec/<name>.md
      (CLAUDE.md, README regen + generate-readme.sh:22,36, docs/*, the 6 skill MDs,
       skills-catalog.json:315/332/338, templates/doc-WORKFLOWS-section.md:70,
       cj-handoff-gate.sh:66-68, cj-portability-audit.sh:22,132, doc-spec.sh:168,
       the 3 files' own prose)
THEN  an adversarial completeness sweep finds NO stale root-path reference to the 3 files
      AND no human-doc (Check 19) gains a work-item ID
```

### Story #10: One lockstep commit [resilience]

```
GIVEN the git mv, registry self-paths, helper fallbacks, the new spec/ orphan scan,
      and its test.sh mirror are interdependent
WHEN  they are all staged and committed together
THEN  validate.sh does not transiently ERROR 3 orphan/missing
      AND the pre-commit hook passes on the single commit
```

## Architecture

<!-- High-level system design for this story. -->

```
  spec/                                   scripts/
  ├── doc-spec.md ◄── parsed by ─────────  doc-spec.sh           (spec/→root→env)
  ├── gate-spec.md ◄── parsed by ────────  gate-spec.sh          (spec/→root→env)
  └── permission-policy.md ◄── parsed by   permission-policy.sh  (spec/→root→env)

  Literal-root-gating callers (taught spec/-then-root, NOT relying on the helper fallback):
    validate.sh  : Check 16 (:716,718) | Check 19 HARD (:821) | Check 20 (:855)
                   Check 21 (:898) | Check 22 (:940,:971) | + new spec/*.md orphan scan
    test.sh      : S94 _S94_POLICY (:88) | S96 _S96_SPEC (:125) | zzz orphan-scan mirror
    CJ_document-release/SKILL.md : self-bootstrap guard (:156,158-163) | Step 6.7.1 parser (:534)

  Unchanged (root-style by construction): the seed, _emit_seed heredoc,
  templates/doc-spec-common.md, the config-test temp-repo fixtures.
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. A consumer (validate.sh check, a skill, a CI job) needs the registry.
2. It calls the helper (`doc-spec.sh`/`gate-spec.sh`/`permission-policy.sh`), OR — for the literal-root-gating callers — first probes spec/-then-root itself.
3. The helper resolves: env override (if set) → `$ROOT/spec/<name>.md` (if exists) → `$ROOT/<name>.md`.
4. In this repo the spec/ path resolves; in the knowledge-base repo (root-only) the root path resolves via the fallback.
5. The parsed `yaml` registry feeds the check / skill / view-generation as before.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Migration scope | Workbench-internal move + back-compat fallback (Approach A) | Ecosystem-wide convention change + migrate knowledge-base + change the seed (Approach B) | B forces a consumer migration + seed/test-#13 churn + a new-adopter break for no extra benefit to this repo |
| Role-clarity signal | Structural `spec/` folder (Approach A) | Header banner inside each file (Approach C) | Operator wants the glance-level signal to be the layout, not a comment |
| Literal-root callers | Teach each one spec/-then-root | Rely on the helper fallback alone | The fallback does NOT cover callers that gate on `[ -f ]` BEFORE invoking the helper — they'd silently SKIP or duplicate-create |
| Seed/bootstrap WRITE path | Stays root-style | Move it to spec/ too | Keeps consumer convention + test #13 byte-identity green |
| Env override position | Outermost (`${ENV:-<spec-else-root>}`) | Nested inside the fallback | test.sh:113 asserts the `/nonexistent` override FAILS validation |
