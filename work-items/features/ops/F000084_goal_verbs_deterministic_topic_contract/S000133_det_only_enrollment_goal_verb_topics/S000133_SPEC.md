## Problem Statement

The workbench operator ships through three cj_goal verbs daily, but none of them
is held to the three-layer topic contract: `goal-feature` + `goal-task` testing
is one bundled agentic-only `cj-goal-eval` label, `goal-defect` has no
`categories:` row at all, and the F000082 enrollment rule would hard-require a
`local-hook`+`agentic` test per topic — a dependency on eval assets the operator
has already decided to remove. The operator needs (a) a contract-engine seam that
can enroll a topic deterministic-only, and (b) the three verbs backfilled as
separately-enrolled topics with real deterministic coverage at every layer,
including NEW CI-nightly chain drills over each verb's helper chain.

## Mental Model

Two enrollment lists, one union runner. `topic_contracts:` keeps the four-point
both-modes rule (portability). The NEW `topic_contracts_deterministic:` requires
three points — CI-push + CI-nightly + local-hook/deterministic — and merely
tolerates agentic rows. Both engine runners (`_run_topic_contract` for Check 30,
`_run_topic_docs` for Check 31) iterate the UNION of the two lists, judging each
topic under its own list's rule; "inactive" only when BOTH lists are empty. Per
verb, the three required points are: an existing-or-new fast smoke at CI-push, a
NEW helper-chain drill at CI-nightly (gated out of the per-PR path by
`TEST_FAST=1`), and an existing deterministic test at local-hook.

### Story #1: Deterministic-only enrollment key [core]

```
GIVEN a spec/test-spec-custom.md carrying topic_contracts_deterministic: [goal-feature, goal-task, goal-defect]
WHEN  bash scripts/test-spec.sh --validate runs
THEN  it exits 0; AND a fixture with an invalid slug (e.g. "Goal_Feature") or a topic present in BOTH
      topic_contracts: and topic_contracts_deterministic: exits non-zero with a named error
```

### Story #2: Det-only arm of --check-topic-contract [core]

```
GIVEN the three det-enrolled topics each carrying CI-push + CI-nightly + local-hook/deterministic rows with front-door docs
WHEN  bash scripts/test-spec.sh --check-topic-contract runs
THEN  the three topics report green with NO agentic requirement; AND portability is still judged by the
      unchanged four-point both-modes rule; AND removing any one of a det topic's three points yields a finding
```

### Story #3: Union iteration + inactive gate [resilience]

```
GIVEN a registry where topic_contracts: is empty but topic_contracts_deterministic: is non-empty
WHEN  _run_topic_contract / _run_topic_docs execute (via --check-topic-contract / --check-topic-docs)
THEN  neither reports "topic contract inactive"; the summary line reads topic contract: enrolled=N findings=M
      with N counting the union; AND a repo with BOTH lists absent still prints the inactive line matched by
      ^(REGISTRY=absent|topic contract inactive)
```

### Story #4: Seed dual-write [integration]

```
GIVEN the deterministic-only flavor added to the spec/test-spec.md topic-axis prose
WHEN  the seed-identity test compares spec/test-spec.md against test-spec.sh --seed output
THEN  they are byte-identical
```

### Story #5: 3-arm negative drill [resilience]

```
GIVEN temp copies of the registry (and a temp docs tree via TESTDOC_OUT) with planted faults
WHEN  the targeted scripts/test.sh drill invokes ONLY the test-spec.sh engine per arm
THEN  arm 1 (CI-nightly row removed) yields the topic-contract finding; arm 2 (re-topic'd agentic eval row
      removed) leaves the det-enrolled topics green; arm 3 (dream doc hidden) yields the topic-docs finding
```

### Story #6: 4 new deterministic tests + TEST_FAST gating [core]

```
GIVEN the defect smoke + the three chain drills implemented per the design's chain-step contracts

      runs them (nightly.yml picks them up by construction)
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. The overlay declares the two enrollment lists + the 11 topic-attributed `categories:` rows.
2. `test-spec.sh` parses both lists (`--validate` guards slugs + cross-list duplicates).
3. `_run_topic_contract` iterates the union: for each det-list topic it checks the three deterministic points (+ front-door docs); for each both-modes-list topic it checks the unchanged four points.
4. `_run_topic_docs` iterates the same union: dream doc + topic subdir (+ per-covered-layer pages) per topic.
5. `validate.sh` Checks 30/31 call the engines, skipping only on `^(REGISTRY=absent|topic contract inactive)`; `scripts/test.sh` parses the `enrolled=N findings=M` summaries and runs the 3-arm drill against temp copies.
6. Per-PR CI runs `TEST_FAST=1 bash scripts/test.sh` (chain drills skipped); `nightly.yml` runs the full `test.sh` (chain drills execute, driving each verb's helper chain in a temp clone).

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Enrollment flavor | A second overlay key `topic_contracts_deterministic:` with its own three-point arm | Declaring agentic rows for the three verbs to satisfy the existing four-point rule | The agentic tests are scheduled for removal; requiring them bakes in a dependency planned for deletion (operator directive) |
| Nightly coverage | NEW per-verb helper-chain drills (Approach B) | Re-declaring existing scripts at CI-nightly (A) or an eval-wiring lint (C) | Real new coverage of the deterministic plumbing; a wiring lint guards assets scheduled for removal |
| Cadence mechanism | `TEST_FAST=1` guard in `scripts/test.sh` (the `test-deploy` pattern) | A new CI workflow file or per-PR execution | CI-push stays fast (operator directive); `nightly.yml` already runs the full suite — zero workflow changes |
| Activity gate | Union iteration in BOTH runners, per-list rules, format-preserving | A separate det-only runner or a changed summary format | Existing grep/summary contracts in `validate.sh` + `test.sh` must keep matching; one runner per check keeps the engines single-owner |
| Local-det fills | Existing deterministic tests (`cj-e2e-gate`, `e2e-local` det half, `post-land-sync`) | New local-hook scripts per verb | Zero new maintenance; documented fallback (re-declare the chain drill at local-hook) if the harnesses are later removed |
| Defect agentic row | None (eval case stays undeclared) | Declaring defect's on-disk eval case | No NEW agentic row anywhere (operator directive) |
