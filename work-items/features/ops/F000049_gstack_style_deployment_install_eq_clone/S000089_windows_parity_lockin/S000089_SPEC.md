## Problem Statement

F000049's last open success criterion is "Windows/Git-Bash copy-mode parity
holds." S4 (S000088) declared the default install install==clone-in-place
(`install_mode: in-place`) and dropped every runtime `.source` reach-back. The S5
design pass empirically verified those S4 changes ALREADY work under Windows
copy-mode — they are platform-neutral by construction (a manifest jq stamp, the
`_cj-shared` deposit is `cp`, the resolution is `_cj-shared`-based). So S5 is not
a parity-code story; it is a **lock-in** story: add a CI assertion so a future
change can't silently regress the parity, document it, and close the epic. The
POSIX-only dir-symlink reinstall-free refinement is **dropped** (it would create a
POSIX/Windows asymmetry — the opposite of parity).

## Mental Model

```
S4:  default install stamps install_mode: in-place + de-couples from .source  (platform-neutral)
S5:  PROVE it under copy-mode  →  windows-smoke.sh asserts:
       (a) copy-mode default install stamps install_mode: in-place (bundle_path == source)
       (b) a copy-installed orchestrator resolves update-check from _cj-shared, no .source
     runs on BOTH lanes:  windows-latest (windows.yml)  +  ubuntu (test.sh:506, FORCE_COPY)
DROP: dir-level skill symlinks (reinstall-free pull) — POSIX-only asymmetry, non-criterion
```

## Acceptance Criteria

### Story #1: copy-mode keeps the in-place receipt [core]

```
GIVEN a hermetic copy-mode (FORCE_COPY) DEFAULT `skills-deploy install`
WHEN  windows-smoke.sh inspects the manifest
THEN  install_mode == "in-place" AND bundle_path == source
```

### Story #2: de-coupled resolution survives copy-mode [core]

```
GIVEN the copy-installed orchestrator skill files
WHEN  windows-smoke.sh inspects them
THEN  the update-check resolves from _cj-shared (the `_UC=` form) with NO manifest .source read
```

### Story #3: both lanes guarded [resilience]

```
GIVEN windows-smoke.sh
WHEN  it runs under windows-latest (windows.yml) AND ubuntu (test.sh:506) AND on a symlink-capable host
THEN  the S5 assertion passes (it uses FORCE_COPY, so it is host-independent)
```

### Story #4: parity + dropped refinement documented [usability]

```
GIVEN the S5 changes
WHEN  a maintainer reads CLAUDE.md "Running on Windows"
THEN  the in-place install==clone copy-mode parity is noted AND the dir-symlink drop + rationale are recorded
```

### Story #5: update-check on the in-place checkout [observability]

```
GIVEN a copy-mode in-place install
WHEN  the update-check nudge resolves
THEN  skills-update-check is found in _cj-shared AND reads manifest source == the in-place checkout
```

### Data Flow

1. `windows-smoke.sh` runs a hermetic `FORCE_COPY` default install into a temp target.
2. It `jq`-reads the temp manifest for `install_mode` + `bundle_path == source`.
3. It greps a copy-installed orchestrator SKILL.md for the `_UC=` (`_cj-shared`) form + absence of a manifest `.source` read.
4. The block is host-independent (FORCE_COPY), so it passes on windows-latest, ubuntu, and macOS.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| S5 scope | Verification-only lock-in + close the epic | Build new copy-mode parity code | The de-risking proved S4's model already holds under copy-mode (platform-neutral); the gap is a missing CI assertion, not missing behavior |
| Dir-symlink reinstall-free pull | DROP | Build it (POSIX-only dir-level skill symlinks) | POSIX/Windows asymmetry (Windows can't symlink) — the opposite of parity; non-criterion; reworks per-file drift detection for a small convenience |
| Assertion placement | `windows-smoke.sh` | `test-deploy.sh` only | windows-smoke runs on BOTH lanes (windows.yml + test.sh:506); one assertion guards real Windows AND the default ubuntu CI |
