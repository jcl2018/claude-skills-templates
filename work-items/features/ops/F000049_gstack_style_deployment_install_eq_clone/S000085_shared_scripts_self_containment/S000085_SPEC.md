## Problem Statement

A workbench operator on a machine (or in a consumer repo) that has the CJ_ skills
installed but NOT the separate source clone cannot run the orchestrator-family
skills: they EXECUTE shared root `scripts/*.sh` (cj-goal-common.sh,
cj-worktree-init.sh, cj-worktree-cleanup.sh, cj-document-release-config.sh)
resolved through `.source` in `~/.claude/.skills-templates.json`, and those
scripts are never deployed. That reach-back is what pins those skills at
`workbench` tier on `/CJ_portability-audit`. This story removes the reach-back
from the happy path so the shared scripts travel with the install — the runtime
de-coupling foundation the rest of the gstack migration (S2–S5) builds on.

> **Implementation reconciliation (ground truth, 2026-06-05).** The parent
> /office-hours design estimated **12 skills** would re-tier. Building it surfaced
> that **12 skills reach `.source`, but only 4 of them EXECUTE shared root
> scripts** and are pinned to `workbench` by that: **`CJ_goal_feature`,
> `CJ_goal_defect`, `CJ_goal_todo_fix`, `CJ_document-release`**. The other 8
> `.source`-reachers touch `.source` ONLY for the passive `skills-update-check`
> nudge (a fail-soft, non-pinning reach the audit already treats as OK) and were
> already `standalone`. Two further `workbench` skills — `CJ_personal-workflow`
> (repo-local reaches) and `CJ_portability-audit` (its own engine) — are pinned by
> patterns OUTSIDE the shared-scripts mechanism and are out of S1 scope. **So S1
> re-tiers exactly the 4 orchestrator-family skills.** Every "12" below should be
> read as "the 4 orchestrator-family skills"; the count is corrected in the load-
> bearing claims.

## Mental Model

Resolution today is 2-tier (repo-local for self-dev, else `.source`). This story
makes it **3-tier**, inserting a deployed shared location between them:

```
shared-script resolution (per skill preamble):
  (1) $REPO_ROOT/scripts/<name>     workbench self-development (unchanged)
  (2) <deployed shared home>        NEW — travels with the install
  (3) .source/scripts/<name>        legacy fallback (retained in S1, removed in S4)
```

Tier (2) is the new behavior; tiers (1) and (3) are preserved, which is what makes
the change non-breaking and reversible.

## Acceptance Criteria

### Story #1: Shared scripts travel with the install [core]

```
GIVEN a clean ~/.claude with no shared scripts deployed
WHEN  skills-deploy install runs from the workbench source
THEN  all 26 shared scripts/*.sh exist at the deployed shared home
```

### Story #2: 3-tier bundle-first resolution [core]

```
GIVEN one of the 4 orchestrator-family skills
WHEN  it resolves a shared script
THEN  it tries $REPO_ROOT/scripts first, then the deployed shared home, then .source —
      in that order, with .source reached only when the first two miss
```

### Story #3: Runs with no source clone present [resilience]

```
GIVEN the CJ_ skills installed but .source unreachable (absent / relocated)
WHEN  a shared-script-dependent skill path runs
THEN  it resolves the script from the deployed shared home and proceeds
      with no ".source missing" failure
```

### Story #4: Re-tiered to local-only [core]

```
GIVEN the 4 formerly-workbench orchestrator-family skills after this change
WHEN  /CJ_portability-audit runs
THEN  each reads local-only (not workbench), and the catalog portability field agrees
```

### Story #5: Non-breaking [integration]

```
GIVEN the workbench's own checkout (repo-local resolution) and a legacy install (.source)
WHEN  the change lands
THEN  both resolution paths still work and ./scripts/validate.sh && ./scripts/test.sh stay green
```

## Architecture

```
                       skills-deploy install
                              │
        ┌─────────────────────┼──────────────────────┐
        ▼                     ▼                       ▼
  per-skill files      26 shared scripts/*.sh    rules/*.md, templates/
  (symlink/copy)        → deployed shared home      (existing behavior)
        │                     │ (NEW deposit step)
        │                     ▼
        │            <deployed shared home>  (O2: _cj-shared/scripts/ | cj-workbench/)
        ▼                     ▲
   /CJ_* skill preamble       │
        │  resolve <name>     │
        ▼                     │
  (1) $REPO_ROOT/scripts ──miss──▶ (2) deployed home ──miss──▶ (3) .source (legacy)
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `scripts/skills-deploy` | claude-skills-templates | Modified | New deposit step: install the 26 shared `scripts/*.sh` to the deployed shared home; checksum-track + own them in the manifest (like templates) |
| 4 orchestrator-family skill preambles | claude-skills-templates | Modified | Insert tier (2) deployed-home resolution between repo-local and `.source` (8 resolution blocks across the 4 SKILL.md files) |
| `skills-catalog.json` | claude-skills-templates | Modified | Re-tier the 4 orchestrator-family skills `workbench → local-only` |
| `scripts/cj-portability-audit.sh` (+ expectations) | claude-skills-templates | Modified | Recognize the deployed shared home so the audit reports `local-only`, not `workbench` |
| `scripts/test.sh` / `tests/` | claude-skills-templates | New | Consumer-repo simulation proving no-`.source` resolution + tier assertions |

### Data Flow

1. `skills-deploy install` copies per-skill files AND deposits the 26 shared `scripts/*.sh` at the deployed shared home (new step), recording checksums + ownership in the manifest.
2. A `/CJ_*` skill preamble resolves a shared script name through the 3-tier chain: `$REPO_ROOT/scripts` (self-dev), else the deployed home, else `.source` (legacy).
3. On a consumer machine with no source clone, tier (1) and (3) miss; tier (2) resolves; the skill proceeds.
4. `/CJ_portability-audit` reads the catalog + scans the preambles and reports the 4 orchestrator-family skills as `local-only`.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| `.source` lifetime | Keep as tier (3) legacy fallback in S1 | Remove `.source` now | Non-breaking + reversible; the live install keeps working; removal is S4 |
| Resolution duplication | One shared idiom reused across the 4 skills | Per-block independent copies | Single source of truth; cheaper to retarget at S2 (bundle) and S4 (drop `.source`) |
| Target tier | `local-only` | `standalone` | Skills still depend on a deployed location outside their own dir until the single-bundle layout (S2) |
| Deposit ownership | Manifest checksum-tracked, like templates | Untracked copy | Enables `skills-deploy doctor` drift detection (P2) + clean `remove` |
