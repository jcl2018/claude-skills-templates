## Problem Statement

A workbench maintainer who installed the install==clone bundle (S2's `--bundle`)
wants to **develop the workbench IN that bundle** — branch, edit, push, open a PR —
without keeping the separate external clone. After S2 the bundle is a git checkout
with the flat `/CJ_*` symlinked into it, so editing there reflects live. But there
is one blocker: `--bundle` clones from the LOCAL `.source` for speed/offline, so
the bundle's `origin` points at that local clone — `git push`/PR does not reach
GitHub. This story makes the bundle a first-class dev checkout you can ship from,
and explicitly DEFERS the dangerous "retire the separate-clone machinery" half.

## Mental Model

```
S2:  --bundle → bundle is a git checkout, flat /CJ_* symlink into it (install==clone)
S3:  + repoint bundle origin → GitHub upstream   (so push/PR works FROM the bundle)
     + skills-deploy bundle-status                (see the dev checkout state)
     = develop-in-place: cd ~/.claude/skills/cj-workbench, branch, edit, push, PR
DEFER to S4: retire .source / the cj-feat-* worktree flow / post-land-sync
```

The separate-clone machinery stays WORKING (made optional, not removed) — additive
and reversible, the same posture as S1/S2.

## Acceptance Criteria

### Story #1: origin repointed to the GitHub upstream [core]

```
GIVEN --bundle cloned the bundle from a local .source and a GitHub upstream is known
WHEN  the install finishes
THEN  the bundle's `origin` remote URL is the GitHub upstream (not the local clone path)
```

### Story #2: bundle-status reports the dev checkout [core]

```
GIVEN a --bundle install completed
WHEN  skills-deploy bundle-status runs
THEN  it prints install_mode=bundle, the bundle path, branch, origin, and tree state
```

### Story #3: non-bundle install reports dev-clone [resilience]

```
GIVEN an install that is NOT in bundle mode
WHEN  skills-deploy bundle-status runs
THEN  it reports install_mode=dev-clone and "Not in bundle mode" (no false bundle claim)
```

### Story #4: additive — machinery untouched [integration]

```
GIVEN the default skills-deploy install and the separate-clone machinery
WHEN  S3 lands
THEN  they are unchanged (no .source / worktree / post-land-sync rip-out)
```

### Story #5: develop-in-place documented [usability]

```
GIVEN the S3 changes
WHEN  a maintainer reads `skills-deploy` usage / the install summary
THEN  develop-in-place + `bundle-status` are documented
```

## Architecture

```
  skills-deploy install --bundle
        │
        ▼
  do_bundle_install
        │  bundle_upstream = SKILLS_DEPLOY_BUNDLE_UPSTREAM | manifest.upstream_url
        │  ensure checkout (S2: clone / ff)
        ▼
   repoint:  git -C <bundle> remote set-url origin <bundle_upstream>   (when differs)
        │
        ▼
   delegate + stamp (S2)
        ▼
   develop in place:  cd <bundle>; branch; edit; push; PR

  skills-deploy bundle-status  →  do_bundle_status (read manifest + git -C <bundle>)
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `scripts/skills-deploy` | claude-skills-templates | Modified | `do_bundle_install` origin-repoint + the new `do_bundle_status` subcommand + dispatcher/usage |
| `scripts/test.sh` | claude-skills-templates | Modified | S000087 hermetic test (origin-repoint + bundle-status + the non-bundle case) |

### Data Flow

1. `--bundle` resolves `bundle_upstream` (env override → manifest `upstream_url`).
2. After ensuring the bundle checkout, it repoints the bundle's `origin` to `bundle_upstream` when they differ.
3. The delegated install + manifest stamp run as in S2.
4. `skills-deploy bundle-status` reads the manifest (install_mode, bundle_path) + the bundle's git state and reports it.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| S3 scope | Develop-in-place enablement; defer the rip-out to S4 | Rip out `.source` + the worktree flow now | The rip-out is subtractive, undesigned, and retires the running machinery — unsafe in one PR |
| Push enablement | Repoint `origin` to the upstream | Always clone from the upstream (network) | Local-`.source` clone is faster/offline; repointing origin gets both speed AND a pushable remote |
| Status surface | A `bundle-status` subcommand | Fold into `doctor` | A dedicated, scannable command for the in-place dev checkout |
