## Problem Statement

A workbench operator wants the gstack experience: the CJ_ family installed as ONE
self-contained git checkout under `~/.claude/skills/`, where the install dir IS the
checkout (install == clone) — zero install-drift, `git pull` to upgrade. Today the
CJ_ family installs as flat symlink dirs pointing at an **external** dev clone
(`~/Documents/projects/...`), so the install is NOT self-contained. S2 adds a
managed-bundle install mode. The parent design flagged this as blocked on **O1**
(how Claude Code discovers `/CJ_*` from a bundle dir); this story resolves O1 and
delivers the mode.

## Mental Model

O1 is answered by the gstack reference bundle: Claude Code discovers skills from
**flat** `~/.claude/skills/<name>/SKILL.md`; gstack **flat-exports** each skill via a
symlink INTO its `~/.claude/skills/gstack/` git checkout. The CJ_ family already
installs as flat symlink dirs — the only delta is the symlink TARGET. So:

```
skills-deploy install --bundle
  ├─ ensure a managed git checkout at the bundle path (clone if absent, else ff)
  └─ delegate to <bundle>/scripts/skills-deploy install
       (its REPO_ROOT = <bundle>, so the existing per-file-symlink install
        symlinks flat ~/.claude/skills/CJ_<name>/* INTO <bundle>/skills/CJ_<name>/*)
  └─ stamp the manifest: install_mode=bundle, bundle_path, bundle_commit, source=<bundle>
```

Additive + opt-in: the default `install` (dev-clone symlinks) is untouched.

## Acceptance Criteria

### Story #1: --bundle ensures a managed git checkout [core]

```
GIVEN no checkout at the bundle path and a reachable clone source
WHEN  skills-deploy install --bundle runs
THEN  a workbench git checkout exists at the bundle path (a .git dir)
```

### Story #2: flat skills symlink into the bundle [core]

```
GIVEN a --bundle install completed
WHEN  the flat ~/.claude/skills/CJ_goal_feature/SKILL.md is inspected
THEN  it is a symlink whose target is <bundle>/skills/CJ_goal_feature/SKILL.md
```

### Story #3: install==clone recorded in the manifest [core]

```
GIVEN a --bundle install completed
WHEN  the manifest is read
THEN  install_mode == "bundle", bundle_path points at the bundle, and source == the bundle
```

### Story #4: additive — default install untouched [integration]

```
GIVEN the default skills-deploy install (no --bundle)
WHEN  it runs
THEN  flat skills symlink to the dev clone (not a bundle) and no bundle marker is written
```

### Story #5: offline bootstrap from a local source [resilience]

```
GIVEN SKILLS_DEPLOY_BUNDLE_SOURCE points at a LOCAL git checkout (no network)
WHEN  skills-deploy install --bundle runs
THEN  the bundle is cloned from that local source and install==clone holds
```

## Architecture

```
  skills-deploy install --bundle
        │
        ▼
  do_bundle_install (scripts/skills-deploy)
        │  resolve bundle path  (SKILLS_DEPLOY_BUNDLE_TARGET | ~/.claude/skills/cj-workbench)
        │  resolve clone source (SKILLS_DEPLOY_BUNDLE_SOURCE | manifest .source | upstream_url)
        ▼
   ensure checkout:  git clone <src> <bundle>   (or  git -C <bundle> pull --ff-only)
        │
        ▼
   delegate:  bash <bundle>/scripts/skills-deploy install   (REPO_ROOT = <bundle>)
        │        → flat ~/.claude/skills/CJ_<name>/* symlink INTO <bundle>/skills/CJ_<name>/*
        │        → _cj-shared scripts + templates + rules deposited FROM <bundle>
        ▼
   stamp manifest:  install_mode=bundle, bundle_path, bundle_commit, source=<bundle>
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `scripts/skills-deploy` | claude-skills-templates | Modified | New `do_bundle_install` + the `--bundle` flag short-circuit in `do_install` + a usage note |
| `scripts/test.sh` | claude-skills-templates | Modified | S000086 hermetic test: install==clone + the default-install-untouched additive guarantee |

### Data Flow

1. `skills-deploy install --bundle` resolves the bundle path + clone source.
2. It ensures a managed git checkout at the bundle path (clone, or best-effort fast-forward).
3. It delegates to the bundle's own `skills-deploy install`; that child's `REPO_ROOT` is the bundle, so the flat `/CJ_*` dirs + `_cj-shared` scripts + templates symlink/deposit from the bundle.
4. It stamps the manifest with the install==clone receipt.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Install mechanism | Delegate to the bundle's own install (REPO_ROOT=bundle) | Parameterize the whole installer on a `SRC_ROOT` variable | O1 showed discovery is unchanged; delegation reuses the install verbatim with zero parameterization — much lower risk |
| Bundle provenance | A managed clone (separate from the dev clone) | Move/designate the existing dev clone | Keeps S2 non-disruptive; making the bundle the dev checkout is S3 |
| Default behavior | Unchanged; `--bundle` opt-in | Flip the default to bundle | Reversible + un-brickable; the flip is S4 |
