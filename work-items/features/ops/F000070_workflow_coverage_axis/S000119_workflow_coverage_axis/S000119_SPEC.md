## Problem Statement

A maintainer of this workbench can document a `CJ_goal_*` workflow in
`spec/workflow-spec.md` (auto-rendered into `docs/workflows/`) with NO test that
the workflow actually runs. The F000066 `behaviors:` axis reserves
`level: workflow` for exactly this, but 0 such behaviors exist and nothing forces
one. The maintainer's pain: a "covered" workflow surface that proves nothing, and
no structural guard against adding a 5th orchestrator with zero workflow test.

## Mental Model

Two halves, deliberately split for honesty:

- **The GATE** (plain CI, registry-only, no API): forward + reverse cross-check
  between the workflow registry (`workflow-spec.sh --list-orchestrators`) and the
  `level: workflow` behaviors in `spec/test-spec-custom.md`. Green from birth;
  enforces "documented-but-untested impossible" everywhere.
- **The EVAL** (nightly / local, needs `ANTHROPIC_API_KEY`): real Claude-driven
  `tests/eval/<skill>/<case>/` runs via `eval.sh` (`claude --print`,
  `--plugin-dir <workbench>/skills`, `--json-schema`), each targeting a
  gstack-independent path (`task` → `halted_at_too_complex`; `feature`/`defect` →
  `--dry-run dry_run_preview`).

The `behavior_coverage:` row is the link: behavior → `suite-eval` unit → the case
`prompt.md` `source` → a live `anchor` literal.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group. -->

### Story #1: Real eval case per missing orchestrator [integration]

```
GIVEN tests/eval/CJ_goal_task/halt-too-complex/, tests/eval/CJ_goal_feature/dry-run-plan/,
      and tests/eval/CJ_goal_defect/dry-run-plan/ each have a prompt.md + expected.schema.json
WHEN  eval.sh runs the case (claude --print, --plugin-dir <workbench>/skills, --json-schema)
THEN  CJ_goal_task asserts end_state/halt_class = halted_at_too_complex with the routing suggestion,
      and CJ_goal_feature/CJ_goal_defect assert end_state = dry_run_preview with a chain-plan/draft marker
```

### Story #2: level:workflow behaviors + behavior_coverage rows [core]

```
GIVEN spec/test-spec-custom.md
WHEN  test-spec.sh --list-behaviors / --list-behavior-coverage is run
THEN  4 level:workflow behaviors exist (one per orchestrator, each with a workflow: field),
      and 4 behavior_coverage rows link each to unit: suite-eval, source: the case prompt.md,
      anchor: a literal present in that prompt (grepped live -F by Check 5)
```

### Story #3: 6th-column parser + enum-checked workflow: field [core]

```
GIVEN the behaviors TSV now has columns id,statement,level,area,purpose,workflow
WHEN  test-spec.sh --validate runs
THEN  workflow: is optional, allowed ONLY on level: workflow rows, MUST equal a declared
      orchestrator (enum-check); the - placeholder round-trips to "" on read; the
      rendered-field work-item-ID lint is preserved
```

### Story #4: forward/reverse gate, HARD + registry-gated [resilience]

```
GIVEN test-spec.sh --check-workflow-coverage and a new validate.sh check
WHEN  validate.sh runs in plain CI (no API)
THEN  forward: every --list-orchestrators name has >=1 matching level:workflow behavior (else FINDING);
      reverse: every level:workflow behavior's workflow: resolves to a declared orchestrator (else FINDING);
      registry-absent (workflow-spec or test-spec) => inactive note + exit 0
```

### Story #5: /CJ_test_audit surfacing [observability]

```
GIVEN /CJ_test_audit
WHEN  the audit runs
THEN  Stage 1 prints test-spec.sh --check-workflow-coverage verbatim (stage1/ prefixed),
      and Stage 2 judges, per level:workflow behavior, whether the linked eval case drives the
      workflow through a real gstack-independent path vs a hollow prompt (FINDING: stage2/behavior:<id>)
```

### Story #6: consumer-absent SKIP [integration]

```
GIVEN a consumer repo with no spec/workflow-spec.md or no spec/test-spec.md
WHEN  the new validate.sh check / --check-workflow-coverage runs
THEN  it reports inactive and exits 0 — no error, no false finding (mirror of Check 24/26/27)
```

### Story #7: --list-orchestrators source [usability]

```
GIVEN workflow-spec.sh --list-orchestrators
WHEN  invoked
THEN  it emits orchestrator-kind names only (reusing _list_sections() + _section_kind()),
      NOT the 6 --list-workflows names (incl. roster) nor the skills-catalog.json jq set
```

## Architecture

<!-- High-level system design. -->

```
                      spec/workflow-spec.md (registry)
                                 |
                  workflow-spec.sh --list-orchestrators
                                 |
                                 v
   spec/test-spec-custom.md ---> test-spec.sh --check-workflow-coverage ---> FINDING / inactive / OK
     (level:workflow behaviors        |  forward: orchestrator -> behavior         |
      + behavior_coverage rows)       |  reverse: behavior.workflow -> orchestrator |
                                      |                                            v
                                      |                                   validate.sh (new check, HARD)
                                      |                                            |
                                      +--> /CJ_test_audit Stage 1 (verbatim)       +--> zzz-test-scaffold fixture
                                      +--> /CJ_test_audit Stage 2 (substance)

   behavior_coverage.source -> tests/eval/<skill>/<case>/prompt.md  (live anchor, -F grep)
                                          |
                                   eval.sh (suite-eval, anchor scripts/eval.sh) -- runs nightly/local w/ API key
```

### Data Flow

1. `validate.sh` (new check) probes registry presence; if `spec/workflow-spec.md` or `spec/test-spec.md` is absent → `inactive` + exit 0.
2. Else it calls `test-spec.sh --check-workflow-coverage`, which resolves `workflow-spec.sh` (repo-local→`_cj-shared`) and reads `--list-orchestrators`.
3. Forward: each orchestrator must have ≥1 `level: workflow` behavior whose `workflow:` equals it → FINDING if missing.
4. Reverse: each `level: workflow` behavior's `workflow:` must resolve to a declared orchestrator → FINDING on an orphan.
5. `/CJ_test_audit` Stage 1 prints the same call verbatim; Stage 2 judges per-behavior substance against the linked eval case `prompt.md`.
6. Separately (nightly/local), `eval.sh` runs the `tests/eval/<skill>/<case>/` cases with the API key and `--json-schema`-validates the emitted JSON.

## Tradeoffs

<!-- Design decisions with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Test mechanism | Real Claude-driven eval case (`CJ_goal_todo_fix` pattern) | Shell fixture / fresh-install fixture / `--dry-run`-stub | Two reviews proved stubs hollow; the eval actually runs the workflow. The operator refused the shell fixture twice. |
| Forward-link | Explicit optional `workflow:` 6th TSV column | Derived heuristic from `statement`/`area` | Deterministic + enum-checkable; closes OQ1. Costs the budgeted parser change. |
| Orchestrator source | `workflow-spec.sh --list-orchestrators` (registry) | `--list-workflows` (incl. roster) / `skills-catalog.json` jq | Roster names aren't orchestrators; the catalog is consumer-absent → breaks registry-gating. Closes OQ2. |
| Generated view | Deferred to a follow-up | Build `docs/tests/workflow-coverage.md` now | Highest blast radius (renderer + Check 26 reverse-sweep exemption + doc-spec row); the gate + Stage-2 substance deliver the value without it. |
