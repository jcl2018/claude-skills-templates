## Scope

Move `skills/company-workflow/` and `templates/company-workflow/` to a new top-level `deprecated/` directory (`deprecated/company-workflow/` with templates as a sub-directory) so `skills/` contains only deployable skills. Refactor `scripts/skills-deploy` (line 260, line 278) and `scripts/validate.sh` (catalog walker at line 30, orphan check at line 71) to derive paths from `skills-catalog.json` `files[]` / `templates[]` instead of hardcoding `skills/{name}/`. Retarget all 7 `MIRROR_SPECS` entries in `validate.sh` Error check 10 so the work-copilot byte-mirror invariant continues to hold. Add a `deprecated/README.md` explaining the directory's purpose. Update `test.sh` to use a `COMPANY_PATH` constant so the next move (if any) is trivial. Pure relocation — no SKILL.md content changes, no behavioral changes.

## Non-Goals

<!-- Explicit non-goals. Things this feature deliberately does NOT do, and
     why. Prevents scope creep during Implement and gives reviewers an
     unambiguous boundary. -->

- Removing `skills/company-workflow/` from the repo entirely — same byte-mirror reason as F000005's out-of-scope. We are *moving* the source, not deleting it.
- Refactoring `personal-workflow/` or `system-health/` to use the same catalog-driven path lookup — both are active and live at the conventional `skills/{name}/`. The catalog-driven refactor in `skills-deploy` and `validate.sh` is principled (works for any path) but only one entry exercises the new behavior today.
- Eliminating the `templates/` top-level directory — `personal-workflow/` templates and `doc-SKILL-DESIGN.md` still live there. Only `templates/company-workflow/` moves.
- Changing the SKILL.md frontmatter of company-workflow (description, allowed-tools, etc.) — out of scope for this feature, same as F000005's out-of-scope. Catalog `status: deprecated` is the canonical signal.
- Auto-uninstalling a previously-installed company-workflow on the next `install` run — same lifecycle stance as F000005. Users own removal via `skills-deploy remove`.
- Eliminating the work-copilot byte-mirror entirely (e.g., make the bundle reference the source at deploy time instead of duplicating files) — bigger architectural change, separate feature if/when needed.
- Adding an `archived` tier (third lifecycle state beyond `deprecated`) — premature; can be added if `deprecated/` proves too coarse.
