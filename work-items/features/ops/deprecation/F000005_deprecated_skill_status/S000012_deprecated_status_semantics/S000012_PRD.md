## Problem Statement

The `status` field in `skills-catalog.json` is decorative: only `scripts/generate-readme.sh` reads it, and every entry today is `"active"`. There is no way to mark a skill as "kept in the repo as upstream truth, but please don't install it on new machines."

The user (a single-developer maintainer of `claude-skills-templates`) has just shipped F000004 (the GitHub Copilot bundle in `work-copilot/`) and migrated to it on the Windows work machine. `skills/company-workflow/` cannot be deleted — `work-copilot/` byte-mirrors several of its files via `scripts/validate.sh` Error check 10 — but the user does not want `scripts/skills-deploy install` to put `company-workflow` into `~/.claude/skills/` on a fresh Mac. Today, it does.

## Mental Model

Three concentric concerns, each owned by a different file. They all read the same catalog signal:

```
catalog (skills-catalog.json: status field)
   |
   +--> install/doctor/remove   (scripts/skills-deploy)        — operational
   +--> validation              (scripts/validate.sh)          — guard
   +--> README rendering        (scripts/generate-readme.sh)   — visibility
```

All three start by reading `skills-catalog.json`; they branch on `status`. The change is additive: extend each consumer's branching to recognize `deprecated`.

## Acceptance Criteria

<!-- Given/When/Then format. One block per user story or logical group. -->

### Story #1: Install skips deprecated by default [core]

```
GIVEN a clean target machine with no ~/.claude/skills/ directory
  AND skills-catalog.json contains an entry { name: "company-workflow", status: "deprecated" }
WHEN  the maintainer runs `scripts/skills-deploy install`
THEN  ~/.claude/skills/company-workflow/ is NOT created
  AND the install summary line for company-workflow shows it was skipped
  AND active skills install normally
```

### Story #2: --include-deprecated installs everything [core]

```
GIVEN the same setup as Story #1
WHEN  the maintainer runs `scripts/skills-deploy install --include-deprecated`
THEN  ~/.claude/skills/company-workflow/ IS created
  AND the install behaves identically to today's behavior on a deprecated entry
  AND active skills also install (no exclusion)
```

### Story #3: One warn line per skipped skill [observability]

```
GIVEN skills-catalog.json has 1 deprecated entry and 3 active entries
WHEN  the maintainer runs `scripts/skills-deploy install` (no flag)
THEN  stdout contains exactly one line matching: WARN: skipping deprecated skill: company-workflow (use --include-deprecated to install)
  AND no other deprecated-related warnings appear
  AND the line is emitted to stdout (or stderr — pick one and document) consistently with the rest of the script
```

### Story #4: Doctor reports deprecated as INFO [usability]

```
GIVEN skills-catalog.json has company-workflow as deprecated
  AND ~/.claude/skills/company-workflow/ does not exist
WHEN  the maintainer runs `scripts/skills-deploy doctor`
THEN  the output line for company-workflow uses an INFO label (not WARN)
  AND the line indicates "deprecated — not installed by default"
  AND the doctor exit code is 0 (deprecated state is not a doctor failure)
```

```
GIVEN skills-catalog.json has company-workflow as deprecated
  AND ~/.claude/skills/company-workflow/ DOES exist (was installed via --include-deprecated)
WHEN  the maintainer runs `scripts/skills-deploy doctor`
THEN  the output line uses INFO, not WARN
  AND the line indicates "deprecated — installed (--include-deprecated)"
  AND the doctor exit code is 0
```

### Story #5: validate.sh rejects typos [resilience]

```
GIVEN skills-catalog.json has an entry with status: "depricated"  (typo)
WHEN  the maintainer runs `scripts/validate.sh`
THEN  the script exits non-zero
  AND the error message identifies the offending entry by name
  AND the message lists the valid status values (active, experimental, deprecated)
```

### Story #6: README has a Deprecated section [usability]

```
GIVEN skills-catalog.json has 1 deprecated entry and 3 active entries
WHEN  the maintainer runs `scripts/generate-readme.sh`
THEN  the rendered README has the active skills under their existing section
  AND deprecated skills appear under a new "Deprecated" section below
  AND each deprecated row shows the skill name, version, and a one-line description
```

## Out of Scope

- Deleting `skills/company-workflow/` from the repo — required by `work-copilot/` byte-mirror via `validate.sh` Error check 10.
- An `archived` tier or further status states — premature.
- Auto-uninstalling already-installed deprecated skills on the next install — too aggressive.
- Per-user / per-machine deprecation profiles — global only.
- In-skill SKILL.md frontmatter banner for deprecation — keep one canonical signal in the catalog.
- Dependency-graph-aware deprecation warnings — no skill-to-skill deps today.

## Assumptions

- The catalog schema currently uses string values for `status` (verified: `"active"` is a string in every entry).
- `scripts/skills-deploy` is bash and parses flags with either `getopts` or a `case` statement (to be confirmed during ARCHITECTURE phase).
- `generate-readme.sh` uses `jq` over `skills-catalog.json` (verified: `jq -r '.[] | ...'` is the current rendering pattern).
- The user is the only consumer of `scripts/skills-deploy install` — there are no third-party scripts that depend on the current install-everything behavior.
- README diffs are baselined in the same PR that flips `company-workflow` to deprecated; no separate "regenerate README" pass needed.
