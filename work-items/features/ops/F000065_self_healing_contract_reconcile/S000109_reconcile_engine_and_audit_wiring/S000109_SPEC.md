## Problem Statement

The two audit skills (`/CJ_doc_audit`, `/CJ_test_audit`) already own the canonical
contract-file format (a 3-column Markdown table, their `--seed` template) and
position (`spec/`, root accepted as a fallback) and already seed a MISSING contract
file. They do nothing useful for a contract file that exists in a non-canonical
shape: a `doc-spec.md` still on the old YAML generation is rejected with
`[doc-sync-no-config] … has no registry table` and a dead stop. The operator wants
the audits to be idempotent across *generations* of the contract — detect + classify
a legacy / duplicated / wrong-position file, refresh it to canonical preserving the
declared rows, and at least prompt (report-borne, since the audit cannot run an
interactive AUQ) — so a future audit in a new or legacy repo "just handles it."

## Mental Model

Two internal phases inside one PR, each verified green before the next:

- **Phase 1 (engines):** `scripts/doc-spec.sh` and `scripts/test-spec.sh` each gain
  a read-only `--classify` (emit `GENERATION`/`POSITIONS`/`DUPLICATE`/`CANONICAL_PATH`)
  and an opt-in `--reconcile` (the ONLY new write path: migrate legacy→canonical
  preserving every declared row, atomic temp→`--validate`-clean→`mv`, `.bak` kept,
  migration report; idempotent no-op on canonical; the `audit_class` asymmetry
  guard). `legacy` requires BOTH "no canonical registry table" AND an old-generation
  signature match — a no-signature no-table file stays the `[doc-sync-no-config]`
  halt, so a malformed canonical file is never clobbered.
- **Phase 2 (skills):** the audit skills' Step-2 "seed if missing" generalizes to a
  reconcile step driven by `--classify` (absent → seed; canonical → ok;
  legacy/duplicate → an advisory `RECONCILE:` directive into the Stage-1 report, NO
  auto-write). An opt-in audit `--reconcile` flag forwards to the engine (standalone
  only — the in-QA path never passes it). The canonical contract-file template is
  documented in the USAGE.md + spec prose; classify/reconcile fixtures are registered
  in `scripts/test.sh` + `spec/test-spec-custom.md`.

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. -->

### Story #1: --classify labels generation + position [core]

```
GIVEN a repo whose contract file is absent / canonical / legacy YAML / duplicated
WHEN  `doc-spec.sh --classify` runs (read-only, no write)
THEN  it emits GENERATION=<canonical|legacy|absent>, POSITIONS=<comma-list>,
      DUPLICATE=<0|1>, CANONICAL_PATH=<spec/ path>,
      AND `legacy` is reported ONLY when no canonical registry table parses AND the
      file matches the old-generation signature (fenced ```yaml + schema_version: +
      docs:); a no-signature no-table file stays the [doc-sync-no-config] halt.
```

### Story #2: --reconcile migrates legacy preserving every row [core]

```
GIVEN a legacy YAML doc-spec fixture with N declared rows (incl. a 40+-row fixture)
WHEN  `doc-spec.sh --reconcile` runs
THEN  it writes the canonical 3-column Markdown table preserving all N rows
      (path→Doc, purpose→Purpose, requirement→Requirement; dropping
      section/audit_class/front_table), atomically (temp → --validate-clean → mv),
      keeps a <path>.bak, emits the migration report (migrated N rows / backup
      written / dropped fields),
      AND a re-run on the now-canonical file is a clean no-op (RECONCILE: already
      canonical).
```

### Story #3: audit_class asymmetry guard [resilience]

```
GIVEN a legacy fixture row at a docs/* path declared audit_class: operational
WHEN  `doc-spec.sh --reconcile` migrates it
THEN  it emits RECONCILE-WARN: <path-row> audit_class was 'operational' but path
      derives 'human-doc' — verify no work-item IDs.
```

### Story #4: test-spec gets the symmetric classify/reconcile [core]

```
GIVEN scripts/test-spec.sh
WHEN  the build adds --classify (+ --reconcile)
THEN  --classify labels absent/canonical/legacy/duplicate symmetrically,
      AND --reconcile migrates whatever legacy form exists (or is a dedup/no-op if
      test-spec never had a divergent on-disk legacy format — still symmetric).
```

### Story #5: read-mostly default, opt-in write, no canonical noise [usability]

```
GIVEN a legacy/duplicate repo and, separately, the canonical workbench
WHEN  a plain (no-flag) `/CJ_doc_audit` runs, then `/CJ_doc_audit --reconcile`
THEN  the plain run emits an advisory RECONCILE: directive into the Stage-1 report
      naming the issue + remedy and writes NOTHING,
      AND `--reconcile` performs the migration,
      AND a canonical repo emits ZERO reconcile lines on a plain run,
      AND the in-QA path never passes --reconcile.
```

### Story #6: suite green + tests registered in both homes [integration]

```
GIVEN the new classify/reconcile fixtures
WHEN  `./scripts/validate.sh` and `./scripts/test.sh` run
THEN  both exit green (validate 0/0), the workbench classifies `canonical` with no
      reconcile noise,
      AND every new tests/*.test.sh is registered in BOTH scripts/test.sh and
      spec/test-spec-custom.md (no unregistered-test gate failure).
```

### Story #7: canonical contract-file template documented [usability]

```
GIVEN skills/CJ_doc_audit/USAGE.md, skills/CJ_test_audit/USAGE.md, spec/doc-spec.md,
      spec/test-spec.md
WHEN  the build adds the canonical contract-file template section
THEN  each names the required file (the general file of each pair), the optional
      file (the *-custom.md overlay), the canonical position (spec/, root fallback),
      and the format (3-column Markdown table).
```

## Architecture

<!-- High-level system design. -->

```
PHASE 1 (engines)                         PHASE 2 (audit-skill wiring + docs + tests)

scripts/doc-spec.sh                       skills/CJ_doc_audit/SKILL.md
   ├─ --classify (read-only)                 ├─ Step 2: seed-if-missing → reconcile
   │    GENERATION/POSITIONS/                 │    absent→seed; canonical→ok;
   │    DUPLICATE/CANONICAL_PATH              │    legacy/dup→advisory RECONCILE:
   │    (legacy = no-table + old sig)         └─ opt-in --reconcile → engine (standalone)
   └─ --reconcile (ONLY new write, opt-in)  skills/CJ_test_audit/SKILL.md  (mirror)
        legacy→canonical, preserve rows,
        atomic temp→--validate→mv, .bak,    skills/CJ_*_audit/USAGE.md + spec/*-spec.md
        report, idempotent,                    └─ canonical contract-file template prose
        audit_class asymmetry guard
                  |                          skills-catalog.json (flag/version — sensitive)
scripts/test-spec.sh  (symmetric)                  |
   ├─ --classify                            tests/*.test.sh (classify + reconcile fixtures,
   └─ --reconcile (scoped/dedup-no-op)         incl. 40+-row legacy, duplicate, asymmetry,
                                               idempotent-re-run)
                                                   |
                                            scripts/test.sh (register + integration drills)
                                            spec/test-spec-custom.md (register units)
```

### Data Flow

<!-- How data moves for the primary use case (a reconcile run). -->

1. `/CJ_doc_audit` (Step 2) invokes `doc-spec.sh --classify`.
2. `doc-spec.sh` probes the contract file(s) at `spec/`-then-root, emits
   `GENERATION`/`POSITIONS`/`DUPLICATE`/`CANONICAL_PATH`.
3. The skill routes on `GENERATION`: `absent` → seed (today's behavior); `canonical`
   → ok; `legacy` / `duplicate` → emit an advisory `RECONCILE:` directive into the
   Stage-1 report (no write).
4. If (and only if) the operator passes `--reconcile` (standalone), the skill forwards
   to `doc-spec.sh --reconcile`, which parses the old entries, writes the canonical
   table atomically (temp → `--validate`-clean → `mv`), keeps a `.bak`, emits the
   migration report + any `RECONCILE-WARN` asymmetry guard line.
5. `/CJ_test_audit` runs the symmetric flow over `test-spec.sh`.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Reconcile mechanism | Approach A: classify + flag-gated migrate, preserve declared rows | B: auto re-seed + prompt | Re-seed replaces a 47-doc registry with the ~10-row template (destructive); A preserves every row |
| "Prompt to update" | Report-borne advisory `RECONCILE:` directive + opt-in flag | Interactive AUQ in the audit (Approach C) | The in-QA subagent can't prompt; C only works standalone and breaks the read-mostly / no-AUQ design |
| Duplicate handling | Report-only; reconcile the canonical copy, never auto-delete | Auto-delete the redundant copy | Deleting on a plain reconcile is unsafe; `--prune-duplicates` deferred |
| `legacy` detection | Gate on an old-generation *signature* (fenced yaml + schema_version + docs) | Label any no-table file as legacy | Without the gate, `--reconcile` would clobber a hand-broken canonical file |
| Migration parser | awk/sed, POSIX-shell / bash-3.2, no python/yaml dep | A yaml-aware converter | Matches the existing `doc-spec.sh` / `test-spec.sh` idiom; no new runtime dep |
| Write atomicity | temp → `--validate`-clean → `mv` + `.bak` | In-place rewrite | Guarantees no half-written registry and a recoverable original |

## Open Questions

| Question | Next check |
|----------|-----------|
| OQ1 — auto-delete duplicates? | v1 reports + reconciles the canonical copy but does NOT delete the redundant file. A future `--reconcile --prune-duplicates` could remove it after the canonical write verifies. Deferred. |
| OQ2 — root→spec relocation? | Root is an accepted position; v1 treats root-only as advisory `wrong-position` (reported, not moved). Relocation is opt-in future work. |
| OQ3 — test-spec legacy signature? | The exact old test-spec on-disk signature must be confirmed from git history during implementation; if test-spec never had a divergent legacy format, `--classify` reduces to canonical/absent + duplicate and `--reconcile` is a dedup/no-op (still symmetric, less converter work). |
