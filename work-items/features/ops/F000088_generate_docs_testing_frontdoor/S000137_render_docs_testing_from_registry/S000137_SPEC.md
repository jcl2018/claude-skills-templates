## Problem Statement

A workbench maintainer (or another machine) who wants to understand, run, audit,
and verify the test suite today has to read four disconnected surfaces
(`spec/test-spec.md`, `docs/philosophy.md`, `docs/test-catalog.md`,
`spec/test-spec-custom.md`). There is no single front door. This story specifies
the generator that emits one non-driftable page, `docs/testing.md`.

## Mental Model

`docs/testing.md` is a GENERATED page (like `docs/test-catalog.md`): fixed template
prose for the narrative, plus registry-derived indexes for the live data. It is
rendered by `test-spec.sh --render-docs` and diffed by `--render-docs --check`, so
the existing `validate.sh` Check 26 keeps it fresh. Nine sections: what-it-proves →
model-at-a-glance → RUN → AUDIT → VERIFY → behaviors index → categories index →
enrolled topics → drill-down links.

## Acceptance Criteria

<!-- Given/When/Then. One block per requirement. #n matches the Requirements # column. -->

### Story #1: docs/testing.md is emitted with all nine sections [core]

```
GIVEN a repo with a merged test-spec registry (general + custom overlay)
WHEN  `test-spec.sh --render-docs` runs
THEN  `docs/testing.md` is written with the nine sections and the
      "GENERATED FILE — do not edit by hand … Check 26 enforces freshness" header
```

### Story #2: --render-docs --check guards the page [integration]

```
GIVEN `docs/testing.md` exists on disk
WHEN  a section is hand-edited and `test-spec.sh --render-docs --check` runs (as validate.sh Check 26)
THEN  the check reports a diff and exits non-zero (the hand-edit is caught)
```

### Story #3: indexes match the live registry [core]

```
GIVEN the merged registry declares 17 behaviors + 28 categories + the enrolled topics
WHEN  `docs/testing.md` is rendered
THEN  its behaviors / categories / enrolled-topics indexes list exactly those rows,
      and adding/removing a registry row changes the rendered page on next render
```

### Story #4: declared in the doc contract, no orphan, no work-item IDs [integration]

```
GIVEN `docs/testing.md` is generated
WHEN  `doc-spec.sh --validate` + `--check-on-disk` run
THEN  the file is a declared generated human-doc (in spec/doc-spec-custom.md),
      present, non-orphan, and contains no [FSTD]NNNNNN work-item IDs (Check 19 clean)
```

### Story #5: idempotent, override-aware, seed unchanged [resilience]

```
GIVEN the renderer honors the TESTDOC_OUT/docs-root override
WHEN  `--render-docs` runs twice (and in a temp-dir drill)
THEN  the output is byte-identical on re-render AND `spec/test-spec.md` stays
      byte-identical to `--seed` (seed-identity test green)
```

## Architecture

<!-- High-level design. Which components change; ASCII data flow. -->

```
spec/test-spec.md ─┐
                   ├─(merge)─> test-spec.sh registry parse ─> _render_testing_md ─> docs/testing.md
spec/test-spec-custom.md ─┘        │  (reuses --list-behaviors / --list-categories / topic_contracts parsers)
                                   │
             --render-docs ────────┤ (write path)
             --render-docs --check ┘ (diff path → validate.sh Check 26)

spec/doc-spec-custom.md ─(declares docs/testing.md as generated human-doc)─> doc-spec.sh --check-on-disk
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `scripts/test-spec.sh` | (this repo) | Modified | Add `_render_testing_md`; wire into `--render-docs` write + `--render-docs --check` diff paths; honor `TESTDOC_OUT`/docs-root override |
| `spec/doc-spec-custom.md` | (this repo) | Modified | Declare `docs/testing.md` as a generated human-doc row |
| `docs/testing.md` | (this repo) | New | Generated front-door page (output only — never hand-edited) |
| `tests/test-spec.test.sh` | (this repo) | Modified | Add a render/idempotency/freshness drill for `docs/testing.md` |

### Data Flow

1. The two-tier test-spec registry (general `spec/test-spec.md` + overlay `spec/test-spec-custom.md`) is merged and parsed by `test-spec.sh`.
2. `_render_testing_md` composes the nine sections: fixed template prose + the behaviors/categories/topics indexes from the reused list-parsers.
3. `--render-docs` writes the composed page to `docs/testing.md` (or the `TESTDOC_OUT` docs-root in drills).
4. `--render-docs --check` re-renders to a temp dir and diffs vs on-disk; `validate.sh` Check 26 runs this and fails on any mismatch.
5. `doc-spec.sh --check-on-disk` sees `docs/testing.md` declared in the overlay and present (no orphan).

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Front-door prose depth | Approach A — concise generated why + link to `philosophy.md`/`test-spec.md` | Approach B — embed the full verification principle verbatim | B duplicates `philosophy.md` (a second copy to keep in sync — the drift this roadmap fights) |
| Freshness enforcement | Reuse Check 26 via `--render-docs --check` | Add a new dedicated validate check | Emitting from the same render path means Check 26 covers it for free — less surface, same guarantee |
| Doc declaration site | `spec/doc-spec-custom.md` overlay | General `spec/doc-spec.md` seed | The seed must stay byte-identical to `--seed`; repo-specific docs belong in the overlay |
