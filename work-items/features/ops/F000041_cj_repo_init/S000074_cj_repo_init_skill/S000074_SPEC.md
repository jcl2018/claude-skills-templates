## Problem Statement

A workbench operator (or an adopting repo's maintainer) has the CJ_ skill family installed
under `~/.claude/` but the target repo is missing the per-repo config files those skills
require at runtime. The result is opaque mid-pipeline HALTs (`[doc-sync-no-config]`) and
exit-1 failures (`/CJ_suggest` with no `TODOS.md`). They need one command that tells them
exactly what is missing and offers to create it from safe generic defaults.

## Mental Model

Three phases, all in one script: (1) **detect** which CJ_ skills are deployed; (2) **map**
each detected skill to its per-repo prerequisite(s) and **verify** the union; (3) on
`--fix`, **scaffold** the missing repo-level prerequisites from inline generic seeds. The
SKILL.md wraps phase 1+2 (always) and gates phase 3 behind one confirm AUQ. `--dry-run`
runs 1+2 only. Re-running when green is a no-op that just reprints the health table.

The prerequisite map:

| Prerequisite | Path (repo root) | Needed by | Absent ⇒ | Seed strategy | Level |
|---|---|---|---|---|---|
| Doc-sync config | `cj-document-release.json` | CJ_document-release, CJ_goal_feature/defect/todo_fix | hard HALT `[doc-sync-no-config]` | generic `schema_version:1` + portable whitelist + 2 generic categories | repo |
| Backlog | `TODOS.md` | CJ_suggest, CJ_goal_todo_fix, CJ_improve-queue | exit 1 | minimal `## Active work` skeleton + format comment | repo |
| Work-item tree | `work-items/{features,defects,tasks}/` | scaffold/implement/qa, CJ_personal-workflow, CJ_goal_feature/defect | soft (scaffold self-creates) | `mkdir -p` the three dirs | repo |
| Workflow assets | `~/.claude/skills/CJ_personal-workflow/` manifest + templates | CJ_personal-workflow + all pipeline phases | hard refusal | report only → "Run: <source>/scripts/skills-deploy install" | install |

## Acceptance Criteria

### Story #1: detect deployed skills + build prereq set [core]

```
GIVEN a repo with one or more CJ_ skills deployed under ~/.claude/ (or repo-local skills/ for self-dev)
WHEN  cj-repo-init.sh runs with no flags
THEN  it reads the deployed-skills manifest (with documented fallbacks) and computes the union of required per-repo prerequisites
```

### Story #2: health table + machine-readable output [observability]

```
GIVEN detection has run
WHEN  the script prints its result
THEN  it emits a human-readable `prereq | needed-by | status` table AND a machine-readable `GAPS=<n>` line plus one line per gap for the SKILL.md to consume
```

### Story #3: --fix scaffolds repo-level prereqs [core]

```
GIVEN repo-level gaps exist (missing TODOS.md / cj-document-release.json / work-items dirs)
WHEN  cj-repo-init.sh --fix runs
THEN  it writes the missing files/dirs from generic inline seeds and never touches install-level gaps
```

### Story #4: --dry-run writes nothing [resilience]

```
GIVEN any repo state
WHEN  cj-repo-init.sh --dry-run runs
THEN  no file or directory is created or modified, and the exit code reflects the gap count
```

### Story #5: exit code contract [core]

```
GIVEN detection has run
WHEN  the script exits
THEN  exit code is 0 when no repo-level gaps remain and 1 when repo-level gaps remain (install-level gaps are reported but do not fail the repo-level contract)
```

### Story #6: config validity beyond existence [security]

```
GIVEN a cj-document-release.json that is present but unparseable or has an unsupported schema_version
WHEN  detection runs
THEN  it is reported as a gap; AND any config scaffolded by --fix is parseable JSON with schema_version 1 (passes validate.sh Check 16)
```

### Story #7: SKILL.md single-confirm wrapper [usability]

```
GIVEN gaps are detected
WHEN  /CJ_repo-init runs
THEN  it prints the table, surfaces exactly ONE AskUserQuestion (scaffold now / dry-run-only / cancel), calls --fix on confirm, re-runs detection, and prints the post-fix table — all in-place, no worktree/ship
```

### Story #8: tests wired into suite [integration]

```
GIVEN the script and skill exist
WHEN  scripts/test.sh runs
THEN  tests/cj-repo-init.test.sh executes and passes, and the zzz-test-scaffold integration fixture is updated to include the new surface
```

### Story #9: clean degradation [resilience]

```
GIVEN the script runs outside a git repo OR with the deployed manifest absent
WHEN  detection runs
THEN  it errors or degrades with a clear message rather than crashing
```

## Architecture

```
/CJ_repo-init (SKILL.md)
  |
  |-- runs --> scripts/cj-repo-init.sh            (detect + verify; prints table + GAPS=n)
  |               |
  |               |-- reads ~/.claude/.skills-templates.json   (fallback: ls ~/.claude/skills/CJ_*; repo-local skills/)
  |               |-- maps detected skills -> prereq set
  |               |-- verifies each prereq (exists? valid?)    (cj-document-release.json: parse + schema_version)
  |               '-- emits health table + GAPS=<n> + per-gap lines
  |
  '-- on gaps: ONE confirm AUQ (prose-owned)
        |-- confirm  --> cj-repo-init.sh --fix    (heredoc seeds: TODOS.md, cj-document-release.json, work-items/ dirs)
        |-- dry-run  --> cj-repo-init.sh --dry-run (no writes)
        '-- cancel   --> stop
```

### Data Flow

1. `/CJ_repo-init` invokes `cj-repo-init.sh` (default mode).
2. Script resolves the deployed-skills set from `~/.claude/.skills-templates.json` (fallback to `ls ~/.claude/skills/CJ_*`, then repo-local `skills/`).
3. Script maps each detected skill to its per-repo prerequisite(s) and verifies each (existence; for `cj-document-release.json`, parse + supported `schema_version`).
4. Script prints the health table and emits `GAPS=<n>` + per-gap lines; exits 0 (no repo-level gaps) or 1 (gaps remain).
5. SKILL.md reads the output; if gaps, it surfaces one confirm AUQ.
6. On confirm, SKILL.md calls `cj-repo-init.sh --fix`, which writes the missing seeds; SKILL.md re-runs detection and prints the post-fix table.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Logic location | bash engine in `scripts/` + AUQ in SKILL.md prose | all logic embedded in SKILL.md bash fences | Unit-testable, idempotent, QA-able, matches repo convention (`skills-doc-sync-check`) |
| Seed storage | inline heredocs in the script | `templates/CJ_repo-init/` deployment surface | Tiny generic files; avoids skills-catalog `templates` + manifest churn |
| Scope of fix | repo-level prereqs only | also install missing `~/.claude/` assets | Installation is owned by `setup.sh` / `skills-deploy`; CJ_repo-init reports but never installs |
