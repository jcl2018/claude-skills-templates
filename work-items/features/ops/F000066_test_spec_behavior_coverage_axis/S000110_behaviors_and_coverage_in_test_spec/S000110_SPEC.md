## Problem Statement

The maintainer of any repo adopting the `test-spec` contract (workbench or
consumer) cannot today *declare* that a specific behavior must be proven by a test.
The contract is closed-world over the existing test inventory: `--check-coverage`
flags an orphaned or mis-anchored test, but a required-but-untested behavior is
structurally invisible — there's nothing to miss. Concrete pain: an options
portfolio repo has no place to assert "adding a short put is valid," so the
absence of that test never surfaces as a finding.

## Mental Model

The contract gains a third, orthogonal axis on top of the existing two:

```
layers[]  — WHERE/WHEN verification fires (local-hook / ci / pipeline-gate / ratchet)
rules[]   — IS the test inventory honest/complete (closed-world over existing tests)
units[]   — one row per verification MECHANISM (each check / suite / hook / workflow)
            └── NEW: behaviors[]          — WHAT the software must prove (open-world; carries `level`)
            └── NEW: behavior_coverage[]   — many-to-many: behavior ↔ test-bearing unit + semantic-evidence source/anchor
```

`level` lives on the *obligation* (the behavior), not the mechanism (a unit can
prove several levels). Two halves of the engine: a **deterministic** structural
check (`test-spec.sh`) verifies the links resolve and the anchor greps live; an
**agent-judged** substance check (`/CJ_test_audit` Stage-2) verifies the test
actually proves the behavior — load-bearing, because the deterministic half alone
merely relocates the blind spot to vague prose.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group. -->

### Story #1: Parser reads the two new blocks cleanly [core]

```
GIVEN an overlay with behaviors: and behavior_coverage: blocks
WHEN  test-spec.sh parses the merged registry
THEN  behaviors flush on `- id:`, behavior_coverage rows flush on `- behavior:` (no id),
      and none of the rules/units/layers/gates parsers absorb the new top-level keys
```

### Story #2: level enum + id-uniqueness gate [core]

```
GIVEN a behavior with level not in {unit,integration,contract,workflow,property} OR a duplicate behaviors[].id
WHEN  any registry-reading subcommand runs (--validate / --list-* / --check-coverage)
THEN  it halts with [test-spec-no-config]
```

### Story #3: coverage-link resolution [core]

```
GIVEN a behavior_coverage row whose .behavior is a typo (0 matches) or duplicated (2+),
      OR whose .unit points at a validate|ci|hook family
WHEN  test-spec.sh --check-coverage runs with behaviors: present
THEN  each case is reported as a finding
```

### Story #4: live-anchor + ≥1-cover [core]

```
GIVEN a behavior_coverage.anchor that does not grep live in its source, OR a behaviors[] row with zero coverage rows
WHEN  --check-coverage runs
THEN  the missing-anchor and uncovered-behavior cases are findings (anchor matched via grep -F, not _fwd_match)
```

### Story #6: consumer parity [resilience]

```
GIVEN a repo whose registry is absent, and separately a repo with units: but no behaviors:
WHEN  --check-coverage runs
THEN  the absent case is REGISTRY=absent + exit 0; the no-behaviors case prints
      "behavior coverage inactive" + exit 0 (no fabricated findings, no reverse-floor)
```

### Story #7: seed byte-identity [core]

```
GIVEN the prose + level enum added to spec/test-spec.md and the --seed heredoc updated in lockstep
WHEN  the seed-identity test runs (diff spec/test-spec.md against test-spec.sh --seed)
THEN  they are byte-identical and schema_version is still 1 with the machine block unchanged
```

### Story #9: agent-judged substance [core]

```
GIVEN a vague, over-claimed, or mis-leveled behavior row linked to a broad test
WHEN  /CJ_test_audit Stage-2 runs
THEN  it emits a finding prefixed stage2/behavior:<id> with cited evidence
```

### Story #10: dogfood green [core]

```
GIVEN the ~8 test-spec behavior + coverage rows in spec/test-spec-custom.md
WHEN  validate.sh and test.sh run on the live tree
THEN  both are green and every dogfood behavior resolves to a real, anchored, test-bearing cover
```

## Architecture

<!-- High-level system design for this story. -->

```
spec/test-spec.md (general seed)            spec/test-spec-custom.md (overlay)
  rules[] + layers[] (UNCHANGED machine block)  units[]  +  NEW behaviors[] + behavior_coverage[]
  + NEW prose: axis + level enum + caveat
        │                                              │
        └──────────────── merge ──────────────────────┘
                          │
              scripts/test-spec.sh
                ├── _parse_{rules,units,layers,gates}_file  (boundary regex += behaviors|behavior_coverage)
                ├── _parse behaviors (- id:) + behavior_coverage (- behavior:)
                ├── _run_registry_gates → Checks 1–2 (schema/enum/id-unique)
                ├── _run_coverage       → Checks 3–6 (link/family/anchor/≥1-cover; gated on behaviors:)
                ├── --list-behaviors / --list-behavior-coverage
                └── --validate lint (work-item-ID-free statement/purpose)
                          │
            ┌─────────────┴───────────────┐
   scripts/validate.sh Check 24      skills/CJ_test_audit/SKILL.md
   (hard loop runs behavior checks)  (Stage-2 per-behavior substance sub-check)
                          │
            tests/test-spec.test.sh + scripts/test.sh integration fixture
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `spec/test-spec.md` | claude-skills-templates | Modified | Seed prose + closed `level` enum; machine block + `schema_version` UNCHANGED |
| `spec/test-spec-custom.md` | claude-skills-templates | Modified | ~8 dogfood `behaviors:` + `behavior_coverage:` rows |
| `scripts/test-spec.sh` | claude-skills-templates | Modified | Parser for the two blocks; 6 checks; 2 plumbing items; `--seed` heredoc in lockstep |
| `scripts/validate.sh` | claude-skills-templates | Modified | Check 24 runs the new behavior checks in the same hard loop |
| `scripts/test.sh` | claude-skills-templates | Modified | Extend the `test-spec` sub-suite + the parallel integration-fixture edit |
| `tests/test-spec.test.sh` | claude-skills-templates | Modified | Parser round-trip + the new deterministic-check drills |
| `skills/CJ_test_audit/SKILL.md` | claude-skills-templates | Modified | New Stage-2 per-behavior substance sub-check |

### Data Flow

1. A maintainer adds a `behaviors:` row (id, statement, level, optional area/purpose) + ≥1 `behavior_coverage:` row (behavior, unit, source, anchor) to the overlay.
2. `test-spec.sh` merges general + overlay; the four existing per-block parsers honor the new boundary regex so the new keys don't bleed.
3. `_run_registry_gates` runs Checks 1–2 (schema/enum/id-uniqueness) for every registry-reading subcommand.
4. `_run_coverage` (gated on `behaviors:`) runs Checks 3–6: behavior resolution, test-bearing-unit join, live `grep -F` anchor, ≥1-cover.
5. `validate.sh` Check 24 calls the behavior checks inside its hard loop; a finding fails the gate.
6. `/CJ_test_audit` Stage-2 reads each behavior and judges substance against the linked test, emitting `stage2/behavior:<id>` findings.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Where `level` lives | On the behavior (obligation) | On `units[]` (mechanism) | One suite legitimately proves multiple levels; leveling the mechanism is wrong (Codex P3). |
| Schema versioning | Stay at `schema_version: 1`, arrays optional + overlay-only, seed machine block unchanged | Bump seed version + extend `SUPPORTED_SCHEMA_VERSIONS` | Seed machine block doesn't change, so a bump is meaningless and breaks the byte-identical-seed test + the one-fenced-yaml-block invariant. |
| Semantic-evidence enforcement | Deterministic check 5 only proves the anchor greps live; the "is this genuine evidence vs a runner path, and does the test actually prove it?" judgment is the agent-judged Stage-2's job | A deterministic `source != unit.source` guard in v1 | Kept minimal; the substance judgment is inherently agent-shaped (P5), and a structural guard would be both leaky and over-strict. |
| Anchor matching | Fixed-string `grep -F` | Route behavior anchors through the family-shaped `_fwd_match` dispatcher | Behavior anchors are arbitrary semantic-evidence prose, not `=== Check N` / runner-path shapes. |
| Coverage scope | v1 = the test-spec contract only, dogfood ~8 rows | Repo-wide behavior inventory + pyramid quotas + diff-aware now | Approach A (minimal honest v1); B is the deferred fast-follow. |
