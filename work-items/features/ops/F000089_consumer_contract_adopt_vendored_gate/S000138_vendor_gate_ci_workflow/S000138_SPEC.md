## Problem Statement

A team adopting the drift-proofing contract in a consumer repo wants structural
drift to red their PR automatically, the same way it does in the workbench â€” with
one command, no per-machine hook discipline, and no `~/.claude` dependency on the
CI runner. Today `skills-deploy install-contract-gate` seeds the contract and
installs a bypassable, `~/.claude`-dependent pre-commit hook, but it does not make
the consumer's *push* enforce the gate. The maintainer running the adopt command
needs the push path gated turnkey.

## Mental Model

`install-contract-gate` has three existing phases for a consumer repo: seed
contracts â†’ complete adoption (refresh surfaces, declare docs) â†’ install the
pre-commit hook. This story appends a fourth and fifth effect, consumer-only and
self-repo-skipped:

```
seed â†’ adopt â†’ hook   (existing)
              â”śâ”€ vendor  : copy 4 engines â†’ <consumer>/.cj-contract/  (new)
              â””â”€ drop    : write .github/workflows/cj-contract-gate.yml (new)
--remove: reverse hook + vendor + drop (sentinel-gated)
```

The gate script already resolves its sibling engines from its own directory as a
last resort, so co-locating all 4 in `.cj-contract/` is what makes CI work with no
`~/.claude`.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group. -->

### Story #1: Vendor the 4 engines [integration]

```
GIVEN a consumer repo (not the workbench self-repo)
WHEN  `skills-deploy install-contract-gate` runs
THEN  `.cj-contract/` contains cj-contract-gate.sh + doc-spec.sh + test-spec.sh + workflow-spec.sh,
      each with a stamped version header, LF line endings, and the executable bit set
```

### Story #2: Drop the CI workflow [integration]

```
GIVEN a consumer repo with no existing cj-contract-gate.yml
WHEN  `skills-deploy install-contract-gate` runs
THEN  `.github/workflows/cj-contract-gate.yml` exists, triggers on pull_request + push:main,
      and its job runs `bash .cj-contract/cj-contract-gate.sh --repo .`
```

### Story #3: Bare-runner gate proof [resilience]

```
GIVEN the vendored `.cj-contract/` and NO `~/.claude/_cj-shared` on the machine
WHEN  `.cj-contract/cj-contract-gate.sh --repo .` runs against (a) a clean seeded contract,
      (b) a planted structural violation, (c) an unadopted repo
THEN  it exits 0 (a), non-zero (b), and clean-SKIP 0 (c)
```

### Story #4: Symmetric `--remove` [usability]

```
GIVEN a consumer repo previously adopted (vendored .cj-contract/ + dropped workflow, both sentinel-marked/unmodified)
WHEN  `skills-deploy install-contract-gate --remove` runs
THEN  `.cj-contract/` and the dropped workflow are removed, and a hand-edited file is preserved
```

### Story #5: Self-repo skip [security]

```
GIVEN the workbench self-repo
WHEN  `skills-deploy install-contract-gate` runs
THEN  no `.cj-contract/` is vendored and no cj-contract-gate.yml is dropped (as seeding/adoption already skip)
```

### Story #6: Hand-authored workflow protection [usability]

```
GIVEN a consumer repo with a hand-authored `.github/workflows/cj-contract-gate.yml` that differs from the auto-drop
WHEN  `skills-deploy install-contract-gate` runs
THEN  the existing workflow is left untouched and a note is emitted (skip-with-note); only a prior sentinel-marked auto-drop is overwritten
```

### Story #7: Adopt doc [usability]

```
GIVEN the workbench
WHEN  a maintainer opens `docs/adopting-the-contract.md`
THEN  it documents run-adopt â†’ commit `.cj-contract/` + workflow â†’ drift reds the PR,
      is declared in spec/doc-spec-custom.md, and contains no work-item IDs
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. Maintainer runs `skills-deploy install-contract-gate` in a consumer repo.
2. The deploy detects it is NOT the self-repo (or skips early if it is).
3. Existing phases run: seed â†’ adopt â†’ hook.
4. Vendor step resolves the source scripts (repo-local `scripts/` or `_cj-shared`) and copies the 4 engines into `.cj-contract/`, stamping + LF-normalizing + `chmod +x`.
5. Drop step renders the CI-workflow template into `.github/workflows/cj-contract-gate.yml` (skip-with-note if hand-authored differs).
6. Maintainer commits `.cj-contract/` + the workflow; on the next push, CI checks out, installs jq, and runs the vendored gate â€” reding the PR on structural drift.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| How CI gets the engines | Vendor into `.cj-contract/` + re-sync via re-deploy | Reusable GH workflow/action; fetch-at-runtime | Offline, self-contained, no external coupling; matches install==clone |
| Where the CI template lives | Match existing skills-deploy idiom (heredoc unless a tracked file is cleaner) | Always a tracked `templates/` file | Consistency with the embedded pre-commit-hook body; avoid a needless new validate guard |
| `--remove` scope | Sentinel-gated (only auto-dropped / unmodified) | Unconditional delete | Never destroy a consumer's hand-edits |
