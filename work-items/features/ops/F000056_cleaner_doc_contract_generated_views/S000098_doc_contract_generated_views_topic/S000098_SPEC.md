## Problem Statement

The maintainer of this workbench wants `doc-spec.md` cleaner: a readable general vs
custom doc list, the doc-contract logic in one obvious philosophy home, and the
registry tidier — WITHOUT introducing a second source of truth. Today the Custom
section of `doc-spec.md` carries a hand-maintained table that duplicates the machine
registry, and the contract's rationale is spread across the registry prose.

## Mental Model

One source of truth (the root `doc-spec.md` registry), two generated readable views
(`docs/doc-general.md` for `section: common`, `docs/doc-custom.md` for `section:
custom`), one logic home (`docs/philosophy.md` `## Topic: Doc contract`). The views
are generated exactly like `README.md` is generated from `skills-catalog.json`; a
`validate.sh` check keeps them in lockstep. The split is in the views (by `section`),
never in the registry.

```
doc-spec.md (one registry, root, config)
   │  doc-spec.sh --render general   --render custom
   ▼
generate-doc-views.sh  ──►  docs/doc-general.md  +  docs/doc-custom.md  (generated views)
   ▲                                                        │
   └──────────── validate.sh Check 23 (regen-to-temp + diff; drift fails)
philosophy.md ## Topic: Doc contract  ◄── the "why" (logic) home
```

## Acceptance Criteria

<!-- Given/When/Then. #n matches the Requirements # column. -->

### Story #1: Render subcommand [core]

```
GIVEN the root doc-spec.md registry
WHEN  I run `scripts/doc-spec.sh --render general`
THEN  it emits a Markdown table (header + |---|) of exactly the 4 `section: common`
      rows (README.md, docs/philosophy.md, docs/workflow.md, docs/architecture.md)
      with Doc · Purpose · Requirement columns, quote-stripped and pipe-escaped,
      deterministic; and `--render custom` emits exactly the 9 `section: custom` rows
      (7 root operational docs + docs/doc-general.md + docs/doc-custom.md).
```

### Story #2: Generator [core]

```
GIVEN scripts/generate-doc-views.sh
WHEN  I run it with `--output-dir <tmp>` twice
THEN  it writes <tmp>/doc-general.md + <tmp>/doc-custom.md each with an
      AUTO-GENERATED header, and the two runs produce byte-identical output (idempotent).
```

### Story #3: Views on disk + declared, same commit [core]

```
GIVEN the generator and the registry
WHEN  the feature lands
THEN  docs/doc-general.md + docs/doc-custom.md exist (generated) AND two
      `section: custom`/`audit_class: human-doc` registry entries declare them, both
      changes in the SAME commit, so validate.sh Check 15a is green (no
      declared-but-missing, no orphan) and Check 19 (no work-item IDs) is green.
```

### Story #4: Slim Custom prose, preserve rationale [usability]

```
GIVEN the doc-spec.md Custom section
WHEN  the hand-written root-operational-docs table is replaced by a pointer
THEN  the "Repo notes" rationale (why root docs stay at root; singular workflow.md;
      no-separate-whitelist) + the `front_table` field explanation are preserved, the
      Common section is byte-identical to the seed (test #13 green), and
      `doc-spec.sh --validate` is clean.
```

### Story #5: Philosophy Doc-contract topic [usability]

```
GIVEN docs/philosophy.md
WHEN  the 2 doc-contract principles are moved into a new `## Topic: Doc contract`
THEN  the topic appears before `## Decision tree` with a registry→views lead-in, the
      front-summary-table topic labels for those rows are updated, and Check 19/20
      stay green (Decision tree remains last).
```

### Story #6: Drift gate [resilience]

```
GIVEN validate.sh Check 23 and the test.sh mirror
WHEN  the generated views differ from a fresh regeneration
THEN  Check 23 ERRORs ("doc views drifted — run scripts/generate-doc-views.sh"); when
      in sync it passes; when the generator is absent it skips cleanly; and test.sh
      generates into a temp dir twice and compares the temp outputs WITHOUT touching
      docs/.
```

### Story #7: README + script-reference consistency [integration]

```
GIVEN generate-readme.sh:23 and CLAUDE.md
WHEN  the feature lands
THEN  the generate-readme.sh:23 docs/ blurb includes the two new views, README.md is
      regenerated to match on disk, and the CLAUDE.md Scripts reference table has a
      row for generate-doc-views.sh.
```

## Architecture

<!-- High-level design. -->

```
                 +---------------------------+
                 |  doc-spec.md (root)        |  <-- single source of truth (config)
                 |  Common seed (untouched)   |
                 |  Custom section (slimmed)   |
                 +-------------+--------------+
                               |
              doc-spec.sh --render general|custom  (NEW: separate awk pass)
                               |
                 +-------------v--------------+
                 | generate-doc-views.sh      |  (NEW: --output-dir, header)
                 +------+--------------+-------+
                        |              |
              docs/doc-general.md   docs/doc-custom.md   (NEW: generated views)
                        ^              ^
                        |  validate.sh Check 23 (regen-to-temp + diff)  (NEW)
                        |  test.sh mirror (temp-only, never docs/)       (NEW)
                        |
   docs/philosophy.md ## Topic: Doc contract  (the "why"/logic home)     (MOVED)
   generate-readme.sh:23 blurb + README regen ; CLAUDE.md Scripts row    (UPDATED)
```

### Data Flow

1. `doc-spec.md` registry entries (Common + Custom) are the input.
2. `doc-spec.sh --render general|custom` walks the entries (separate awk pass) and emits a Markdown table per section, quote-stripping `key: "…"` wrappers and pipe-escaping cells.
3. `generate-doc-views.sh` wraps each render with an AUTO-GENERATED header and writes `docs/doc-general.md` + `docs/doc-custom.md` (default `--output-dir docs/`).
4. `validate.sh` Check 23 regenerates into a temp dir and diffs against `docs/`; any diff fails. `test.sh` does the temp-twice-compare without touching `docs/`.
5. README regen + the philosophy topic + the CLAUDE.md row keep the human-facing surfaces consistent.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Where the views live | Generated `docs/` views from one root registry | Hand-maintained `docs/` files (literal proposal) | No second list to keep in sync; preserves the "one file" contract |
| Registry stays at root | Keep `doc-spec.md` at root (config) | Relocate into `docs/` | ~13 hardcoded `./` call sites + every consumer repo would break |
| `--render` parsing | Separate awk pass (mirror `_list_front_table_docs`) | Extend `_parse_registry` | 3-col TSV mis-binds a 4th field onto `audit_class`; values are quoted free-form |
| Check 23 source | Generator-based (header-safe) regen-to-temp + diff | Bare `--render` diff | The generator owns the header on both sides, so the diff is header-safe |
| `test.sh` coverage | Stdout/temp-only (temp-twice-compare) | Generate into `docs/` then diff | The EXIT trap doesn't restore `docs/`; writing there corrupts the tree |
| Common seed | Untouched (out of scope) | Slim/edit the Common section too | Byte-identical to the seed (test #13); shared by consumer repos |
