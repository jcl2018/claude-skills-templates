## Problem Statement

A workbench contributor (human or agent) authoring a new human-readable doc has no machine-checked signal for WHERE it belongs. Explanation docs are supposed to live in `doc/` (F000034 moved PHILOSOPHY/ARCHITECTURE/SKILL-CATALOG there and registered them in a tracked-doc/ manifest enforced by Check 15), but nothing stops a new `FOO.md` from landing at the repo root instead of `doc/FOO.md`. The root surface is supposed to be limited to a small, justified set of pinned/operational docs (README, CLAUDE, CHANGELOG, CONTRIBUTING, TODOS) plus tooling-pinned configs — but that limit is implicit prose-knowledge at best, and entirely unenforced. The pain: the top-level surface silently drifts as the repo grows, and the F000032→F000037 doc-infra investment (which made the doc surface coherent) decays without manual vigilance. This story closes the gap by codifying the root-vs-`doc/` boundary in CLAUDE.md and enforcing it with a new `validate.sh` Check 17 — with zero file moves.

## Mental Model

Two symmetric manifests partition the top-level doc surface:

```
top-level *.md placement
        │
   ┌────┴─────────────────────────────┐
   │                                   │
 doc/*.md                          root *.md
   │                                   │
 F000034 tracked-doc/ manifest     F000038 Tracked root docs allowlist
 (CLAUDE.md, audit_class per entry) (CLAUDE.md, reason per entry)
   │                                   │
 enforced by validate.sh Check 15   enforced by validate.sh Check 17
```

A new explanation doc → `doc/` + a tracked-doc manifest entry (Check 15 catches an unregistered `doc/` file). A new root `*.md` → justified + added to the root allowlist with a `reason:` (Check 17 catches an un-allowlisted root file). Drift on either side fails validate.sh. Configs stay at root (tooling-pinned, ~246 refs to skills-catalog.json); the convention documents that but adds no config enforcement in v1.

## Acceptance Criteria

### Story #1: CLAUDE.md convention section + allowlist manifest [core]

```
GIVEN the workbench CLAUDE.md
WHEN  the F000038 change is applied
THEN  CLAUDE.md contains a new H2 section `## Doc placement convention (root vs doc/)`
      adjacent to the F000034 "/document-release workbench audit conventions" section
AND   the section states the prose rule (explanation docs → doc/ + tracked-doc manifest;
      root *.md → allowlist; configs stay at root; skills/templates/work-copilot/
      work-items/tests docs out of scope)
AND   a prose comment line just ABOVE the YAML block (NOT inside the fence) states the
      two load-bearing constraints (no `#`-leading lines in the block; heading text
      matched literally)
AND   a `### Tracked root docs allowlist` YAML block lists exactly 5 entries, each a
      `- path:` with a `reason:`: README.md, CLAUDE.md, CHANGELOG.md, CONTRIBUTING.md,
      TODOS.md
AND   the block contains NO `#`-leading comment lines
```

### Story #2: validate.sh Check 17 [core]

```
GIVEN scripts/validate.sh
WHEN  the F000038 change is applied
THEN  a new Check 17 is inserted AFTER Check 16, headed `=== Check 17: root-doc placement allowlist ===`
AND   it parses the allowlist via awk: arm on `/^### Tracked root docs allowlist$/`,
      disarm on `/^#/` (ANY heading), print `$3` of `flag && /^- path:/` lines
AND   it enumerates root *.md via `find . -maxdepth 1 -type f -name '*.md'`
AND   for each root *.md NOT in the allowlist it emits `  ERROR: root doc <f> is not in
      the CLAUDE.md 'Tracked root docs allowlist'; ...` and does `ERRORS=$((ERRORS+1))`
AND   for each allowlist entry whose file is missing it emits `  ERROR: <p> is in the
      CLAUDE.md root-docs allowlist but missing from disk` and does `ERRORS=$((ERRORS+1))`
AND   it uses the inline echo+ERRORS form (prefix `  ERROR:`), NOT the older fail()
      helper (prefix `  FAIL:`)
AND   it counts entries once (`N_ALLOW=$(echo "$ALLOWED_ROOT_MD" | grep -c . || true)`)
      and prints `  PASS: root *.md allowlist parsed (N entries)` when N_ALLOW > 0
```

### Story #3: clean PASS + synthesized violation [resilience]

```
GIVEN the F000038 change applied on the PR HEAD
WHEN  `./scripts/validate.sh` runs with no stray root docs
THEN  Check 17 prints `PASS: root *.md allowlist parsed (5 entries)` and validate.sh
      exits 0 with 0 errors / 0 warnings
WHEN  `touch STRAY.md` is run at the repo root and `./scripts/validate.sh` runs again
THEN  validate.sh exits non-zero AND its output contains
      `  ERROR: root doc STRAY.md is not in the CLAUDE.md`
WHEN  `rm STRAY.md` is run and `./scripts/validate.sh` runs again
THEN  validate.sh exits 0
```

### Story #4: test.sh zzz-test-scaffold integration assertion [integration]

```
GIVEN scripts/test.sh's zzz-test-scaffold integration test
WHEN  the F000038 change is applied
THEN  the integration test, in its scaffolded fixture repo, does `touch STRAY.md` at
      root and asserts validate.sh exits non-zero AND output contains the literal
      `  ERROR: root doc STRAY.md is not in the CLAUDE.md`
AND   then does `rm STRAY.md` and asserts validate.sh exits 0
AND   `./scripts/test.sh` exits 0 on the PR HEAD (the new assertion runs + passes)
```

### Story #5: ancillary artifacts + no collateral churn [core]

```
GIVEN the F000038 change
WHEN  the PR is prepared
THEN  VERSION is bumped from 6.0.3 to the next free slot (likely 6.0.4; confirmed via
      ./scripts/check-version-queue.sh)
AND   CHANGELOG.md has a new `### Added` entry in user-forward voice naming F000038 +

AND   no SKILL.md / USAGE.md / skills-catalog.json / manifest-JSON file is modified
AND   README.md, CLAUDE.md, and all 4 root config files remain at root byte-for-byte
      unchanged (zero file moves)
```

## Architecture

<!-- High-level system design for this story. Which components are affected?
     How do they interact? Include an ASCII diagram for any non-trivial data flow. -->

```
CLAUDE.md
  ## Doc placement convention (root vs doc/)
    prose rule
    <constraint comment: no # lines in block; heading literal>     ← prose, above fence
    ### Tracked root docs allowlist
    - path: README.md         reason: ...
    - path: CLAUDE.md         reason: ...
    - path: CHANGELOG.md      reason: ...
    - path: CONTRIBUTING.md   reason: ...
    - path: TODOS.md          reason: ...
        │
        │  (parsed by)
        ▼
scripts/validate.sh  Check 17
    ALLOWED_ROOT_MD = awk '/^### Tracked root docs allowlist$/{flag=1;next}
                           /^#/{flag=0}
                           flag && /^- path:/{print $3}' CLAUDE.md
    ROOT_MD_ON_DISK = find . -maxdepth 1 -type f -name '*.md' | sed 's#^\./##' | sort
        │
        ├─ 17-orphan:  for f in ROOT_MD_ON_DISK: f ∉ ALLOWED_ROOT_MD → ERROR + ERRORS++
        ├─ 17-missing: for p in ALLOWED_ROOT_MD: [ ! -f p ]          → ERROR + ERRORS++
        └─ N_ALLOW = grep -c . ; [ N_ALLOW -gt 0 ] && PASS line
        │
        ▼
    Summary block (unchanged): ERRORS>0 → exit 1

scripts/test.sh  zzz-test-scaffold integration
    (in scaffolded fixture repo)
      touch STRAY.md → run validate.sh → assert non-zero + grep "  ERROR: root doc STRAY.md is not in the CLAUDE.md"
      rm STRAY.md    → run validate.sh → assert exit 0
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. A contributor (or /document-release at Step 2) reads CLAUDE.md `## Doc placement convention (root vs doc/)` to learn where a new doc belongs.
2. At commit time, the pre-commit hook runs `./scripts/validate.sh`.
3. Check 17 reads the `### Tracked root docs allowlist` block from CLAUDE.md via awk → `ALLOWED_ROOT_MD`.
4. Check 17 enumerates root `*.md` via `find . -maxdepth 1` → `ROOT_MD_ON_DISK`.
5. 17-orphan: any on-disk root `*.md` not in the allowlist → `  ERROR:` + `ERRORS++`.
6. 17-missing: any allowlist entry whose file is absent → `  ERROR:` + `ERRORS++`.
7. Check 17 prints `PASS: root *.md allowlist parsed (N entries)` when N_ALLOW > 0.
8. The Summary block exits 1 if `ERRORS > 0` — blocking the commit on drift.
9. Separately, `scripts/test.sh`'s zzz-test-scaffold integration test synthesizes a STRAY.md in a fixture repo and asserts the orphan ERROR fires + clears on removal, locking the behavior in CI.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Where the allowlist lives | YAML block in CLAUDE.md, parsed by Check 17 (Approach B) | Hardcoded bash array in validate.sh + separate CLAUDE.md prose (Approach A) | A splits the allowlist from its rationale into two drifting places, no co-located `reason:`, not symmetric with F000034. B is single-source, self-documenting, symmetric, reuses Check 15's parse shape. |
| Where the allowlist lives | YAML block in CLAUDE.md (Approach B) | New JSON config file (Approach C) | C over-engineers 5 filenames and adds another root config surface — ironic for a feature about keeping the root tidy. (Mirrors F000037's own "JSON Schema file is over-engineering for v1" call, applied in the opposite direction.) |
| awk disarm trigger | Disarm on ANY heading (`/^#/`) | Disarm only on `^###` (Check 15's form) | The allowlist is the last `###` subsection under its `##` section; `^###`-only would over-capture `- path:` lines from a following `##` section. Disarm-on-any-heading is strictly more robust. |
| ERROR-emit form | Inline `echo "  ERROR:"; ERRORS=$((ERRORS+1))` (Checks 15/16 form) | Older `fail()` helper (`  FAIL:` prefix) | Checks 12–16 abandoned `fail()`; matching the inline form keeps Check 17 consistent with its neighbors and lets the test.sh assertion grep for the right literal. |
| Enforcement severity | ERROR-strict (exit 1) | Warning-only | Matches the repo ethos (F000037 strict-required; Checks 12–16 ERROR-strict). Safe day-one because all 5 current root docs are allowlisted. |
| Config enforcement | Document the rule in prose; no config-file check in v1 | A sibling "tracked root configs" manifest + check | Configs are tooling-pinned (~246 refs) + stable; enforcement adds churn for no gain now. Deferred to v2. |
| Scope | Codify + enforce, zero file moves | Move configs/docs into a single folder (original ask) | The explanation docs already moved (F000034); configs are tooling-pinned. Moving them is high-blast-radius churn for no functional gain. The real gap was the unenforced boundary. |
