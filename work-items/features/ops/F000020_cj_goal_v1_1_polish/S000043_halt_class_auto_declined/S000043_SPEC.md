## Problem Statement

Today, when `/CJ_goal` encounters a sensitive-surface TODO body, `goal.sh:296` halts with `halted_at_sensitive_surface_user_declined`. Under bash there is no AUQ tool ‚Äî the script auto-defaults regardless of whether a human is present. The end_state name "user_declined" is therefore a misnomer: nobody declined; the gate fired by design because no AUQ tool was reachable. The halt-class lookup table maps `_user_declined` to STOP (correct interpretation given the name), so `/loop /CJ_goal` halts here on a routine "needs human" gate that under unattended-loop semantics should defer (skip-list) and continue.

The user (operator running /loop /CJ_goal) wants to grind through the backlog without halting on auto-default gates.

## Mental Model

Under bash today: only `_auto_declined` ever fires (continue). Under future interactive AUQ at orchestrator layer: `_user_declined` fires when human explicitly rejects (STOP).

## Acceptance Criteria

### Story #1: goal.sh emits _auto_declined [core]

```
GIVEN  /CJ_goal hits the sensitive-surface gate at goal.sh:296 under bash (no AUQ tool)
WHEN   the auto-default fires
THEN   the emitted end_state is `halted_at_sensitive_surface_auto_declined`
AND    is NOT `halted_at_sensitive_surface_user_declined`
```

### Story #2: halt-class table updated [core]

```
GIVEN  goal.sh's halt-class lookup table (and SKILL.md's documentation table)
WHEN   I look up `halted_at_sensitive_surface_auto_declined`
THEN   it appears in the continue column (not STOP)
AND    parallels `halted_at_preflight` (same continue semantics)
```

### Story #3: /loop /CJ_goal continues past the gate [core]

```
GIVEN  /loop /CJ_goal is running and the next candidate row matches sensitive-surface regex
       (bypassing WI-A's pre-filter, e.g. via interactive `/CJ_goal "fragment"` from inside /loop)
WHEN   the gate fires and emits `_auto_declined`
THEN   /loop reads the halt-class as continue
AND    iterates to the next loop iteration (skip-list updated)
AND    does NOT print "loop stopped" or terminate
```

### Story #4: _user_declined remains reserved [resilience]

```
GIVEN  current bash code (post-WI-B)
WHEN   I grep for `_user_declined` references
THEN   the only occurrences are in commented-out code, SKILL.md taxonomy table (marked "future"), or the halt-class lookup table (marked STOP)
AND    no active emit-site emits `_user_declined`
```

### Data Flow

1. /CJ_goal evaluates a candidate TODO body; sensitive-surface regex matches.
2. goal.sh:296 reaches auto-default branch (bash has no AUQ tool).
3. `end_state="halted_at_sensitive_surface_auto_declined"` is set.
4. skip-list updated (defer gate; row surfaces on next interactive run).
5. journal entry written.
6. exit with end_state.
7. /loop reads end_state, looks up halt-class ‚Üí "continue", iterates.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| End state naming | `_auto_declined` (disposition discriminator) | `_loop_declined` (caller discriminator) | Disposition is the right axis: same logic applies in cron/daemon/etc., not just /loop. No caller env contract introduced. |
| Introduce both end states upfront | Just rename `_user_declined ‚Üí _auto_declined`; keep `_user_declined` reserved | Add both `_auto_declined` AND emit `_user_declined` from a new interactive path | YAGNI ‚Äî interactive AUQ at orchestrator layer is a separate future change with no consumer today. Defer until needed. |
| Halt-class for `_auto_declined` | continue (mirror `_preflight`) | STOP (be conservative) | Auto-default + skip-list IS the defer mechanic; same semantic as preflight rejection. STOP would defeat WI-B's purpose. |
| Keep WI-A's pre-filter alongside WI-B's rename | Both ship together (defense-in-depth) | Drop WI-B if WI-A's pre-filter holds | Bypass paths (interactive fragment from inside /loop, regex update drift) keep WI-B load-bearing. |
