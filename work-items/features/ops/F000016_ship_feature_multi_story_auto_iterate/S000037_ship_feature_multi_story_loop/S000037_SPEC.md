## Problem Statement

`run.md` Branch (b) halts with unexecutable manual instructions after
multi-story scaffold. The operator — anyone running `/CJ_ship-feature` on a
multi-story feature — must manually drive impl+qa+ship+land for each child story,
negating the automation value. This story replaces that halt with an auto-iterate
loop that ships each child story independently.

## Mental Model

Three sequential phases per child story:
1. **Git setup** — fetch origin main, create child branch, copy scaffold files from feature branch, commit.
2. **Pipeline** — dispatch `CJ_personal-pipeline --work-item-dir --suppress-final-gate`; wait for green.
3. **Ship** — invoke /ship (AUQ fires here: diff-review gate for this child), then /land-and-deploy.

After each child: restore to feature branch. On failure: halt, report, restore. Resume guard
at loop start prevents re-shipping already-merged children on re-run.

## Acceptance Criteria

### Story #1: Auto-iterate instead of halt [core]

```
GIVEN /CJ_ship-feature is invoked on a multi-story feature design doc
WHEN Branch (b) detects N ≥ 1 child user-story dirs (S[0-9]*) at scaffold complete
THEN the loop begins without printing "Per-child invocation needed" halt message
AND for each child: branch is created, scaffold copied, pipeline dispatched, /ship invoked, /land-and-deploy invoked
```

### Story #2: Isolated git state per child [core]

```
GIVEN the loop is processing child CHILD_NAME
WHEN git-ops runs
THEN a new branch ${FEATURE_NAME}--${CHILD_NAME}-<timestamp> is created off origin/main
AND CHILD_REL (relative path of child dir) files are copied from FEATURE_BRANCH
AND a commit is made with message "scaffold: ${CHILD_NAME} (from ${FEATURE_NAME} feature scaffold)"
```

### Story #3: Resume guard [core]

```
GIVEN a re-run after child 1 already merged and child 2 failed
WHEN the loop processes child 1
THEN it detects the already-merged branch (gh pr list --state merged)
AND prints "Child S000001: already merged. Skipping."
AND increments CHILDREN_DONE without dispatching pipeline
```

### Story #4: Failure halt with clean repo state [core]

```
GIVEN pipeline returns end_state != green for child CHILD_NAME
WHEN the failure branch executes
THEN END_STATE is set to "halted_at_pipeline"
AND CHILDREN_FAILED is updated with CHILD_NAME
AND git checkout returns to FEATURE_BRANCH
AND write_state() is called
AND the loop breaks with a clear error message listing CHILD_NAME and remaining children
```

### Story #5: State persistence [integration]

```
GIVEN a multi-story run completes (all children green)
WHEN write_state() is called after the loop
THEN the state file contains: CHILDREN_TOTAL, CHILDREN_DONE, CHILDREN_FAILED, CHILD_PR_URLS
AND END_STATE is "green"
```

### Story #6: Telemetry correctness [observability]

```
GIVEN a multi-story run (any outcome)
WHEN Step 6.1 telemetry is written
THEN the JSONL record contains multi_story_mode: true (not multi_story_scaffold_only)
AND contains multi_story_children_shipped: <count of successful children>
```

### Story #7: Multi-story summary [observability]

```
GIVEN all children shipped successfully
WHEN Step 6.2 green summary is printed
THEN the output includes "Children shipped: N/N" and a list of PR URLs
```

## Architecture

```
ship-feature.md Branch (b) entry point
         |
  MULTI_STORY=1 detected
         |
  preamble: source state, detect FEATURE_BRANCH, MAIN_BRANCH, CHILDREN
         |
  N > 3? --> AUQ: "N children — inline Skills may overflow context. Proceed?" 
         |   (v1 guard; v2 auto-dispatch as subagents)
         |
  +----- loop over CHILDREN -----+
  |                               |
  | resume guard:                 |
  |  gh pr list --merged          |
  |  match ${FEATURE_NAME}--${CHILD_NAME}* |
  |  if match: skip (CHILDREN_DONE++) |
  |                               |
  | git fetch origin MAIN_BRANCH  |
  | git checkout -b CHILD_BRANCH  |
  |   origin/MAIN_BRANCH          |
  | git checkout FEATURE_BRANCH   |
  |   -- CHILD_REL                |
  | git commit scaffold message   |
  |                               |
  | pipeline subagent dispatch    |
  |  --work-item-dir CHILD_DIR    |
  |  --suppress-final-gate        |
  |                               |
  | parse PIPELINE_END_STATE      |
  |                               |
  | if green:                     |
  |   /ship (AUQ gate fires here) |
  |   /land-and-deploy            |
  |   CHILDREN_DONE++             |
  |   git checkout FEATURE_BRANCH |
  | else:                         |
  |   CHILDREN_FAILED+=CHILD_NAME |
  |   git checkout FEATURE_BRANCH |
  |   write_state                 |
  |   break                       |
  +----- end loop ----------------+
         |
  all green? --> write_state(END_STATE=green) --> Step 6
  partial?   --> END_STATE=halted_at_pipeline --> Step 6
```

### Data Flow

1. Branch (b) detects CHILDREN via `find $WORK_ITEM_DIR -maxdepth 1 -mindepth 1 -type d -name 'S[0-9]*' | sort`
2. Loop: for each CHILD_DIR in CHILDREN
3. Resume guard: `gh pr list --state merged` checks for prior-run merges
4. Git setup: fetch, checkout new branch from origin/main, sparse-copy scaffold from feature branch, commit
5. Pipeline dispatch: Agent subagent with `--work-item-dir CHILD_DIR --suppress-final-gate`
6. On green: Skill /ship (AUQ fires — diff review gate), then Skill /land-and-deploy
7. On failure: break loop, restore to feature branch, write state
8. After loop: write state, call Step 6 (telemetry + summary)

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Sequential child loop | Sequential | Parallel branches | Parallel creates race conditions on main. Sequential is safe and ensures each child's diff is clean vs post-merge main. |
| Resume guard via gh pr list | Merge check on branch name prefix | Re-dispatch all children | Re-dispatching already-merged children would create duplicate PRs and duplicate version bumps. Resume guard is idempotent. |
| v1: inline Skills for /ship + /land-and-deploy | Inline (Skill tool) | Agent subagent dispatch | Inline is simpler for N ≤ 3. Subagent adds ~200 token overhead and async complexity. v2 upgrade path is documented. |
| Child branch timestamp suffix | ${FEATURE_NAME}--${CHILD_NAME}-YYYYMMDD-HHMMSS | Sequential counter | Timestamp prevents collisions between re-runs. Sequential counter requires global state to be reliable. |
