## Problem Statement

`CJ_personal-pipeline` accepts only a design-doc path today. Child user-stories
produced by feature scaffolding have no design doc — only TRACKER.md, SPEC.md, and
TEST-SPEC.md in the child dir. The pipeline must be extended to accept a pre-scaffolded
work item dir directly, so `run.md` can dispatch per-child pipeline runs
without a design doc. The operator running the pipeline today cannot use it on
pre-scaffolded dirs at all.

## Mental Model

The pipeline has two input modes after this story:
- **Design-doc mode (existing):** `--design-doc <path>` (implicit). Full scaffold → impl → qa.
- **Work-item-dir mode (new):** `--work-item-dir <dir>`. Skips scaffold; runs impl → qa on
  an already-scaffolded dir. No design doc required.

Both modes converge at Step 4 (post-scaffold gate). All downstream steps (4→8) are
identical. The mode difference is visible only in Step 1 (arg parsing) and Step 2
(idempotency branch — Branch (e) bypasses footer search and Phase 1).

## Acceptance Criteria

### Story #1: Flag parsing [core]

```
GIVEN CJ_personal-pipeline is invoked with --work-item-dir <path> and --suppress-final-gate
WHEN Step 1 arg parsing runs
THEN WORK_ITEM_DIR_OVERRIDE is set to <path>, SUPPRESS_FINAL_GATE is set, and DESIGN_DOC is ""
AND no "design doc not found" or "design doc must be under ~/.gstack/projects/" error fires
```

### Story #2: DESIGN_DOC validation block gated [core]

```
GIVEN WORK_ITEM_DIR_OVERRIDE is set
WHEN Step 1 validation runs
THEN the DESIGN_DOC validation block ([ -n "$DESIGN_DOC" ] || ...) is skipped
AND the --work-item-dir validation block runs: dir exists check + TRACKER.md exists check
```

### Story #3: Branch (e) routing [core]

```
GIVEN WORK_ITEM_DIR_OVERRIDE is set and a valid work-item dir is provided
WHEN Step 2 runs
THEN Branch (e) fires: footer search is skipped, Phase 1 (scaffold subagent) is skipped
AND a journal entry "[orchestrator] --work-item-dir mode: using pre-staged dir at <path>; scaffold skipped." is written to TRACKER.md
AND pipeline continues directly to Step 4
```

### Story #4: Step 4 sub-step 1 carve-out [core]

```
GIVEN --work-item-dir mode is active (DESIGN_DOC is "")
WHEN Step 4 runs
THEN sub-step 1 (footer write-back confirm) is skipped entirely
AND sub-steps 2–4 (boundary check, multi-story guard, shape confirm) run normally
```

### Story #5: Telemetry mode field [core]

```
GIVEN --work-item-dir mode is active
WHEN Step 9.1 telemetry is written
THEN the JSONL record contains "work_item_dir_mode": true
```

### Story #6: Step 9.3 summary with empty DESIGN_DOC [observability]

```
GIVEN --work-item-dir mode is active (DESIGN_DOC is "")
WHEN Step 9.3 summary is printed
THEN the "Design:" line reads "Design: (work-item-dir mode — no design doc)"
AND no blank line appears in the summary output
```

## Architecture

```
CJ_personal-pipeline invocation (Step 1)
        |
        v
  arg parser loop
        |
   --work-item-dir <path>?
        |
       YES                        NO (design-doc mode, existing)
        |                              |
  set WORK_ITEM_DIR_OVERRIDE          set ARGS+=(<path>)
  DESIGN_DOC=""                       DESIGN_DOC="${ARGS[0]}"
        |                              |
  validate: dir exists?          validate: file exists?
  TRACKER.md exists?             under ~/.gstack/projects/?
        |                              |
        +----------Step 2 merge--------+
                        |
                        v
               Branch (e)?  (WORK_ITEM_DIR_OVERRIDE set)
                        |
                       YES  --> skip footer search, skip Phase 1
                               journal entry
                               --> Step 4 (sub-step 1 skipped)
                        |
                        NO  --> normal Branch (a/b/c/d) routing
```

### Data Flow

1. Operator invokes `CJ_personal-pipeline --work-item-dir <path> [--suppress-final-gate]`
2. Step 1 parser: sets `WORK_ITEM_DIR_OVERRIDE=<path>`, `SUPPRESS_FINAL_GATE=1` (if set), `DESIGN_DOC=""`
3. Step 1 validation: verifies `<path>` exists and contains TRACKER.md; sets `WORK_ITEM_DIR=realpath(<path>)`
4. Step 2: Branch (e) fires; footer search skipped; Phase 1 (scaffold subagent) skipped; journal entry written
5. Step 4: sub-step 1 skipped; sub-steps 2–4 run (boundary check, multi-story guard, shape confirm)
6. Steps 5–8 run normally (impl, validate, qa, post-qa gate)
7. Step 9.1: telemetry written with `work_item_dir_mode: true`
8. Step 9.3: summary printed with `Design: (work-item-dir mode — no design doc)`

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Skip only Step 4 sub-step 1 | Carve-out in prose state | Skip entire Step 4 | Sub-steps 2–4 (boundary check, multi-story guard, shape confirm) are valid and needed in --work-item-dir mode. Only sub-step 1 reads DESIGN_DOC. |
| `~/.gstack/projects/` path guard in `else` branch | Conditional on `WORK_ITEM_DIR_OVERRIDE` absent | Remove guard entirely | Guard is still needed for design-doc mode; removing it weakens validation. Moving it to the `else` branch is correct scoping. |
