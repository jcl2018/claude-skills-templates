## Problem Statement

The workbench maintainer (and any consumer repo adopting the doc contract)
reads a two-tier model where the general tier claims only 4 docs, while 6 more
docs every adopting repo needs are mislabeled `custom`, the portable seed
still frames the contract as "four human docs," and no rule anywhere states
that general docs are required. The contract argues with itself, and a fresh
bootstrap under-declares what a repo actually gets.

## Mental Model

One portable contract (the Common seed, byte-identical in three places), one
per-repo registry, two generated views. After this story the general tier is
10 docs and carries one explicit rule — "General docs are required" — enforced
by machinery that already exists (Check 15/17 declared⇒on-disk, the
`/CJ_document-release` stub-scaffold, the seed declaring all 10 on bootstrap)
plus one new ADVISORY audit line; nothing new halts.

```
Common seed (3 byte-identical copies)          workbench registry (spec/doc-spec.md)
  spec/doc-spec.md Common section                10 section: common  (6 flipped)
  templates/doc-spec-common.md          ◄──►      3 section: custom
  scripts/doc-spec.sh _emit_seed                       │ doc-spec.sh --render
       │ 10 common entries, portable strings           ▼
       │ + "General docs are required" rule      docs/doc-general.md (10 rows)
       ▼                                         docs/doc-custom.md  (3 rows)
/CJ_document-release: states tier logic; Step 6.7 advisory
  "registry missing general-contract doc(s)" on the contract file's verdict
```

### Data Flow

<!-- Primary use case: a doc-release run after the change. -->

1. `/CJ_document-release` resolves the registry (`spec/doc-spec.md`, spec/-then-root) and reads 10 `section: common` + 3 `section: custom` entries.
2. Step 6.7 writes `doc-spec.sh --seed` to a temp file and runs `DOC_SPEC_PATH=<temp> doc-spec.sh --render general`, taking column 1 as the general-contract set.
3. Each general-contract path is matched against the repo registry's declared paths (basename equivalence for `doc-spec.md`); any miss becomes `stale: registry missing general-contract doc(s): <paths>` on the contract file's verdict line (advisory).
4. Stub-scaffold creates any declared-but-missing doc; for the two views it renders real content via `doc-spec.sh --render general|custom` with a portable header (plain stub on render failure); TODOS.md converges with the lazy-create path.
5. The verdict block surfaces to the RESULT + scratch file as today; the positive all-current line appears only when every verdict is `up-to-date`.

## Tradeoffs

<!-- Design decisions with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| How "required" is enforced | Existing machinery (Check 15/17 + stub-scaffold + seed) + one advisory audit line | Approach C: `--list-general-docs` + hard validate.sh check | New check ⇒ test.sh fixture churn; hard-gate posture rejected at the premise gate; deferable as a TODOS row |
| Scope of the restatement | Approach B: full Common-prose rewrite + skill statement + philosophy | Approach A: seed-minimal entry flip | A leaves the "four human docs" framing beside the new rule — the seed argues with itself |
| General-set enumeration in the audit | `--seed` → temp file → `DOC_SPEC_PATH` → `--render general` | Hand-parse the seed yaml, or `--list-declared` | Reuses the parser; render-general filters by `section: common`, so a future custom seed entry cannot over-enumerate |
| Contract-file path matching | Basename equivalence (`spec/doc-spec.md` satisfies `doc-spec.md`) | Exact-path match | Mirrors doc-spec.sh's spec/-then-root resolution; exact match would false-positive the workbench every run |
| View stub shape | Render-first with portable header, plain-stub fallback | Workbench-header stub naming `spec/doc-spec.md` + `scripts/generate-doc-views.sh` | Born satisfying "kept matching the registry"; workbench paths don't exist in a root-style consumer repo |
| Seed requirement strings | Portable (no workbench script references); workbench registry keeps specialized strings | One shared string set | Premise 4; established `front_table` specialization precedent |
