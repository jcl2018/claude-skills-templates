## Problem Statement

Concurrent same-machine cj_goal worktrees collide on the next work-item ID when
both reach `/CJ_scaffold-work-item` Step 5.1 before either has pushed. The
existing 3-source check (local / open-PRs / origin) cannot see a not-yet-pushed
sibling, so two worktrees both mint the same ID; the clash is found only at
merge/`/ship` and recovered manually. The user is the cj_goal operator running
several Conductor/Claude sessions in parallel worktrees of one clone; the pain
is post-merge renumbering (~4 runs hit per a recent retro).

## Mental Model

Add a 4th ID source layered on top of the existing three: an **atomic claim
directory** in the shared `.git` common-dir. The claim is made at scaffold time,
before any commit/push/PR, and is instantly visible to every sibling worktree
(they share one `.git`). `mkdir` is the compare-and-swap — it fails if the dir
already exists. The helper has three phases: (1) lazy reap, (2) idempotent
same-branch reuse, (3) a bounded atomic claim loop.

```
Sources 1-3 (point-in-time, caller-computed)        Source 4 (continuous, atomic)
  local tree ─┐                                        mkdir .git/cj-id-claims/<ID>
  open PRs  ──┼─► HIGHEST (--floor) ──► cj-id-claim.sh ─────────► CLAIMED_ID
  origin/main ┘                          (claims strictly above max(floor, live claims))
```

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group. -->

### Story #1: Single claim above floor [core]

```
GIVEN no live claims for prefix F and --floor 47
WHEN  cj-id-claim.sh --prefix F --floor 47 runs
THEN  stdout contains CLAIMED_ID=F000048 and exit code is 0
```

### Story #2: Concurrent race returns distinct IDs [resilience]

```
GIVEN two cj-id-claim.sh invocations launched in parallel with the same --prefix and --floor
WHEN  both run to completion (repeated for 20+ rounds)
THEN  the two CLAIMED_ID values are distinct every round, with no duplicate
```

### Story #3: Reap on origin [resilience]

```
GIVEN a claim dir for an ID whose ${PREFIX}NNNNNN_*_TRACKER.md already exists on origin/main
WHEN  the helper runs its reap phase
THEN  that claim dir is removed and the ID is not counted toward the next claim
```

### Story #4: Reap on TTL [resilience]

```
GIVEN a claim dir whose mtime is older than --ttl-hours
WHEN  the helper runs its reap phase
THEN  that claim dir is removed and the ID is not counted toward the next claim
```

### Story #5: Same-branch idempotent reuse [usability]

```
GIVEN a live claim whose meta.branch == the current branch AND no work-item dir exists for that ID
WHEN  cj-id-claim.sh runs again on the same branch
THEN  it reuses the existing ID (refreshing mtime), prints that CLAIMED_ID, and exits 0 — the ID does not advance
```

### Story #6: Helper-absent fallback [integration]

```
GIVEN scripts/cj-id-claim.sh is absent or not executable
WHEN  scaffold.md Step 5.1 runs
THEN  NEW_ID is minted by the 3-source printf fallback and scaffold proceeds normally
```

### Story #7: Prefix isolation + absolute common-dir [resilience]

```
GIVEN an existing F-claim and a request for an S-claim (and a request issued from a linked worktree)
WHEN  cj-id-claim.sh --prefix S runs
THEN  the S-ID is independent of the F-claim AND the claim is created under the absolute shared .git common-dir (visible from the root checkout)
```

## Architecture

<!-- High-level system design for this story. -->

```
scaffold.md Step 5.1
   │  computes HIGHEST = max(LOCAL_MAX, PR_MAX, ORIGIN_MAX)   [Sources 1-3]
   │
   ├─ _CLAIM="$(git rev-parse --show-toplevel)/scripts/cj-id-claim.sh"   # worktree's own script
   ├─ if [ -x "$_CLAIM" ]: NEW_ID=$("$_CLAIM" --prefix "$PREFIX" --floor "$HIGHEST" | sed -n 's/^CLAIMED_ID=//p')
   └─ [ -n "$NEW_ID" ] || NEW_ID=$(printf "${PREFIX}%06d" $((10#$HIGHEST + 1)))   # fail-soft fallback
                       │
                       ▼
        scripts/cj-id-claim.sh
          CLAIM_ROOT="$(cd "$(git rev-parse --git-common-dir)" && pwd)/cj-id-claims"   # SHARED .git, absolute
          1. reap (on-origin OR mtime > ttl)         [REAP INVARIANT: never deletes a live winner]
          2. idempotent same-branch reuse (live claim, branch match, no dir yet)
          3. atomic loop: new_id = max(floor, live-claim-max)+1; mkdir CLAIM_ROOT/$new_id || retry
             on success: write meta {branch,pid,iso-ts}; echo CLAIMED_ID=$new_id; exit 0
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. Scaffold Step 5.1 computes `HIGHEST` from Sources 1-3 (local tree, open PRs, origin/main).
2. Step 5.1 invokes `cj-id-claim.sh --prefix $PREFIX --floor $HIGHEST`.
3. The helper resolves the absolute `CLAIM_ROOT` under the shared `.git` common-dir and `mkdir -p`s it.
4. Reap phase removes claim dirs that are on origin/main or older than the TTL.
5. Idempotent-reuse phase reuses a live same-branch claim (if no work-item dir exists yet) and returns its ID.
6. Atomic loop computes `new_id = max(floor, live-claim-max)+1` and `mkdir`s the claim dir; on EEXIST it re-reads and retries.
7. On a winning `mkdir`, the helper writes `meta` and prints `CLAIMED_ID=<ID>`.
8. Step 5.1 captures `CLAIMED_ID`; if empty (helper absent/failed), it falls back to the 3-source `printf`.

## Tradeoffs

<!-- Design decisions made in this spec, with rejected alternatives and rationale. -->

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Atomic primitive | `mkdir` CAS | `git update-ref` zero-oid CAS | mkdir is simplest, mtime gives trivial TTL reaping, portable; update-ref needs an object + ref cleanup + ref-date plumbing for zero v1 benefit. |
| Location | helper script (`scripts/cj-id-claim.sh`) | inline ~12 lines in scaffold.md | helper is unit-testable in isolation (race/reap/prefix); inline violates engine-in-script and is untestable. |
| Claim-root path | absolute (`cd "$(...)" && pwd`) | raw `git rev-parse --git-common-dir` | a relative root gives agents with different cwd different roots, silently breaking the CAS. |
| Reaping | lazy (on-origin OR mtime > TTL) | aggressive PID-liveness reaping | ID gaps are harmless, so lazy reaping suffices and avoids platform-specific liveness checks. |
| Scope | same-machine / shared `.git` only | cross-machine ref replication | matches the actual failure (worktrees); cross-machine stays covered post-push by Sources 2+3. |
