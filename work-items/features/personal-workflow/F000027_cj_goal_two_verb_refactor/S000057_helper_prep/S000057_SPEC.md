## Problem Statement

The author needs shared, testable plumbing for the two new verb skills, and the existing `cj-worktree-init.sh` guard rejects the `feature`/`defect` callers those skills require. Without a deterministic common helper, worktree/telemetry/PR-check logic gets duplicated as fragile per-skill prose; without an early smoke harness, the feature path stays unvalidated until PR #2.

## Mental Model

Three independent, deterministic surfaces shipped together as the foundation layer: a validator extension (callerвЖТprefix), a new common helper (phase/mode-gated bash), and a standalone smoke harness (feature-path shape). All three are exercised by `test.sh`; none depend on the verb skills existing.

## Acceptance Criteria

<!-- Given/When/Then format. One block per requirement or logical group. -->

### Story #1: New callers accepted [core]

```
GIVEN cj-worktree-init.sh with the extended --caller validator
WHEN  invoked with --caller feature (then --caller defect)
THEN  the prefix resolves to cj-feat (then cj-def) and the script exits 0
```

### Story #2: Deterministic common helper [integration]

```
GIVEN scripts/cj-goal-common.sh
WHEN  invoked with --phase ship --mode feature (and other phase/mode combos)
THEN  it performs the deterministic worktree/telemetry/PR-check operation for that phase+mode and exits 0
```

### Story #3: Early feature smoke harness [resilience]

```
GIVEN the feature smoke harness, with the /cj_goal_feature skill NOT yet present
WHEN  the harness runs
THEN  it validates the feature path shape (worktree + leaf-subagent dispatch) and passes
```

### Story #4: Non-regressive for existing callers [resilience]

```
GIVEN the extended validator
WHEN  invoked with --caller cj-run (and cj-todo, cj-inv)
THEN  each resolves to its existing prefix unchanged and test.sh stays green
```

### Data Flow

<!-- How does data move through the system for the primary use case? -->

1. A verb skill calls `cj-worktree-init.sh --caller feature` вЖТ validator `case` matches вЖТ prefix `cj-feat` вЖТ worktree created.
2. The skill calls `cj-goal-common.sh --phase <p> --mode feature` for each deterministic step (telemetry write, PR check).
3. CI runs `tests/cj-goal-feature-smoke.test.sh` + `tests/cj-worktree-init.test.sh` to validate the path and guard against regression.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Where common logic lives | Deterministic `cj-goal-common.sh` bash with mode flags | LLM-followed shared `tail.md` | Bash is testable and drift-free; the LLM doc becomes a mode-flag orchestrator (the coupling the refactor removes). |
| When to validate the feature path | Early smoke harness right after the `--caller` change | Wait for the `/cj_goal_feature` skill (S000059) | Defect-first sequencing leaves the feature tail unexercised until PR #2; validate early instead. |
