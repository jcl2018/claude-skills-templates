## Problem Statement

`/implement-from-spec` and `/qa-work-item` reject any work-item whose `type:` frontmatter field is not `user-story`. A user who scaffolds a defect or task via `/scaffold-work-item` (which accepts all types) cannot then run the rest of the F000010 pipeline — they have to drop back to manual implementation. This is direct friction documented in TODOS.md #6 and surfaced as a meta-finding during F000011 dogfood on 2026-05-08.

The asymmetry breaks the pipeline's contract. The fix: both skills detect type from frontmatter and read type-appropriate input artifacts per `personal-artifact-manifests.json`.

## Mental Model

Today: `/implement-from-spec` and `/qa-work-item` are user-story-only. The "type guard" appears at start, errors out, exits.

After: type guard becomes a type-dispatch table. The skill detects `type:` from frontmatter, looks up which input artifacts to read for that type, and the rest of the skill (boundary check, sensitive-surface AUQ, idempotency, journal entries, post-write check) is type-agnostic — those layers already use the manifest.

```
                    ┌─────────────────┐
                    │ Read TRACKER.md │
                    │ frontmatter     │
                    └────────┬────────┘
                             │ type: <T>
                             ▼
                    ┌─────────────────────────┐
                    │ Per-type input lookup:  │
                    │                         │
                    │ user-story → SPEC+DESIGN│
                    │ defect → RCA+test-plan  │
                    │ task → task-spec+test-pl│
                    │ feature → pick child AUQ│
                    └────────┬────────────────┘
                             ▼
                    ┌─────────────────┐
                    │ Existing flow:  │
                    │ - boundary check│
                    │ - sensitive AUQ │
                    │ - implement/qa  │
                    │ - post-write chk│
                    └─────────────────┘
```

## Acceptance Criteria

### Story #1: Defect implement happy path [core]

```
GIVEN a defect work-item exists at work-items/defects/<component>/D000xxx_<slug>/
  AND the dir contains a complete _TRACKER.md (type: defect), _RCA.md, _test-plan.md
  AND Phase 1 gates are green
WHEN  the user runs /implement-from-spec work-items/defects/<component>/D000xxx_<slug>/
THEN  the skill reads RCA + test-plan as the SPEC-equivalent
  AND proposes/implements the change per the test-plan rows
  AND writes a journal entry [impl-pass] with the work-item type recorded
  AND post-write boundary check passes
```

### Story #2: Defect QA happy path [core]

```
GIVEN a defect work-item with Phase 2 implementer-owned gates green and a [impl-pass] journal entry
WHEN  the user runs /qa-work-item work-items/defects/<component>/D000xxx_<slug>/
THEN  the skill reads test-plan as the test-spec equivalent
  AND runs the test-plan rows as smoke-equivalent (no E2E subagent dispatch in v1)
  AND writes a [qa-pass] journal entry on green
  AND transitions Phase 2 gates per template
```

### Story #3: Type detection error path [core]

```
GIVEN a work-item with malformed or missing type: field in TRACKER.md frontmatter
WHEN  the user runs /implement-from-spec or /qa-work-item on that dir
THEN  the skill exits with a clear error: "Error: TRACKER.md frontmatter missing or malformed type: field; cannot dispatch."
  AND no implementation or QA action is taken
```

### Story #4: User-story regression [resilience]

```
GIVEN an existing user-story work-item (e.g., S000020 from F000011)
WHEN  the user runs /implement-from-spec or /qa-work-item on the user-story dir
THEN  the skill behaves identically to v1.10.0 (reads SPEC + DESIGN for implement; TEST-SPEC for QA)
  AND no regression in journal output, AUQ behavior, or idempotency
```

### Story #5: Error table cleanup [core]

```
GIVEN the SKILL.md error-handling tables in /implement-from-spec and /qa-work-item
WHEN  a reviewer reads the table after this story merges
THEN  the "Wrong type (not user-story)" row is removed
  AND replaced with "Frontmatter type: missing or malformed" guidance
```

### Story #6: Per-type idempotency [resilience]

```
GIVEN a completed defect work-item with [impl-pass] journal entry and Phase 2 gates green
WHEN  the user re-runs /implement-from-spec on that defect dir
THEN  the skill prints "INFO: D000xxx already implemented; nothing to do."
  AND exits 0 without writing
```

## Architecture

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `/implement-from-spec` skill | `skills/implement-from-spec/SKILL.md` + `implement.md` | Modified | Add type-dispatch at start; per-type input-artifact resolution; update error table; update usage examples |
| `/qa-work-item` skill | `skills/qa-work-item/SKILL.md` + `qa.md` | Modified | Add type-dispatch; per-type input-artifact resolution; update error table |
| Test fixtures | `skills/implement-from-spec/fixtures/` (and qa equivalent if exists) | Modified | Add a defect fixture (TRACKER + RCA + test-plan) for the new code path |
| `personal-artifact-manifests.json` | `skills/personal-workflow/personal-artifact-manifests.json` | Read-only | Source of truth for per-type artifact lists; NOT modified |

### Data Flow

1. User invokes `/implement-from-spec <work-item-dir>` (or `/qa-work-item`).
2. Skill reads `<work-item-dir>/<ID>_TRACKER.md` frontmatter and extracts `type:`.
3. Skill consults the per-type input table (above) to determine which files to read as SPEC equivalent.
4. Skill reads those files; rest of flow (boundary check, sensitive-surface AUQ, propose-and-confirm, write code, journal) proceeds as today.
5. Post-write boundary check via `/personal-workflow check` validates output compliance for the work-item type.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Per-type branching | Generalize each skill (Approach A) | Sibling skills per type (B); Dispatcher skill (C); Accept (D) | A: pipeline uniformity, fewer names to remember, existing type-detection plumbing in /scaffold-work-item to model on. B explodes skill count. C adds indirection without removing per-type code. D loses the dogfood path for #5. |
| Defect implement input | RCA + test-plan (test-plan is de-facto SPEC) | RCA only; test-plan only | RCA tells implementer what NOT to revert; test-plan tells what to build. Both needed. |
| Defect QA tier | All test-plan rows = smoke-equivalent | Split test-plan into smoke + E2E sub-tiers | Defect test-plans don't have the smoke/E2E split today. Splitting later if needed is non-breaking; v1 ships uniform. |
| Feature-level invocation | Preserve existing AUQ-pick-child | Auto-pick next un-shipped child | Explicit beats clever; user can ship one child at a time without surprise. |
