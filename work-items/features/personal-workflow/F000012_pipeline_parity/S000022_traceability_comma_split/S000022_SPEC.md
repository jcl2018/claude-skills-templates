## Problem Statement

`skills/personal-workflow/check.md` Step 18 (Cross-Reference Traceability) describes AC extraction as "Find the `## Smoke Tests` table; extract all values from the `AC` column → `smoke_acs`." The prose doesn't specify how to handle multi-AC cells. An LLM following the prose with field-by-field exact matching would treat `AC-1, AC-2, AC-3` as a single string that never matches the individual P0 story numbers — producing false `[UNTESTED]` findings on every multi-AC P0 entry in real TEST-SPECs.

The bug is in the prose, not the code. `check.md` is interpreted by the LLM running the skill. The fix is tightening the prose to be unambiguous about comma-handling, with a worked example pinning the ordering relative to the existing placeholder filter.

## Mental Model

`check.md` is a runtime spec — the LLM reads it during `/personal-workflow check` invocations and follows the steps. Step 18 has 6 numbered sub-steps. Sub-step 3 ("Parse TEST-SPEC.md for AC values") is where comma-split must happen, BEFORE the placeholder filter that drops `^AC-\{[a-zA-Z_]+\}$`.

```
TEST-SPEC AC column
  │
  ▼
For each cell:
  ├── split on comma
  ├── trim whitespace
  └── for each token:
       ├── filter out placeholders (AC-{n}, AC-{name})
       └── add to ac_set
  │
  ▼
ac_set = smoke_acs ∪ e2e_acs
  │
  ▼
For each P0 story #N:
  └── if AC-N not in ac_set: flag [UNTESTED]
```

```

        "For each cell in the AC column, split on comma and trim whitespace.
         Each resulting token (matching AC-N format after the placeholder filter)
         contributes one value to smoke_acs / e2e_acs."
  AND the instruction is unambiguous (no "may" or "consider" hedging)
```

### Story #2: Worked example present [core]

```
GIVEN Step 18 sub-step 3
WHEN  the reader scans below the rule
THEN  a worked example shows:
        Input row: | S2 | core | AC-1, AC-2, AC-3 | description |

        Step 2: trim whitespace → ["AC-1", "AC-2", "AC-3"]
        Step 3: apply placeholder filter (no placeholders here) → ["AC-1", "AC-2", "AC-3"]
        Step 4: add each to ac_set → ac_set ∪= {AC-1, AC-2, AC-3}
```

### Story #3: Comma-split runs before placeholder filter [resilience]

## Architecture

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| Step 18 prose | `skills/personal-workflow/check.md` (lines 339-371) | Modified | Tighten sub-step 3 instruction; add worked example block; preserve existing edge-case list |
| F000010 fixtures | `work-items/features/personal-workflow/F000010_pipeline_skills/` | Read-only | Used as verification target; no changes |
| Test fixtures (optional) | new fixture under `skills/personal-workflow/fixtures/` if traceability tests live there | New (optional) | Minimal SPEC + TEST-SPEC pair with multi-AC cells; runnable by test.sh |

### Data Flow

1. User invokes `/personal-workflow check <dir>`.
2. Skill reaches Step 18 (matching SPEC + TEST-SPEC pair).
3. Sub-step 3: for each AC column cell, split on comma + trim, then apply placeholder filter, then add to set.
4. Sub-step 4: for each P0 story #N, set-membership-check `AC-N` in `ac_set`.
5. If found: PASS (no flag). If not found: `[UNTESTED]` finding.

The change is in step 3's prose only. Steps 1, 2, 4, 5, 6 are unchanged.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Implementation surface | Prose tightening + worked example | Standalone parser script | `check.md` is intentionally LLM-interpreted (D000007 trust model: templates + WORKFLOW.md + check.md are sources of truth, AI executes them). A script changes the architecture. |
| Example placement | Inline under sub-step 3 | Separate "Examples" section at end | Inline is closer to the rule it illustrates; readers find it without scrolling. |
| Edge-case list | Preserved unchanged | Expanded with comma-split edge cases | Three existing edge cases are sufficient; comma-split is the rule itself, not an edge case. |
| Verification approach | Run check on F000010; spot-check S000018/S000019 P0 findings | Build a separate fixture | F000010 is the real-world test surface that motivated the fix; using it as verification is direct evidence. |
