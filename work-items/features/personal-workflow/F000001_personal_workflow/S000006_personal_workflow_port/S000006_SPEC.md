## Problem Statement

F000004 gives `/company-workflow` an external knowledge folder: engineers export
`AI_KNOWLEDGE_DIR`, drop categorized markdown under it, declare `.knowledge.yml`
surfacing modes, and Claude picks up both always-on guidance and on-demand
domain content automatically. `/personal-workflow` — the parallel validation
skill for personal-dev work items — has none of it. Engineers who use both
skills (company day job + personal side project) hit a cliff: the same
`$AI_KNOWLEDGE_DIR` works in one repo and silently does nothing in another.

This story closes that gap via a mechanical parity port. No new knobs, no new
schema, no new design decisions. Everything F000004 shipped for company-workflow
also works for personal-workflow.

## Mental Model

```
Before this story:
  /company-workflow ──► reads $AI_KNOWLEDGE_DIR ──► injects always-on + on-demand
  /personal-workflow ──► (nothing — knowledge env var is invisible to this skill)

After this story:
  /company-workflow ──► reads $AI_KNOWLEDGE_DIR ──► injects always-on + on-demand
  /personal-workflow ──► reads $AI_KNOWLEDGE_DIR ──► injects always-on + on-demand
                                                    (same env var, same schema,
                                                     same opt-in marker, same
                                                     warnings, same shipping
                                                     test helper)
```

The port is three copy-and-adapt moves, no more:

1. **SKILL.md §Knowledge Resolution** — copy company-workflow's block; adjust the `/company-workflow` reference to `/personal-workflow` in docstrings.
2. **SKILL.md §Knowledge Loading** — copy company-workflow's block; reuse the shared `scripts/test-helpers/knowledge.sh` as-is.
3. **WORKFLOW.md §Knowledge Configuration** — copy company-workflow's section; update command names in the prose.

`scripts/test.sh` gains a mirror of T000003's 11-case assertion block, pointed at personal-workflow's SKILL.md.

## Acceptance Criteria

### Story #1: resolution parity [core]

```
GIVEN AI_KNOWLEDGE_DIR is set to a valid directory
AND   company-workflow's SKILL.md and personal-workflow's SKILL.md both exist (deployed or in-repo)
WHEN  each skill's Knowledge Resolution block executes
THEN  both set $_KNOWLEDGE_DIR to the same resolved path
AND   neither emits a warning
AND   both exit 0 with empty stdout
```

### Story #2: warning-output parity [core]

```
GIVEN AI_KNOWLEDGE_DIR is unset
WHEN  personal-workflow's Knowledge Resolution block runs
THEN  exactly one stderr line is emitted
AND   the line names the variable
AND   the line is the same shape as company-workflow's (allowing only the skill-name reference to differ)
AND   exit code is 0

AND
GIVEN AI_KNOWLEDGE_DIR is set to a path with control characters or >200 chars
WHEN  the resolution block runs
THEN  the warning is sanitized identically to company-workflow (control chars stripped, truncated at 200)
```

### Story #3: always-on parity [core]

```
GIVEN $_KNOWLEDGE_DIR/coding/.knowledge.yml declares surface: always
AND   $_KNOWLEDGE_DIR/coding/ contains one or more *.md files (nested or flat)
AND   the current repo has .claude/knowledge-enabled
WHEN  /personal-workflow runs any command
THEN  the emitted Always-On Knowledge section lists the same paths that /company-workflow would list for the same fixture
AND   load order is deterministic (lexical, matching /company-workflow)
```

### Story #4: on-demand parity [core]

```
GIVEN a category with .knowledge.yml { surface: on-demand, triggers: [pricing engine] }
AND   the current repo has .claude/knowledge-enabled
WHEN  the user's latest message mentions "pricing engine"
THEN  /personal-workflow emits the same On-Demand Knowledge Candidates block that /company-workflow would
AND   Claude Reads the same set of paths under that block

AND
WHEN  the user's latest message mentions neither "pricing" alone nor "engine" alone
THEN  no on-demand files from that category are surfaced
```

### Story #5: per-repo opt-in parity [security]

```
GIVEN AI_KNOWLEDGE_DIR resolves to a valid directory with always-on + on-demand categories
AND   the current repo has NO .claude/knowledge-enabled marker
WHEN  /personal-workflow runs
THEN  no Always-On Knowledge section is emitted
AND   no On-Demand Knowledge Candidates section is emitted
AND   S000004's resolution warning still fires if the env var is misconfigured (the opt-in gate does not suppress resolution warnings)
```

### Story #6: zero regression [resilience]

```
GIVEN a clean personal-workflow install with no AI_KNOWLEDGE_DIR and no .claude/knowledge-enabled
WHEN  /personal-workflow check is run on an existing work-items/ tree
THEN  stdout is byte-identical to a run against the pre-port SKILL.md
AND   stderr contains only the AI_KNOWLEDGE_DIR=unset warning (identical to /company-workflow's)
AND   exit code is unchanged
```

### Story #8: docs read correctly [usability]

```
GIVEN a reader who only uses /personal-workflow
WHEN  they read personal-workflow/WORKFLOW.md §Knowledge Configuration
THEN  every command reference is /personal-workflow (not /company-workflow)
AND   the F000004 backlink points at the same feature (both skills share it)
AND   no unexplained company-workflow concepts are mentioned
```

## Out of Scope

- Any change to `AI_KNOWLEDGE_DIR` semantics, `.knowledge.yml` schema, or the opt-in marker
- New surfacing modes (anything beyond `always` and `on-demand`)
- Per-skill knowledge directories (single env var stays shared)
- Moving the shared bash into a sourced helper file — P2 Story #9, deferred unless a maintenance pain surfaces
- Changes to `personal-artifact-manifests.json` (knowledge is runtime, not scaffolding)
- Changes to `/company-workflow` (this port is additive)
- Port of any F000004 follow-ups not yet scoped into S000004/S000005 (e.g., soft-cap warnings for always-on bytes, if deferred)

## Assumptions

- S000005 has landed on `main` before Phase 2 starts. Without it, there's no Loading block to port.
- S000004's Resolution block is stable (it shipped in PR #38; no changes expected before this port begins).
- `scripts/test-helpers/knowledge.sh` exists and is reusable once S000005 lands.
- The deployed install paths for both skills (`~/.claude/skills/{company,personal}-workflow/`) are structurally symmetric — the port can copy bash verbatim without path-resolution rework.
- `.claude/knowledge-enabled` is the final opt-in marker name (decided during S000005's Track phase).
- Users who run both skills in the same repo will place the opt-in marker once and have both skills honor it.

## Overview

Mechanical parity port. Copy the two shipped bash blocks (`## Knowledge Resolution` from S000004; `## Knowledge Loading` from S000005) from `skills/company-workflow/SKILL.md` into `skills/personal-workflow/SKILL.md`, adapt only skill-name references, mirror the WORKFLOW.md `## Knowledge Configuration` section the same way, and add a T000007 test block to `scripts/test.sh` that mirrors T000003's assertions pointed at the new file.

No new design decisions. No new dependencies. No new env vars, schemas, or markers. The point of this story is to eliminate the asymmetry where `/company-workflow` reads the user's knowledge folder and `/personal-workflow` ignores it, without re-opening any decision F000004 already made.

## Architecture

```
Before:
  skills/company-workflow/SKILL.md
    ## Path Resolution
    ## Knowledge Resolution       (S000004 — shipped)
    ## Knowledge Loading          (S000005 — in flight)
    ## Template Registry
    ...

  skills/personal-workflow/SKILL.md
    ## Preamble
    ## Path Resolution
    ## Stale Rules Detection
    ## Overview                   ← no knowledge blocks
    ...

After:
  skills/company-workflow/SKILL.md   (unchanged)

  skills/personal-workflow/SKILL.md
    ## Preamble
    ## Path Resolution
    ## Stale Rules Detection
    ## Knowledge Resolution       ← MIRRORED from company-workflow
    ## Knowledge Loading          ← MIRRORED from company-workflow
    ## Overview
    ...

  scripts/test.sh
    # T000003 block — 11 cases against company-workflow/SKILL.md
    # T000007 block — 11 parallel cases against personal-workflow/SKILL.md  ← NEW
```

The block placement inside `personal-workflow/SKILL.md` is **after Stale Rules Detection, before Overview**. Same reasoning as company-workflow: resolution must run after the skill has verified its own assets exist (Path Resolution) but before any documentation-style sections the user might read to understand the skill. Stale Rules Detection is personal-workflow-specific (flags the old `~/.claude/rules/work-items.md`) and belongs to Path Resolution's prelude, so Knowledge Resolution slots in right after it.

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `skills/personal-workflow/SKILL.md` | claude-skills-templates | Modified | Add `## Knowledge Resolution` and `## Knowledge Loading` sections mirrored from company-workflow |
| `skills/personal-workflow/WORKFLOW.md` | claude-skills-templates | Modified | Add `## Knowledge Configuration` section mirrored from company-workflow |
| `scripts/test.sh` | claude-skills-templates | Modified | Add T000007 test block (parallel to T000003) asserting the mirrored blocks against personal-workflow/SKILL.md |

| `skills/company-workflow/*` | claude-skills-templates | Unchanged | The port is additive; company-workflow source is not touched |

### Data Flow

1. User exports `AI_KNOWLEDGE_DIR=/path/to/knowledge` in their shell profile.
2. User drops `.claude/knowledge-enabled` in the current repo root (per-repo opt-in).
3. User invokes `/personal-workflow check` (or any personal-workflow command).
4. SKILL.md's Path Resolution runs → sets `$_SKILL_DIR` + `$_TMPL_DIR`.
5. Stale Rules Detection runs → emits legacy warning if applicable.
6. **Knowledge Resolution runs** → sets `$_KNOWLEDGE_DIR` or emits an unset/invalid warning to stderr.
7. **Knowledge Loading runs** → if marker present, enumerates categories, parses `.knowledge.yml`, emits `## Always-On Knowledge` + `## On-Demand Knowledge Candidates` blocks.
8. Overview + Usage + other sections follow (user reads and executes the command as before).

Step 6 mirrors the company-workflow behavior shipped in S000004. Step 7 mirrors what S000005 will ship. Both are copied verbatim (modulo skill-name strings).

## API Changes

No API changes. All changes are internal to personal-workflow's SKILL.md and its test harness. The only user-visible change is that `AI_KNOWLEDGE_DIR` is now honored by `/personal-workflow` in addition to `/company-workflow` — an additive behavior change documented in WORKFLOW.md.

## Design Decisions

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Scope | Single story, both blocks ported together | Two stories (resolution now, loading later) | See S000006_TRACKER journal — avoids half-ported cutover |
| Duplication vs shared helper | Lift-and-shift (accept bash duplication for v1) | Extract Resolution + Loading to `skills/_shared/knowledge.sh`, source from both | Lift-and-shift is the minimum-viable port. Extraction is a broader refactor; S000006 tracks it as P2 Story #9, not a requirement |
| Block position in SKILL.md | After Stale Rules Detection, before Overview | Anywhere else (after Overview, after Usage, etc.) | Matches company-workflow's "resolve infra before explaining yourself" order; Stale Rules is personal-workflow-specific setup that must run first |
| Opt-in marker | Reuse `.claude/knowledge-enabled` (same as company-workflow) | Separate marker per skill (`.claude/knowledge-enabled-personal`) | The marker answers "does THIS repo want knowledge?"; it's a repo-level concern, not a skill-level one. Splitting would double the setup burden for no gain |
| Env var | Reuse `AI_KNOWLEDGE_DIR` (same as company-workflow) | Separate env var per skill | Same reasoning as the marker — the knowledge store is a user-level resource, not skill-scoped. Splitting forces dual config |
| Test harness | Add a parallel T000007 block in `scripts/test.sh` that mirrors T000003's 11 cases against personal-workflow/SKILL.md | Extract T000003 into a parameterized helper that both skills invoke | Parameterization is the right end-state but out of scope for v1; mirror-then-maybe-dedupe is the safer sequence (T000003 has landed and been reviewed; reopening it for parameterization risks regressing company-workflow) |
| WORKFLOW.md docs | Mirror the entire `## Knowledge Configuration` section, rename commands in prose only | Write a personal-workflow-specific section from scratch | Same content, same schema, same setup — writing from scratch invites subtle drift |
| Docs backlink | Same `work-items/features/F000004_knowledge_integration/` reference | Separate backlink per skill | Both skills share the same feature; one backlink is correct |
