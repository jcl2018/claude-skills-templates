## Problem Statement

The implementer of S000030 (and any change to `pipeline.md`) needs verified answers to two questions before editing:

1. Does the synthesized stub design doc satisfy `/scaffold-work-item`'s parser surface as-is?
2. Do the stub's `(none, brief mode bypasses ...)` placeholders avoid matching any Step 8.5 taste-fork scan pattern?

Without those answers, S000030 risks cascading rework: edit pipeline.md, run the fixture, hit a parser failure or a Step 8.5 false-positive, and have to either re-shape the stub template or escalate the entire feature to Approach B mid-implementation.

## Mental Model

Two read-only enumeration legs against existing skill source files. Each leg produces a verdict (yes/no) and an action recommendation. The combined output is a single journal note on S000029_TRACKER.md.

```
[S000029 spike]
  |
  +-- Phase 0.a: read scaffold.md → enumerate parser fields → verdict + action
  |
  +-- Phase 0.b: read pipeline.md Step 8.5 → enumerate scan surface → verdict + action
                                                                                    |
                                                                                    v
                                                              [combined journal note: 10–15 lines]
                                                                                    |
                                                                                    v
                                                       [extend | harden | escalate, recorded explicitly]
```

## Acceptance Criteria

### Story #1: parser surface verified [core]

```
GIVEN  the synthesized stub template (in F000015_DESIGN.md) and `skills/scaffold-work-item/scaffold.md`
WHEN   the spike enumerates every design-doc field the parser consumes
THEN   the journal note states "stub satisfies parser surface: yes" OR "stub satisfies parser surface: no, missing fields {X, Y, ...}"
```

### Story #2: Step 8.5 scan surface verified [core]

```
GIVEN  the synthesized stub template and `skills/personal-pipeline/pipeline.md` Step 8.5
WHEN   the spike enumerates the sections + patterns Step 8.5 scans for taste / user-challenge surfaces
THEN   the journal note states "placeholders inert: yes" OR "placeholders inert: no, pattern {P} matches placeholder {Q}"
```

### Story #3: combined journal note captured [observability]

```
GIVEN  both legs of the spike completed
WHEN   the combined output is written
THEN   `S000029_TRACKER.md` Journal contains a 10–15 line entry with: parser fields enumerated, Step 8.5 scan surface enumerated, both verdicts, and an explicit extend/harden/escalate action
```

### Story #4: extension scope is single-file [resilience]

```
GIVEN  Phase 0.a verdict is "no, missing fields"
WHEN   the recommended action is recorded
THEN   the action names a single file to extend (the stub template inside `pipeline.md` Step 0a synthesis block, or — if not feasible there — F000015_DESIGN.md's stub template; pick one)
```

### Story #5: escalation reason captured [usability]

```
GIVEN  recommended action is "escalate to Approach B"
WHEN   the action is recorded
THEN   a corresponding entry appears in TODOS.md with the rejection reason for the simpler Approach A
```

### Components Affected

| Component | Repo | Change Type | Description |

| S000029 TRACKER.md Journal | claude-skills-templates | Modified | Append combined Phase 0 note (10–15 lines) |
| TODOS.md | claude-skills-templates | Modified | Optional — only if action = escalate (Approach B reason captured) |

### Data Flow

1. Read `skills/scaffold-work-item/scaffold.md` — extract every reference to design-doc parsing (Step 2 "Read the Design Doc" + downstream consumers in Steps 3, 4, 7, 8, 10).
2. Build the parser-field list (frontmatter keys + section headers + content patterns).
3. Cross-reference against the F000015 stub template; mark each field present / missing.
4. Read `skills/personal-pipeline/pipeline.md` Step 8.5 — extract every scan pattern (Tradeoffs/SPEC paths, top-level design-doc sections if any).
5. Build the scan-surface list.
6. Cross-reference against the stub's `(none, brief mode bypasses ...)` placeholders; mark each placeholder safe / matches.
7. Write the combined journal note + action to `S000029_TRACKER.md`.
8. If action = escalate, append a TODOS.md entry capturing the reason.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Output format | 10–15 line journal note on the tracker | Separate `phase0-findings.md` doc | The output is short, action-oriented, and lives where S000030's implementer will look. A separate doc adds discoverability cost without proportional payoff. |
| Action recording | Explicit extend / harden / escalate keyword | Free-form prose recommendation | Keyword forces the decision; prose lets it slip. |
| Spike scope | Read-only — no pipeline.md edits in S000029 | Combined spike + minimum stub-template fix | Keeps scope tight. If the verdict is "extend," S000030 applies the extension as part of its own change set, with the spike as input. |
| Escalation surface | TODOS.md (single line) | A new D-ticket | TODOS.md is lower-overhead for a deferred decision; if Approach B becomes a real follow-up, it can graduate to a feature ticket. |
