## Problem Statement

The current personal-workflow templates + manifest + validator encode an artifact set (4/4/2/3 with feature-summary, PRD, ARCHITECTURE, milestones as separate docs) that no longer matches how the engineer actually works. The workflow today is `/office-hours → /personal-workflow templates → implement → smoke + E2E → /ship → /land-and-deploy` but: (1) `/office-hours` outputs are not first-class artifacts in the work-item directory; (2) PRD and ARCHITECTURE for user-stories are written together as one "lock the structure doc" act but live in two separate files; (3) feature-summary.md largely duplicates TRACKER.md fields; (4) milestones.md is often empty for small features.

This story introduces the new template + manifest + validator shape that subsequent stories migrate to. After this story lands, all NEW work items can be scaffolded with the new shape; historical items still match the old shape and will be migrated by S000015.

## Mental Model

Three layers change here, in dependency order:

1. **Templates** (the source of truth for structural rules) — create doc-SPEC.md + doc-ROADMAP.md, delete 4 old doc templates, edit doc-DESIGN.md + doc-TEST-SPEC.md, rewrite 2 tracker templates, lightly edit 1 tracker template.
2. **Manifest** (`personal-artifact-manifests.json`) — declares which artifacts each work-item type requires. Bumped to v3.0.0 with new types.feature and types.user-story arrays.
3. **Validator** (`check.md`) — Step 18 (cross-reference traceability) needs filename + section name updates; Steps 11 and 16 auto-follow because they read the manifest + templates at runtime.

Side effects: WORKFLOW.md absorbs the narrative changes (7 line edits); SKILL.md gets a version bump.

## Acceptance Criteria

### Story #1: New feature scaffolding produces TRACKER+DESIGN+ROADMAP [core]

```
GIVEN the new templates + manifest are in place
WHEN  I scaffold a hypothetical F999_smoke_test feature directory under /tmp using the new templates
THEN  the directory contains exactly F999_TRACKER.md + F999_DESIGN.md + F999_ROADMAP.md
AND   `/personal-workflow check` on the directory returns PASS for ARTIFACTS
```

### Story #2: New user-story scaffolding produces TRACKER+DESIGN+SPEC+TEST-SPEC [core]

```
GIVEN the new templates + manifest are in place
WHEN  I scaffold a hypothetical S999_smoke_test user-story directory under /tmp
THEN  the directory contains exactly S999_TRACKER.md + S999_DESIGN.md + S999_SPEC.md + S999_TEST-SPEC.md
AND   no PRD.md or ARCHITECTURE.md present
```

### Story #3: check.md understands the new artifact set [core]

```
GIVEN a synthetic new-shape work item under /tmp
WHEN  I run `/personal-workflow check </tmp/synthetic-feature/>`
THEN  the report shows ARTIFACTS PASS for TRACKER, DESIGN, ROADMAP
AND   no [MISSING] for old artifact names
```

### Story #4: Step 18 PRD↔TEST-SPEC traceability works on SPEC.md [core]

```
GIVEN a new-shape user-story with SPEC.md containing `### P0 (Must-Have)` table with rows numbered 1, 2
AND   TEST-SPEC.md with `## Smoke Tests` containing AC-1, AC-2 entries (filtered, not literal `AC-{n}` placeholders)
WHEN  I run `/personal-workflow check` on the user-story directory
THEN  Step 18 reports `[PASS] All P0 stories have TEST-SPEC coverage`
AND   no [UNTESTED] for stories 1 or 2
```

### Story #5: No stale artifact references in active skill surfaces [core]

```
GIVEN the templates + manifest + check.md + WORKFLOW.md + SKILL.md are updated
WHEN  I run `grep -rEn "PRD\.md|ARCHITECTURE\.md|feature-summary|doc-milestones" skills/personal-workflow/ templates/personal-workflow/`
THEN  no matches in active skill files (excluding error messages, line 84/218/220/365 incidentals which become SPEC, and deprecated/, work-items/)
```

## Out of Scope

- Migration of historical work items — owned by S000015.
- Examples, fixtures, repo-level docs (CONTRIBUTING, PHILOSOPHY, template-registry, scripts) — owned by S000016.
- F000008 self-migration — happens during S000015's sweep.
- Backward-compat / legacy mode in check.md for old-shape items — explicitly rejected by design.
- VERSION bump and CHANGELOG entry — owned by F000008's Phase 3 (`/ship` step).

## Assumptions

- v1.4.0 (TEST-SPEC restructure) is shipped on origin/main (verified: commit abe411c, PR #57).
- The new doc-SPEC.md preserves PRD's `### P0/P1/P2` sub-sections inside `## Requirements` so Step 18 parser stays a filename-only change. (Per design Big Decision #6.)
- `/personal-workflow check` walks `work-items/` not `deprecated/work-items/`, so deprecated F000003 work items don't pollute the check report.
- Engineer can scaffold synthetic work items under `/tmp` for smoke testing without polluting the real `work-items/` tree.
