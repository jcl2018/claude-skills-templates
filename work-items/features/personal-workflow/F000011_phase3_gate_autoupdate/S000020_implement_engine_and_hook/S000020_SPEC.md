## Problem Statement

After every `/ship` + `/land-and-deploy` cycle on a personal-workflow work-item, Phase 3 lifecycle gates in the tracker stay UNCHECKED. Today's session shipped 4 PRs (S000017, S000019, S000018, D000016 work) and all 4 carry stale Phase 3 state. Manual gate updates are unmemorable in practice — direct evidence: I forgot 4 times today.

The fix has two parts: (1) an inference engine that reads external state and writes the inferable gates, (2) an auto-trigger so the engine fires without requiring a new manual step.

## Mental Model

```
INPUT                                            OUTPUT
─────                                            ──────
work-items/{type}s/{component}/                  (1) Phase 3 gates auto-marked:
  {ID}_{slug}/                                       - [x] /personal-workflow check passed
    └── {ID}_TRACKER.md  ◄── primary input           - [x] Smoke tests pass in CI
                                                     - [x] /ship — PR created
       │                                             - [x] /land-and-deploy — merged + deployed
       ▼                                             - [x] All children shipped (if any)
  /personal-workflow check --update                  - [x] /document-release (if recent docs commit)
       │                                             - [ ] E2E walked manually  (NEVER auto-marked)
       ├── reads gh PR state                         (2) ## PRs section gets merged PR link
       ├── reads gh PR checks                        (3) ## Journal gets [gates-update] entry
       ├── reads child trackers (recursive)
       └── writes Phase 3 checkboxes
                                                 (4) Idempotent: re-run = NO-OP

scripts/post-merge-hook.sh                       (5) Hook trigger:
       │                                             - fires after `git pull` on main
       ├── if-on-main check                          - finds touched work-item dirs
       ├── git log @{1}..HEAD --name-only            - runs --update on each
       └── per-touched-dir: invoke --update
```

## Acceptance Criteria

### Story #1: Engine writes inferable Phase 3 gates [core]

```
GIVEN  a work-item dir whose PR is merged + deployed (e.g., S000017 after this session)
       AND `gh` is authenticated and online
WHEN   /personal-workflow check --update <work-item-dir> is invoked
THEN   gates "/ship — PR created", "/land-and-deploy — merged and deployed",
       "Smoke tests pass in CI", and "/personal-workflow check — validation passed"
       are marked [x] in the tracker
       AND `E2E walked manually` is left UNCHECKED
       AND `## PRs` section has the merged PR link with status
       AND `## Journal` has a new [gates-update] entry summarizing what changed
```

### Story #2: Idempotency [resilience]

```
GIVEN  a work-item dir where /personal-workflow check --update has already run successfully
WHEN   /personal-workflow check --update is re-invoked on the same dir
THEN   no checkboxes change state
       AND no duplicate PR link is added to `## PRs`
       AND no duplicate [gates-update] journal entry is added (or: the new entry says "no changes")
       AND the skill exits clean
```

### Story #3: Manual override preserved [resilience]

```
GIVEN  a work-item where the user manually marked `E2E walked manually` as [x]
WHEN   /personal-workflow check --update runs
THEN   the [x] on `E2E walked manually` is NOT downgraded to [ ]
       AND no other gates are downgraded (additive-only contract)
```

### Story #4: Engine handles `gh` offline [resilience]

```
GIVEN  `gh` is unauthenticated or network is offline
WHEN   /personal-workflow check --update runs
THEN   the structural check (existing /personal-workflow check behavior) still runs
       AND `/ship — PR created` and `/land-and-deploy` gates are skipped (left unchanged)
       AND the engine prints a warning naming the gh failure
       AND the skill exits clean (no crash)
```

### Story #5: Hook fires on main after pull [core]

```
GIVEN  the post-merge hook is installed and HEAD is on main
WHEN   `git pull origin main` lands new commits that touch work-items/**/*_TRACKER.md
THEN   the hook script runs
       AND for each touched work-item dir, `/personal-workflow check --update <dir>` is invoked
       AND the user sees a one-line summary per work-item processed
```

### Story #6: Hook silently no-ops on feature branch [core]

```
GIVEN  the post-merge hook is installed and HEAD is on a feature branch
WHEN   `git pull` or `git merge` runs on that branch
THEN   the hook detects HEAD != main
       AND exits 0 silently (no-op, no warning)
```

### Story #7: Hook failure doesn't block git [resilience]

```
GIVEN  the post-merge hook script has a bug (e.g., gh missing, permission error)
WHEN   `git pull` triggers the hook
THEN   the hook prints a warning to stderr
       AND exits 0 (not the actual error code)
       AND the git operation completes normally
       AND the user can investigate / fix manually later
```

### Data Flow

1. User runs `/ship` → PR created
2. PR auto-merges (CI passes)
3. User runs `git pull` on main locally → fast-forward
4. Git fires post-merge hook
5. Hook checks branch == main, runs `git log @{1}..HEAD --name-only`, filters to `work-items/.*_TRACKER.md`, dedup to dirs
6. For each touched dir, hook invokes `/personal-workflow check --update <dir>` (via shell call)
7. Engine: structural check (unchanged), Step 13.5 inference (new), Step 13.6 PR linking (new), Step 13.7 journal entry (new), Tier 2 re-validation (unchanged)
8. Engine writes the tracker, prints summary
9. Hook exits 0 (best-effort)

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Engine: flag vs new skill | `--update` flag on existing check | New `/personal-workflow update-gates` skill | Keeps surface small; reuses existing path resolution + boundary check infra |
| Hook trigger | post-merge | post-checkout / post-rewrite | Post-merge fires on `git pull` after ship (the natural step); others would re-fire on branch switches |
| `E2E walked manually` policy | Explicit-exclude (never auto-mark) | Auto-mark on PR-merge-as-proxy | Contract: engine reflects verifiable reality. PR-merge doesn't prove E2E was walked |
| Idempotency strategy | Additive-only (never downgrade [x] → [ ]) | Bidirectional sync | Protects against user manually unchecking; engine doesn't fight user intent |
| Hook failure mode | Best-effort (warn, exit 0) | Block git operation on hook error | Git stays usable; user investigates manually |
| Children inference depth | One level (direct) | Multi-level recursion | v1 simplicity; revisit if multi-level features ship |
| `/document-release` heuristic | docs: commit on main touching work-item paths | Explicit `[doc-release]` marker | v1 ships with low-risk over-mark; v2 can tighten with explicit marker |
| Cross-machine merge coverage | Accept gap | Add cross-machine sync | Major scope expansion; not worth v1 cost |

## Open Questions

| Question | Next check |
|----------|-----------|
| When PR title doesn't include the work-item ID, how does engine find the PR? | Implementation: try `gh pr list --search "in:title <ID>"`; fall back to grep commit messages for the PR ref; if neither finds it, leave `/ship` gate unchanged with a warning |
| Multiple work items touched by a single PR (e.g., a chore PR that updates 3 trackers) | Implementation: hook handles each dir independently; same PR is linked to all three; should be fine |
| Hook ordering relative to existing pre-commit hook | Implementation: post-merge runs after merge completes; pre-commit runs before commit. No collision |
| Hook on `git rebase` (which can fast-forward main locally) | v1: post-merge hook fires on rebase too if main moves. Should be safe given idempotency |
