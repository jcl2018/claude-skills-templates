## Problem Statement

The /personal-pipeline orchestrator design assumes two Agent-tool subagent
behaviors that have not been verified in this workbench: (a) AskUserQuestion
calls inside a subagent bubble to the human user, and (b) subagents reliably
emit a controlled `RESULT: <key>=<value>` final line. The author is
chjiang (the workbench owner); the pain today is that pipeline.md cannot
be designed concretely until these two questions have YES/NO answers.

## Mental Model

Two independent probes, written as throwaway shell scripts. Each probe
constructs a Claude Code subagent invocation with a known-shape prompt,
captures the subagent's behavior, and emits a verdict line. The findings
report is hand-written after running both probes.

```
operator → probe-auq.sh    → claude -p / Agent → AUQ inside? → captured behavior
operator → probe-result.sh → 5 × (claude -p / Agent) → RESULT line? → hit/miss tally
                                                                          ↓
operator hand-writes findings.md (YES/NO + recommended action)
```

## Acceptance Criteria

### Story #1: AUQ-bubble probe [core]

```
GIVEN  the spike harness is set up and Claude Code is on a known version
WHEN   I run `tests/spike/subagent-capabilities/probe-auq.sh`
THEN   the script prints either "VERDICT: AUQ_BUBBLES=yes" (if the AUQ surfaced to me)
       or "VERDICT: AUQ_BUBBLES=no" (with a sub-classification: hang | error | auto-cancel)
       and exits 0 in either case (verdict is data, not failure)
```

### Story #2: RESULT-line reliability probe [core]

```
GIVEN  the spike harness is set up
WHEN   I run `tests/spike/subagent-capabilities/probe-result.sh` (5 trials)
THEN   the script prints "VERDICT: RESULT_LINE_HITS=N/5" where each hit is the subagent's
       final message ending with `RESULT: STATUS=ok` (no trailing prose)
       and exits 0
```

### Story #3: Findings report [usability]

```
GIVEN  both probes have been run
WHEN   I read tests/spike/subagent-capabilities/findings.md
THEN   I see: (1) Claude Code version + model overlay used,
              (2) probe-auq verdict + sub-classification,
              (3) probe-result hit/miss tally,
              (4) recommended downstream action for F000014/S000027 — one of:
                   "design as-is",
                   "redesign Phase 2 to pre-collect AUQs",
                   "add parser leniency",
                   "both redesigns"
```

### Story #4: Hang/error distinction [resilience]

```
GIVEN  probe-auq.sh runs the AUQ probe
WHEN   the subagent does NOT successfully bubble the AUQ
THEN   the verdict line includes a sub-classification (hang | error | auto-cancel)
       so the parent design can pick the right redesign
```

## Architecture

```
tests/spike/subagent-capabilities/
├── probe-auq.sh         # bash; spawns subagent with AUQ prompt, captures output
├── probe-result.sh      # bash; spawns subagent 5 times with RESULT prompt, tallies hits
└── findings.md          # markdown; hand-written after probes run
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| tests/spike/subagent-capabilities/ | claude-skills-templates | New | Three new files under a new dir; throwaway after findings written |

### Data Flow

1. Operator runs `probe-auq.sh` → script invokes `claude -p` (or uses Agent inline) with a prompt that has the subagent call AskUserQuestion.
2. Script captures stdout/stderr + return code; classifies as bubbled/hang/error/cancel.
3. Operator runs `probe-result.sh` → script loops 5 times, each time spawning a subagent with a "do X, end with RESULT: STATUS=ok" prompt.
4. Each iteration parses the subagent's final assistant message via `grep -E '^RESULT: '`; hits tallied.
5. Operator hand-writes findings.md combining the two verdicts + recommended downstream action.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Spike shape | shell scripts | proper skill with frontmatter + catalog entry | Throwaway after findings; no need to deploy or version |
| Probe count | 5 trials | 1 trial | 1 is anecdote; 5 is signal. Cost is ~5 minutes total subagent runtime |
| Findings format | hand-written markdown | auto-generated JSON | Findings need recommended-action prose; LLM judgment beats template fill |
| Probe both AUQ variants? | yes if both callable | only the default | Asymmetry between MCP and native AUQ would change the design |
