## Problem Statement

`/personal-pipeline` users (currently: just chjiang at scale=1) face a small
but recurring decision tax: should this run be `--auto` or default-manual?
After v1.14.0 shipped the `--auto` flag, lived experience confirmed the
manual path is dead-by-policy — nothing outside personal habit recommends it.
This story removes the decision: auto IS the experience. The orchestrator
runs through with auto-decision classification (Mechanical / Taste /
User-Challenge), surfaces only Taste + User-Challenge-Approved decisions at
Step 8.5's final approval gate, and halts on Halt-at-Gate User Challenges.
The flags `--auto` and `--manual` are accept-and-discard for backwards compat.

## Mental Model

The change is structural deletion + structural promotion, not new logic:

```
BEFORE (v1.15.1)                       AFTER (v1.16.0)
─────────────────                      ─────────────────
SKILL.md                               SKILL.md
  Usage: /pp [--auto] <doc>              Usage: /pp <doc>
  ## Auto Mode (50 lines)                (deleted)
  Two examples (auto + manual)           One example
                                       
pipeline.md                            pipeline.md
  AUTO_MODE=false                        (deleted — variable removed)
  case --auto) AUTO_MODE=true ;;         case --auto|--manual) ;;  # discard
  ## Auto Mode Overlay                   ## Auto Mode Overlay
    Active when $AUTO_MODE=true            (framing dropped — overlay IS the flow)
  Per-step: Manual mode: …               Per-step: single unconditional paragraph
            Auto mode (Step N): …          (using auto-mode behavior)
  Step 8.5: Skip if $AUTO_MODE=false     Step 8.5: always fires (carve-outs unchanged)
  _MODE=$([ ... ] auto || manual)        _MODE="auto"
  Closing: Auto mode (`$AUTO_MODE=true`) Closing: single-mode narration
           in both modes …                       (drop "in both modes")
```

The Auto Mode Overlay's substance (6 principles, decision classification,
$DECISION_LOG schema, Step 8.5 logic) is preserved — it just stops being
gated by `$AUTO_MODE=true` and starts being the only flow.

### Data Flow

1. User runs `/personal-pipeline <design-doc-path>` (with or without `--auto`/`--manual`)
2. Step 1 parses args; the case statement matches `--auto|--manual)` and discards both as silent no-ops; no `$AUTO_MODE` variable is set
3. Steps 2-8 execute the auto-decision flow (Mechanical / Taste / User-Challenge classification) unconditionally; the prior conditional gating (`if $AUTO_MODE=true`) is gone
4. Each Taste or User-Challenge-Approved decision appends a row to `~/.gstack/analytics/personal-pipeline-auto-decisions.jsonl` ($DECISION_LOG)
5. User-Challenge-Halt-at-Gate decisions halt the pipeline at the originating step (unchanged)
6. Step 8.5 fires unconditionally subject to (a) $DECISION_LOG-empty short-circuit (silent pass) and (b) end-state carve-out for `halted_at_gate`/`subagent_crashed` (both unchanged)
7. Step 9 sunset criterion check runs interactively (unchanged — by design, not auto-decided)
8. Telemetry JSONL row appends with `mode: "auto"` literal (no conditional eval)

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Flag handling | Accept-and-discard `--auto` AND `--manual` | (a) Hard error on `--auto`; (b) Hard error on `--manual` only; (c) Keep `--auto` as no-op but error on `--manual` | Symmetric forgiveness preserves existing scripts and muscle memory. Hard-erroring breaks footers in old design docs and CI scripts. Asymmetric handling (option c) creates a new "two ways" mental model in the very change that's eliminating one. |
| Manual code path | Delete (Approach B) | (A) Keep manual reachable as `--manual` flag; (C) Two-step ship with one-week soak | Approach A leaves dead-by-policy debt and doesn't match the user's D2.A intent ("flag disappears entirely"). Approach C splits one design into two PRs at scale=1 where telemetry-driven safety is theatre. Approach B ships the polarity flip, deletion, docs, and reversal note in one coherent change with `git revert` as rollback. |
| Auto Mode Overlay substance | Promote (move to main flow, strip conditional framing) | Delete and re-author later if needed | Substance (~200 lines: 6 principles, decision classification, $DECISION_LOG schema, Step 8.5 logic) is valuable. Promotion preserves it; only ~10-15 lines of conditional framing get deleted. A future revert restores conditionals (~1 hour rewrap), not re-authored substance. |
| Telemetry `mode` field | Stay in v1.16.0 emitting `"auto"` literal; defer deletion to v1.17.0 (TODOS.md) | Delete in v1.16.0 in the same PR | Avoids breaking external JSONL readers mid-flight even at scale=1. One release of grace is cheap insurance; the deferral is logged as a TODOS.md follow-up so it doesn't get lost. |
| Versioning | MINOR (1.16.0) | MAJOR (2.0.0) | Removed flag is accept-and-discard (zero break for existing invocations). Default behavior changed but the result envelope (pipeline runs through, Step 8.5 surfaces decisions, sub-skills callable individually) is preserved. Mirrors v1.13.x → v1.14.x precedent (which was a UX default change with backwards-compat scaffold). |
| Reversal documentation | Explicit "reverses S000028 premise 1" phrase in CHANGELOG | Rebrand as "evolution" or omit the cross-reference | Premise 5: an honest reversal paper trail is a higher discipline bar than "ship and forget." Future readers (including future-me) need to see "I changed my mind, here's why" without rebranding. Validated by Success Criteria #8 (`grep` for the exact phrase). |
| Step 8.5 silent opt-out | None — Step 8.5 stays mandatory on green-or-recoverable (P3) | Add `--silent` flag to skip Step 8.5 | Rejected in D2 — would have added a new flag while killing one. The empty-state short-circuit (no Taste, no User-Challenge-Approved) already produces a silent pass when nothing surfaces. |
