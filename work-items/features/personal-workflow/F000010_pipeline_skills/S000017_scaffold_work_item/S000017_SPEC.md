## Problem Statement

The user (chjiang, solo developer maintaining this workbench) currently scaffolds work items by directing Claude inline: "scaffold a feature for this design." Claude reads `WORKFLOW.md`, `personal-artifact-manifests.json`, and `templates/personal-workflow/*`, then writes a directory tree. This works but is conversational — not reproducible across sessions, not enforceable, and the AI's attention is split between scaffolding mechanics and ongoing context.

A `/scaffold-work-item` skill codifies the operation: takes a design-doc path, produces a compliant work-item directory tree, runs `/personal-workflow check` to verify. One keystroke replaces the conversation.

## Mental Model

```
INPUT                                    OUTPUT
─────                                    ──────
~/.gstack/projects/{slug}/               work-items/{type}s/{component}/
  ...-design-*.md                          {ID}_{slug}/
       │                                     ├── {ID}_TRACKER.md
       │                                     ├── {ID}_DESIGN.md
       ▼                                     ├── (feature: {ID}_ROADMAP.md)
  parsed for: title,                         ├── (user-story: {ID}_SPEC.md +
              recommended approach,          │   {ID}_TEST-SPEC.md)
              alternatives,                  └── (feature: {child-slug}/
              success criteria                       └── {S_ID}_*.md ...)
       │
       ▼                                  + (idempotency check passes)
  /scaffold-work-item                     + (/personal-workflow check passes)
       │
       ├── reads templates/personal-workflow/*
       ├── reads personal-artifact-manifests.json
       ├── reads WORKFLOW.md (hierarchy + scaffolding rules)
       └── (optional, v1 deferred) Agent tool for validator subagent
```

## Acceptance Criteria

### Story #1: Feature-level scaffold [core]

```
GIVEN  a feature-level design doc at ~/.gstack/projects/{slug}/{user}-{branch}-design-{datetime}.md
       AND the current branch is feature/feat-* (matches feature pattern)
       AND work-items/features/personal-workflow/F000009_skills_update_check/ is the highest existing F-prefix
WHEN   /scaffold-work-item ~/.gstack/projects/{slug}/{user}-{branch}-design-{datetime}.md is invoked
THEN   work-items/features/personal-workflow/F000010_{slug}/F000010_TRACKER.md exists
       AND F000010_DESIGN.md, F000010_ROADMAP.md exist
       AND at least one user-story child dir (S000017_{slug}/) exists with 4 artifacts
       AND /personal-workflow check on the new directory returns PASS
```

### Story #2: User-story-level scaffold [core]

```
GIVEN  a user-story-level design doc
       AND the current branch is story-* (matches user-story pattern)
WHEN   /scaffold-work-item is invoked
THEN   a single user-story dir is created with TRACKER + DESIGN + SPEC + TEST-SPEC
       AND no feature dir is created
       AND /personal-workflow check returns PASS
```

### Story #3: Boundary check passes [core]

```
GIVEN  a freshly scaffolded work-item directory
WHEN   /scaffold-work-item finishes step 7 (write directory)
THEN   it invokes /personal-workflow check <new-dir>
       AND if check PASSES → exit clean
       AND if check FAILS → AskUserQuestion to surface violations
```

### Story #4: Idempotency [resilience]

```
GIVEN  a work-item directory already scaffolded for design doc D
WHEN   /scaffold-work-item D is invoked a second time
THEN   the skill detects "already done" via filesystem state + /personal-workflow check at start
       AND prints "F000010 already scaffolded at <path>; nothing to do"
       AND exits without writing anything
```

### Story #5: Boundary check at start (Premise 1.3) [resilience]

```
GIVEN  a stale work-item directory where templates have drifted (e.g., new required section added)
WHEN   /scaffold-work-item D is invoked
THEN   /personal-workflow check at START surfaces the drift
       AND the skill refuses to write more on top of stale state
       AND AskUserQuestion to either repair the existing dir or abort
```

### Story #6: Branch type AskUserQuestion [usability]

```
GIVEN  the current branch is `main` (no type prefix)
WHEN   /scaffold-work-item D is invoked
THEN   AskUserQuestion: "Branch 'main' doesn't match a type pattern. What type should I use? feature / user-story / task / defect"
       AND the user's answer is used for type
```

### Story #7: Multi-story decomposition AskUserQuestion [usability]

```
GIVEN  a feature-level design doc with 3 alternatives in its "Recommended Approach"
WHEN   /scaffold-work-item D is invoked AND type is feature
THEN   AskUserQuestion: "Proposed user-story children: A, B, C with slugs <a-slug>, <b-slug>, <c-slug>. Confirm or override?"
       AND the user's answer determines which user-stories are scaffolded
```

## Architecture

```
+--------------------+
| /scaffold-work-item|
+--------------------+
         |
         | 1. read design-doc-path arg
         |
         v
   +-------------+
   | Read inputs |
   |             |
   |  - design   |  reads ~/.gstack/projects/{slug}/...-design-*.md
   |  - manifest |  reads skills/personal-workflow/personal-artifact-manifests.json
   |  - WORKFLOW |  reads skills/personal-workflow/WORKFLOW.md (hierarchy rules)
   |  - templates|  reads templates/personal-workflow/*.md (2-level fallback)
   +-------------+
         |
         | 2. derive type from branch / AskUserQuestion
         |
         v
   +-------------+
   | Plan tree   |
   |             |
   |  - type     |  feature / user-story / task / defect
   |  - ID       |  scan work-items/, increment highest existing
   |  - slug     |  from design doc title or AskUserQuestion
   |  - children |  for feature, AskUserQuestion to confirm N user-stories
   +-------------+
         |
         | 3. boundary check at start (Premise 1.3)
         |     /personal-workflow check on existing
         |     dir if it exists; idempotency detect
         |
         v
   +-------------+
   | Already done?
   |   yes → NO-OP, exit
   |   no  → continue
   +-------------+
         |
         v
   +-------------+
   | Write tree  |
   |             |
   |  - mkdir -p work-items/{type}s/{component}/{ID}_{slug}/
   |  - mkdir -p .../child-slugs/ for feature children
   |  - fill templates with placeholders + content distillation
   |  - write all required artifacts per manifest
   +-------------+
         |
         | 4. boundary check at end (Premise 1.3)
         |
         v
   +-------------+
   | /personal-workflow check <new-dir>
   |   PASS → print path, exit
   |   FAIL → AskUserQuestion to surface
   +-------------+
         |
         v
   +-------------+
   | (P1) Append Status: SCAFFOLDED footer to source design doc
   +-------------+
         |
         v
       DONE
```

### Data Flow

1. User runs `/scaffold-work-item <design-doc-path>`
2. Skill reads design doc, manifest, WORKFLOW.md, and required templates
3. Skill determines type (from branch, with AskUserQuestion fallback)
4. Skill scans `work-items/` for highest existing ID of that type, increments
5. Skill runs `/personal-workflow check` at start: catches drift in existing work items, detects idempotent state
6. Skill plans the directory tree (feature: feature dir + N user-story children with confirmation; user-story: single dir)
7. Skill writes the directory tree, filling placeholders and distilling content from the design doc
8. Skill runs `/personal-workflow check` at end on the new directory: catches its own write violations
9. Skill optionally appends `Status: SCAFFOLDED → <path>` footer to the source design doc
10. Skill prints the new directory path for the user to copy into the next skill invocation

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Type detection | Branch name + AskUserQuestion fallback | Always AskUserQuestion | Branch naming is the existing convention; matching reduces friction in the common case |
| Multi-story decomposition | AskUserQuestion to confirm N + slugs | Auto-decompose from design's "Recommended Approach" | Design docs vary; auto-magic produces wrong decomposition often enough to require explicit confirmation |
| Validator subagent | Defer to v2 (covered by 1.3A boundary check) | Always run validator subagent | Source design says optional; boundary check is a tighter, faster pattern |
| Source design doc footer | Append `Status: SCAFFOLDED → ...` (P1) | Don't modify the source design doc | Traceability matters when 50+ design docs accumulate; small footer is non-invasive |
| Idempotency mechanism | Filesystem state + /personal-workflow check | `.in-progress` markers | Premise 1.1: idempotency over markers; fewer files, same recovery |
