## Problem Statement

After `/scaffold-work-item` produces a user-story directory, the user today manually directs Claude to read SPEC + DESIGN and write code. This is the highest-friction step of the day-to-day workflow because the user has to provide context inline ("here's the SPEC, here's the codebase, please implement"). The `/implement-from-spec` skill removes that conversation by taking a user-story dir, reading the handoff docs, and writing code per SPEC architecture decisions, then transitioning the tracker through Phase 1 → Phase 2 lifecycle gates as they're met.


## Acceptance Criteria

### Story #1: Implement code from SPEC [core]

```
GIVEN  a user-story directory with valid SPEC.md (P0 requirements + AC + Architecture)
       AND DESIGN.md providing context
       AND tracker in Phase 1 with all Phase 1 gates checked
WHEN   /implement-from-spec <user-story-dir> is invoked
THEN   code is written/edited matching SPEC's Components Affected and Data Flow
       AND tracker journal has new category-grouped entries (decision / finding / implementation)
       AND tracker Phase 2 gates check off as their preconditions are met
       AND /personal-workflow check on the user-story dir returns PASS
```

### Story #2: Idempotency [resilience]

```
GIVEN  a user-story whose SPEC has already been implemented (Phase 2 gates green)
WHEN   /implement-from-spec is re-invoked
THEN   skill detects "already implemented" via /personal-workflow check at start + tracker state
       AND prints "S000017 already implemented; nothing to do"
       AND exits without writing
```

### Story #3: Feature dir mistake [usability]

```
GIVEN  the user passes a feature dir (e.g., F000010_pipeline_skills/) by mistake
WHEN   /implement-from-spec is invoked
THEN   skill prints "F000010 has 3 user-story children:"
       AND lists S000017, S000018, S000019 with their slugs
       AND AskUserQuestion: "Which one to implement?"
```

### Story #4: Sensitive surface confirmation [usability]

```
GIVEN  a SPEC whose architecture requires changes to skills-catalog.json
WHEN   /implement-from-spec runs the implementation step
THEN   before committing the catalog change, AskUserQuestion: "This change touches skills-catalog.json. Proceed?"
       AND user's answer determines whether the change lands
```

### Story #5: Propose-and-confirm default [usability]

```
GIVEN  a SPEC with multiple architecture forks or non-trivial scope
WHEN   /implement-from-spec is invoked WITHOUT --auto
THEN   skill writes proposed changes to a preview (e.g., diff format)
       AND AskUserQuestion: "Apply these changes? Yes / Modify / Cancel"
       AND user's answer determines next step
```

### Story #6: --auto for trivial changes [usability]

```
GIVEN  a SPEC with ≤2 files touched, no sensitive surface change
WHEN   /implement-from-spec --auto is invoked
THEN   skill writes changes directly without proposing first
       AND tracker journal records "auto-mode" in the entry metadata
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `skills/implement-from-spec/` | claude-skills-templates | New | New skill directory with SKILL.md, fixtures/ |
| `skills-catalog.json` | claude-skills-templates | Modified | New catalog entry, status: experimental |
| Various source files | varies (per user-story) | Modified | Implementation per SPEC for the user-story being implemented |
| TRACKER.md (per user-story) | varies | Modified | Journal entries, lifecycle gate transitions |

### Data Flow

1. User runs `/implement-from-spec <user-story-dir>` (optionally `--auto`)
2. Skill validates input is a user-story dir; on feature dir, lists children + AUQ
3. Skill runs `/personal-workflow check <user-story-dir>` at start; refuses if input invalid
4. Skill reads SPEC.md, DESIGN.md, TRACKER.md
5. Skill plans the implementation: components affected, data flow, tradeoffs, sensitive surface check
6. If `--auto` and trivial: write directly. Else: propose diff, AUQ to confirm
7. Skill writes code via Read/Edit/Write tools
8. Skill updates tracker: journal entries (decision/finding/implementation), Phase 2 gates
9. Skill runs `/personal-workflow check <user-story-dir>` at end; AUQ on violations
10. Skill exits with summary ("wrote N files, journal entries M added")

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Granularity | User-story-only input | Accept feature dir, auto-iterate | Issue 1.2A: explicit > implicit; auto-iteration hides per-story failures |
| Default mode | Propose-and-confirm | Write directly always | Safety default; `--auto` opt-in for trivial cases |
| Sensitive surface | AskUserQuestion before commit | Always require confirmation | Targeted protection on high-blast-radius surfaces only |
| Code reviewer subagent | Defer to v2 | Always run code reviewer | Source design says optional; /qa-work-item is the safety net |
| Subagent for taste decisions | None in v1 | Two subagents agree/disagree pattern | 2x cost, marginal value; defer until /qa misses things |
