## Problem Statement

A Copilot user on the Windows work box installs `work-copilot/` via
`scripts/copilot-deploy.py install`, opens VS Code with Copilot Chat, and
asks procedural questions about the workflow:

- *"What are the workflow phases?"*
- *"Show me an example architecture doc."*
- *"What's the rationale for the PRD format?"*
- *"How do I write a task tracker?"*

Today (post-v0.14.0), Copilot answers these questions from training data —
not from any work-copilot artifact, because the bundle doesn't ship a
procedural backbone (`WORKFLOW.md`), how-to guides (`reference/`),
rationale notes (`philosophy/`), or example artifacts (`examples/`). The
Claude Code home-box user has all of these in `skills/company-workflow/`;
the Copilot work-box user does not. As a result the work-box workflow is
"second class" on the artifact-completeness axis — same templates and same
manifest, but materially less ambient knowledge for Copilot to draw on.

This story closes the gap by mirroring those artifacts byte-identically
into `work-copilot/`, and updating `copilot-instructions.md` to point at
them so Copilot reads the bundle file instead of recalling from training.

## Mental Model

Five mirror operations, one design:

```
skills/company-workflow/                  work-copilot/
├── WORKFLOW.md                  ──>      ├── WORKFLOW.md
├── reference/                   ──>      ├── reference/
│   └── guide-*.md  (7 files)             │   └── guide-*.md
├── philosophy/                  ──>      ├── philosophy/
│   └── rationale-*.md (3 files)          │   └── rationale-*.md
├── examples/                    ──>      ├── examples/
│   └── example-*.md (14 files)           │   └── example-*.md
└── fixtures/                    ──>      └── fixtures/
    └── (5 missing/drifted)                   └── (5 closed)
```

The bundle-side files are byte-identical copies enforced by
`scripts/validate.sh` Error check 10 (extended in T000011 to a config-driven
`MIRROR_SPECS` array). Copilot reads them via direct file references in
`copilot-instructions.md` — no `.prompt.md` slash-command needed for these
artifacts; they are reference material, not validators.

## Acceptance Criteria

### Story #1: Workflow phases [core]

```
GIVEN  a target repo with work-copilot/ installed under .github/
AND    Copilot Chat is open in VS Code with copilot-instructions.md as ambient context
WHEN   the user asks "what are the workflow phases?"
THEN   Copilot's response cites .github/work-copilot/WORKFLOW.md by path
AND    the cited content matches the phase definitions in skills/company-workflow/WORKFLOW.md
```

### Story #2: Reference guides [core]

```
GIVEN  the same setup as Story #1
WHEN   the user asks "how do I write a TEST-SPEC?" (or any guide-*.md topic)
THEN   Copilot's response cites .github/work-copilot/reference/guide-test-spec.md by path
AND    the cited content matches skills/company-workflow/reference/guide-test-spec.md
```

### Story #3: Philosophy rationales [core]

```
GIVEN  the same setup as Story #1
WHEN   the user asks "why is the PRD structured this way?"
THEN   Copilot's response cites .github/work-copilot/philosophy/rationale-PRD.md by path
AND    the cited content matches skills/company-workflow/philosophy/rationale-PRD.md
```

### Story #4: Example artifacts [core]

```
GIVEN  the same setup as Story #1
WHEN   the user asks "show me an example architecture doc"
THEN   Copilot's response cites .github/work-copilot/examples/example-doc-ARCHITECTURE.md by path
AND    the cited content matches skills/company-workflow/examples/example-doc-ARCHITECTURE.md
```

### Story #5: Drift detection [resilience]

```
GIVEN  the work-copilot/ bundle with WORKFLOW.md + reference/ + philosophy/ + examples/ + fixtures/ mirrored
AND    scripts/validate.sh Error check 10 has been extended to a MIRROR_SPECS array (T000011)
WHEN   a maintainer modifies one side of any mirror pair (e.g., edits work-copilot/reference/guide-task.md without editing skills/company-workflow/reference/guide-task.md)
AND    runs scripts/validate.sh
THEN   validate.sh exits non-zero with a [DRIFT] message naming the diverged file pair
```

### Story #6: Closed fixture gap [resilience]

```
GIVEN  the bundle's fixtures/ directory
WHEN   the maintainer compares each fixture to its upstream counterpart via cmp -s
THEN   all 5 previously missing/drifted fixtures are byte-identical:
       - invalid-bad-frontmatter.md (flat)
       - invalid-missing-lifecycle.md (flat)
       - invalid-wrong-order.md (flat)
       - valid-feature-dir/DESIGN.md (nested)
       - valid-feature-dir/TRACKER.md (nested, drift resolved)
```

### Story #7: 8 KB budget [usability]

```
GIVEN  work-copilot/instructions/copilot-instructions.md after the v2 "Bundle layout" pointer section is added
WHEN   the maintainer runs `wc -c < work-copilot/instructions/copilot-instructions.md`
THEN   the output is ≤ 8192
AND    the file contains literal references (grep -F) to each bundle-dir path:
       - "work-copilot/WORKFLOW.md"
       - "work-copilot/reference/"
       - "work-copilot/philosophy/"
       - "work-copilot/examples/"
       - "work-copilot/fixtures/"
```

### Story #8: Installer auto-pickup [core]

```
GIVEN  a fresh target repo with no .github/work-copilot/ directory
WHEN   the maintainer runs `python scripts/copilot-deploy.py install <target>`
THEN   the install summary shows all new mirror artifacts as `installed`:
       - .github/work-copilot/WORKFLOW.md
       - .github/work-copilot/reference/guide-*.md (7 files)
       - .github/work-copilot/philosophy/rationale-*.md (3 files)
       - .github/work-copilot/examples/example-*.md (14 files)
       - .github/work-copilot/fixtures/*.md (5+ files)
AND    no code change to scripts/copilot-deploy.py is required
       (verified in plan-eng-review D1: rglob("*") routes everything not in prompts/ or instructions/)
```

### Story #9: Doctor reports new artifacts [observability]

```
GIVEN  a target repo with work-copilot/ installed
WHEN   the maintainer runs `python scripts/copilot-deploy.py doctor <target>`
THEN   the doctor report enumerates each new mirror artifact under its bundle-relative path
AND    reports OK status for each artifact whose checksum matches the source
AND    reports DRIFT status for any artifact whose checksum differs (e.g., user-edited)
```

### Story #10: Budget enforcement location [observability]

```
GIVEN  the v2 realignment is in flight
WHEN   the maintainer authoring this PRD checks scripts/test.sh and scripts/validate.sh for an existing 8192-byte gate on copilot-instructions.md
THEN   one of the following is true:
       (a) an existing check is found, documented in S000010_ARCHITECTURE.md, and reused, OR
       (b) no existing check is found; S000010 adds a `wc -c` gate (≤8192) to scripts/validate.sh or scripts/test.sh
AND    the chosen location is documented in S000010_ARCHITECTURE.md
```

### Story #11: Bundle layout grouping [usability]

```
GIVEN  work-copilot/instructions/copilot-instructions.md after v2 updates
WHEN   the maintainer reads the file
THEN   a single section header (e.g., "## Bundle layout") groups the new artifact pointers
AND    the section lists each bundle-internal path with a one-line description
AND    no inlined content from the mirrored files appears in copilot-instructions.md (file-path references only)
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Mirror coverage | 100% of upstream artifacts byte-identical | `cmp -s` per file in `MIRROR_SPECS` (run by `scripts/validate.sh` Error check 10) |
| Fixture parity | 5/5 previously missing/drifted fixtures closed | `cmp -s` per fixture against upstream |
| Instructions budget | `copilot-instructions.md` ≤ 8192 bytes | `wc -c < work-copilot/instructions/copilot-instructions.md` in CI |
| Installer auto-pickup | 0 code changes to `scripts/copilot-deploy.py` | git diff of `scripts/copilot-deploy.py` over the S000010 PR is empty (or limited to comment/test changes) |

## Out of Scope

- Knowledge integration (`$AI_KNOWLEDGE_DIR`, two-tier surfacing,
  `bin/knowledge-helpers.sh`, `knowledge-doctor`) — explicitly deferred to a
  follow-up feature. See F000004_DESIGN.md v2 "Not in scope" for rationale.
- `bin/` mirroring — Copilot has no shell execution at prompt time, so any
  mirrored shell scripts would be dead weight. `bin/knowledge-helpers.sh`
  goes away when the knowledge-integration follow-up ships its
  Copilot-native redesign.
- Generating `copilot-instructions.md` from `WORKFLOW.md` automatically —
  source of truth stays in `work-copilot/instructions/`, copied on install
  (per Decision #6). Auto-generation is a possible follow-up if drift
  becomes real.
- Automated Copilot-citation E2E — the manual E2E checklist is sufficient
  for v2; an automated harness is a follow-up if work-box adoption proves
  the value.

## Assumptions

- Copilot in VS Code reads `.github/copilot-instructions.md` as ambient
  context for every chat (verified pre-v0.14.0; v1 of F000004 depends on
  this).
- Copilot will read additional bundle files when their paths are referenced
  in `copilot-instructions.md` and the user's question matches the
  reference. (To be verified in S000010 manual E2E; if Copilot proves
  reluctant to follow path references, fall back to inlining critical
  pointers — but the 8 KB budget makes full inlining infeasible.)
- `scripts/copilot-deploy.py` already routes new bundle artifacts via
  `bundle_dir.rglob("*")` without per-dir special cases (verified in
  plan-eng-review D1, 2026-04-26).
- The 8 KB budget on `copilot-instructions.md` is a Copilot-platform
  constraint, not a self-imposed convention. Treat it as fixed.
