## Problem Statement

The company follows a formal work item specification: structural validation rules
(contract.json), 5 tracker types with Jira/TFS integration, 8 doc types, reference
guides for AI doc generation, and philosophy docs explaining lifecycle rationale.
These templates now live in `templates/company-workflow/` within the skill. Without
enforcement, engineers follow them manually, work items drift from the spec, and
nobody notices until review time.

The company-workflow skill packages this entire spec as a standalone Claude Code skill.
Install it in any repo. No gstack. No /review, /ship, /docs check. It just works.

## Doc-Driven Development Workflow

The company-workflow skill enables a 3-step doc-driven development approach.
Documents are first-class artifacts — not afterthoughts. The validate command
enforces structural compliance at every step.

### Step 1: Generate Initial Docs

The engineer gives Claude the big picture. Claude generates the initial work item
documents using two inputs per template type:

- **Template** (`templates/company-workflow/*.md`) — the structural skeleton
- **Example** (`skills/company-workflow/examples/*.md`) — a filled-in instance showing tone, depth, and conventions

For each work item type, Claude reads the template for structure and the
corresponding example for content style:

- Feature tracker + milestones from `tracker-feature.md` / `doc-milestones.md`
- User story tracker + PRD + ARCHITECTURE + TEST-SPEC from their respective templates
- Task trackers + test plans for decomposed work

At this stage, run `company-workflow validate` to ensure the generated docs meet
the structural contract (required fields, section order, lifecycle phases).

### Step 2: Align the Big Picture

The engineer works on the docs — especially feature milestones, user story PRDs,
architecture decisions, and test specs — to align the big picture with reality:

- Refine acceptance criteria in trackers
- Flesh out PRD user stories and acceptance criteria
- Make architecture decisions and record tradeoffs
- Map test cases to requirements in TEST-SPEC
- Adjust milestones and dependency graphs

Run `company-workflow validate` iteratively during this step to ensure documents
stay compliant with the company format as you edit. File mode catches structural
violations; directory mode catches missing artifacts and frontmatter drift.

### Step 3: Implement and Iterate

Implementation follows the aligned docs. For each task:

1. Read the parent user story's PRD and ARCHITECTURE for context
2. Implement according to the architecture decisions
3. Run `company-workflow validate` on modified docs after updates
4. Update tracker: move through lifecycle phases, add journal entries
5. Verify against TEST-SPEC criteria

The validate command acts as a continuous compliance gate:
- After editing any doc: file mode catches structural drift
- After completing work: directory mode catches missing artifacts
- Before shipping: full validation confirms spec compliance

## Mental Model

```
company-workflow (standalone, zero external deps)
    |
    +-- validate <file>    contract.json structural rules -> exit 0/1
    +-- validate <dir>     artifact completeness per type -> [PASS]/[MISSING]/[DRIFT]
    |
    reads: contract.json, company-artifact-manifests.json, templates/company-workflow/,
           examples/ (AI reads during doc generation)
    writes: nothing (read-only)
    depends on: nothing external
```

One command, two modes. Pass a file for structural validation, pass a directory for
artifact completeness.

- **File mode**: checks a single file against contract.json (required fields, sections, order, 4-phase lifecycle). Exit 0/1.
- **Directory mode**: checks a work item directory for artifact completeness (are all required files for the type present? correct frontmatter? no unresolved placeholders?). Reports [PASS]/[MISSING]/[DRIFT].

## Lifecycle Boundary

The company spec uses a 4-phase lifecycle (Track, Implement, Review, Ship). The
workbench uses 3-phase. This is intentional divergence. The skill preserves the
company spec exactly, including `workflow_type`, `url`, verbose sub-gate checkboxes,
and the review type.

## Acceptance Criteria

### Stories #1-3: Validate [shipped]

```
GIVEN a company tracker exists
WHEN  I run company-workflow validate
THEN  exit 0 if all contract.json rules pass
AND   exit 1 with violations on stderr
AND   checks: required fields, section order, lifecycle phases, min checkboxes
```

### Stories #4-7: Directory Mode Validate

```
GIVEN a feature directory with tracker but no PRD
WHEN  I run company-workflow validate on that directory
THEN  output includes [MISSING] PRD.md

GIVEN an artifact missing workflow_type in frontmatter
WHEN  I run company-workflow validate on that directory
THEN  output includes [DRIFT] — missing required field

GIVEN one complete work item directory per type
WHEN  I run company-workflow validate on each directory
THEN  feature=5, defect=3, task=2, userstory=5, review=2 required
AND   complete items get [PASS] for all artifacts
```

### Stories #13-14: Examples

```
GIVEN the examples/ directory exists in the skill
WHEN  I list its contents
THEN  there is one example file per template (13 examples)
AND   each example has valid frontmatter matching its template's required fields
AND   each example contains realistic filled-in content (no {PLACEHOLDER} values)

GIVEN skills-deploy install has been run
WHEN  I check ~/.claude/skills/company-workflow/examples/
THEN  all 13 example files are present
```

## Delivery

The skill is designed to be installed on any machine via `skills-deploy install`.
Delivery copies the complete skill package to the target machine:

- `~/.claude/skills/company-workflow/` — SKILL.md, contract.json, manifests, reference/, philosophy/, examples/, fixtures/
- `~/.claude/templates/company-workflow/` — all 13 company templates

After install, the skill resolves paths via the 2-level fallback chain (repo root
first, then `~/.claude/`). No gstack or other dependencies required.

### Install on another machine

```bash
# From the workbench repo:
./scripts/skills-deploy.sh install

# Or copy manually:
cp -r skills/company-workflow/ ~/.claude/skills/company-workflow/
cp -r templates/company-workflow/ ~/.claude/templates/company-workflow/
```

Reference guides, philosophy docs, and examples are included in the skill
directory. Examples are read by AI during doc generation; reference and
philosophy docs are for human reference.

## Out of Scope

- Automated sync from company spec upstream
- Converting workbench work items to company format
- Scaffolding / create command (use workbench work-items.md rules for scaffolding)
- Hierarchy / placement rules (use /docs check for that)
- Recursive directory validation (validate each directory separately)
- Any gstack integration (analytics, /review, /ship)


## Assumptions

- Templates live in `templates/company-workflow/` within the skill repo (no external paths)
- skills-deploy handles subfolder templates (shipped in v0.4.0)
- The repo is the source of truth for template evolution
