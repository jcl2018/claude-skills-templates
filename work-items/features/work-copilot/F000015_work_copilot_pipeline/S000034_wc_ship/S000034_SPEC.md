## Problem Statement

Copilot users need a prompt that synthesizes a PR description from the work-item's tracker journal, `receipts.qa.ac_ids_covered`, and `receipts.implement.commits_since_scaffold`. The prompt prints the description to chat (and optionally to `PR-DESCRIPTION.md`), writes a `receipts.ship` block with `pr_opened: false`, and exits. The user opens the PR manually on GitHub and flips `pr_opened: true` afterwards.

## Mental Model

`/wc-ship` is a 5-step prompt: validate → read inputs → synthesize PR description → print/write artifact → write receipts.ship. It is the only receipt-writing prompt with a warn-and-write Working-Tree Rule.

## Requirements

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | Pre-flight /validate | Copilot user | the prompt to fail-fast on structural drift | I don't ship a broken work-item |

| 3 | core | PR description synthesis | Copilot user | the prompt to synthesize a coherent PR description from journal entries + receipts.qa.ac_ids_covered + receipts.implement.commits_since_scaffold | I get a clipboard-ready artifact |

| 6 | resilience | Working-Tree Rule (warn) | Copilot user | a warning if the working tree is dirty, NOT a hard-stop | I can still get the PR description for paste |

## Acceptance Criteria

### Story #1: pre-flight /validate [core]

```
GIVEN a work-item path
WHEN /wc-ship is invoked
THEN the prompt calls /validate first
 AND aborts on DRIFT/MISSING
```

### Story #2: input loading [core]

```
GIVEN a work-item with tracker, PRD/RCA, and (optionally) PR-DESCRIPTION.md template
WHEN /wc-ship reads inputs
THEN the prompt loads:
  user-story → tracker + PRD + (no PR-DESCRIPTION; user-stories ship via task children)
  defect → tracker + RCA + PR-DESCRIPTION.md template
  task → tracker + PR-DESCRIPTION.md template
```

### Story #3: PR description synthesis [core]

```
GIVEN inputs are loaded
WHEN /wc-ship synthesizes the description
THEN the output contains:
  - One-line summary derived from the tracker title
  - "What changed" section from receipts.implement.commits_since_scaffold (list of commit one-liners)
  - "ACs verified" section from receipts.qa.ac_ids_covered
  - "Open risks" section from receipts.implement.open_risks
  - "Tracker" link to the work-item path
```

### Story #4: chat print + file write [core]

```
GIVEN the description is synthesized
WHEN /wc-ship outputs
THEN the description is printed in chat (formatted for clipboard paste)
 AND written to <work-item-path>/PR-DESCRIPTION.md (unless user explicitly opts out)
```

### Story #5: receipts.ship write [core]

```
GIVEN the description is produced
WHEN /wc-ship writes the receipt
THEN receipts.ship contains:
  phase: 4
  completed_at: <ISO>
  pr_description_synthesized_from: [TRACKER.md, PRD.md or RCA.md, commits <sha_start>..<sha_end>]
  pr_url: null
  pr_opened: false
  next_legal: [merge]
```

### Story #6: Working-Tree Rule warn-and-write [resilience]

```
GIVEN /wc-ship is about to write receipts.ship
WHEN it prompts the user to paste `git status --porcelain -- <files_touched>`
THEN if the paste shows uncommitted entries the prompt PRINTS A WARNING
 AND PROCEEDS to write the receipt (does NOT abort)
 AND the warning includes "Note: PR description was synthesized from an unpushed working tree; verify before opening PR."
```

### Story #7: post-ship instructions [usability]

```
GIVEN /wc-ship has written receipts.ship
WHEN it prints the final summary
THEN the chat includes "After opening the PR on GitHub, edit this tracker's receipts.ship: flip pr_opened: true and fill pr_url with the PR URL."
```

### Story #8: YAML parse-failure resilience [resilience]

```
GIVEN tracker frontmatter is malformed
WHEN /wc-ship attempts to parse it
THEN the prompt aborts with "tracker frontmatter could not be parsed — fix manually before re-invoking"
```

### Data Flow

1. User invokes `/wc-ship <work-item-path>`.
2. Prompt calls `/validate`.
3. Prompt reads tracker (parses receipts.qa, receipts.implement), PRD/RCA, PR-DESCRIPTION.md template.
4. Prompt synthesizes description from receipts + journal.
5. Prompt prints description to chat.
6. Prompt writes `<work-item-path>/PR-DESCRIPTION.md` via `editFiles` (unless opted out).
7. Prompt asks for `git status --porcelain` paste; warns if dirty but proceeds.
8. Prompt reads-merges-writes `receipts.ship` block to tracker frontmatter.
9. Prompt prints post-ship instructions.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Working-Tree Rule severity | Warn-and-write | Hard-stop | Description is useful even unpushed; warning surfaces risk. |
| `pr_opened` source of truth | Manual user flip | Detect from receipts.ship.pr_url | URL can be set without flag flip; flag is unambiguous. |
| Artifact placement | `<work-item>/PR-DESCRIPTION.md` + chat | Chat-only | File serves as history artifact; opt-out flag for users who don't want it. |
