## Problem Statement

Copilot users need a read-only status compiler that reads all phase receipts from a work-item (or a design-doc) and `.git/HEAD`, computes drift math against the receipt schema, and prints a synthesized status block in a fixed format. The prompt has zero file writes; it is purely diagnostic. Output drives next-legal action selection.

## Mental Model

`/wc-pipeline` is a 4-step prompt: read receipts → read `.git/HEAD` (via codebase tool, no shell) → compute drift math (5 rules) → print status block. Two input modes: work-item path (full drift math) and design-doc path (frontmatter-state report). Read-only `tools:` array enforces read-only behavior at the harness level.

## Acceptance Criteria

### Story #1: read-only tools array [core]

```
GIVEN the prompt frontmatter is inspected
WHEN looking at tools:
THEN it contains exactly: codebase, search, searchResults
 AND does NOT contain: editFiles
```

### Story #2: two-mode input [core]

```
GIVEN /wc-pipeline <path>
WHEN path is a work-item directory
THEN the prompt reads <work-item>/<ID>_TRACKER.md and parses receipts
GIVEN path is a design-doc file (.md under .github/work-copilot/designs/)
WHEN /wc-pipeline runs
THEN the prompt parses receipts.investigate from the design-doc frontmatter
 AND reports DRAFT/APPROVED/SCAFFOLDED state + next legal
```

### Story #3: receipt read [core]

```
GIVEN a work-item tracker with one or more of receipts.{investigate,scaffold,implement,qa,ship}
WHEN /wc-pipeline runs
THEN each present receipt is parsed and its `completed_at` timestamp is captured
 AND each absent receipt is recorded as "Missing"
```

### Story #4: .git/HEAD read [core]

```
GIVEN the codebase tool can read .git/HEAD
WHEN /wc-pipeline executes the stale check
THEN the prompt reads .git/HEAD (path is "<repo-root>/.git/HEAD") via codebase tool
 AND extracts the SHA from the file (handles both "ref: refs/heads/<branch>" and direct SHA forms)
```

### Story #5: Missing rule [core]

```
GIVEN any of receipts.{scaffold,implement,qa,ship} is absent
WHEN /wc-pipeline prints output
THEN each absent receipt is shown as "? <phase> (not yet run)"
```

### Story #6: Stale (binary) rule [core]

```
GIVEN receipts.implement.latest_sha_at_implement is set
WHEN /wc-pipeline compares against current .git/HEAD SHA
THEN if SHAs string-equal: print "HEAD matches" — no drift
     if SHAs differ: print "HEAD has moved past receipts.implement.latest_sha_at_implement — code/doc desync possible; re-run /wc-implement"
 AND the printed output does NOT include a commit count (binary signal only)
```

### Story #7: Coverage holes rule [core]

```
GIVEN receipts.qa.ac_ids_uncovered is non-empty
WHEN /wc-pipeline prints output
THEN each entry is shown as "AC-N has no test row; re-enter /wc-implement or add row to TEST-SPEC."
```

### Story #8: Diff audit rule [core]

```
GIVEN receipts.qa.diff_audit.changed_files_without_tests is non-empty
WHEN /wc-pipeline prints output
THEN each entry is shown as "<file> changed since last [qa-*] entry — has no checklist coverage; /wc-qa re-entry recommended."
```

### Story #9: Ship-not-opened rule [core]

```
GIVEN receipts.ship exists with pr_opened == false AND completed_at is older than 24h from now
WHEN /wc-pipeline prints output
THEN it prints "/wc-ship printed a PR description but no PR was opened yet — re-run /wc-ship or open PR manually."
GIVEN pr_opened == true OR completed_at is within last 24h
WHEN /wc-pipeline runs
THEN this rule does NOT fire
```

### Story #10: Next-legal rule [core]

```
GIVEN all receipts read
WHEN /wc-pipeline computes next legal
THEN the output is union of all `next_legal` arrays minus phases that are already completed
 AND printed as "NEXT LEGAL: <comma-separated phase names>"
```

### Story #11: status block format [core]

```
WHEN /wc-pipeline prints
THEN the output matches this template:
  WORK-ITEM: <ID>_<slug>
  ──────────────────────────────────
  ✓ scaffold       (<completed_at>)
  ✓ implement      (<completed_at>) HEAD: <sha> → current: <sha> (MATCHES|MOVED)
  ✗ qa             AC-N uncovered; <file> changed without test
  ? ship           (not yet run)
  ──────────────────────────────────
  STALE: <stale-message OR "no drift">
         For exact count, run: git log <sha>..HEAD --oneline | wc -l
  COVERAGE: <coverage holes OR "none">
  DIFF AUDIT: <changed-without-tests OR "none">
  SHIP-NOT-OPENED: <message OR "n/a">
  NEXT LEGAL: <comma-separated phases>
```

### Story #12: review-type degenerate tolerance [core]

```
GIVEN a tracker with type: review and receipts.implement having empty files_touched, commits_since_scaffold, ac_ids_targeted
WHEN /wc-pipeline runs
THEN the receipt is treated as COMPLETE (✓), not drifted
 AND coverage/diff audit rules do NOT fire on the empty arrays
```

### Story #13: receipt schema mismatch [resilience]

```
GIVEN a receipt with an unrecognized required field shape (e.g., latest_sha_at_implement is not a 40-char hex)
WHEN /wc-pipeline parses
THEN it prints "receipt schema mismatch in receipts.<phase>; upgrade /wc-pipeline or fix manually"
 AND continues to print partial output for other phases
```

### Story #14: "For exact count" hint [usability]

```
GIVEN the stale rule fires
WHEN /wc-pipeline prints the stale message
THEN it includes "For exact count, run: git log <sha>..HEAD --oneline | wc -l" on a separate line
```

### Story #15: One-line summary [observability]

```
WHEN /wc-pipeline prints the status block
THEN the final line is "/wc-pipeline summary: <N>/<5> phases complete; <D> drift signals; next /wc-<phase>"
```

### Data Flow

1. User invokes `/wc-pipeline <path>` (work-item OR design-doc).
2. Prompt reads target file via `codebase` tool; parses frontmatter; determines mode.
3. For work-item: parses all 5 receipts.
4. For design-doc: parses only `receipts.investigate`.
5. Prompt reads `<repo-root>/.git/HEAD` via `codebase` tool; extracts SHA.
6. Prompt computes 5 drift rules.
7. Prompt prints status block.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Stale check granularity | Binary (HEAD matches OR moved) | Commit count via `git log` | Copilot has no shell; binary is the only available signal. Print the `git log` command as user-paste hint for those who want a count. |
| Mutation capability | None (read-only `tools:`) | Allow editFiles for self-healing fixes | Status compiler should be a printer; mutations belong to upstream phase commands. Read-only `tools:` enforces at the harness level. |
| Review-type detection | Empty arrays in receipts.implement → treat as complete | Always require non-empty arrays | Review type is degenerate by design (no code shipped, just notes); flagging it would create false positives. |
| Multi-receipt sequence | Overwrite-per-phase (V1 spec) | Append-only journal of receipts | V1 keeps the most recent receipt per phase; append-only is V2 if drift history surfaces. |
