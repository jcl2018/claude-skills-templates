## Problem Statement

Copilot users need a prompt that reads per-type input artifacts, walks through a code change proposal interactively, and writes a `receipts.implement` block. The receipt must be readable by `/wc-qa` (S000030) and `/wc-pipeline` (S000035) — its schema is locked by S000030.

## Mental Model

`/wc-implement` is a 6-step prompt: validate → per-type input read → walkthrough plan + edits → user-paste git data → Working-Tree Rule gate → receipt write. Per-type dispatch is the key complexity: 5 types, 5 input contracts, plus a degenerate "review" path.

## Requirements

### P0 (Must-Have)

| # | Tag | What it asks | As a... | I want to... | So that... |
|---|-----|-------------|---------|-------------|------------|
| 1 | core | What runs first? | Copilot user | call `/validate` to fail-fast on structural drift | I don't implement against a broken work-item |
| 2 | core | How does per-type dispatch work? | Copilot user | have the prompt read the tracker's `type:` field and load the right input artifacts | the right context lands in the conversation |
| 3 | core | What's the walkthrough cadence? | Copilot user | propose a plan in chat → user confirms → edit code → re-confirm | I review every code change before it lands |
| 4 | core | How are SHAs captured? | Copilot user | paste `git rev-parse HEAD` and `git log --oneline <scaffold_sha>..HEAD` when prompted | `receipts.implement` records the correct commit window |
| 5 | resilience | Working-Tree Rule? | Copilot user | paste `git status --porcelain -- <files_touched>` and have the prompt hard-stop on dirty entries | drift math stays honest |
| 6 | core | What's `receipts.implement`'s shape? | downstream prompt author | a documented schema conforming to S000030's contract — `phase: 2`, `completed_at`, `latest_sha_at_implement`, `commits_since_scaffold`, `files_touched`, `ac_ids_targeted`, `open_risks`, `next_legal: [qa]` | /wc-qa and /wc-pipeline can read it |
| 7 | core | How is the review-type degenerate path handled? | Copilot user | when `type: review`, write the degenerate receipt (empty arrays + 1-line `open_risks` summary) | review work-items don't get falsely flagged as drifted |
| 8 | core | Feature-type delegation? | Copilot user | when `type: feature`, see "pick a child user-story" prompt and delegate to that child | feature-level implements route correctly |

## Acceptance Criteria

### Story #1: pre-flight /validate gate [core]

```
GIVEN a work-item path
WHEN /wc-implement is invoked
THEN the prompt calls /validate first and aborts on DRIFT/MISSING
```

### Story #2: per-type input dispatch [core]

```
GIVEN a tracker with type: <one of feature|user-story|task|defect|review>
WHEN /wc-implement reads inputs
THEN the prompt loads:
  user-story → PRD.md + ARCHITECTURE.md + TEST-SPEC.md
  defect     → RCA.md + test-plan.md
  task       → TRACKER.md + test-plan.md
  feature    → feature-summary + DESIGN + milestones (then prompt to pick child)
  review     → review-notes (then walkthrough only; degenerate receipt)
```

### Story #3: walkthrough cadence [core]

```
GIVEN per-type inputs are loaded
WHEN /wc-implement proposes a change
THEN the prompt prints a plan in chat
 AND waits for user confirmation before any editFiles call
 AND re-confirms after each logical edit chunk
```

### Story #4: SHA capture [core]

```
GIVEN /wc-implement is about to write the receipt
WHEN it captures commit metadata
THEN the prompt asks the user to paste `git rev-parse HEAD` (captured as latest_sha_at_implement)
 AND asks the user to paste `git log --oneline <scaffold_sha>..HEAD` (captured as commits_since_scaffold)
```

### Story #5: Working-Tree Rule hard-stop [resilience]

```
GIVEN /wc-implement is about to write receipts.implement
WHEN it prompts the user to paste `git status --porcelain -- <files_touched>`
THEN if any uncommitted entries appear the prompt refuses to write
 AND prints "Please commit those files first and re-invoke /wc-implement; I'll wait."
```

### Story #6: receipt schema [core]

```
GIVEN /wc-implement completes
WHEN it writes the receipt
THEN receipts.implement contains:
  phase: 2
  completed_at: <ISO>
  latest_sha_at_implement: <40-char SHA>
  commits_since_scaffold: [<list of SHAs>]
  files_touched: [<list of paths>]
  ac_ids_targeted: [<list>]
  open_risks: [<list of one-line strings>]
  next_legal: [qa]
```

### Story #7: review-type degenerate receipt [core]

```
GIVEN tracker type: review
WHEN /wc-implement completes
THEN receipts.implement contains:
  files_touched: []
  commits_since_scaffold: []
  ac_ids_targeted: []
  open_risks: ["<one-line: what was reviewed and what action was taken>"]
  next_legal: [qa]
 AND /wc-pipeline tolerates these empty arrays as a valid completion state
```

### Story #8: feature-type delegation [core]

```
GIVEN tracker type: feature with N child user-stories
WHEN /wc-implement runs
THEN the prompt lists the children in chat
 AND asks the user to pick one
 AND re-invokes /wc-implement on the picked child path
 AND does NOT write receipts.implement on the feature tracker itself (children carry their own receipts)
```

### Story #9: edit chunking [usability]

```
GIVEN a multi-file change
WHEN /wc-implement makes edits
THEN each editFiles call covers one logical change (one file or one tightly-coupled multi-file change)
 AND the prompt confirms with the user between chunks
```

### Story #10: YAML parse-failure resilience [resilience]

```
GIVEN tracker frontmatter is malformed
WHEN /wc-implement attempts to parse it
THEN the prompt aborts with "tracker frontmatter could not be parsed — fix manually before re-invoking"
```

### Data Flow

1. User invokes `/wc-implement <work-item-path>`.
2. Prompt calls `/validate`.
3. Prompt reads tracker frontmatter, extracts `type:`.
4. Prompt loads per-type input artifacts via `codebase` tool.
5. Prompt enters walkthrough: propose → user confirm → editFiles → re-confirm.
6. Prompt asks for `git rev-parse HEAD` paste.
7. Prompt asks for `git log --oneline <scaffold_sha>..HEAD` paste.
8. Prompt asks for `git status --porcelain` paste (hard-stop gate).
9. Prompt reads whole tracker, merges receipts.implement, writes whole back.
10. Prompt prints summary.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Mode | Walkthrough only | Walkthrough + `--auto` | Copilot has no AUQ; --auto is risky without an explicit "review this proposed plan" gate. V2 candidate. |
| Per-type review degenerate path | Empty arrays + 1-line open_risks | Skip receipts.implement for review type | Skipping leaves /wc-pipeline thinking the phase is unrun; explicit empty receipt is honest. |
| Feature delegation | Pick child via chat, re-invoke on child | Auto-pick first incomplete child | User control is the V1 default; auto-pick is surprising. |
