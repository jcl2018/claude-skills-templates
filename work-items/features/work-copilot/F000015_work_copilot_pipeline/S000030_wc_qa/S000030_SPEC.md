## Problem Statement

Copilot users at the company need a structured walkthrough of QA-time checks for any work item — test-row checklist, AC coverage verification, diff audit since the last `[qa-*]` entry, Working-Tree Rule gate, and a receipt block that downstream prompts can read. As build #1 of Approach C, this prompt also **locks the receipt schema** for all downstream phase commands.

## Mental Model

`/wc-qa` is a 9-step prompt: validate → read test rows → cross-reference ACs → user-paste git log → working-tree gate → walk checklist → journal entries → receipt write → READY_FOR_SHIP gate. Receipts live in tracker YAML frontmatter under `receipts.qa`; the prompt uses the "read whole tracker, parse YAML, merge, write whole tracker" edit pattern.

## Acceptance Criteria

### Story #1: pre-flight /validate gate [core]

```
GIVEN a work-item path
WHEN /wc-qa is invoked
THEN the prompt instructs Copilot to call /validate <work-item-path> as the first step
 AND aborts with the /validate error message if /validate reports DRIFT or MISSING
```

### Story #2: test-row checklist [core]

```
GIVEN a valid work-item path containing a test-plan.md or TEST-SPEC.md
WHEN /wc-qa runs after /validate passes
THEN the prompt reads the right file (test-plan for defect/task, TEST-SPEC for user-story)
 AND prints each row as a numbered checklist entry
```

### Story #3: AC coverage cross-reference [core]

```
GIVEN PRD/SPEC/RCA in the work-item
WHEN /wc-qa builds the checklist
THEN the prompt extracts AC IDs (AC-1, AC-2, ...) from these docs
 AND flags any AC that has no corresponding test row as "AC-N uncovered"
```

### Story #4: diff-audit user-paste pattern [core]

```
GIVEN a work-item with at least one prior [qa-*] journal entry
WHEN /wc-qa reaches the diff-audit step
THEN the prompt asks the user to run `git log --name-only --since='<ISO timestamp>' --pretty=format:''` and paste the output
 AND parses the paste, cross-references files with test-plan rows, and flags changed-files-without-tests
```

### Story #5: first-run fallback to receipts.scaffold SHA [core]

```
GIVEN a work-item with no prior [qa-*] journal entry
WHEN /wc-qa reaches the diff-audit step
THEN the prompt reads `receipts.scaffold.completed_at` (and any associated SHA captured in receipts.scaffold)
 AND uses that as the diff-audit baseline instead of `<ISO timestamp>`
```

### Story #6: Working-Tree Rule hard-stop [resilience]

```
GIVEN /wc-qa is about to write receipts.qa
WHEN the prompt instructs the user to paste `git status --porcelain -- <files_touched>`
THEN if the paste shows any uncommitted entries the prompt refuses to write
 AND prints "Please commit those files first and re-invoke /wc-qa; I'll wait."
```

### Story #7: journal entries [core]

```
GIVEN /wc-qa has walked the checklist
WHEN each row's pass/fail/skip is captured
THEN the prompt appends `[smoke-pass]` or `[qa-fail: <reason>]` entries to the tracker's ## Journal section
 AND each entry includes the date and the AC IDs touched
```

### Story #8: receipt schema [core]

```
GIVEN /wc-qa completes successfully
WHEN it writes the receipt
THEN the receipt block under `receipts.qa` in tracker frontmatter contains:
   phase: 3
   completed_at: <ISO timestamp>
   test_rows_run: <int>
   ac_ids_covered: [<list>]
   ac_ids_uncovered: [<list>]
   diff_audit:
     changed_files_without_tests: [<list>]
   journal_entries: [<list of journal lines written>]
   ready_for_ship: <bool>
   next_legal: [<phase names>]
```

### Story #9: READY_FOR_SHIP gate [usability]

```
GIVEN /wc-qa has written receipts.qa
WHEN the prompt prints its final line
THEN it prints "READY_FOR_SHIP: yes" or "READY_FOR_SHIP: no" with blocker list if no
```

### Story #10: YAML parse-failure resilience [resilience]

```
GIVEN tracker frontmatter is malformed
WHEN /wc-qa attempts to parse it
THEN the prompt aborts with "tracker frontmatter could not be parsed — fix manually before re-invoking"
 AND does NOT attempt a partial-edit recovery
```

### Data Flow

1. User invokes `/wc-qa <work-item-path>` in Copilot Chat.
2. Prompt calls `/validate <work-item-path>` (existing prompt) — abort on DRIFT.
3. Prompt reads `test-plan.md` or `TEST-SPEC.md` via `codebase` tool.
4. Prompt reads PRD/SPEC/RCA via `codebase`; extracts AC IDs.
5. Prompt asks user to paste git log output; parses the paste.
6. Prompt asks user to paste git status output; hard-stops if uncommitted entries.
7. Prompt walks checklist in chat back-and-forth.
8. Prompt reads whole tracker file, parses YAML frontmatter, merges new `receipts.qa` block, writes whole tracker via `editFiles`.
9. Prompt prints READY_FOR_SHIP summary.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Receipt location | Tracker YAML frontmatter under `receipts.qa` | Separate `receipts.yaml` file per work-item | Frontmatter keeps receipts version-controlled with the tracker, diffable in PRs, discoverable without manifest. |
| First-run diff-audit baseline | `receipts.scaffold` SHA | `HEAD~N` | SHA is deterministic; N is fuzzy and depends on commit cadence. |
| YAML-edit approach | Read whole, parse, merge, write whole | Surgical line-level edit | Surgical edits from Copilot are unreliable; precedent: `validate.prompt.md`. |
| Working-Tree Rule severity | Hard-stop | Warn-and-write with `dirty: true` flag | /wc-qa writes receipts the orchestrator's drift math depends on; honesty beats permissiveness. |
