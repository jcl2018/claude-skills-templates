## Problem Statement

Copilot users at the company need a scoping prompt that conducts a structured conversation (problem, target user, narrowest wedge, key risks), synthesizes a design doc with the required YAML frontmatter, and writes a `receipts.investigate` block — without an AUQ tool and without shell access. The prompt also requires three per-target-repo domain knowledge files to anchor the conversation in real context; `copilot-deploy.py` must install skeletons for those files on first install and preserve user-filled content on re-install.

## Mental Model

`/wc-investigate` is a 5-step prompt: read domain context → grep codebase → walk scoping chat → synthesize design doc → write receipts.investigate. The output is a design doc at `.github/work-copilot/designs/<slug>-design-<datetime>.md` with required frontmatter (status, work_item_type, scaffolded_to: null, receipts.investigate). `/wc-scaffold` consumes this on its next invocation.

## Acceptance Criteria

### Story #1: domain context loading [core]

```
GIVEN .github/work-copilot/domain/ exists with N .md files and M .template.md files
WHEN /wc-investigate runs
THEN the prompt reads all N .md files (skipping the .template.md skeletons)
 AND uses them as ambient context for the conversation
```

### Story #2: codebase grep [core]

```
GIVEN the user mentions an entity (function name, file path, concept) in their initial prompt
WHEN /wc-investigate runs
THEN the prompt uses `search` and `searchResults` tools to find references in the codebase
 AND uses the findings as conversation context
```

### Story #3: scoping conversation [core]

```
GIVEN /wc-investigate has loaded context
WHEN it conducts the scoping chat
THEN the prompt asks (in plain chat): problem statement, target user, narrowest wedge, key risks
 AND waits for user replies between questions
```

### Story #4: design-doc synthesis [core]

```
GIVEN the scoping conversation is complete
WHEN /wc-investigate writes the design-doc
THEN the file path is .github/work-copilot/designs/<slug>-design-<datetime>.md
 AND the body contains sections: Problem Statement, Approaches Considered, Recommended Approach, Open Questions, Success Criteria
```

### Story #5: design-doc frontmatter [core]

```
GIVEN the design-doc is written
WHEN its frontmatter is inspected
THEN it contains:
  title: "<topic>"
  generated_by: /wc-investigate
  generated_at: <ISO>
  status: DRAFT
  work_item_type: <feature|user-story|task|defect|review>
  scaffolded_to: null
  receipts:
    investigate:
      phase: 0
      completed_at: <ISO>
      design_doc: <relative path to this file>
      inputs_read: [<list of domain files + codebase paths>]
      outputs:
        proposed_type: <type>
        scope_summary: "<one-line>"
      next_legal: [scaffold]
```

### Story #6: domain skeleton templates ship [core]

```
GIVEN the bundle is checked into the workbench
WHEN inspecting work-copilot/domain/
THEN files exist:
  domain-knowledge.template.md
  coding-conventions.template.md
  architecture-overview.template.md
 AND each contains a commented placeholder ("# Replace with...") plus a short example
```

### Story #7: first-install rule [core]

```
GIVEN `python3 scripts/copilot-deploy.py install <target-repo>`
WHEN the target's .github/work-copilot/domain/<name>.md already exists
THEN copilot-deploy.py prints "[KEEP-USER] <name>.md" and skips
 AND if .github/work-copilot/domain/<name>.md does NOT exist
THEN copilot-deploy.py writes the skeleton (stripping .template.md → .md)
```

### Story #8: designs/.gitkeep [core]

```
GIVEN `python3 scripts/copilot-deploy.py install <target-repo>`
WHEN target's .github/work-copilot/designs/ does not exist
THEN copilot-deploy.py creates the folder and writes a zero-byte `.gitkeep`
 AND if the folder already exists
THEN it skips (idempotent)
```

### Story #9: slug derivation [usability]

```
GIVEN the conversation's topic
WHEN /wc-investigate derives a slug
THEN slug is lowercase, [a-z0-9_-]+, no spaces or capitals, max 40 chars
```

### Story #10: codebase-search resilience [resilience]

```
GIVEN the user's prompt mentions an entity not found in the codebase
WHEN search returns no results
THEN the prompt prints "no codebase matches for <entity> — proceeding with domain context only"
 AND continues the conversation
```

### Story #11: resume in-progress investigation [usability]

```
GIVEN a prior /wc-investigate run wrote a design-doc with status: DRAFT but the conversation was cut short
WHEN /wc-investigate is re-invoked on the same topic
THEN the prompt offers to resume from the existing draft
 OR start fresh
```

### Data Flow

1. User runs `python3 scripts/copilot-deploy.py install <target-repo>` once.
2. copilot-deploy.py writes the 3 domain skeletons (stripping `.template.md`) IF target file does not exist; otherwise skips with `[KEEP-USER]`.
3. copilot-deploy.py creates `<target>/.github/work-copilot/designs/.gitkeep` (idempotent).
4. User fills `.github/work-copilot/domain/<name>.md` files manually.
5. User runs `/wc-investigate <topic>` in Copilot Chat at the target repo.
6. Prompt reads domain files (skipping `.template.md`), greps codebase, walks conversation.
7. Prompt writes design-doc with frontmatter + receipts.investigate.
8. `/wc-scaffold` later consumes the design-doc.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Skeleton format | `.template.md` suffix; install strips suffix | Manifest-tracked install state | Suffix signal is simple and discoverable; manifest adds bookkeeping. |
| Domain file count | 3 (knowledge/conventions/architecture) | 1 (combined) or 5+ (over-decomposed) | 3 matches the natural problem split for scoping conversations. |
| Scoping format | Free-form chat with 4 forcing questions | Structured AUQ-style prompts | Copilot has no AUQ; documented forcing questions keep the conversation on script without runtime AUQ. |
