## Problem Statement

Copilot users need a scaffold prompt that takes a `/wc-investigate`-produced design doc (or a hand-authored stub with the right frontmatter) and produces a compliant work-item directory tree with all required artifacts, a valid `receipts.scaffold` block, and an updated design-doc lineage marker. The prompt must be idempotent and enforce the design-doc-required invariant.

## Mental Model

`/wc-scaffold` is an 8-step prompt: read design-doc frontmatter (idempotency + invariant check) → read manifest+templates → pick ID → write tree → /validate → copy receipts.investigate → write receipts.scaffold (pending_commit: true) → update design-doc status. The work-item directory becomes the visible state machine's root; `receipts.scaffold` becomes the lineage root that downstream prompts read.

## Requirements

## Acceptance Criteria

### Story #1: idempotency NO-OP [core]

```
GIVEN a design-doc with status: SCAFFOLDED and scaffolded_to: <path> pointing to existing dir
WHEN /wc-scaffold is invoked
THEN the prompt prints "Already scaffolded at <path>; nothing to do."
 AND exits without writing
```

### Story #2: design-doc-required invariant [core]

```
GIVEN a design-doc that lacks status, work_item_type, OR receipts.investigate
WHEN /wc-scaffold is invoked
THEN the prompt aborts with "design doc is missing required frontmatter (status, work_item_type, receipts.investigate); hand-author or re-run /wc-investigate."
```

### Story #3: ID picker [core]

```
GIVEN existing work-items/ with highest F-prefix at F000NNN
WHEN /wc-scaffold runs against a feature design
THEN it picks F000(NNN+1) (zero-padded) as NEW_ID
```

### Story #4: per-type tree write [core]

```
GIVEN a design-doc with work_item_type: <one of feature|user-story|task|defect|review>
WHEN /wc-scaffold writes the directory tree
THEN it creates the right artifacts per the manifest:
  feature → TRACKER + DESIGN + ROADMAP
  user-story → TRACKER + DESIGN + SPEC + TEST-SPEC
  defect → TRACKER + RCA + test-plan
  task → TRACKER + test-plan
  review → TRACKER + review-notes
```

### Story #5: /validate gate at end [core]

```
GIVEN the directory tree is written
WHEN /wc-scaffold calls /validate <new-dir>
THEN if /validate reports DRIFT/MISSING the prompt prints the violations and exits
 AND does NOT mark the design-doc SCAFFOLDED
```

### Story #6: lineage copy [core]

```
GIVEN a design-doc with receipts.investigate
WHEN /wc-scaffold writes the new tracker
THEN the new tracker's frontmatter contains receipts.investigate copied from the design-doc
```

### Story #7: receipts.scaffold write [core]

```
GIVEN scaffolding completed and /validate passed
WHEN the prompt writes the receipt
THEN receipts.scaffold contains:
  phase: 1
  completed_at: <ISO>
  work_item_id: <NEW_ID>
  work_item_dir: <path>
  artifacts_written: [<list>]
  validate_result: PASS
  pending_commit: true
  next_legal: [implement]
```

### Story #8: design-doc update [core]

```
GIVEN scaffolding succeeded
WHEN the prompt updates the source design-doc
THEN design-doc frontmatter status: SCAFFOLDED
 AND scaffolded_to: <work-item-dir>
 AND no other frontmatter fields are modified
```

### Story #9: YAML parse-failure resilience [resilience]

```
GIVEN design-doc frontmatter is malformed
WHEN /wc-scaffold attempts to parse
THEN the prompt aborts with "design-doc frontmatter could not be parsed — fix manually before re-invoking"
```

### Data Flow

1. User invokes `/wc-scaffold <design-doc-path>`.
2. Prompt reads design-doc via `codebase` tool; parses frontmatter; checks idempotency + invariant.
3. Prompt reads `.github/work-copilot/copilot-artifact-manifests.json` + templates via `codebase`.
4. Prompt greps `work-items/` for highest ID per type; picks next.
5. Prompt writes directory + artifacts via `editFiles`.
6. Prompt calls `/validate <new-dir>`.
7. Prompt copies `receipts.investigate` into new tracker (read-whole-parse-merge-write).
8. Prompt writes `receipts.scaffold` block with `pending_commit: true`.
9. Prompt updates design-doc frontmatter: `status: SCAFFOLDED`, `scaffolded_to: <path>`.

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Idempotency marker | YAML frontmatter (`status:`, `scaffolded_to:`) | Footer line | Frontmatter is structured; footer is brittle to manual edits. |
| Design-doc-required | Hard invariant | Allow hand-prompt scaffold without design | Without a design-doc root, /wc-pipeline can't chain drift math. Hand-authored stubs allowed with required frontmatter — that's the escape hatch. |
| ID picker | grep `work-items/` | grep `work-items/` + `gh pr list` | Copilot has no shell; `gh pr list` requires user-paste which adds friction. V1 punts to local-only; V2 candidate. |
