## Problem Statement

Even with the `/validate` prompt installed (S000007), Copilot has no
context about _how_ the engineer works on work items: the 3-phase
lifecycle, the ID scheme, the hierarchy rules, where to scaffold new
items. Without an always-on context file, the engineer has to re-explain
conventions every session, and Copilot invents directory layouts or
template structures that don't match the spec. A `copilot-instructions.md`
makes the conventions ambient: Copilot knows them by default, and the
engineer can focus on the work itself.

## Mental Model

Three surfaces, different concerns:

```
always-on                         on-demand                     reference (read by prompt)
---------                         ---------                     -------------------------
copilot-instructions.md           validate.prompt.md            copilot-artifact-manifests.json
(S000009 — this story)            (S000007)                     templates/
                                                                reference/

"Here is HOW we work."            "Check THIS item."            "Structural rules and examples."
```

## Acceptance Criteria

### Story #1: Ambient workflow awareness [core]

```
GIVEN a target repo with `.github/copilot-instructions.md` installed
WHEN  I ask Copilot "how do I add a user story under F000004?"
THEN  it proposes `work-items/features/F000004_*/S0000NN_*/` with correct ID increment
AND   it mentions required artifacts (TRACKER, PRD, ARCHITECTURE, TEST-SPEC)
```

### Story #2: Scaffolding guidance [core]

```
GIVEN the instructions file includes directory layout rules
WHEN  I ask "where does a defect live?"
THEN  Copilot answers `work-items/defects/D000NNN_{slug}/` with TRACKER + RCA + test-plan
```

### Story #3: Point to the validator [core]

```
GIVEN an engineer asks "is my work item compliant?"
WHEN  Copilot answers
THEN  it instructs them to run `/validate <path>` rather than hand-checking
```

### Story #4: Stay within budget [core]

```
GIVEN the final instructions file
WHEN  I measure its byte size
THEN  size <= 8192 bytes (8 KB)
AND   no single section exceeds 1 KB
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| File size | <= 8 KB | `wc -c copilot-instructions.md` |
| Convention accuracy | 100% match against `WORKFLOW.md` | Manual spot-check of every claim |
| User trust | subjective | engineer reports "Copilot now proposes the right path first try" |

## Out of Scope

- Embedding the manifest inline (too large; already on disk)
- Teaching Copilot to author full trackers (that's the `/validate` prompt's cousin, not this story)
- Per-repo customization (single canonical instructions file; repos inherit as-is)

## Assumptions

- Copilot respects `.github/copilot-instructions.md` on the target work machine
- The user's workflow won't diverge significantly between the Claude Code
  workbench and the work machine; if it does, we fork the file.
