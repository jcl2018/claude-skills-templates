## Problem Statement

The engineer uses `/company-workflow check` daily on a Claude Code machine to
validate work items against a template-driven spec. On their Windows work
machine they use GitHub Copilot, which has no equivalent slash command and no
way to run the existing SKILL.md. Today they either skip validation or
manually cross-check trackers against the spec — both lose signal. The
validator has to be expressible as a Copilot `.prompt.md` file so Copilot
chat can run the same structural/manifest checks without any Anthropic
tooling.

## Mental Model

```
Claude Code side                      Copilot side (this story)
-----------------                     -------------------------
/company-workflow check  <---port---> /validate (prompt file)
  reads SKILL.md                        reads validate.prompt.md
  reads manifest.json                   reads manifest.json (same format)
  reads templates/*.md                  reads templates/*.md (same files)
  emits [PASS]/[MISSING]/[DRIFT]        emits [PASS]/[MISSING]/[DRIFT]
```

The prompt file is the Copilot-native analog of a SKILL.md. Same inputs,
same output contract, different runtime.

## Acceptance Criteria

### Story #1: Run validate from Copilot chat [core]

```
GIVEN work-copilot is installed in my repo's `.github/prompts/`
WHEN  I type `/validate work-items/features/F000004_work_copilot/` in Copilot chat
THEN  Copilot reads the prompt, reads the manifest, reads the trackers
AND   emits one line per required artifact with [PASS]/[MISSING]/[DRIFT]
```

### Story #2: Same manifest-driven rules [core]

```
GIVEN `copilot-artifact-manifests.json` describes `feature` as requiring TRACKER + milestones
WHEN  I validate a feature directory missing milestones.md
THEN  output includes `[MISSING] milestones.md`
```

### Story #3: Match output contract [core]

```
GIVEN a tracker with a required frontmatter field absent
WHEN  I validate it
THEN  output includes `[DRIFT] TRACKER.md: missing required field <field>`
AND   the line shape matches company-workflow's output exactly
```

### Story #4: Work in file and directory modes [core]

```
GIVEN the prompt is passed a single tracker path
WHEN  validated
THEN  file mode runs (structural rules only, no manifest check)

GIVEN the prompt is passed a directory path
WHEN  validated
THEN  directory mode runs (manifest + each artifact's structural rules)
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Runtime parity | 100% of manifest checks run identically | Run both `/company-workflow check` and `/validate` on the same item; diff the output |
| Cold-start usability | < 30s from `/validate` to first result | Wall clock in Copilot chat |
| Cross-repo reuse | Works in 2 repos without edits | Install in claude-skills-templates and 1 work repo |

## Out of Scope

- Writing or fixing work items (read-only like the company skill)
- Any Bash/shell execution (Copilot prompts can't run shell at prompt time)
- Template discovery beyond the bundled `.github/prompts/templates/`
- Parity with `/personal-workflow check` (different spec, different story)

## Assumptions

- GitHub Copilot supports `.prompt.md` files in `.github/prompts/` on the
  user's work machine version of VS Code
- Copilot can read files from the workspace when the prompt instructs it to
- The manifest format from company-workflow is reusable as-is (JSON schema
  hasn't drifted)

## Overview

Port the company-workflow validator into a Copilot-native prompt file. The
validator is already expressed as prose + checklists in
`skills/company-workflow/SKILL.md`; we translate it into a `.prompt.md` that
Copilot loads when the user types `/validate`. The manifest
(`copilot-artifact-manifests.json`) and the templates directory are
delivered alongside the prompt and are read by Copilot at prompt time via
the workspace file API.

## Architecture

```
.github/                              (inside the target repo)
  copilot-instructions.md             always-on context (S000009)
  prompts/
    validate.prompt.md                THIS STORY — the slash command
  work-copilot/
    copilot-artifact-manifests.json   same schema as company manifest
    templates/                        mirrored from templates/company-workflow/
    reference/                        optional guides for the model

Copilot chat flow:
  user types /validate <path>
    -> Copilot loads validate.prompt.md
    -> prompt instructs model: "read manifest, read templates, read <path>"
    -> model uses Copilot's file tool to read workspace files
    -> model emits [PASS]/[MISSING]/[DRIFT] lines + summary footer
```

### Components Affected

| Component | Repo | Change Type | Description |
|-----------|------|------------|-------------|
| `work-copilot/prompts/validate.prompt.md` | claude-skills-templates | New | The slash-command prompt |
| `work-copilot/copilot-artifact-manifests.json` | claude-skills-templates | New | Copy of company manifest, renamed for namespace clarity |
| `work-copilot/templates/` | claude-skills-templates | New | Mirror of `templates/company-workflow/` |

### Data Flow

1. User invokes `/validate <path>` in Copilot chat
2. Copilot injects `validate.prompt.md` as the system/user prompt
3. The prompt tells the model to read `copilot-artifact-manifests.json`
4. Model resolves work-item type from the tracker's `type:` frontmatter
5. Model reads each required artifact's template to derive structural rules
6. Model reads each required artifact in the target directory
7. Model emits one status line per artifact + a summary footer

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Copilot model hallucinates manifest contents instead of reading the file | Med | High | Prompt explicitly instructs "read the file, do not recall" and names the path |
| `.prompt.md` format changes in a future Copilot release | Low | Med | Pin documentation link, add a smoke test in S000008's installer |
| Output drift between Claude Code and Copilot runtimes | Med | Med | Ship fixtures in the bundle; manually diff outputs before declaring parity |
| Prompt too long for Copilot context window | Low | Med | Keep prompt body under 2 KB; link to reference guides instead of embedding |

## Tradeoffs

| Decision | Chosen | Rejected Alternative | Why |
|----------|--------|---------------------|-----|
| Runtime surface | `.prompt.md` in `.github/prompts/` | `.chatmode.md` | Prompts are slash-invokable per-message, matching Claude slash-command UX |
| Manifest schema | Reuse company schema 1:1 | Invent a Copilot-specific schema | One spec, two runtimes — drift is the enemy |
| Output format | Identical `[PASS]`/`[MISSING]`/`[DRIFT]` | Copilot-native rich cards | Grep-ability + diff parity with company-workflow |
| Shell execution | None (pure prompt) | Node.js helper in `.github/scripts/` | Copilot prompts can't shell out; adding a script doubles the install surface |
| File reads | Model uses Copilot's file tool | Embed templates inline in the prompt | Inline would bloat context and couple the prompt to template versions |
