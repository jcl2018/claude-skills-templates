## Problem Statement

The work-copilot bundle (prompts, manifest, templates) is useless until it
lands inside the target repo's `.github/` directory where Copilot looks for
it. `skills-deploy` today installs into `~/.claude/`, which Copilot doesn't
read. We also hit D000005 with CRLF + jq on Windows, so the installer must
be cross-platform. This story covers: a new installer entry point,
idempotent install + drift detection, a doctor check, and a repo-level
validate.sh rule that keeps the bundle's templates in sync with
`templates/company-workflow/`.

## Mental Model

```
source of truth                       delivered into target repo
---------------                       --------------------------
templates/company-workflow/  --copy--> work-copilot/templates/
skills/company-workflow/SKILL.md (logic) --> work-copilot/prompts/validate.prompt.md (S000007)
                                       --install--> <target-repo>/.github/work-copilot/
                                       --install--> <target-repo>/.github/prompts/validate.prompt.md

     (repo-local, so Copilot Chat sees it per-workspace)
```

## Acceptance Criteria

### Story #1: Install into `.github/` [core]

```
GIVEN a target repo with no `.github/work-copilot/`
WHEN  I run the installer pointing at the repo
THEN  `.github/work-copilot/` contains the manifest, templates, and reference
AND   `.github/prompts/validate.prompt.md` exists
```

### Story #2: Works on Windows [core]

```
GIVEN a Windows 11 machine with Python 3.10+ and Git for Windows
WHEN  I run the installer
THEN  it completes without invoking bash, jq, grep, or awk
AND   file contents are not corrupted by CRLF conversion
```

### Story #3: Idempotent re-install [core]

```
GIVEN an install has already run
WHEN  I run it again
THEN  files with matching checksums are skipped
AND   no duplicate files are created
AND   exit status is 0
```

### Story #4: Drift detection [core]

```
GIVEN an installed file has been hand-edited
WHEN  I run install without `--overwrite`
THEN  the installer refuses, prints `[DRIFT] <path>`, and exits non-zero
AND   with `--overwrite`, the file is replaced and a warning is printed
```

### Story #5: Doctor report [core]

```
GIVEN a partially-installed bundle
WHEN  I run `copilot-deploy doctor <target-repo>`
THEN  output lists `[MISSING]`, `[DRIFT]`, `[ORPHAN]` entries
AND   exit status reflects health (0 clean, 1 issues)
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Install time on Windows | < 10s | Wall clock on the actual work machine |
| Cross-platform smoke pass | 100% (macOS + Windows) | CI matrix in GitHub Actions |
| Drift false positive rate | 0% | Hand-check after running install + no edits |

## Out of Scope

- Uninstall (deferred — unlikely to be needed for now)
- Pushing the bundle via GitHub API (manual clone + install is fine)
- Rolling out to multiple repos in one command (loop externally)
- Upstream sync from upstream company spec (F000003 already punts on this)

## Assumptions

- Target repos allow `.github/work-copilot/` and `.github/prompts/`
  (standard Copilot locations; no org policy blocks them)
- Python 3.10+ is available on the user's Windows work machine (verified
  before committing to Python)
- The user has write access to the target repo's working tree
