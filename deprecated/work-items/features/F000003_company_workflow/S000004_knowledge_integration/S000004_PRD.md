## Problem Statement

An engineer using the `company-workflow` skill wants Claude to consult their house coding guidance (e.g. cpp style) and company-specific domain knowledge while working on work items. Today, no such discovery mechanism exists: the engineer has to copy-paste relevant docs into every prompt, or watch Claude make guesses that contradict house conventions. The knowledge already exists in their filesystem — the skill just can't find it.

This story delivers the full knowledge-integration feature in three layers:

- **Layer 1 — Env-var resolution:** the skill discovers the knowledge folder via `$AI_KNOWLEDGE_DIR` and warns when unset / invalid. No content is loaded yet.
- **Layer 2 — Always-on loading:** guidance the engineer wants Claude to apply *by default* — coding style, team conventions, house patterns. Categories marked `surface: always` are pulled into Claude's context on every invocation.
- **Layer 3 — On-demand matching:** domain deep-dives, runbooks, language references, team-specific jargon — only useful when the user is asking about them. Categories declare `surface: on-demand` with a `triggers` list; when the prompt mentions any trigger, Claude loads that category.

All three layers share the category tree, the `.knowledge.yml` parser, and the test fixture builder. The split is for shipping convenience (Layer 1 in PR #38; Layers 2 + 3 in PRs #40 + #41), not architectural separation.

## Mental Model

**The only enforced contract is the two `surface` modes — `always` and `on-demand`.** Everything else (category names, nesting depth, file naming) is user-defined. The diagram below uses `coding/`, `runbooks/`, `notes/` for illustration; the skill discovers whatever top-level subdirs exist at runtime via `list_categories()`.

```
$_KNOWLEDGE_DIR (resolved by Layer 1 from $AI_KNOWLEDGE_DIR)
├── <any-name>/                   ← category (user picks the name)
│   ├── .knowledge.yml { surface: always, triggers: [] }
│   ├── <any-subdir>/
│   │   └── <any>.md              ← loaded EVERY invocation (always-on)
│   └── <any>.md                  ← loaded EVERY invocation (always-on)
├── <any-name>/                   ← category
│   ├── .knowledge.yml { surface: on-demand, triggers: ["pricing engine", PE] }
│   └── <any>.md                  ← loaded ONLY if prompt matches a trigger
└── <any-name>/                   ← category, no .knowledge.yml
                                  ← treated as on-demand + empty triggers (never loads)
```

Concrete example (illustrative; not enforced):

```
$_KNOWLEDGE_DIR/
├── coding/        .knowledge.yml { surface: always }       ← cpp/errors.md, style.md
├── runbooks/      .knowledge.yml { surface: on-demand,
│                                   triggers: ["pricing engine", PE] }
│                                                            ← pricing.md (loads on match)
└── notes/                                                   ← no yml; inert
```

Per-invocation flow:
1. **Layer 1**: resolve `$AI_KNOWLEDGE_DIR` → `$_KNOWLEDGE_DIR` (or empty + warning).
2. **Layer 2**: enumerate categories. For each `surface: always`, emit `## Always-On Knowledge` block listing absolute paths; instruct Claude to Read them all.
3. **Layer 3**: for each `surface: on-demand` with non-empty triggers, emit `## On-Demand Knowledge Candidates` block (root + triggers + file paths); instruct Claude to tokenize the latest user message and Read paths from any matched category.

Bash handles discovery + file enumeration (deterministic, testable). Claude handles prompt matching for on-demand (the prompt is only visible to Claude). One shared `.knowledge.yml` parser, one shared file enumerator, one shared malformed-yml warning path.

| 4 | always-on / core | Does the skill automatically include my coding guidance in every invocation? | engineer who configured a `surface: always` category | have Claude read my house style without my asking | style violations drop without friction |

| 10 | on-demand / resilience | Does absence of any trigger keyword leave on-demand categories alone? | engineer asking an unrelated question | have Claude NOT load domain knowledge my prompt didn't invoke | context stays focused |
| 11 | on-demand / core | Does matching multiple categories load all of them? | engineer whose prompt mentions two domains | pull in both runbook and security guides | I get the union, not an arbitrary pick |

## Acceptance Criteria

### Story #1: env var resolution [core]

```
GIVEN the engineer has exported AI_KNOWLEDGE_DIR=/path/to/knowledge
WHEN  the company-workflow skill is invoked
THEN  the skill captures the resolved path in a skill-internal variable and proceeds to run the requested command
```

### Story #2: warn when unset [usability]

```
GIVEN AI_KNOWLEDGE_DIR is unset or empty
WHEN  any company-workflow skill command runs
THEN  the skill emits exactly one warning line on stderr naming the variable and pointing to setup docs
AND   exit code is unchanged (0 on success)
```

### Story #3: warn when path invalid [resilience]

```
GIVEN AI_KNOWLEDGE_DIR is set to a path that does not exist or is not a directory
WHEN  any company-workflow skill command runs
THEN  the skill emits a warning mentioning the configured path and the reason
AND   exit code is unchanged (0 on success)
```

### Stories #4 + #5: always-on loading with nesting [core]

```
GIVEN $_KNOWLEDGE_DIR = /tmp/k
AND   /tmp/k/coding/.knowledge.yml contains "surface: always"
AND   /tmp/k/coding/ contains a.md and cpp/b.md
WHEN  the skill runs any command
THEN  Claude's context includes the contents of a.md and cpp/b.md
AND   load order is deterministic (a.md before cpp/b.md, lexical path order)
```

### Story #6: malformed yml resilience [resilience]

```
GIVEN two categories (one always-on, one on-demand), each with `.knowledge.yml`
AND   one of the yml files is broken
WHEN  the skill runs
THEN  the valid category loads (always-on) or emits as candidate (on-demand)
AND   a single warning names the broken yml file
AND   exit code is unchanged
```

### Story #7: missing yml = silent skip [core]

```
GIVEN a category directory with no .knowledge.yml
WHEN  the skill runs
THEN  no files from that category are loaded
AND   the category is not emitted as an on-demand candidate
AND   no warning is emitted (missing yml is not an error)
```

### Story #8: single-word on-demand trigger match [core]

```
GIVEN category /k/runbooks/ with .knowledge.yml { surface: on-demand, triggers: [pricing] }
AND   /k/runbooks/overview.md contains unique canary CANARY_RB_1
WHEN  user asks "explain the pricing flow"
THEN  Claude Reads /k/runbooks/overview.md before answering
AND   its reply can cite CANARY_RB_1
```

### Story #9: multi-word phrase trigger [core]

```
GIVEN category with triggers: ["pricing engine"]
AND   a file under the category
WHEN  user asks "how does the pricing engine handle rounding?"
THEN  the category is loaded (phrase match)

AND
GIVEN same category
WHEN  user asks "what is pricing?"
THEN  the category is NOT loaded (the word "pricing" alone is not the phrase "pricing engine")
```

### Story #10: no match means no loading [resilience]

```
GIVEN any number of on-demand categories with any triggers
WHEN  the user's prompt contains none of those triggers
THEN  none of those categories' files are Read
AND   the request is answered from Claude's baseline context + always-on knowledge only
```

### Story #11: multi-match loads all [core]

```
GIVEN two on-demand categories: runbooks (triggers: [pricing]) and security (triggers: [auth])
WHEN  user asks about pricing auth
THEN  files from BOTH categories are Read
```

### Story #12: diagnostic visibility [observability]

```
GIVEN the engineer wants to verify knowledge folder detection
WHEN  they run the knowledge-doctor diagnostic
THEN  the resolved knowledge dir path (or "unset"), per-category surface mode, and trigger lists appear in the output
```

### Story #16: case-insensitivity [usability]

```
GIVEN trigger declared as "cpp"
WHEN  user writes "CPP", "Cpp", or "cpp"
THEN  the category is loaded in all three cases
```

## Success Metrics

| Metric | Target | How Measured |
|--------|--------|-------------|
| Warning visibility when unset | 100% of skill invocations with unset var emit the warning | Tier 2 E2E tests: invoke skill with env unset, assert warning on stderr |
| Always-on category content reaches Claude | 100% of declared always-on files | E2E test: inject unique string into a knowledge file, verify Claude can quote it back |
| Trigger hit rate (intended on-demand matches found) | ≥ 95% across author-specified test cases | Tier 2 E2E canary tests |
| False positive rate (unwanted on-demand matches) | < 5% | Tier 2 suite: prompts that should NOT match |
| Zero regression | `validate` output byte-identical w/ and w/o categories | Diff on existing fixtures |
| One-typo blast radius | A malformed yml in category A does not block category B | Integration test: mix valid + malformed |

## Out of Scope

- Editing or authoring `.knowledge.yml` (user responsibility)
- Hot-reload when knowledge files change mid-session (each invocation reads fresh)
- Cross-category dependencies (a file in A requires a file in B) — categories are independent
- Format validation for the *markdown* content (only yml is validated)
- Fuzzy / semantic matching for on-demand (embeddings, stemming, synonyms) — explicit triggers only
- Priority/scoring between matched on-demand categories (P2 Story #19)
- Per-file ignore syntax (P2 Story #18)
- Matching against prior conversation turns — latest user message only
- Multi-source resolution (env var + per-repo overlay) — deferred follow-up
- Per-repo opt-in marker — dropped from desired design (2026-04-25)

## Assumptions

- The engineer's shell environment is inherited by the skill's bash blocks (standard Claude Code skill behavior)
- Stderr is visible to the user in their Claude Code session
- The engineer's knowledge folder has ≤ ~20 top-level categories (no pagination needed)
- Claude can reliably tokenize the latest user message and match against a small keyword list (≤20 triggers per category × ≤20 categories)
- The total size of always-on content is the user's responsibility (they marked it always-on); skill provides observability + caps but does not enforce a low ceiling
- Cross-context isolation is the user's responsibility — control which categories are `surface: always`, which carry triggers, or unset `$AI_KNOWLEDGE_DIR` per shell
