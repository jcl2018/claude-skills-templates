## Symptom

Two drift symptoms between `/company-workflow`'s contract.json and its own templates. Both surface every time the skill is deployed and a fresh tracker is scaffolded.

1. **Issue 1 — `workflow_type` frontmatter drift.** Templates emit `workflow_type` and `url` in YAML frontmatter. `contract.json` lists `[name, type, status, created, updated]` as required, with `[repo, branch]` recommended; neither `workflow_type` nor `url` appear. But `SKILL.md` line 126 documents the file-mode validator's company-specific check as `workflow_type, url`, and line 145 shows an example violation `missing required field "workflow_type"` — so the validator enforces these fields despite contract.json being silent. Reproduces every time a new tracker is scaffolded.
2. **Issue 3 — Section-order drift.** Feature/story templates insert `## Acceptance Criteria` and defect templates insert `## Reproduction Steps` between `## Lifecycle` and `## Todos`. The contract's section-order check expects the order `Lifecycle, Todos, Log, PRs, Files, Meetings, Insights, Journal` — neither optional section is in the contract, and there's no `order_skip_absent` flag. So brand-new files fail validation on section order. (Defect tracker passes today only by accident — the validator has no explicit position for `Reproduction Steps`, so the failure is undefined-behavior.)

Frequency: **always**, on any fresh scaffold of feature, user-story, or defect.

These are the first and third of three drift symptoms originally filed under D000003 on 2026-04-16. Issue 2 (artifact duplication) was kept under D000003 because it's a pure manifest fix. Issues 1 and 3 are bundled here because they share the same architectural blocker (how to enforce template-vs-contract round-trip when the validators are LLM-driven, not executable).

## Reproduction Steps

1. Deploy company-workflow skill: `scripts/skills-deploy install`
2. In a consumer repo (ai-content), scaffold a fresh feature, user-story, and defect tracker per the skill's WORKFLOW.md
3. Open the generated `*_TRACKER.md` files
4. **Observe (Issue 1):** YAML frontmatter contains `workflow_type:` even though `contract.json` does not list it as required
5. **Observe (Issue 3):** `## Acceptance Criteria` (or `## Reproduction Steps` for defects) sits between `## Lifecycle` and `## Todos`
6. Run `/company-workflow validate` against the scaffolded files
7. **Observe:** validator flags Issue 3 (section order); Issue 1 only "passes" because the company-specific check uses an implicit superset of required fields

**Environment:** macOS Darwin 25.3.0; consumer repo ai-content @ master; skills-catalog source `local`; `/company-workflow` deployed via `skills-deploy install`.

## Root Cause

**Root cause:** Two coupled causes:

1. **Surface cause:** company-workflow's `contract.json` and `templates/company-workflow/*` were maintained as two independent sources of truth, with no automated round-trip test. Issue 1 is the required-frontmatter list and validator's implicit requirement set drifting. Issue 3 is the section-order rule and the templates that should satisfy it drifting.
2. **Structural cause (inherited blocker):** the validators (`/personal-workflow check`, `/company-workflow validate`) are LLM-driven SKILL.md files run by Claude Code, not executable scripts. There is no validator binary that bash CI can invoke. So the obvious fix — a round-trip fixture runner that scaffolds + execs the validator — is unimplementable as originally designed. This must be resolved before Issue 1 + 3 fixes can be ship-gated by an automated test.

**Location:**
- Surface: `skills/company-workflow/contract.json`, `skills/company-workflow/SKILL.md` (line 126), `templates/company-workflow/tracker-{feature,user-story,defect}.md`
- Structural: `skills/personal-workflow/check.md`, `skills/company-workflow/SKILL.md` (the entire validator-as-LLM-skill architecture)

## Fix Description

Two phases. Cannot start phase 2 until phase 1 is decided.

### Phase 1: Resolve the architectural rethink (BLOCKING)

Re-run `/office-hours` with the LLM-validator constraint surfaced upfront. Pick from five design-space options:

- **A. Subset-validator-in-bash** — runner re-implements contract.json structural checks (frontmatter required, sections required, section order, lifecycle phases) in pure bash + jq + grep. Catches Issues 1 + 3 fully. ~120 lines. Duplicates a subset of validator logic.
- **B. Pre-rendered fixtures + git-diff** — commit one scaffolded fixture per (skill, type). Runner re-scaffolds and diffs against the committed fixture. Catches template drift but not contract.json drift directly.
- **C. Drop the runner** — pre-commit lint that grep-checks templates against contract.json fields/sections. Less safety net but smaller surface.
- **D. Self-test inside the SKILL itself** — `/personal-workflow check --self-test` invocation runs the validator on freshly-scaffolded templates as part of the skill. Lives in Claude Code, not bash CI. Loses CI gate.
- **E. Shared bash library** — extract structural validation logic into `scripts/lib-validate.sh` that both the SKILL.md and the round-trip runner consume. Single source of truth, executable. Heaviest refactor but architecturally cleanest.

CI-gating preference favors A or E. Interactive-Claude-Code-only coverage favors D.

### Phase 2: Apply the contract/template fixes (after Phase 1)

**Issue 1.**
- Pre-step: grep ai-content for `workflow_type` to confirm universal presence. If absent anywhere, demote to recommended (grandfather policy). If present everywhere, required is safe.
- Add `workflow_type` to `contract.frontmatter.required` (or recommended per pre-step).
- Add `url` to `contract.frontmatter.recommended` (NOT required) for the same grandfather reason.
- Update `SKILL.md` line 126 to reference `contract.json` instead of inlining the field list.

**Issue 3.**
- Add `Acceptance Criteria` and `Reproduction Steps` to `contract.sections.optional`.
- Insert both in `expected_order` between `Lifecycle` and `Todos`.
- Add `order_skip_absent: true`.
- Add `type_specific_optional` mirroring personal-workflow's pattern. tracker-task and tracker-review are unaffected.

**Validation per Phase 1 choice:** run the chosen drift-detection mechanism and confirm both real skills pass.

Bump skill version. Update CHANGELOG with both contract changes.
