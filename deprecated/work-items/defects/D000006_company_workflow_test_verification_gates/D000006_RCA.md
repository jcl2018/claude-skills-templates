## Symptom

Two related gaps in `templates/company-workflow/` surfaced when walking through the Phase 2 → Phase 3 transition of a real work item:

1. **Tracker Phase 2 gates do not reference the linked test docs.** A defect tracker can be marked Phase 2 complete with `Regression test added` ticked even if no case in the sibling `test-plan.md` has been executed and marked Pass. A task tracker has no test-related Phase 2 gate at all, despite owning a `test-plan.md` per the manifest. A user-story tracker has `Acceptance criteria met` but never references the sibling `TEST-SPEC.md` Test Matrix.
2. **test-plan vs TEST-SPEC scope contract is undocumented.** Both templates ship without any top-of-file commentary stating their intended scope. New authors generating a task `test-plan.md` and a user-story `TEST-SPEC.md` from the same skill have no signal that one should be **concrete** (single fix or task) and the other **broader** (entire user story, mapped to PRD acceptance criteria across happy/edge/error paths).

Frequency: **always**, on every freshly scaffolded company-workflow tracker / test doc.

Reported during a `/company-workflow` walk-through. Both issues are pure template/contract content gaps; no runtime crash.

## Reproduction Steps

1. Open `templates/company-workflow/tracker-defect.md` Phase 2 — `Gates:` lists `Regression test added` (state, not verification)
2. Open `templates/company-workflow/tracker-task.md` Phase 2 — `Gates:` has no test-related entry; the manifest still requires the task to ship a `test-plan.md`
3. Open `templates/company-workflow/tracker-user-story.md` Phase 2 — `Gates:` lists `Acceptance criteria met`; never references `TEST-SPEC.md` even though that's the artifact mapping the AC → test cases
4. Open `templates/company-workflow/doc-test-plan.md` and `templates/company-workflow/doc-TEST-SPEC.md` — neither has a top-of-file comment stating intended scope
5. **Observe:** the lifecycle never gates on the linked test doc, and the test docs themselves never tell the author what level of detail to write at

**Environment:** macOS 25.3.0; templates as committed in `templates/company-workflow/` on branch `claude/nostalgic-volhard`.

## Root Cause

**Root cause:** The company-workflow lifecycle gates were authored to track **scaffolding completeness** (was the doc created? did a commit land?) rather than **content verification** (did the author actually run the cases listed in the linked test doc?). The test docs themselves describe their *structure* (matrix, table) but not their intended *scope*. The result is two missing contracts:

1. The tracker → test-doc loop is open: trackers reference test docs implicitly (via the manifest) but never gate on their contents.
2. The test-doc scope contract is implicit: a `test-plan.md` for a defect and a `test-plan.md` for a task render from the same template with no scope guidance, and neither template explains how it differs from `TEST-SPEC.md`.

**Location:**
- `templates/company-workflow/tracker-defect.md:22-28` — Phase 2 Gates list
- `templates/company-workflow/tracker-task.md:23-26` — Phase 2 Gates list (no test gate)

## Fix Description

Two-part template-only fix; no validator changes required for the minimum landing.

### Part 1 — Phase 2 test-verification gates

Edit each tracker template's Phase 2 `Gates:` list:

- `tracker-defect.md`: change `Regression test added` to `Regression test added AND all cases in test-plan.md marked Pass`
- `tracker-task.md`: add new gate `All test cases in test-plan.md marked Pass`
- `tracker-user-story.md`: add new gate `All P0 cases in TEST-SPEC.md marked Pass; remaining cases marked Pending/Skip with reason`
- `tracker-feature.md`: add new gate `Each child user-story's TEST-SPEC has all P0 cases Pass`

These are author-asserted gates, matching the rest of the lifecycle today. They do not require validator changes. If validator enforcement is desired later (post-D000004 architectural choice), the gate text becomes the assertion target for a `## Test Verification` section.

### Part 2 — test-plan vs TEST-SPEC scope contract

Add a top-of-file comment to each test-doc template stating intended scope:

- `doc-test-plan.md`:
  ```markdown
  <!-- Scope: ONE fix (defect) or ONE task. Cases must be concrete and reproducible.
       For broader coverage of a user story, use TEST-SPEC.md instead. -->
  ```
- `doc-TEST-SPEC.md`:
  ```markdown
  <!-- Scope: ENTIRE user story. Test Matrix must cover every PRD acceptance criterion
       across happy/edge/error paths. For a single fix or task, use test-plan.md instead. -->
  ```
- `doc-test-plan.md`: also generalize the title placeholder from `{Defect Name} — Regression Test Plan` to `{ITEM_NAME} — Test Plan` (UPPER_SNAKE per the canonical placeholder convention in WORKFLOW.md, NOT `{Item Name}` — the validator's `\{[A-Za-z_]+\}` regex misses any placeholder containing spaces, so `{Item Name}` would silently bypass unresolved-placeholder detection); generalize `parent: {DEFECT_ID}` → `parent: {ITEM_ID}` for the same reason. Mirror BOTH changes into `templates/personal-workflow/doc-test-plan.md` since personal-workflow shares the same artifact for tasks and defects
- `WORKFLOW.md`: add a one-paragraph "test-plan vs TEST-SPEC" subsection under Scaffolding Conventions

Mirror Part 2 into `templates/personal-workflow/doc-test-plan.md` and `templates/personal-workflow/doc-TEST-SPEC.md` so both skills enforce the same convention.

**Why not alternative fixes:**
- **Validator enforcement (contract.json sections.required → "Test Verification")** — heavier; would also need the section in every existing tracker; defer until D000004's architectural mechanism lands so this defect either rides it or doesn't duplicate it.
- **Add a separate `## Test Verification` section** — clean structurally but doubles the maintenance surface vs. one extra gate line; the gate-line approach matches the existing lifecycle pattern.
- **Rename `test-plan` and `TEST-SPEC` to clarify scope** — touches the manifest, every existing work item's filenames, and `contract.json`; cost is high vs. a one-line scope comment.
