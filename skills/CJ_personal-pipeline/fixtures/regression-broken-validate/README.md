# Fixture: regression-broken-validate

Step 6 — post-implement gate halt on validate.sh failure.

## What it tests

Orchestrator's post-implement gate catches a validate.sh failure and halts
before Phase 3 dispatches. Confirms:
- Failure output captured to tracker journal as `[gate-red]`
- AUQ surfaces with abort/fix/override options
- Telemetry end_state=halted_at_gate

## Setup

This fixture is harder to inject deterministically — it depends on the implement
subagent producing output that breaks validate.sh. Two approaches:

### Approach A (recommended): poison the catalog mid-pipeline

1. Create a fixture design doc that asks for a no-op work-item:

   ```bash
   PROJECTS=~/.gstack/projects/jcl2018-claude-skills-templates
   FIXTURE_DOC="$PROJECTS/fixture-broken-validate-$(date +%s).md"
   cat > "$FIXTURE_DOC" <<'EOF'
   # Design: Broken-validate fixture

   Generated by /office-hours on 2026-05-09 (FIXTURE)
   Branch: feat/CJ_personal-pipeline
   Status: APPROVED
   Mode: Builder

   ## Problem Statement
   Fixture: produces a work-item the orchestrator can implement, but the
   /CJ_implement-from-spec dispatch is rigged to leave the catalog in a broken
   state to verify the post-implement gate halts.

   ## Recommended Approach
   Touch a marker file. (The poisoning happens via in-pipeline manual edit; see Setup step 2.)

   ## Components Affected
   | Component | Repo | Change Type | Description |
   |-----------|------|------------|-------------|
   | tests/.broken-validate-marker | claude-skills-templates | New | One-line file |
   EOF
   ```

2. After Phase 2 dispatch but BEFORE the post-implement gate runs, you have to
   manually corrupt `skills-catalog.json` (e.g., remove a required field from
   any entry) in another shell. This is timing-sensitive and operator-driven.

   This is the v1 limitation: deterministic injection of a Phase 2 output that
   breaks validate.sh requires either a custom implement subagent prompt
   override (out of v1 scope) or in-pipeline operator action.

### Approach B: synthetic full-cycle outside the orchestrator

Manually run the 3 individual skills + corrupt the catalog at the same point:

```bash
# Skip the orchestrator; verify validate.sh halts in isolation
cp skills-catalog.json /tmp/catalog.bak.json
jq 'del(.[0].name)' skills-catalog.json > /tmp/broken.json
mv /tmp/broken.json skills-catalog.json
./scripts/validate.sh  # should fail
mv /tmp/catalog.bak.json skills-catalog.json  # restore
```

Then trace what the orchestrator's post-implement gate WOULD have done given
the same broken catalog. This doesn't actually exercise the orchestrator
end-to-end but verifies the gate logic in isolation.

## Invocation

Approach A:

```bash
/CJ_personal-pipeline "$FIXTURE_DOC"
# After "Phase 2 — implement subagent" log line appears, in another shell:
cp skills-catalog.json /tmp/catalog.bak.json
jq 'del(.[0].name)' skills-catalog.json > /tmp/broken.json
mv /tmp/broken.json skills-catalog.json
# Wait for post-implement gate; it should halt
mv /tmp/catalog.bak.json skills-catalog.json  # restore (after halt observed)
```

## Expected outcome

- Step 6 detects validate.sh exit != 0
- Tracker journal gets `[gate-red] post-implement validate.sh failed`
- AskUserQuestion: abort / re-dispatch / override
- On abort: telemetry end_state=halted_at_gate

## Cleanup

```bash
rm -f "$FIXTURE_DOC" tests/.broken-validate-marker
mv /tmp/catalog.bak.json skills-catalog.json 2>/dev/null  # safety
# Remove the fixture work-item dir if it was scaffolded before halt
rm -rf work-items/tasks/<fixture-id>/
```

## v2 follow-up

A deterministic injection harness (e.g., a fixture-specific subagent prompt
that instructs Phase 2 to deliberately break the catalog) would let this
fixture run end-to-end without operator timing. Defer until a real-world
post-implement gate failure surfaces and a regression test is needed.
