### Portability gate misses config-file deps read via trailing-arg or variable indirection — only the `[ -f ]` guard form is caught (P2, M)
`scripts/cj-portability-audit.sh` exists to catch a skill declaring a more-portable tier than its real deps justify (`skills-catalog.json` / `template-registry.json` / `VERSION` / `CLAUDE.md` are workbench-tier). But `is_exec()` (~L306-322) classifies a config literal as EXECUTED only when it sits immediately after a run-cue (`[ -f`, `bash`, `sh`, `source`, `.`, statement-start, `$(`). The dominant read idioms escape entirely: **trailing-arg reads** (`jq '...' skills-catalog.json`, `cat VERSION`, `grep x CLAUDE.md` — left-neighbor is the filter/command-name, not a run-cue) and **variable indirection** (`_CATALOG="$ROOT/skills-catalog.json"; jq ... "$_CATALOG"` — the literal is only on the assignment line, never on the executed read line). The ONE form caught is the existence-GUARD `[ -f skills-catalog.json ]`, which does not even read the file — the gate hears the knock but not the read. Proven by isolated A/B probes (a `standalone`-declared skill reading the catalog passes `portable` in both literal-direct and variable forms). Live false-negative: `CJ_document-release` (declared `local-only`) executes `jq ... "$_CATALOG"` against `skills-catalog.json` at `skills/CJ_document-release/SKILL.md:604` and the audit reports `portable` (FINDINGS=0). Never caught because every finding-positive test fixture plants only a ROOT-SCRIPT reach (`bash "$ROOT/scripts/x.sh"`) — a separate, WORKING detection path — so the config-read path is unexercised. D000032 deliberately TIGHTENED `is_exec` to kill a quoted-seed-literal false positive (a `"CLAUDE.md",` array element the engine WRITES); that tightening overshot and now misses real trailing-arg reads too.
**Root cause (verified):** `is_exec` inspects only the token immediately LEFT of the literal.
