There's no clean, repeatable way to uninstall (disable) a `status: deprecated` skill from a machine — `skills-deploy remove` can't do it when the deployed copy is foreign-owned, and there's no "remove all deprecated" command. Gaps:
- **Ownership refusal.** `do_remove` skips any skill whose deployed file isn't owned by the invoking checkout (the `owner != "ours"` guard); `--force` only bypasses the *reverse-dependency* check, not ownership. When the deployed symlink points into a different checkout (parent root / another worktree), a worktree-invoked remove is a no-op. (Same ownership-from-invoking-checkout root as the deploy-install row below + PR #172.)
- **Symlink-only deletion.** `do_remove` only unlinks symlinks; it won't clean a copy-based install.
- **No bulk/deprecated selector.** You must name each skill; there's no `--deprecated` selector that finds all `status: deprecated` catalog entries and removes them.
- **The working fallback (what we did this session):** manually `rm -f ~/.claude/skills/<name>/*` (the symlinks) + `rmdir` the dir + `jq 'del(.skills["<name>"])'` on `~/.claude/.skills-templates.json` (atomic write). A workflow should encapsulate exactly this.
**Scope to decide:** (a) a `skills-deploy remove --deprecated` (or `purge-deprecated`) subcommand that removes all `status: deprecated` skills' deployed copies + prunes the manifest, idempotent, with a `--force-ownership` escape for foreign-owned symlinks; and/or (b) a small `/CJ_*` skill wrapping it for one-keystroke use.
